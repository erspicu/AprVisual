# 別動被測物 —— 一次「探針效應」的教訓,與 Gemini 的儀器級修正通則

> 教學向文章。講一個我們真的踩到的坑:為了讓一個測試通過,我們對模擬電路「動了手腳」,
> 結果弄壞了另一個**完全無關**的測試 —— 而那個手腳其實一次都沒作用過。接著講 Google Gemini
> 給的一套通則怎麼把整件事導正,以及任何寫模擬器 / EDA / 離散事件系統的人都能帶走的原則。
> English version: `2026-07-08-probe-effect-instrument-grade-shims.en.md`。

## 1. 背景:我們在做什麼

AprVisual.S1 是把 NES 的兩顆晶片(2A03 CPU、2C02 PPU)**逐電晶體**模擬的開關級(switch-level)
引擎 —— 直接跑 Visual6502 式的真晶粒 netlist。我們拿它去跑 blargg 等人的硬體精確度測試 ROM,
絕大多數直接通過。

少數「差一點」的測試,需要我們補上一小段**行為層修正(shim)**。原因是:那份正確行為,住在
「純 CPU+PPU 開關級傳播」根本表達不出來的地方 —— 類比物理(電荷衰減、上電閂鎖沉降)、
**同半週期賽跑**、或雙晶片板級整合裡約 1 dot 的走線延遲。真機對這些都有明確答案,我們只是
把那份規格補回來。

問題來了:**shim 要「怎麼掛」上去,才不會弄壞別的東西?** 這篇就是在講這個「怎麼掛」的血淚。

## 2. 出事了:一個「零開火」的 shim,弄壞了無關的測試

我們要修 `double_2007_read`。第一版做法很直覺 —— **在 netlist 圖裡「加幾個節點和電晶體」**
來實現修正邏輯(9 個假節點 / 假電晶體)。

詭異的是,這個 shim 在那次執行中**一次都沒觸發**(zero fire)。照直覺,「沒作用 = 沒影響」。

結果:一個**完全無關**的測試 `dma_2007_read`,從 PASS 翻成一個**真機根本不可能產生**的
圖樣(讀出 X=00)。

> 一個從不開火的東西,怎麼會弄壞別人?

## 3. 為什麼:改了圖,就重擲了「對齊彩票」

引擎在載入時會做 **class-major 節點重編號** —— 為了效能,把節點依類別重排成一塊塊連續的 id
區間。這代表:

1. 你一旦「加了節點」,**每一個節點的 id 都位移了**。
2. 而 id 的順序,正是引擎每半週期 settle 波(把電路解到穩態的鬆弛迴圈)**波內的處理順序**。
3. 偏偏有一整類測試,結果取決於**同半週期賽跑** —— 例如「DMA 停頓 vs 匯流排讀取」誰先誰後。
   這種賽跑在二值穩態模型裡只能被**單向**裁決,而裁決往哪邊倒,對 settle 順序敏感。

把三點串起來:

> **加幾個假節點 → 全體 id 位移 → settle 波內順序改變 → 那場同波賽跑翻面 → 無關測試翻紅。**

關鍵領悟:**任何載入期的圖結構變更,都會替「每一個同波賽跑類測試」重擲骰子。** 就算你加的
東西從不作用也一樣 —— 因為傷害不在「它做了什麼」,而在「它讓圖變了」。

打個比方:這像在一台精密天平**旁邊**多放一顆螺絲。你沒去碰天平的秤盤,但整個系統的平衡點
已經悄悄改了。

## 4. 第二個錯:per-test 過度擬合

踩到坑之後,有個很自然、但一樣錯的反射動作:「那我讓這個 shim **只對這一個測試**開啟就好
(per-test scope)。」

Gemini 一句點破:這是**過度擬合 / 技術債**。

道理是:一個代表「真實物理傳播延遲」的 shim,是**那台機器的性質**,不是「某個測試的性質」。
它應該對**所有**測試一致生效。如果你 per-test,等於在說「這個物理現象只在跑這個測試時存在」——
那不是模擬,是對著答案作弊。

## 5. Gemini 的通則:儀器化,別改被測物(DUT)

核心一句話:**Instrument, don't modify the DUT(Device Under Test)。**

把被測電路當成實驗室裡桌上的一顆待測晶片。你不會為了量它,而在它身上**焊一塊新電路**——
你會用**探針 / 夾具**從外面施力,量完就收。

### 5.1 絕對覆蓋式 force / release(EDA 標準做法)

測試儀器需要的是「**絕對覆蓋權**」,不是改線路。EDA 業界的標準答案是 VPI 式的 `force` / `release`:
暫時把某個節點釘在某個值,結束後放開,**完全不動電路拓樸**。

### 5.2 實作:InstClampLow / InstClampHigh / InstRelease(零圖足跡)

我們照這個通則做了三個 runtime API:直接把 `Gnd`(或 `Pwr`)旗標 **OR 進**目標節點。群組解算時,
這等價於「這個節點接了一條導通的 vss 路徑」—— 落在群組解析 LUT 的**最高優先序**,壓得過主動驅動。

**關鍵:零圖變更。** 沒有新節點、沒有新電晶體、id 不位移、settle 順序不變 → **不重擲任何彩票**。
效能路徑(benchmark)的 golden checksum 掛不掛這些 shim 都**逐位元相同**。

紀律:只在測試模式作用;只施加在沒有靜態 Gnd 的節點;釋放責任在呼叫者(reset 重建旗標時
自然清掉殘留)。

### 5.3 全域,而非 per-test

runtime 儀器級 shim 既然是「絕對覆蓋、零圖足跡」的力,就該**全域生效** —— 它代表機器的物理,
對每個測試都成立。

實測驗證:把三個這類 shim(double_2007、even_odd 寫路徑延遲、OAM-DMA 匯流排持值)全部改成
**測試模式全域預設**,再跑一次完整 146 測回歸 —— **零附帶傷害**:該過的過、該紅的還是只有那
一個忠實偏差。

### 5.4 真正該「換」的東西:Socket Pattern(插座模式)

有一種變更是**正當**的:換掉可插拔的**治具(fixture)**,例如手把 —— 它本來就是外接週邊。
但「換手把」在我們的引擎裡**也是載入期圖變更**,一樣會踩探針效應。Gemini 的解法是 Socket Pattern:

1. **兩階段載入**:Phase 1 只載入不可變的 DUT(主機 + 手把**插座邊界**),做完 class-major
   重編號、**鎖死 id 0~N**;Phase 2 才依需要掛上行為手把。
2. **尾端掛載**:治具的所有新節點,一律分配 **> DUT 最大 id** 的尾端編號。
3. **邊界對接**:治具的輸出 pin,接到 Phase 1 已經定型的插座節點。

結果:**插不插手把,DUT 內部的 id / 圖 / BFS 展開樹 100% 同構** → 零探針效應。手把於是可以
「全域可用」而不動任何對齊。

### 5.5 分清楚:DUT 修正 vs 治具變更

Gemini 幫我們把兩類變更分開,這是整件事的思考骨架:

| 變更 | 本質 | 正解 |
|---|---|---|
| tie 改接(板卡把本應浮接的 pin 綁到 vss)| **DUT 建模瑕疵** | **全域**修正(不能 per-test,否則等於用「錯版主機板」跑其餘測試)|
| 換手把模組(閘級 → 行為級)| **治具變更**(可插拔外設)| per-test 概念正確,但用 **Socket Pattern** 做到零擾動 |

### 5.6 防呆:Graph Fingerprint(圖指紋)

本次 bug 的本質 = 「無意間改了圖,卻沒意識到要重驗對齊敏感的測試」。解法很簡單也很有效:

- Phase 1 完成後(掛任何治具**之前**)算一枚 DUT 指紋:節點總數 + 邊總數 +(嚴謹版)排序後
  adjacency 的 hash。
- 寫死進測試框架;每次跑前驗證,不符就丟 `DutGraphMutatedException`。
- 效果:下次有人動了 board def / 圖結構,CI 立刻報「**對齊已重擲,請重驗 ppu_vbl_nmi 並更新
  指紋**」—— 把**未知的隨機失敗**變成**已知的預期變更**。

## 6. 結果

三個 runtime 儀器級 shim 全域化後,乾淨全量回歸 **146/1(99.3%,147 測)**,零附帶傷害。唯一殘留的 FAIL
是一個**真正的忠實偏差**(`cpu_dummy_writes_oam` —— RP2C02G 版次專屬的 OAMADDR 寫入損毀 bug,
真 G 版硬體本來就過不了),那是「該紅才對」,不在本文討論範圍(詳見忠實偏差 Q&A)。

## 7. 可帶走的原則(給寫模擬器 / EDA / 任何離散事件系統的人)

1. **不要為了量測而改動被測物。** 用外掛的 force / release,不要改拓樸。
2. **警惕「零開火也有害」。** 傷害可能來自你「加了什麼」,而不是「它做了什麼」—— 尤其當引擎
   對節點順序敏感時。
3. **物理性質要全域,別 per-test。** per-test 才通過 = 過度擬合,不是模擬。
4. **分清 DUT bug 與治具變更。** 前者全域修,後者用插座模式換。
5. **把不可變性寫成檢查。** Graph Fingerprint 讓「悄悄改了圖」變成「CI 立刻尖叫」。
