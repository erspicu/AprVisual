## ① 設計描述

### 1.1 合成規則（依 NMOS 解析優先鏈：GND 贏 → VCC/pull-up → 外部驅動 → 保留前值 → 0）

對 node N（有 `DriveInfo` di）：

**(A) `di.Hybrid` → `nextExpr[N] = null`**（= 這個 node 由 S2.5 的 hybrid bridge 用 switch-level 算；IR 引擎看到 `nextExpr == null` 就知道）。

**(B) 非 hybrid**：拆 di 的「驅動源」，照優先序組：
- `pd = di.PullDown`（可能 `null` = 無下拉路徑，或一個 `Expr`；不會是 `Complex` —— Complex 已被標 hybrid）。
- `inPasses` = `di.Passes` 裡 `OwnerDrives == false` 的（= 對方驅動 N）；非 hybrid 的 N，passes 的 `OwnerDrives` 都非 null（null 的那些 —— bidirectional / charge-sharing / 碰 bus —— 都已 `MarkHybrid`）。`OwnerDrives == true` 的（N 驅動對方）不影響 N 的值，忽略。
- `puFallback` = pull-up 帶來的「沒被下拉、沒被 pass 驅動時」的值：
  - `StaticLoad` / `StrongVcc` → `Const(true)`（恆拉高；弱或強對布林值都一樣）
  - `Conditional` → `Mux(di.PullUpCond, Const(true), Hold(N))`（precharge：clock on → 1，否則 hold —— domino 求值後保持的 1）
  - `None` → `Hold(N)`（純 dynamic：沒驅動就 hold）
- **組起來**：
  - `passChain` = `inPasses` 由後往前折：`fallback`；對每個 `(M, cond)` → `Mux(cond, Node(M), 上一層)`。沒 inPasses → `passChain = puFallback`。
    - ≥2 個 inPasses：first-wins 近似（真正並導通是 contention，但乾淨設計不會 —— 它們是 mux，互斥）。**S2.2 先這樣，note 一下。**（→ 也許 Gemini 建議：≥2 inPasses 且條件非顯然互斥 → 標 hybrid？）
  - `pd == null` → `nextExpr[N] = passChain`（沒下拉路徑 → 永遠不會被拉 0）。
  - `pd != null` → `nextExpr[N] = Mux(pd, Const(false), passChain)`（下拉導通 → 0；GND 贏，蓋過 pull-up 和 pass —— 因為導通的 pass 把 M 併進同一個 group，但 N 自己的下拉讓 group 有 Gnd flag → 0）。
- **特例化（讓常見情況的 Expr 乾淨）**：
  - `pd != null && pu ∈ {StaticLoad, StrongVcc} && inPasses 空` → `nextExpr[N] = Not(pd)` ← 教科書 inverter / NAND / NOR / AOI（`Mux(pd, false, true)` 由 `Mux` 的 smart ctor 收成 `Not(pd)`）。
  - `pd == null && pu ∈ {StaticLoad, StrongVcc} && inPasses 空` → `Const(true)`（恆拉高、無下拉）。
  - 其餘照上面的通式。

> 關鍵語義（為什麼 pull-down 蓋過 pass、pass 蓋過 pull-up）：switch-level 裡導通的 pass 把 N 和 M 併成同一個 group → group 值 = `resolve(group 內所有 flag)`。N 自己的下拉導通 → group 有 Gnd flag → 0（不管 M）。N 沒下拉但 pass 導通 → group = {N, M}，N 的 pull-up 是弱 flag、M 的值是 State/PullUp/Gnd flag → M 的值蓋過 N 的弱 pull-up（M=0 → 0；M=1 → 1）。N 沒下拉、pass 沒導通 → group = {N}，pull-up → 1（或 hold）。

### 1.2 組合 vs 序向（初判；S2.3 精修）

- N 是**序向**若 `nextExpr[N]` 裡出現 `Hold(N)`（它保留自己的前值 → latch）。
- 否則 N 暫標**組合**。
- **註**：cross-coupled latch（N 的 expr 引用 /N、/N 的 expr 引用 N —— 誰都沒 `Hold(self)`，但兩個合起來是個環）—— 這需要 dependency graph + SCC，是 **S2.3** 的事。S2.2 只做「有 `Hold(self)` → 序向」的初判；S2.3 會把 SCC 裡的 node 也標序向、把環打斷成 `Prev(...)`。
- hybrid node（`nextExpr == null`）→ 既不算組合也不算序向（它走 switch-level）。

### 1.3 IR 覆蓋率

- 也報：`Hold(self)` 引用了哪些 node（序向 node 初判數）、`nextExpr` 是 `Const`/`Not(...)`/`Mux(...)`/`Hold(...)` 各幾個（結構分佈）。

這是一份針對 S2.2 設計的務實、具體的技術 review。整體來說，S2.2 的邊界劃分得非常乾淨（明確把 SCC 和模擬器建置推遲到後續步驟），這能讓這個步驟專注於「單一節點的本機邏輯推導」。

針對你的 5 個開放問題與設計細節，以下是具體評估與建議：

### 1. 合成規則（§1.1）與優先序：Pull-down > Pass > Pull-up
**結論：完全正確，精準對應 Switch-level 的 Group Resolution，不用改。**

*   **原理解析：**
    *   當 Pass 導通時，N 與 M 被併入同一個 Group。
    *   如果 **N 自己有 Pull-down 導通**：整個 Group 會獲得強烈的 Ground Flag（因為 NMOS 傳遞 0 是無損的）。此時無論 M 那邊傳來什麼（即使 M 有 Pull-up 或被其他邏輯驅動為 1），強 0 都會贏。所以 `pd != null -> 0` 是對的，PD 絕對蓋過 Pass。
    *   如果 **N 沒 Pull-down 但 Pass 導通**：N 的弱 Pull-up（如果有）會被 M 傳遞過來的 Driven State (0 或 1) 蓋過。所以 Pass 蓋過 Pull-up 是對的。
    *   如果 **都沒導通**：退回 N 自己的 Pull-up 或 Hold。
*   **有沒有漏掉的情況？**
    如果 M 本身也是 floating 或只有弱 Pull-up 怎麼辦？你在 S2.1 的 `OwnerDrives` 與 `Hybrid` 標記已經防堵了這個問題。非 Hybrid 的 node，其 inbound pass 對方必定有明確的強驅動值。這套邏輯完美自洽。

### 2. ≥2 個 inbound pass 的處理（First-wins vs Hybrid vs 條件 Resolve）
**結論：使用 First-wins（巢狀 `Mux`）是正確且最佳的選擇。不要標 Hybrid。**

*   **為什麼 6502/2C02 會有這個？** 這是標準的 Multiplexer (Mux) 結構。例如 ALU 的輸入選擇、或多個暫存器寫入同一條內部 bus。
*   **為什麼 First-wins 是對的？** 在乾淨的數位設計中，這類 Mux 的控制訊號**必然是互斥的（One-hot）**。如果同時有兩個導通且值不同，在實體電路上就是 Short-circuit（Contention）。既然控制條件互斥，那麼 `Mux(cond1, M1, Mux(cond2, M2, fallback))` 的**順序根本不重要**，結果邏輯上是等價的。
*   **對計畫的影響 / 具體行動：** 照你的計畫寫 `passChain` 迴圈摺疊即可。在註解標明「Assumes mutually exclusive pass selects (one-hot)」。不要為此 fallback 到 Hybrid，否則你的 IR 覆蓋率會大幅下降（很多純邏輯的資料路徑會被誤判）。

### 3. Precharge/Domino Node 的公式
**結論：公式 `Mux(pd, False, Mux(precharge_cond, True, Hold(N)))` 完全正確。**

*   **2C02 的實際情況：** 2C02 的 OAM/Sprite 邏輯中，確實有大量 Domino logic。`precharge_cond` 通常來自時鐘（如 `clk0`）或由時鐘 gating 出來的訊號。在預充期（Precharge phase），Pull-down 路徑的求值電晶體通常會被 Clock 關閉（互斥），所以 `pd` 和 `precharge_cond` 在正常運作下不會同時為 True。
*   **公式的防禦性：** 如果因為某種怪異情況導致 `pd` 和 `precharge_cond` 同時為 True，你的公式讓 `pd` (False) 贏。這符合 NMOS 下拉強於 PMOS/Depletion 上拉的物理特性（Ground 贏 VCC）。
*   **具體行動：** OK，照做。這個公式非常漂亮且通用。

### 4. 組合 vs 序向的初判（只看 `Hold(self)` 夠嗎？）
**結論：對於 S2.2 來說，絕對夠了。**

*   **邏輯分析：** 在排除 Cross-coupled 迴圈（S2.3 負責）的前提下，一個節點如果能「記住前一拍的狀態」，在布林邏輯中**唯一的表現形式**就是它的 Next-state 表達式依賴於它自己（即 `Hold(self)`）。這涵蓋了所有 Dynamic Latches（純靠寄生電容 hold 值的地方）。
*   **對計畫的影響：** 這樣做能清楚分離「Dynamic Latch（本機即可判定）」與「Static Latch / SRAM（需要全域圖論分析）」。S2.2 的職責很清晰。

### 5. `Expr` 的 Smart Constructor (idioms)
**結論：需要補上與 `False/True` 結合的邏輯閘化簡，這對降低 IR 複雜度極度重要。**

你已經提了 `Mux(c, True, False) = c` 和 `Mux(c, False, True) = Not(c)`。
但因為你有 `Mux(pd, False, passChain)` 這種結構，當 `passChain` 被化簡為某個訊號時，會出現大量 `Mux(cond, False, B)` 的形式。請務必補上以下 Idioms：

*   **具體行動：在 `Expr.Mux` 裡加入這 4 條化簡規則：**
    1.  `Mux(c, False, b)  => And(Not(c), b)`    *(如果 c 發生則為 0，否則看 b)*
    2.  `Mux(c, a, False)  => And(c, a)`         *(如果 c 發生看 a，否則為 0)*
    3.  `Mux(c, True, b)   => Or(c, b)`          *(如果 c 發生則為 1，否則看 b)*
    4.  `Mux(c, a, True)   => Or(Not(c), a)`     *(如果 c 發生看 a，否則為 1)*
*   **為什麼重要？** 很多標準的 AOI/OAI 閘或帶有 Enable 的邏輯，在經過 Pull-down 樹和 Pass 樹的解析後，最初會長得像巢狀 Mux。這些規則能把它們瞬間壓平成標準的 `And/Or/Not` 樹，不僅印出來可讀性大增（給 Spot-check 用），S2.4 直譯器的效能也會更好。
*   至於 `Mux(c, a, Hold(self))`，**不要化簡**，保留 Mux 語義，這是 D-Latch / Enable-Latch 的標準特徵，保持原樣對後續分析最有利。
