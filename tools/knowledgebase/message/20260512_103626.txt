# S2.1 — 導通分析 / `DriveInfo`（每個 node 的「驅動結構」）

> S2 的第 1 步：用 S2.0 的 `NetlistGraph`（node 角色 + transistor kind + pass 方向欄位），算出每個非 `Input`/`Supply` node 的「驅動結構」`DriveInfo` —— 它被拉到 GND 的布林條件、它的 pull-up 種類、透過 pass 連到哪些別的信號 node（含方向）。**還不組 `Expr` 的 next-state（那是 S2.2）；還不碰 SCC（S2.3）；不改模擬行為。**
>
> 工作流程（`00_計畫總覽.md` §3）：① 設計描述（本段）→ ② 丟 Gemini 分析 → ③ 評估 & 採用 → ④ 實作 → ⑤ 驗證。
>
> **承接 S2.0 學到的**：depletion load 在這份網表裡**不是電晶體、是 `node.Pullups > 0` 標記**（→ static pull-up 看 `Pullups`，不要去 `TransistorKind` 找 `PullUpLoad`）；`PullUpStrong` 很少（5 顆，= S1.5 留的「綁到 vcc」connection）；`PullUpActive` ≈ 897 顆（真正「條件拉到 vcc」的電晶體 = precharge / push-pull / 動態邏輯）；`PullDown` ≈ 16126（直接到 GND 的葉子）、`Pass` ≈ 9747（transmission gate + 下拉鏈中段）。`Input` = 72（含 handler hook node / reset 網路 / 板上腳 / gate-only 內部 node —— 這些 S2.1 直接跳過、不算 `DriveInfo`）。

---

## ① 設計描述

### 1.1 目標 & 輸出

對每個 node `N`（`role ∈ {Internal, Bus}`，跳過 `Supply` / `Input`）算 `DriveInfo[N]`：

```csharp
sealed class DriveInfo
{
    public Expr? PullDown;          // 「N 被拉到 GND」的布林條件（沿 PullDown+Pass 子圖做串並聯歸約）；
                                    //   null = 無下拉路徑；ComplexExpr = 歸不出乾淨式子（→ S2.2 標 hybrid）
    public PullUpKind PullUp;        // None / StaticLoad（node.Pullups>0）/ StrongVcc（有 PullUpStrong 電晶體）
                                    //   / Conditional（有 PullUpActive 電晶體 → 條件式；Expr = OR 那些 gate）
    public Expr? PullUpCond;         // 只有 PullUp==Conditional 時有意義
    public List<PassLink> Passes;    // 透過 Pass 電晶體連到的別的信號 node
    public bool Hybrid;              // 這個 node 的驅動結構太亂、抽不出乾淨式子 → 整個 node 留 switch-level
    public string? HybridReason;     // debug 用
}
enum PullUpKind { None, StaticLoad, StrongVcc, Conditional }
struct PassLink { public int Other; public Expr Cond; public PassDir Dir; }   // Dir 從 NetlistGraph.PassDirection 來（這步會把 Unknown 推導掉）
```
`Expr` 型別本身在 S2.2 才正式定（這步先用一個最小子集：`ConstExpr(bool)` / `NodeRefExpr(int)` / `AndExpr` / `OrExpr`；`ComplexExpr` 當 sentinel「歸不出來」）。S2.1 不引入 `NotExpr` / `MuxExpr` / `HoldExpr`（那些在 S2.2 組 next-state 時才用）。

### 1.2 `PullDown` 的算法 —— 串並聯歸約 + 路徑展開 fallback

「N 被拉到 GND」= 在「目前哪些電晶體 ON」這個 symbolic 設定下，存在一條從 N 到 `vss` 的導通路徑。把相關電晶體當邊、gate signal 當邊的「啟用條件」：

- **相關電晶體** = N 的 `C1c2s` 裡 kind 為 `PullDown` 或 `Pass` 的（`PullDown` 的另一端是 `vss`；`Pass` 的另一端是別的信號 node —— 要遞迴往那邊找到 `vss`）。
- **遞迴邊界**：往 `Pass` 的另一端 `M` 走時，只有當 `M` 自己是「下拉網路的一部分」才繼續（heuristic：`M` 有到 GND 的下拉路徑、且 `M` 沒有自己的 pull-up、`role` 不是 `Bus`）。若 `M` 看起來是個獨立的被驅動 node（有 pull-up，或本身是別人的閘輸出）→ **不要把它當成 N 的下拉網路的一部分**；那個 `Pass` 是真的 transmission gate（mux），歸到 `Passes` 裡，不歸到 `PullDown`。→ 這是「pass transistor 串聯下拉鏈」vs「pass transistor mux」的關鍵 disambiguation（S2.0 把這種電晶體一律標 `Pass`，S2.1 在這裡分開）。
- **串並聯歸約**：從 N 開始 DFS 到 `vss`，把走過的子圖做 series-parallel reduction：
  - 一條鏈（每個中間 node 度數 = 2、只在這條鏈上）→ 鏈上所有電晶體 gate 的 **AND**。
  - 同一對端點之間的多條鏈 → **OR**。
  - 遞迴歸約。`gate == Npwr` 的電晶體 → 該邊條件恆 `Const(true)`（在 AND 裡可省、在 OR 裡讓整個 OR 變 true）。
- **非串並聯（橋接）fallback**：若子圖歸不成純串並聯（Wheatstone-bridge 式結構）→ 退成「枚舉所有 N→vss 的簡單路徑，每條路徑 = 路徑上 gate 的 AND，全部 OR 起來」。設路徑數上限（例如 64）；超過 → `PullDown = ComplexExpr`、`Hybrid = true`、`HybridReason = "pulldown net not series-parallel and >64 paths"`。
  - 註：6502/2C02 的下拉網路絕大多數是乾淨的串並聯（NMOS AOI 結構）；橋接式很少。先用 64 上限，看實際命中率再調。

### 1.3 `PullUp` 的判定

對 node `N`：
- `node.Pullups > 0` → `PullUp = StaticLoad`（depletion load 標記；弱拉高，輸給任何下拉）。
- N 的 `C1c2s` 裡有 kind == `PullUpStrong` 的電晶體（gate==Npwr、另一端 vcc）→ `PullUp = StrongVcc`（無條件接 VCC —— 但因 NMOS 的「GND 贏」優先鏈，行為上跟 StaticLoad 一樣是「下拉導通則 0、否則 1」，差別只在 bus-fight strength，那種情況反正 hybrid）。
- N 的 `C1c2s` 裡有 kind == `PullUpActive` 的電晶體（gate 是信號）→ `PullUp = Conditional`、`PullUpCond = OR(那些電晶體的 gate)`（precharge / 動態 / push-pull 上半）。
- 以上同時成立的優先序：`StaticLoad` / `StrongVcc` 蓋過 `Conditional`（有靜態上拉的話，conditional 那個是多餘的；實際上一個 node 通常只有一種）。都沒有 → `PullUp = None`（純 dynamic / pass 驅動的 node）。

### 1.4 `Passes` + pass 方向推導（Gemini §7-2 採用）

對 N 的每顆「不算進下拉網路」的 `Pass` 電晶體（另一端 `M`、gate `g`）：加 `PassLink { Other = M, Cond = NodeRef(g), Dir = ? }`。方向 `Dir`：
- N 有下拉網路（`PullDown != null && != Complex`）或有 pull-up，而 `M` 是「純 dynamic」（無下拉、無 pull-up、`role` 非 `Bus`）→ `Dir = AtoB`（A = 有 driver 那端 = N → 驅動 M）。從 M 的角度看同一顆 → `Dir = BtoA`（一致）。
- 兩端都有 driver（或都是 `Bus`）→ `Dir = Bidirectional`、那個 pass 網路 → `Hybrid = true`（兩端的 node 都標 hybrid），`HybridReason = "bidirectional pass between two driven nodes"`。
- 兩端都「純 dynamic」（charge sharing）→ `Dir = Bidirectional`、`Hybrid = true`、`HybridReason = "pass between two floating dynamic nodes (charge sharing)"`（Gemini §7-5：6502 真依賴的場景極少，先警告 + hybrid，看 test ROM）。
- 一個 `Bus` node 連到一個普通 node 的 pass：方向看普通 node 那邊有沒有 driver；通常 `Bus` 是「外部 + 內部都驅動」→ 偏 `Bidirectional` → hybrid（呼應 S2.0 把 `cpu.db`/`ppu.io_db` 標 `Bus` 的理由）。

> 這步順便把 `NetlistGraph.PassDirection[]` 從 `Unknown` 填掉（推不出的留 `Bidirectional`）。

### 1.5 `BusResolver.RefineBuses()`（S2.0 留的待辦，這步做掉）

用 `(N 有內部 driver：`PullDown != null` 或 `PullUp != None` 或被 `AtoB` 的 pass 驅動) ∩ (N 被某個行為級 handler `WriteBits`)` 把 `NetlistGraph.Role` 裡更多 node 標成 `Bus`。「被 handler `WriteBits`」的清單從哪來：`WireCore.Handlers.cs` 的 `AttachRamLikeHandler` / `AttachVideoHandler` 解析出來的 `_d[7:0]` / `d[]` / dataOut node list —— 這些目前沒存起來，這步要在那邊加個 `HandlerWrittenNodes` 集合（或讓 `AttachMemoryHandlers` 回傳 / 累積）。被新標成 `Bus` 的 node → 連帶 `DriveInfo.Hybrid = true`（雙向 bus 先 hybrid，呼應 §1.4 + Gemini §7-... bus-fight 沒 strength 維度）。

### Q1: 串並聯歸約 vs 路徑展開 Fallback vs BDD/Fixpoint
**結論：【設計 OK，保持串並聯 + Fallback，不需 BDD】**
*   **非串並聯（橋接）在 6502/2C02 下拉網路有多常見？**
    *   **極少。** 1980 年代的 NMOS 標準邏輯（AOI/OAI）下拉網路 99.9% 都是純粹的串並聯結構。
    *   如果你在 NMOS 網表裡看到「橋接」結構，那通常是 **Pass Transistor Logic（例如 ALU 的進位鏈、Barrel Shifter、或 BCD 校正網路）**，而不是直接通往 GND 的 Pull-Down 網路。
*   **對計畫的具體影響 / 該怎麼改：**
    *   不需要 BDD 或 Reachability Fixpoint（殺雞用牛刀，且難以轉回人類可讀的 Expr）。
    *   目前的 `Series-Parallel Reduction + 枚舉 Fallback` 是最佳解。
    *   **建議微調：** 將路徑上限從 64 **下修到 16 或 32**。因為如果一個節點到 GND 有超過 16 條路徑，它有 99% 的機率根本不是純下拉網路，而是你的 heuristic（Q2）判斷錯誤，把一大坨 Mux/Pass Logic 當成下拉網路展開了。提早報 `Complex` + `Hybrid` 可以幫你抓出分類錯誤的電晶體。

---

### Q2: 「Pass 是下拉鏈中段 vs Mux」的 Disambiguation Heuristic
**結論：【目前的 Heuristic 有危險，建議改用 `Fan-out (GateDriven)` 判定】**
*   **風險點：** 你的原設計是「看另一端 `M` 有無下拉、無 Pull-up、非 Bus」。這在 NMOS 動態邏輯會誤判。例如：一個 Precharge 節點（有 `PullUpActive`）透過串聯的 NMOS 求值並拉到 GND。如果你因為它有 Pull-up 就在往下拉的過程中停住，你會把一個完整的動態 NAND/NOR 結構腰斬。
*   **更好的物理特徵（The Missing Heuristic）：**
    *   在 NMOS 佈局中，**「下拉鏈的中間節點」純粹是電晶體串聯的物理節點，它絕對不會去驅動任何其他電晶體的 Gate**。
    *   反之，**「Transmission Gate / Mux 的輸出」是一個真正的邏輯訊號，它一定會去接某個電晶體的 Gate（Fan-out > 0）**。
*   **對計畫的具體影響 / 該怎麼改：**
    *   在遞迴找 VSS 時，當遇到 `Pass` 電晶體通往節點 `M`，是否繼續將其視為下拉網路的條件改為：
        `繼續遞迴 = (M 不是 Bus) && (M 沒有驅動任何電晶體的 Gate，即 M 在所有電晶體的 C1c2s 裡，作為 gate 的次數 == 0)`。
    *   如果 `M` 有驅動別的 Gate，它就是一個獨立的邏輯節點。這個 `Pass` 就是 Mux/Transmission Gate，歸入 `Passes` 集合，**停止該路徑的 PullDown 搜尋**。
    *   這個改動能完美切開「Stack 內部節點」與「訊號傳遞節點」，不會被 Precharge / Bootstrap 騙倒。

---

### Q3: Pass 方向推導 (Driver vs Dynamic vs Charge Sharing)
**結論：【設計極佳，完全切中 NMOS 特性】**
*   **6502/2C02 裡會不會把很多 Pass 留成 `Bidirectional` / `Hybrid`？**
    *   會有一些，但數量在可接受範圍內。真正的 Charge Sharing（例如某個 Register 寫入另一個獨立浮動的 Latch）在 6502 裡確實存在（主要由 Phase 1 / Phase 2 clock 隔離）。
    *   Bootstrap 節點在 Visual6502 中通常是被標記為 PullUpActive + 某種驅動，如果兩端都有 Driver 發生「打架（Fight）」，標記為 Hybrid 留給後續或 switch-level 處理是最安全的。
*   **對計畫的具體影響 / 該怎麼改：**
    *   原封不動保留這組 Heuristic。S2.1 的目標不是「完美解析所有方向」，而是「把顯然有向的 Pass 找出來（佔 90%+），剩下的難題用 `Hybrid` 框起來」。

---

### Q4: 雙向 Bus 一律標 `Hybrid` 會不會太保守？
**結論：【以 S2.1 的階段來說，這樣剛好，不要在這裡做 Mux】**
*   **分析：** 你提議的 `Mux(cpu_writing, cpu_drives_db, Hold(db))` 其實是 **「Next-State 的邏輯描述 (S2.2)」**，甚至牽涉到外部 Handler 的行為模型。
*   **S2.1 的職責是物理拓撲。** 物理上，`cpu.db` 就是一個同時被多方驅動、且可能帶有大量 Pass 連線的巨大節點。
*   **對計畫的具體影響 / 該怎麼改：**
    *   保持原設計：`RefineBuses` 後一律將這些 Bus 節點的 `DriveInfo.Hybrid` 設為 `true`。
    *   **不要在 S2.1 引入 `HoldExpr` 或 `MuxExpr`**。這會破壞 S2.1 只做「靜態導通分析」的純粹性。
    *   把「Handler 覆寫」跟「Bus Fight 解析」留到 S2.2 甚至 S2.4 去做。現在標 Hybrid 只是為了讓 DFS 歸約不要跨過 Data Bus 亂竄。

### Q5: DriveInfo 結構與 Expr 最小子集
**結論：【精準，沒有漏掉 Corner Case】**
*   **關於 `NotExpr`：** 你的直覺完全正確！在 NMOS Switch-level 中，一個反相的訊號 `/xxx` 本身就是一個獨立的 Node。它的下拉網路條件就是 `NodeRef(xxx)`。我們在描述**「導通條件（Conductance）」**時，本來就不需要 `Not`。`Not` 是電壓翻轉的結果（PullDown 導通 -> 節點電壓為 0 -> 邏輯 0），那是 S2.2 的事。
*   **對計畫的具體影響 / 該怎麼改：**
    *   維持現狀，S2.1 嚴格限制 `Expr` 只能有 `Const`, `NodeRef`, `And`, `Or`, `Complex`。
    *   這確保了 `DriveInfo.PullDown` 表示的純粹是「開關網路的串並聯布林函數」，而不是邏輯閘的抽象。

### 其他實作與驗證建議 (補充)

1.  **PullUpKind 的涵蓋範圍：**
    你的判定規則 `StaticLoad / StrongVcc 蓋過 Conditional` 非常正確。NMOS 的 Depletion 負載通常是常開的（Static）。如果有 `PullUpActive`，那通常是動態邏輯的 Precharge。兩者很少同時存在，若同時存在，Static 的物理特性確實會支配未驅動時的狀態。
2.  **Verify Plan 的 Expectation 修正：**
    在 §1.7 中提到 spot-check `Pass-mux` 節點。結合 Q2 的建議：一個真正的 Mux 節點，其特徵是它**「有驅動別人的 Gate」**。你可以在 `TestDriveAnalysis` 手刻測試時，故意讓 Mux 的輸出端去接一個假電晶體的 Gate，以驗證新的 Heuristic 能正確截斷 PullDown 搜尋並把它歸類為 `Passes`。
3.  **效能考量：**
    整片 NES 大約 14600 個 Internal Node。S2.1 的 DFS 加上串並聯歸約，用 C# 跑起來應該在 100 毫秒內就能完成。這是 O(V+E) 等級的操作，只要你的 `Complex` Fallback 提早（例如 32 條路徑）截斷，就不會有效能問題。
