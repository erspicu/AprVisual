這是一個非常經典且深入的硬體行為與模擬器架構交集的問題。你的假設非常精準，你觀察到的現象完全是模擬器實作架構導致的假影（Artifact）。

以下為你詳細剖析這個問題，並給出實作建議：

### 1. 這是根本限制還是模擬器的假影？
**結論：這是 100% 的模擬器時戳粒度假影，完全可以（也應該）修復。**

在真實 NES 硬體上，PPU open-bus 的電容放電是連續的類比物理過程。CPU 執行 `LDA $2001` 時，會在該指令的特定週期（絕對定址的第 4 個 CPU cycle）精確取樣資料匯流排上的電壓。
因此，真實硬體的量化誤差是 **1 次緊迴圈的時間（約 16 CPU cycles / 48 PPU dots）**，而不是 1 個 Frame。

**為什麼你的模擬器會出現 1 Frame 的量化？**
根據你提供的 `NowDots()` 算法，你的模擬器很可能採用了「批次執行（Batch Execution）」架構：CPU 一次跑完一整條 scanline 或一整幀（Frame），然後 PPU 再追上來（Catch-up）。
在那個緊迴圈中，CPU 一直在跑，但因為 PPU 還沒被排程執行，PPU 的內部狀態 `scanline` 和 `ppu_cycles_x` **處於凍結狀態**。直到 CPU 跑完了一幀的 cycle 配額，PPU 才更新。這導致你在迴圈中呼叫 `NowDots()` 時，拿到的永遠是「這幀開始時的靜止時戳」，衰減判斷只能在幀邊界觸發，從而產生 1858 次迴圈（約 1 幀）的巨大階梯。

---

### 2. 模擬器端如何修復？（改為逐 Dot / 逐 Master-Clock 細粒度）
要解決這個問題，不需要放棄 Lazy Evaluation，也不需要把模擬器改成極度耗能的 Cycle-accurate 交錯執行，只需要**改變「目前時間」的基準源**。

**實作解法：以 CPU 為基準的 Global Clock**
由於是 CPU 在主動輪詢（Polling），`NowDots()` 的計算不應該依賴 PPU 的狀態（`scanline` 等），而應該依賴 **CPU 已經執行的總週期數**。

1.  **維護全局時鐘：** 在 CPU 核心中維護一個 `uint64_t total_cpu_cycles`，每次 CPU 執行指令時，嚴格累加該指令消耗的 cycle 數。
2.  **改寫 NowDots()：**
    ```cpp
    // NTSC 系統下，1 CPU Cycle = 3 PPU Dots
    uint64_t NowDots() {
        return cpu->total_cpu_cycles * 3;
    }
    ```
3.  **Lazy 衰減偵測：**
    ```cpp
    // 當 CPU 寫入 $2002 時
    latch_stamp_dots = NowDots(); // 記錄當下時間

    // 當 CPU 讀取 $2001 時
    if ((NowDots() - latch_stamp_dots) >= decay_period_dots) {
        latch_value = 0; // 觸發衰減
    }
    ```
**效能與正確性注意事項：**
*   **效能代價極低：** 這完全是純數學計算，維持了 Lazy Evaluation 的高效，不需要頻繁 context switch。
*   **Sub-instruction timing（次指令級計時）：** `LDA $2001` 耗時 4 cycles，但實際的「讀取動作」發生在第 4 個 cycle。為了極致的精確度，當 CPU 呼叫 PPU 的 Read function 時，傳入的 `NowDots` 應該是 `(total_cpu_cycles_before_instruction + 3) * 3`。

---

### 3. ROM 端如何提高暖端解析度？（繞過 16 cycles 限制）
即使模擬器修好了，真實硬體的量化誤差仍是 16 cycles (約 9µs)。若想在暖端（衰減極快）提高解析度，ROM 端有幾種進階技巧：

**A. 縮緊迴圈 (Tighter Loop) - 可行，代價低**
改變判定邏輯，只測 bit 7（透過 N 旗標）。
```assembly
WaitDecay:
    LDA $2001   ; 4 cycles
    BPL Done    ; 2 cycles (沒跨頁), 假設衰減時 bit 7 變 0
    JMP WaitDecay ; 3 cycles (或者展開迴圈避免 JMP)
```
你可以把迴圈壓到 7~9 cycles，直接讓解析度翻倍。

**B. 相位偏移量測 (Phase-shifted Equivalent-Time Sampling) - 極力推薦**
這是量測理論中對抗量化誤差的經典作法。既然取樣頻率（迴圈時間）固定，我們可以**改變起跑點**。
1. 第一輪：寫入 $2002 後，**不延遲**，進入 16 cycles 迴圈，記錄 Count0。
2. 第二輪：寫入 $2002 後，加入 `NOP` 延遲 **2 cycles**，進入迴圈，記錄 Count1。
3. 第三輪：延遲 4 cycles... 以此類推直到延遲 14 cycles。
因為物理衰減與 CPU 時鐘是異步的，這 8 次量測會因為起始相位的不同，在衰減發生的臨界點產生不同的 Count（例如差 1）。將這 8 次的 Count 相加或取平均，你的**等效時間解析度就達到了 2 CPU cycles**！

---

### 4. 關於絕對地板與解析度極限（本來就無解嗎？）
**結論：是的，這是物理學與取樣定理的根本限制，只能認命。**

因為溫度與衰減時間是 Arrhenius 關係（指數衰減 $t \propto e^{E/kT}$ 或類似的非線性關係）。
*   在極高溫（熱端）：$\Delta t$ 變得極小。當溫度變化 1 度導致的衰減時間變化**小於 1 次量測解析度**（即使加上相位偏移，極限也是 1 CPU cycle ≈ 559ns），你就再也分不出溫度的差異了。
*   這是所謂的「感測器飽和區」。任何基於計時的感測器，在信號頻率超過取樣頻率時都會面臨這個問題。這不是模擬器或程式寫錯，這是不可違背的數學事實。

---

### 5. 綜合建議（針對教學/驗證用 ROM）

對於一個要展示技術與驗證硬體行為的 Demo，我的取捨建議如下：

1.  **必須修復模擬器（第一優先）：**
    把 `NowDots()` 改為基於 `total_cpu_cycles`。這不僅是為了解決這個溫度計的問題，這是讓你的 AprNes 從「粗糙的批次模擬器」邁向「高精確度時序模擬器」的關鍵基石。不改這個，未來做 Sprite Zero Hit, MMC3 IRQ, 或視差滾動都會遇到嚴重的時序對齊問題。
2.  **ROM 的實用範圍裁切：**
    不需要為了突破 300°C 的物理極限去寫無比複雜的 sub-cycle 測量。在 ROM 的 UI 上，明確設定一個「有效測量範圍」（例如 10°C ~ 80°C）。
    當 count 低於某個門檻（例如判定溫度 > 90°C），直接顯示 `"> 90°C (Out of Range)"` 或 `"MAX"`。這在工程上是最負責任的做法，也符合真實感測器的 datasheets 規範。
3.  **（選做）實作相位偏移展示：**
    如果你想讓這個 ROM 成為一個「硬核」的教學 Demo，可以實作上面提到的「相位偏移量測（Phase-shift）」。這不但能把 80°C 附近的解析度變得平滑，還能作為展示「如何用軟體突破硬體取樣率極限」的絕佳教材，這會讓你的 Demo 非常令人驚豔。
