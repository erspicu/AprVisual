Excellent. This is a fantastic, well-defined micro-optimization problem. Here is the concrete, honest assessment you asked for.

### 1. Automatic Stop for `FirstTouchCaptureHc`

Yes, a coverage-saturation stop is the most principled and cheapest approach. Order-stability is too expensive.

*   **Concrete Stopping Rule:** Stop capturing when the number of *newly* touched nodes in a measurement chunk falls below a small fraction of the *total* nodes touched so far. This adapts to ROMs with different active node counts.
*   **Metric:** `new_touches_in_chunk / total_touched_nodes`
*   **Implementation:**
    *   Define a check interval, e.g., `chunk_hc = 1024`.
    *   After each chunk, calculate `new_touches = (current_total_touched - prev_total_touched)`.
    *   **Stop if `new_touches < (current_total_touched * 0.005)`**. (i.e., new discoveries in the last chunk are less than 0.5% of the discovered set). Also add a hard ceiling (e.g., 262144 hc) to prevent infinite loops on weird workloads.
*   **Threshold Rationale:** A percentage-based threshold is superior to a fixed number because it scales with the size of the active working set. The 0.5% value is a starting point; it represents deep diminishing returns. For a game with 4000 active nodes, this means you stop when you find fewer than 20 new nodes in a ~1ms chunk.

### 2. Automatic Stop for `CaptureWarmupHc`

Yes, there is a much better, non-arbitrary signal for "reset transient is over".

*   **Concrete Stopping Rule:** End the warmup period a fixed, small number of half-cycles *after* the CPU fetches its first instruction from the reset vector (`$FFFC`/`$FFFD`).
*   **Metric:** A boolean flag: `has_fetched_from_reset_vector`.
*   **Implementation:**
    *   In the instrumented CPU model, when processing an instruction fetch cycle, check if `address == 0xFFFC`.
    *   When this happens, set a flag and record the current half-cycle count (`reset_fetch_hc`).
    *   Set `CaptureWarmupHc = reset_fetch_hc + 512`.
*   **Threshold Rationale:** The 1024 hardcode is a guess. The reset vector fetch is the exact, semantic moment the system transitions from hardware power-on state to executing the ROM's first line of code. The extra `+512` hc gives the initial boot code a moment to set up the stack and initial PPU state before you start the capture, which is what the warmup is truly for. This is far more robust than any statistical measure of queue activity.

### 3. Is "First-Touch Order" the Right Metric to Stabilize?

**Yes.** Your existing empirical result is king here. You measured that the pruned first-touch order is the key that provides the +6.2% lift, while RCM-style locality was neutral. Trust your measurement.

The theoretical reason this works is that first-touch order is a direct proxy for the *causal dependency chain* of signal propagation in the circuit. Since your bottleneck is dependent memory loads, laying out the nodes in memory to match the order they are accessed in the dependency chain is the ideal way to improve cache prefetcher efficiency and reduce latency. Stabilizing for frequency or a generic locality metric would ignore this crucial temporal aspect of the dependency chain, which you've already proven matters most. Do not change this objective.

### 4. Online Auto-tune with a Micro-benchmark?

**No. Do not do this.**

The load-time cost is prohibitive and the potential benefit is unmeasurable.

*   **Cost:** To get a stable measurement that can reliably distinguish between two keys whose performance difference is <1%, you would need to run a benchmark for several seconds, not milliseconds. Your self-timed micro-benchmark would need to be longer than the capture window itself. This would turn a 0.3-second load penalty into a 5-10 second penalty.
*   **Benefit:** The difference in performance between the key from a 32768-hc window and one from a "perfect" window determined by saturation is almost certainly going to be deep inside your 1% measurement noise floor. You are spending seconds of load time to chase a 0.2% gain you can't even prove exists.

### 5. Bottom Line: ROI Ranking

Here is the blunt ranking, from best to worst ROI.

1.  **(a) Leave the hardcode.** This is the highest-ROI option. Your current parameters are simple, fast enough, and provide a proven 6.2% gain. The engineering principle of "if it ain't broke, don't fix it" applies strongly. The probability that any further tuning will yield a measurable (>1%) gain is extremely low.

2.  **(b+e) Implement both automatic stops.** If you feel compelled to replace the "magic numbers", the two rules from Q1 (coverage-saturation) and Q2 (reset-vector fetch) are the correct ones to implement. They are principled, cheap, and have a small chance of improving robustness or slightly reducing load times on some ROMs. The development cost is minimal. Do them together. This is the only change I would even consider.

3.  **(c) Order-stability auto-stop.** Bad ROI. The computational cost at load time is too high for a questionable benefit.

4.  **(d) Online self-timed pick.** Terrible ROI. The load-time cost is immense, and the benefit is guaranteed to be below your measurement noise floor. This is a textbook example of premature, over-engineered optimization.

**Final Advice:**
The hardcode is fine. You found a good local optimum. Be proud of the +6.2% and move on. If you have a spare afternoon and the magic numbers bother your engineering sense, implement the reset-vector warmup stop and the coverage-saturation capture stop. They are technically "more correct". But do not expect to measure a performance difference. Your time is better spent elsewhere.
