**每半週期的工作流程**（`WireCore.Recalc.cs` + `.Group.cs`）：

```
StepCycle:
  RunHandlerChain()                              # clk toggle、memory/video callbacks
  ProcessQueue():                                # BFS over dirty-set
    while RecalcListNext non-empty:
      swap current/next queue (double-buffered)
      for each nn in current queue:
        if RecalcHash[nn] != 0:                  # dirty flag #1（佇列 dedup）
          RecalcHash[nn] = 0
          RecalcNode(nn):
            ComputeNodeGroup(nn):
              clear _inGroup of previous group   # O(prev group size)
              AddNodeToGroup(nn) (recursive):
                if _inGroup[nn] != 0: return     # dirty flag #2 / early return（群 dedup）
                _inGroup[nn] = 1
                _groupBuf[_groupCount++] = nn
                _groupFlags |= NodeInfos[nn].Flags
                # walk channels to normal nodes / GND / VCC:
                for each transistor t in TlistC1c2s[nn]:
                  (gate, other) = t
                  if NodeStates[gate] != 0:      # transistor conducts iff its gate is high (NMOS)

                for each t in TlistC1gnd[nn]: if NodeStates[gate]: _groupFlags |= Gnd
                for each t in TlistC1pwr[nn]: if NodeStates[gate]: _groupFlags |= Pwr
              newState = FlagsToState[_groupFlags & 0xFF]   # 256-entry LUT
            for each m in _groupBuf:
              SetNodeState(m, newState):
                if NodeStates[m] == newState: return        # early return #1（最關鍵！）
                NodeStates[m] = newState
                for each t in TlistGates[m]:                # m gates these transistors
                  enqueue both channel endpoints of t       # gate-fanout enqueue
  InvokeCallbacks()                              # memory writes、video frame writes
  Time++
```

### 3. Bit-packing 的細節 (CSR / ELLPACK / SELL-C-σ 針對 AVX2)

### 4. Conduction Matrix Rebuild 是不是 Deal-breaker？

**是。絕對是 Deal-breaker。**

*   **重建成本**：你要遍歷 27K 個電晶體去更新 $A$。就算你用 SIMD 來加速重建，這仍然是 $O(M)$。S1 的整個 half-cycle 只有 $O(D \times \text{degree}) \approx 10K$ 步。你光是**準備**矩陣的時間，S1 早就把這半週期的模擬跑完了。
*   **Incremental Update 的真相**：你想「只更新 gate 變了的那些列」。等等，你要怎麼知道哪些 gate 變了？你需要一個 BFS queue 來追蹤變動的 node... **這不就是 S1 現在正在做的事情嗎？！**
*   **總結**：S1 的 `TlistC1c2s[nn]` 本質上就是一個**即時、免維護、隨選即用 (On-demand)** 的動態 adjacency list。任何試圖把它物化 (Materialize) 成二維矩陣 $A$ 的行為，都是效能退化。

### 5. 有沒有比 H 更好的方向？(你沒想到的)

你似乎專注在「拓樸圖搜尋」上，但 NES 晶片有一個物理特性被你忽略了：**它是可以被 Levelize (階層化 / 拓撲排序) 的**。

NES 的電晶體可以嚴格分為兩類：
1.  **純邏輯閘 (Logic Gates)**: 佔 86%。訊號流向是單向的 (Unidirectional)，沒有 feedback (除非是 Latch，那是在 clock 邊緣觸發)。
2.  **傳輸閘/匯流排 (Pass-Transistors / Buses)**: 佔 14%。訊號雙向 (Bidirectional)，需要 BFS/收斂。

**新方向提案：DAG-Extraction + Split-Phase Simulation**
*   **Build-time**：做 SCC (強連通分量) 分解。把所有的「單向邏輯閘」提取出來，做拓撲排序 (Topological Sort)。把它們拍平成一個純粹的、無需 BFS、無分支的指令流 (或位元運算流)。
*   **Runtime**：
    *   **Phase 1 (Logic)**: 依序遍歷單向陣列，直接計算狀態。完全不需要 BFS queue，不需要 dirty flag，就是純粹的 $O(N_{logic})$ 直行。因為無分支且 memory access pattern 極佳，這段可以極速執行 (甚至可以用 SIMD 將 64 個 state bits 平行計算)。
    *   **Phase 2 (Pass-Bus)**: 把 Phase 1 產生的變動當作 seed，餵給 S1 原本的 BFS queue，去解決那 14% 需要雙向解析的節點。
*   **為什麼這會贏？**：雖然 Phase 1 是 $O(N)$，但它是**無分支、零 overhead** 的 $O(N)$。S1 的 BFS 雖然是 $O(D)$，但有大量的 `if` 判斷、hash 檢查、queue 操作。在現代 CPU 上，執行 10000 個連續的 `A = B & C` 比執行 500 次複雜的 queue 入出操作還要快。

### 6. 整體判斷與天花板 (潑冷水時間)

這個 `math-algos` branch 的結論：

*   **把這套系統硬塞進「線性代數 / SIMD SpMV」的框架，天花板是 0.5x 到 0.8x（也就是會變慢）**。因為你把 Event-Driven (稀疏) 強行轉成 Data-Parallel (密集)。
*   **S1 本身的演算法已經非常接近 Switch-level Simulation 的理論極限**。它是一個極度優秀的 Incremental Solver。
*   如果只做 **G (Layout 最佳化)**，預期可達 **1.2x ~ 1.4x** (~65K hc/s)。
*   如果採用我提的 **DAG-Extraction (拆分單向邏輯與雙向匯流排)**，預期可達 **2x ~ 3x** (~100K - 140K hc/s)。
*   **要達到 10x 以上的跨越？** 不可能在「純圖形 / 代數」上達成。你必須放棄每個 half-cycle 的電晶體級精確模擬，走向 (1) **Static Compilation (把圖轉成 C# 代碼直接編譯)** 或 (2) **Macro-modeling (把整個 adder / counter 替換成 high-level 行為模型)**。

**給 User 的建議回覆：**
「線性代數與 SIMD 擅長處理 dense / predictable 的運算，而我們的 NES 模擬器核心是一個 sparse / unpredictable 的事件傳播圖。每半週期僅約 3% 的節點活躍。將 3% 的事件放進全域矩陣中用 SIMD 運算，矩陣維護與資料打包的成本將遠大於 SIMD 帶來的加速。這條路徑在純 switch-level 模擬的範疇下已被證偽。我們應將精力轉向資料局部性 (Data Locality / RCM 排序) 或靜態邏輯提取。」
