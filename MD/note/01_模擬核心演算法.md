# MetalNES — 模擬核心演算法

核心檔案：`source/metalnes/wire_module.cpp`（2340 行），裡面有兩個類別：

- `Wires` — 網表組裝（載入模組、建 node/transistor、處理 connection），實作 `wire_module` 介面
- `wire_compute` — 真正的模擬引擎（recalc、node group、求值、傳播）

本文件講 `wire_compute`。它**本質上是 Visual6502 `chipsim.js` 演算法的 C++ 工程化升級版**，但加了一堆關鍵優化與一個重要的求值改良。

---

## 1. 資料結構

### 1.1 節點狀態：拆成三個陣列（SoA！）

```cpp
using node_value = uint8_t;            // 0 / 1

struct nodeinfo {
    uint8_t flags = 0;                 // node_state | node_pull_up | node_set_high | ...
    int connections = 0;               // c1c2s + gates 的數量（= 約略電容大小）
    int _tlist_gates;                  // 索引：此節點當 gate 的電晶體的 (c1,c2) 攤平表
    int _tlist_c1c2s;                  // 索引：此節點當 channel、對端是普通節點 的 (gate,other) 攤平表
    int _tlist_c1gnd;                  // 索引：此節點當 channel、對端是 GND 的 gate 清單
    int _tlist_c1pwr;                  // 索引：此節點當 channel、對端是 VCC 的 gate 清單
};

// 在 Wires 裡：
std::vector<node_value> _node_states;  // [node_id] -> 0/1     ← 熱資料，每步狂讀
std::vector<nodeinfo>   _node_infos;   // [node_id] -> nodeinfo
std::vector<node_ptr>   _node_ptrs;    // [node_id] -> node*   ← 冷資料（名字、segments、callback）
std::vector<wire_transistor> _transistors;
std::vector<nodeID>     _transistor_list;  // 一大坨攤平的鄰接表，null(0) 結尾的子清單
```

**重點 1：SoA**。`_node_states`（每步被狂讀寫的 hot data）和 `node`（名字、polygon、callback 等 cold data）分開存。這正是我們設計文件 §10.1 強調的 SoA 概念，MetalNES 已經這樣做了 —— 雖然它沒做 bit-slicing（每個 instance 一個 byte，不是一個 bit）。

**重點 2：`node_flags`（8 bit）**

```cpp
enum node_flags {
    node_state        = (1 << 0),   // 目前邏輯狀態（也用來代表「浮接時保留的前值」）
    node_pull_up      = (1 << 1),   // 有 pull-up（來自 segdef '+' 或 module pullups:[]）
    node_set_high     = (1 << 2),   // 被外部強制拉高（setHigh()）
    node_set_low      = (1 << 3),   // 被外部強制拉低（setLow()）
    node_pwr          = (1 << 4),   // 這個節點是 VCC
    node_gnd          = (1 << 5),   // 這個節點是 GND
    node_forcecompute = (1 << 6),   // 特殊：若 group 同時連到 gnd+pwr，互相抵消（給某些 bus 用）
    node_has_callback = (1 << 7),   // 此節點有註冊 callback（記憶體 watch 等）
};
```

### 1.2 三組 transistor 清單的妙處

在 `reset()` 裡，每個節點 `nn` 把自己 touch 到的電晶體**預先分類**成三組（`wire_module.cpp:1447-1498`）：

- `_tlist_gates`：此節點當 **gate** 的電晶體，攤平成 `(c1, c2, c1, c2, ..., 0)`。當這個節點狀態變了，就要把這些 c1/c2 重算。
- `_tlist_c1c2s`：此節點當 **channel 端**、且對端是普通節點的，攤平成 `(gate, other, gate, other, ..., 0)`。建 group 時走這些。
- `_tlist_c1gnd` / `_tlist_c1pwr`：此節點當 channel 端、**對端剛好是 GND / VCC** 的，只存 `(gate, gate, ..., 0)`。

→ 「我這個節點現在有沒有路徑通到 GND？」變成「掃一下 `_tlist_c1gnd`，看有沒有哪個 gate 是 high」—— **O(只連到 GND 的電晶體數)**，而不是掃全部 channel。這是相對 Visual6502（每次都掃 `c1c2s` 並判斷對端是不是 ngnd/npwr）的一大加速。

## 2. 模擬主迴圈

### 2.1 每個 half-cycle：`step_cycle()`

```cpp
void step_cycle() {
    bool clk = isNodeHigh(_clk);
    if (clk) setLow(_clk); else setHigh(_clk);   // 翻轉主時鐘節點 → 觸發 recalc 連鎖
    if (_handler) _handler();                     // 跑 handler 鏈（clock handler、nes_system handler...）
    if (_trace) dump();
    _compute->nextCycle();                        // 重置 recalc 計數
    _time++;
}

void step(int count) { for (int i = 0; i < count; i++) step_cycle(); }
```

`step(count)` 由 `wire_thread` 的模擬執行緒每幀呼叫一大批（預設 1024 步）。

### 2.2 變動傳播：`recalcNodeList` → `processQueue`

```cpp
void recalcNodeList(nodeID nn)  { enqueueNode(nn); processQueue(); }

void processQueue() {
    int iteration = 0;
    while (!_recalc_list_next.empty()) {
        if (++iteration == 100) printf("iteration warning %d\n", iteration);  // ← 只警告，不停！
        _recalc_iteration++;
        _recalc_list_next.swap(_recalc_list);          // 雙緩衝
        _recalc_hash_next.swap(_recalc_hash);
        for (nodeID nn : _recalc_list) {
            _recalc_count++;
            if (_recalc_hash[nn]) {                    // hash 去重
                recalcNode(nn);
                _recalc_hash[nn] = 0;
            }
        }
        _recalc_list.clear();
    }
    _wires->invoke_callbacks();                          // 收斂後才觸發 callback（記憶體讀寫等）
}
```

**對照設計文件 §11.2**：Visual6502 是 `for(j=0;j<100;j++)` 硬上限。MetalNES 改成「迭代到 `_recalc_list_next` 空為止（= 真正收斂），到 100 次只印警告不停手」。這比 Visual6502 好一點（不會在 100 次就硬截斷），但**仍然沒有「迴路偵測」**—— 它賭電路會在合理次數內收斂。實務上 NES 的電路確實會收斂（因為真實電路也會穩定），但這是賭運氣，不是證明。

### 2.3 單一節點重算：`recalcNode`

```cpp
void recalcNode(nodeID nn) {
    if (is_pwr_gnd(nn)) return;
    bool newState = computeNodeGroup(nn);          // 1) 算出這個 group 的值
    for (nodeID gnn : _group) setNodeState(gnn, newState);   // 2) 把值寫給 group 裡每個節點
    if (_group_flags & node_has_callback) { ... enqueue_callback ... }   // 3) 觸發 callback
}
```

### 2.4 建 group：`addNodeToGroup`（遞迴 DFS）

```cpp
void addNodeToGroup(nodeID nn) {
    for (auto gnn : _group) if (gnn == nn) return;       // 已在 group 就跳過
    nodeinfo &ns = _node_infos[nn];
    _group.push_back(nn);

    if (ns.connections > _max_connections) {             // ★ 記住「連線最多」的節點及其狀態
        _max_state = _node_states[nn];
        _max_connections = ns.connections;
    }
    _group_flags |= ns.flags;                            // ★ 把 flags OR 進來

    // 沿著導通的電晶體展開
    if (ns._tlist_c1c2s) { ptr = &_transistor_list[ns._tlist_c1c2s];
        while (*ptr) { gate = *ptr++; c2 = *ptr++;
            if (_node_states[gate]) addNodeToGroup(c2); } }     // gate 為 high → 導通 → 加入

    if (ns._tlist_c1gnd) { ptr = ...;
        while (*ptr) { gate = *ptr++;
            if (_node_states[gate]) { _group_flags |= node_gnd; break; } } }   // 連到 GND

    if (ns._tlist_c1pwr) { ptr = ...;
        while (*ptr) { gate = *ptr++;
            if (_node_states[gate]) { _group_flags |= node_pwr; break; } } }   // 連到 VCC
}
```

→ 邊建 group 邊累積兩樣東西：(a) `_group_flags`（OR 起來的所有 flag），(b) `_max_state`（group 裡 connections 最多那個節點的狀態）。

### 2.5 求 group 的值：`getNodeValue` + 查表

```cpp
node_value getNodeValue() {
    if (_group_flags & node_forcecompute) {              // forcecompute 特例
        if ((_group_flags & node_gnd) && (_group_flags & node_pwr)) {
            _group_flags &= ~node_gnd;  _group_flags &= ~node_pwr;   // gnd 和 pwr 互相抵消
        }
    }
    if (!_group_flags) return _max_state;                // ★ 完全沒 flag（純浮接 group）→ 最大電容節點 wins
    return _flags_to_state[_group_flags];                // ★ 否則查 256-entry 表
}
```

**`_flags_to_state[256]` 是在 `reset()` 時用 `flagsToState()` 預算好的**。`flagsToState(flags)` 的優先順序：

```text
1. forcecompute 且同時 gnd+pwr → 清掉 gnd 和 pwr
2. gnd      → false      （GND wins）
3. pwr      → true
4. set_high → true        （外部驅動高）
5. set_low  → false       （外部驅動低）
6. pull_up  → true        （pull-up）
7. state    → true（保留前值；上次是 high）
8. else     → false
```

**對照設計文件 §8**：這就是我們設計文件講的 NMOS 三層規則的 MetalNES 版本：

| 設計文件規則 | MetalNES 對應 |
|--------------|---------------|
| 有路徑到 GND → 0 | `node_gnd` flag |
| 否則有路徑到 VCC / pull-up → 1 | `node_pwr` / `node_pull_up` flag |
| 否則保留前狀態 | `node_state` flag；**或** group 全浮接時用 `_max_state` |

> ⚠️ 注意：和 Visual6502 一樣，MetalNES 把「真的被驅動到 high」與「上次這個 group 就是 high」混在同一個 `node_state` flag 裡。這正是設計文件 §15.3 點出的「dynamic node = previous state 是工程抽象」的具現。對 cycle-accurate 數位行為夠用，對 charge sharing / 洩漏不夠 —— 但 MetalNES 用一個小聰明緩解：當整個 group 都浮接時（沒任何 flag），不是無腦保留某個值，而是回傳**連線最多（電容最大）那個節點的狀態**，這是對 charge sharing 的一個粗略物理近似。

### 2.6 寫回狀態並繼續傳播：`setNodeState`

```cpp
void setNodeState(nodeID gnn, node_value newState) {
    if (_node_states[gnn] != newState) {
        _node_states[gnn] = newState;
        nodeinfo &ns = _node_infos[gnn];
        if (ns._tlist_gates) {                           // 這個節點是某些電晶體的 gate
            ptr = &_transistor_list[ns._tlist_gates];
            while (*ptr) { c1 = *ptr++; c2 = *ptr++;
                enqueueNode(c1);                          // 那些電晶體的 c1 要重算
                if (!newState && !is_pwr_gnd(c2)) enqueueNode(c2);  // 變 OFF 時 c2 也要（可能斷開了）
            } }
    }
}
```

`enqueueNode` 把節點塞進 `_recalc_list_next`，用 `_recalc_hash_next` 去重。雙緩衝（`_recalc_list` / `_recalc_list_next`）讓「這一輪要處理的」和「下一輪要處理的」分開。

## 3. 三個值得偷學的優化

| 優化 | 做法 | 效益 |
|------|------|------|
| **flags → state 查表** | group 的所有 flag OR 成一個 8-bit 值，索引進 256-entry 預算表 | 求值從一串 if 變成一次陣列存取 |
| **transistor 預分類** | 每個節點的 channel 電晶體預先分成「對普通節點 / 對 GND / 對 VCC」三組 | 「連到 GND 嗎」變成掃一個小清單 |
| **攤平鄰接表 + null 結尾** | `_transistor_list` 一大塊連續記憶體，子清單 0 結尾 | cache-friendly，沒有 per-node 的 vector 間接 |
| (SoA) | `_node_states`（hot byte 陣列）與 `node*`（cold）分開 | 走 group 時只碰 hot 陣列 |
| (雙緩衝 recalc list + hash) | `_recalc_list` / `_recalc_list_next` swap，hash 去重 | 不重複處理同一節點 |

## 4. 跟我們設計文件的差距（重要）

MetalNES 的求值引擎做到了設計文件的 Phase 1–2（CPU evaluator + pull-up/float + 收斂式 settle），但**沒做**：

1. **沒有迴路偵測 / SCC**：純靠迭代收斂 + 100 次警告。設計文件 §6.4 / §10 主張要做三色 DFS / Tarjan，把回授群縮成超節點、分類為 static latch / dynamic / bus loop。MetalNES 沒這層 —— 它的「迴路」就靠電路自己穩定。
2. **沒有 IR**：節點求值是直譯（每步重跑 group 演算法），沒有把電路「編譯」成布林表達式樹。設計文件 §8.2 主張先建 `Expr` IR。
3. **沒有 codegen**：沒有 emit C++ / Verilog / CUDA。
4. **沒有 bit-slicing / 多 instance**：一個 byte 一個 instance，單執行緒一台 NES。設計文件 §10.2 主張用 `uint32/64` 一條表達式算 32/64 台。
5. **沒有等價驗證框架**：靠跑 test ROM 對結果，不是 node-by-node 比對 CPU vs CUDA（因為根本沒有 CUDA backend）。

→ **這 5 點正是我們專案相對 MetalNES 的「往前推進空間」**。如果我們的目標是「邊跑邊比對 → 抽出 IR → emit CUDA → 大規模平行」，MetalNES 是很好的 CPU 參考起點，但它本身不是終點。
