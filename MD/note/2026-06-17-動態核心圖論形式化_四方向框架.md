## 0. 四方向框架(使用者提案)

- **階段**:靜態準備(**inspector**)↔ 動態執行(**executor**)。這正是 inspector–executor 結構(論文參考文獻 [7–9])。

> **用語更正**:把「節點縮減」改成「**事件剪枝 + 資料重排**」。位元精確要求**節點集合不可變**(讓內部節點消失=S2 的抽象,S1 不允許);lowering / 常數摺疊實測整顆晶片僅 ~0.16% 可砍(已是確定死路)。靜態階段省的是「每半週期要重算的**事件數**」與「存取**延遲**」,不是節點數。

## 1. 靜態受質(時不變結構)

節點集 `V`(|V| ≈ 數萬,2A03+2C02 整機),電晶體集 `T`。每個 NMOS pass transistor `t∈T`:
- 閘 `g(t) ∈ V`;通道兩端 `c1(t), c2(t) ∈ V`。
- 導通條件:`s(g(t)) = 1`(閘為高)時,`c1–c2` 連通。NMOS-only,c1/c2 可互換。

由 `T` 在 `V` 上導出三個結構:

1. **可切換通道圖**(無向、帶守衛):每個 `t` 貢獻一條邊 `e_t = {c1(t),c2(t)}`,守衛為 `g(t)`。
2. **影響有向圖** `D = (V, A)`:arc `(u → w)` ⟺ `u` 是某 `t` 的閘、`w` 是該 `t` 的端點(`w∈{c1(t),c2(t)}`)。語意:`u` 狀態改變 ⇒ 可能改變 `w` 所在分量 ⇒ 需重算 `w`。對應 `NodeTlistGates[u]`。
3. **供電關聯**:節點可經「被某閘控制的電晶體」接到 GND / VCC 軌;該閘為高時,所在群取得 `Gnd` / `Pwr` 旗標(供電軌不進群走訪,折成旗標)。對應 `GndCount` / `PwrCount`。

每節點另有時不變屬性:靜態旗標 `φ(v) ⊆ {PullUp, ForceCompute, HasCallback, …}`,與**電容代理** `cap(v) = NodeConnections(v) =` 通道度 + 閘度。

---

## 2. 動態狀態與群解析

狀態 `s : V → {0,1}`(`NodeStates`)。外加動態驅動旗標 `δ(v) ∈ {SetHigh, SetLow, ⊥}`(由 handler / 時鐘設定)。

**導通子圖**(依賴 s):
```
E(s) = { e_t : s(g(t)) = 1 } ,   G(s) = (V, E(s))
```
`G(s)` 的連通分量 = 等電位群。記 `v` 的分量為 `K_s(v)`。

**群旗標聚合**(交換、冪等 → 半格,`∨ = OR`):
```
Λ(C) = ⋁_{v∈C} φ(v)  ⊕ (C 內有導通 gnd 閘 ⇒ Gnd)  ⊕ (有導通 pwr 閘 ⇒ Pwr)  ⊕ δ 貢獻
```
冪等 + 交換 ⇒ **走訪順序無關** ⇒ BFS/DFS 等價(這是「DFS 沒有比較快」的數學原因)。

**解析函數** `val(C)`:
- `Λ(C) ≠ ∅`(被驅動):`val = ρ(Λ(C))`,`ρ` 是固定優先級解碼(256-entry LUT,`FlagsToStateOf`):
  `ForceCompute∧Gnd∧Pwr ⇒ 抵銷兩者` → `Gnd⇒0` → `Pwr⇒1` → `SetHigh⇒1` → `SetLow⇒0` → `PullUp⇒1` → `State⇒1` → `else⇒0`。
- `Λ(C) = ∅`(純浮動):`val = s( argmax_{v∈C} cap(v) )` — **電容支配的保持前值**(tie 由走訪順序定,seed 先入)。對應 `GetNodeValue` 浮動分支。

> **關鍵**:`ρ` 對「驅動類」是無記憶純函數;但浮動分支**依賴當前 `s`**(保持前值)→ `val` 不是 `E(s)` 的純函數,而是 `(E(s), s)` 的函數。這個記憶性是後面「F 非單調」的根。

---

## 3. 安定 = 非同步不動點迭代

定義「一次全解析」算子 `F`:
```
F(s)(v) = val( K_s(v) )
```
注意 `G(s)`、因而分量 `K_s(·)`,**本身是 `s` 的函數**(閘也是節點 — **自我參照**)。故 `F : s ↦ s'` 是「在由 `s` 自己定義的分割上,重解每個分量」。**這個自我參照就是為何沒有靜態排程的根源**:你要解的圖,由你正在解的狀態決定。

**安定態(quiescent)= 不動點** `s*`:`s* = F(s*)`。

**半週期**:翻轉時鐘節點的驅動旗標擾動 `s`,再以**非同步(chaotic / Gauss–Seidel)迭代**逼近不動點。引擎不每回合重算整個 `F`,只重解被上一波改動「弄髒」的分量 — 髒前緣沿 `D` 傳播:
```
改動集 Δ  →  重算  ⋃_{v∈Δ} N⁺_D(v)  的分量  →  直到無改動(不動點)
```
雙緩衝波(`RecalcList` / `RecalcListNext`)實現 Gauss–Seidel 波序(下一波用最新值)。

**事件剪枝 = `D` 上的「不變弧」**:P-1~P-4 是靜態過近似,證明某些弧 `(u→w)` 在 `w` 的結構類下「`u` 改動不可能改變 `K(w)` 的解」,故可不 enqueue。例:turn-off 進入「無驅動孤立單例」(浮動保持前值)⇒ 必無改動 ⇒ 跳過(對應 `SetNodeState` 的 range-compare;range 是把該結構類在載入時排成連續 id 區段)。

## 4. 量到的結構事實(界定難度 = 數學簡化的邊界條件)

- **分量極小**:平均導通群 ≈ **1.13–1.4 節點**(多數節點是單例,走 fast-path);真正進 BFS 解析器的走訪中 ~**77% size-2**、~16% size-3、~5.7% ≥4;BFS 深度均值 ≈ 1.13。⇒ 穩態 `G(s)` 近似「稀疏 near-matching」(孤點 + 2-clique 為主)。
- **影響圖無 feed-forward**:`D` 的 SCC 凝聚 ~**94% 是單一巨 SCC**,只有 ~6% 前饋可壓。⇒ 無法把 `F` 靜態分層成 DAG(無 topological schedule)。
- **F 非單調**:浮動保持 + 電容 tie-break 使 `F` 依賴 `s`、非格上單調算子 ⇒ 無 Kleene 唯一最小不動點;是真正的非同步鬆弛,原則上可振盪(故設 pass 上限;實測安定 ≤ ~45 波)。

## 5. 下一階段的數學把手(候選 — 尚未推導;每條附已知實測狀態,避免重蹈)

僅列「形式結構自然邀請的數學角度」,**本階段不展開**:

- **(a) 解析半格 `Λ` 的增量維護**(monoid 部分和):成員旗標變動時增量更新聚合,免重 OR 整群。⚠ 已知:counter-fastpath 實測 −6%(群太小,維護成本 > 省下)。
- **(b) 動態連通性**(edge insert/delete 下 incremental CC):理論上每步 work 可界在「改動區」。⚠ 已知:群 ~1.4,資料結構 overhead 壓垮(bitset-BFS −156×)。
- **(c) SCC 凝聚 / 分層**:94% 巨 SCC 是牆,只有 ~6% 前饋可拓樸排程。→ 這也是「為何無靜態 DAG」的**精確陳述**(本身即嚴謹負結果,可入論文)。
- **(d) 不動點單調化 / 收斂界**:能否 refine 出一個格使 `F` 單調?或證明非單調是本質的(電容 tie-break 的資訊論下界)。→「F 非單調」之證明本身有發表價值。
- **(e) 半環矩陣化 settle**:把安定寫成解析半環上的迭代不動點求解(稀疏「矩陣-向量」式),問是否有可利用結構(分塊 / 低秩 / RCM 後帶狀)。⚠ 已知:RCM 重排只買到 locality(+3.6 / +6%),非漸近 work。

## 7. 數學推導階段:第一個候選 — group_flags turn-off skip(Gemini 諮詢 + 評估)

2026-06-17 以「純數學等價簡化」視角諮詢 gemini-3.1-pro-preview(完整紀錄:`tools/knowledgebase/message/20260617_012358.txt`)。目標明確:**用等價變換消掉那 84% 的「重算後無變化」事件以增速,不是證明負面**。Gemini 給出一個具體、可證、正中靶心的候選。

### 7.1 提案:元件旗標快取(group_flags)的 turn-off 跳過

**turn-off 定理(已驗,正確)**:節點 `c` 目前狀態 `V`,其某條通道斷開使元件分裂。因子集旗標聚合單調遞減 `Λ(C_new) ⊑ Λ(C_old)`:
- `c=1` 只可能翻成 0 ⟺ 舊元件含 0 驅動者(Gnd / SetLow)。
- `c=0` 只可能翻成 1 ⟺ 舊元件含 1 驅動者(Pwr / SetHigh / PullUp / 保持態)。
- 浮動子群每成員同前值 ⇒ 電容 tie-break 退化成 identity,不會自發翻轉。

故:**若 `c` 舊元件的 `Λ` 不含「反向極性」驅動者,則任何子集都不含 ⇒ `c` 必維持原值 ⇒ 該 enqueue 可證明 no-op,跳過**。

**維護(等價 invariant)**:每次群解析寫回時,把該群的 `Λ`(一個 byte)寫到每個成員的 `group_flags[]`。`group_flags[c]` = `c` 上次被解析時所在元件的 `Λ`;靠「任何合併(turn-on)會 enqueue 端點 → 重解 → 刷新 group_flags」維持時效。

**熱路徑跳過測試(turn-off enqueue 迴圈,端點 `c`)**:
```
lam = group_flags[c]
if NodeStates[c]==1 ? (lam & MASK_0_DRIVERS)==0 : (lam & MASK_1_DRIVERS)==0:  skip c
else:  enqueue c
```
(turn-on 那半 = `s[u]==s[w]` 跳過 = **我們已上線的 P-1**;Gemini 等於獨立重推,印證推理可信但無新東西。)

### 7.2 評估(對到已測死路)

- **真新 & 與 P-5z 不同**:P-5z 釘「解析後的值」→ stale 發散(其 golden 是 survivorship);group_flags 釘「旗標 OR 集合」→ 單調、不受 tie-break 影響 → 繞過 P-5z 發散點。比靜態 P-2/3/4 更廣:抓的是**當前狀態**下的 no-change(動態殘留),不是全狀態皆成立的(靜態)。
- **經濟風險(與死路 #4 / #5 / counter-fastpath 同型)**:
  1. 維護要在**每次解析寫回**多寫一個 byte——連 ~70% 單例熱路徑都得寫(singleton / B1-pair / BFS 三條路徑都要)。
  2. 跳過測試要在熱 enqueue 迴圈多 `group_flags[c]` + `NodeStates[c]` 兩個**隨機 byte 載入**;現行 P-2/3/4 是**純暫存器比較、零載入**。加 dependent load 正好逆行於「記憶體延遲」這條最痛軸。
- **正確性 wrinkle(原型要顧)**:ForceCompute 群 Gnd&Pwr 互銷,MASK 測試需正確處理(別把「同時有 Gnd、Pwr、FC」誤判極性)。
- **預估**:偏中性/負。但**不等於任何已測死路、數學乾淨、正中靶心** → 過「最小原型 + checksum gate」門檻(beat-S1 規則)。量了才算數。

### 7.3 我自己的第二條(Gemini 沒接,正交)

**結構/值分離**:把「大多時間不變的元件骨架」與「時鐘切換的那一小撮邊」分離,只在骨架上重算被時鐘真正動到的部分。與 group_flags 互不衝突,可獨立探。

### 7.4 驗證協議(原型用)

- **bit-exact**:golden checksum,**SMB1 + full_palette 都驗**(避開 P-5z survivorship 陷阱:單一 ROM 過關不算數)。
- **timing**:interleaved-paired(預建兩 binary、交替 base/exp、中位數 + trimmed-mean + 配對勝場)。
- **分支實作 `group-flags-skip`**;beat-S1 才併,否則記為死路。

### 7.5 結論(2026-06-17 量測完畢:確認死路,不併 main)

兩步都做了、都驗了,結論與 7.2 的「偏中性/負」預估一致 ——

**bit-exact(全過)**:原型(`bdb9bcf`,存原始 Λ flags)修掉「記憶體節點不可跳過」後,full_palette `--extra-ram` checksum 在 300k/400k/1M 全中金牌
(`0x794A43ABDF169ADA` / `0x9174E19D961CB6E5` / `0x6D4CCBCE2E9CD599`)。
P2(`c83fe87`,改存預算好的 1-byte predicate、熱讀只剩一筆 byte 載入)同樣三長度全中。

**timing(15 輪 interleaved-paired,400k,`--pin`,中位數 hc/s)**:

| 版本 | median | vs main | paired |
|---|---|---|---|
| main(無 GF) | 134,104 | — | — |
| proto(原始 Λ flags,`bdb9bcf`) | 127,481 | −4.94% | — |
| **exp(GF + P2 predicate,`c83fe87`)** | 128,391 | **−4.26%** | **exp>main 0/15** |
| | | exp>proto +0.71% | exp>proto 10/15 |

**判讀**:P2 確實照設計拔掉了讀取端的 `NodeStates[c]` 載入 + state 選遮罩,對原型有 +0.71%(bit-exact);
但整個 group_flags 機制仍比 main 慢約 4%(exp 對 main 0/15,完全沒贏過)。

**根因(= 死路 #4/#5/counter-fastpath 同型,latency-bound)**:這機制無法擺脫兩筆成本 ——
(1)每次解析(~70% singleton + B1-pair + BFS 三條寫回路徑)多寫一筆 predicate;
(2)turn-off 迴圈仍多一筆相依 `canChange[c]` byte 載入。
而 main 既有的 range-compare prune(`c<rS` / `c<rA||c>=rB`)是**零載入暫存器比較**。
GF 多省下的少量 enqueue 付不起這兩筆;P2 只能砍掉讀取端兩筆相依載入中的一筆(`NodeStates[c]`),寫回端那筆 store 仍在 → 淨值仍負。

**處置**:`group-flags-skip` 分支保留為實驗紀錄(P2 是乾淨的 +0.71% bit-exact 小改),**不併入 main**。
新增第 8 條同型死路:任何「動態殘留 no-change」culling 只要在熱路徑新增 predicate 維護寫 + 相依讀,就會被 latency-bound 反噬,即使理論上更廣、即使 bit-exact。
