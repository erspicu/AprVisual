## 一、統一發現(整條 ALERead 調查的收束)
三顆看似無關的測試 = **同一個根因:S1 的 CPU→PPU 跨晶片效應早約 1 CPU cycle(24hc=3 dot)**。
1. **ALERead**(boing2k7):$2007 讀取的 ALE 效應在 S1 落 **dot 226**(半週期 trace 實證),應在 **dot 229**。
   八位閂鎖因此捕捉 $2CF0 的 $F0,而非 $2FFF 的 $FF → 無 hybrid address 汙染。read_2007_trigger 早在 dot 224。
2. **dot-339**:$2001 開渲染,渲染 enable 到 dot-339 決策早 3 dot。現用外科手術 clamp(PpuWriteDelayGlobal,24hc)。
3. **even_odd**:$2001 相對 dot-339 skip ~1-dot 偏移。現用窄窗寫延遲 shim(PpuWriteDelay,vpos261/hpos338-339,16hc)。

**三方交叉佐證**:NESdev wiki(hybrid address「reads from an address the PPU never outputted」)+
AprNes/TriCNES(顯式 $2007 SR pipeline + 八位閂鎖 + 手調 offset=7,三顆全過)+ 半週期 trace + 5 輪 Gemini。

## 二、Gemini 的統一修法(a_m6_cpu_ppu_phase_20260715.md,公開)
**根因**:CPU→PPU 的走線電容 + Pad buffer + 內部同步的**物理傳播延遲被 switch-level 抹零**。
**為何前人全域延遲會壞**:粒度太粗 —— **$2002 讀取(非同步狀態直通)** vs **$2001 寫入 / $2007 觸發
(需內部同步 + 時脈邊緣捕捉)** 的內部延遲**不對稱**。無腦全延遲把 $2002 也推後 → 打壞 NMI-edge 8 顆。
**乾淨正解 = 非對稱介面延遲**:
- **只延遲 $2001 寫入 + $2007 存取**對 PPU 生效 **24hc(1 CPU cycle)**。
- **放過 $2002 讀取**(保住 NMI golden alignment)。

## 三、施力點(已定位 + 半週期實測 2026-07-15)
- PPU I/O chip-enable:`ppu.io_ce`(**active-low**:實測 io_ce=0 = PPU 被存取)。`cpu.rw`(1=read)。
- 暫存器選擇:`ppu.io_ab[2:0]`(**7=$2007、1=$2001**)。資料:`ppu.io_db[7:0]`。
- **實測 ALERead $2007 讀取的介面時序(v=3,[aehc] 探針,已 commit)**:
  ```
  dots 223-224: ioce=0 ioab=7 rw=1 r2007=1   ← CPU 的 $2007 讀取存取在這
  dots 225+   : ioce=1 r2007=0               ← 存取結束
  dot 226     : ale=1 rd=0(ReadALE 的 ALE 異常)← 早 3 dot!應在 dot 229
  dot 228     : chrAb=$2FFF($FF 在此但 ale=0)
  ```
  → 延遲這個 ioce=0(dots 223-224)24hc → 移到 226-227 → ALE 異常移到 dot 229 → 八位閂鎖抓 $2FFF 的 $FF。
- 現有可參考:`PpuWriteDelay`(System.cs:1785+,delay bkg/spr_enable 下游)、`PpuWriteDelayGlobal`
  (dot-339)、`InstClampLow/High/Release`(儀器級 force)。

## 三-b、實作的兩個關鍵子問題(2026-07-15 實測發現,務必處理)
1. **$2007 讀取分兩部分**:(a) CPU **立即**讀「讀取緩衝」(返回上次 fetch 的值)、(b) PPU 重抓 VRAM 填緩衝
   (**ReadALE**,boing2k7 汙染的來源,可延遲)。**無腦延遲整個 io_ce 會害 CPU 讀不到緩衝 → 破壞讀取。**
   → 要延遲的是 **(b) ReadALE(VRAM 重抓)**,不是 (a) CPU 讀緩衝。AprNes 的 read/write latch 分開正是為此。
2. **CPU 24hc 後放開匯流排** → 延遲時 io_ab/io_db 已變 → **必須捕捉-重播**:存 (io_ab, io_db, rw),24hc 後
   重播進 PPU。crude「clamp io_ce」會 LOSE 存取(PPU 永遠沒看到)→ 無效。
   → 這是真正的 delay pipeline(非單純 clamp),對應 AprNes/TriCNES 的 delay_queue。

## 四、實作設計(捕捉-重播 delay pipeline,像 AprNes/TriCNES delay_queue)
在 StepCycle(每 hc)插入 CPU→PPU 橋接延遲:
1. **偵測**:每 hc 看 io_ce 是否對 **$2001-write(io_ab=001 & rw=write)或 $2007(io_ab=111)** 生效。
2. **捕捉**:進 delay queue:`{fireTime = now+24, io_ab, io_db, rw}`。
3. **壓抑立即生效**:clamp `io_ce` 非致能(InstClampHigh `_io_ce` 或 clamp io_ce 對應極性)24hc,
   讓 PPU 這 24hc 內不處理該存取(**$2002 讀取不碰,維持 golden alignment**)。
4. **重播**:到 fireTime,drive io_ce + io_ab + io_db + rw 進 PPU(force 該存取),讓 PPU 現在才「看見」。
**注意**:CPU 24hc 後已放開匯流排,所以**必須捕捉-重播**(不能只 clamp io_ce,否則重播時匯流排已變)。
這正是為什麼要 queue 存 (io_ab, io_db, rw)。

## 五-b、S1 特有的實作難點(2026-07-15 實作嘗試中發現)
1. **re-assert 高難**:延遲一個信號 = 抑制自然脈衝(dt<24)+ 重新拉高延遲脈衝(dt 24~38)。
   InstClampLow(Gnd 位階、必勝)抑制沒問題;但**重新拉高**要 InstClampHigh(Pwr/VCC),而 group LUT
   優先序 **GND > VCC/pull-up > 外部驅動** → 若網表此時驅動該節點低,Pwr 壓不過。**這正是 dot-339 shim
   當年只能「鉗互補節點 /bkg_enable low」而非「InstClampHigh 主節點」的同一個坑。** → re-assert 需找
   目標信號的**互補節點**(如 `/read_2007_trigger` 若存在)鉗低,或改延遲 io_ce 存取本身(見下)。
2. **候選施力節點(S1 有豐富的 $2007 狀態機節點)**:`read_2007_trigger`(dot 224 亮)、
   `reading_or_writing_2007_or_hpos_eq_255_and_rendering`(疑似 ReadALE 對 ppu.ale 的貢獻,對應 AprNes
   `PPU_ALE = ReadALE || ...`)、`read_2007_reg`、`read_2007_ended_1/2`、`read_2007_output_vrambuf`
   (讀緩衝輸出,**延遲時要小心別斷了 CPU 讀緩衝**)、`db_buf_read_2007_ended0-7`。
3. **建議實作順序**:先用 io_ce(active-low,CPU 側、我不鉗)當**延遲觸發器**,產生延遲版的
   `reading_or_writing_2007_or_...`(ReadALE 貢獻)→ 只移 ALE 相位、不碰讀緩衝路徑。先驗「ALE 異常
   從 dot 226 移到 dot 229 + 八位閂鎖抓 $FF」再談 $2001-write。

## 五-c、實作嘗試 #1 的決定性負面結果(2026-07-15,單一節點延遲不可行)
試了「延遲 read_2007_trigger 24hc」(io_ce/自然沿觸發,suppress 主 InstClampLow + re-assert 互補
InstClampLow(/read_2007_trigger)):
- **抑制成功**:r2007=1 樣本 12→1(自然脈衝壓掉)。
- **re-assert 失敗**:互補鉗沒回傳,dots 226-227 沒重現 r2007=1。**根因:延遲時網表正 drive
  read_2007_trigger 低,LUT 優先序 GND>VCC>外部 → InstClampHigh(Pwr)/SetHigh(外部)/互補鉗
  全壓不過網表的 drive-low。**
- **結論:延遲任何「網表主動驅動」的內部信號在 S1 不可行(re-assert 高必敗)。** →
  **唯一可行 = io_ce 捕捉-重播**:在延遲後的正確時間(dot 227)重播整個 CPU 存取(io_ce/io_ab/io_db/rw),
  讓**網表自己自然生成** read_2007_trigger,不用硬拉。這是把「延遲信號」轉成「延遲存取事件」的關鍵轉向。
  重播時 io_ce 是 active-low,可 InstClampLow(io_ce)強制存取(Gnd 必勝),io_ab/io_db 各位元同法鉗到
  捕捉值 —— 全部往「低」鉗沒有 re-assert-high 問題。**這是下一個實作方向,乾淨可行。**

## 六、風險 / 開放問題
- 重播的匯流排 drive:force io_ab/io_db/io_ce 進 PPU 需正確極性 + 時機;錯了會連鎖壞 $2004/$2006/$2005。
- 24hc 是否精確(可能非對稱:$2001 vs $2007 略不同;先統一 24 再微調)。
- $2007 存取含讀+寫;讀觸發 ReadALE、寫觸發 WriteALE,延遲量可能不同(AprNes 的 read/write latch 分開)。
- 這是 M6(相位)× M3(跨晶片延遲)的合體;若成,是 S1a 統一假說的第一個實證。

## 八、最終定性:純開關級天花板,S1a M5(行為層 $2007/373)可破

**六輪 Gemini + NESdev + AprNes/TriCNES + 實作實證的最終收束(2026-07-15)**:

**ALERead 在「純開關級」是天花板**,是 **A4(跨晶片傳播延遲)× B(板級 74LS373 未建模)× 二值引擎架構
(GND>VCC>Drive → 無法注入匯流排 '1' / 無法延遲被驅動信號)** 的三層交集。這是整個 141 顆裡唯一同時
踩滿三層的。dot-339/even_odd 同 A4 根因但汙染打在渲染邏輯內部節點、有下游觀察點可鉗(可修);
ALERead 的汙染打在**板級 74LS373**且**資料相依**,下游 shim 要嘛過度開火(BOL 打壞 2007Stress)、
要嘛無法精確。**上游延遲又撞 force-high 牆(§五-b/c)。所以純開關級不可修。**

**但 S1a M5(板級元件庫 + 行為層 $2007 時序)可以精確修** —— 而且架構上完全合法:
- **S1 本來就是混合體**:兩晶粒開關級 + 「晶粒以外一切」行為層(RAM/ROM/手把 CD4021/mapper)。
  **74LS373 是板級晶片、不在網表 → 行為層模它和 CD4021 同級,合法。**
- **作法 = port AprNes/TriCNES 的行為層 $2007/AD-bus 模型**(現成參考碼):
  1. **行為層八位閂鎖**(code byte,非網表節點 → **繞過 force-high 牆**,想寫什麼寫什麼)。
  2. **行為層 $2007 SR pipeline**(readLatch/writeLatch + ReadALE,**關鍵**:給正確相位的 ALE →
     八位閂鎖在 ppuAddressBus=$2FFF 那刻捕捉 $FF)。相位用 offset 校準(AprNes offset=7)。
  3. CHR 讀取位址 = (S1 高六位 | 行為層八位閂鎖)。
- **為什麼精確不過度開火**:AprNes 的 SR pipeline 給**精確的 $2007-read 相位**(非 BOL 的時間窗啟發式,
  非虛擬 373 抓早相位的 ppu.ab)→ 八位閂鎖只在 boing2k7 真正發生時開火 → **2007Stress 不受影響**。
- **代價/哲學**:ALERead 從「開關級結果」變「行為層結果」(這一小段 $2007 時序改行為層補)。對 74LS373
  (板級)完全 OK;$2007-read 相位是晶粒內部,行為層模它是「承認純開關級這小段做不到、改行為層」 ——
  和 even_odd/dot-339 下游 shim 同精神,規模大一點。**呼應 thesis:那 1.1% 板級/類比本就預期行為層補。**
- **工程量**:中等(一狀態機 + 一閂鎖 + 相位校準);風險低(行為層、test-mode、不碰金 checksum)。
  參考:`AprNesRef/AprNes/NesCore/ppu_new.cs`(PPU_DATA_Pipeline_Step1:readLatch/ReadALE/PPU_ALE)、
  `ppu_dispatch.cs`(ppuOctalLatch 更新 + `(baseAddr&0xFF00)|ppuOctalLatch`)。

**一句話**:140/141 不是「S1 修不了」,而是「**純開關級**修不了;S1a 的合法行為層板級模型(M5)可精確破,
到 141/141」。這把天花板從「絕對極限」精確化成「純開關級極限 + 已知的 S1a 破法」。
