# Question: can two consecutive node-state writes in a switch-level simulator be MATHEMATICALLY merged into one that does strictly less work?

## Context
Optimized C# port of the Visual6502 switch-level NES simulator (event-driven relaxation / "settle"). Each half-cycle a recalc queue is drained; popping a node re-evaluates its conducting transistor group, then writes the resolved value and enqueues affected neighbours for the next wave.

## The "pair path" (B1) — the spot in question
When a popped node `nn`'s conducting group is provably exactly two nodes `{nn, o}` (this is 77% of all group walks), we resolve the group's single value `v` (0 or 1) and write BOTH nodes:

    SetNodeState(nn, v);   // (A)
    SetNodeState(o,  v);   // (B)

`SetNodeState(x, v)` does exactly:
  1. if `state[x] == v`  →  return (no change, no enqueue at all).
  2. `state[x] = v`.
  3. enqueue x's FAN-OUT: walk x's gate-list (a per-node contiguous list of transistor endpoint ids), and for each endpoint id `c`: if `hash[c]==0` then `nextList[count++]=c; hash[c]=1;` (dedup via one global hash bitmap over all nodes).

## Hard constraints (bit-exactness — the result checksum must be identical)
- ORDER MUST hold: (A) fully completes (write THEN its enqueue walk) BEFORE (B) starts. The next-wave append order == next-wave pop order == Gauss-Seidel iteration order; reordering still converges but through different intermediate states → different checksum. So I cannot freely interleave.
- `nn` and `o` have DIFFERENT, dynamically-chosen gate-lists; the pair `{nn,o}` is different on every pop, so the two lists cannot be pre-merged at load time.
- Both nodes get the SAME value `v`.
- Dedup `hash[]` is a single global bitmap shared across the whole wave.

## The actual question
Is there a MATHEMATICAL / algorithmic identity that merges (A)+(B) into ONE computation doing strictly LESS work (fewer memory loads / fewer instructions) — NOT merely sharing setup (I already know I can share the base pointers, the queue-state hoisting, and the v==0/v!=0 branch)?

Concretely:
- Can the two gate-list walks be fused into one pass, given both append to the same dedup'd nextList with the same semantics?
- Is there an identity exploiting that both write the same `v`, or that `o` is `nn`'s conducting neighbour, that removes work?
- Or is the per-node gate-list walk fundamentally irreducible — two distinct lists must each be walked, and the only wins are the setup-sharing I already have?

Please be concrete and skeptical; if the honest answer is "no real algorithmic merge exists, only setup-sharing", say so.

## Background — do NOT re-suggest these known dead ends
- Bottleneck is DEPENDENT-LOAD LATENCY (pointer chase: read endpoint id → read state[id]/hash[id]), NOT cache capacity (L1d miss ~1.3%, working set is L1-resident). perf: backend stall ~19%, frontend ~7%.
- Already tried and FAILED: IR/AOT/codegen and GPU (all slower); bitset-parallel BFS (156x slower — average walk is only ~1.4 nodes); per-chip parallel (15x slower — too little work per wave); merging scattered state byte-loads into 64-bit word loads (only ~2.2% of co-read pairs share a 64-bit word).
- The transistor graph is ~94% one large bidirectional SCC; no static DAG / topological schedule is extractable.
- Single-threaded, memory-latency bound. Instruction count and op-cache pressure both already measured.

========================================================================
RESPONSE
========================================================================
The honest, skeptical, and mathematically rigorous answer is **No. There is no mathematical or algorithmic identity that merges these two updates into strictly less work.** 

The per-node gate-list walk is fundamentally irreducible under your hard constraints. You have already extracted the maximum possible theoretical efficiency via setup-sharing.

Here is the concrete breakdown of exactly *why* no algebraic/algorithmic reduction exists, proving that any further attempts down this specific path will be dead ends.

### 1. Information-Theoretic Irreducibility
To do strictly *less* work (e.g., fewer memory loads), you would need to predict the contents of `o`'s gate-list, or its dedup `hash[]` status, without loading them. 

In a CMOS/NMOS circuit graph, the source/drain network (which connects `nn` and `o` in a conducting group) is topologically orthogonal to the gate-drive network (which dictates the gate-lists). Knowing that `o` is the conducting neighbour of `nn` provides **zero bits of mathematical information** about which transistors `o` drives. Because there is no graph isomorphism mapping "connected via source/drain" to "shares driven gates", `o`'s gate list must be explicitly loaded and walked every time.

### 2. The Independent Guard Constraint
You are executing conceptually:
```c
bool do_A = (state[nn] != v);
bool do_B = (state[o]  != v);
```
Because `state[nn]` and `state[o]` are separate bytes in memory (and as you noted, >97% do not share a 64-bit word), they require two distinct memory loads. 

More importantly, `do_A` and `do_B` are entirely independent. There are 4 possible dynamic execution paths (neither, A-only, B-only, both). To fuse the two gate-list walks into a single contiguous algorithmic pass, you would need a single pre-merged gate-list for the pair `{nn, o}`. 
However:
* A merged list `A_then_B` is only valid if *both* `do_A` and `do_B` are true.
* If only `do_A` is true, you must walk only `A`. 
* Even though the set of edges (pairs) is statically bounded by the number of transistors, generating static `A_only`, `B_only`, and `A_then_B` arrays for every transistor would drastically inflate your structural working set, destroying your currently excellent L1d residency. 

### 3. The Strict Ordering & Hash Check Constraint
To maintain your checksum, `A`'s walk must append to `nextList` before `B`'s walk. 
If an endpoint `c` exists in both `A`'s list and `B`'s list, `A` will enqueue it and set `hash[c] = 1`. `B` will then read `c`, check `hash[c]`, see the `1`, and skip the enqueue.

To "merge" or "skip" work, you would need an identity that mathematically proves `hash[c] == 1` for `B`'s walk so you can skip the dependent load of `hash[c]`. But because the global `hash` is mutable and wave-dependent, the only way to know if `c` was enqueued by `A` (or a prior node in the wave) is to load `hash[c]`. 

### Summary of the Math
For any given pair `{nn, o}`, the minimum required operations to maintain strict Gauss-Seidel queue ordering and bit-exactness are:
1. $Load(State[nn])$ and $Load(State[o])$ — Irreducible (orthogonal memory locations).
2. $Load(GateList[x])$ — Irreducible (variable dynamic lengths, orthogonal to source/drain topology).
3. $Load(Hash[c])$ — Irreducible (wave-dependent dynamic state).

There is no Boolean logic trick, bitwise identity, or graph-theory property that bypasses these dependent loads. 

### The Only Way Forward: MLP (Memory Level Parallelism)
Since you are bottlenecked by dependent-load latency (pointer chase) rather than cache capacity or instruction count, your only remaining avenue is hiding the latency of `B` behind `A`.

You cannot algorithmicly reduce the work, but you can explicitly interleave the fetches to force the CPU to issue the loads for `B` while waiting on the dependent loads for `A`, **without** violating the commit order.

If you haven't already, decouple the state-checks from the walks:

```c
// 1. Issue independent state loads in parallel
int s_nn = state[nn];
int s_o  = state[o];

bool do_nn = (s_nn != v);
bool do_o  = (s_o  != v);

// 2. Commit states (independent, order doesn't matter for the writes themselves)
if (do_nn) state[nn] = v;
if (do_o)  state[o]  = v;

// 3. Issue gate-list pointer loads (Software Prefetching / Hoisting)
// The CPU backend can resolve these pointers while doing the branch for do_nn
ushort* list_nn = gate_lists[nn];
ushort* list_o  = gate_lists[o];
int len_nn      = gate_lengths[nn];
int len_o       = gate_lengths[o];

// 4. Strictly ordered walks
if (do_nn) {
    for (int i = 0; i < len_nn; i++) {
        ushort c = list_nn[i];
        if (hash[c] == 0) { hash[c] = 1; nextList[count++] = c; }
    }
}

if (do_o) {
    for (int i = 0; i < len_o; i++) {
        ushort c = list_o[i];
        if (hash[c] == 0) { hash[c] = 1; nextList[count++] = c; }
    }
}
```

This does exactly the same amount of algorithmic work, but it pulls the latency-bound fetches (`state`, `gate_lists`) up. If your backend stall is 19% due to pointer chasing, convincing the OOO window to overlap the array-base pointer fetches for `o` while blocked on the `hash[c]` fetches for `nn` is the absolute ceiling for optimizing this exact pair-path.
