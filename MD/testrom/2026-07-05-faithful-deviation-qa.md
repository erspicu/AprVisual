# 忠實偏差深入解說 Q&A —— 為什麼有些測試「不通過」反而是對的

> 報告裡有兩個長期 FAIL:`cpu_dummy_writes_oam` 是我們**不打算讓它通過**的
> 忠實偏差;`10-even_odd_timing` 則在 2026-07-05 **再定性**為「本模擬器雙
> netlist 整合的已知限制(約 1-dot 絕對相位偏移,待仲裁)」—— 不是真機的
> 性質,也還不是修好的東西。這份文件用 Q&A 的形式,把讀者(尤其是寫過
> 模擬器的朋友)最可能想問的問題,連同完整的證據一次說清楚。
> English version: `2026-07-05-faithful-deviation-qa.en.md`。
> 各主張的完整證據鏈見[知識庫](00_測試修復知識庫_總綱.md)與報告頁卷宗。

## 0. 什麼樣的 FAIL 才能算「忠實偏差」?

我們給這個分類設了明確的門檻,必須**同時**滿足:

1. **測試作者自己的文件**記載該行為依硬體而異(readme / 原始碼註解 / 畫面自述)
2. **社群文獻佐證**(NESdev wiki / 論壇)
3. 可行時,做**獨立模型交叉驗證**(例:emu-russia 對同一晶粒的獨立逆向)
4. **凡真機行為確定的,我們一律修**——本專案已修 10+ 項,含三個類比賽跑
   shim;這個分類不是「修不動就放進來」的箱子,而是修復手段用盡後、
   證據顯示「真機本來就這樣」的殘餘
5. 每一條都附**可驗證的預測**——歡迎任何人用真機資料檢驗

## Case 1:`10-even_odd_timing` —— 上電對齊與一個尚未仲裁的 1-dot 整合偏移

### Q|聽說真的 NES 有機會十項全過 ppu_vbl_nmi。你們的立場是什麼?

**A:** 和社群一致:**真機可以**。blargg 是在真實 NTSC 主機上開發並反覆驗證
整套測試的,公認存在能單次上電十項全過的「golden alignment」。我們的模擬器
目前十項過九項(NMI-edge 家族 8 項 + 其他),10-even_odd 這一項在我們固定的
對齊下 FAIL —— 而且我們把完整的量測結果公開:

- 上電對齊空間已**完整枚舉**(不是抽樣):K=0..5 掃描證明中間偏移量化塌縮回
  4 個相對相位類(除頻器對以整 clk0 週期重啟);「CPU ÷12 自由運轉、PPU ÷4
  在 /res 釋放時重啟」是探針實測的湧現行為,與社群黑箱量測的共識一致。
- 在**本模型**上,NMI-edge 家族 passes {7,5}、10-even_odd passes {1,3},
  零交集 —— 但真機有 golden alignment,所以這個零交集**不是真機的性質**。

### Q|那零交集是哪裡來的?

**A:** 我們的診斷(2026-07-05,經外部技術諮詢檢驗後定案):這是**雙 netlist
板級整合的絕對相位偏移**,量級約 1 dot,目前尚未仲裁。Visual2A03 與
Visual2C02 各自是準確的晶片模型,但把兩顆晶片黏在一起的「板級」部分有
三個候選偏移源(依可能性排序):

1. **$2002 讀取的跨晶片路徑被理想化為零延遲**——真機上 CPU 的 /RD 抵達 PPU、
   PPU 把 D7(VBL 旗標)送回匯流排,橫跨兩個時脈域、有實體傳輸時間;
   模擬中這段是 0,可能讓 CPU「早一個 dot」讀到 VBL。
2. reset 釋放啟動 PPU 除頻器的精確時刻(pad 上升時間等未建模)。
3. 時鐘輸入 pad 到內部除頻器的緩衝級延遲,兩顆晶片可能不同。

偏移的效果正好是把兩組測試的通過相位錯開,製造出「互斥」的表象。

### Q|為什麼選對齊 7?為什麼不乾脆把偏移修掉?

**A:** 對齊 7 是 blargg NMI-edge 測試校準的相位(8 項對 1 項)。至於修偏移:
在「量出偏移是多少」之前就去調它,等於把上電相位當成第二個手調參數 ——
那會重蹈我們對照組(TriCNES 作者手調 offset 直到全過,原始碼註解自承
「I don't know why, but this passes all the tests if this is 7, so...?」)
的路。我們選擇先公開「模型內完備枚舉 + 零交集」的量測,再做仲裁實驗:
用 emu-russia 的 **PPUSim**(同一顆 2C02 晶粒的獨立電路逆向;DMC 案例已用
它做過雙模型互證)交叉比對 `BIT $2002` 從 /RD 到資料送回的**絕對
master-clock 延遲**。量出偏移、修正、然後 NMI 家族與 10-even_odd 應當在
同一對齊下同時通過 —— 那才是把這一項真正修好的方式。

**歡迎檢驗:** 若你有 NTSC NES-001 真機,在同一次上電錄到 05-nmi_timing 與
10-even_odd_timing 都 PASS 的影像,等於直接示範 golden alignment ——
我們很樂意把它收進卷宗當作仲裁的另一路證據。

## Case 2:`cpu_dummy_writes_oam` —— 一顆版次專屬的硬體 bug

### Q|這個測試主要在驗 CPU 的 dummy write,你們過不了會不會其實是 CPU 有 bug?

**A:** 這一點有直接的對照組可以排除:姊妹測試 **`cpu_dummy_writes_ppumem`
用完全相同的 dummy-write 驗證機制、只是改用 PPU 記憶體當載具——我們通過**。
CPU 的兩次寫入行為是被獨立驗證過的;`_oam` 變體卡住的位置,正好是它自己
在畫面上聲明「真機上不可靠」的那個讀回載具。

### Q|AprNes 和 TriCNES 都通過這兩個測試,行為層做得出來,為什麼電晶體級反而不行?

**A:** 關鍵在 **PPU 版次**,NESdev 現行版寫得很清楚:

> "Writing to OAMADDR on the **2C02G** causes OAM corruption — this usually
> seems to **copy sprites 8 and 9 (address $20) over the 8-byte row at the
> target address**."
> "It is known that in the **2C03, 2C04, 2C05, and 2C07, OAMADDR works as
> intended**."

整理成時間線:

- 我們鎖定的基準機型 **RP2C02G,正是帶有這顆 OAMADDR 寫入損毀 bug 的版次**;
  後期 PPU(2C03/04/05/07)把它修掉了。
- blargg 的測試需要寫 $2003(OAMADDR)來定位讀取——在真的 G 主機上
  就會觸發損毀。他 2005 年在畫面上寫的「模擬器常過、真 NES 不過」,
  和 2026 年社群定位出的機制完全吻合。
- 行為層模擬器(AprNes / TriCNES)實作的是**修正版次的理想 OAMADDR**,
  所以通過;Mesen2 則把 G 的行為做成選配,名字就叫
  `EnablePpuOamRowCorruption`(row 損毀,與 NESdev 的 8-byte row 敘述
  一致),開啟後同樣不會通過。
- 我們的 OAM 與定址邏輯是 G 晶粒描繪出來的實體電路,損毀行為是**湧現**的。
  換句話說,「通過」的方法是把 G 的矽晶行為換成修正版次的理想陣列——
  那會讓模擬離我們鎖定的機器更遠,而不是更近。

### Q|引用測試作者的警告,就能當作不修的理由嗎?

**A:** 值得看一下作者的原文(截圖在報告頁,任何人可重跑):

> "Requirement: OAM memory reads MUST be reliable. This is often the case on
> emulators, but **NOT on the real NES**."

作者把「模擬器常過、真機不過」印在測試開頭的畫面上;搭配上一題的 NESdev
機制敘述,警告與機制互相印證。配套證據:同作者的 `oam_read` readme 記載
**同一台真機**不同次上電出現四種圖樣、其中三種以 Failed 收場——我們的
判定就落在這個已記載的家族裡(目前恰好落在通過的圖樣;會隨引擎狀態變動,
而「會變」本身正是這類行為的特性)。

### Q|你們的損毀圖樣,和 NESdev 說的「sprite 8/9 列複製」逐位元一樣嗎?

**A:** 還沒驗證到那個深度,這裡如實說明:我們尚未逐位元比對 netlist 的
損毀圖樣與 NESdev 敘述(cell 也還沒建模電荷衰減——已文件化的缺口)。
我們目前主張的是**類別一致**:G 版次「寫 OAMADDR → OAM 讀回不可靠」的
行為類別,而不是特定位元圖樣(作者自己也寫「這些值大概是我這台 NES
獨有的」——精確圖樣本來就依機而異)。逐位元比對列為開放工作。

**歡迎檢驗(版次對照版):** 依這裡的分析可以預期:真 **RP2C02G** 主機上
`cpu_dummy_writes_oam` 不會通過、`oam_read` 呈四圖樣抽籤;而 **2C03/2C05
(RGB PPU)** 主機上兩者應該通過。有 Vs./RGB 改機的朋友可以直接驗證——
無論結果偏向哪邊,都是有價值的資料。

## 關於「忠實偏差」這個分類本身

### Q|會不會任何修不好的 bug,都可以被說成「這是類比現象」?

**A:** 這個顧慮很合理,所以我們用可查核的紀錄來回答:

1. **修掉的遠多於歸類的**:計分從 115/26 走到 141/4,已修 10+ 項——
   其中三項(DMC latch、ALU latch、frame-IRQ)本身就是「類比賽跑」:
   只要真機行為是確定的,我們就修,不歸類。
2. 分類有**入場門檻**(§0 的五條),每條都附**可驗證的預測**。
3. 有獨立模型互證的先例:DMC 案例我們實際跑了 emu-russia 的 APUSim,
   兩個獨立的矽晶級模型行為一致、行為層模型則通過——這是可重複的
   實驗,不是說法。
4. 分類是**雙向的**:oam_read 曾經以失敗圖樣列在這裡,引擎修正後落到
   通過圖樣,我們就把它移出失敗清單並改寫條目——分類跟著證據走。

## 結語

cpu_dummy_writes_oam 這類忠實偏差的結構:**測試把「作者那台機器的特定物理」編碼成了期望值**。
行為層模擬器實作社群整理出的共識規則,自然全過;電晶體級模擬重現的是
一顆特定晶片的物理,於是在「物理本來就依機而異」的地方誠實地分歧。
我們希望提供的,是每一個分歧點的**電路級解釋、版次對照、和可驗證的邊界**——
讓「為什麼不通過」變成能被檢驗的知識,而不是一句帶過的藉口。
