Here is the brutally honest, engineering-focused assessment of where your remaining performance lives.

First, the reality check: **Even if you eliminate 70% of your nodes and cut events in half, you will speed up by roughly 3x to 5x.** That takes you from 115K to ~500K half-cycles/s. **You will still be ~80x short of 42.95M real-time.** Bridging a 370x gap requires moving off dynamic event-driven switch-level simulation entirely (e.g., statically scheduled compiled C++ for the logic, leaving only the buses event-driven). 

However, if your goal right now is to push the event-driven behavior-correct model to its absolute limit, here are the answers to your specific levers.

### 1. Sequential / Latch Abstraction (The 8,311 no-pull-up nodes)
**The Verdict:** **DO IT.** This is your biggest untapped goldmine. 
**Why it's different from your failed AOT approach:** Your AOT macro-block approach failed because it lost sparsity—it forced you to evaluate the *entire* register file or ALU every cycle, ruining the I-cache and doing useless work. An **Event-Driven Sequential Primitive** retains sparsity. You collapse a 4-transistor dynamic latch into one state-holding node. You only queue an event for it when its specific `clock` or `data_in` toggles. It is sparse execution of dense state. 
**What's hiding in the 8,311 nodes:** The 2C02 (PPU) contains 256 bytes of OAM (2048 bits) and 32 bytes of Palette RAM (256 bits). The 2A03 contains CPU registers. Because NMOS dynamic memory relies on pass-transistors trapping charge on gate capacitance, these act as switch-level nodes. OAM alone is likely chewing up ~6,000 to 8,000 nodes.
*   **Realistic Node Ceiling:** **~30% to 40% (4,000 - 5,500 nodes).** Collapsing OAM, Palette RAM, and CPU registers into array-backed primitive nodes will gut your no-pull-up node count.
*   **Realistic Compute Ceiling:** Huge. You eliminate the expensive BFS group-resolution for thousands of memory pass-transistors.
*   **Cheapest Load-Time Measurement:** Iterate your pass-transistors. Find every pass-transistor where: 1) the gate is connected to a known clock/control line, and 2) the drain connects *exclusively* to the gate input of a pull-down transistor (pure capacitance). Count these drains. Multiply by the typical bit-cell node count.

### 2. The 1,076-Node Channel Component (Bus/Decoder)
**The Verdict:** **ABSTRACT IT.** 
**Why:** A 1,076-node Channel-Connected Component (CCC) is fatal for an event-driven switch simulator. Whenever a node in this component toggles, your engine likely fires a BFS that touches up to 1,000 nodes to resolve charge sharing and drive strength. This is almost certainly the main data bus + register file read/write pass-network, or OAM evaluation buses.
*   **How:** Abstract it as a **Tristate/Mux Bus Primitive**. Instead of a switch-graph, treat the bus as a single behavioral entity with `N` writers and `M` readers. When a writer's pass-gate control (e.g., `read_enable`) goes high, O(1) propagate the writer's value to the bus, and queue events for the currently connected readers. 
*   **Realistic Node Ceiling:** Only cuts ~1,000 nodes (**~7%**), but drastically reduces BFS time complexity.
*   **Cheapest Load-Time Measurement:** Run a mock BFS from a bus line in that component. Count how many nodes are purely pass-transistor sources/drains that never drive a gate. That’s your extractable bus fabric.

### 3. Deeper Logic Folding (Fanout-2, Multi-level chains)
**The Verdict:** **DEAD END. SKIP IT.**
**Why:** As you noted, pure logic abstraction tops out at ~15%. You have 1,110 fanout-2 nodes. Even if you aggressively fold half of them, you save ~500 nodes. Extracting a full 8-bit adder block (like the 2A03 ALU) saves maybe 150 nodes. The pattern-matching complexity (recognizing carry-lookahead chains in NMOS) is monumental, and the runtime payoff is negligible.
*   **Realistic Node Ceiling:** **2% to 4%** max. Diminishing returns hit hard here.
*   **Cheapest Load-Time Measurement:** Count fanout-2 nodes where *both* consumers are purely pure-logic pull-down inputs.

### 4. Event-Count Reduction (Clock Phase Gating)
**The Verdict:** **HIGHLY LUCRATIVE.** 
**Why:** The NES uses a strict two-phase non-overlapping clock (Phi1 / Phi2). In authentic NMOS, logic transitions ripple through a combinatorial block, causing glitches (multiple events per node) before settling at a pass-transistor that is currently turned OFF by the clock. Your simulator is currently evaluating all those intermediate glitches.
*   **How (Phase-Region Gating):** If you know a sub-net ends at a Phi2 pass-transistor, and the system is currently in Phi1, *you don't need to evaluate that sub-net yet*. Better yet: delay the propagation of events through logic gates until the clock phase that actually samples them opens. Alternatively, use **Activity-based evaluation**: if an event reaches a pass-transistor that is OFF, drop the event. Do not BFS.
*   **Realistic Event Ceiling:** **20% to 35% fewer events.**
*   **Cheapest Load-Time/Run-Time Measurement:** Add a counter to your event loop. Every time an event is popped, check if the node's only consumers are pass-transistors whose gate control is currently 0. If yes, increment a `Wasted_Events` counter. Run for 1 frame. `Wasted_Events / Total_Events` is your exact ceiling.
