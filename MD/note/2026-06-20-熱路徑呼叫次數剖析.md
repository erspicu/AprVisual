# S1 熱路徑 chain — 各 method 呼叫次數與比例剖析(`[hotpath-calls]` profiler)

> 日期:2026-06-20 · 引擎:`src/AprVisual.S1`(main) · 量測:`full_palette.nes` 400,000 half-cycles `--extra-ram` · checksum golden `0x9174E19D961CB6E5`(DEBUG 插樁 bit-exact,Release byte-identical)

## 目的
在 **DEBUG 條件編譯**下,於熱路徑 call-chain 的每個 method 進入點放一個計數器,量出整個效能測試中**每個 method 的呼叫次數 + 佔比 + 每 pop 平均呼叫數**,看工作量實際分布在哪裡。`#if DEBUG` only —— Release 完全不受影響(計數增量全部編譯掉)。

## 熱路徑 chain 結構(call graph)
```
StepCycle (每半週期)
  └─ ProcessQueue            ← settle 驅動;一個半週期內反覆掃 work-list 直到收斂
       └─ RecalcNode(nn)     ← 每 pop 一次(分母);依 IsPureLogic[nn] 分派:
            ├─ cls1 / cls2-all-off → RecalcNodeFast(nn)        ── O(1) 單節點解析
            ├─ cls2-some-on(2 節點)→ B1 pair path(inline)   ── 不呼叫下面三個
            └─ cls0 / 多節點 → ComputeNodeGroup(nn)            ── 慢路徑
                                 ├─ AddNodeToGroup(seed)       ── 迭代式 BFS 走訪
                                 └─ GetNodeValue()             ── 群組解析(256-entry LUT)
       (上述任一解析出新值 → SetNodeState(nn, v) 寫值 + enqueue 鄰居到下一 wave;
        群組解析會對「每個群組成員」各呼叫一次 SetNodeState)
```

## 量測結果

| Method | 呼叫次數 | 佔 chain 總呼叫 | 每 RecalcNode pop |
|---|--:|--:|--:|
| ProcessQueue | 416,897 | 0.1% | 0.0023 /pop |
| **RecalcNode** | **182,457,809** | **29.1%** | **1.0000 /pop**(分母) |
| RecalcNodeFast | 128,033,212 | 20.5% | 0.7017 /pop |
| ComputeNodeGroup | 27,766,908 | 4.4% | 0.1522 /pop |
| AddNodeToGroup | 27,766,908 | 4.4% | 0.1522 /pop |
| GetNodeValue | 27,766,908 | 4.4% | 0.1522 /pop |
| **SetNodeState** | **231,777,980** | **37.0%** | **1.2703 /pop** ← 最熱 |
| chain 總計 | 625,986,622 | 100% | — |

同一次跑的相關 profiler(交叉印證):
- `[settle-pass-dist]`:ProcessQueue calls = 416,897,每次 settle 平均 **12.08 passes**(max 45,p99 33)。
- `[bfs-depth-dist]`:AddNodeToGroup walks = 27,766,908,BFS 深度 **mean 1.25**(MAX 14,p99 3)。

## 解讀
1. **SetNodeState 是呼叫量第一名(37%、1.27/pop)。** 因為群組解析會對群組裡每個成員各寫一次,所以 >1/pop。它內含的 **turn-on/turn-off enqueue 迴圈就是全引擎最熱的程式碼**(與先前 branch-dist/Rule B 量到 turn-off enqueue n≈125.7M、turn-on n≈108.5M 一致)。
2. **每個 pop 的解析路徑分布(三者相加 = 100%)**:
   - **70.2%** → `RecalcNodeFast`(O(1) 單節點;= cls1 靜態 + cls2 動態全-off)
   - **15.2%** → `ComputeNodeGroup` 慢路徑 BFS(cls0 + 多節點群組)
   - **~14.6%** → B1 雙節點 pair path(inline 解析,不進上面三個)
3. **慢 BFS 只佔 15.2% 的 pop**,且 `ComputeNodeGroup ≡ AddNodeToGroup ≡ GetNodeValue`(各 27.77M,1:1:1)印證 BFS chain 一次建群只走一次 walk + 一次 resolve;B1 pair path 不另呼叫 GetNodeValue(自行 inline)。
4. **ProcessQueue 極少(0.1%)** —— 它是粗粒度的「每半週期 settle 驅動」,真正的工作在它底下的 per-pop 迴圈。
5. 整體形狀與已知特性吻合:~70% 單節點、稀疏活動、群組小(深度 1.25)。**熱點集中在 SetNodeState(寫+enqueue)+ RecalcNode(分派)+ RecalcNodeFast(快解析)三者(合計 86.6% 的呼叫)**;最佳化若要動,得動這三者(但歷史上它們都已高度最佳化,見記憶體延遲調查結論)。
