# Settle-pass 分佈量測(ProcessQueue 每次跑幾個 wave)2026-06-08

加了一個 DEBUG-only profiler(`[settle-pass-dist]`,程式在 `WireCore.Recalc.cs` 的 `SettlePassTally` + TestRunner 報表),統計每次 `ProcessQueue()` 要跑幾道 settle wave(`iteration` 計數)才收斂。**DEBUG 專用、Release 零成本、計數與 Release 完全相同(同演算法)。** 目的是回頭檢視 `MaxSettlePasses` 防呆機制(Release 已移除該 cap)到底離真實尾端多遠。

## 量測設定

`full_palette.nes` · `--bench-hc 300000` · `--extra-ram`(標準驗證點),checksum `0x794A43ABDF169ADA` = golden(bit-exact,profiler 不影響行為)。

統計涵蓋**所有** `ProcessQueue` 呼叫 —— 每半週期的時鐘 settle + 較小的 handler settle(記憶體寫入經 WriteBits)都算進去。

## 結果

```
ProcessQueue calls = 310,905   (~1.04 次 / hc)
mean = 12.09 passes   max = 45
p50 = 9   p90 = 19   p99 = 33   p99.9 = 34   p99.99 = 34
```

## 解讀

- **最少深度是 7**:幾乎沒有 2~6 pass 的 settle(只有 4 次是 1 pass,屬開機/邊界)。代表每次時鐘邊緣 settle 都要 ≥7 道波才能把訊號傳過 netlist 的關鍵鏈。
- **雙峰(~8 與 ~16)**:64% 集中在 7~10、20% 集中在 15~17。幾乎可確定是**兩個時鐘相位**(rising / falling 邊做的傳播量不同)。
- **尾巴到 34、極少數到 45**:p99=33、絕對最大 45,且 35~45 只有零星 1~2 次。
- **`MaxSettlePasses=128` 防呆綽綽有餘**:實測最大才 **45**,cap 在 ~2.8× 之上,**永遠不會觸發** —— 它純粹是開發期「收斂壞掉」的 tripwire,Release 已整個移除(該 in-loop guard 在 Release 量過 +2.77% 的代價,所以拿掉)。先前筆記「max 45 / p99 in [33-64]」獲得確認。

> 旁註:`passes` 是 settle **深度**(波數),不是每波處理的節點數;節點層面的浪費由另一支 `[waste-profile]`(no-change pop)量測。兩者互補。

## 重現方式

```
dotnet run -c Debug --project src/AprVisual.S1 -- --benchmark <rom> --bench-hc 300000 --extra-ram --system-def-dir <data> 
# 讀 "# [settle-pass-dist] ..." 那段(直方圖 + p50/p90/p99/p99.9/max)
```
