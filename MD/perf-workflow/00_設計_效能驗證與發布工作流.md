# 效能驗證與發布工作流 — 設計文件

> 建立 2026-06-19。一套自動化 pipeline:**自動量測各 release 的效能參數 → raw data → 處理成 JSON →
> 部署到 `https://baxermux.org/myemu/AprVisual/version/`**,可用 **HTML 檢視 / PHP API / 純 JSON** 三種方式
> 撈各版的各種效能檢測參數。`/version` 下分 **x64** 與 **arm64** 兩個獨立區域(本期先做 x64 boost;ARM 待 Pi 環境就緒)。

---

## 1. 目標與範圍
- **誰**:Zen2 桌機(Ryzen 7 3700X)**boost 模式**(真實世界、不鎖頻、best-3-of-5)為 x64 主指標。
- **要什麼**:不只一個吞吐量數字,而是**每版一組完整參數**(吞吐量分布、鎖頻週期數、IL/機械碼大小、checksum、環境)。
- **自動**:一道指令完成「探索 release → (快取)取得二進位 → 跑 N 次 → 彙整 → 產 JSON → 部署」。
- **可查**:部署後可由瀏覽器(HTML 圖表)、程式(PHP API)、或直接抓 JSON 三種途徑取得。
- **非目標(本期)**:CI 自動觸發(先手動 `pwsh` 觸發)、ARM 實測(Pi 修好後接 §10)、與 leaderboard 排行榜合併(兩者並存、互不干擾)。

---

## 2. 架構(四階段)

```
                       本機 (Windows / Zen2)                              遠端 baxermux.org
  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐      ┌─────────────────────────┐
  │ 1 COLLECT │ → │ 2 PROCESS │ → │  (data)  │ → │ 3 DEPLOY  │ ──→ │ 4 VIEW  /myemu/.../version │
  │ 跑各版binary│   │ raw→彙整  │   │ x64.json │   │ FTPS 上傳 │      │  HTML · PHP API · JSON    │
  └──────────┘   └──────────┘   └──────────┘   └──────────┘      └─────────────────────────┘
   raw JSONL       best-3/CV       schema驗證      只傳異動            x64 區 / arm64 區(獨立)
```

- **1 COLLECT**(`tools/perf/collect_x64.ps1`):探索 release、取得二進位(快取)、每版跑 5 次,輸出 **raw JSONL**(一行一次跑)。
- **2 PROCESS**(`tools/perf/build_json.py`):raw JSONL + 靜態 metadata → 彙整(best-3 平均、median、min/max、CV)→ 產出 **`x64.json`**(通過 schema 驗證)。
- **3 DEPLOY**(`tools/perf/deploy_version.ps1`):FTPS 把 `x64.json` + 前端(html/php/js)上傳到 `/public_html/myemu/AprVisual/version/`。
- **4 VIEW**:靜態 `index.html` 執行時 fetch `data.json` 繪圖(資料更新只需重傳 JSON);`api.php` 提供程式化查詢;`data.json` 可直接抓。

> **總指令**:`pwsh tools/perf/run_all.ps1 -Platform x64 -Deploy`(串 1→2→3)。不加 `-Deploy` 只產本機 JSON。

## 3. 資料模型(JSON schema)

每平台一個 `data.json`:`meta`(環境,整批一次)+ `versions[]`(每版指標 + metadata)。

```jsonc
{
  "schema": 1,
  "platform": "x64",
  "cpu": "AMD Ryzen 7 3700X (Zen2)",
  "mode": "boost",                       // boost | locked
  "generated": "2026-06-19T12:00:00Z",
  "golden_checksum": "0x9174E19D961CB6E5",
  "realtime_hc_s": 42954552,             // NES 即時門檻(算 ×倍距離用)
  "env": {
    "os": "Windows 11 26200", "dotnet_sdk": "11.0.100-preview.4",
    "cores": 8, "base_ghz": 3.6, "boost_ghz": 4.4,
    "l1d_kb": 32, "l2_kb": 512, "l3_mb": 32, "ram_gb": 32,
    "pinned": false, "locked": false, "cooled_idle": true, "bench_hc": 400000, "runs": 5
  },
  "versions": [
    {
      "version": "2026.06.19", "date": "2026-06-18",
      "commit": "270259a", "tfm": "net11.0",
      "title": "disasm 驅動 codegen 微優化",
      "desc": "讀 JIT x64 找的 5 個 bit-exact 優化",
      "milestone": false,
      "metrics": {
        "hc_s_best3": 138708, "hc_s_median": 132690,
        "hc_s_min": 129900, "hc_s_max": 139500, "hc_s_cv_pct": 0.94,
        "hc_s_runs": [139500, 138900, 137700, 132690, 129900],
        "cyc_per_hc_locked": 31000,          // 選配:鎖頻量(抗噪)
        "il_size": 136, "native_size": 5321, // ProcessQueue
        "checksum": "0x9174E19D961CB6E5", "bit_exact": true,
        "realtime_x": 309                    // = realtime_hc_s / hc_s_best3
      }
    }
    // ... 其餘版本
  ]
}
```

- 設 `schema` 版號便於日後演進。
- `bit_exact = (checksum == golden_checksum)`;**任一版 false 應在報告醒目標示**(資料完整性門檻)。
- `cyc_per_hc_locked`、`il_size`、`native_size` 為選配(若該批有跑鎖頻/JitDisasmSummary)。

## 4. 指標目錄(要記什麼參數)

| 類別 | 欄位 | 來源 / 怎麼量 |
|---|---|---|
| **吞吐量(主)** | `hc_s_best3` / median / min / max / runs[] / cv_pct | release binary `--benchmark --bench-hc 400000` ×5,boost |
| **週期(抗噪,選配)** | `cyc_per_hc_locked` | QueryProcessCycleTime + 鎖頻(PERFBOOSTMODE 0),(400k−40k)/360k |
| **碼大小** | `il_size` / `native_size` | `DOTNET_JitDisasmSummary=1 DOTNET_TieredCompilation=0` → `WireCore:ProcessQueue` |
| **正確性** | `checksum` / `bit_exact` | benchmark 輸出的 NodeStates checksum vs golden |
| **即時距離** | `realtime_x` | `42,954,552 / hc_s_best3` |
| **版本 metadata** | version/date/commit/tfm/title/desc/milestone | 人工維護的 `metadata.csv`(沿用報告的逐版說明) |
| **環境** | cpu/clock/cache/ram/os/dotnet/pinned/locked | 收集時自動偵測 + 手填一次 |
| **(ARM 後續)** | IPC / L1d-miss / L1i-miss / branch-miss / temp / freq | `perf stat` + `vcgencmd`(見 §10) |

## 6. COLLECT(x64,boost)
1. **探索 + 快取**:`gh release list` 取所有 `benchmark-*`;每個 zip 若 `tools/perf/releases/` 已有就跳過,否則 `gh release download` 下載(符合「下載後保留重複使用」)。
2. **跑**:解壓 → `AprVisual.S1.exe --benchmark <rom> --bench-hc 400000 --extra-ram` ×5,解析每次的 `rate:` 與 `checksum`。
3. **(選配)鎖頻週期**:`-Locked` 開關 → 跑前 `powercfg PERFBOOSTMODE 0` + QueryProcessCycleTime 量 cyc/hc,跑後還原 `PERFBOOSTMODE 2`(沿用既有 cpu_lock/restore 機制與「測完恢復」原則)。
4. **(選配)碼大小**:`DOTNET_JitDisasmSummary` 跑一次(短 hc)抓 IL/native。
5. **輸出**:`raw/x64_<timestamp>.jsonl`,一行一次跑:`{version, run, hc_s, checksum, ...}`。
- 第一版若無版號資訊、舊版用 HC=300000 → 在 metadata 標注;一律以 **HC=400000** 重量(與報告一致)。
- 機器須**冷機、idle**(boost 才準);建議可選 `-Pin`(thread affinity + High priority)降變異。

## 7. PROCESS
`build_json.py`:讀 raw JSONL → 依 version 分組 → 算 best-3 平均 / median / min / max / CV → 併 `metadata.csv` 的逐版說明 → 偵測環境 → 輸出 `out/x64.json` → **schema 驗證**(欄位齊、型別對、checksum 比對 golden 標 `bit_exact`)。
> 重跑可累加(以 version 為 key 覆寫),不必每次全跑;`metadata.csv` 是唯一人工維護點。

## 8. DEPLOY(FTPS)
`deploy_version.ps1`:讀 `etc/ftp.txt`(gitignored)組 curl `-K` cfg(`--ssl-reqd`/insecure/`user=acct@host:pass`),`--ftp-create-dirs` 上傳:
- `out/x64.json` → `/version/x64/data.json`
- `web/*`(index.html/api.php/app.js/app.css)→ `/version/`(僅前端變動時)
> 沿用既有 FTPS 部署慣例,**絕不覆寫** `/myemu/AprVisual/` 的 `config.php`/`data`。資料更新只傳一支 JSON。

---

## 9. VIEW —— 三種存取

### (a) HTML 儀表板 `/version/`(+ `app.js`)
靜態頁,**執行時 fetch `x64/data.json`** 繪製:雙指標圖(boost hc/s + 鎖頻 cyc/hc)、逐版說明卡、完整表、即時距離、平台 tab(x64 | arm64)。資料更新 = 重傳 JSON 即可(不必改 HTML)。風格沿用 `MD/REPORT/perf-history*.html`。

### (b) PHP API `/version/api.php`(回 JSON,帶 CORS)
| 請求 | 回應 |
|---|---|
| `?platform=x64` | 整份 `x64.json` |
| `?platform=x64&version=2026.06.19` | 該版單筆 |
| `?platform=x64&metric=hc_s_best3` | `[{version,value}...]`(畫圖用) |
| `?platform=x64&latest=1` | 最新版摘要 |
| `?platform=x64&format=csv` | CSV 匯出 |
| (無參數) | 平台清單 + 各自最新摘要 |
作法:讀對應 `x64/data.json` 再過濾輸出(唯讀、無寫入、無 secret)。

### (c) 純 JSON
直接抓 `/version/x64/data.json`(給外部工具 / 論文 / 自動化)。

## 11. 實作階段
1. **P1 — x64 collect + json**:`collect_x64.ps1` + `build_json.py` + `metadata.csv`,本機產出 `x64.json`(可先用我已量的 perf-history 數據回填驗證 schema)。
2. **P2 — 前端**:`web/index.html` + `app.js`(fetch JSON 繪圖)+ `api.php`,本機開 `data.json` 測。
3. **P3 — deploy**:`deploy_version.ps1` FTPS 上 `/version/`,線上驗證 HTML / API / JSON 三路。
4. **P4 — arm64**:Pi 修好後接 §10。
5. **P5(選配)**:每次 release 後自動跑(掛進 `release_benchmark.ps1` 尾段或排程)。
