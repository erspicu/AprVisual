RESPONSE
========================================================================
Your troubleshooting and isolation are excellent. The fact that standard PMC EBP works perfectly while IBS captures exactly zero samples—without throwing an initialization or driver lock error—strongly points to a hardware-to-OS interrupt routing failure. The IBS hardware is arming, overflowing, and then silently hanging or having its interrupt dropped by the Windows HAL.

Here is the concrete, prioritized analysis and steps to resolve this, addressing your hypothesis and questions directly.

### Q1 & Q2: The x2APIC / LVT Routing Hypothesis (Ranked Root Causes & Fixes)

**Your hypothesis is highly probable.** 
Standard PMCs trigger the standard Performance Monitor Interrupt (PMI), which uses the standard LVT PC register. 
**IBS does NOT use the standard LVT PC.** It requires the driver to allocate and program an **Extended Interrupt LVT (EILVT)**. AMD CPUs provide 4 EILVTs. 

If Windows 11 has transitioned the APIC to **x2APIC mode**, APIC registers must be accessed via MSRs rather than MMIO. If the AMD uProf driver (`AMDCpuProfiler.sys`) relies on legacy MMIO to program the EILVT, or if the Windows 11 HAL strictly guards EILVT allocations in x2APIC mode, the driver will silently fail to program the interrupt. The IBS counter overflows once, waits for an interrupt service routine that never fires, and stops.

**How to check current APIC mode in Windows:**
Open an admin PowerShell and check the Windows System Event Log for the Kernel-Boot APIC initialization:
```powershell
Get-WinEvent -ProviderName "Microsoft-Windows-Kernel-Boot" | Where-Object {$_.Id -eq 153 -or $_.Id -eq 20} | Select-Object -First 5 -Property TimeCreated, Message
```
*(Alternatively, use Sysinternals `Coreinfo.exe -v` to see if the hardware supports x2APIC, though it doesn't definitively tell you Windows is using it).*

**How to FIX / Force xAPIC (Legacy APIC) mode:**
Instead of hunting for BIOS settings (which consumer boards often hide), force Windows to use standard xAPIC via the Boot Configuration Data (BCD). Open an elevated command prompt:
1. `bcdedit /set x2apicpolicy Disable`
2. `bcdedit /set uselegacyapicmode Yes`
3. **Reboot.**

*If IBS starts working after this reboot, your hypothesis is 100% confirmed.* (To revert later: `bcdedit /deletevalue x2apicpolicy` and `bcdedit /deletevalue uselegacyapicmode`).

### Q3: BIOS Settings Gating IBS on Zen 2

There is no dedicated "Enable IBS" toggle. However, specific settings interfere with the APIC, MSR accessibility, and interrupt routing:

1. **SVM Mode (Secure Virtual Machine):** Turn this **OFF**. Even with Windows VBS/HVCI off, an enabled SVM can cause the BIOS to load ACPI tables that reserve certain hardware features or change how Windows handles hypervisor-level intercepts.
2. **IOMMU:** Turn this **OFF** (or set to "Disabled", not "Auto"). IOMMU handles interrupt remapping. A bug in the AMD AGESA / IOMMU ACPI tables can cause the EILVT interrupts to be misrouted or dropped.
3. **Local APIC Mode:** If visible in your BIOS (often under Advanced -> CPU Configuration or AMD CBS), explicitly change it from "Auto" to **"Compatibility"** or **"xAPIC"**. 

*Settings like Global C-State Control, CPB, or CPPC do not prevent sample collection; they only affect the clock frequencies and latency calculations.*

### Q4: uProf 5.3.521 Bugs and Windows 11 Driver Conflicts

Windows 11 (especially 24H2/25H2) introduced strict enforcement regarding how drivers can register NMI (Non-Maskable Interrupt) callbacks. IBS is typically routed as an NMI to bypass IRQL masking. If the `AMDCpuProfiler.sys` driver uses an older API to hook the NMI, Windows 11 silently drops the hook.

1. **Clean Reinstall / Downgrade:** If the BCD fix above fails, the 5.3 driver might be broken on your specific Windows build. Downgrade to **uProf 4.2** (which used a slightly different driver architecture).
2. **The required cleanout sequence:**
   * Close uProf.
   * `sc stop AMDProfilerLoadService`
   * `sc delete AMDProfilerLoadService`
   * `sc stop AMDCpuProfiler`
   * `sc delete AMDCpuProfiler`
   * Go to `C:\Windows\System32\drivers` and physically delete `AMDCpuProfiler.sys`.
   * Reboot.
   * Install uProf.

### Q5: Other Windows-Side Causes (Security Residue)

You verified VBS is off via WMI, but Windows 11 is notorious for "residue" security policies that block MSR/APIC access at the HAL level.
Double-check these specific registry keys from an Admin prompt:
```cmd
reg query "HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard\Scenarios\HypervisorEnforcedCodeIntegrity"
reg query "HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard"
```
Ensure `Enabled` is `0`.

Furthermore, **Vulnerable Driver Blocklist**: Windows 11 strictly blocks drivers it deems vulnerable. Check Windows Security -> Device Security -> Core Isolation -> **Microsoft Vulnerable Driver Blocklist**. If this is ON, it might be silently blocking the specific MSR-write commands the AMD driver uses to arm IBS, while allowing the standard PMC MSRs. Toggle it OFF for testing.

### Q6: Alternatives for Data-Address Attribution on Zen 2

**If IBS is genuinely dead on this machine, are there alternatives?**

**Hardware/Platform alternatives:**
*No.* On AMD architecture, IBS (specifically the `IbsDcLinAd` register populated by IBS-Op) is the *only* hardware mechanism that captures the exact data linear address of a cache miss. Core PMCs only capture the Instruction Pointer (RIP). (Intel has PEBS Data Linear Address; AMD relies entirely on IBS for this).

**Software/ETW alternatives:**
ETW (Event Tracing for Windows) with the `ProcessorProfile` or `SystemProfiler` flags uses the exact same underlying PMC hardware as uProf. It will give you the instruction pointer, but **not** the data address.

**The Practical Workaround (Static Analysis + PMC):**
If you are strictly limited to per-instruction (PMC) profiling:
1. Use uProf to find the exact *Instruction Pointer (RIP)* causing the `L2_CACHE_ACCESS_FROM_L1_DC_MISS`.
2. Map that RIP back to your .NET assembly (using uProf's JIT unwinding or WinDbg).
3. Look at the x64 disassembly for that instruction (e.g., `mov rax, [rcx + 18h]`).
4. If you know the class/struct layout of your native arrays, you can infer *which* data structure is missing the cache based on the register offset being accessed by that specific instruction.

### Summary Action Plan
1. Open Admin CMD and run the **two `bcdedit` commands** to force legacy APIC mode, then reboot. (This takes 2 minutes and is the most likely fix).
2. Disable **SVM** and **IOMMU** in the BIOS.
3. Completely purge the `AMDCpuProfiler.sys` driver using `sc delete` and downgrade to an older uProf version.
