這是一次非常漂亮且深入的硬體級除錯。你所收集的探針數據直接切中了問題的核心。針對你的三個問題，以下是基於 NES 硬體架構與電晶體級行為的**最終判定與建議**。

### 1. `held == $FF` 是過擬合還是通則？
**結論：這是 100% 的過擬合 (Overfitting)，純屬巧合，絕非通則。**

**硬體原理解釋：**
*   **74LS373 是數位元件，不吃浮接 (High-Z) 的自然電壓：** 你提到 High-Z 時匯流排可能有弱上拉或寄生電容導致 `$FF` (Open bus 現象)。在 NES 的 CPU 側確實有此現象，但在此處**不適用**。因為提供低八位位址的是 **74LS373 閂鎖器 (Latch)**。閂鎖器的特性是：在 `ALE` (Address Latch Enable) 訊號下降沿 (High to Low) 時，鎖住當時 AD0-7 的數位狀態。一旦鎖住，不管 PPU 的 AD 腳後來變成 High-Z 還是拔掉，閂鎖器的輸出就是死死咬住上一次鎖住的 0/1 值。
*   **為什麼剛好是 `$FF`？** 在 `boing2k7` 觸發 desync 的那一刻前，PPU 最後一次正常執行的 fetch，很可能是讀取 Nametable (`$2FFF`，低八位是 `$FF`) 或是某個剛好為 `$FF` 的 Attribute byte。74LS373 忠實地鎖住了這個 `$FF`。
*   **為什麼是過擬合？** 如果換另一個測試 ROM，或者你在 `boing2k7` 觸發 desync 的前一個週期，讓 PPU 去讀 `$2F55`，那麼閂鎖器持有的就會是 `$55`。如果你用 `held == $FF` 來當作 BOL 的觸發條件，只要未來遇到一個背景是 `$00` 的遊戲發生 desync，這個 Patch 就會失效。因此，它不能作為判據。

### 2. 有沒有比 BOL 更理想的做法？（你的想法非常接近真理）
你提到的這段話：*「偵測『當前 fetch 的低八位是否與上一次 ALE 期真正被驅動上匯流排的低八位』不一致」* —— **這不只是一個判據，這正是 74LS373 閂鎖器在現實世界的物理定義！**

S1 引擎最大的盲點在於：它把 PPU 當成一個黑箱，直接拿 PPU 內部計算出的位址 (如 `$0F04`) 丟給外部記憶體。但**在真實的 NES 裡，CHR 記憶體從來沒有直接接在 PPU 的 AD0-7 上！** CHR 記憶體的 A0-A7 是接在 74LS373 的輸出端。

**終極解法：實作一個「虛擬 74LS373 (Virtual Latch)」來取代目前的 BOL Shim。**
與其在 `$2007` 讀取窗內做骯髒的攔截，不如直接在 S1 的記憶體介面層補上這個缺失的硬體：

1.  **監聽 ALE 腳位：** 在 S1 中掛一個探針在 `ALE` 輸出腳上。
2.  **模擬 Latch 行為：** 當偵測到 `ALE` 發生 **下降沿 (1 -> 0)** 時，將當時 S1 的 `AD0-7` (或 `ab0-7`) 的值存入一個變數 `virtual_74ls373`。
3.  **全局位址覆寫 (Global Address Override)：** 在 S1 向外部 CHR ROM/RAM 發起**任何讀/寫請求**時，不要直接用 S1 算出來的 14-bit 位址，而是強制將位址組裝為：
    *   `High 6 bits` = PPU 的 `A8-A13` 腳位狀態 (或 S1 算出的高六位)
    *   `Low 8 bits` = `virtual_74ls373`

**為什麼這能完美區分 2007Stress 和 ALERead，而且不需要任何 If/Else 判斷？**
*   **在 `2007Stress` (正常讀取) 中：** ALE 會正常脈衝。`virtual_74ls373` 會被正確更新為當前的低八位。因此，`High_6 | virtual_74ls373` 組出來的位址，會**完全等於** S1 原本想讀的正常位址。這就解釋了為什麼兩者一致，不會有任何誤殺。
*   **在 `ALERead` (Desync 發生) 中：** 因為 PPU 內部的時序亂掉，它在 fetch 時**沒有正確發出 ALE 脈衝** (或者脈衝時機錯了)。因此，`virtual_74ls373` **不會更新**，它保留了上一次的舊值 (`$FF`)。當 S1 嘗試用 `$0F04` 去讀取時，我們的 Virtual Latch 邏輯會把它強制改成 `$0FFF` (High_6 `$0F` | Old Latch `$FF`)。這精準重現了 NESdev 描述的 "hybrid address"！

### 3. 我的誠實判斷與下一步行動建議

這是一個分水嶺。你現在面對的不是軟體 bug，而是 S1 架構上少模擬了一顆外部 IC (74LS373) 所導致的必然結果。

**我的專業建議：**
1.  **放棄 `held == $FF` 和 `$2007` 專用 Shim：** 這是治標不治本的特例，未來一定會破功。
2.  **花 1~2 小時嘗試實作「Virtual Latch」：**
    *   按照上述邏輯，直接攔截 S1 對 VRAM/CHR 的定址。
    *   這不需要 S1 支援 High-Z！因為只要 S1 的 **ALE 腳位模擬是準確的**（即：在 desync 時 ALE 確實沒有正確 trigger），這個虛擬 Latch 就能完美吃下 S1 輸出的 0/1 訊號，並自動產生 Hybrid Address。
3.  **如果 Virtual Latch 成功了：** 恭喜你，你不只修好了一個測試，你是幫 S1 補完了一個關鍵的硬體架構，這個解法無懈可擊。
4.  **如果 Virtual Latch 失敗了：** 這代表 S1 在 desync 時，**連 `ALE` 腳位的電晶體行為都模擬錯了**（例如它在 desync 時依然錯誤地發出了 ALE 脈衝，導致 Latch 被更新）。如果真的是這樣，那就觸及了 S1 電晶體網表的死穴。**此時，請毫不猶豫地退回 140/141 的狀態。** 把這個 Test 標記為 `UNSUPPORTED_DUE_TO_ENGINE_LIMITATION`，在文件裡寫明：「需要精確的 ALE 時序與 74LS373 閂鎖器分離模擬，S1 架構無法天然重現」。

這是一場很棒的追根究底。去試試 Virtual Latch 吧，期待你的結果！
