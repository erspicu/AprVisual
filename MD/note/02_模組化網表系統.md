# MetalNES — 模組化網表系統

MetalNES 最有設計感的部分：**用一套統一的「電晶體 + 線網」描述法，把任意數量的晶片組合成一台機器**。這正是我們設計文件講的「全封裝晶片架構」，只是 MetalNES 把它做成了一個資料驅動的模組系統。

相關檔案：`wire_defs.h/.cpp`、`wire_node_resolver.h/.cpp`、`wire_module.cpp`（`Wires` 部分）、`data/system-def/*.js`。

---

## 1. 模組定義檔（`.js`）

每個 `.js` 檔是一個 `var module = { ... }` 的 JSON-like 物件，由 `wire_defs.cpp` 用自家 JSON parser 讀。欄位：

| 欄位 | 意義 |
|------|------|
| `name` | 模組類型名 |
| `description` | 說明文字 |
| `pins` | `[[pin_no, "name"], ...]` — DIP 接腳號 ↔ 內部節點名 |
| `nodenames` | `{ "name": id, ... }` — 節點名 → 節點 id（可內嵌，或用 `nodenames_files` 引外部檔） |
| `transdefs` | `[["name", gate, c1, c2, [bb], [geom], weak], ...]` — 電晶體（gate/c1/c2 可以是 id 或名字字串） |
| `segdefs` | `[[node, "+"/"-", layer, x1,y1,...], ...]` — 矽片 polygon + pull marker |
| `modules` | `[["prefix", "type", 0], ...]` — 子模組實例 |
| `connections` | `[["from_expr", "to_expr"], ...]` — 把節點接在一起 |
| `pullups` | `["name", ...]` — 額外標記為 pull-up 的節點（除了 segdef `+` 之外） |
| `forceCompute` | `["expr", ...]` — 標記為 `node_forcecompute` 的節點（同時連 gnd+pwr 時抵消，給 bus 用） |
| `memory` | `{ "name": size, ... }` — 宣告一塊行為級記憶體（由 handler 讀寫，不建電晶體） |
| `nodenames_files` / `transdefs_files` / `segdefs_files` | 引用外部 `.js`（給大型晶片用，例如 2A03 的 transdefs 一萬多行就放外部檔） |

### 範例：2A03（晶片 wrapper）

```javascript
var module = {
  name: "2A03",
  description: "NES custom CPU",
  pins : [ [1,'snd1'], [2,'snd2'], [3,'res'], [4,'ab0'], ..., [29,'clk_in'], [28,'db0'], ..., [40,'vcc'] ],
  nodenames_files: ["2a03/nodenames.js"],
  transdefs_files: ["2a03/transdefs_named.js"],
  segdefs_files:   ["2a03/segdefs.js"],
};
```

→ 真正的網表在 `data/system-def/2a03/` 目錄下：`nodenames.js`、`segdefs.js`、`transdefs_named.js`（用了命名節點的版本，比 `transdefs.js` 可讀）、還有 `transdefs2.js` 等變體。2C02 同理，多了 `nodenames_palram.js` / `nodenames_oamram.js`（palette RAM 和 OAM 的節點），並有 `forceCompute: ["spr_d[]"]`（sprite 資料 bus）。

### 範例：74HC04（標準反相器，全建成電晶體！）

```javascript
var module = {
  name: "74HC04", description: "Simple logic inverter",
  pins : [ [1,'1A'], [2,'1Y'], [3,'2A'], ..., [14,'vcc'] ],
  nodenames : { vcc:1, vss:2, '1A':11, ..., '1Y':21, ..., '6Y':26 },
  transdefs: [
    ['t', '1A', '1Y', 2],   // [name, gate='1A', c1='1Y', c2=2(vss)]  → 1A 高時把 1Y 拉到 vss
    ['t', '2A', '2Y', 2], ['t', '3A', '3Y', 2], ['t', '4A', '4Y', 2], ['t', '5A', '5Y', 2], ['t', '6A', '6Y', 2],
  ],
  segdefs: [ ['1Y','+',0], ['2Y','+',0], ['3Y','+',0], ['4Y','+',0], ['5Y','+',0], ['6Y','+',0] ],  // 每個輸出有 pull-up
};
```

**這就是經典 NMOS 反相器**：輸入高 → 電晶體導通 → 輸出被拉到 GND（=0）；輸入低 → 電晶體關 → pull-up 把輸出回到 high（=1）。`74HC04` 其實是 CMOS，但 MetalNES 用 NMOS 模型近似它的數位行為 —— 這完全 OK，因為**整個引擎只認識電晶體**，所以 support chip 也只能用電晶體描述。`74LS139`（解碼器）、`74LS373`（latch）、`74LS368`（三態 buffer）也都是這樣手刻成電晶體網路。

### 範例：SRAM2K（記憶體用宣告，不用電晶體）

```javascript
var module = {
  name: "SRAM2K", ...
  memory : { "ram": 2048 },     // 宣告一塊 2KB 記憶體，名字 "ram"
  // + pins (a[10:0], d[7:0], cs, /oe, /we) + 少量電晶體（位址解碼/三態控制邏輯）
};
```

→ 2KB SRAM 若真的用電晶體模擬（6T cell × 2048 × 8 ≈ 十萬顆電晶體）會很浪費，所以 MetalNES 把它當「行為級小黑盒」：宣告 `memory:{}`，由 `handler_ram` watch 它的控制腳（見 03 號文件）做正常 array 讀寫並驅動資料腳。**這是「transistor-level for 自訂晶片，behavioral for 通用記憶體」的混合策略**，跟我們設計文件 §12.1 的 bus 整合思路一致。

→ 這份 `nes-001.js` 基本上就是「在資料裡畫出 NES-001 的電路圖」。`cart-mmu0.js` 再往下一層描述卡匣內部（IRQ 接 vcc 停用、VRAM 鏡像方式）；`cart-mmu0-prgrom.js` / `cart-mmu0-chrrom.js` / `cart-extraram.js` 各自宣告 `memory:{}` 並接到卡匣邊緣。實際 PRG/CHR 資料由 `system.cpp::setupRom()` 從 `.nes` 檔灌進去。

### 2.5 晶片模擬分類彙整（S1 實際載入）

整理「誰跑電晶體網表、誰是軟體行為模型、誰只是 stub」，避免被晶片名稱誤導。 電晶體數為 2026-05-30 實際清點 `transdefs` 得到:

| 元件 | 定義檔 | 模擬方式 | 電晶體數 | 說明 |
|---|---|---|---|---|
| **2A03**（CPU,改版 6502 + APU） | `2a03.js` (+ 外部 transdefs) | ✅ **電晶體網表** | **10,946** | 完整,主角 |
| **2C02**（PPU） | `2c02.js` (+ 外部 transdefs) | ✅ **電晶體網表** | **16,794** | 完整 |
| **74LS373**（u2,位址 latch） | `74LS373.js` | ✅ **電晶體網表** | 82 | 8-bit latch,TTL 手刻成 NMOS |
| **74LS139**（u3,位址解碼） | `74LS139.js` | ✅ **電晶體網表** | 38 | `$2xxx→PPU` / `$0xxx→RAM` / `/ROMSEL` |
| **74LS368**（u7/u8,手把三態 buffer） | `74LS368.js` | ✅ **電晶體網表** | 14 | 6 三態 buffer,~2 顆/ch |
| **74HC04**（u9,反相器 PPU A13） | `74HC04.js` | ✅ **電晶體網表** | 6 | 6 個 NMOS inverter,完整 |
| **手把 ×2**（port0/1） | `nes-pad.js` → `4021.js` → `pslatch.js` ×8 | ✅ **電晶體網表** | ~176/個 | `nes-pad`(8) + `4021`(wrapper,自己 0 顆,內含 8×`pslatch`,每個 21 顆 = 168)。 4021 是 composition wrapper 不是 stub |
| **CIC**（u10,nes-cic1） | `nes-cic1.js` | ⚠️ **stub,非真實鎖區晶片** | **1** | **只有 1 顆電晶體 + 1 個 pull-up** ── 把 `cic_rst_in` 反相成 `/cic_rst1`(reset inverter)。 真正 CIC 的 4-bit MCU + seed 比對鎖區邏輯**完全沒模擬**(NROM 不需要鎖區檢查)。 名字叫「security chip」但實作上只是個 reset 反相殘樁 |
| **工作 RAM / VRAM**（u1/u4,各 2KB） | `SRAM2K.js` | 🔧 **hybrid**:控制腳網表 + 陣列行為 | 10(控制腳) | cs/we 解碼 glue 是真電晶體;**cell 陣列**由 `memory:{func<ram>:3}` + `AttachMemoryHandlers` 行為模型 |
| **extra RAM 8KB**($6000) | `cart-extraram`→`SRAM8K.js` | 🔧 **hybrid**(同上) | 18(控制腳) | blargg 測試工作 RAM(`--extra-ram`) |
| **PRG-ROM / CHR-ROM** | `ROM32K.js`/`ROM8K.js` | 🔧 **hybrid**(同上) | 10(控制腳) | `memory:{func<rom>:3}`;陣列行為,控制腳網表 |
| **主時鐘** | `clk` 節點 | 🔧 **純軟體** | — | `AttachClockHandler` 每 half-cycle toggle |
| **影像輸出** | `ppu.pclk1` | 🔧 **純軟體** | — | `AttachVideoHandler`(螢幕輸出,非 NES 電路) |

> 驗證紀錄(2026-05-30):逐檔清點 `transdefs` 確認 **CIC 是唯一的 stub**(1 顆)。 其餘宣稱「網表」的晶片電晶體數都與其功能相符(74HC04=6 個 inverter、74LS373=82 顆 latch 等),沒有第二個誤標。 兩處原表格不夠精確之處已修正:(1) 手把鏈正確是 `nes-pad → 4021 → pslatch×8`(原本漏寫 4021 中間層);(2) 記憶體晶片並非 0 電晶體,而是「控制腳網表 + cell 陣列行為」的 hybrid。

**一句話**:所有 NES 內部邏輯晶片(2A03/2C02/4 顆 74xx/手把)100% 跑電晶體網表;記憶體 cell 陣列、主振盪器、螢幕輸出用軟體行為模型;**CIC 是只做 reset 反相的最小 stub,不是真鎖區晶片**(2026-05-30 查證更正,先前曾誤列為「完整 CIC 網表」)。

## 3. 組裝引擎：`Wires::addInstance` / `addConnection`

### 3.1 節點 id 空間配置

`wire_node_resolver` 管理一個全域 node id 空間。每次 `addInstance` 一個子模組，就用 `allocNodes(alignment, maxNodeID+1)` 切出一段連續 id（對齊），然後把該模組所有「本地 id」加上 offset，重新註冊全名（`prefix.name`）→ 全域 id。

→ 所以最後整台 NES 是**一個扁平的 node 陣列**，每個 node 有個全名（`cpu.a0`、`u3.1/Y1`、`port0.shift.L0.q`…）。`func<clock>` / `func<rom>` / `func<ram>` / `func<video_out>` / `func<audio_out>` 這種特殊節點名是給 handler 用 wildcard（`*func<rom>`）去找的「掛鉤點」。

### 3.2 connection = 一顆永遠導通的電晶體

```cpp
bool Wires::addConnection(wire_instance_ptr instance, nodeID from, nodeID to) {
    if (from == to) return true;
    if (is_pwr_gnd(from)) std::swap(from, to);
    std::string name = getNodeName(from) + "<>" + getNodeName(to);
    addTransistor(instance, name, npwr /* gate = VCC = 永遠 on */, from, to);  // ★
    return true;
}
```

**把兩個節點接在一起 = 加一顆 gate 接 VCC（永遠導通）的電晶體。** 這樣 group 演算法自然就會把它們 union 進同一個 group。優雅地重用了現有機制 —— 不需要「節點合併」這種破壞性操作。

### 3.3 register 與陣列語法

`wire_node_resolver::resolveNodes(expr)` 支援幾種展開語法：

- `"a[7:0]"` → `[a7, a6, a5, a4, a3, a2, a1, a0]`（注意是大端，`[7:0]` 從高到低）
- `"a[]"` → `[a0, a1, a2, ...]` 直到查不到為止
- `"x|y|z"` → `[x, y, z]`（用 `|` 串多個）
- `"*func<rom>"` → 所有名字符合 `*func<rom>` 的節點（wildcard）
- 查不到的元素 → 預設用 `ngnd` 補（這樣 register 寬度不對也不會 crash）

`wire_register` 包裝一組 node id，提供 `read()`（把這些節點當作一個整數讀出來）/ `write(v)`（把整數每個 bit 設到對應節點）。`handler_nes_system` 就用 `wire_register_field(_cpu_a, "cpu.a[7:0]")` 這種方式把「CPU 累加器」當一個 8-bit register 讀。

## 4. 對我們的啟示

1. **「一切都是電晶體 + 線網」是個好的統一抽象**。自訂晶片、TTL 邏輯、connection 全用同一套資料結構，引擎只要實作一次。我們的 C# parser/IR 也應該保持這個統一性。
2. **大型晶片的網表放外部檔**（`transdefs_files: [...]`），module wrapper 只放 pins + 引用。我們的專案也該這樣分離 — pin 對映、模組組裝是「設計」，網表是「資料」。
3. **記憶體不要用電晶體模擬**。`memory:{}` 宣告 + handler 行為級讀寫。我們設計文件 §12 已經這麼說，MetalNES 確認了這是對的。
4. **connection 用永遠導通的電晶體實作**，避免破壞性的節點合併 —— 這個小技巧值得抄。
5. **`func<...>` / wildcard 掛鉤點**：模組裡留特殊命名節點，讓外部 handler 用 wildcard 找到並掛上行為。這是「網表（資料）」與「外部行為（程式碼）」之間乾淨的介面。
6. **`forceCompute` 處理 bus 爭用的特例**（同時連 gnd+pwr 時抵消）—— 提醒我們真實電路會有「兩個 driver 打架」的狀況，求值規則要能處理（哪怕只是個 hack）。
