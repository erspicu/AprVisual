### 1. γ.1 的「兩步代換」對嗎？

**答：對，針對 Size-2 是對的，這是基於「雙穩態電路（Bistable Circuit）的定點迭代（Fixpoint Iteration）」原理。**

*   **為什麼 size-2 代換 2 步會收斂？** 在數位邏輯抽象下，一個 size-2 的 cross-coupled pair 如果是一個合法的 Latch（無論是 static 還是 dynamic），其行為就是一個定點方程式 $a = f(b, \text{prev}), b = g(a, \text{prev})$。將 $b$ 代入 $a$ 得到 $a' = f(g(a, \text{prev}), \text{prev})$。如果這是一個雙穩態 Latch，這個代數表達式化簡後一定只會依賴 `Prev(a)`、`Prev(b)` 與外部訊號。如果代入兩步後，表達式**仍然**強烈依賴 `Node(a)` 或 `Node(b)`（且無法用布林代數消去），說明這不是 Latch，而可能是震盪器（Oscillator）或存在 race condition —— 這時候你反而**不該**把它提出 SCC。
*   **Size-3 該不該做？** **不建議直接做**。Size-3 通常不是簡單的 Latch，而是包含邏輯延遲的迴路（例如 Ring Oscillator 或者是某種透過三個 node 傳遞的 Glitch-filter）。對 size-3 盲目做 3 步代換風險較高，很容易抽錯邏輯。
*   **一般化的閉式解（BDD/SAT）？** 你的直覺很精準。理論上，任何 size-$k$ 的 SCC 都可以看作一個具有 $k$ 個內部狀態的 FSM。對於 $k \le 4$ 的微小 SCC，用 ROBDD（Reduced Ordered BDD）建立轉移關係，並計算 Fixed-point 是**最穩健且數學上絕對正確**的做法。但這方法**絕對無法擴展到 703**（狀態空間爆炸 $2^{703}$）。
*   **結論：** γ.1 就照你的提議，實作一個「代數代換 + 布林化簡器」，限定只跑 $k=2$。這能以極低的實作成本殺掉 ~4400 個節點，投資報酬率極高。

### 2. γ.2 的 Clock-domain 分類器與大 SCC 切割

**答：千萬不要去「Pattern-match」計數器或 LFSR，那會讓你陷入泥淖。你只需要找到「Register Boundary（暫存器邊界）」。**

*   **怎麼認 Clock-like node？** 你**不需要**知道它是分頻器還是主頻。在 Switch-level 轉 Gate-level 的過程中，所謂的「Clock」只是一個**「用來控制 Pass-transistor（傳輸閘）的控制端訊號」**。
*   **切割策略（Stage B：Pass-transistor Cut）：**
    1. 掃描大 SCC 內的所有節點。
    2. 尋找被 Pass-transistor 驅動的節點（NES 的 NMOS 邏輯中，Latch 的輸入幾乎 100% 來自 Pass-transistor）。
    3. 將這個 Pass-transistor 的「資料輸入端」或「輸出端」切斷，標記為 `Prev()`。
    4. 重抽 `NextExpr`：`NextExpr[latch] = Mux(pass_gate_control, pass_gate_data, Prev(latch))`。
*   **為什麼不用管 703 是什麼 Counter？** 一旦你把 703-node SCC 裡面所有的 Latch 輸入端都正確切斷並加上 `Prev()`，這個 703 的巨大環就會**瞬間瓦解（Shatter）**成一堆有向無環圖（DAG）。那些 Counter 的加法器、Carry chain、LFSR 的 XOR，本質上只是夾在 `Prev(latch_A)` 和 `NextExpr[latch_B]` 中間的一大坨「純組合邏輯（Combinational Logic）」。你**完全不需要**手寫 IR 去描述「這是一個 Counter」，你的邏輯綜合器會自然把這些 NAND/NOR/XOR 展開成一個巨大的、但完全 acyclic 的布林表達式！這才是硬體驗證管線的威力。

### 3. 優先序與範圍

**答：先 γ.1 (Commit) → 處理同義節點 (Q5) → 再 γ.2。**

1.  **γ.1 是 Low-hanging fruit**：程式碼簡單，影響範圍明確（只動 size-2）。做完立刻 commit，這會讓你的 coverage 從 46.3% 跳升到 70%+，士氣大振，也能讓剩下的圖變乾淨。
2.  **大 SCC 絕對值得花力氣**：703-node 的 DMC 不只是為了跑某些特定的遊戲，而是**為了能上 GPU**。只要有一個 node 留在 SCC 裡，GPU 就必須在那裡跑 while-loop 直到收斂，這對 GPU 架構（SIMT）是毀滅性的。

### 4. 「夠」的判準與 GPU Fallback

**答：對於 GPU Bit-slicing 而言，及格線是「Sequential Node 100% 被抽離為 Prev() 邊界，Combo Node 100% 變為 DAG」。**

*   **剩 500-1000 個能不能豁免？** **不行**。在 GPU 裡，如果你有 64 個 NES 模擬器在同一個 Warp 裡 bit-slicing（每個 thread 負責一個 bit），只要其中一個模擬器的某個 macro-cell 進入 SCC oscillation 或需要 iteratively evaluate，**整個 Warp 的 64 個模擬器都要陪著它等**（Warp divergence）。
*   **GPU 有無 Fallback？** 實務上很難。如果你讓 GPU 算一半，把那 500 個難搞的 node 拷貝回 CPU 算，算完再拷貝回 GPU，PCIe/Memory 傳輸延遲會把你的效能吃光。
*   **最終手段：** 如果最後真的剩幾十個 unnamed tangle 抽不出 Latch 模型，你可以對它們實作一個「Micro-iteration block」：在 GPU/CPU codegen 時，將這幾個 node 包在一個 `for (int i=0; i<3; ++i) { ... }` 的微型迭代迴圈裡強制收斂（假設它們只是某種複雜的 combo 延遲），但這非常破壞美感，應極力避免。

### 5. 漏掉的：前綴 / 同一節點的多版本

**答：這是神來之筆（Critical Insight）。這叫「Node Aliasing / Buffer Removal」，必須在更早的階段做！**

*   在 NMOS/CMOS 網表中，因為 fan-out 限制，往往會有一大堆 Buffer 節點（`pcm_t5`、`pcm_+t5`）或 Inverter 節點（`pcm_/t5`）。
*   **強烈建議在「建立 NextExpr / 拓撲排序前」新增一個 Lowering Pass**：
    *   找出所有單純的 Buffer（`NextExpr[a] = Node(b)`），將圖中所有對 `a` 的引用全部替換為 `b`。
    *   找出所有單純的 Inverter（`NextExpr[a] = Not(Node(b))`），將引用替換為 `!b`。
*   **為什麼這很重要？** 這會大幅減少圖的節點數，而且很多看似 size-4、size-6 的 SCC，在經過 Aliasing 之後，會現出原形，變成標準的 size-2 cross-coupled pair！這能讓你的 γ.1 抓到更多東西。

### 6. 關於最終 Codegen 後端 (S3 之後的方向感)

**答：強烈推薦 (c) LLVM via .NET。對你的使用情境，(a) 和 (b) 都有嚴重的工程與效能瓶頸。**

*   **為何不選 (a) Roslyn 字串：**
    35,000 行 flat block 加上數萬個區域變數，會觸發 Roslyn 和 .NET JIT 的惡夢。Roslyn 的 Parser 會很慢，而 .NET 的 RyuJIT 主要是為「小而美的物件導向方法」最佳化的，當它遇到有三萬個區域變數的方法時，Register Allocation（暫存器分配）算法會耗費極長時間，甚至直接 fallback 到極差的 unoptimized 機器碼。
*   **為何不選 (b) Reflection.Emit IL：**
    雖然跳過了 Roslyn parsing 變快了，但最終還是要過 RyuJIT，上述的巨大 Block 暫存器分配問題依舊存在。
*   **為何 (c) LLVM 是最佳解：**
    硬體模擬界（例如 Verilator）產生巨量 flat SSA 程式碼是家常便飯。LLVM 的架構就是為了處理這種巨型 IR 而生的。
    1. **最佳化能力：** LLVM 的 `-O3` 能夠對這 35,000 行進行跨越全域的 Constant Folding、Dead Code Elimination 和 Instruction Combining。這會把你的網路圖進一步壓縮。
    2. **Register Allocation：** LLVM 的 greedy register allocator 處理巨型基本區塊（Basic Block）的表現遠超 RyuJIT。
    3. **GPU Retargeting：** 這是一擊必殺的理由。同一份 LLVM IR，透過 .NET LLVM binding，可以一鍵 Retarget 到 CPU (x86/ARM) 或者是 NVPTX (CUDA)。如果你用 C# emit，你未來要上 GPU 就必須寫另外一套 emit 字串去生成 CUDA C / PTX 碼，維護兩套 Codegen 後端會讓你發瘋。用 LLVM，你只需維護一套 `IRBuilder` 的呼叫邏輯。
