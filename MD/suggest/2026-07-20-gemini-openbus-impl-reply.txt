Here is the concrete, implementation-oriented guide to replacing your hand-tuned open-bus behavior with a principled, physically-derived model in your switch-level NES engine.

### 1. THE RIGHT MODEL: Lazy-Evaluated Timestamps
In a discrete 0/1 event-driven engine where `hc` is 23.28 ns and the decay is ~600 ms (~25 million `hc`), **timestamp + threshold (lazy evaluation)** is absolutely the correct approach. 

*   **Why not an Event Wheel?** Scheduling an event 25,000,000 cycles in the future requires expanding your event queue depth massively for a phenomenon that rarely matters. If a bus is driven again 2 cycles later, you have to find and cancel that future event. Heavy and unnecessary.
*   **Why not a Leaky Integrator / Charge Counter?** Adding continuous decrement math to a floating node during the hot inner loop wastes CPU cycles.
*   **The Lazy Timestamp Model:** When the network settles, if a bus node has a continuous driven path to VDD/GND, record `last_driven_hc = current_hc` and `last_val = state`. When a node resolves to *floating* (no driven path), its state is `1` only if `last_val == 1` AND `(current_hc - last_driven_hc) < decay_threshold_hc`.
*   **Failure Modes of Lazy Eval:** If a floating line is tied to an *asynchronous* edge-sensitive input, lazy evaluation might miss the exact `hc` the 1->0 transition occurs, because the decay is only evaluated when the node is "looked at" (computed). **Mitigation:** The NES data bus is synchronously sampled by $\phi_2$ / read-enable signals. By evaluating the decay condition inside your `wire_compute` specifically when resolving floating states, it perfectly mimics the real timing without scheduling future events.

### 2. THE DECAY LAW + COMPUTING THE CONSTANT
**The Physics:** A floating external bus node is dominated by **reverse-biased PN junction leakage** to the p-type substrate (which is tied to GND). Subthreshold leakage of the IO drive transistors also exists, but also leaks to GND. 

Because reverse-bias leakage current ($I_{leak}$) is roughly independent of the reverse voltage (until avalanche), it acts as a constant-current sink. Therefore, the decay is **Linear**, not an exponential RC curve. 
$$ \frac{dV}{dt} = \frac{-I_{leak}}{C_{bus}} $$
$$ V(t) = V_{OH} - \left( \frac{I_{leak}}{C_{bus}} \right) t $$

**The Flip:** The switch-level model transitions from 1 to 0 when $V(t)$ crosses the NMOS input logic-low threshold ($V_{IL}$).

**The Derivation:**
1.  **$V_{OH}$ (Output High):** NMOS depletion load typically pulls an unloaded bus to $\approx 4.5\text{V}$ to $5.0\text{V}$.
2.  **$V_{IL}$ (Input Low Threshold):** TTL-compatible NMOS inputs recognize a 0 at roughly $\approx 1.5\text{V}$.
3.  $\Delta V = V_{OH} - V_{IL} \approx 3.0\text{V}$ to $3.5\text{V}$.
4.  $t_{decay} = \frac{C_{bus} \cdot \Delta V}{I_{leak}}$
5.  **Convert to `hc`:**
$$ \text{Threshold}_{hc} = \frac{C_{bus} \cdot \Delta V}{I_{leak} \cdot t_{hc}} $$
*(Where $t_{hc} = 23.28 \times 10^{-9}$ seconds).*

### 3. PER-BIT vs WHOLE-BYTE
The 8 data lines decay **independently**. They do not decay in unison. 
*   **Physical reality:** Each data trace on the NES motherboard has a slightly different length, via count, and neighboring traces. More importantly, manufacturing variance in the PN junction area of the IO transistors means $I_{leak}$ varies bit-by-bit.
*   **Test-accuracy:** Real NES open-bus tests observe that a holding 0xFF byte rots bit-by-bit into a mixture (e.g., 0xEF -> 0xEE -> 0x40 -> 0x00) over a window of a few tens of milliseconds. 
*   **The Spread:** The per-line spread is roughly $\pm 10\%$ to $\pm 20\%$ of the base threshold. 
*   **Representation:** Your model *must* use per-bit timestamps and per-bit thresholds. Whole-byte is not test-accurate if someone writes a ROM that spins checking for the exact order in which the bits drop.

### 4. REFRESH SEMANTICS
A bus node's timestamp is refreshed (reset to `current_hc`) **if and only if there is an active path to VDD or GND during the settled half-cycle.**
*   CPU Write? Refreshes.
*   Mapped ROM/RAM/PPU Read? Refreshes.
*   CPU reading unmapped space (Open Bus)? **Does NOT refresh.** 
    * *Nuance:* When the CPU reads open bus, the read-buffer pass transistors open, momentarily sharing charge between the massive external $C_{bus}$ and the tiny internal CPU bus capacitance. Physically, this drops the voltage by a microvolt. It does *not* recharge it. In a discrete model, ignore it: an open-bus read simply samples the floating state.

### 5. THE THRESHOLD / FLIP
*   **Direction:** The line strictly decays **1 $\rightarrow$ 0**. Leakage is to the substrate (GND). 0 $\rightarrow$ 1 does not happen on the NES data bus because there are no pull-up resistors and no significant leakage path to VDD.
*   **Grid Placement:** The flip occurs instantly on the `hc` where `current_hc - last_driven_hc > threshold_hc`.

### 6. IMPLEMENTATION IN THE ENGINE
Keep this out of the generalized transistor solver to save CPU time. Handle it at the bus-latch boundary or as a post-pass on the 8 external data bus nodes.

**Data Structure:**
```c
struct OpenBusNode {
    uint64_t last_driven_hc;  // Timestamp of last strong drive
    uint32_t threshold_hc;    // Computed limit (e.g., ~25,000,000 + variance)
    uint8_t  last_val;        // 0 or 1
};
OpenBusNode ext_bus[8];
```

**Where to put it:**
Hook into the end of `wire_compute` (after the switch-level matrix has settled for the half-cycle), specifically intercepting the 8 external DB pins.

```c
// Post-settle step (only runs for the 8 external DB nodes)
for (int i = 0; i < 8; i++) {
    Node* n = &netlist.nodes[DB0_NODE + i];
    
    if (n->is_driven) {
        // Node has path to VDD/GND. Refresh it.
        ext_bus[i].last_driven_hc = current_hc;
        ext_bus[i].last_val = n->state;
    } else {
        // Node is floating. Evaluate decay.
        if (ext_bus[i].last_val == 1) {
            uint64_t age = current_hc - ext_bus[i].last_driven_hc;
            if (age > ext_bus[i].threshold_hc) {
                n->state = 0; // The flip!
            } else {
                n->state = 1; // Still holding charge
            }
        } else {
            n->state = 0; // Was driven to 0, stays 0
        }
    }
}
```
**Interaction Hazards:**
1.  **PPU IO-Latch (2C02 internal):** The PPU has its own internal open bus (e.g., reading status $2002 bits 0-4 returns the last value on the *PPU's internal data bus*). Make sure your external `ext_bus` array maps *strictly* to the motherboard traces connecting the dies, distinct from the internal PPU capacitance.
2.  **Test-mode Toggle:** Wrap the `age > ext_bus[i].threshold_hc` block in an `if (enable_decay_model)` to keep your golden unit-tests perfectly deterministic when comparing against short-running visual6502 traces.

### 7. FROM TUNED TO DERIVED (The Follow-up Plan)
How to replace your magic `600ms` number with a derived constant:

**1. The Parameters to Estimate (The Inputs):**
*   **$C_{bus}$:** Sum up: DIP40 pin ($\sim 4$pF) + PCB trace ($1.5$pF/inch $\times \sim 4$ inches) + Cart edge ($\sim 2$pF) + ROM input ($\sim 3$pF). Total is roughly $15$ to $25$ pF per line.
*   **$I_{leak}$:** For early 80s $3\mu m$ NMOS at room temperature ($25^\circ$C), reverse-biased diode leakage is roughly $10$ to $50$ pA per pin. Since there are multiple pins on the bus (CPU, PPU, WRAM), assume total leakage of $50$ to $100$ pA.
*   **$\Delta V$:** $\approx 3.5$V.

**2. The Bench Measurement:**
To calibrate the variance, run this on real hardware:
*   Write `0xFF` to a floating address.
*   Wait $N$ cycles.
*   Read the address, write result to zero-page (or output to controller port for a scope).
*   Sweep $N$ from $100$ms to $1000$ms.
*   *Result:* You will see a waterfall. Bit 2 might drop at $510$ms, Bit 7 at $580$ms. This gives you the exact per-bit thresholds to seed your `threshold_hc` array variance.

**3. The Honest Floor (What is fundamentally un-derivable):**
You can never compute a single *universal* absolute threshold from first principles, because **$I_{leak}$ is exponentially dependent on temperature.**
$$ I_{leak} \propto T^3 \exp\left(\frac{-E_g}{kT}\right) $$
A real NES in a $30^\circ$C room will decay open-bus values almost **twice as fast** as one in a $20^\circ$C room. 

**The clean engineering solution:** Calculate the base `threshold_hc` equation (as given in Section 2) using $C_{bus} = 20\text{pF}$ and $I_{leak} = 80\text{pA}$. This yields $\sim 600$ms. Hardcode this derivation in your code as documentation, but allow a simulation parameter `ambient_temp_variance` to scale $I_{leak}$, and apply a fixed $\pm 10\%$ PRNG spread to the 8 bits on emulator initialization to replicate the bench waterfall. This gives you a physically rigorous, computed model that embraces the analog reality of the hardware.
