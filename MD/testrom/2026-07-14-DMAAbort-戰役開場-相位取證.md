# Explicit/Implicit DMA Abort 戰役:開場取證(相位問題)

日期:2026-07-14 · 對象:`$479` ExplicitDMAAbort / `$478` ImplicitDMAAbort(皆 run4 `$0A` = FAIL err2)
狀態:**取證中 —— 微架構逐拍正常,分歧縮到 ±10 週期的細相位;下一步 AprNes 事件時間線比對**

## 1. 測試機制(全破解)

- err2 = **答案鍵比對**:16 個 X(0-15)掃相位,量測「DMA 中止編排後下一次 DMA 的相位」寫入 `$50-$5F`,
  與硬體答案鍵 `04 04 04 04 04 04 03 04 01 01 00 00 00 00 00 00` 逐位比對。
- **量測 = open-bus DMA 碰撞偵測器**(`CalculateDMADuration`):迴圈 `LDA $4000`(open bus,平常讀
  `$40`=自己的運算元高位),每圈滑 560 週期讓讀取點相對 DMA(576 週期/位元組)每圈提前;撞上 DMA
  那圈讀到 **DMC 樣本位元組 `$00`** → BEQ 跳出,Y=圈數=相位。
- X 掃相位:`Clockslide64_Minus_A`,X 越大 disable(`STA $4015`=0)越早,掃過目標 DMA 的 4 週期窗。
- 速率:`$4E` = rate idx $E = **72 週期/位元 → 576 週期/位元組**(idx $F 才是 54/432)。

## 2. S1 量測結果與硬體差異

S1 全 X 讀 **0**;硬體 `4,4,4,4,4,4,3,4,1,1,0,0,0,0,0,0`。
**X≥10(完全中止段)S1 與硬體一致**;分歧只在 X=0-9(未中止/部分中止段)。

## 3. 微架構顯微鏡(rdy + pcm 節點,全部合理)

工具:rdy 停機時間線 + pcm raw-id 顯微鏡(`cpu.#14059`=pcm_dma_active、`#11094`=loadbuf、
`#11093`=loadsr、`#11102`=shiftsr、具名 `pcm_en`/`pcm_lc[11:0]`)。X=0 迭代逐拍:

```
15874307 loadsr(SR 取走 buffer → buffer 空)
15874379 dma+loadbuf(目標 DMA 重抓,buffer 滿)     ← stall 84t ≈ 3.5 cyc
15874559 en 1→0(disable —— 晚了 ~7.5 cyc,X=0 本來就該 miss ✓)
  en=0 期間:shiftsr 照每 1728t(72 cyc)走 ✓(硬體輸出單元也不停)
15888131 loadsr(SR 又取走 buffer —— en=0 也照搬 ✓ 硬體同)→ buffer 空
15888263 en 0→1(re-enable)
15888443 dma+loadbuf(立即重抓 —— buffer 空,合法 ✓)
```

**每一步單獨看都符合硬體語意。** 分歧在整體相位:S1 的 re-enable 後重抓 = 目標 DMA + **586** 週期
(自然槽位 576 + ~10 抖動);答案鍵 X=0-9 段吃的就是這 ±10 週期。

CheckDMATiming(Y=4 閘)通過 → 基本 load-DMA 時序與硬體一致;DMASync 碰撞輪詢正常
(stall @ab=$4000 可直接當同步標記)。

## 5. Dmc4015AbortShim 迭代史(v1-v4,2026-07-14)

**已實作**:`EnableDmc4015AbortShim`(WireCore.System.cs;TestRunner 預設**註解關閉**)——
`pcm_en` 下降沿 → 倒數 → 屆時若停機中(rdy=0)且 fetch 未見(`#14059` 未脈衝)→
`InstClampLow(cpu.#14039)`(rdy 下拉 gate;rdy 靠 4 顆 pullup 自然回高)。
**RESULT-WRITE hook**($478/$479 寫入瞬間傾印 ZP $50-$5F)= 16 格量測直讀,迭代迴圈 3 分鐘。

| 版本 | 參數 | 16 格結果(vs 鍵 `…03 04 01 01 00…`)| 判讀 |
|---|---|---|---|
| v1 | D=7 ticks(單位搞錯:**Time=主時鐘半週期,24/CPU 週期**)| 全 no-kill(檢查點在寫後 0.3c)| 單位教訓 |
| v1b | D=84(3.5c)hold=10 | X=7,8,9 殺 → `…03 03 03 03 00…` | X=7 誤殺;殺太晚只省 1c(03 非 01)|
| v1c | D=36(2c,AprNes X=9 實測 write+2)hold=10 | X=6,7 殺 → `…02 03 04 04 00…` | **S1 的寫入會推遲停機**(halt 等讀週期)→ 固定 D 無法分辨 X |
| v4 | D=36 + **邊界窗**(kill 僅當 statusOff − 上次 `pcm_loadsr` 邊界 < 120t)hold=96 | **測試掛死**(60 幀無判定)| 鉗 96t 卡死 pcm sequencer(疑似反覆重 halt)|

**關鍵發現**:硬體的 kill 判準錨在**位元組邊界**(DMA 自然槽位),與寫入位置無關;
S1 的停機會被寫入推遲,不能拿停機當錨。真正乾淨的取消點是**請求 latch**(#14039 的上游)——
一次性雙邊翻轉(仿 DL shim),而非長鉗 halt gate。

## 6. 破關:v5/v6(2026-07-14)

**v5(retire-early)**:不再對抗 latch —— 拆 fan-in 發現 `#14039`(halt 儲存節點,自帶
pullup)/`#15737`(放電側,retire 信號 `#14063` 已拉低)/`#11483`(連接閘)構成動態
latch cell,而 **`#11466`(ACLK 相位)按著連接閘 = 停機時長的本體**。kill =
`InstClampLow(#11466)` 鉗住 72t(打開 retire 路,netlist 自己放電)再放 —— 相位節點
每 settle 被主動重驅,自癒。點狀翻 #14039/#15737 都會被 pullup/上游彈回(三連敗重演)。

**v6(邊界格點錨定)**:量測 = 邊界距離 71t→03、47t→01、23t→00 —— **kill 效果被 ACLK
量化**:硬體的中止一律落在 boundary+48t 格點。規則:statusOff(D=84)時 rdy=0 且
fetch 未見且 dist<60t → 排程 kill 於 `max(now, boundary+48t)`。

**結果:16/16 完美命中答案鍵**;`$479` 1/1 PASS、**`$478` Implicit 同 shim 直接翻綠
($09 = oracle 精確值,含 variant)**;golden checksum 不變、OpenBus/LAE 哨兵 PASS。
回歸窗(補記)。
