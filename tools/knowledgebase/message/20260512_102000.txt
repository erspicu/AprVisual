## ① 設計描述

### 1.1 目標 & 範疇

- 輸入：`WireCore` 在 `ComposeSystem`（含 `EnableLowering` → `LowerNetlist()`）之後的 build-time 表 —— `_nodes: List<Node?>`（dense、`1==Npwr`、`2==Ngnd`、`0` 保留）、`_transistors: List<Transistor>`（每筆 `{Gate, C1, C2, IsWeak, Name}`，`AddTransistor` 已把 supply 正規化到 `C2`）、`_nodeByName` / `_nameByNode`。
- 輸出：一個 `AprVisual.Sim.Logic.NetlistGraph`（靜態 class，跟 `WireCore` 同 assembly，讀 `WireCore.Nodes` / `WireCore.Transistors`），裡面：
  - `NodeRole[] role`（indexed by node id）
  - `TransistorKind[] kind`（indexed by transistor index）+ `bool[] isWeak`（直接抄 `Transistor.IsWeak`，方便）
  - `PassDir[] passDir`（indexed by transistor index；只有 `kind==Pass` 的有意義，初值 `Unknown`，S2.1 填）
  - 一個 `Build()` 把上面填好，加幾個統計查詢（`CountByRole()`、`CountByKind()`、`Describe(nodeId)`）
- **不做**：pull-down 表達式、pass 方向推導、任何 `Expr`、任何模擬行為改動。那是 S2.1+。

### 1.2 `NodeRole`

```
enum NodeRole { Supply, Input, Bus, Internal }
```
- **`Supply`** = `{ Npwr(1), Ngnd(2) }`。直接設。
- **`Input`** = 「從來不被網表本身驅動，只被外部碼（handler / `ResetNes`）`SetHigh`/`SetLow`」的 node。判定 heuristic：`role != Supply` 且 `node.C1c2s.Count == 0`（從來不是任何電晶體的 channel 端）且 `node.Pullups == 0`。→ 抓到 `clk`（clock handler 每 half-cycle toggle 它；它只當一堆電晶體的 gate，自己沒 channel）、可能還有幾個 config strap。
  - 註：`res` **不算** `Input` —— NES 的 `res` 線板上有 pull-up、CIC 也驅動它，S1.5 lowering 又把 `res`/`cpu.res`/`ppu.res` 那些 always-on connection 併成一個 super-node 了，所以它有 pull-up + 有 driver → `Internal`。我們在 `ResetNes` 裡 `SetHigh/SetLow(res)` 是「外部 override」，IR 引擎處理 override 跟 S1 一樣（直接蓋值傳播），不需要把它標 `Input`。
- **`Bus`** = 雙向（既被網表驅動、又被 handler 驅動）—— S2.0 先做**最小但確定對**的版本：把 `cpu.db[7:0]`（CPU 資料匯流排）跟 `ppu.io_db[7:0]`（PPU 對 CPU 的資料線）resolve 出來標 `Bus`（這兩條確定是雙向的：CPU 寫的時候 CPU 驅動、讀的時候 RAM/ROM/PPU handler 驅動）。其餘雙向 node（RAM/PPU 內部的資料線、`_d[7:0]` 那些 handler 寫的腳）**先不標** —— 等 S2.1 有了 `DriveInfo`（知道哪些 node「有 driver」）+ 從 handler 取得「被 `WriteBits` 的 node 集合」，再精準地把 `(有 driver) ∩ (被 handler 寫)` 標成 `Bus`。S2.0 階段 `Bus` 是「定義好的類別、先粗填」。
- **`Internal`** = 其餘全部。

### 1.3 `TransistorKind`

```
enum TransistorKind { PullDown, PullUpStrong, PullUpLoad, PullUpActive, Pass, Dead }
```
對每筆 `Transistor t`（已正規化：若有 supply 在 channel 端，它在 `C2`）：
1. `t.Gate == Ngnd`（gate 接 GND → 永遠 OFF）→ **`Dead`**。（S1.5 lowering 已丟掉這種，理論上不會出現；保留以防萬一。）
2. channel 端碰 GND（`t.C1 == Ngnd || t.C2 == Ngnd`）→ **`PullDown`**。（這是「直接下拉到地」的葉子；經由其他信號 node 串聯才到 GND 的，那些電晶體會是 `Pass`，串並聯結構由 S2.1 算。`gate == Npwr` 的「無條件接地」也歸 `PullDown`，S2.1 看到 `gate` 是 `Npwr` 就知道條件恆真。）
3. channel 端碰 VCC（`t.C1 == Npwr || t.C2 == Npwr`，設 `sig` = 非 VCC 的那一端）：
   - `t.Gate == Npwr`（無條件接 VCC）→ **`PullUpStrong`**。
   - `t.Gate == sig`（gate 接自己 source / 被拉的 node —— 經典 NMOS depletion load 的接法）→ **`PullUpLoad`**。
   - 否則（gate 是別的信號 node → conditional pull-up，superbuffer / push-pull 的上半 / precharge 的 clocked pull-up）→ **`PullUpActive`**。
4. 兩個 channel 端都是信號 node（非 supply）→ **`Pass`**（`passDir = Unknown`，S2.1 填）。
- `isWeak[i] = t.IsWeak`（depletion / weak 旗標）—— S2.1 分 `PullUpLoad` 是強是弱、pass 方向推導時會用。

> 為什麼這樣分（呼應 Gemini review §7-3、§7-4）：S2.1 抽 `Expr` 時要分得出「static load（恆拉高的負載）」「strong VCC tie」「conditional / clocked pull-up（precharge）」「active pull-up（superbuffer，要做 `Mux(pullup, 1, Mux(pulldown, 0, Hold))`）」。S2.0 把這個一級分類做好，S2.1 才有 base。

### 1.4 `PassDir`

```
enum PassDir { Unknown, AtoB, BtoA, Bidirectional }
```
- `Pass` 電晶體上一個欄位，S2.0 一律 `Unknown`。
- S2.1 的方向性推導（Gemini §7-2）會填：典型「`C1` 側有 PULLDOWN 網路、`C2` 側純 dynamic（只有 pass 連進來）→ `AtoB`」（A=有 driver 那端）。推不出單一方向（如 2C02 OAM 讀寫匯流排）→ 留 `Bidirectional`，那個 pass 網路 → hybrid（S2.1 標）。
- S2.0 只是把欄位開出來；不推導。

### 1.5 IR 跟 S1 共用 `NodeStates`（X / 初始化的前置決策）

呼應 `00_計畫總覽.md` §1.4 / Gemini §7-6：S2.0 這步順便**確認**（不需要改 code，只是文件化決策）：IR 引擎（S2.4）會跟 S1 共用那個 `byte* NodeStates`（0/1，不追 X）；power-on 的「未定」由「`ResetNes(full)` 先跑 S1 的 `Reset()` + `RecomputeAllNodes()` settle 到決定性穩態，IR 引擎再接手」這條路解掉。所以 S2.0 不需要為 X 做任何資料結構。

### 1.6 程式落點

- 新檔 `src/AprVisual/Sim/Logic/NetlistGraph.cs`：
  ```csharp
  namespace AprVisual.Sim.Logic
  {
      internal enum NodeRole { Supply, Input, Bus, Internal }
      internal enum TransistorKind { PullDown, PullUpStrong, PullUpLoad, PullUpActive, Pass, Dead }
      internal enum PassDir { Unknown, AtoB, BtoA, Bidirectional }

      internal static class NetlistGraph
      {
          public static NodeRole[] Role = [];           // [nodeId]
          public static TransistorKind[] Kind = [];     // [transistorIndex]
          public static bool[] IsWeak = [];             // [transistorIndex]
          public static PassDir[] PassDirection = [];   // [transistorIndex]
          public static bool Built;

          public static void Build() { /* fill from WireCore.Nodes / WireCore.Transistors */ }
          // stats / introspection
          public static (int supply,int input,int bus,int internal_) CountByRole();
          public static int[] CountByKind();             // length = enum size
          public static string Describe(int nodeId);     // "name#id role=… pulldownTransistors=… pullup=… pass=…"
      }
  }
  ```
- `TestRunner` 加 `--dump-graph`（或在 `--dump-system` 末段呼叫）：`ComposeSystem` → `NetlistGraph.Build()` → 印各 role 的 node 數、各 kind 的 transistor 數、幾個 spot-check（`vcc`/`vss`=Supply、`clk`=Input、`res`=Internal、`cpu.db0`=Bus、某個確定是 depletion load 的 transistor=`PullUpLoad`）。
- **不碰** `WireCore` 既有的東西（`Reset()` / `Step()` / handlers）—— S2.0 是純新增。`NetlistGraph.Build()` 由 `--dump-graph` / 之後的 S2 步驟顯式呼叫，正常 `LoadSystem` 流程不呼叫它（不影響模擬）。

這份 S2.0 設計文件非常務實。「先建表、不推導邏輯、不碰模擬」的切入點十分精準，能把複雜的 S2 拆解得很好。

以下針對你的 5 個問題進行具體的技術 Review：

### 1. `NodeRole` / `TransistorKind` 的分類與 NMOS 結構（回應 §1.8 開放問題 1, 2）

**結論：你的 1-Transistor 分類法非常正確。不要在 S2.0 試圖辨識 Superbuffer 或 Bootstrap，那會掉進陷阱。**

*   **為什麼 S2.0 的分類夠用了？**
    *   **Depletion load 接法**：你已經完美區分了 `Gate == source` (`PullUpLoad`) 和 `Gate == Npwr` (`PullUpStrong`)。這是最關鍵的基底。
    *   **Superbuffer / Push-Pull**：這在 NMOS 中是由「一顆 PullDown + 一顆 Active PullUp」組成。在 S2.0 看單顆電晶體時，它的上半部因為 Gate 接的是另一個信號（非 Npwr 也非自己），會被準確標為 `PullUpActive`；下半部會被標為 `PullDown`。在 S2.1 組合分析時，只要看到一個 node 同時具備這兩個 driver，就知道它是 push-pull。S2.0 做到這樣就滿分了。
    *   **Bootstrap**：在 Visual6502 網表中，bootstrap 電容通常沒有顯式元件，而是靠動態節點電容與 pass transistor 來模擬。對於驅動端，它看起來就像一個 `PullUpActive` 或 `Pass`。S2.0 只要如實標記即可，時序和電壓推升（overdrive）的問題交給 S2.1/S2.4 的狀態保留（Hold）機制處理。
    *   **Ratioed Inverter / Tri-state / 下拉鏈**：這些都需要看「網表拓撲（多顆電晶體）」，屬於 S2.1 串並聯分析的範疇。單顆電晶體層級，它們就是 `PullDown` 和 `Pass`。
*   **建議修改 / 確保**：
    *   你的 `TransistorKind` 邏輯已經無懈可擊，**不需增加新類別**。保持它作為「1-T（單顆電晶體）層級的絕對物理特徵分類」。

### 2. `Input` 的 Heuristic 與 `res` 判定（回應 §1.8 開放問題 1）

**結論：判斷標準正確，`res` 標為 `Internal` 是非常清醒的決定。**

*   **關於 `res`**：你的分析完全正確。因為板子上有 pull-up，CIC 也會拉它，所以它在網表內「有合法 driver」。我們在 C# 用 `SetHigh/Low` 寫它，本質上是「強制覆寫（Override）」，而不是它天生只是個輸入腳。保持它為 `Internal`，能確保 S2 抽出來的 CIC 和 pull-up 邏輯不會被誤刪。
*   **會不會漏抓？手把輸入（Controller D0）**：
    *   手把輸入腳（如 CPU 上的 `$4016` D0）在 NES 板子上通常接了 pull-up 電阻，且不被網表內部任何電晶體驅動（除非被當作 channel 端，但通常只接 gate 或直接進 buffer）。
    *   因為它有 pull-up，你的 Heuristic（`node.Pullups == 0`）會把它踢出 `Input`，變成 `Internal`。
    *   **這有關係嗎？沒關係。** 只要 IR 引擎（S2.4）允許 Handler 對 `Internal` node 進行 Override 寫入（就像 S1 的蓋值傳播），那它叫 `Input` 還是 `Internal` 根本不影響模擬結果。
*   **建議**：Heuristic 夠了。維持現狀。

### 3. `Bus` 的分階策略（回應 §1.8 開放問題 3）

**結論：先粗填 `cpu.db` / `ppu.io_db` 的分階策略非常好，強烈建議採用。**

*   **影響**：S2.0 的目標是「建立骨架並通過 CI 驗證」。如果現在去撈 `WireCore.Handlers` 裡被 `WriteBits` 的 node，你需要處理跨模組的資料依賴，會拖慢 S2.0 的進度。
*   **為什麼 S2.0 粗填沒關係？** 因為 `Bus` 這個標籤在 S2.0 只是「存進陣列裡」，沒有任何邏輯依賴它。直到 S2.1/S2.2 處理雙向 Pass 方向和驅動強度時才需要精確的 `Bus` 集合。
*   **建議**：S2.0 寫死這 16 條線（加上 alias 解析）即可。在 S2.1 實作 `DriveInfo` 時，再寫一個方法 `BusResolver.RefineBuses()` 自動算出完整的交集 `(有內部 driver) ∩ (有外部 handler write)` 來覆寫。

### 4. 程式落點架構（回應 §1.8 開放問題 4）

**結論：絕對要用「獨立的 class」，千萬不要放進 `WireCore` 的 partial。**

*   **影響**：`WireCore` 是 S1（執行期模擬）的引擎核心；`NetlistGraph` 是 S2（編譯器/分析器）的資料結構。兩者的生命週期和關注點完全不同。
*   **該怎麼改**：
    *   採用你設計的 `internal class NetlistGraph`。
    *   **微調建議**：把它改成可實例化的 class（`internal class NetlistGraph { ... }`，不要 `static`），然後提供一個工廠方法 `NetlistGraph.BuildFrom(WireCore)`。這能避免全域狀態污染，萬一未來你想寫 Unit Test 跑兩塊不同的小網表來驗證 S2.1 的邏輯提取，非 static 會好用一百倍。

### 5. 驗證計畫（§1.7）

**結論：除了 spot-check，S2.0 必須加入「互斥與窮舉（Mutually Exclusive & Exhaustive）」的斷言（Assertion）。**

*   **該怎麼改**：在你的 `--dump-graph` 或測試代碼中加入以下 Assertions：
    1.  **窮舉檢驗**：
        *   `Count(PullDown) + Count(PullUpStrong) + Count(PullUpLoad) + Count(PullUpActive) + Count(Pass) + Count(Dead) == TotalTransistors`
        *   `Count(Supply) + Count(Input) + Count(Bus) + Count(Internal) == TotalNodes`
    2.  **合理性交叉檢驗（Sanity Check）**：
        *   遍歷所有 `PullDown`：斷言其 `C1 == Ngnd || C2 == Ngnd` 必成立。
        *   遍歷所有 `PullUpLoad`：斷言其 `Gate == C1 || Gate == C2` 必成立，且另一端為 `Npwr`。
        *   遍歷所有 `PullUpStrong`：斷言其 `Gate == Npwr`，且 `C1 == Npwr || C2 == Npwr`。
        *   斷言 `Dead == 0`（驗證 S1.5 的優化成果）。
*   這只需要 10 行 LINQ，但能 100% 保證 S2.0 的分類邏輯沒有漏網之魚（例如把某個奇葩接法的電晶體分錯類或漏算）。

### 總結

S2.0 的設計非常扎實，邊界劃分得非常清晰。「不提前做推導」是這一步成功的關鍵。
請直接依照你的 ① 設計去實作，只要把 `NetlistGraph` 改為非靜態（可實例化），並在測試中加上「窮舉斷言」，這個 PR 就可以穩穩落地了。
