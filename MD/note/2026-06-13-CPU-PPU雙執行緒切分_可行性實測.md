# CPU‖PPU 雙執行緒切分(PDES)可行性實測 — 2026-06-13

## 緣起

使用者構想:既然 netlist 相依性太強、細粒度多執行緒無望(per-chip 慢 15×、bit-parallel
慢 156×),改用「兩條 thread 各跑一份完整 netlist、錯開 n / n−1 步、以整個 step 為單位交換
資訊」。先與 Gemini 對撞分析,再實測。

## 分析結論(我 + Gemini 一致)

- **字面上「兩份相同 netlist 錯開」死路**:相同輸入→相同輸出=純 2× 冗餘;改成投機執行則命中率
  ~0%(時鐘每 hc 翻、會變的 ~3% 節點正是混沌級聯部分,預測=計算)。
- **唯一站得住腳的形式 = 功能切分**:2A03 與 2C02 當兩個弱耦合 logical process(PDES),
  只在「真正互動的匯流排事件」同步,而非像當年 per-chip 嘗試那樣每個 settle wave 同步。
- Gemini 的悲觀預測:因主時脈接到一切、共用匯流排是開關級電氣節點,跨界 churn 會 ≥1/hc →
  退化成 per-chip 的 15× 慘案。給了一個便宜的殺死測試門檻:跨界傳播 ≤0.05/hc 才值得做。

## 實測(DEBUG profiler `[cpu-ppu-boundary]`,WireCore.Recalc.cs InitBoundaryDiag/BoundaryPopTally)

nes-001.js 的 CPU↔PPU 切點 = `io_db[7:0]≡cpu.db`、`io_ab[2:0]≡cpu.ab`、`io_ce`(PPU 選擇)、
`cpu.nmi≡ppu.int`、`cpu.rw≡ppu.io_rw`。按模組前綴(cpu./ppu.)標記每個節點的域。

| 指標(/hc) | full_palette 400k | SMB 3M |
|---|---|---|
| io_ce(真正的 CPU↔PPU 交握) | 0.0119 | **0.0070** |
| nmi | 0 | ~0(workload 仍在 init) |
| 資料匯流排 db(含 CPU↔RAM,PPU tri-state 時看不到→不算跨晶片) | 0.14 | 0.14 |
| 兩晶片同 hc 都活動 | 12.5% | 12.5% |
| 只有 PPU 活動 | 87.5% | 87.5% |
| 只有 CPU 活動 | **0%** | **0%** |
| pop 分布 cpu/ppu/board | — | 10% / 36% / 54% |

checksum 不變(0x9174E19D961CB6E5 @400k full_palette)→ 插樁無擾動。

## 判讀

1. **可行性:通過便宜殺死測試。** 真正的跨晶片交握 ~0.007–0.012/hc « 0.05 門檻。Gemini 的
   「≥1/hc」預測被推翻 —— 關鍵是 PPU 未被選到時輸出 tri-state,共用匯流排雖每 CPU 週期都變,
   PPU 看不到、不需傳。切點窄(~13 條線),粗粒度同步不會被淹。**這是本專案 4 次平行化死路
   以來,第一個碰到不立刻死的方向。**
2. **收益:翻不了盤。** PPU 幾乎每個 hc 都在跑、CPU 只 1/8 hc 動且 cpu-only=0 → 87.5% 時間
   線上拆 CPU 出去毫無平行可言。pop 分布 board 54% / ppu 36% / cpu 10%,PPU 側怎麼算都是
   重邊,且 PPU 本體是一整塊 SCC 不能再切。**理論加速上限 ~1.1–1.5x(最可能 ~1.2x)**,
   對 316× 的即時差距無意義。
3. **工程代價高**:切點位元精確 rendezvous(重現該 hc 內 Gauss-Seidel 順序)、$2002 輪詢讀
   會逼 CPU thread 停等 PPU(保守式 PDES 痛點)、board 54% 要乾淨分側。

**裁決:當加速手段不值得做(~1.1x vs 高風險並行工程);當研究結論值得留 —— 推翻了「這引擎
所有平行化都死」的全稱命題,正確版本是「細粒度平行化死於同步成本;粗粒度功能切分可行但因
PPU 工作獨大而收益微薄」。**

## 補測:兩側工作平衡 + 加速上限(2026-06-13,SMB 3M hc)

升級 profiler 為「依模組統計 pop + 把每顆 board 晶片用 SideOf() 歸 CPU/PPU 側」:

```
pops by module:  ppu 36.3% | cpu 10.1% | cart 0.4% | clk 0.2% | u3 0.1% | 其餘(無名/board)~53%
PDES 兩側:       cpu-side 10.5% | ppu-side 36.3% | neutral(無名 board/clock)53.2%
2-way 上限:       2.75x(假設 53% neutral「免費」——不可能).. 1.12x(neutral 算進重邊)
```

無名節點(GetNodeName 回傳數字 → 各自成桶)佔 53%,pop 數卡在 3M=每 hc 一次 / 2M / 1.5M 等
時鐘諧波上 → 是時鐘分配樹 + PPU 繪圖匯流排 + board 內部,真實工作、不可能免費。

**上限 ≈ 1.12x 的硬證明(不需精算無名節點分側):cpu-only = 0%、ppu-only = 87.5%。** 即 87.5%
的 hc CPU 晶片完全沒動,CPU thread 那段時間全閒置;CPU 側總工作僅 ~10.5% 且全擠在 12.5% 的
「兩者都動」hc。重邊(PPU)扛 ~88–90% → 加速上限 ≈ 1/0.89 ≈ **1.12x**。完整 flood-fill 只會
微調小數點,改不了結論。負載結構(PPU 全程滿載、CPU 大多閒置)讓 2-way 切分天生賺不到。
