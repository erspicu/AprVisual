### 1 & 2. The Unsoundness of Staleness (Why Wall #1 and #2 are Fatal)

Here is the exact proof of why reading a stale `DominantGate` breaks your hard bit-exact requirement:
1. At wave step $T_0$, node `C` is resolved. The Main thread determines its actual dominant driver is `G2`.
2. The Worker thread lags. `DominantGate[C]` still holds the stale value from the previous wave: `G1`.
3. At wave step $T_1$ (same wave, microseconds later), gate `G2` turns off. The Main thread walks `G2`'s gated transistors, looking at endpoint `C`. 
4. Main evaluates the skip: `DominantGate[C] != 0 && DominantGate[C] != nn`.
5. Since it reads the stale value, it evaluates: `G1 != 0 && G1 != G2`. This evaluates to **TRUE**.
6. Main **SKIPS** the re-evaluation of `C`. 
7. **DIVERGENCE:** `C` was actually driven by `G2`. Because `G2` turned off, `C` should have lost its supply and either floated or changed state. Because you skipped it, `C` remains erroneously powered. Your 64-bit checksum just failed.

To prevent this, Main *must* wait for Worker to process `C`. Because `C` was resolved just microseconds ago, you introduce a synchronous inter-core handshake on a microsecond loop. You will lose 1000% performance, not 15%.

### 5. THE REAL ESCAPE: Single-Threaded Data Layout & Domain Shrinking
You are losing 15.4% because you are doing **100% of the maintenance work** to dodge a tiny fraction of the graph, and you are paying cache penalties to do it. You must fix the single-threaded memory physics.

Here are the specific ways to turn P-5 positive, in order of likelihood to work:

#### Escape A: AoS Packing (Eliminate the Write Stream Cost)
Your `DominantGate` maintenance cost is heavily driven by writing to a completely separate 29 KB array (`ushort* DominantGate`). Every write is a potential cache miss and a separate hardware prefetch stream. 

**The Fix:** Pack `NodeStates`, `PruneMask`, and `DominantGate` into a single 4-byte struct.
```csharp
[StructLayout(LayoutKind.Sequential, Pack = 1)]
public struct NodeHotData {
    public byte State;       // 1 byte
    public byte PruneMask;   // 1 byte
    public ushort DomGate;   // 2 bytes
}
```
Allocate a single 58.8 KB `NodeHotData*` array. 
*Why this wins:* When the Main thread's turn-off walk reads `pruneMask[c1a]`, the CPU pulls the 64-byte cache line containing that node. Because `DomGate` is in the **same 4-byte struct**, `DomGate[c1a]` is already in the L1 register. During `RecalcNode`, when you write the new `State` and the new `DomGate`, they hit the exact same cache line simultaneously. You have completely eliminated the secondary memory stream.

#### Escape B: Statically Shrink the Domain (Don't compute what can't be skipped)
Why are you executing the branchy count-and-capture for all 14,700 nodes? 
A node can *only* benefit from the P-5 skip if it is a channel endpoint of a pass-transistor that gets turned off. Many nodes in your graph are purely gates, or sit in static pull-up/pull-down paths that never get skipped this way.

**The Fix:** At load time, flag nodes that can *actually* be skipped. Steal a bit in `NodeInfos` or `PruneMask` (e.g., `IsBypassCandidate`).
Inside `RecalcNodeFastDom`:
```csharp
if ((NodeInfos[nn].Flags & IsBypassCandidate) != 0) {
    // DO the branchy count-and-capture and write DominantGate
} else {
    // Just do the branchless OR. Skip DominantGate entirely.
}
```
If only 40% of your nodes are valid targets for this skip, you just deleted 60% of the -15.4% overhead while keeping 100% of the +3.4% gain.

#### Escape C: Bitwise Count-and-Capture (Kill the Branches)
If `RecalcNodeFastDom` turned your branchless OR into a branchy count, use SWAR (SIMD Within A Register) or bit-tricks to remove the branches.
Assuming your gate max count per node is small (e.g., ≤ 8), you can read the states of all GND supply gates into a single `ulong` (one byte per state, 0x00 or 0x01).
A simple bitwise population count (e.g., `ulong gndVector = ...; int gndCount = BitOperations.PopCount(gndVector);`) gives you the count branchlessly. If `gndCount == 1`, you can use `BitOperations.TrailingZeroCount(gndVector)` to mathematically derive the index of the single active gate, mapping it back to the node ID via a lookup, entirely branch-free.
