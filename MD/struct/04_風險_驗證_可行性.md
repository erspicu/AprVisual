## Part 1：可行性分層

這個專案技術門檻高，遠不只是「優化模擬器」。它要把晶片逆向、圖論分析、邏輯抽取、時序抽象、GPU 平行通通串成一條自動化管線。技術企圖心很大，但學術／研究價值也很大。

### Layer 1：局部成功，明顯可行

包含：

- 讀 Visual2A03 / Visual2C02 網表
- 建圖模型
- 通電 / 浮島 / 迴路分析
- 對小區域抽出 boolean / 序向模型
- emit CPU evaluator 或局部 CUDA evaluator

**這一層明顯可行**，做得好就已經很有價值，因為它本身就是一條「從晶片拓樸到可驗證邏輯模型」的工具鏈。這不是玩具。

### Layer 2：CPU 級的整顆 cycle-accurate 模型

**有條件可行，但工程量很大**。難點不是 parser，而是：

- 動態節點與電容狀態的正確抽象
- bus sharing、pass transistor、回授結構的分類
- phase-based settle 是否真的穩定
- 證明抽取出來的模型確實等價

如果做到這一層，就是一份**少見的 Golden Model**，對模擬器驗證、逆向研究、未文件化行為分析都很有價值。

### Layer 3：整顆 NES 常駐 GPU

**理論誘人，但應視為終局架構，不是首要目標**。到這個階段，問題不再只是邏輯正確性，還會出現：

- 產生的 code 大到難以舒服編譯
- CUDA 記憶體布局與 branching 壓力
- 極高的 debug 成本
- PPU / mapper / APU 整合後的驗證爆炸

務實成功路徑是：

1. 先 Layer 1（局部）
2. 再 Layer 2（CPU 級）
3. 再 Layer 3（GPU 批次）

不要直接跳終局。

### 真的做得出來，價值高嗎？

**高，而且不只一個面向**。

- **技術價值**：少見的 switch-level → executable-logic pipeline，正好踩在 RE / EDA / emulator / HPC 的交叉點
- **工程價值**：可作為手寫模擬器的 validator；協助找 cycle-level bug、illegal opcode、邊界 timing
- **研究價值**：transistor-netlist abstraction 本身就是有意思的研究主題；做到 bit-sliced GPU evaluator 對大規模平行模擬與 AI 環境生成有實質用途
- **稀有性**：這類人不多，能把「晶片網表 → 邏輯抽取 → GPU 執行」串起來的更少

但要對商業價值務實一點：直接商業價值不會自動很高，受眾偏窄；學術與技術價值高，實用價值要看做到哪裡。

### 結論

- 局部工具鏈：**值得做、成功率不低**
- 整顆 2A03 Golden Model：**很難，但很有價值**
- 整顆 GPU 常駐 NES：**最 impressive，也最高風險、最高成本**

> 真正最有價值的里程碑不是「在 GPU 上跑幾萬台 NES」。
> 真正最有價值的里程碑是**建出一份值得信任、可驗證的 switch-level 邏輯抽取核心**。
> 這才是這個專案的靈魂。

## Part 2：三大最可能的技術失敗點

### #1（最致命）：動態節點與記憶體行為抽象錯誤

這是整個專案**最可能讓你做出「看起來合理、實際上錯」結果的核心風險**。

困難來自於：

- switch-level 的很多 state element 不是顯式 DFF
- 有些節點靠浮接與寄生電容保留值
- 有些節點透過 pass transistor 被充放電
- 簡化規則「`gnd / high / hold`」雖然好用，但不一定能涵蓋所有真實行為

如果這個抽象錯了：

- CPU evaluator
- CUDA backend
- Verilog 輸出

**全部都只是更快地算出錯的答案**。

可怕的是它不會大聲失敗。在簡單例子上看起來都對，**到 2A03 / 2C02 真實子區域或邊界 timing 時才漂移**。

### #2：回授、共享 bus、雙向傳輸結構分類失敗

第二大風險：**把複雜雙向網路過度簡化成普通邏輯樹**。

困難來自於：

- 雙向 pass transistor
- 共享 bus
- 不等於普通 latch 的 loop
- SCC / loop detection 找出結構，但**不會自動證明結構的語意**

分類失敗常見後果：

- 共享 bus 島被錯誤切成孤立節點
- 回授區被誤認為普通組合邏輯
- 該用 `current_state` 的區域被強制展開成巨大 boolean 表達式

直接導致：

- 不收斂
- 表達式爆炸
- 模型語意與真實電路不一致

### #3：驗證不足，看似厲害實則不可信

第三大風險不是演算法本身，而是**驗證策略太弱**。

這類專案很容易進入下面這個狀態：

- parser 跑得起來
- code generator 跑得起來
- CUDA kernel 也跑得起來
- 局部 trace「看起來合理」

但仍然**無法證明**：

- graph 是對的
- 抽出的邏輯與電路等價
- 動態狀態抽象正確
- CPU 與 CUDA backend 一致

> 這類專案最大的風險之一是：做了大量工程，但結果**不可信**。

### 排名

如果硬要排：

1. 動態節點與 memory 行為抽象錯誤
2. 回授 / bus / pass-transistor 結構分類錯誤
3. 驗證不足

前兩個是模型層面的失敗。第三個是方法論層面的失敗。三者**任何一個不穩**，整個系統就會 impressive 但不可信。

## Part 3：分層驗證策略

驗證**必須分層**。每一層都要有自己的證據，不能仰賴下一層。

### 3.1 Parser 驗證

- node / transistor 計數對得上原始檔
- 重要 pin 都找得到（`vcc` / `vss` / `clk0` / `res` / `rw` / 位址 bus / 資料 bus）
- pull marker（`+`）通通沒有遺失

### 3.2 Graph 驗證

- gate / source / drain mapping 正確
- 鄰接建構正確
- 在通電條件下的連通分量看起來合理

### 3.3 手刻迷你網表

每個都要附**期望真值表**與**期望 phase 行為**：

- inverter
- NAND
- NOR
- 三態 pass transistor
- dynamic latch

### 3.4 真實小區域

不要從整顆 CPU 開始，挑一小塊：

- reset chain
- 一個 register bit
- ALU carry path 一段

### 3.5 對照參考模型

可比對的對象：

- Perfect6502 風格實作
- Visual6502 自己跑出來的 trace
- 已知的 reset / fetch / opcode 序列

### 3.6 CPU / CUDA 等價

**用同一份 IR**：

- CPU evaluator 跑一遍
- CUDA evaluator 跑一遍
- 兩邊結果**節點對節點完全一致**

這是必要條件。否則無法分辨 bug 是在模型層還是 backend 層。

---

## Part 4：給自己的鐵律

- 任何「看起來對」的結果都不算數，必須有 trace 證據
- 任何 backend 在沒有 CPU 對照之前都不可信
- 任何「跑固定次數 settle」的策略都要有警告與超出記錄
- 任何「為了快」而省略的物理行為必須**寫在註解裡**並標註「未來要修」
