Context: we generate a ~100-page A4 PDF book from a single HTML file via Edge headless
`--print-to-pdf` (Chromium). The HTML contains internal anchor links: citations
`<a class="cite" href="#ref15">[15]</a>` pointing to `<li id="ref15">` in the references
section, and TOC links `<a href="#ch05">`. Chromium correctly converts these to internal
GoTo link annotations — in the raw Chromium output the links work.

Then we post-process with Python pypdf (6.13) + reportlab to add footer page numbers and
an outline, like this:

    reader = PdfReader(path)
    writer = PdfWriter()
    for i, page in enumerate(reader.pages, 1):
        overlay = <reportlab one-page canvas with the page number>
        page.merge_page(overlay)      # stamp number onto the reader's page object
        writer.add_page(page)         # append page-by-page into a fresh writer
    writer.add_outline_item(...)      # bookmarks
    writer.write(out)

Symptom: in the final PDF the link annotations still EXIST (276 /Annots counted) but
clicking a citation does nothing — the internal GoTo destinations appear to be dead.
Outline bookmarks added via add_outline_item DO work.

My hypothesis: rebuilding the document page-by-page with `writer.add_page(page)` breaks
Chromium's internal link destinations — the annotations' /Dest (or /A << /S /GoTo /D >>)
arrays reference page objects (or named destinations in the catalog's /Dests or
/Names tree) of the ORIGINAL document; pypdf's add_page does not remap those references,
and the catalog's name tree is not copied at all, so the GoTo targets dangle.

My planned fix: clone the whole document instead, then mutate in place:

    writer = PdfWriter(clone_from=path)   # full xref-level clone, links preserved
    for i, page in enumerate(writer.pages, 1):
        page.merge_page(overlay_i)        # stamp onto the WRITER's pages
    writer.add_outline_item(...)
    writer.write(out)

(text extraction for locating bookmark headings is done from a separate PdfReader
before any mutation).

Questions:
1. Is the diagnosis right (page-by-page add_page breaks internal GoTo / named
   destinations from Chromium PDFs)?
2. Is PdfWriter(clone_from=...) + in-place merge_page the correct minimal fix, or is
   writer.append(reader) (which has destination-transfer logic) safer? Any pypdf
   gotchas with merge_page on cloned pages (content-stream mutation breaking links
   that live in /Annots — they shouldn't, annotations are page-dict level)?
3. Anything else that silently kills Chromium internal links in pypdf post-processing
   that we should watch for?

**1. Is the diagnosis right?**
Yes. Your diagnosis is perfectly accurate. Chromium `--print-to-pdf` uses Named Destinations for HTML anchor links. These live at the document catalog level in the `/Names` tree. Using `add_page()` in a loop creates a new catalog but fails to copy the `/Names` tree. The `/Annots` on the pages are copied, but they reference named destinations that no longer exist, rendering them dead.

**2. Is `clone_from` + in-place `merge_page` the correct minimal fix vs `append()`?**
**`clone_from` is the best and cleanest fix.** 
* It was explicitly designed for this "read-modify-write" workflow. It guarantees document-level structures (like the `/Names` tree, existing metadata, etc.) are perfectly preserved without needing manual reconstruction.
* `writer.append(reader)` *does* contain logic to transfer named destinations, but `clone_from` is faster and less prone to edge-case translation bugs during object transfer.
* **Gotchas with `merge_page`?** None here. `merge_page` modifies the `/Contents` array and `/Resources` dictionary of the page. It completely ignores and preserves the `/Annots` dictionary, so your links will not be damaged.

**3. Anything else that silently kills Chromium links?**
* **Overlay Z-index / Transparency:** Ensure your reportlab overlay canvas does not have a solid white background (make sure it's transparent). While PDF `/Annots` technically float on a UI layer above the `/Contents` stream, some PDF viewers' hit-testing fails if a solid object is drawn directly over the annotation bounding box.
* **TOC/Outline wiping:** If the original Chromium PDF happened to generate any bookmarks, `clone_from` retains them. If you add your own via `add_outline_item()`, they will be appended. If you want to *replace* an existing outline, you would need to clear `writer.outline` first (though Chromium headless usually outputs zero bookmarks, so this is rarely an issue). 

Your planned fix using `PdfWriter(clone_from=path)` with a separate `PdfReader` for text extraction is the exact optimal path for `pypdf` >= 3.1.0.
