## 1. 重現參數(以後忘了查這裡)

### checksum 驗證(determinism gate)
```
dotnet <AprVisual.etc.dll> --cpu-bench src/AprVisual.etc/netlists/<chip> --chip <chip> \
       --bench-hc 500000 --warmup 50000 --rounds 3 --workload nop
```
**golden checksum**(這組參數下逐位元固定):
| chip | golden checksum |
|---|---|
| 6502 | `0x97C10609CF8BF86F` |
| 6800 | `0x713AF51D83FAAC5E` |
| z80  | `0x0CE4E8586F898641` |
（與 `MD/note/2026-06-16-*` 一致。⚠️ checksum 隨 `bench-hc`/`warmup` 而變 —— 換參數就對不上,不是壞了。）

### 速率(hc/s)量測方法(cross-cpu.html「數字的出處 / Where the numbers come from」)
- NOP-sled、**每輪 1,000,000 半週期、6 輪取穩態中位數**(前幾輪 .NET JIT 分層/PGO 暖機會偏低)、釘核(`--pin`)。
- CLI:`--cpu-bench <dir> --chip <c> [--naive] [--workload nop|fuzz|reset] --bench-hc 1000000 --rounds 6`。
- 同機 AMD Ryzen 7 3700X、.NET 11 x64 Release。C++ 對照:`tools/cpp-naive` + `tools/cpp-ours`(clang/LLVM -O3),載入 C# `--export-netlist` 匯出的網表。JS 基線:`tools/visual6502-node`(Node headless)。
- 6800 / Z80 比照原始 harness 跳過 weak(depletion)電晶體。

### kHz 換算(cross-cpu.html「換算成 kHz」段)
- 裸 CPU 一個 cycle = 2 個 half-cycle → **模擬時脈 = hc/s ÷ 2 ÷ 1000 (kHz)**。
- published(涼機)hc/s → kHz:6502 157,154→~78.6kHz、6800 94,921→~47.5kHz、z80 62,449→~31.2kHz。

## 2. 2026.06.20 優化同步到 etc（本次）
- 兩個優化套到 `src/AprVisual.etc/Sim/WireCore.cs`(turn-off `gatesOff` 的 `StableDedupeItems`)+ `WireCore.FastPath.cs`(`RecalcNodeFast` 讀 packed `GndPwr`、特化 gp==1 / gp==2)。etc 的 NodeInfo 早已有 packed `GndPwr` 欄位,直接可套。
- **bit-exact 驗證**:同參數 with-opt vs pre-opt(stash 比對,預設參數)三顆全 MATCH;且在上方 golden 參數下三顆 checksum 逐位元 = published golden → 優化在小網表也行為中性。
- 速率:本次重量受室溫(熱)影響、比 published 略低,**不是優化退步**(同 NES boost 問題)。要更新 cross-cpu.html 數字需涼機重量,或用 thermal-neutral A/B 的增益% 套到 published 基線。

## 3. 雷點
- bare-CPU 數字也是 boost/thermal 敏感(memory-latency-bound),單跑/暖機差很多 → 要涼機 + 中位數。
- etc 是獨立 fork,有自己的 WireCore 副本;S1 的優化要手動同步過來(本筆記記錄一次同步流程)。
