## Part 1：Visual6502 JavaScript 模擬器

### 1.1 檔案分工

| 檔案 | 角色 |
|------|------|
| `wires.js` | 載入 segdefs / transdefs，建立 `nodes` / `transistors` 兩張表 |
| `chipsim.js` | **核心模擬演算法**（recalcNodeList / recalcNode / getNodeGroup / getNodeValue） |
| `expertWires.js` | UI / debug / trace（646 行，主要是視覺化） |
| `nodenames.js`、`segdefs.js`、`transdefs.js` | MOS 6502 revD 的網表資料 |
| `chip-6800/`、`chip-z80/` | 6800、Z80 的同型資料（同一個模擬器都能跑） |
| `macros.js` | 高階指令包裝（reset、step、bus 模擬） |
| `memtable.js` | RAM / ROM 模擬 |
| `testprogram.js` | 預設要跑的 6502 機器碼 |

### 1.2 節點與電晶體的記憶體布局（`wires.js`）

```javascript
// setupNodes
nodes[w] = {
  segs: [],          // 視覺化用 polygon
  num:  w,
  pullup: seg[1]=='+',  // ← 只從 segdef 第二欄取 '+'
  state: false,
  gates: [],         // 此節點當作 gate 的所有電晶體
  c1c2s: []          // 此節點當作 channel 端點（c1 或 c2）的所有電晶體
};

// setupTransistors
transistors[name] = {name, on, gate, c1, c2, bb};
nodes[gate ].gates .push(trans);
nodes[c1   ].c1c2s.push(trans);
nodes[c2   ].c1c2s.push(trans);
```

兩個觀察：

- 模擬器**只有 `pullup`，沒有 `pulldown` 從 segdef 來**。`pulldown` 只在執行期被 `setPd()` / `setLow()` 用來「驅動外部 pin 拉低」。換言之，segdef 上的 `-` 在原始模擬器裡其實**沒有特別處理**。
- `setupTransistors` 在偵測到 `c1==ngnd` 或 `c1==npwr` 時會把它換到 `c2`。意思是：**c1/c2 順序不可信賴**，必須當成可交換。

### 1.3 模擬主迴圈

`recalcNodeList` 是核心：

```javascript
function recalcNodeList(list){
  for (var j = 0; j < 100; j++) {     // ← 硬上限 100 次
    if (list.length == 0) return;
    list.forEach(recalcNode);
    list = recalclist;
    recalclist = [];
    recalcHash = [];
  }
}
```

`recalcNode(node)`：

1. 用 `getNodeGroup(node)` 把目前**全部電氣連通**的節點集合起來（沿著 ON 的電晶體 BFS）
2. 用 `getNodeValue()` 決定這個 group 應該是 0 / 1
3. group 內每個節點若狀態改變，則翻轉它所控制的全部電晶體 ON/OFF，並把那些電晶體的 c1/c2 推進 recalclist

這就是把「change」沿著電氣連通與閘控關係**漣漪式傳播**。

### 1.4 群組求值（最關鍵的 30 行）

```javascript
function getNodeValue(){
  if (arrayContains(group, ngnd)) return false;   // GND 永遠贏
  if (arrayContains(group, npwr)) return true;    // VCC 次之
  for (var i in group){
    var n = nodes[group[i]];
    if (n.pullup)   return true;     // 任何 pull-up → high
    if (n.pulldown) return false;    // （只有外部驅動會設這個）
    if (n.state)    return true;     // 群組內任一節點 state 為 high → high
  }
  return false;                       // 否則 → low
}
```

**這就是設計 docs 裡反覆提到的 NMOS 三層規則的原型版本**：

| 規則 | 對應條件 |
|------|---------|
| GND 路徑贏 → 0 | group 包含 `vss` |
| 否則 VCC / pull-up → 1 | group 包含 `vcc` 或任一節點 `pullup=true` |
| 否則保留前狀態 | group 內任一節點 `state=true` |
| 預設 | 0 |

### 1.5 對照設計文件的修正點

我們的 C# 版本相對於這份原始實作，要**有意識地**修這些東西：

1. **取消 100 次硬上限**，改成「settle 到收斂或達到 max」並在超出時 log warning（見 [`01_技術指南.md` §11.1](./01_技術指南.md)）。
2. **保留 `PullType`，但同時保留 `pulldown` 標記**（segdef 的 `-` 在 2A03 / 2C02 也許有用，至少在 parser 階段不要丟）。
3. **`getNodeValue` 的「state→high」是「保留前狀態」的代用品**。我們的版本要明確分辨「真的有路徑通到高源」vs「只是這個 group 上次的值是 high」 — 這兩件事在原始模擬器裡被混為一談，是 dynamic node 抽象的根本缺口。
4. 原始模擬器**沒有顯式的浮島偵測** — 它靠 `state` bit 隱式達成 hold 行為。我們的版本應顯式做 floating island 分類，並能輸出哪些節點在哪個 phase 是 floating。

### 1.6 設計文件提到的「100 次 settle」其實就在這

設計 docs 反覆警告「不要把跑三次當成正確性的證明」，原始模擬器的對應物就是 `recalcNodeList` 裡的 `for (j=0; j<100; j++)`。**100 次只是經驗法則**，不是收斂證明。

## Part 2：Visual2A03 / Visual2C02 資料集

### 2.1 檔案大小對比

| 檔案 | 行數 |
|------|------|
| `visual2a03-segdefs.js` | 27,716 |
| `visual2a03-transdefs.js` | 11,034 |
| `visual2a03-nodenames.js` | 2,265 |
| `visual2c02-segdefs.js` | 35,568 |
| `visual2c02-transdefs.js` | 16,946 |
| `visual2c02-nodenames.js` | 2,642 |
| 6502 revD（對照） | seg 8246 / trans 3512 / names 949 |

**2A03 ≈ 6502 的 3 倍規模；2C02 ≈ 6502 的 4–5 倍**。實作時 parser 必須能處理這個量級，graph 也要設計得夠精簡（不要每個節點都吃太多記憶體）。

### 2.2 segdefs.js 格式（與 6502 相同）

```javascript
[node_id, pull, layer, x1,y1, x2,y2, x3,y3, ...]
//  ↑     ↑     ↑      ↑————————————————————
//  |     |     |       polygon 多邊形座標
//  |     |     +-- layer (視覺化用)
//  |     +-- '+' / '-'  ← PullType（最重要的非可選欄位）
//  +-- 該 polygon 屬於哪個電氣節點
```

注意：

- 同一個 `node_id` 會在 segdefs 出現**很多次**（一個節點由多塊矽 / 金屬組成）。
- Parser 在去重時要決定如何處理 `+` / `-` 衝突（同一節點的不同 segment 標不同 pull 是否應發 warning？— 應該要）。

### 2.3 transdefs.js 格式（2A03/2C02 多了一個欄位！）

**6502** 原版：
```javascript
['t0', 357, 558, 217, [bbox], [geom]]
//      gate src drn
```

**2A03 / 2C02**：
```javascript
['t12798', 10357, 13170, 10001, [bbox], [geom], false]
//                                                ↑
//                                  第 7 欄：weak / depletion 標記
```

**這個 boolean 第 7 欄**是 6502 版本沒有的東西。它代表這顆電晶體是**弱驅動**（depletion-mode load 或類似的 pull-up 元件）。原始 visual6502 模擬器**會讀這個欄位**（在後期 NES 衍生版本裡），把弱電晶體與強電晶體分開處理。

> ⚠️ **重要**：原 6502 版的 `chipsim.js` 不會處理這個欄位。我們的 C# parser **必須**保留它，因為對 NMOS 邏輯抽取至關重要（弱拉高 vs 強拉低正是「GND wins」規則的物理基礎）。

### 2.4 nodenames.js — 一座金礦

這是整個資料集**最有價值的部分**，遠超過視覺化 polygon。

#### 2A03 nodenames 重點

外部 pin（NES 級）：

```
pwr, gnd, res, rw, db0..db7, ab0..ab15
clk_in, phi2, clk0, clk1out, clk2out
nmi, irq, rdy, so, sync
out0, out1, out2, joy1, joy2, snd1, snd2
```

**已命名的內部 6502 暫存器位元**（驗證用！）：

```
a0..a7      ← Accumulator
x0..x7      ← X register
y0..y7      ← Y register
pcl0..pcl7  ← PC low（第一儲存節點）
pclp0..pclp7 ← PC low（pre-incremented，第二儲存節點）
"~pclp0".."~pclp7" ← 反向（自動產生的別名）
```

**這意味著：你不需要從拓樸結構去猜哪些節點是 ALU 累加器** — 它們**已經被命名了**。

對驗證策略的影響：

- 只要在 reset 後跑幾個 cycle，把 `a0..a7` / `x0..x7` / `pcl0..pcl7` 的 trace dump 出來
- 直接對照 Perfect6502 / 真實 NES 的 register state
- **這條 path 是現成的 Golden Model 比對接口**

#### 2C02 nodenames 重點

PPU 的時鐘系統：

```
clk0, _clk0, pclk0, pclk1, pclk0_en, pclk1_en
```

PPU 是**內部多相時鐘**（CPU 的 clk 進來後再分頻成 pclk0/pclk1，並各有 enable），這驗證了設計 docs 中「PPU 比 CPU 複雜得多」的說法。

PPU 的 bus 多工：

```javascript
ab7:2772,    db7:2772,   // 同一節點！
ab6:2773,    db6:2773,
...
ab0:1991,    db0:1991,
ale:1611,                 // Address Latch Enable
```

**`db[7:0]` 與 `ab[7:0]` 在 PPU 上是同一條 bus** — 透過 `ale` 訊號分時多工。這是 PPU 最容易在實作上踩雷的地方，**parser 不要把它們當成不同節點處理**。

`/ext_out0..3` 這種**斜線開頭**是「低有效（active-low）」訊號的命名慣例。Parser 對節點名稱用字應寬容處理（接受 `/`、`#`、`~`、`_` 等前綴）。

## Part 3：對 C# 實作的具體影響

把上面的觀察整理成 parser / model 必須處理的事項：

### 3.1 Parser

- 接受 `segdefs.js` 中**同一節點多 segment** 的去重
- **保留** `PullType`：`+` 與 `-` 都要存（不要學原版只存 `+`）
- 接受 transdefs **第 7 欄的 weak / depletion 標記**（2A03/2C02 才有）
- 解析 `nodenames.js`：支援 `/`、`#`、`~`、`_` 前綴，以及如 `"#pclp0": 1227` 這類引號語法
- 處理 PPU 的「同節點兩個名字」（`ab7 == db7`）— 不要報錯，要當成 alias

### 3.2 NetTransistor 多一個欄位

```csharp
public sealed class NetTransistor
{
    public int     Id     { get; init; }
    public NetNode Gate   { get; init; } = null!;
    public NetNode Source { get; init; } = null!;
    public NetNode Drain  { get; init; } = null!;
    public bool    IsWeak { get; init; }   // ← 第 7 欄；6502 沒有，2A03/2C02 有
}
```

`IsWeak == true` 的電晶體在邏輯抽取時是 **pull-up / depletion load** 的角色，**不是**普通的 pull-down 路徑。

### 3.3 NodeState 要分辨四種狀態

不要急著只用 `bool[]`。原始模擬器把「保留前狀態」與「真的被驅動到 high」混淆了。我們應該分辨：

```csharp
public enum NodeValue
{
    Low,            // 強制為 0（有 GND 路徑）
    High,           // 強制為 1（有 VCC / strong pull-up 路徑）
    HoldPrevious,   // 浮接、靠寄生電容保留
    Undefined       // 衝突或未初始化
}
```

第一版可以先用 `bool` + `bool isFloating` 兩個欄位，但要**結構上保留升級空間**。

### 3.4 NodeGroup 演算法直接照抄即可

`getNodeGroup` 那個 BFS（沿 ON 電晶體的 c1/c2）是正確的。我們的 C# 版本就照搬：

```csharp
public sealed class ConductionAnalyzer
{
    public HashSet<int> FindConnectedComponent(
        NetlistGraph graph, int startNodeId, NodeState state)
    {
        var visited = new HashSet<int>();
        var queue = new Queue<int>();
        queue.Enqueue(startNodeId);
        while (queue.Count > 0)
        {
            int n = queue.Dequeue();
            if (!visited.Add(n)) continue;
            if (n == graph.GroundId || n == graph.PowerId) continue;
            foreach (var tx in graph.Nodes[n].ConnectedChannels)
            {
                if (!IsTransistorOn(tx, state)) continue;
                int other = tx.Source.Id == n ? tx.Drain.Id : tx.Source.Id;
                if (!visited.Contains(other)) queue.Enqueue(other);
            }
        }
        return visited;
    }
}
```

注意 `if (n == ground || n == power) continue;` — 不再從供電節點出發繼續走，但**它們已經被加進 group**。和原版一致。

### 3.5 Reset 與外部驅動的對應

原始模擬器 `setHigh(name)` / `setLow(name)` / `setPd(name)` 是用「強制改 pullup/pulldown 標記」來驅動 pin。我們的 C# 版本可以做得更乾淨：用一個 `Dictionary<int, NodeValue> ExternalDrives` 表示「這幾個 pin 目前被外部強制成什麼」，evaluator 在 `getNodeValue` 等價邏輯裡優先讀它。這樣外部驅動是顯式的、可記錄到 trace 裡。

## Part 4：交叉檢查清單

當 Phase 0–1 完成時，下面這些事**全部**要能驗證：

- [ ] 用 2A03 segdefs 建出來的節點數 ≈ nodenames 內提到的最大 ID
- [ ] 用 2A03 transdefs 建出來的電晶體數 ≈ 11000 多
- [ ] 所有 `nodenames.js` 列出的 pin（pwr / gnd / res / rw / db / ab / clk0 ...）都能在 graph 裡找到
- [ ] 任意挑一個 transistor，它的 gate / c1 / c2 三個 node id 都存在於 graph
- [ ] 對 `pwr`、`gnd` 從供電做 BFS（所有電晶體都 ON 的條件下），覆蓋率應該幾乎是全 chip
- [ ] 把 6502 的 segdefs / transdefs 餵進去，能跑出和原始 JS 模擬器一致的 reset 後 trace（把 a0–7、x0–7、pcl0–7 dump 出來比對）

最後一條是**最有力的驗證方法**：用同一份 6502 資料集驗證你的 C# evaluator 與 JS evaluator 等價。這是免費的 Golden Model，**比手刻網表測試更接近真實使用情境**。
