# S2.4 — IR 直譯器（`IrEngine`）+ S2.5 hybrid bridge（合併到這步做）+ S2.6 等價 gate

> S2 的第 4 步（+ 5、6 合併）。S2.0~S2.3 已產出：`nextExpr[N]`（per node 的布林 next-state Expr，~94.2% 的 node 有；hybrid 的 `null`）+ `IsSequential[]` + `Hybrid[]`（847 個 hybrid，主要是真 multi-driver bus）。S2.4 = 一個跟 S1 平行存在的 backend：用 IR evaluate 那 ~94% 的 node、hybrid 的 847 個跑 S1 的 switch-level recalc（= S2.5 的 bridge），兩者共用 `WireCore.NodeStates`，handler chain（clock / RAM/ROM / video）照跑。然後 S2.6 = 逐 node、逐 half-cycle 跟 S1 對照（這就是整個 S2 的過關 gate）。**不改 S1 既有行為**（IrEngine 是新增的；S1 的 `Step()` 路徑不動）。
>
> 工作流程（`00_計畫總覽.md` §3）：① 設計描述 → ② 丟 Gemini → ③ 評估採用 → ④ 實作 → ⑤ 驗證。可分多 firing（①寫設計+丟Gemini ②topo+SCC+evaluator ③Step 迴圈+bridge ④等價 harness+驗證+commit）。
>
> **承接**：`Expr`{Const,NodeRef(現值),Not,And,Or,Mux,Hold(電容保值=上一拍值),Prev(上一拍值，打斷環),Complex}；S1 的 `Step(count)` = 每次 `RunHandlerChain(); if(TraceLevel) CaptureTraceLine(); Time++;`（clock handler 翻 clk）；`ResetNes(full)` = 清 RAM + `Reset()` + `RecomputeAllNodes()`（S1 settle 到決定性穩態）+ assert/run/deassert res；`WireCore.NodeStates` 是 `byte* 0/1`；`RecalcNodeList(span)` / `ProcessQueue` 是 S1 的 recalc。

## ① 設計描述

### 1.1 IR build（一次性，`LoadSystem` 後 / `IrEngine.Build()`）

從目前的 `WireCore` 網表（已 compose + lowering）：`NetlistGraph.BuildFrom()` → `DriveAnalysis.Analyze()` → `NextStateModel.Build()` → `SccModel.Build()` → 得 `nextExpr[]`、`isSequential[]`、`hybrid[]`。然後：

1. **建「現值依賴圖」**：`nextExpr[v]` 裡每個 `NodeRef(M)` → 一條 `v → M` 的依賴（`Hold(M)`/`Prev(M)` **不算**邊 —— 那是上一拍）。hybrid node（`nextExpr == null`）不在圖裡（它走 switch-level；但別人引用它的 *現值* 是 OK 的 —— bridge 會把它的現值放在 `NodeStates`）。
2. **Tarjan 找殘餘 SCC**（size>1，或 size-1 但 `nextExpr[v]` 引用 `NodeRef(v)`）—— 這些是 S2.3 沒打斷的（複雜回授 / cross-phase pipeline）。第一刀：**把殘餘 SCC 的所有 node 標 hybrid**（`nextExpr[v] = null`、`hybrid[v] = true`），丟回 switch-level。（之後可選做 S2.3 的 Stage D —— 用 `clk0`/`clk1` 互斥認出「假環」、用 `Prev` 打斷 —— 來救回一些；先看殘餘 SCC 有多少。）
3. **拓樸排序剩下的 non-hybrid node**（依現值依賴圖；hybrid 不在裡面）→ `evalOrder[]`（一個 node id 的陣列，按「先算被依賴的、後算依賴別人的」）。
4. **`compactExpr`**：把每個 non-hybrid node 的 `nextExpr` 攤平成「一串 stack-machine 指令」或一棵小樹的指標形式 —— 第一刀直接用 `Expr` 樹 + 遞迴 eval（簡單；S3 才優化成攤平）。
5. **hybrid node 清單** + 「IR node 中引用了 hybrid node 現值的」清單（bridge 用）。

### 1.2 evaluate 一個 node（`EvalExpr(Expr e)`）

遞迴：`Const(b)` → `b ? 1 : 0`；`NodeRef(m)` → `NodeStates[m]`（現值）；`Hold(m)` / `Prev(m)` → `prevStates[m]`（上一拍值）；`Not(x)` → `1 - eval(x)`；`And(l,r)` → `eval(l) & eval(r)`；`Or(l,r)` → `eval(l) | eval(r)`；`Mux(c,a,b)` → `eval(c) != 0 ? eval(a) : eval(b)`（short-circuit OK）；`Complex` → 不該出現（hybrid 的 `nextExpr` 是 null 不是 Complex）。

### 1.3 `Step(1)` 一次（mirror S1 的 `Step` 但用 IR）

```
RunHandlerChain();                              // clock toggle / RAM-ROM callbacks / video — 跟 S1 一樣，會直接寫 NodeStates（外部驅動 + override）
Array.Copy(NodeStates, prevStates, NodeCount);  // 快照「這一拍開始時」的狀態 → Hold/Prev 讀這個
foreach v in evalOrder: NodeStates[v] = EvalExpr(nextExpr[v]);   // 一遍：按拓樸序算 non-hybrid node（讀 NodeStates 的現值 + prevStates 的上一拍值）
// ── bridge：hybrid node 還沒算；用 S1 的 switch-level recalc 補 ──
//   enqueue：(a) handler 改過的 node（handler 內部已 SetHigh/Low → 已 enqueue 了？要確認 handler 路徑），(b) 上面 IR-eval 改了值的 node（讓 ProcessQueue 從它們傳播到相連的 hybrid node），(c) hybrid node 本身（讓它們被重算）。
//   ProcessQueue();   // S1 的 recalc：會 re-settle hybrid node（用 group-BFS）；對 non-hybrid node 它會 re-derive，理論上跟 IR 算的一樣 → 不變 → 不再傳播 → 收斂快。
if (TraceLevel != 0) CaptureTraceLine();
Time++;
```
細節要小心：(1) handler（`SetHigh`/`SetLow`）會走 S1 的 `SetNodeState` → 已經 enqueue 了相鄰 node。所以 handler 改完後 recalc queue 裡已經有東西。(2) IR-eval 直接寫 `NodeStates[v]` —— 沒走 `SetNodeState` → 沒 enqueue。所以 IR-eval 完要把「值有變的 IR node」手動 enqueue（`EnqueueNode(v)`）。(3) 然後 `ProcessQueue()` 跑 S1 的 recalc —— 它會處理 queue 裡的（handler 改的 + IR 改的 + 連帶 enqueue 的 hybrid node），用 group-BFS re-settle。對 non-hybrid node：ProcessQueue 會用 `ComputeNodeGroup` re-derive 它的值 —— 如果 IR 算對了，結果跟現值一樣 → `SetNodeState` 不觸發變化 → 不傳播。對 hybrid node：ProcessQueue 第一次算它（從 raw 0/pull-up 狀態？不對 —— `prevStates` / `NodeStates` 已有它上一拍的值；ProcessQueue 的 `getNodeValue` 對「無 flag 的浮島」會回 `_maxState`（最大 cap 的 node 的現值）= 上一拍值 —— 所以 hybrid node 的 hold 行為由 S1 的浮島機制自然處理）。→ 整個收斂後 `NodeStates` 就是這一拍的正確結果。
- **第一刀的「ProcessQueue 也 re-derive non-hybrid node」是浪費**（IR 已經算對了）—— 但**正確**，而且 ProcessQueue 對「值沒變」的 node 不傳播 → 不會無限。S3 才優化（讓 ProcessQueue 只碰 hybrid 區）。S2.4 要的是「跑得對」。
- **更大的擔心**：IR-eval 的「一遍拓樸序」夠不夠？non-hybrid node 之間 —— topo 排好了 → 一遍夠。但 non-hybrid node 引用 hybrid node 的*現值* —— hybrid node 是 ProcessQueue 之後才算的，IR-eval 時 hybrid node 還是上一拍值。所以 IR node 用了「上一拍的 hybrid 值」算自己 → 然後 ProcessQueue 算 hybrid → hybrid 變了 → ProcessQueue 傳播到相連的 IR node → 那個 IR node 被 ProcessQueue re-derive（用 group-BFS，不是 IR）→ 變對。Hmm —— 所以「IR node 緊鄰 hybrid node」的會被 ProcessQueue 用 switch-level 重算。OK，正確（只是那些 node 沒享受到 IR 加速）。或者：IR-eval → ProcessQueue → 再 IR-eval（吃 ProcessQueue 算好的 hybrid 現值）→ 再 ProcessQueue → 迭代到收斂。第一刀先用「IR-eval 一遍 → ProcessQueue 一遍」+ cap-and-warn（若 ProcessQueue 改了 > N 個 IR node，警告 —— 表示 IR/hybrid 邊界耦合太緊，要看）。

### 1.4 等價（S2.5/S2.6）

- **初值**：IrEngine 跟 S1 共用 `NodeStates`。`LoadSystem` 跑完（含 `ResetNes(full)` 的 S1 settle）後，`NodeStates` 是決定性穩態；IrEngine 接手 —— `prevStates` 初值 = `NodeStates`。
- **`--engine ir`**（CLI flag）：`LoadSystem` 後若 `--engine ir` → 用 `IrEngine.Step` 取代 `WireCore.Step`（`RunFrame` / `--test` / `--trace` / `--ppu-dump` 都改吃 IrEngine.Step）。
- **`--trace-cmp <rom> [--cycles N]`**：跑 S1 N 個 half-cycle、每拍快照 `NodeStates` 進 list；`ResetNes(full)`；跑 IrEngine N 拍、每拍快照；diff 兩個 list → 報第一個不一致的 half-cycle + 哪些 node + 它們的名字/`nextExpr`。N capped（~2000，記憶體 ~30MB；夠涵蓋 reset 序列 + 幾百個指令）。長跑用 `--test --engine ir` / `--trace --engine ir`（功能比對：blargg `$6000` 簽章、CPU state）。
- **S2 過關條件**：`--trace-cmp` 對某 ROM 前 ~2000 half-cycle bit-identical；`--test --engine ir` 跑得通 S1 跑得過的 test ROM；hybrid 覆蓋率記下（94.2%）；dump `ir_debug.json`（每 node 的 `nextExpr` / 組合-序向 / hybrid / evalOrder）。

### 1.7 開放問題（丟 Gemini）

1. hybrid bridge 的「IR-eval 一遍 → ProcessQueue 一遍」第一刀 —— 對嗎？會不會收斂不了 / 漏算？「IR node 緊鄰 hybrid node」的會被 ProcessQueue 用 switch-level 重算（沒享受 IR 加速）—— 可接受嗎？要不要「IR-eval → ProcessQueue → IR-eval → … 迭代到收斂」？
2. 殘餘 SCC（S2.3 Stages B-E 沒做 → S2.4 用 Tarjan 找出來 → 標 hybrid）—— 6502/2C02 上會有多少？有沒有一定是「cross-phase 假環」（用 `clk0`/`clk1` 互斥能打斷）的？該不該 S2.4 就做那個 Stage D（找 clk0/clk1、打斷假環、用 `Prev`）來救回，還是先全標 hybrid 看數量？
3. `Hold(M)` vs `Prev(M)` —— 我把它們都當「上一拍（這一拍 IR-eval 開始前）的值」= 讀 `prevStates`。對嗎？有沒有情況 `Hold` 該是「這一拍內被驅動前的值」（更細的時間粒度）？6502 的 dynamic latch（`Mux(we, data, Hold(self))`）—— `Hold(self)` = 上一個 half-cycle 的值，對嗎？
4. `--engine ir` 的 `Step` 跟 S1 的 `Step` —— S1 的 `Step` 是 `RunHandlerChain(); ...; Time++`，沒有顯式 recalc（recalc 是 handler 的 `SetHigh`/`SetLow` 觸發的 `SetNodeState` → enqueue → 然後... 等等，ProcessQueue 什麼時候跑？）。我得先搞清楚 S1 的 `Step` 裡 recalc 是在哪一步跑的（`RunHandlerChain` 內？還是 `SetNodeState` 馬上 settle？）—— 這影響 IrEngine.Step 的結構。
5. 等價 harness：snapshot-compare（跑 S1 全程快照 → reset → 跑 IR 全程快照 → diff）vs lockstep（兩個 NodeStates 同步比）—— 共用 `NodeStates` 的話只能 snapshot-compare、N capped。夠嗎？長跑只比 CPU state 夠嗎？

## ② Gemini 分析回饋

> `gemini-3.1-pro-preview-customtools`，log：`tools/knowledgebase/message/20260512_113929.txt`。摘要 —— **方向非常準確，照做**：
- **§1.7-Q1 hybrid bridge（IR-eval 一遍 → enqueue 變動 → ProcessQueue 一遍，無迭代）**：完全 OK，第一刀就該這樣。會收斂（ProcessQueue 是 S1 的 group-BFS，本身收斂；IR 只是幫 94% 的 node 預先算好精確值）。IR 緊鄰 hybrid 的 node 被 ProcessQueue 用 switch-level 重算 —— 完全可接受（「IR 預測，switch-level 校正」）。**絕對不要 IR↔ProcessQueue 迭代**（複雜、可能無窮震盪）。**加一行 log：統計 ProcessQueue 改了多少個 *non-hybrid* node 的值** —— 少（<50）表示 IR 邊界預測準；極大就記下留 S3。
- **§1.7-Q4 / 總結-Q2 S1 Step 的 recalc 時機 —— 實作時最大的地雷，先看 code**：在 visual6502/MetalNES 架構，handler 的 `SetHigh(clk)` / `SetNodeState()` **通常內部就直接觸發 ProcessQueue/Recalc**。若 S1 的 handler 已觸發 recalc → `RunHandlerChain()` 後整個晶片已被 S1 算完一遍收斂 → 此時 IR-eval 只是「再證明一次 S1 的值對」（對驗證沒差，但 IR 沒在「驅動」網表）。→ **Case A（最好）**：handler 只 enqueue、外部 `Step()` 才顯式 `ProcessQueue` → IrEngine.Step 照原計畫。**Case B（常見）**：handler 寫值立刻 settle → IrEngine.Step 裡照跑 handler（S1 settle 局部）、強行快照 `prevStates`、跑 IR、若跟 S1 有差異覆寫、最後再 `ProcessQueue` 同步 hybrid。S2 先求對，S3 再砍 S1 的多餘 recalc。
- **§1.7-Q3 Hold/Prev = 讀 prevStates 快照**：OK，理解正確。`Prev` 打斷假環 / `Hold` = pass 關閉時電容保值；在零延遲拓樸排序的布林網裡，`Mux(we, data, Hold(self))` —— `we`=1 選 `data`（現拍最新值）、`we`=0 必須維持「這一拍開始前」的值；同一 half-cycle 內讀 `Hold(M)` **絕不該**讀到同 half-cycle 內其他分支對 M 的臨時修改（拓樸序不保證同層先後）→ 讀 `prevStates[M]`（half-cycle 初快照）對 6502 dynamic latch 嚴格正確。
- **§1.7-Q5 等價 harness：snapshot-compare 是最優解**：lockstep（兩份 NodeStates）會牽扯 handler 改寫硬體狀態的巨大重構（不想動 S1）→ snapshot-compare 唯一解。2000 half-cycle（~300 CPU 指令）看似少，但 switch-level→boolean 的 bug 通常**在 reset 序列前 ~50 拍就全面炸裂**（bus 衝突沒解好、latch 寫錯邊）。mismatch 時印「哪一拍、哪幾個 node、該 node 的 nextExpr」是 debug 救命神器。長跑用 `--test --engine ir`（CPU state / `$6000` 簽章對 → 整台 NES 等價），不比每個內部 node。
- **§1.7-Q2 / 總結-Q3 殘餘 SCC**：**先全部標 hybrid，S2.4 不做 Stage D**（卡進 clk0/clk1 互斥分析會迷失在圖論裡、延誤核心 IR engine 驗證）。Tarjan 掃出 size>1 / 自引用的 SCC → 無腦全標 hybrid（`nextExpr=null`）。可能讓 hybrid 從 847 暴增到 2000 —— **完全沒關係**，只要 ProcessQueue bridge 能動、跑起來功能對，S2 就過關。「消滅殘餘 SCC」留 S2.7 / S3。
- **§1.7 合併 S2.4/2.5/2.6**：**必須合併，分開無法驗證**（IR 算的值要透過 bridge 餵給 hybrid 網，否則 hybrid 浮接、下一拍崩；沒等價閘不知 S2.4 寫對沒）。**開發順序**：① `IrEngine.Build()`（Tarjan + topo sort + `prevStates` 配置）② snapshot-compare 的 test runner（先拿 S1 自己比自己確保 harness 沒寫錯）③ `IrEngine.EvalExpr()` + `Step()` ④ 掛上去跑 2000 拍、無盡 debug。

## firing 8 起：**driving 模式**（IR 真的驅動 netlist + ProcessQueue hybrid bridge + S3 加速的底子）

> checking 模式（firing 2-7）= 「IR 算、跟 S1 settle 完的值比」（S2.6 過關）。driving 模式 = 「IR 取代 S1 的 settle，只剩 hybrid node 跑 S1 的 group-BFS」—— 這才是 S2.4 計畫原本要的東西、也是 S3 benchmark 的對象。先設計 + 丟 Gemini（firing 8），再實作（firing 9+）。

### ① 設計描述（driving 模式）

**現狀盤點（已查 S1 程式）**：S1 是 **Case B** —— `SetHigh/SetLow(nn)` = `ns.Flags |= SetHigh; RecalcNodeList(nn)` = `EnqueueNode(nn); ProcessQueue();`，所以 handler（`AttachClockHandler` 的 `SetLow/SetHigh(clk)`、memory callback driving `_d[7:0]` 之類）一呼叫就立刻整片 settle。`StepCycle()` = `RunHandlerChain(); if(TraceLevel) CaptureTraceLine(); Time++;` —— recalc 全藏在 handler 的 `SetHigh/Low` 裡。`ProcessQueue` 末會 `InvokeCallbacks()`（memory/video 的 fake-transistor callback）。`SetNodeState(nn, v)`：寫 `NodeStates[nn]`、若 `nn` gate 了 transistor 就 enqueue 那些 channel-end。`RecalcNode(nn)` = `ComputeNodeGroup(nn)`（group-BFS + 256-LUT）→ 對 group 每個 member `SetNodeState`。

**新增需求**：要讓 IR「驅動」，就不能讓 handler 觸發整片 settle。→ 加一個 `WireCore.DeferRecalc` 旗標：set 時，`RecalcNodeList(nn)` / `SetHigh/Low/Float` 只做 `EnqueueNode(nn)`（設好 flag）、**不**呼叫 `ProcessQueue`。這是對 S1 唯一的（小、向後相容的）改動 —— `DeferRecalc==false`（預設）時 S1 行為完全不變。

**`IrEngine.Build()` 加（driving 模式需要的）**：
1. **「現值依賴圖」**：對每個 non-hybrid `v`，`NextExpr[v]` 裡每個 `NodeRefExpr(M)` → 邊 `v→M`（`HoldExpr(M)`/`PrevExpr(M)` **不算邊** —— 那讀 `prevStates` 快照）。hybrid node 不在圖裡（引用 hybrid 的*現值* OK —— bridge 後 `NodeStates` 會有它）。
2. **Tarjan 找殘餘 SCC**（size>1，或 size-1 但 `NextExpr[v]` 引用 `NodeRef(v)`）—— S2.3 沒打斷的。**第一刀全標 hybrid**（`NextExpr[v]=null; Hybrid[v]=true`；不做 Stage D）。記下殘餘 SCC 的數量/大小（希望少、小）。
3. **拓樸排序剩下的 non-hybrid node**（依現值依賴圖，「先算被依賴的、後算依賴別人的」）→ `evalOrder[]`（int[]）。
4. **boundary node 清單**：handler 驅動的 input-ish node（`clk`、`res`、memory callback 的 `_d[7:0]`、…）—— driving 模式要在 IR-eval 前先把它們的 `NodeStates` 更新好（見下）。第一刀：「`Role==Input` 或被 handler/callback 寫過的 node」（`HandlerWrittenNodes` 已有）。

**`IrEngine.StepOne()`（driving 模式）**：
```
1. prevStates ← NodeStates                          // Hold/Prev 讀這個（half-cycle 初快照）
2. WireCore.DeferRecalc = true; RunHandlerChain(); WireCore.DeferRecalc = false;
   // handler 的 SetHigh/Low 只設 flag + enqueue（沒 ProcessQueue）→ 此時 NodeStates 對 clk/db/… 還是舊值，但 recalc queue 裡有它們
3. 把 boundary node 的 NodeStates 更新好：對 queue 裡（或 boundary list 裡）的每個 input node n，RecalcNode(n)（= ComputeNodeGroup → SetNodeState）
   —— input node 的 group 通常就 {n} 自己（只 gate transistor、無 channel），所以這只更新 NodeStates[n] + enqueue n 的 gate-fanout neighbor（進 next queue，先不處理）
4. foreach v in evalOrder: NodeStates[v] = EvalExpr(NextExpr[v])   // 一遍拓樸序算 non-hybrid node（讀現值 NodeStates + 上一拍 prevStates）；現在 clk 等 boundary 已是新值
   —— 同時：值有變的 v → EnqueueNode(v)（讓 bridge 從它傳播到相鄰 hybrid node）
5. foreach hybrid h: EnqueueNode(h)                  // 讓 bridge 重算 hybrid node
6. WireCore.ProcessQueue();                          // S1 的 group-BFS：re-derive non-hybrid（=IR 算的值 → SetNodeState 無變化 → 不傳播 → 收斂快）；compute hybrid（依賴 IR-eval'd + boundary node；hybrid 的 hold 靠 S1 「浮島→max-cap node 現值=prevStates」自然處理）；末 InvokeCallbacks() = memory/video
   —— log：統計 ProcessQueue 改了幾個 *non-hybrid* node 的值（少 = IR 邊界準；極大 = 記下留 S3）
7. if (TraceLevel != 0) WireCore.CaptureTraceLine();  Time++;
```

**為什麼一遍夠（不用 IR↔ProcessQueue 迭代）**：non-hybrid 之間拓樸排好 → 一遍精確。non-hybrid 引用 hybrid 的*現值* —— IR-eval 時讀的是 hybrid 上一拍值（step 1 之後沒人改 hybrid 的 NodeStates），然後 step 6 ProcessQueue 算 hybrid → 若 hybrid 變了 → 傳播到相鄰的那個 non-hybrid node → ProcessQueue 用 group-BFS re-derive 它（不是 IR）→ 變對。所以「IR node 緊鄰 hybrid node」的會被 ProcessQueue 用 switch-level 重算（沒享受 IR 加速）—— 可接受（Gemini firing-1 已確認「IR predicts, switch-level corrects」、「絕對不要 IR↔ProcessQueue 迭代」）。

**`--engine ir`**：`TestRunner` 的 `Step`/`RunFrame`/`--trace`/`--test`/`--benchmark` 處 dispatch：`--engine ir` → `IrEngine.StepOne()`（driving）取代 `WireCore.Step(1)`。預設 / `--engine interp` = S1。**checking 模式保留**（`--trace-cmp` 不變 —— 它是 S2.6 gate，已過）。

**S2.5 = 上面 step 5-6 的 bridge**（hybrid node 跑 S1 的 ProcessQueue，邊界守在 latch 上 —— 因為大部分 hybrid 是 bus，I/O 多半是前一個 half-cycle 已穩的 latch 訊號 → 一遍夠）。**S2.6（driving 版）**：`--trace-cmp --engine ir` —— 跑 IR-driven N 拍、跟 S1-driven 比每拍 observable node（應該全等，因 checking 模式已證 `EvalExpr(NextExpr[v]) == S1 settled value`）。

### ② 開放問題（丟 Gemini）

1. **boundary node 的處理（step 2-3）**：`DeferRecalc` 模式下 handler 的 `SetHigh(clk)` 只設 flag + enqueue、不寫 `NodeStates[clk]`。IR-eval（step 4）需要 `NodeStates[clk]` 是新值。我的方案：step 3 對 queue 裡的 input node 各做一次 `RecalcNode(n)`（更新它的 `NodeStates` + enqueue 它的 fanout，但不處理 fanout）。問題：(a) 怎麼乾淨地「只處理一層」？`ProcessQueue` 是 loop 到收斂的；我要一個「processOneLevel」（grab 當前 queue、對每個 node `RecalcNode`、不 loop）。直接 inline 一段就好，還是有更乾淨的作法？(b) 有沒有 input node 的 group **不是** `{n}` 自己（例如 `res` 連到別的東西）？那 `RecalcNode(n)` 會 enqueue 一串 → 還是「只一層」就好（剩下交給 step 6）。(c) memory callback driving `_d[7:0]` —— callback 是在 `InvokeCallbacks()`（ProcessQueue 末）呼叫的、不是 `RunHandlerChain` 裡。所以 step 2 的 `RunHandlerChain` 只跑 clock toggle + nes-system handler。memory callback 要等 step 6 的 `ProcessQueue` 末才跑（那時 `cpu.ab`/`cpu.rw` 已被 IR-eval 算好）→ callback 讀 ab/rw、driving `_d[7:0]`（= 一個 hybrid bus，因為 `RefineBuses` 把 `_d[]` 標 Bus 了）→ 但 callback driving 完又 enqueue → ProcessQueue 繼續跑 → OK。這樣對嗎？memory callback 在 step 6 ProcessQueue **內**跑（不是之後）、它驅動的 `_d[]` 是 hybrid → bridge 自然處理 —— 我理解對嗎？

2. **`prevStates` 快照時機（step 1 在 `RunHandlerChain` 前）**：checking 模式是 `prevStates ← NodeStates; WireCore.Step(1)`（即快照在 handler 前）→ 所以 `Hold(M)` = handler 跑之前的值（= 上一拍 settle 完的值）。driving 模式我也 step 1 先快照。但 `Hold(clk)`（若有人引用）= 舊 clock 值（toggle 前）—— 應該沒人引用 `Hold(clk)`，但確認一下：6502 的 dynamic latch `Mux(we, data, Hold(self))` 的 `we` 是 clock 衍生的 —— `Hold(self)` 是 latch node 自己上一拍的值（latch 不會因 clock toggle 就變），所以快照在 handler 前後對 `Hold(self)` 沒差。對嗎？

3. **`ProcessQueue` 對 non-hybrid node 的 re-derive**：step 6 ProcessQueue 會對被 enqueue 的 non-hybrid node 跑 `RecalcNode` = `ComputeNodeGroup` → 理論上算出跟 IR 一樣的值（checking 模式已證）→ `SetNodeState` 無變化 → 不傳播。但 `ComputeNodeGroup` 是 group-BFS（會把整個 transistor group 走一遍）—— 對「一個 non-hybrid node 緊鄰一堆 non-hybrid node」的情況，這個 group 可能很大 → 一次 ProcessQueue 還是走了很多 node（雖然不傳播）。S2.4 第一刀不在乎（求對）；S3 才優化「讓 ProcessQueue 只碰 hybrid 區」（怎麼做？—— 例如 enqueue 時跳過 non-hybrid node、只 enqueue「IR 改了值 *且* gate 了某個 hybrid-相關 transistor」的；或建一個「hybrid 區的 transistor 子集」給一個瘦身版 ProcessQueue）。先確認 S2.4 第一刀「ProcessQueue 全跑」是對的、不會無窮 / 不會把 IR 算對的覆寫錯？

5. **`--engine ir` 的整體驗證**：driving 模式跑起來後，`--trace-cmp --engine ir`（IR-driven vs S1-driven 比每拍 observable node）應該全等（因 checking 模式已證）—— 但 bridge 的 ProcessQueue 對 hybrid node 的算法跟 S1 一樣（就是 S1 的 ProcessQueue）→ hybrid node 也應該全等。所以理論上 driving 模式 0 mismatch。對嗎？還有什麼會讓 driving ≠ checking 的（除了 bug）？例如：ProcessQueue 對 non-hybrid node 的 re-derive 順序跟 IR 的拓樸序不同 → 但都收斂到 fixpoint → 同值。對嗎？

### ④ 實作筆記（firing 9 —— driving 模式基礎建設：`DeferRecalc` + 依賴圖 + Tarjan + topo sort）

實作了 driving 模式需要的**基礎建設**（`StepOne` driving 版 + `--engine ir` + 驗收 = firing 10）：

1. **`WireCore.DeferRecalc` flag**（`WireCore.Recalc.cs`）—— set 時 `RecalcNodeList` / `SetHigh/Low/Float` 只 `EnqueueNode`、不 `ProcessQueue`（呼叫者自己 own settle）。預設 false → S1 行為完全不變。加 `ProcessQueueOneLevel()`（= `ProcessQueue` 的 while-loop 一次迭代 —— 處理當前 queue 裡的 node、re-derive 每個、把 fanout enqueue 進下一層；不 loop、不 `InvokeCallbacks`）—— driving 模式用它把 handler-chain 的 boundary 變動（clk…）flush 進 `NodeStates`。`EnqueueNode` / `ProcessQueue` / `RecalcNode` 改 `internal`（給 `IrEngine` 用）。
2. **`IrEngine.Build()` 加 `BuildEvalOrder()`**：
   - **現值依賴圖**：對每個 `NextExpr[v]!=null` 的 v，`CollectNodeRefs(NextExpr[v])`（只收 `NodeRefExpr.Id` —— `Hold`/`Prev` 不算邊）→ `deps[v]`（IR-covered 的）+ `dependents[M]`。
   - **Tarjan SCC**（遞迴；DFS spanning-tree 深度 = 閘深度，幾十；路徑在 hybrid/Input/序向 leaf 終止 → 遞迴安全）→ size>1 或 self-loop 的 SCC → 標 `InScc[c]=true`（**不**改 `NextExpr[c]` —— checking 模式仍用它、deps-on-it 讀 S1 的值）。log SCC 數量/大小。
   - **Kahn topo sort** 剩下的「Drivable」node（`NextExpr!=null && !InScc`，且只 count Drivable 的 dep）→ `EvalOrder[]`（deps 先）。`DrivingCoveredCount = EvalOrder.Length`。
   - `CheckInChecking` / `IrCoveredCount` 仍基於 `NextExpr!=null`（不受 InScc 影響）→ checking 模式覆蓋率不變（85%）。
   - 加 `--diag-node` 已有；無新 CLI（driving 的 `--engine ir` 是 firing 10）。
3. **不碰 S1 執行期**（`DeferRecalc` 預設 off；`IrEngine.Build` 只在分析 / `--trace-cmp` 用）。
