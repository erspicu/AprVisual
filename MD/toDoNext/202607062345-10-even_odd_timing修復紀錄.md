## 原始問題

乾淨基準在 K=1 對齊下:

```text
FAIL(3) @ frame 75
output: 08 07
```

ROM 預期輸出:

```text
08 08 09 07
```

失敗點代表 `$2001` 開啟背景渲染相對 PPU odd-frame dot-339 skip decision 太晚/太早錯位 1 dot。前一輪廣域 `--ppu-write-delay N` 掃描曾把每個 `bkg_enable/spr_enable` 轉態都延遲,結果 N={6,8,10,12,14} 全部 TIMEOUT,原因是 `10-even_odd_timing` 本身用 cycle-precise 的多次 `$2001` 寫入做量測,廣域延遲會破壞測試流程。

## 診斷重點

1. PPUSim 仲裁顯示 PPU 內部 dot 行為一致:
   - VBL set 在 `(vpos=241,hpos=1)`
   - odd-frame skip decision 在 pre-render line dot 339
   - 因此不是 PPU 內部相位錯,而是 CPU->PPU `$2001` 寫入效果路徑被理想化為 0 delay。

2. `--probe-2001` 顯示關鍵轉態分裂:
   - A=4 enable case 在 `vpos=261,hpos=337`,原本已正確,不能延遲。
   - A=5 enable case 在 `vpos=261,hpos=338`,需要延遲。
   - disable 側也需要同樣窄窗,但 falling edge 的夾法不同。

3. falling edge 不能用 `InstClampHigh(bkg_enable)`:
   - `InstClampHigh` 只是加 PWR 類驅動。
   - 若 netlist 同時有 active low/GND 類驅動,GND priority 仍會贏。
   - 所以 old=1 要保持 high 時,改成夾互補節點 `ppu./bkg_enable` 或 `ppu./spr_enable` 為 low。

## 修復方式

修改位置:

- `src/AprVisual.S1/Sim/WireCore.System.cs`
- `tools/testrom/run_tests.py`
- `tools/testrom/catalog.json`
- `tools/testrom/gen_catalog.py`

### WireCore shim

`--ppu-write-delay N` 仍維持 opt-in,預設關閉。修復後不再延遲所有 `$2001` 轉態,只在 skip decision 的 late side window 生效:

```text
vpos == 261
hpos <= 339
transition dot >= 338
```

邏輯:

- old=0,new=1:夾主節點 low,保持 disabled 狀態 N half-cycles。
- old=1,new=0:夾互補節點 low,等效保持 enabled 狀態 N half-cycles。
- `10-even_odd_timing` 實測採用 `N=16`。

核心條件:

```csharp
if (v != 261 || h > 339) return false;
if (oldValue == 0 && newValue != 0) return h >= 338;  // late enable
if (oldValue != 0 && newValue == 0) return h >= 338;  // late disable
```

互補節點:

```text
ppu./bkg_enable
ppu./spr_enable
```

這是本次能修掉 disable case 的關鍵;先前只夾 high 會讓 A=5 disable 仍失敗。

### Runner/catalog 接入

在 catalog 加入 per-ROM 欄位:

```json
"ppuWriteDelay": 16
```

`run_tests.py` 讀到該欄位時自動加:

```text
--ppu-write-delay 16
```

同時把該 ROM 的 `typicalFrames` 從失敗時的 75 改成 147,避免 runner 的 `1.5x typicalFrames` 快砍上限在 PASS verdict 前把測試中止。

`gen_catalog.py` 也同步加入同樣規則,避免重建 catalog 時設定消失。
