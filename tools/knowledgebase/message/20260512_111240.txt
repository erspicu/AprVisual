# S2.3 — 回授 / SCC 分析（S2.3a 帶條件依賴圖 + Tarjan；S2.3b 打斷假 SCC + cross-coupled latch 回收 → 序向）

## ① 設計描述

### 1.1 S2.3a — 帶條件的依賴圖（Conditional Dependency Graph）+ Tarjan SCC

**為什麼不建純靜態圖**（Gemini 的 S2-總覽 review 已採用）：在 NMOS 網表上對純靜態圖跑 Tarjan 會得到「跨多個 latch」的巨大無意義 SCC —— 因為 pass transistor 在硬體上常雙向 / 串成環。要建**帶條件的依賴圖**：

- **node 集合**：所有 `nextExpr[N] != null` 的 node（含 §1.2 回收進來的 cross-coupled latch）。hybrid 的（回收不了的）不進圖（它們走 switch-level）。
- **邊 + 條件**：`nextExpr[N]` 裡每個 `NodeRef(M)` → 一條 `N → M` 的邊，**邊上帶這個引用的「啟用條件」**（在 `nextExpr` 的 AST 裡，這個 `NodeRef(M)` 出現在哪個 `Mux` 的哪一分支下 → 條件 = 那些 Mux cond 的 AND/它們的 Not 的組合；最外層無條件 → `True`）。`Hold(M)` / `Prev(M)` **不產生組合邊**（那是「上一輪的值」，跨 cycle）。
- **Tarjan SCC**（對組合邊；忽略邊條件先找 SCC，再用條件判哪些是假環）。size-1 無自環 → 不算 SCC（純組合 node）；size-1 有自環（`nextExpr[N]` 引用 `NodeRef(N)` —— 不是 `Hold(N)` 的話 → 真的自己引用自己 → 怪，可能是 ratioed feedback → 標 hybrid 或當 latch）；size≥2 → SCC。

### 1.2 cross-coupled latch 回收（從 hybrid → 序向）

S2.1 把「bidirectional pass between two driven nodes」標 hybrid。其中最常見的是 **2-node cross-coupled latch + write pass**：node `Q` 有下拉（條件 `pd_Q`，通常 = `NodeRef(/Q)` 或一個含 `/Q` 的式子）、static pull-up、一個 pass 到 `data`（gated by `we`）；對稱地 `/Q` 有下拉 `pd_/Q`（含 `Q`）、static pull-up（可能沒有 write pass，或有第二個 pass）。

**辨識 pattern**（保守 —— 認不出的維持 hybrid）：
- `Q` 是 hybrid、`HybridReason` 含 "bidirectional pass between two driven nodes"；
- `Q` 的 hybrid 來源 pass 連到 `data`、gated by `we`；`data` 是「有 driver 的真 node」（`HasDriver(di[data])`，非 hybrid 或就算 hybrid 也有值）；
- `Q` 有下拉 `pd_Q`，且 `pd_Q` 引用某個 node `R`（= `/Q` 的角色），而 `R` 的下拉 `pd_R` 引用 `Q` → 確認是 2-node cross-couple；
- `Q` 有 static pull-up（`PullUpKind.StaticLoad`/`StrongVcc`）—— 沒有的話可能是 dynamic latch，另一種 pattern。

**合成**（把 `Q` 當「state node」—— 它有 write port）：
- `nextExpr[Q] = Mux(we, Node(data), Not(prev_pd_Q))`，其中 `prev_pd_Q` = `pd_Q` 但裡面的 `NodeRef(R)`（= `/Q`）換成 `Prev(R)`（打斷環）。化簡：若 `pd_Q == NodeRef(R)`（最常見：Q 的下拉就是 /Q 一個 gate）→ `Not(Prev(R)) = Prev(Q)`（因 `R = !Q` 恆成立）→ **`nextExpr[Q] = Mux(we, Node(data), Prev(Q))`** = 標準 D-latch。
- `R`（= `/Q`）：它沒 write port → 當組合，跟原本 S2.2 給的一樣（`nextExpr[R] = Not(pd_R)`，`pd_R` 引用 `Node(Q)` —— 讀 Q 的現值）。但 `R` 在 S2.1 也可能被標 hybrid（如果 `R` 也有個 bidirectional pass）—— 那 `R` 也要回收（對稱）。
- `Q` 標序向（`IsSequential[Q] = true`）；`R` 標組合。
- 把 `Q` 從 hybrid 名單拿掉（`di[Q].Hybrid = false`）；`nextExpr[Q]` 設成上面的。

**其他 pattern**（先列、firing 看狀況做）：
- **dynamic latch with feedback**（沒 static pull-up，但有個 feedback inverter 維持值 + write pass）—— 類似，`nextExpr[Q] = Mux(we, Node(data), Prev(Q))`。
- **set/reset latch**（有兩個 input pass，set 和 reset）—— `nextExpr[Q] = Mux(set, True, Mux(reset, False, Prev(Q)))` 之類。
- **多 write port**（一個 reg bit 被多個來源寫，各自的 we）—— `Mux(we1, data1, Mux(we2, data2, Prev(Q)))`（one-hot 假設，跟 S2.2 一樣）。
- 認不出的 / >2-node 強耦合 / bus-style 回授 → **維持 hybrid**，記 reason。

### 1.3 S2.3b — 打斷剩下的真 SCC

回收完 cross-coupled latch 後，重建依賴圖，再 Tarjan。剩下的 SCC：
- **cross-phase pipeline 假環**：環上有一條邊的條件需要 `clk0`（或含 `clk0` 的 AND）、另一條需要 `clk1`，而 `clk0`/`clk1` 互斥 → 這個環在邏輯上不存在（它是個跨 phase 的 pipeline，不是同一拍的組合回授）→ **打斷**：把環上「在 phase 邊界更新」的那些 node 標序向，環內引用換 `Prev(...)`。判定「`clk0`/`clk1` 互斥」：先找出 `clk0`/`clk1`（或它們的對應 node）—— 從 nodenames 或 from 「`clk1 = !clk0`」這種結構推；保守起見也可以只認「直接是 `clk0` 或 `Not(clk0)` / `clk1`」的條件。
- **同 phase async loop（RS latch、真回授）**：環裡沒有 clock 互斥 → 真的是個 latch / 回授結構。若能認出「state node + set/reset/data input」→ 改寫成序向（同 §1.2 的精神）；認不出 → **維持 hybrid**（整個 SCC 的 node 都標 hybrid）。
- **bus-style 回授 / >2 強耦合**（典型在 2C02 的 OAM bus / sprite bus）→ 維持 hybrid（沒 strength 維度，bus-fight 解不了）。

### 1. 6502/2A03/2C02 的 Register Bit 結構與寫入 (對應 Q1, Q2)

**事實與知識：**
6502 與 2C02（NMOS）的靜態暫存器（如 A, X, Y, PC 等）**絕大多數就是依靠 Ratioed Logic (比例邏輯) 來寫入的 cross-coupled inverter pair**。
它們沒有所謂的 break-before-make（切斷回授再寫入）複雜結構，而是簡單粗暴的：**寫入端（Pass Transistor + 驅動源）的尺寸/驅動力被設計得比回授 Inverter 的 Pull-up/Pull-down 大得多**。當 `we` 打開時，外部訊號直接「蓋過」回授迴路的弱驅動，強行改變狀態。

**對這個計畫的具體影響 / 該怎麼改：**
*   **合成模型 `Mux` 完全正確：** 你提出的 `nextExpr[Q] = Mux(we, Node(data), Prev(Q))` 是對 Ratioed Write 最完美的高階邏輯抽象。在邏輯層面，它等同於「如果寫入致能，不管現值為何都覆蓋；否則保持上一拍的值」。**你不需在 IR 中引入強度（Strength）維度**，強度衝突在硬體上是靠物理尺寸解決，在 IR 裡靠 `Mux` 的分支優先級解決即可。
*   **如何挑選 State Node：** 「有 write pass 連接的那個 node」就是 State Node (Master)。另一個只是組合邏輯的 `Not(State)`。你的 Pattern 辨識條件非常穩健，請直接採用。
*   **2C02 的特例：** 2C02 的 VRAM-addr (v/t) 和 Sprite Shift Register 有很多是 **Dynamic Latch**（沒有 static pull-up，只靠寄生電容 hold 值，再過一個 Inverter 輸出）。對於這種，S2.2 可能根本沒有標為 Hybrid（如果只有單向 Pass 寫入），只會產生 `Hold(Q)`。若它們被誤判為 Hybrid，用你計畫中的 Dynamic Latch with feedback pattern 抽出來即可。

### 2. S2.3 該做到哪個程度？ (對應 Q3)

**事實與知識：**
S2 的核心目標是「大幅度提高 IR 覆蓋率以換取效能」，而不是「100% 抹除 Hybrid」。5534 個 bidirectional pass 佔了 Hybrid node 的 95% 以上。

**對這個計畫的具體影響 / 該怎麼改：**
*   **先只做最簡單的 2-node cross-coupled + write pass**。把這幾千個 node 處理掉，覆蓋率就能從 60% 衝到 85~90% 以上。
*   **暫緩 SR Latch、Multi-write 與 Dynamic Latch**。SR Latch 在 6502 中通常用於中斷觸發器 (NMI/IRQ) 等控制邏輯，數量不多（可能幾十個）。Multi-write 的暫存器如果不符合單純的 2-node 結構，留在 S2.3a 維持 Hybrid 跑 S1 switch-level 模擬，對整體效能的拖累微乎其微。
*   **效益評估：** 做到 2-node 回收後，直接跑 `--dump-scc`。如果 IR 覆蓋率 >85%，直接收工進 S2.4。不要在這裡過度陷入 corner case 的泥淖。

### 3. Cross-phase pipeline 假環的判定與時鐘識別 (對應 Q4)

**事實與知識：**
6502 的兩相時鐘 (phi1, phi2) 是由外部輸入的 `CLK0` 經過內部分相電路產生的**非重疊 (Non-overlapping) 時鐘**。這意味著在邏輯上 `phi1 AND phi2 == False` 永遠成立。如果一個環的前半段需要 `phi1` 打開，後半段需要 `phi2` 打開，這個環在同一個時間切片 (Tick) 內是不通的，這就是標準的跨相管線，本質是序向，不是組合回授。

**對這個計畫的具體影響 / 該怎麼改：**
*   **不要用語法結構硬推 Mutex：** 從 AST 裡推導 `Not(clk0)` 很脆弱，因為中間可能過了好幾個 gate。
*   **作法：設定「Mutex Pair 名單」**。在你的系統配置 (或 `NetlistGraph` 的某個 Configuration) 中，直接**寫死輸入或關鍵 Clock Node 的 ID / Name**。例如：已知 6502 的 node `1234` 是 phi1，node `5678` 是 phi2。
*   **打斷演算法：** 當 Tarjan 找出一個 SCC 後，走訪環上所有邊的啟用條件 (Condition)。如果邊 A 的條件包含 `NodeRef(phi1)`，邊 B 的條件包含 `NodeRef(phi2)`，且 `(phi1, phi2)` 在你的 Mutex 名單中，則判定為假環。
*   **打斷點選擇：** 隨便挑環上一顆「由時鐘 gated」的 Node，將其引用改為 `Prev(...)`，標記為序向即可。

### 4. Bus-style / 大於 2 的強耦合 SCC (對應 Q5)

**事實與知識：**
2C02 的 OAM Bus 和 Sprite 匯流排是靠多個驅動器進行 Wired-OR / Wired-AND 或具有複雜 Priority 結構的總線。如果沒有精細的強度模型與總線衝突解決機制，硬轉布林邏輯極易出錯（例如 sprite zero hit 的 timing 很容易爛掉）。

**對這個計畫的具體影響 / 該怎麼改：**
*   **絕對維持 Hybrid。** 完全不要試圖在 S2.3 把 2C02 的 OAM bus 抽成 IR。S1 (WireCore) 裡有完整的群組解析與 GND/VCC/Pull-up 優先鏈，這對於處理這種 bus fight 是完美的。這正是 S2 架構保留 Hybrid Fallback 的最大價值。
*   一旦 Tarjan 找出的 SCC 不是你認識的 latch pattern，**直接將該 SCC 內的所有 Node 標記回 Hybrid**。

### 5. 整體演算法流程的微調 (對應 Q6)

你設計的流程（先建圖 -> Tarjan -> 條件分析 -> Latch 回收 -> 打斷）邏輯上可行，但**順序微調一下會更乾淨、效能更好**：

**建議的 S2.3 執行順序：**

1.  **階段 A：Pattern 靜態回收 (S2.3a 預處理)**
    *   在建圖**之前**，直接遍歷所有的 Node。
    *   利用 §1.2 的 Heuristic 找出 `CrossCoupledLatch`。
    *   直接修改其 `NextExpr` 為 `Mux(we, data, Prev(Q))`，將對應 Node `IsSequential = true`，並從 `Hybrid` 集合中剔除。
    *   *理由：* 這些 latch 的結構極度固定（就是兩個 Node 互相依賴），不需要動用 Tarjan 就能找出。提早回收可以大幅減少建圖與 Tarjan 的節點數。
2.  **階段 B：建帶條件的依賴圖 (Conditional Dependency Graph)**
    *   對剩下的 (包含非 Hybrid 與剛轉為序向的 Node) 建立有向圖。注意：`Prev(X)` **不產生邊**。
3.  **階段 C：Tarjan 找 SCC**
    *   這時候找到的環，就是「階段 A 漏掉的未知結構」或「Cross-phase 假環」。
4.  **階段 D：SCC 條件分析與打斷 (S2.3b)**
    *   檢查每個 SCC 的邊條件。若符合 Clock Mutex 規則（phi1/phi2 互斥），使用 `Prev()` 打斷，標記被切斷的節點為序向。
5.  **階段 E：無解 SCC 退回 Hybrid**
    *   剩下的 SCC（既不是 Mutex 假環，也解不開），代表這是複雜組合回授或 Bus 結構。
    *   **將該 SCC 內的所有節點 `Hybrid = true`**，把它們丟回給 S1 的 WireCore 處理。

### 總結

你的設計方向非常務實且精準。請大膽實作 **2-node CrossCoupledLatch Pattern 匹配 + `Mux(we, data, Prev(Q))`**。這一步做完，NES 模擬的 IR 化就已經跨過最難的門檻，後續 S2.4 的直譯器效能將會有爆發性的成長。不用擔心漏掉少數複雜的 Latch，退回 Hybrid 是極其安全的容錯機制。
