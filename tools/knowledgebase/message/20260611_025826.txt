# THE DESIGN (call it "class-major renumbering with self-captured first-touch locality")

The engine's per-node hot data is SoA arrays indexed by node id (a 16-byte NodeInfo, six 1-byte arrays). Node ids were historically the physical-layout order from the netlist files. We now renumber ALL node ids at load, with a two-part sort key:

1. PRIMARY KEY = static "prune class" (4 classes derived from each node's electrical structure: can-skip-on-turn-off x is-turn-on-unsafe). Sorting class-major makes each class ONE CONTIGUOUS ID RANGE, so the hottest loop's per-endpoint mask lookups (PruneMask[c] & bit — an L1-resident but DEPENDENT byte load feeding the enqueue decision chain) become register compares against 3 boundary constants (skip <=> c < S; unsafe <=> c < A || c >= B). The mask array is still computed at every Reset as GROUND TRUTH and every node's bits are verified against the range-implied bits before the first settle; on mismatch the engine falls back to safe-degenerate boundaries (prunes disabled, still correct). This part alone measured +3.6% (the engine is dependent-load-chain bound, NOT cache-miss bound — we measured that the hot set is cache-resident and that pure locality plays, prefetch, and footprint shrinks are all neutral).

2. SECONDARY KEY (within each class block) = the TRUE FIRST-TOUCH ORDER of the production settle cascade, SELF-CAPTURED at load: LoadSystem runs three passes. Pass 0 builds with identity ids just to classify the prune bits. Pass 1 rebuilds class-major with a temporary static key — its ranges verify, so the event-count prunes are ON — then warms the chip past reset (~1K half-cycles) and runs ~32K half-cycles through a COLD COPY of the settle loop that additionally records each node's first pop (the hot ProcessQueue is untouched; the copy is only used here). The captured order is translated back to identity ids through pass 1's permutation. Pass 2 does the final rebuild with class bits + captured order. Total load cost ~1.3 s; the whole rebuild is possible because the workload is a deterministic simulation and a rebuild is cheap. Correctness is layout-independent by construction: the only id-order-dependent site (the power-on recompute sweep) and the state checksum both iterate in ORIGINAL id order through the permutation, so all golden checksums are unchanged.

*   **The Paradigm:** What you are doing is a variant of the **Inspector-Executor model** originally developed for irregular applications (sparse matrices, unstructured meshes). An "inspector" pass determines memory access patterns at runtime, and the "executor" pass runs the optimized schedule.
*   **Key Citations:**
    *   *Saltz et al., 1991* ("A manual for the CHAOS runtime library"). The foundational work on Inspector-Executor.
    *   *Ding & Kennedy, LCPC 1999* ("Improving cache performance in dynamic applications through data and computation reorganization"). Discusses reorganizing arrays at runtime based on traversal patterns.
    *   *Strout et al., PLDI 2003* ("Compile-time composition of run-time data and iteration reorderings").
*   *Why your online capture beat offline:* You proved the **Stale Profile Problem** (classic PGO vs. JIT dynamic profiling). Offline profiles suffer from workload drift. By doing it online dynamically, you captured the exact initialization-state specific trace.

**B. Graph Vertex Reordering (Execution-Trace vs. Structure)**
*   **The Paradigm:** Structural sorts (RCM, Gorder, Rabbit) fail here because switch-level networks (bidirectional pass transistors) do not map topologically to execution flow. You reinvented **Trace-Driven Memory Layout** for graph nodes. 
*   **Key Citations:**

*   **The Paradigm:** In academia, this is known as **Index Space Partitioning**, **Predicate-Based Array Partitioning**, or **Enum Partitioning**. You are encoding static type properties into the memory address / index bounds to eliminate a dependent load and a branch.
*   **Key Citations:**

### 4. Microarchitectural Explanation for Finding (b)

*Finding: Production-schedule ordered layout (+6.2%) beat non-production layout (+0%) even at identical cache-line density (105.9 vs 110 lines/half-cycle).*

This is a textbook demonstration of **Temporal vs. Spatial Locality in Hardware Prefetching and MLP (Memory-Level Parallelism)**.

1.  **Hardware Prefetcher Streaming:** Modern L1/L2 prefetchers (like Intel's Spatial/Stream prefetchers) do not merely look at *which* lines are accessed (density); they heavily rely on the **stride and monotonic sequence** of accesses. If your layout mimics the exact production schedule, the event queue accesses memory in a sequentially ascending order. The prefetcher recognizes this `$PC` + stride pattern and pre-loads the dependent data. The non-production order had the same lines hot, but likely accessed them in a scattered pseudo-random sequence within that half-cycle, breaking the prefetcher's confidence.
2.  **Rob (Reorder Buffer) Utilization:** Even if data is L1-resident, out-of-order execution engines process dependent chains faster if the loads don't conflict or alias. Sequential access patterns maximize the efficiency of the Load/Store Queue (LSQ) and allow the CPU to maintain high MLP.
3.  **Cache Line Contention / Thrashing:** In the non-production layout, even if the density per *half-cycle* is the same, the density per *micro-window* (e.g., a 50-instruction window) might be worse. If the production layout packs events that happen *immediately* after one another into the same cache line, a single fetch serves 4-5 events consecutively. The non-production layout might require bouncing between 4 different hot cache lines back-and-forth within the same half-cycle, increasing L1 cache port pressure.

*Relevant Literature:* Search for **"Spatial vs Temporal Data Locality"** and **"Hardware Prefetcher Training via Data Layout"**. (e.g., *Somogyi et al., ISCA 2006, "Spatial Memory Streaming"*).
