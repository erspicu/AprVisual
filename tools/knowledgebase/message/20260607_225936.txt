### 1. 預先計算的靜態結構 (Static Structure)

在載入階段，我們不再讓時脈（`clk`）依賴通用的 `NodeTlistGates[clk]` 陣列。為了配合 P-1 ~ P-4 的運行時剪枝，我們必須將 `clk` 扇出（Fanout）展開為**兩個完全展平的陣列**（分別對應 Rising 和 Falling 邊緣）。

你需要將每個受影響的端點（$c1$ 和 $c2$）獨立打包，只保留剪枝所需的最少資料，以達到最大的快取密度：

```c
// Precomputed at load time
struct ClockFanoutEndpoint {
    NodeId target;      // The node to conditionally enqueue
    NodeId peer;        // The opposite side of the transistor (needed for P-1..P-4)
    uint8_t prune_mask; // Precompiled rules: is this NMOS/PMOS? Cap-guarded? etc.
};

// Two linear, flat arrays
ClockFanoutEndpoint ClkRisingList[NUM_RISING_ENDPOINTS];
ClockFanoutEndpoint ClkFallingList[NUM_FALLING_ENDPOINTS];
```
*注意：一個電晶體會產生兩個 Endpoint 條目（分別檢查 $c1$ 和 $c2$），這能避免在迴圈內做 `if (target == c1)` 的分支判斷。*

### 2. 取代 `StepCycle` 的具體 Pseudocode

我們將跳過 `EnqueueNode(clk)`，直接強行修改 `clk` 的狀態，並直接掃描預先編譯好的陣列來把資料節點推入佇列。

```c
StepCycle():
    // 1. Toggle Clock directly (Bypass enqueue & ProcessQueue pop)
    bool is_rising = !NodeStates[clk];
    NodeStates[clk] = is_rising; // Set raw state immediately
    
    // 2. Iterate flat array based on edge
    if (is_rising) {
        ProcessClockEdge(ClkRisingList, NUM_RISING_ENDPOINTS);
    } else {
        ProcessClockEdge(ClkFallingList, NUM_FALLING_ENDPOINTS);
    }

    // 3. Normal settle for the rest of the chip
    ProcessQueue();


// The dedicated loop for Clock Bypass
function ProcessClockEdge(ClockFanoutEndpoint[] list, int count):
    for i = 0 to count - 1:
        endpoint = list[i]
        
        // --- RUNTIME PRUNING (MUST still be done) ---
        state_target = NodeStates[endpoint.target]
        state_peer   = NodeStates[endpoint.peer]
        
        // Evaluate precompiled P-1..P-4 rules using current state
        if CheckStaticPruneRules(endpoint.prune_mask, state_target, state_peer):
            continue // Pruned!
            
        // If it survived pruning, enqueue it for Wave-1
        EnqueueNode(endpoint.target)
```

### 4. 效能量化 (Quantify: Cache Win vs Event Reduction)

**回答：這純粹是一個 Data-Oriented Design (DOD) 的記憶體快取最佳化 (Cache-friendly win)，不是算法上的事件縮減。**

*   **複雜度：** 原本是 $O(N)$，現在依然是 $O(N)$ （$N$ 為 `clk` 控制的電晶體數量）。
*   **效能差異：**
    *   原本：讀取 `Transistor` 陣列（可能有 16 bytes），提取 $c1, c2$，根據類型判斷導通狀態，然後執行 P-1~P-4。
    *   現在：讀取緊湊的 `ClockFanoutEndpoint` 陣列（8 bytes），循序存取 (Sequential Access)，完美命中 CPU Prefetcher，並且移除了分支（NMOS/PMOS 的邊緣差異已經在編譯期被分開到兩個陣列了）。
*   **真實比例估算：** 在一個已經實作 P-1 到 P-4 的 NES 引擎中，這個改動頂多讓你每個半週期省下幾萬個 CPU 指令。對於 3MHz 的模擬器，**這大約只能帶來 +1% 到 +3% 的淨效能提升**。如果你期待的是突破性的 >10% 提升，這個方案**做不到**。

### 5. 時脈子樹的回應可以被記憶化 (Memoization) 嗎？

**回答：絕對不行（Dead on arrival）。**

*   **位元精確 (Bit-exact) 是否允許？** 理論上可以，前提是你的 Cache Key 包含所有依賴狀態。
*   **為什麼不可行？** 時脈子樹下一層的反應，完全取決於下游節點（Data lines, registers, buses）**當前**的狀態組合。
    在 NES 運行時，下游狀態的變化是極度混亂且不可預測的（取決於 CPU 指令、PPU 掃描線、匯流排競爭等）。這意味著：
    1. 你無法預期相同的下游狀態會頻繁出現。
    2. 要構建一個 Cache Key 來代表「當前下游狀態」，你必須去讀取那 $N$ 個端點的 `NodeStates`。
    3. **致命傷：** 「讀取這 $N$ 個端點狀態來計算 Cache Key」的記憶體存取成本，**已經完全等同於**「直接執行一次 P-1~P-4 剪枝」的成本！
*   如同你自己剛測試過的 dominant driver 快取（成本 -15.4%），為了維護或比對這種動態狀態，額外的讀寫開銷會徹底吃掉任何可能省下的計算時間。
