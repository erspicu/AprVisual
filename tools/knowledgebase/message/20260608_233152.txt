Here is the blunt, concrete evaluation of your engine and the proposed static precomputations.

### a) STATIC "Potential Group" Precomputation
**Bit-exact?** Yes.
**Will it help?** **NO. (Expected win: 0%)**

**The structural reason:** You noted that 94% of the netlist is one giant bidirectional pass-transistor SCC. In NMOS designs like the 6502/2C02, almost everything eventually connects to the main internal data/address buses via pass transistors. If you ignore gate states (the ON subset), the "potential group" for any node on or adjacent to a bus is effectively the entire 94% SCC. 
Because the static superset is massive, it provides no useful bounded subset. You cannot use it to size local arrays, skip dedup (`_inGroup`), or shortcut the BFS. The runtime ON-subset is so aggressively pruned by gate states (down to your measured 1.4 nodes) that the static topology is useless for bounding the traversal. 

### b) CACHE-LAYOUT Precomputation (Node Renumbering)
**Bit-exact?** Yes.
**Will it help?** **YES. (Expected win: 5% - 12%)**

**The structural reason:** You are memory-latency bound, and `NodeInfo` is 16 bytes (4 per cache line). When Node A’s BFS walks to Node B, or Node A enqueues Node C, fetching the downstream node incurs an L2/L3 penalty if it's not in the same cache line. You want nodes that frequently communicate to live in the same 64-byte chunk.
*However, do not use standard RCM.* RCM is for minimizing matrix bandwidth in linear solvers; it has no concept of dynamic execution frequency. 

**The Ordering Heuristic:** You need **Profile-Guided Heavy-Edge Matching**. 
1. Instrument your emulator to dump a trace of Super Mario Bros for a few frames. 
2. Record a weighted graph of every time Node A reads Node B’s state (channel walk) AND every time Node A enqueues Node B (gate trigger). 
3. Feed this weighted graph into a static partitioner (like METIS) or a greedy clustering algorithm to pack the heaviest 4-node cliques into contiguous IDs. 
Since the average BFS is only 1.4 nodes, the *enqueue* edge (gate -> source/drain) is actually where most of your random memory jumps happen. Order for the dynamic hot paths, not the static topology.

### c) OTHER Static Precomputations
Here are two specific precomputations/structural changes that target your exact bottleneck. 

#### 1. Static Array Packing / Bit-packing (The Smoking Gun)
**Bit-exact?** Yes.
**Expected win: 15% - 25%**

You stated: *L1d is ~32KB and already contended by NodeStates + RecalcHash x2 + _inGroup + PruneMask (each ~14.7KB).*
This is your actual memory-latency floor. Fetching data for a single node requires hitting up to 5 separate cache lines. Because their combined size (~73KB) exceeds L1d, you are guaranteeing L1 thrashing on every random node jump. 
**The fix:** Statically bit-pack all 5 arrays into a **single 14.7KB `byte[]` array**. 
*   Bit 0-1: State (0/1/Z)
*   Bit 2: _inGroup
*   Bit 3: RecalcHash0
*   Bit 4: RecalcHash1
*   Bit 5: PruneMask
By doing this, a node's entire volatile state fits in **one byte**. The entire 14.7K node state array now permanently locks into L1d, leaving the rest of the L1d cache entirely free for your `NodeInfo` structs. (To clear `_inGroup`, do not memset the array; since your walk is ~1.4 nodes, simply unset the bit on the visited nodes as you finish the BFS).

#### 2. Static Sub-Graph Isomorphism (Standard Cell Merging)
**Bit-exact?** Yes.
**Expected win: 8% - 15%**

You lowered always-on transistors, but the netlist is still full of standard NMOS logic structures (e.g., a pull-up to PWR and pull-down to GND via pass transistors forming an inverter or NAND). 
Write a static load-time pass to detect these rigid standard-cell subgraphs and replace them with a single custom "Macro Node." 
*Why it works here:* It statically eliminates internal nodes and their associated event queue overhead. If a 3-node inverter becomes 1 node, you've removed two nodes from `NodeStates` and two potential enqueues. 
*To keep it bit-exact:* You only need the whole-chip checksum to match. When the macro node calculates its state, have it update a dummy array with the deterministic states of the "hidden" internal nodes so the FNV checksum remains identical.

### The Blunt Truth
If you implement profile-guided renumbering and pack your 5 parallel arrays into a single L1-resident array, you will hit the absolute structural floor of scalar event-driven simulation. Because of the dynamic nature of Bryant relaxation on a 94% SCC, **no static topological precomputation can bypass the event queue or the need to evaluate the ON-subset dynamically.** If you need 390x to reach real-time, no amount of node-level tweaking will get you there; but the cache and layout fixes above will comfortably squeeze out the remaining 20-30% left in this engine.
