## The hot loop and the EXACT dependent-load chain (the bottleneck)
ProcessQueue pops node id `nn` from a work-list, then RecalcNode(nn) does:
  (1) load NodeInfo[nn]            -- SoA record: this node's transistor-list offset+count, pull flags, prune mask   [addr depends on nn]

Here is a skeptical, concrete, architecture-level breakdown of how to squeeze out that final 15-20% backend stall. 

You have correctly identified that you are bound by **L1d load-to-use latency**, not bandwidth or cache misses. Your L1d miss rate is 1.13%. **The data is already in L1.** The Cortex-A76 takes ~4 cycles to resolve an L1 load. Here is the exact verdict on your questions.

### 1. DATA LAYOUT: Refuting the "Fat Node" (Denormalization)
**Do NOT denormalize gate states.** It will wreck your performance, and your own telemetry already proves it. 
*   **The Math:** You noted that the `SetNodeState` write path is ~10x hotter than the fast-path resolution. If you denormalize (cache a gate's state into the `NodeInfo` of every node it connects to), you are shifting the cost from the read-side to the write-side. 
*   **The Cost:** To eliminate 1 scattered read (4 cycles), every time a gate flips, you must walk its fan-out and issue scattered *writes*. Writes dirty the cache lines and consume store-buffer slots. A clock line flipping will suddenly trigger thousands of scattered L1 writes.
*   **Verdict:** Denormalization is logically identical to the "active-ON-gate counters" you already tried, just using bits instead of integers. Since the counter approach was a net -6% regression, denormalizing the states will yield the exact same regression. Avoid.

### 2. ORDERING: Refuting RCM for Latency
**RCM (or any spatial bandwidth-minimization ordering) will do absolutely nothing for a latency problem.**
*   **Why?** RCM clusters non-zeros to minimize the maximum topological distance. In hardware terms, it maximizes *spatial locality* (packing neighbors into the same 64-byte cache line). This prevents L1 capacity/conflict misses.
*   **Your reality:** Your `NodeStates` array is ~25KB (1 byte * 25k nodes). The Cortex-A76 has a 64KB L1d. **Your entire hot state already fits in L1**, which is why your miss rate is a microscopic 1.13%. 
*   **The hard truth:** An L1 hit takes 4 cycles whether the byte is at offset 0x10 or 0x20 in the cache line. Clustering addresses won't make the L1 hit any faster. Your pop-order capture (+6%) worked because it improved *temporal* locality within the CPU's Out-of-Order window, but further static DAG clustering is a dead end.

### 3. MLP: Software Pipelining Across Pops
**Why did `__builtin_prefetch` fail?** Because the A76 has a 128-entry Reorder Buffer (ROB). The hardware Out-of-Order engine is *already* seeing the next work-list pop, computing the address, and fetching `NodeInfo[queue[head+1]]` while you stall on the current node's gates. The hardware is already perfectly pipelining the independent loads.

**How to actually get cross-pop MLP:** You must interleave the *dependent* loads via **Speculative Interleaved Execution**. Since 70% of your pops are size-1 (no state mutation, no BFS), you can process them in pairs with zero risk to bit-exactness.

```c
// Take two independent nodes from the queue
int n1 = queue[head];
int n2 = queue[head+1];

// Issue both dependent chains explicitly in registers
NodeInfo i1 = NodeInfos[n1];
NodeInfo i2 = NodeInfos[n2];

// Extract their gates and load states
// (See Q4 below for how to do this branchlessly)
bool n1_is_size1 = check_size_1(i1);
bool n2_is_size1 = check_size_1(i2);

if (n1_is_size1 && n2_is_size1) {
    // BOTH are provably no-ops. Commit both instantly.
    // Bit-exactness is guaranteed because neither mutated state.
    head += 2; 
    continue;
}
// Fallback: If either requires BFS, process n1 normally, then n2.
```
This overlaps the `NodeStates[gate]` latency of `n1` with `n2`. If both are size-1 (which mathematically happens 0.70 * 0.70 = 49% of the time), you halve the load-to-use stall for those pops.

### 4. THE 70% SIZE-1 CASE: The Branchless Parallel Load (Highest EV)
This is where your serialization bottleneck actually lives. Currently, your transistor walk creates a serial control dependency: `load gate -> test -> branch (or load next)`.

**To break the dependent-load chain, decouple the "Is it size-1?" check from the BFS execution, and force the loads to issue in parallel.**

The A76 has two independent load units; it can issue two L1 loads per cycle. But a `for` loop over transistors blocks this.
1.  **Pad the layout:** Ensure node 0 is a dummy node where `NodeStates[0] == 0` ALWAYS.
2.  **Pack the inline gates:** Since 25k nodes fit in a `uint16_t`, and most nodes have $\le 4$ transistors, pack exactly four 16-bit gate IDs into a single 64-bit integer inside `NodeInfo`. Unused slots are set to `0` (the dummy node).
3.  **Branchless OR check:** 

```c
// Inside NodeInfo: uint64_t packed_gates; 
uint64_t pg = info.packed_gates; // 4x 16-bit gate IDs

// Unroll to force the compiler to issue scalar L1 loads in parallel
uint8_t s0 = NodeStates[(uint16_t)(pg)];
uint8_t s1 = NodeStates[(uint16_t)(pg >> 16)];
uint8_t s2 = NodeStates[(uint16_t)(pg >> 32)];
uint8_t s3 = NodeStates[(uint16_t)(pg >> 48)];

// Single bitwise OR. 
// s0|s1|s2|s3 is 0 IF AND ONLY IF all incident transistors are OFF.
if ((s0 | s1 | s2 | s3) == 0) {
    // 70% of the time, you hit this fast-path.
    // You completed 4 gate checks in ~5 cycles total, instead of 16.
    return; 
}

// Fallback to the standard BFS traversal only if something is ON
```
By making the loads unconditional and branchless, the A76 will issue `s0/s1` in cycle 1, and `s2/s3` in cycle 2. You collapse a sequential pointer-chasing loop into a parallel block. This is 100% bit-exact.
