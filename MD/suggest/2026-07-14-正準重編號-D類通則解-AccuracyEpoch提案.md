# 正準重編號:D 類的通則解 —— Accuracy Epoch 核心提案

**日期**:2026-07-14
**性質**:正式設計提案(未排程;排 Accuracy Epoch,掛牌 bank 之後)
**一句話**:把節點編號從「載入順序的任意標籤」換成「載入期按物理正準鍵排定的
確定性標籤」,讓引擎既有的 id-順序機制自動變成正準仲裁 —— 熱路徑零改動、
零成本,D 類樂透從根消滅。

---

## 1. 問題回顧:D 類的兩條 id 依賴通道

1. **浮接群平手判定**:電容代理(現=連接數)同分 → 編號決勝;
2. **同波事件串行化(主犯)**:settle 波內「同時」事件的處理順序 = 佇列順序 =
   編號順序;真矽用連續時間物理裁決的競速,引擎用編號裁決。
   鐵證:u7/u8('368 孿生)一對一錯,只因編號不同(OpenBus err6)。

譜系:Visual6502 浮接判定 = 走訪順序第一個 state=true(最粗);MetalNES/S1 =
連接數代理;本提案 = 分層加權面積 + 結構特徵(見 §3)。

## 2. 核心機制:載入期正準重編號(不是跑時排序)

引擎所有 id-順序依賴吃的是**節點編號本身**。因此不在熱路徑加任何比較 ——
**載入期重編號一次**,現有機制原封不動地變成正準仲裁:

- 與既有 class-major renumber(+3.56% locality win)合成:**桶間順序不動**
  (保留 locality 分桶),**桶內排序鍵**從「原始 id」換成「正準鍵」;
- 熱路徑差異 = 0 條指令;成本全部在載入期(一次排序)。

### 兩個獨立收益

1. **孿生必同命**:同構電路 → 同正準鍵排序 → 同判決;u7/u8 荒謬根除;
2. **圖變更不再重擲全域樂透**:編號由結構+物理決定,不再依賴載入/檔案列順序
   → 補一顆管只擾動局部 → **Socket Pattern 的「載入期圖變更=大爆炸」問題
   大幅消解**,未來 netlist patch、模組置換、儀器掛載全部變安全操作。
   (這個操作面收益可能比 D 類本身更值錢。)

## 3. 正準鍵的構成(草案)

`key = (class, layeredArea, structHash, degree, ...)` 字典序:

1. **class**:既有 class-major 桶(不動,保 locality);
2. **layeredArea**:分層加權面積(tools/netlist/node_area.py 已驗證資料管線)
   `w_metal·A_metal + w_poly·A_poly + w_diff·A_diff`,權重初值可用製程單位面積
   電容比,方向(升/降序)用硬體答案卷校準;
3. **structHash**:節點鄰域的結構雜湊(gate 數、tlist 組成、鄰居 class 序列)——
   面積同分時的結構性次鍵,保證孿生一致;
4. **degree / 原始 id**:最終決勝(此時已是真·同構,選誰都一致)。

物理方向假設:面積→電容→速度(小電容先settle)→「小鍵先處理」近似
「快節點先到」。這是啟發式,不是保證 —— 見 §5 誠實界線。

## 4. 校準與驗證閘

1. **答案卷校準**:AC 141 顆(尤其 D 類受害者:OpenBus err6 家族、DL 站點)
   作為鍵方向/權重的校準集;自由度極低(全域標量,非 per-site),過擬合可控;
2. **孿生一致性測試**:自動枚舉同構站點對(u7/u8、joy 通道、db 位元),驗證
   判決一致 —— 這是 D 類根除的直接量測;
3. **效能閘**:桶內重排動 locality → interleaved-paired 20 輪(Zen2 + Pi5),
   要求不輸基線;
4. **checksum 重定基準**:必變(仲裁全面換鍵),與 Accuracy Epoch 其他項
   (強度 LUT、E 類補丁常開、閂鎖自動重述)打包一次重驗:AC 全套 + testrom
   全家族 + 遊戲畫面 CRC 語料庫 + AprNes 差分。

## 5. 誠實界線

- **一致 ≠ 正確**:正準化根除「樂透性」(任意性、重擲性),但「同命判決是否
  符合矽」取決於鍵與真實物理的相關性。單一全域靜態全序是連續時間的粗模型:
  理論上可能存在兩站點同結構類別、卻需要相反先後(RC 差在面積抓不到的維度:
  路徑電阻、驅動強度)。殘餘偏離從「抽籤」降格為「可校準的已知量」;
- 需要窗口語意(A2)或管線延遲(A4)的行為,重編號不觸及 —— 那是
  timing-calibration 資料檔的領域。

## 6. 在「零手寫特例」全圖中的位置

| 缺陷類 | 通則機制 | 手寫特例? |
|---|---|---|
| D 樂透 | **本提案:正準重編號** | 免 |
| A1 比例打架 | 強度 LUT(同一份面積資料餵) | 免(LAE/LXA 自然解出) |
| A3 透明閂鎖 | 自動偵測+重述(結構衍生) | 免手寫 |
| B 板卡/類比 | Sidecar 時戳資料檔 | 免手寫 |
| E 資料缺陷 | 網表補丁(t13032b 已實證) | 免 |
| A2/A4 連續時間 | 機制通則化 + 校準常數資料化 | ⚠ 唯一殘餘;無靜態等價物 |

終態:**沒有一行認站點的代碼**;shim 清單收斂為「自動衍生機制 + 三個資料檔
(netlist-patch / analog-sidecar / timing-calibrations)」。

## 7. 實作順序建議(Epoch 內)

1. 正準鍵計算器(離線工具,先產報告:孿生對 + 鍵分佈 + 同分率);
2. renumber 桶內換鍵(引擎旗標,預設 OFF)→ 孿生一致性測試 + 效能閘;
3. 鍵方向/權重校準(AC 答案卷迴圈);
4. 與 Epoch 其他項打包,重定基準,全量重驗,轉正。
