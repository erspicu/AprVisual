# Fast-path 家族再評估:B1 成對路徑採用(+~0.6%),其餘候選全數封死(2026-06-12)

> 起因:使用者要求對靜態 fast-path(cls1)與 R-1 動態 fast-path(cls2)再評估演算法擴展空間。
> 方法:3 代理平行(實測分布 / 擴展設計 / 死路對抗複核),全數據實測。
> 結果:**一個活候選(B1)→ 實作 → 採用(commit 2749105)**;其餘全滅、附量測數字存檔。

## 起點分布(main,full_palette 100k hc,Debug 計數 = Release 等價)

- 節點:cls1 = 3,929(26.7%)、cls2 = 10,784(73.2%)、cls0 ≈ 16 個(callback 假目標 6 + FC 8 + …)
- pops = 42.85M;**fast 解析 60.4% / BFS 走訪 39.5%(16.94M)**
- **BFS 群組大小:size-2 佔 77.1%(= 30.5% of ALL pops)**、size-3 16%、size-4+ 5.7%;深度 mean 1.13
- no-change 80.1%(PullUp 42.0% + Supply 38.1% 結構殘餘,無靜態子集 —— 三度確認)

## B1 成對路徑(採用)

**機制**:cls2 派發掃描遇到第一顆 ON 閘時,先證明群組恰為 {nn, o} 再就地解析,跳過
_groupBuf/_inGroup 整套機制。回退條件(全部在任何寫入之前):自迴路 o==nn、種子有第二顆 ON 閘、
o 是 overflow 或帶 HasCallback/FC、o 有通往第三節點的 ON 通道。
**逐位元等價義務**(對照 BFS 逐項覆製):RecalcHash[o]=0(成員清除,漏了就發散)、
flags = 兩節點 Flags | 兩邊供電閘 OR-all、浮接 tie-break = 嚴格大電容勝/平手種子勝
(精確等於 GetNodeValue 對 [nn,o])、寫回順序 seed→o(= 下一波入列順序 = Gauss-Seidel 語意)。

**驗證**:Debug golden @100k、Release 三黃金(300k/400k/1M)、selftest、SMB1 1000 萬 hc checksum 全過;
pops 數完全不變(42,848,807 —— 純同義重寫)。實際吃下 **8.34M pops(19.5%)**(保守回退吃掉其餘)。

**量測**(interleaved-paired 20 輪 ×4 批):未鎖頻兩批 +0.84%(14/20)/ −1.10%(7/20)—— 兩批基準中位差 2%,
熱漂移主導、不可信;**鎖頻(base 3.6GHz)兩批:+0.53%(13/20)/ +0.70%(12/20),合計 25/40** → 採用。
教訓重申:當天環境噪音大時,sub-1% 改動只有鎖頻批次可信(WebSite 量測注意事項早已寫明,這次親身重演)。

**Rust 移植(cae1dab):+14.45% 中位數、20/20、鎖頻 cv ~0.3% —— 巨勝。**
同一個演算法:C# +0.6% vs Rust **+14.5%**,專案史上最大的跨引擎**倍率**不對稱(24×,非正負翻轉)。
原因:Rust 的 add_node_to_group 是**遞迴呼叫**且無 inline-payload 拆分,19.5% 的成對 pop
原本每次都付一整條遞迴呼叫鏈;C# 的 BFS 早已迭代化+全內聯,所以只剩簿記可省。
Rust 新峰值 best-of-5 **118,259 hc/s**(舊最佳 103,120,排行榜 id 31 → 待更新)。
golden:0x9174E19D961CB6E5 @400k + 0x6D4CCBCE2E9CD599 @1M ✓。
這個案例反向印證了 [[jit-vs-llvm-recursive-inline]]:.NET JIT 拒絕內聯遞迴(所以 C# 早被迫迭代化、
反而把錢提前賺完),Rust/LLVM 留著遞迴(錢一直在桌上,B1 一次掃走)。

## 同輪殺掉的候選(連數字一起存,防止重提)

| 候選 | 實測母體 | 死因 |
|---|---|---|
| cls3 callback 單例 fast-path | 6 節點、67,968 pops = **0.16%** | 結構上可行(已驗證假電晶體都在 c1gnd 桶),但母體死;且其 pop 100% no-change(值恆 0,唯一作用是 EnqueueCallback) |
| FC 節點進 cls2 | 8 節點、101,120 pops = **0.24%** | LUT 已含 FC 互消、位元等價可行,但 8 顆全是 overflow 匯流排節點,通道常 ON → 多半照樣落 BFS |
| 入列時提示位(hash 帶 turn-on/off 來源) | — | **分析性死刑**:提示在有利方向(走快路)不可信 —— 同波後續 pop 可在凍結提示後把閘打開,過期的「快」提示=算錯;安全方向只會多走 BFS。維護真值 = counter-fastpath(−6.0%)老路 |
| BFS 落入時重用部分掃描 | mean first-ON index < 2 | 省的是剛摸過的 L1 熱行重讀、餵預測分支 —— cls-range 同型(空氣) |
| epoch 式 hash 清除 | — | 8-bit epoch 每 ~21 次 ProcessQueue 繞回 → 假「已入列」→ 發散;省的又是非鏈上 store |
| 分類別佇列 / 波內重排 / 批次 pop | — | 波內 Gauss-Seidel 順序是語意;且 cls 載入餵預測分支、本來就不在鏈上(cls-range 先例) |
| 波內群組快取 / memo | — | 群組成員是 live 狀態的函數(深度 p50=1,無可攤提的重複大走訪);有效性追蹤 = P-5 維護地板 |
| 無分支入列(容忍重複項) | — | 入列分支 97.7% 單向(已測)= 預測掉的分支無利可圖;且爆 RecalcListNext 容量保證 |

## 家族總結

fast-path 家族在 B1 之後**正式收斂**:60.4% 的 pops 已經在 ~5 載入的最短相依鏈上;
走訪 pops 唯一夠大的子集(size-2,30.5%)已由 B1 接手其可證明的部分;cls0 全部僅 0.40%;
剩餘時間 = 已最小化主體上的相依鏈延遲 + 80.1% 結構性 no-change 殘餘(無靜態子集)。
**三條件檢查表**(靜態 + 熱 + 在被攜帶的相依鏈上)維持為本引擎所有新提案的守門標準。
