## 1. 最重要的知識:FAIL 的三分類法

拿真晶片 netlist 跑 test ROM 還會錯,原因只有三類(已寫進報告頁科普章節):

1. **行為層整合缺口** —— 兩顆晶粒以外的一切(RAM/卡帶/時脈/手把/板線)都是
   行為層重建;錯的通常不是「值」而是「哪個瞬間、浮接與否、殘值多久」。→ 修。
2. **機型差異** —— NES 是家族不是一台機器;測試 ROM 校準自作者那台。→ 對照
   釘死的機型基準據實處理。
3. **數位抽象極限(類比)** —— 開關級 = 數位抽象;真晶片是類比 NMOS。
   衰減、上電隨機、**時序賽跑**、**比例式匯流排打架**都會被二值量化改寫。
   → 若真機行為確定(G 版次共識):文件化 shim 補正;若真機本身就依機而異:
   忠實偏差 + 證據卷宗。

## 2. 引擎語意極限(第 3 類的物理學 —— 本專案最深的新知)

quiescent-settle 二值開關模型(每半週期解到穩態)在四個地方和真矽晶分道:

### 2.1 同半週期賽跑,極性依實例而定(不存在全域規則)

| 案例 | 賽跑 | 真矽晶裁決 | 二值模型裁決 |
|---|---|---|---|
| DMC pcm_latch(t14402)| 資料下降沿 vs apu_clk1 關門沿 | **資料贏**(時脈衰減期 pass gate 仍導通)| 門先關 → 少捕捉一拍 |
| ALU 輸入閂鎖(alua/alub)| 匯流排崩塌 vs SBADD/DBADD 關門 | **門贏**(hold time 滿足)| 崩塌穿門 → 閂鎖被汙染 |

- 關鍵:穩態語意下「最終 gate=0 → 斷」,波內順序不存在 → 同步賽跑必然
  單向裁決;真矽晶靠類比傳播深度逐實例決定。
- **佐證力最強的一發**:emu-russia APUSim(同晶粒獨立電路逆向)在 DMC 案例
  以同樣方式失敗(讀 $10;我們 $00;真機 $80)—— 兩個獨立矽晶模型同敗,
  行為層模擬器(TriCNES)建構上通過 → 證明是抽象極限,不是誰的 bug。
- 工程形式:**帶極性的文件化閂鎖 shim 表**(edge-capture 模式 / hold 模式)。

### 2.2 比例式(ratioed)匯流排打架被 GND-wins 量化

- LXA($AB)的 magic 常數 = AC 弱驅動 vs 資料閂鎖驅動在合併 SB/IDB 線上的
  類比強度賽;真機依晶片/溫度得 $EE/$FF(NESdev + TriCNES 註解),
  G 版次共識 $FF;我們的 LUT(GND 先贏)判成 $00。
- 有趣對照:**ANC 的 A&imm 也是同一種 bus 合併湧現的**(wired-AND),
  而那個結果和真機一致 —— 同機制、不同節點,類比賽的贏家不同。

### 2.3 群組解析 LUT 優先序的操作性後果

`GND > VCC/pull-up > 外部驅動 > hold` —— 意味著**主動驅動的控制線用
SetLow 壓不住**(群組內 VCC 導通勝過外部 drive-low)。
→ shim 不能靠壓制控制線;要嘛作用在浮接/動態節點,要嘛持續驅動。

### 2.4 儲存元件的三種質地(shim 施力點選擇)

| 質地 | 例子 | 單發 force 是否黏住 | 正確手法 |
|---|---|---|---|
| pass-gate 動態閂鎖 | pcm_latch、alua/alub、A/X 暫存器 | ✅(門關後浮接保持)| drive→settle→release |
| 交叉耦合再生對 | palette cell、pcm_irq | ❌ 單側被彈回 | **雙側互補**驅動再 release |
| 主動再生迴路 | P 旗標(p1/p7 每 cycle 從上游重載)| ❌ 一個 cycle 內被沖 | **雙側成對 + 持續驅動跨數 cycle** 再 release(迴路各相位都捕捉新態後自持)|
| 閘級 TTL 閂鎖模組(pslatch 族)| nes-pad 的 CD4021 | ❌ **結構性不可驅動**:閂鎖 pass-gate 反向驅動輸入節點,群組內 GND 恆勝,「放開的按鍵」永遠寫不進去 | 整個模組**行為層化**(nes-pad-behavioral + Joypad handler;與卡帶同級)|

### 2.5 上電/對齊是離散抽籤

- CPU÷12 自上電自由運轉、PPU÷4 在 /res 釋放時重啟 → 相對相位只有
  **4 種**({1,7,5,3}),K=0..5 掃描證明偶數相位物理上不存在(K=2≡K=1、
  K=4≡K=3,除頻器以整 clk0 週期量化)。
- NMI-edge 家族(8 個)passes {7,5};10-even_odd passes {1,3} —— **完備枚舉
  下零交集(本模型)**。定格對齊 7(K=1);此零交集**現已在對齊 7 用窄窗寫延遲 shim 補平、even_odd 通過**(§3.1 #13),從源頭仲裁偏移仍列後續(見下段再定性)。
- **再定性(2026-07-05,Gemini 諮詢 q/a_even_odd_doctrine)**:真機公認存在
  golden alignment 單次上電十項全過(blargg 在真機開發驗證整套)—— 零交集
  **不是真機性質**,而是我們雙 netlist 板級整合的 **~1-dot 絕對相位偏移
  (未仲裁)**。候選源(依可能性):$2002 跨晶片讀取路徑被理想化為零延遲、
  reset 釋放啟動 PPU 除頻器的時序、時鐘 pad 緩衝延遲未建模。TriCNES 手調
  offset=7 全過,與 golden-alignment 共識互相印證(通過區間在真矽晶上有交集)。
  仲裁實驗:PPUSim 交叉比對 BIT $2002 的絕對 master-clock 延遲(/RD→D7)。
  相位空間模型本身(4 類、CPU÷12 自由運轉、PPU÷4 /res 重啟)與社群共識一致。

### 2.6 探針效應:載入期圖變更會重擲彩票(2026-07-05,Gemini 諮詢定案)

**實測事實**:為 double_2007 掛 9 個假節點/假電晶體(shim 全程**零開火**)後,
無關的 dma_2007_read @K=1 從 PASS 翻成真機不存在的第三圖樣(X=00)——
純因 class-major 重編號整體位移 → settle 波內部順序改變 → 「DMA halt vs read」
這場彩票類賽跑翻面。**任何載入期圖變更都會重擲所有同波賽跑類測試的骰子。**

**通則**(諮詢紀錄 `tools/knowledgebase/q/a_dbl2007_footprint_20260705.md`):
測試儀器需要的是「絕對覆蓋權」,不是圖變更 —— EDA 標準做法 = VPI 式 force/release。
per-test 開關被判定為過度擬合/技術債;代表物理傳播延遲的 shim 應**全域生效**。

**實作**:`InstClampLow`/`InstRelease`(WireCore.Recalc)—— runtime 把 `Gnd` 旗標
OR 進節點;群組解算等價於「接了一條 vss 導通路徑」(LUT 最高位階,壓得過主動驅動),
**零圖變更**,golden checksum A/B 逐位元不動。紀律:test-mode shim 限定、只用於無
靜態 Gnd 的節點、釋放責任在呼叫者(ResetNes 重建旗標,自然清殘留)。

**開火守門的三個誤傷教訓**(每一條都有實測屍體):
1. `/r2007` 低 ≠ CPU 在讀:解碼毛刺會拖長窗口 —— 對取指週期接地會把 opcode $8D 磨成 $8C(STY)。
2. DMA 停頓時 ab 停在 $2007 的殭屍週期 CPU 不取樣;重載落在**脈衝之間**也一樣 —— 真取樣在幾週期後,真機早傳播到位。
3. 重載落在 **φ1** 同理(距下個取樣沿超過半週期)。賽跑窗的正確定義=**重載落在同一個活躍 φ2 相位內**(實測落沿距離:micro dt=1、blargg@K=1 dt=7)。

### 3.4 基礎設施回歸:路徑推導出來的模擬組態(2026-07-09,已修)

**不是引擎 bug,是測試框架 bug** —— 但它一度讓**所有 class-A 測試**假性失敗,值得記進來。

- **症狀**:ROM 打包進 repo(`romBase` 從 `nes-test-roms-master/checked` 改成 `tools/testrom/roms`)後,
  4 個 apu_mixer 乃至任何 `$6000` 測試全部 `detection=none / timeout`。畫面判定(class B)不受影響。
- **根因**:`WireCore.LoadSystem()` 靠 **ROM 路徑字串**(`nes-test-roms` / `nes_test`)決定要不要載入
  `cart-extraram` —— blargg `$6000` 協定的 work RAM 就在那個模組裡。新路徑兩個關鍵字都不含 →
  RAM 沒掛 → 簽章區不存在 → runner 從頭到尾看不到 `$6000`。
- **最小證明**(同一顆 DLL、同一顆 ROM `11-special.nes`):bundled 路徑不加 `--extra-ram` → timeout;
  bundled 路徑加 `--extra-ram` → pass(f=11);舊路徑不加 → pass(f=11,因為命中 heuristic)。
- **修復**(commit `23ddd89`):test mode 在 `LoadSystem()` 前顯式 `WireCore.ForceExtraRam = true`
  (`Test/TestRunner.Test.cs`)。連 `dotnet ... --test <任意路徑>` 的直接用法也一併修到,
  且 benchmark 不走 `RunOneTest()`,golden checksum 不受污染(實測仍 `0x794A43ABDF169ADA`)。
- **不是**:幀預算太緊(`detection=none` 代表簽章從未出現)、ROM 損毀(md5 相同)、
  DLL 重建 / .NET preview 破壞 bit-exact(**golden checksum 全程未變**)。這三條當時都被錯誤懷疑過。

**通則(與 §2.6「別動被測物」同族)**:
> **凡是影響模擬組態的條件,一律由 CLI flag 或 test mode 顯式指定,絕不由資料夾/檔名推導。**

**防呆(已實作,commit `5a5a831`)**:`run_tests.py` 在 build 完、派工前會自動跑兩個 canary ——
300k golden checksum(`0x794A43ABDF169ADA`)與一顆短的 `$6000` 測試
(`nes_instr_test/rom_singles/11-special.nes`,期望 `detection=6000`,第 11 幀出判定);
**任一不符就 abort**,不讓 6–8 小時的回歸跑在壞掉的判定路徑上。兩項合計約 70 秒;
急著除錯時可用 `--no-canary` 略過。中止訊息刻意寫成純 ASCII(壞掉的 build 常在 cp950 主控台看)。

## 5. 事實對照法(方法論)

每個深度調查都遵循同一套「包圍」流程,已驗證有效:

1. **黑箱定位**:測試自己的輸出(失敗行/錯誤碼)+ 原始碼讀期望值
2. **micro-ROM 顯微**:自製最小重現,窮舉輸入,對照文件語意
3. **datapath 顯微**:op-probe/bus-trace 半週期級,先驗資料路再驗控制路
4. **外部權威仲裁**:NESdev wiki(100thCoin 版)、AccuracyCoin/TriCNES
   原始碼與註解、blargg readme/source、emu-russia breaks/APUSim
5. **獨立模型交叉**:APUSim 跑同場景 —— 同敗=抽象極限;不同敗=我們的 bug
6. **修復後三級驗證**:micro-ROM 全組合 → 目標 blargg ROM → 家族/全量回歸
