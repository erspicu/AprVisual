## firing 14 起：**S3.0 —— driving 模式 bridge 瘦身**（不要 re-derive IR-covered node）→ 把 ~8× 慢拉回 ~par 甚至更快

> firing 13 的 benchmark：driving 模式比 S1 慢 ~8×。瓶頸 = `StepOneDriving` 把值有變的 IR node `SetNodeState(v,nv)+EnqueueNode(v)` → gate fan-out（含一堆 IR-covered node）全 enqueue → step 5 的 `ProcessQueue` 對 queue 裡的東西全跑 `RecalcNode`=`ComputeNodeGroup`（group-BFS）→ 等於把 IR 已經算對的 node 再用 switch-level 重算一遍（無變化、不傳播，但 `ComputeNodeGroup` 的成本付了）。**S3.0 = 讓 ProcessQueue 跳過「IR 已算對、不需要它」的 node**。先設計 + 丟 Gemini（firing 14），再實作（firing 15+）。

### ① 設計描述

**核心**：`ProcessQueue` 在 driving 模式裡只該做兩件事 —— (a) 算 hybrid + InScc node（IR 沒模型化的），(b) 觸發 callback（memory / video）。對「IR-covered（`NextExpr != null && !InScc`）且沒有 callback flag」的 node `nn`，`RecalcNode(nn)` = `ComputeNodeGroup(nn)`（昂貴）→ 算出跟 IR 一樣的值 → `SetNodeState` no-op → 不傳播；唯一「有用」的副作用是 callback-enqueue，但 `nn` 沒 callback flag 所以也沒這個。→ **這種 node 在 ProcessQueue 裡可以直接 skip**（不 `ComputeNodeGroup`）。

**為什麼安全（callback 不會漏）**：memory callback 監看 `{cs, /we, addr[], dataBus[]}`（這些 node 有 `HasCallback` flag）。callback 在「某個含 callback-node 的 group 被 `RecalcNode`」時被 enqueue。一個 callback-node `c`（如 `addr_i` = `BA_i` = `u2.q_i`，74LS373 latch 輸出）值有變時：step 4 的 IR eval 算到它（它在 `EvalOrder`、`NextExpr` 是 Stage A2 給的）→ `SetNodeState(c, nv)` + （因為 `c` 有 callback flag）`EnqueueNode(c)` → `ProcessQueue` 的 `RecalcNode(c)` 跑（`c` 有 callback flag、不 skip）→ callback enqueue。✓ 所以 callback-node 自己**直接被 enqueue + recalc**（不依賴「它 group 裡的某個鄰居被 recalc 順便帶到它」）→ 即使 skip 掉那些鄰居（IR-covered no-callback node），callback 還是會 fire（addr 變 → callback fire；addr 沒變 → callback 不該重 fire，skip 也對；就算 S1 在「addr group 裡別的 node 變但 addr 沒變」時會冗餘 fire，少 fire 那次也無害 —— memory callback 重讀同位址 = no-op）。

**改動**：
1. `IrEngine.Build()` 算 `bool[] OkToSkipInRecalc` —— `v` 是 IR-covered（`NextExpr[v] != null && !InScc[v]`）**且** 沒有 `HasCallback` flag（`(WireCore.NodeInfos[v].Flags & HasCallback) == 0`）。
2. `StepOneDriving` step 4：`for v in EvalOrder: { nv = EvalExpr(NextExpr[v]); if (NodeStates[v] != nv) { WireCore.SetNodeState(v, nv); if (!OkToSkipInRecalc[v]) WireCore.EnqueueNode(v); } }` —— `SetNodeState` 寫值 + enqueue `v` 的 gate fan-out（hybrid 傳播需要、便宜）；只有「不該 skip 的」（= hybrid/InScc/有 callback 的，這在 `EvalOrder` 裡其實只有 callback node）才 `EnqueueNode(v)` 讓 `RecalcNode(v)` 跑。
3. `WireCore.ProcessQueue`：driving 模式時（一個新 flag `WireCore.SkipRecalcOf` = `IrEngine.OkToSkipInRecalc` 或 null）—— 處理 queue 裡的 `nn` 時若 `SkipRecalcOf?[nn] == true` → `RecalcHash[nn]=0; continue;`（跳過 `RecalcNode`）。`StepOneDriving` step 5 前設 `WireCore.SkipRecalcOf = OkToSkipInRecalc`，step 5 後設回 null（讓 S1 的正常 `Step` 路徑不受影響）。
4. 不碰 S1 的 `Step`/`RunFrame`/handler 邏輯（`SkipRecalcOf` 預設 null → ProcessQueue 行為不變）。

**驗證**：`--trace-cmp --engine ir`（driving）仍 **0 mismatch**（skip 的 node IR 已算對；callback 仍 fire）；`--benchmark --engine ir` —— 看快多少（預期 ~5-8×、回到 par 附近）；`--trace-cmp`（checking 模式）不受影響（這只改 driving 路徑）；`--selftest` 全 PASS；`--trace`（S1）跟前一個 commit 一致。

**開放問題（丟 Gemini）**：
1. 「skip IR-covered no-callback node 的 `RecalcNode`」—— callback 安全性的推理（callback-node 自己直接被 enqueue+recalc，所以 skip 鄰居無害；S1 的「冗餘 fire」少了也無害）對嗎？有沒有漏掉的情況（例如：某個 hybrid node `h` 的 group 裡有一個 IR-covered no-callback node `m` —— 我 skip 了 `m` 的 `RecalcNode`，但 `h` 的 `ComputeNodeGroup` 還是會走到 `m`（讀 `NodeStates[m]` = step 4 的 IR 值）→ 沒問題吧？skip 的是 `RecalcNode(m)`、不是「把 `m` 從 `h` 的 group 拿掉」）？
2. 還有沒有更省的（例如：根本不要 enqueue `v` 的 gate fan-out，改成 step 5 只 enqueue「值有變且 gates 到某個 hybrid/InScc-相關 transistor 的 IR node」+ 所有 callback node + 所有值會變的 hybrid node —— 但「值會變的 hybrid node」要先算才知道、雞生蛋）？還是「`SetNodeState` 的 gate-fanout enqueue 全留、`ProcessQueue` 跳過 IR-covered no-callback」這個 cut 就夠好（簡單、安全）？
3. 預期：這個 cut 後 driving 模式相對 S1 大概多快？瓶頸會移到哪（hybrid+InScc 的 `ComputeNodeGroup` ~6500 個/step？IR eval 的 Expr 樹遞迴 ~50K visit/step？）—— 下一步該往哪打（提升 IR coverage 減少 hybrid？攤平 IR eval？）？

每條建議標明「對這個計畫的具體影響 / 該怎麼改」。某設計 OK 不用改也直說。

### ② Gemini 分析回饋 + ③ 實作試做後 revert（S3.0，log `..._134915.txt`）

**實作試做後 revert**：照 Gemini 的設計做了 —— `IrEngine.OkToSkipInRecalc[]`（`NextExpr[v]!=null && !InScc[v]`）、`WireCore.SkipRecalcOf` flag + `ProcessQueue` 裡 `if (SkipRecalcOf?[nn]==true) continue;`、`StepOneDriving` step 4 改 `SetNodeState(v,nv)` only（拿掉 `EnqueueNode(v)`）+ step 前後設/清 `SkipRecalcOf`。但測下來 **driving 模式從 0 mismatch 變大量 mismatch**（CPU state 在 reset 序列就崩 —— `S=05` 而非 `FD`、`DB=00` 而非 `24`，= reset vector 從 `$FFFC` 讀不回來 → memory callback / data path 出問題）。root cause（推測）：skip `RecalcNode(v)` for IR-covered `v` —— 但 `RecalcNode(v)` 不只算 `v`、它 `ComputeNodeGroup(v)` 走整個 group + 對每個 group member `SetNodeState`；某些 IR-covered `v` 跟一個 hybrid node / callback-watched node / 板間 buffer 輸出在同一個 channel-connected group 裡，`RecalcNode(v)` 是把值傳播到那些 member 的途徑 —— skip 掉 → 那些 member 變 stale → memory callback 沒拿到正確 address / data path 斷。Gemini（跟我）的「`ComputeNodeGroup(h)` 自己會走到 `m`、`m` 不用 recalc」推理對 hybrid `h` 成立（`h` 不被 skip），但漏了「`v` 跟 callback-target / 跨晶片 buffer 輸出在同一 group、而那些不一定會因為別的東西被 enqueue→recalc」的情況。→ **revert 整個 S3.0 實作**（回 firing 13 的 `b65efd4` code state）。

**下一步修法**（未來 firing）：skip `RecalcNode(v)` 的判據要更保守 —— 只有當 `v` 的 *channel-graph connected component*（忽略 gate 條件、用 union-find 算的「v 透過 channel 可能連到的所有 node」）裡**沒有** hybrid node、**沒有** InScc node、**沒有** callback target node 時才能 skip（這保證 `v` 的 group 永遠是純 IR-covered，`RecalcNode(v)` 一定是冗餘）。component 一次算好（用 firing 1 lowering 已有的 union-find 思路、但這次是「所有 channel transistor 都當邊」的保守版）。代價：可 skip 的 node 可能只有一部分（buses / hybrid 區的 component 很大、把一票 IR node 拖下水）→ 加速沒原本預期的 ~8× 那麼多，但安全。或者：別動 `ProcessQueue`，改成 step 4 直接寫 `NodeStates[v]`（不 `SetNodeState`、不 enqueue），step 5 只 enqueue「值會變的 hybrid+InScc node」（這要先知道哪些會變——但其實 step 4 改了哪些 IR node 的值是知道的，那些 IR node 的 gate-fanout 裡的 hybrid/InScc/callback node 就是要 enqueue 的）—— 即「精準 enqueue」而非「全 enqueue 再 skip」。

### ④⑤ 實作筆記 + 驗證（firing 16 —— S3.1：flat IR eval（EvalOrder 編成 stack-machine instruction stream））

接 firing 15 的選項 (b) —— 不碰脆弱的 bridge，先做 IR eval 攤平（Gemini 說的「低垂果實」）。實作（全在 `IrEngine`、不碰 `WireCore` / S1）：

- **`CompileFlatProgram()`**（`Build()` 裡、`BuildEvalOrder()` 後）：對 `EvalOrder` 裡每個 `v`，post-order 走 `NextExpr[v]` 的 `Expr` 樹 → emit stack-machine 指令（op codes：`LoadNode(id)` / `LoadPrev(id)` / `Const0` / `Const1` / `Not` / `And` / `Or` / `Mux` / `StoreNode(id)`），最後 emit `StoreNode(v)`。整個 `EvalOrder` 編成一條大指令流（`byte[] _flatOp` + `int[] _flatArg`、~68K 指令 = ~6824 node × ~10 指令）+ 一個 eval stack（`byte[] _flatStk`，深度 = 最大 `Expr` 樹巢狀深度）。語意完全對齊 `EvalExpr`。
- **`RunFlatProgram()`**：一個 `for` 迴圈跑那 ~68K 指令、用 `_flatStk` 當 eval stack（`LoadNode` → `stk[sp++]=NodeStates[arg]`、`And` → `stk[sp-1]&=stk[--sp]`、`Mux` → pop b,a,c push `c!=0?a:b`、`StoreNode` → `nv=stk[--sp]; if(NodeStates[v]!=nv){SetNodeState(v,nv);EnqueueNode(v);}`）。cache-friendly（陣列循序存取）、無 virtual dispatch / pointer chasing。
- **`StepOneDriving` step 4**：`foreach v in EvalOrder: EvalExpr(NextExpr[v])`（遞迴）→ 換成 `RunFlatProgram()`（flat loop）。
- **checking 模式不變**（`StepOne` 還用 `EvalExpr` —— checking 比的是更多 node（含 InScc）、不在 perf 路徑上）。

**驗證**：`dotnet build` 0/0（Debug+Release）、`--selftest` 全 PASS、`--trace branch_timing/1 --cycles 8 --engine ir` 跟 S1 逐 cycle 一致、`--trace-cmp branch_timing/1 --cycles 2000`（checking）0 mismatch、**`--trace-cmp ... --engine ir`（driving）0 mismatch**（flat eval 語意正確）、`--trace branch_timing/1 --cycles 8`（S1）跟 commit `5d4c7e1` 一致（沒碰 S1）。

**benchmark**（`branch_timing/1` 2 frames = 1,369,455 half-cycles，Release）：
| engine | real time | vs S1 | 備註 |
|---|---|---|---|
| S1 switch-level | 29.7 s | 1× | |
| IR-driving（recursive `EvalExpr`，firing 13）| 232.7 s | ~8× 慢 | |
| **IR-driving（flat program，firing 16）** | **159.6 s** | **~5.4× 慢** | flat eval 給 **~1.46× 加速** |

→ flat IR eval（S3.1）給 driving 模式 ~1.46× 加速（8× 慢 → 5.4× 慢）—— IR eval 那塊變快了，但**剩下的瓶頸是 bridge ProcessQueue**（re-derive 一堆 IR node 的 `ComputeNodeGroup`），那需要 `SkipRecalcOf` 的修法（firing 14-15 試過會壞、要 instrument 才能 debug）。方向對、還沒到「比 S1 快」。

> **下個 firing**：(a) 給 driving 模式加 lockstep-diff instrument → debug `SkipRecalcOf` 為何壞 → 修好它（這是把 5.4× 慢拉回 ~par 甚至更快的關鍵刀）；或 (b) Stage B/D 提升 IR coverage（46.3% → 更高 → bridge 自然變輕）；或 (c) flat program 進一步優化（const-fold、CSE、把 `byte` stack 換成 register-allocated 的 SSA temps 之類）。

### ④⑤ 實作筆記 + 驗證（firing 17 —— S3.0 修法三（instrument → diagnose → channel-component + dep-closure 雙檢查）—— 修對了，但只 184 個 skippable）

接 firing 15 的計畫：先 instrument。加：(1) `WireCore.SkipRecalcOf` flag + `ProcessQueue` 裡 `if(SkipRecalcOf?[nn]==true) continue;`（inert when null）；(2) `IrEngine.OkToSkipInRecalc[]` + `DebugSkipRecalc` flag + `--debug-skip-recalc` CLI；(3) `TraceCmpDrive` 在 mismatch 時印第一個發散 node 的細節（`ir`/`s1`/`prevSelf` 值 + `nextExpr` + 它引用的 node 在 ir/prev/s1 的值 + 哪些被 `[skip]`）。

**diagnose**（`--trace-cmp --engine ir --debug-skip-recalc --cycles 30`，先用 firing-14 的「`OkToSkip = NextExpr!=null && !InScc`」）：第一個發散 = `t=2, BA8#18 ir=1 s1=0`，`nextExpr = !n8442 & (hold(n18) | n1535)`，`refs: 8442#8442=ir:0(prev 0,s1:1)[skip]  ...`。→ **root cause：`8442` 在 IR-eval 時用了一個 hybrid node 的舊值（`8442` 的 `NextExpr` 透過某個 gate 引用了一個 hybrid node `h`；`h` 是 ProcessQueue 在 step 4 之後才算的、所以 step 4 算 `8442` 時讀到 `h` 的舊值）→ `8442` 的 IR 值「暫時錯」，需要 `RecalcNode(8442)`（在 ProcessQueue 裡、等 `h` 被算對之後、`h` 變動傳播到 `8442`）來修正 → 但 `8442` 被 `[skip]` → 沒修 → `8442` 保持錯 → 傳播到 `BA8` 等 → 整個崩**。firing 15 的「channel-component all-IR」檢查漏了這個 —— `h` 透過 *gate* 引用 `8442`、不是 channel-neighbor，所以 `8442` 的 channel-component 可能全 IR、但 dep-closure 含 `h`。

**修法（雙檢查）**：`OkToSkipInRecalc[v] = NextExpr[v]!=null && !InScc[v] && (a) v 的 channel-graph connected component 全是 IR-covered-non-InScc（處理「v 驅動一個 hybrid h via channel、skip RecalcNode(v) 就傳不到 h」）&& (b) v 的 NodeRef dep-closure 全是 IR-covered-non-InScc（Input node OK —— step 3 已解析）（處理「v 透過 gate 引用一個 hybrid h、skip RecalcNode(v) 就拿不到修正後的 h」）`。(a) 用 union-find（transistor 當邊、不穿 vcc/vss）；(b) 在 `EvalOrder` 序（deps 先）一個 pass：`depImpure[v] = (NextExpr[v] 引用某 m，m 是 real hybrid bus（`NextExpr[m]==null && nodes[m].C1c2s.Count>0`）或 InScc) || (某 IR-covered 引用 m 的 depImpure[m])`。

**結果**：`--trace-cmp --engine ir --debug-skip-recalc`（skip 啟用）→ **0 mismatch over 2000 / 12000 half-cycles（branch_timing/1）+ 4000（instr_test-v5/10-branches）** —— 修對了！→ 把 skip 改成 default-on（移除 `DebugSkipRecalc` gate + `--debug-skip-recalc` CLI；`StepOneDriving` 永遠設 `SkipRecalcOf = OkToSkipInRecalc`）。**但量到只有 184 個 node bridge-skippable**（out of 6824 IR-driven）—— buses（`db`/`sb`/`adl`/…）channel-連到太多東西、control signals 衍生自 buses，所以幾乎每個 IR node 不是 (a) 的 component 裡有 hybrid bus 就是 (b) 的 dep-closure 裡有 hybrid → 只 184 個「純」node（probably 一些 board glue / 孤立的小邏輯閘）。→ **這個 skip 給的加速 ~0**（skip 184 個 `RecalcNode` out of ~thousands/step）。瓶頸（bridge ProcessQueue re-derive 一堆 IR node 的 `ComputeNodeGroup`）**還在**。

- `dotnet build`（Debug + Release）：**0/0**。`--selftest`：**全 PASS**。`--trace branch_timing/1 --cycles 8`（S1）跟 `d1d1ab0` 一致。`--trace-cmp`（checking）0 mismatch。`--trace-cmp --engine ir`（driving，skip default-on）0 mismatch（2000/12000 + 另一個 ROM 4000）。flat program `35,137` 指令（firing 16 的 ~68K 是高估、實際 IR exprs 比較小）。

> **下個 firing**：bridge 瓶頸的真正解 —— (a) **precise-enqueue**：`StepOneDriving` step 4 不要 `EnqueueNode(v)`、改成 step 5 前精準算「哪些 hybrid/InScc/callback node 的 input（= 某個變動的 IR node 或 boundary node）gates 了它的 transistor」→ 只 enqueue 那些 +「v 驅動的 hybrid h（v 在 h 的 channel-component 裡且 v 變了）」→ ProcessQueue 只跑那一小撮（這比「全 enqueue 再 skip」省得多、但要小心 callback / channel-propagation）；或 (b) **Stage B/D 提升 IR coverage**：把 5705 個 InScc node 救回來（少 hybrid/InScc → bridge 自然輕、且更多 node 變 skippable）—— 這也是 IR coverage 從 46.3% 往 85% 拉的正路；或 (c) **接受 driving 模式目前 ~par-ish、focus 在「IR 的 *結構*（flat boolean program）是 GPU bit-slice 的對象」這個角度**（單一實例 par、但 IR 可平行）。

### firing 18 —— S3 per-instance speedup 的盤點 / 決策（為何難、要往哪走）

firing 13-17 在追「IR-driving 比 S1 快」，結果：flat eval（S3.1）給 ~1.46×（8×慢→5.4×慢），bridge-thinning（skip RecalcNode）只有 184 個 skippable（~0 加速）。盤點瓶頸 + 決策：

**為何 bridge 是瓶頸（~5× S1 的 settle）**：driving 模式 `StepOneDriving` step 5 的 `ProcessQueue` 處理「step 4 的 IR-eval 改動的 *每個* IR node 的 gate-fanout（含一堆 IR-covered channel-end）+ 那些 IR node 本身（`EnqueueNode(v)`）」→ 對 queue 裡的東西全跑 `RecalcNode`=`ComputeNodeGroup`（group-BFS，靠近 buses 的 node 是「大 group walk」）→ 多數是 no-op（IR 算對了）但成本付了 → bridge 做了 ~thousands 個 `ComputeNodeGroup`/step，比 S1 的 settle（只 re-derive *變動的* node）多 ~5×。flat eval 本身 ~1.3% 的時間，~98% 在 bridge。

**為何「skip IR-covered node 的 RecalcNode」沒用**：要安全（不漏掉「v 驅動的 hybrid h via channel」+「v 透過 gate 引用的 hybrid h」的傳播），只能 skip「channel-component 全 IR 且 dep-closure 全 IR」的 node —— 但 buses channel-連到太多、control signals 衍生自 buses → 只 184 個合格。

**真正要快，需要其中之一**（都是大工程，都先 Gemini）：
1. **重設計 bridge —— 不 re-derive 任何 IR node**：step 4 直接寫 `NodeStates`（不 `EnqueueNode(v)`），step 5 只處理 hybrid+InScc node 本身（不處理 IR node 的 gate-fanout）。難點：(a) hybrid `h` 要被 enqueue —— `h` 的 input（一個 gate signal `g` 或一個 channel-neighbor）變了才需要重算；step 4 知道哪些 IR node 變了 → 哪些 hybrid 的 gate signal 變了 → 哪些 channel-neighbor 變了 → 精準 enqueue 那些 hybrid。要預先建「hybrid h 的 input set（gate signals + channel-neighbors）」的反向索引。(b) callback —— callback `target` node gated by watched nodes（cs/addr/…，那些是 IR node）→ 那些 IR node 變了 → 要 enqueue `target` → 已經被 (a) 涵蓋（`target` 是個 hybrid-ish node、它的 input 是 watched node）。(c) hybrid `h` 變了 → 它的 channel-group 裡的 IR node 要更新（`SetNodeState`s the group members in `ComputeNodeGroup(h)`）—— 這自動發生。**預期**：bridge 只跑 ~847 hybrid + ~變動的 InScc + callback → 比 S1 settle 少很多（IR 那 46% 不重算）→ driving < S1。
2. **Stage B/D 提升 IR coverage 46.3% → ~85%**：救回 5705 個 InScc node（大半在 2C02 的 tangle —— clock divider/counter/pixel pipeline/sprite logic，IR 抽出來的 model 對它們是錯的（toggle 模型化成 clear 之類）→ 要重抽（認 T-FF / counter / shift-register pattern → 給對的 `Mux(clk_edge, Not(Prev), Hold)` 之類））。大工程，2C02 結構複雜。但這是 IR coverage 往 85% 拉的正路、bridge 也自然輕（hybrid+InScc 從 6552 降到 847）。
3. **接受 driving ~5.4× 慢、改變 S3 的詮釋**：「per-instance 速度」不是這個 pipeline 的價值點（CLAUDE.md：「The value is the translation pipeline ... not the GPU」）；S3 的「比 raw switch-level 快很多」應詮釋為「IR 的 *結構*（flat boolean program、~35K 指令、無 group-BFS）是可 bit-slice 的 —— S4 在 GPU 上跑 N 個實例 ≈ 1 個實例的成本 → 對 N 個實例的 aggregate throughput 遠超 S1」。flat-IR-eval'd、equivalence-verified 的 IR + bridge 就是 S4 的 input。

**決策**：S2.0~S2.6（translation pipeline + 等價驗證）**完成**。S3.1（flat eval）完成。S3 的 per-instance 加速 = 開放問題，路徑 #1（重設計 bridge）或 #2（Stage B/D）；下一個 firing 從 #1 開始（先寫設計丟 Gemini —— 它比 #2 小、且直接攻瓶頸）。詮釋 #3 是 fallback（若 #1/#2 都太難）。

> **下個 firing**：S3.2 設計 —— 重設計 bridge（精準 enqueue、不 re-derive IR node），丟 Gemini，再實作。

## firing 19 起：**S3.2 —— bare-2A03 CPU-only 測試平台**（「S3 CPU proof」的正確量測對象）

> firing 18 的決策：S3 per-instance 加速 = 開放問題。重新審視 —— CLAUDE.md 寫 **「S3 — CPU proof: the IR interpreter must be ... benchmarked as much faster than the raw switch-level interpreter」**。關鍵字 **CPU proof** —— S3 量的是 **2A03（CPU）** 的 IR vs switch-level，不是整片 `nes-001`（CPU + 2C02 + 板間 TTL）。但目前 `--engine ir` driving 整片板子 → 2C02 那 ~703-node tangle + 一堆 InScc node 把 IR coverage 拉到 46.3%、bridge 變重 → 拖慢。**bare-2A03 rig** = 只 compose `2a03` 模組（modified 6502 + APU die）+ 一個 behavioral flat 64KB RAM/ROM handler + 把 `nmi`/`irq` 綁高（inactive）+ clock handler → 一個 headless 2A03。沒有 2C02 → InScc 大幅減少（2C02 tangle 沒了）→ IR coverage 對 CPU 應該 ~90%+ → bridge 只剩 CPU 自己的 buses（db/sb/adl/adh/idb + 資料 latch）~幾百個 hybrid node → bridge 輕得多 → IR-driving 對 CPU 應該 ≥ par 甚至 > S1。**這才是「S3 CPU proof」該量的東西**。也獨立有用：`--test` blargg 跑得快很多（沒 2C02 的 settle 負擔）、CPU bug 隔離。

### ① 設計描述

**`2a03` 模組的 pins**（`ref/metalnes-main/data/system-def/2a03.js`）：`ab0..ab15`（位址匯流排，輸出）、`db0..db7`（資料匯流排，雙向）、`rw`（R/W，輸出 —— high=read）、`phi2`（M2 記憶體時鐘，輸出）、`nmi`/`irq`（中斷，active-low 輸入）、`clk_in`（master clock 輸入）、`dbg`（debug 腳）、`snd1`/`snd2`（音訊輸出）、`res`（reset，active-low 輸入）、$4016/$4017 控制器相關（輸出 strobe / 寫腳）。VCC=pin40, VEE=pin20（地）。

**`ComposeSystem2a03Bare()`**（`WireCore.System.cs` 新增）：`ResetBuild()` → `LoadModuleDef(SystemDefDir, "2a03")` → `AddInstance(2a03def, "")`（prefix 空 → pin 名直接是 `clk_in`/`ab0`/`db0`/…）→ `if (EnableLowering) LowerNetlist()`。**不**載 `nes-001`/`cart-mmu0`/2C02。

**新 handler：`AttachFlatRamHandler(int sizeBytes = 65536)`**（`WireCore.Handlers.cs` 新增）：一個 behavioral 64KB array（`byte[] _flatMem`）；`AddCallback(trigger = ab[15:0] + rw + db[7:0], () => { int a = ReadBits(ab); if (NodeStates[rw] != 0) WriteBits(db, _flatMem[a]); else _flatMem[a] = (byte)ReadBits(db); })` —— read：把 `_flatMem[a]` 推上 `db`；write：把 `db` 存進 `_flatMem[a]`。（觸發在 `ab`/`rw`/`db` 任一變 —— 跟現有的 `AttachRamLikeHandler` 同套路、但沒 `cs`、addr 是全 16-bit。）

**`LoadSystem2a03Bare(NesRom rom)`**（`WireCore.System.cs` 新增）：`ComposeSystem2a03Bare()` → 把 `rom.PrgRom` copy 進 `_flatMem`（16KB → mirror 到 $8000 跟 $C000；32KB → $8000-$FFFF；其餘清 0）→ `AttachClockHandler()`（改成找 `clk_in`，或加參數 `AttachClockHandler(string nodeName = "clk")`）→ `AttachFlatRamHandler()` → `ResolveCachedNodes()` → `ResetNes(full: true)`（reset 序列在 `res` 上；reset 前先把 `nmi`/`irq` 綁高 —— 用 `pullups`？不行，`ResetBuild` 後才有 node；改成 `ResetNes` 裡 reset 完 `SetHigh("nmi"); SetHigh("irq")`，或在 compose 後給那兩個 node 一個 forced-high flag）。

**CLI**：`--system 2a03`（或 `--cpu-only`）—— `--test`/`--trace`/`--trace-cmp`/`--benchmark` 都查這個 flag、改用 `LoadSystem2a03Bare` 而非 `LoadSystem`。

**S3 量測**：`--benchmark <blargg.nes> --system 2a03`（S1）vs `--benchmark <blargg.nes> --system 2a03 --engine ir`（IR-driving）→ 看 IR-driving 對 CPU-alone 多快（預期 ≥ par、希望 > S1 —— 這就是 S3 CPU proof）。`--test <blargg.nes> --system 2a03`（S1 + IR 都跑、比 $6000 簽章）確認功能對。`--trace-cmp <rom> --engine ir --system 2a03` 確認 IR-driving 對 CPU-alone 0 mismatch。

**開放問題（丟 Gemini）**：
1. `2a03` 模組的 pin 接法 —— `nmi`/`irq` 綁高（inactive）、`phi2` 是 CPU 輸出（flat RAM handler 要不要 trigger 在 `phi2` 邊緣，還是 `ab`/`rw`/`db` 變就夠？）、$4016/$4017 控制器讀 —— bare rig 沒控制器，讀 $4016/$4017 會返回 flat RAM 的 garbage（blargg CPU 測試不讀控制器 → OK）對嗎？`dbg` 腳要不要綁？`snd1`/`snd2` 不接 OK 嗎（它們是輸出）？
2. flat RAM handler —— trigger 在 `ab + rw + db` 任一變（跟 `AttachRamLikeHandler` 同）—— 會不會有「write 時 `db` 是 CPU 在驅動、read 時 handler 在驅動 `db`」的 contention（在 S1 的 group 解析裡，handler `WriteBits(db, ...)` = `SetHigh/SetLow(db_i)` → 那是個強驅動；CPU 也驅動 `db` —— read cycle 時 CPU 的 `db` driver 應該是 hi-Z/弱）？這在現有 `AttachRamLikeHandler`（驅動 `_d[]` 內部匯流排、不是外部 `db` 腳）裡是怎麼處理的 —— bare rig 直接驅動外部 `db` 腳會不會不一樣？
3. `--system 2a03` 是否就是「S3 CPU proof」該量的對象 —— 還是該量「2a03 + 2C02 但不含板間 TTL」之類的中間配置？（我覺得 bare-2A03 + flat RAM 是最乾淨的「CPU only」—— TTL/2C02 都是 CPU 外的東西。）
4. 預期：bare-2A03 的 IR coverage（vs 整片 46.3%）、InScc 數（vs 5705）、bridge 大小（vs 6552 hybrid+InScc）—— 我猜 IR coverage ~90%+、InScc ~少數百、hybrid ~幾百（CPU 的 buses）。對嗎？driving 對 CPU-alone 會 > S1 嗎（瓶頸 = ~幾百個 bus `ComputeNodeGroup`/step vs S1 的 ~幾百-幾千）？

每條建議標明「對這個計畫的具體影響 / 該怎麼改」。某設計 OK 不用改也直說。
