## 1. 設計總結(投稿時的 contribution 敘述底稿)

**一句話:** 在事件驅動的電晶體級(switch-level)模擬器中,於載入時對全部節點 ID 做一次重編號,
排序鍵 =(主鍵)靜態剪枝類別 +(次鍵)生產級聯的真實初次觸碰順序(由引擎**自我捕捉**),
使熱迴圈的剪枝查表變成暫存器區間比較、且資料排列吻合真實事件排程 —— C# 引擎合計 **+10%**(+3.56% 範圍化 + +6.17% 自捕捉鍵),位元等價。

### 1.1 主鍵:範圍剪枝(range-prune)

- 4 個剪枝類別(可跳過關斷 × 開啟不安全)按類別排序後各自成為**一段連續 ID 區間**:
  `[3,A)` skip∩unsafe / `[A,S)` skip∩safe / `[S,B)` noskip∩safe / `[B,∞)` noskip∩unsafe(supply 1,2 搭 skip 順風車;假手把節點延伸最後一塊)。
- 熱迴圈的 `PruneMask[c] & bit`(L1 常駐但**相依**的 byte 載入,餵入入列決策鏈)變成
  `c < S`(關斷跳過)與 `c < A || c ≥ B`(開啟不安全)—— 每端點檢查刪掉一條相依載入 + 15KB 陣列下架。
- **適用條件三閘門**(實證歸納):(1) 靜態事實 (2) 熱 (3) **位於被攜帶的相依鏈上**。
  反例:IsPureLogic 三類做 cls-range = −0.28% 中性 —— 其載入餵的是 ~70% 可預測的分支(延遲被投機隱藏),不在串行鏈上。
- 正確性:PruneMask 每次 Reset 仍計算為基準真相,首次 settle 前逐節點比對範圍推導位元;
  不符 → 退化安全邊界(剪枝關閉、supply 守住、依然正確)= JIT 的 deopt guard 模式。

### 1.2 次鍵:自我捕捉初次觸碰順序(self-captured first-touch)

- 三遍載入:pass 0 原始編號下分類剪枝位元(只 Reset)→ pass 1 以暫時靜態鍵(clk 盲 BFS)做類別為主鍵重建
  (區間驗證通過 → 剪枝**開啟**)+ 暖機 1024 hc + 用 settle 迴圈的**冷儀器化副本**捕捉 32768 hc 的真實初次觸碰順序
  (熱 ProcessQueue 完全不動)→ 經排列翻譯回原始 ID → pass 2 最終重建。總載入成本 ~1.3 s。
- 正確性與排列無關是**構造出來的**:唯一依賴 ID 順序的點(開機重算掃描)與狀態 checksum 都透過排列以原始 ID 順序迭代 → 三條 golden checksum 不變。
- 對工作負載漂移(不同手把輸入、不同流程、步數 1M→5M)**構造上免疫**:每次載入從晶片本人重新導出,沒有過期 profile 問題。

## 2. 實證數據(論文的 evaluation 章底稿)

平台:AMD Ryzen 7 3700X,Windows 11,.NET 11 / Rust(rustc release);
方法:full_palette 400k hc、`--pin`、interleaved-paired(逐輪交替、checksum 閘門、報告中位數+配對勝場);
golden:`0x794A43ABDF169ADA` @300k / `0x9174E19D961CB6E5` @400k / `0x6D4CCBCE2E9CD599` @1M,全程不變。

| 實驗 | 結果 | 判定 |
|---|---|---|
| 範圍剪枝(C#,51e046d) | **+3.56%** 中位數,19/20 | 採用 |
| 範圍剪枝(Rust,665acf5) | **+2.90%**,20/20;區塊邊界 A=460/S=1275/B=7532 與 C# **完全相同** | 採用(跨引擎驗證免費) |

|---|---|---|

## 3. 先行研究對照(Gemini 2026-06-11 諮詢,引用清單)

Gemini 總評:「Inspector-Executor 式、profile 導引、資料導向的記憶體重排器,帶 JIT 式 deopt guard,應用於 switch-level EDA 模擬」——
**各組件皆有出處,但這個綜合體罕見且值得發表。**

### 3.1 各組件的學術譜系

**A. 線上同進程資料重排(我們的整體範式)= Inspector-Executor 模型**
- Saltz et al., 1991 — "A manual for the CHAOS runtime library"(奠基作)
- Ding & Kennedy, LCPC 1999 — "Improving cache performance in dynamic applications through data and computation reorganization"
- Strout et al., PLDI 2003 — "Compile-time composition of run-time data and iteration reorderings"
- 我們的「線上自捕捉勝過期離線 profile」= 經典 Stale Profile Problem(靜態 PGO vs 動態 JIT 剖析)

**B. 圖頂點重排:執行軌跡 vs 結構**
- 結構排序(RCM、Gorder SIGMOD'16、Rabbit Order IPDPS'16)在這裡失效:雙向 pass-transistor 網路的拓撲不映射到執行流。
- Chilimbi & Larus, PLDI 1999;Chilimbi, PLDI 2001 — "Efficient Representations and Abstractions for Quantifying and Exploiting Data Reference Locality"(Temporal Data Streams)
- Han & Abdelrahman, SC 2013 — profile 導引軌跡排序(SpMV/GPU)
- PDES 文獻(Fujimoto 1990)有邏輯進程分組,但**單執行緒、軌跡式排列在離散事件模擬中文獻極稀疏** ← 我們的空隙

**C. 主鍵把戲(靜態謂詞 → ID 區間比較,刪相依載入)**
- 學界稱 Index Space Partitioning / Predicate-Based Array Partitioning / Enum Partitioning;
  無單一名作 —— 被視為編譯器優化 / Data-Oriented Design 模式(Data Layout Optimization for Branch Elimination 傘下)。

**D. 驗證模式(重算基準真相、驗證、退化回退)= Dynamic Deoptimization / Guarded Execution**
- Hölzle, Chambers, Ungar, PLDI 1992 — "Debugging Optimized Code with Dynamic Deoptimization"
- Arnold & Ryder, PLDI 2001 — "A Framework for Reducing the Cost of Instrumented Code"

### 3.2 實務上的雙胞胎

1. **遊戲引擎 ECS Archetype 排序**(Unity DOTS/Burst、Flecs)—— 主鍵把戲的一比一對應:按組件擁有狀況把實體排進連續記憶體塊,迴圈零 `has(component)` 檢查。
2. **JIT / 複製式 GC 重定位** —— 複製順序(BFS/DFS 追蹤)決定共同存取物件的空間區域性;我們的 pass 1/pass 2 = 為精確事件級聯客製的 tracing-GC 重定位遍。
3. **資料庫叢集索引 / 範圍分割** —— `(Class, CapturedOrder)` 上的 clustered index;`WHERE Class < 3` 直接 seek 連續頁。
4. **Verilator / 編譯式邏輯模擬器** —— 靜態拓撲攤平;我們的模擬器是事件驅動 switch-level(本質循環/動態)不能靜態編譯 → 我們等效做了「**用暖機執行軌跡對動態事件做 AOT 編譯步驟**」。

### 3.3 可發表性判定 + 防撞關鍵字

## 4. 發現 (b) 的微架構解釋(Gemini;論文 discussion 章素材)

「等密度下生產順序仍贏」= 硬體預取與記憶體級平行度(MLP)中**時間 vs 空間區域性**的教科書級展示:

1. **硬體預取器串流訓練** —— 現代 L1/L2 預取器(Intel spatial/stream)不只看「哪些行被摸」(密度),
   重度依賴**步幅與單調遞增序列**。排列吻合生產排程 → 事件佇列以近乎遞增的順序存取記憶體 → 預取器建立信心預載;
   非生產順序同樣的行是熱的,但在半週期內以散亂偽隨機序觸碰,打斷預取器。
2. **ROB / LSQ 利用率** —— 即使資料在 L1,循序存取模式最大化 Load/Store Queue 效率、維持高 MLP。
3. **微視窗密度** —— 每半週期密度相同,但每 ~50 指令視窗的密度可能不同:
   生產排列把「緊接著發生」的事件打包進同一 cache line(一次抓取服務 4–5 個連續事件);
   非生產排列在同半週期內於 4 條熱行間來回彈跳,增加 L1 port 壓力。

文獻錨點:Somogyi et al., ISCA 2006 — "Spatial Memory Streaming";
搜尋詞:"Spatial vs Temporal Data Locality"、"Hardware Prefetcher Training via Data Layout"。

**Rust 正負翻轉(C# +6.17% / Rust −1.08%)是這個解釋的天然對照組:**
同一張網表、同一把鍵、位元等價,僅後端程式碼生成不同(.NET JIT vs LLVM;分支 vs branchless 熱路徑)→
區域性鍵的價值是**引擎×微架構特定的**,不是網表內在性質。論文裡可作為「layout 收益不可移植」的證據點。
