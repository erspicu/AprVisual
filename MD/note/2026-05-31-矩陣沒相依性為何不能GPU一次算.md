# 矩陣化後既然沒相依性,為何不能用 GPU 一次算?

> 2026-05-31 討論筆記(接續 `2026-05-31-網表當2D矩陣處理-SPICE矩陣法評估.md`)。
> 結論先講:**我們做過 —— S4 的 GPU(bit-sliced D3D11/HLSL)後端實測比 CPU 事件驅動的 S1 慢約 10–18×。** 原因:矩陣只殺掉「節點走訪順序」這一種相依性,真正綁住你的「迭代相依 + 時間相依」還在,而那兩者才是擋住「GPU 一次算」的;再加上它逼你 oblivious(算全部節點)、丟掉 ~1.4 活躍節點的稀疏性。GPU 是吞吐機器,我們是延遲 + 稀疏 + 時間序列問題 —— 用錯工具。

## 1. 相依性沒消失,只是「換了位置」
矩陣化只殺掉**節點間的走訪順序**相依。但綁住你的相依性還在:
1. **不是「一次算」,是「迭代到收斂」**:直接解是 LU(消去本身序列相依);離散邏輯是 **relaxation 迭代到 fixpoint**(我們 settle 跑 12–45 passes)。**每個 pass 依賴上一個 pass → pass 之間有 barrier**(相依性從「節點間」搬到「迭代間」)。GPU 能平行單一 pass 內的節點,但 pass 之間仍序列。
2. **時間軸序列**:半週期 N+1 依賴 N settle 完的狀態 → 單一 instance **無法跨時間平行**;只能平行「一個半週期內的 settle」,那是極小工作 + pass barrier。
3. **線性解本身有相依**:稀疏 LU 消去序列、GPU 上極難平行;迭代解(SpMV)好平行但要很多次全系統迭代才收斂。

## 2. 它逼你走 oblivious —— 丟掉稀疏性
矩陣全解 = 每個 pass 算全部 14.7K 節點;但事件驅動下平均只有 **~1.4 個**真的變。GPU 平行算那 14.7K,等於做 ~一萬倍多餘的工作;加上 **kernel launch + CPU↔GPU 來回(每半週期都要把結果傳回 CPU 才能推進時鐘)** 的延遲,打不過 CPU 上「只碰 1.4 個節點」。→ oblivious 實測 121× 慢、GPU 10–18× 慢的根因。

## 3. 它本來就是離散,不是連續
「矩陣化」真正意思是回到 **SPICE 式連續解**(電壓/電流、Newton 迭代),那是 switch-level 特地逃離的昂貴東西。離散 0/1 + 「GND 贏/pull-up/hold」不是線性代數解。

## 4. GPU 真正的舞台:很多個 instance,不是一個
GPU 是**吞吐**機器(thousands of cores 要 thousands 份獨立工作)。真正會贏的場景是 **bit-sliced 同時模擬上千台 NES**(每 lane 一台),才餵得飽、才攤平 launch 成本。但:
- 那是**不同目標**(我們現在單一 instance 求快)。
- 各 instance 活躍節點不同 → SIMD lane **發散**,逼每 lane 做 oblivious 全節點更新 → 得平行到 **>121 台**才追平「CPU 單台事件驅動」,而且**單台速度沒變快、也一樣達不到 real-time**。

## 誠實底線
**瓶頸是「延遲 + 稀疏 + 時間序列」,不是「吞吐不足」。** GPU 解吞吐問題;我們是「每半週期等一個極小、依賴前一步的結果」的延遲問題 —— 拿大砲打蚊子,還每打一隻就 CPU↔GPU 來回一次。實測 **GPU 10–18× 慢、bit-parallel 156× 慢、per-chip 平行 15× 慢**,同一根因。

→ 一句話:**矩陣讓「節點相依」消失,但「迭代相依 + 時間相依」還在,而後兩者才擋住 GPU one-shot;再加上它逼 oblivious、丟稀疏性 —— GPU 對單一 instance 必輸(S4 已驗證)。GPU 只在「同時跑很多台」才有意義,而那是另一個目標。**
