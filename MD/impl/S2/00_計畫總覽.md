## 0.1 範疇（依 `MD/struct/08` 2026-05-12 路線圖調整）

- **整片板子的網表**：2A03（CPU）+ 2C02（PPU）+ 板上 74xx 膠合（74LS373/139/368、74HC04）+ CIC（nes-cic1）+ 手把（nes-pad）+ NROM mapper 邏輯（nes-cart72 / cart-mmu0）。**不是「先 CPU 後 PPU」** —— 抽取是一套演算法跑整張圖；人為把 CPU/PPU 隔開反而要設計「IR 區 ↔ switch-level 區」的跨區協定（= 失敗點 #1 的溫床）。
- **記憶體陣列維持行為級**（u1/u4 SRAM 的 2KB、cart PRG-ROM/CHR-ROM/CHR-RAM、`$6000` 工作 RAM）—— 它不是電晶體網表，是 `byte[]` + handler，是**模型的邊緣**；S2/S3/S4 都一樣 call out 到那個陣列。S2 不對它抽邏輯（它沒邏輯）。
- **「分階段」按結構複雜度，不按晶片**：先抽良性結構（反相器 / NAND / NOR / 串並聯下拉網路 / 明確的 precharge-evaluate）→ 那一刀就 cover 晶片大部分 → 高覆蓋率 IR + 少數還沒處理的區域 fallback 回 switch-level（hybrid）→ 隨著處理更多結構類型，覆蓋率往 100% 爬。「最難的結構」（shift register / OAM DRAM / 共享 bus / 強耦合 dynamic）剛好大多落在 2C02。
- 只 NROM（mapper 0）—— 跟 S1 一致。

## 1. 核心問題：怎麼從「開關層級網表」抽出「邏輯」？

開關層級模型在每個瞬間做的事：給定所有 gate node 的狀態 → 整張網表依「目前導通的電晶體」分成連通群 → 每群的值 = `resolve(群裡所有 node 的 flags OR 起來)`（優先鏈：GND 贏 → VCC/pull-up → 外部驅動 → 保留前值 → 0；純浮島：capacitance 最大的 node 勝）。

「抽出邏輯」= 把這整套塌成「`next_state[每個 node] = f(current_state[每個 node], 外部輸入)`」，並把 `f` 表達成布林電路（`Expr` DAG）。對 NMOS 電路，`f` 的結構**大部分**很乾淨：

### 1.1 良性結構（佔晶片大部分）—— 「下拉網路 + 負載」

標準 NMOS 邏輯閘 = 一個「**下拉網路**」（電晶體串並聯接到 GND）+ 一個「**負載**」（depletion-load 弱拉高 / pull-up 接 VCC）。閘輸出 node `N`：

```
N = NOT( 下拉網路導通 )
下拉網路導通 = 對 gate signals 的布林式（串聯 = AND、並聯 = OR）
```
- 反相器：`N = NOT(a)`；NAND：`N = NOT(a AND b)`；NOR：`N = NOT(a OR b)`；AOI/OAI：`N = NOT((a AND b) OR c)`…
- 把「下拉網路」當「N→GND、邊權是 gate signal 的圖」做**串並聯歸約**得到布林式；歸不出純串並聯（橋接結構）→ 退一步用「N→GND 的路徑 OR、每條路徑上 gate 的 AND」；路徑數爆炸 → 標 hybrid。

### 1.2 難 / 需要小心的結構（少數、大多在 2C02）

- **pass transistor / transmission gate —— 而且雙向**：NMOS 的 pass transistor**沒有方向**；S1 是靠「誰的驅動強度大」決定 A 驅動 B 還是 B 驅動 A。所以「純 pass fan-in = Mux」這個假設不夠 —— **S2.1 要做方向性推導**（典型：A 側有 PULLDOWN 網路、B 側純 dynamic（只有 pass 連進來）→ 方向 A→B）；推不出單一方向的 pass 網路（如 2C02 OAM 讀寫匯流排）→ **不要硬轉 Mux，直接 hybrid**。能定向的 → `N = Mux(en, M, Hold(N))`，多顆 → mux 鏈。
- **active pull-up / superbuffer（push-pull）**：6502 的 clock generator / address pin 等用一顆 depletion NMOS 當 *active* pull-up（gate 接信號而非短路到 source）→ 形成 push-pull。這種 node：`N = Mux(pullup_cond, 1, Mux(pulldown_cond, 0, Hold(N)))`，並 assert `!(pullup_cond AND pulldown_cond)`。→ S2.0 的電晶體分類要能辨識 `ACTIVE_PULLUP`（vs 接 VCC 的 strong pull-up vs 接自己 source 的 static depletion load）。
- **precharge-evaluate（domino logic）**：2C02 sprite 邏輯大量用 —— phase1 透過（conditional / clocked）pull-up 預充至 1，phase2 下拉網路求值。Expr 長 `N = Mux(phi1, 1, NOT(pulldown_eval))`。落在現有設計範圍內，但 **S2.1 要把「預充用的 pull-up」辨識成 conditional pull-up，不是 static load**。
- **dynamic node（電容存值）**：沒有下拉路徑也沒 pull-up，只在某些 phase 被 pass 進來、其他時候靠寄生電容保持 → `next[N] = (被驅動) ? 驅動值 : prev[N]` = `HoldExpr`。
- **回授環（cross-coupled latch）**：兩個 node 互相當對方下拉網路的一部分 → SCC → 序向元件（register bit、SR latch）→ `next = sel ? data : prev`，指回上一輪狀態（`NodeRefExpr`/`prev[...]`）。CPU/PPU 裡幾乎每個 register bit 都是這種，而且**靠 pass transistor 隔離 + 兩相時鐘**（見 S2.3）。
- **charge sharing（電荷分享）**：兩個 capacitance 不同的 dynamic node 經 pass 相連、沒 VCC/GND → 強電荷覆蓋弱電荷（Visual6502 有 node area capacitance）。6502 真正靠這個決定邏輯值的場景極少（多半只在預充放電）。**初期：兩端都 floating 的 pass 導通 → 標警告 + hybrid**，看 test ROM 有沒有真依賴。
- **共享 / tri-state bus**：多驅動者、可能 tri-state → 勝負由 *driving strength* 決定（MOSSIM 的 signal strength）。但我們的 `Expr` IR 是**純布林、沒有 strength 維度** → bus-fight 區域 → **hybrid**。

### 1.3 結論

「抽邏輯」= 對每個 node 算「驅動結構」（下拉式 / pull-up 種類（static load / VCC strong / conditional / active）/ pass 進來的分支 + 方向），把良性的塌成 `Expr`，難的標 hybrid 或丟 SCC 分析，最後得到「組合邏輯 DAG + 序向 node（clock edge 更新）+ 少量 hybrid 區」的 IR。

### 1.4 等價驗證的兩個前置決策（X / floating / 初始化）

開關層級在剛開機時 node 可能 floating / 未定，要幾個 cycle 才全部初始化。為了不為這個吃苦：

- **不在 IR 引擎裡做三值邏輯（X）**。理由：S1 的 `NodeStates` 本來就是 `byte` 0/1（不追 X）—— power-on 是 `Reset()`（pull-up→1、其餘→0）+ `RecomputeAllNodes()` 把網表 settle 到一個**決定性**的穩態。所以根本沒有「S1 那邊是 X」的對齊問題。
- **IR 引擎的初始 state = S1 settle 完的 power-on state**：IR 引擎跟 S1 **共用同一個 `NodeStates` 陣列**，`ResetNes(full:true)` 跑完（含 `RecomputeAllNodes()` 的 S1 settle）後 IR 引擎接手 —— 自動從 S1 的穩態開始。等價檢查就是「IR 引擎從這個穩態演化 vs S1 引擎從同一穩態演化，逐 half-cycle 比 node-state 陣列」。
- **`HoldExpr` 語義必須對齊 S1**：當 S1 某 node 失去所有驅動 → 靠 capacitance 保持前值（再被導通進來的群覆蓋）。`HoldExpr(N)` 的行為要跟 S1 「保留前值 → 被群覆蓋」的優先鏈完全一致（這也是為什麼 charge-sharing 那種「強覆蓋弱」要 hybrid —— 純 `prev[N]` 表達不了）。
- （備案）若這個簡化在某些區域對不齊：選項 B = 前 N 個 reset cycle 關閉逐 node 檢查、把 S1 的 state array 整個蓋給 IR 當初值、從穩態開始對；選項 A = IR 真的上三值邏輯（`Expr` eval 支援 X 傳播）。先不做，撞到再說。

## 2. 我設計的 S2 步驟拆解

> 編號 S2.0～S2.6（S2.3 拆成 a/b）。每一步都有自己的設計檔 + Gemini review + 驗證。S2.4（IR 直譯器）之前的 step 都不改 S1 行為（只是建分析資料 / 平行 backend）。

### S2.0 — 靜態圖 + 角色標記（無邏輯抽取）

**做什麼**：從 lowered netlist（`_nodes`/`_transistors`，整片板子）建分析用的 `NetlistGraph`：
- **node 角色** `NodeRole`：`SUPPLY`（vcc/vss）/ `INPUT`（從不被網表驅動、只被 handler·reset `SetHigh/Low` 的 —— 至少 `clk`、`res`）/ `BUS`（雙向：既被網表又被 handler 驅動，如 `db[]`、PPU/RAM 資料線）/ `INTERNAL`（其餘）。
- **電晶體分類** `TransistorKind`：`STATIC_LOAD`（depletion load，gate 接自己 source 或 weak 旗標）/ `STRONG_PULLUP`（channel 端是 VCC、gate 永遠 1）/ `ACTIVE_PULLUP`（depletion NMOS、gate 接信號 → push-pull 的上半）/ `PULLDOWN`（channel 端是 GND，或經串聯鏈到 GND）/ `PASS`（兩 channel 端都是信號 node）。
- **每顆 PASS 電晶體上預留 `Direction` 欄位**（`Unknown` / `A_to_B` / `B_to_A`），S2.1 推導後填。
- **共用 `NodeStates`**：確認 IR 引擎之後跟 S1 共用那個 `byte*` 陣列（power-on 的 X 問題見 §1.4）。

**資料結構**：`src/AprVisual/Sim/Logic/NetlistGraph.cs`（`Logic/` 子目錄）；沿用 build-time 的 `Node.Gates`/`C1c2s`，外掛角色 / kind / direction 的平行陣列。
**驗證**：count 各角色的 node、各 kind 的 transistor；spot-check（vcc/vss=SUPPLY、clk/res=INPUT、`cpu.db0`=BUS、某個確定是 depletion load 的=STATIC_LOAD）。**不影響模擬行為**。

### S2.1 — 導通分析（每個 node 的「驅動結構」+ pass 方向）

**做什麼**：對每個非 INPUT/SUPPLY node `N`，算 `DriveInfo`：
- `pullDown : Expr?` —— 「N 被拉到 GND」的布林條件（沿 PULLDOWN 子圖做以 N 為源、GND 為匯的串並聯歸約；非串並聯橋接 → 退 path-enumeration 設爆炸上限；超限 → `Complex`/hybrid）。
- `pullUp` 種類：`StaticLoad` / `StrongVcc` / `Conditional(Expr)`（如 precharge 的 clocked pull-up）/ `Active(Expr)`（superbuffer）/ `None`。
- `passes : List<(int otherNode, Expr cond, Direction dir)>` —— 透過 PASS 電晶體連到別的信號 node 的分支；**方向性推導**：A 側有 PULLDOWN 而 B 側純 dynamic → A→B；兩側都有 driver / 推不出 → 整個 pass 網路 → hybrid（不硬轉 Mux）。

**演算法**：PULLDOWN 子圖串並聯偵測（series 鏈 / parallel 分支；橋接 → path enum + 上限）；PASS 電晶體單獨收集 + 方向推導 heuristic（參考 perfect6502 `netlist_to_c.py` 的 pass-grid 化簡，見 §6）。
**輸出**：`DriveInfo[nodeId]`（含 hybrid 標記）。
**驗證**：spot-check 已知閘 —— 某反相器 node：`pullDown == NodeRef(gate)`、`pullUp == StaticLoad`、`passes` 空；某 NAND：`pullDown == And(a,b)`；某 register bit 的 cross-coupled pass 對：方向推不出 / 互指（交給 S2.3）。

### S2.2 — `Expr` IR + 良性結構抽取

**做什麼**：
- `Expr` 型別：`ConstExpr(bool)` / `NodeRefExpr(int)`（讀現值）/ `PrevExpr(int)`（讀上一輪值，由 S2.3 打斷環時引入）/ `NotExpr` / `AndExpr` / `OrExpr` / `MuxExpr(cond,a,b)` / `HoldExpr(int)`。
- 由 `DriveInfo` 組 `nextExpr[N]`：
  - 下拉 + static/strong pull-up、無 pass → `Not(pullDown)`（反相器/NAND/NOR/AOI）。
  - 有 active pull-up + 下拉 → `Mux(pullup_cond, 1, Mux(pulldown_cond, 0, Hold(N)))`，並記下 assertion `!(pullup_cond AND pulldown_cond)`（執行期可選擇性檢查）。
  - 有 conditional（clocked）pull-up（precharge）→ `Mux(precharge_cond, 1, Not(pulldown_eval))`。
  - 純 pass fan-in（已定向）→ `Mux(cond1, M1, Mux(cond2, M2, … Hold(N)))`。
  - 純被動上拉、無下拉無 pass → `Const(true)`（除非被 pass 驅動的 dynamic → `Hold`）。
  - 兩端都 floating 的 pass 導通 / 推不出方向 / pullDown 是 `Complex` / bus-fight → **標 hybrid**，記到 hybrid 名單。
- 標每個 node：**組合 / 序向**（初判：`nextExpr` 引用回自己或透過明顯的環 → 序向；S2.3 精修）。
- 算**覆蓋率**：有 `nextExpr` 的 node 數 / 全部。

**輸出**：`Expr?[] nextExpr`（null=hybrid）、`bool[] isSequentialHint`、hybrid 名單、覆蓋率。
**驗證**：覆蓋率報告；dump 幾個 node 的 `Expr` 樹目視；spot-check 反相器/NAND/NOR/AOI 的式子；hybrid 名單初看（應該主要是 2C02 的 OAM/bus/sprite 區）。

### S2.3a — 帶條件的依賴圖（**不要建純靜態圖**）

**做什麼**：建**Conditional Dependency Graph**：`nextExpr[N]` 引用 `M` → 加邊 `N → M`，**邊上帶這個引用的「啟用條件」**（例如 `Mux(phi1, M, Hold(N))` 裡的 `M` → 邊 `N→M` 條件 `phi1`；無條件引用 → 條件 `true`）。
**為什麼**：在 NMOS 網表上對**純靜態**圖跑 Tarjan 會得到巨大無意義的 SCC —— 因為幾乎所有 latch 都靠 pass transistor 隔離、pass 在硬體上常雙向 / 串成環，Tarjan 會把整條資料路徑（甚至跨多個 latch）包進一個巨大 SCC。
**驗證**：邊數 / 帶條件邊的比例；spot-check 一個 register bit 的兩條 cross-coupled 邊，各帶互斥的 clock 條件。

### S2.3b — 動態 SCC 打斷（用兩相時鐘）→ 序向 node

**做什麼**：在 conditional dependency graph 上找環。利用 2A03/2C02 明確的**兩相時鐘 `clk0`/`clk1` 互斥**：
- 若一個環上有一條邊需要 `clk0==1`、另一條需要 `clk1==1` → 這個環在邏輯上不存在（是跨 phase 的 pipeline，不是組合回授）→ **打斷它，把那些 node 標成序向（latch）**，環內被打斷的引用換成 `PrevExpr(...)`。
- 剩下的真 SCC = 同 phase 內、不被 clock 隔離的 asynchronous loop（RS latch、bus 回授）→ 分類：**static latch**（有清楚 set/reset/hold → `next = sel ? data : prev`）/ **dynamic**（refresh 路徑 → `next = prev` 除非被驅動）/ **bus feedback / 強耦合**（→ hybrid）。良性的改寫成序向；難的標 hybrid。
- 處理完：SCC 外的圖是 DAG → 拓樸排序（組合部分的求值順序）。

**輸出**：序向 node 清單（精修版）、改寫後的 `nextExpr`（環打斷、引用換 `PrevExpr`）、剩餘真 SCC 的分類、組合部分的拓樸序、hybrid 名單（增補）。
**驗證**：剩餘 SCC 的數量 / 大小（應該很少、很小）；確認「SCC 外無環」；spot-check（某 register bit 確實被當「被 clock 打斷的環 → 序向」處理）；覆蓋率（IR 能完整處理的 node 比例）。

### S2.4 — IR 直譯器

**做什麼**：一個跟 S1 平行存在的 backend（`src/AprVisual/Sim/Logic/IrEngine.cs`），跟 S1 **共用 `NodeStates`** + 自己一份 `prevStates`（給 `PrevExpr`/`HoldExpr` 用）：
- 把「INPUT / BUS 被 handler·reset 設的值」當這一拍的輸入。
- 組合 node 按 S2.3b 的拓樸序 evaluate `nextExpr`（讀現值）。
- 序向 node 在 clock phase 邊界更新（讀 `prevStates`，寫 `NodeStates`；phase 邊界後 `prevStates ← NodeStates`）。
- 兩相 master clock（clk0 high/low half-cycle）、half-cycle 推進 —— 對齊 S1 的 `Step()`。
- handler chain（clock toggle、RAM/ROM、video）照跑（外部驅動者）。
- 殘餘真 SCC（S2.3b 後應該很少）→ 在那一小區做 settle + cap-and-warn。
- hybrid node → 交給 S2.5 的 bridge。

**輸出**：`IrEngine` + CLI（`--engine ir` 之類）能跑 ROM；保留 `--engine interp`（= S1）做對照。
**驗證**：跑得通 S1 跑得過的 test ROM（先不要求逐 node 一致 —— 那是 S2.6）；`--trace` 看 CPU state 像不像；`--benchmark` 順手量一下 IR vs interp（S3 的前哨）。

### S2.5 — Hybrid bridge（fallback；邊界守在 latch 上）

**做什麼**：對標 hybrid 的 node/區域，留在 switch-level 引擎跑；IR + switch-level 共用 `NodeStates`；每個 half-cycle：IR-eval IR node → switch-level-settle hybrid node → 若兩邊有同 cycle 組合耦合則來回迭代到一致。**關鍵**：
- 這是**一個引擎內部**的 IR 區 ↔ hybrid 區橋接，不是「CPU vs PPU」切割；是 **fallback**，hybrid 名單該隨 S2.2/S2.3 處理更多結構而縮向 0。
- **盡量讓 hybrid 區域的邊界落在 latch（序向元件）上** —— 如果 hybrid 區的 I/O 都是前一個 half-cycle 已穩定的 latch 訊號，那這一拍 IR 跟 switch-level 各 evaluate 一次即可，不用「來回迭代」。
- 加 **迭代次數監控**：若某 hybrid 區每 cycle 要迭代 >5 次 → 警告，優先把那個區域的連接 node 標出來 / 想辦法消掉。

**輸出**：hybrid bridge；hybrid 覆蓋率（希望很小）；迭代次數統計。
**驗證**：hybrid + IR 混跑的整體 trace；hybrid node 清單 + 每區迭代次數。

### S2.6 — 等價 gate（S2 過關）

**做什麼**：IR 引擎（+ hybrid）的 **`NodeStates` 陣列、每個 half-cycle、對全部 named node（或全部 node）**，跟 S1 開關層級引擎的逐一比對 —— 必須 **bit-identical**（初值是 S1 settle 完的 power-on state，見 §1.4）。任何不一致 → 回 S2.1/S2.2/S2.3 修；對照 perfect6502 / Javitto·QuietSnooze 的 schematic 判斷「是演算法錯，還是這真的需要 hybrid」（§6）。再加：所有 test ROM 仍 pass；dump `ir_debug.json`（每個 node 的 `nextExpr`、SCC/序向分類、hybrid 名單、覆蓋率）供人工檢視。

**S2 過關條件**：
- ✅ IR 抽取對整片板子完成，能 dump `ir_debug.json`。
- ✅ IR 引擎跑得通 S1 跑得過的那批 test ROM。
- ✅ IR 引擎與 S1 開關層級引擎的 trace **逐 node、逐 half-cycle 完全一致**（硬條件）。
- ✅ 回授 / SCC 區域被正確分類（不展開爆炸 / 不收斂）。
- （放寬條款）頑固區域可 hybrid，但記清楚、標 IR 覆蓋率 ≥ X%、整體 trace 仍一致。

## 4. 不在 S2 範疇

- **IR 優化**（常數摺疊 `A && true ⇒ A`、CSE、死碼、攤平成線性指令拿掉指標間接）= **S3.1**。S2 只要**正確**的 IR + 直譯器；S3 才讓它快。
- **codegen**（C++ / Verilog / CUDA bit-sliced）+ GPU 大規模平行 = **S4**。
- **效率證明 / benchmark 報告**（IR-CPU ≫ netlist-CPU）= **S3**（S2.4 後可順手用 `--benchmark` 量一下當前哨）。
- mapper > 0、AI 大規模訓練環境、整顆 NES 常駐 GPU kernel = S4 之後 / stretch。

## 5. 不採用 / 暫不做的（記下來免得重複糾結）

- **IR 上三值邏輯（X）**：暫不做（§1.4）—— 改用「IR 跟 S1 共用 NodeStates、初值 = S1 settle 後的穩態」這個簡化；撞到再上。
- **直接全網表 BDD / SAT-based 等價**：不採用 —— 2C02 的 ratioed / dynamic / tri-state 區會讓 BDD 爆 / 表達不出 strength；hybrid 策略才保證專案推得動（Gemini 也認同）。
- **`Expr` IR 帶 driving-strength 維度**（做完整的 MOSSIM 式 strength resolution）：暫不做 —— bus-fight 區域直接 hybrid 處理掉；之後若 hybrid 佔比太高再考慮。

## 6. 參考資料（S2 用得到的）

- **Perfect6502（Michael Steil）** —— Visual6502 的 C 高效轉譯版。看 `netlist_to_c.py`：它有實用的「節點狀態預先計算」+「pass-transistor 網格化簡」heuristic，已解掉 6502 內部網路的串並聯 + 雙向推導 → **S2.1 開工前先 clone 進 `temp/` 研究一下**（`gh repo clone mist64/perfect6502 temp/perfect6502_ref` 或類似）。
- **Bryant, "MOSSIM II" / "A Switch-Level Model and Simulator for MOS Digital Systems"（1980s）** —— 開關層級模擬的經典；**signal strength**（Driven / Pull-up / Floating-capacitance）的概念。我們的 `Expr` IR 沒有 strength 維度 → bus-fight → hybrid（理由見 §1.2 / §5）。
- **Javitto / QuietSnooze 等的 2C02 逆向分析** —— 手動標出哪些是 dynamic latch、哪些是 domino logic。S2.3b 抓出詭異 SCC、或 S2.2 標 hybrid 時，對照他們的 schematic 判斷「演算法錯 vs 真的需要 hybrid」。
- 我們自己的 `ref/visual6502-master`（chipsim.js / wires.js / expertWires.js）、`ref/metalnes-main`（wire_module.cpp，S1 的來源）、`MD/note/01`（recalc 演算法走查）。

1. **S2.3 拆成 S2.3a（帶條件依賴圖）+ S2.3b（用兩相時鐘打斷假 SCC）** —— 不要對純靜態圖跑 Tarjan（會得到跨多個 latch 的巨大無意義 SCC）；conditional dependency graph + `clk0`/`clk1` 互斥來判「跨 phase pipeline ≠ 組合回授」。← §2 S2.3a/b、§1.2。

3. **S2.0 加 `ACTIVE_PULLUP` 辨識（superbuffer/push-pull）**；S2.2 對這種 node 用 `Mux(pullup_cond, 1, Mux(pulldown_cond, 0, Hold))` + assert `!(pullup&&pulldown)`。← §1.2、§2 S2.0/S2.2。
4. **precharge 的 pull-up 要辨識成 conditional（clocked）pull-up，不是 static load**；Expr 形 `Mux(precharge_cond, 1, Not(pulldown_eval))`。← §1.2、§2 S2.1/S2.2。
5. **charge sharing**：兩端都 floating 的 pass 導通 → 警告 + hybrid（6502 真依賴的場景極少）。← §1.2、§2 S2.2。
6. **等價驗證的 X / 初始化問題**：採「IR 跟 S1 共用 NodeStates、初值 = S1 settle 後的決定性穩態、從穩態開始逐 cycle 對」（Gemini 的選項 B），不上三值邏輯（選項 A 暫不做）。← §1.4、§5。
7. **S2.5 hybrid bridge 的效能隱患**：邊界守在 latch 上（前 half-cycle 穩定的 latch 訊號 → 不用來回迭代）；加迭代次數監控、>5 次/cycle 就警告。← §2 S2.5。
8. **參考資料**：perfect6502 `netlist_to_c.py`（pass-grid 化簡 + 狀態預算 heuristic）、Bryant MOSSIM II（signal strength）、Javitto/QuietSnooze 2C02 逆向。← §6。

**不採用的**：IR 上三值邏輯（選項 A）—— 太重，先用選項 B；`Expr` 帶 strength 維度 —— bus-fight 直接 hybrid。（理由記在 §5。）
