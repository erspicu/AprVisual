# 執行路線圖 — 四階段

> **這是目前的執行計畫**。它把 `02_實作路線圖與TODO.md` 的 Phase 0–10 重新組織成 4 個有明確「過關條件」的大階段，並把整個計畫**錨定在「先把 MetalNES 改寫成 C# 版本」**這個決策上。

## 總覽

| 階段 | 名稱 | 一句話 | 過關條件（gate） |
|------|------|--------|------------------|
| **S1** | C# 改寫 MetalNES 核心 | 把 `ref/metalnes-main` 的開關層級模擬引擎改寫成 C#，當作後續一切的**基礎平台** | C# 版能跑通 MetalNES 跑得過的 blargg test ROM；對同一參考序列，節點 trace 與 MetalNES / chipsim.js 一致 |
| **S2** | netlist 高階化成 IR | 從開關層級網表**抽取**出組合邏輯 + 序向更新規則（`Expr` IR） | IR 直譯器對同一批 test ROM 的 trace 與 S1 開關層級直譯器**逐節點完全一致** |
| **S3** | CPU 跑、證明可行 + 證明效率大幅改善 | IR 版只在 CPU 上跑，**用 benchmark 證明它比純 netlist 版快很多**，且正確性不打折 | benchmark 報告：相同 ROM、相同 cycle 數，IR-CPU 的 cycles/sec ≫ netlist-CPU；同時所有 test ROM 仍 pass。**這是 S4 的 go / no-go** |
| **S4** | Codegen + GPU 加速 | 從 IR emit C++ / Verilog / **CUDA bit-sliced kernel**，做大規模平行模擬 | CUDA backend 與 CPU IR 直譯器**逐節點等價**（單 instance 與整批都要）；有「N 台 NES / 秒」的吞吐量數字；codegen 流程可重現 |

每個階段之間都有一道**等價驗證 gate**——這正是設計文件 `04_風險_驗證_可行性.md` 把「驗證不足」列為三大失敗點之一的對應防線。**沒過 gate 不准進下一階段**。

範圍策略（貫穿四階段）：

- **先 2A03（CPU）、後 2C02（PPU）**。S1 可以把兩顆都搬進來（MetalNES 本來就有），但 **S2 的 IR 抽取從 2A03 開始**，PPU 延後（PPU 的 shift register / sprite evaluation / CHR bus 複雜得多）。
- **只 NROM（mapper 0）**。跟 MetalNES 一樣（它就是 "No MMU support"）。
- mapper、AI 大規模訓練環境、「整顆 NES 常駐 GPU kernel」都是 S4 之後或 stretch goal。

## S1 — C# 改寫 MetalNES 核心（基礎平台）

**目標**：得到一個能跑、且驗證過的 C# 開關層級模擬器。這是「繼續前進的基礎」——後面 S2/S3/S4 都站在它上面。

**為什麼是改寫 MetalNES 而不是從 chipsim.js**：MetalNES（`ref/metalnes-main`）已經是 chipsim.js 演算法的工程化升級版（flags 查表、transistor 預分類、SoA、雙緩衝 recalc list、收斂式 settle），而且把 bus 整合、真實 reset、support chip、handler 機制、test ROM 驗證全做完了。詳見 `MD/note/`。我們抄它的架構，不重新摸索那 4 個月。

### S1 子步驟

| # | 內容 | 對應 MetalNES / 設計文件 |
|---|------|--------------------------|
| S1.1 | **Parser**：讀網表，建 graph。先支援 MetalNES 的 `.js` 模組格式（`data/system-def/`），或 Visual6502 原始三件套。**保留** transdefs 第 7 欄 weak/depletion flag、segdef 的 `+` / `-` pull marker、節點名稱（含 `/` `#` `~` `_` 前綴） | `wire_defs.cpp` / `02_模組化網表系統.md` |
| S1.2 | **Graph 模型**：`NetNode` / `NetTransistor` / `NetlistGraph`。同時保留兩種關係：**控制**（誰是誰的 gate）與**通電**（導通時誰跟誰短接）。`NetTransistor.IsWeak` 來自第 7 欄 | `wire_module.h` 的 `wire_transistor` / 設計文件 §5 |
| S1.3 | **求值引擎**：把 `wire_compute` 改寫成 C#——node group BFS、`getNodeValue` 優先鏈、flags→state 256-entry 查表、transistor 預分類成「對普通節點 / 對 GND / 對 VCC」三組、SoA（hot `byte[] nodeStates` vs cold `Node[]`）、雙緩衝 recalc list + hash 去重、迭代到收斂（設上限 + 超出 warning）。**改進**：`NodeValue` 分四態（`Low` / `High` / `HoldPrevious` / `Undefined`），不要學它把 hold 跟 high 混在同一個 flag；**真的用** `IsWeak` 區分強拉低 vs 弱拉高 | `wire_module.cpp` ~L1500–2030 / `01_模擬核心演算法.md` / 設計文件 §8 |
| S1.4 | **module 組裝**：connection = gate 接 VCC 的永遠導通電晶體；instance node-id 空間配置（對齊切段、本地 id → 全域 id）；名稱解析（`a[7:0]` / `a[]` / `x\|y\|z` / `*wildcard` 展開）；`wire_register` 包裝一組 node 當整數讀寫 | `Wires::addInstance` / `addConnection` / `wire_node_resolver.cpp` |
| S1.5 | **handler / 系統層**：時鐘 handler（每 step 翻轉 clk 節點）；RAM/ROM 行為級 handler（watch cs / /we / addr / data，做 array 讀寫並回驅資料腳）；callback = 假電晶體連假節點；**真實 reset**（`setHigh(res); step(192); setLow(res)`，不是「假裝 NOP」）；iNES 載入 + NROM | `handler_*.h` / `system.cpp` / `03_系統整合與週期推進.md` / 設計文件 §11.3、§12 |
| S1.6 | **trace 基建**：指定一串節點/register 當欄位，每個 trace 點 dump 一行（像邏輯分析儀）；環形緩衝；可 dump 成 `cpu_eval_trace.json` | `handler_log` / `logger.cpp` / 設計文件附錄 B |
| S1.7 | **驗證**：(a) 子電路 truth-table 測試（手刻 inverter / NAND / pass transistor / dynamic latch，或直接用 MetalNES 的 4021 / pslatch 網表）；(b) 用同一套 `nes-test-roms` + `$6000` 簽章自動偵測；(c) 每幀 dump `cpu.a/x/y/p/s/pcl/pch`、`cpu.ab/db` 對照已知正確值或 Perfect6502 | `chip_tests.cpp` / `handler_nes_system::check_unit_test` / `04_驗證與測試策略.md` / 設計文件 §13 |

### S1 過關條件

- ✅ C# 版能載入 2A03（至少）的網表並建出 graph，node/transistor 計數對得上
- ✅ 子電路 truth-table 測試全 pass
- ✅ 跑得通 MetalNES 跑得過的那批 blargg test ROM（CPU 類至少：`blargg_nes_cpu_test5`、`instr_test-v3` 的一部分），`$6000` 簽章顯示 pass
- ✅ 對某個固定參考序列（例如 reset 後前 N 個 cycle），C# 版的 `a/x/y/p/s/pc` trace 與 MetalNES（或 chipsim.js）**逐 cycle 一致**
- ✅ 有 trace dump 能力（給後面比對用）

> 過了這道 gate，我們就有了「繼續前進的基礎」。**注意：這一階段刻意不碰 IR、不碰 GPU。** 它的唯一目的是「有一個我們完全掌握、驗證過的開關層級模擬器」。它也順便成為後面所有等價驗證的 golden reference。

## S2 — netlist 高階化成 IR

**目標**：從開關層級網表**抽取**出一個更高階的表示——每個節點的「下一個狀態 = 某個布林表達式 / 保留前值」——並用一個「IR 直譯器」執行它，行為要跟 S1 的開關層級直譯器一模一樣。

**這是整個專案最有價值、也最危險的部分**（設計文件 §15.3 + `04` 號文件的失敗點 #1：dynamic node 抽象錯誤）。MetalNES **完全沒做這層**——它永遠是直譯開關層級。我們要走到這裡。

### S2 子步驟

| # | 內容 | 對應設計文件 |
|---|------|--------------|
| S2.1 | **靜態圖建構**：標記供電節點（VCC/VSS/GND）、外部 pin（`clk0` `res` `rw` `ab[]` `db[]` …）、建鄰接 | §6.1 |
| S2.2 | **通電圖 / 浮島分析**：對每個 clock phase，決定哪些電晶體 ON，算連通分量；判斷每個節點/分量是「連到 GND」「連到 VCC/pull-up」還是「浮島」 | §6.2、§7.3 |
| S2.3 | **迴路 / 回授偵測**（MetalNES 跳過的部分）：三色 DFS / Tarjan SCC，把回授群縮成超節點，分類為 **static cross-coupled latch / dynamic storage / bus-style 回授**。停止遞迴防無限展開，但更重要的是**把序向邏輯與組合邏輯分開** | §6.4、§10 |
| S2.4 | **布林表達式抽取**：對良性區域（反相器、NAND/NOR、串並聯下拉網路、明確的 precharge/evaluate 區）→ 抽成 `Expr` 樹（串聯=AND、並聯=OR、有路徑到 GND→0、無下拉且有 pull-up→1）；對 dynamic 區域 → `HoldExpr`；對回授 → `NodeRefExpr` 指向「上一輪狀態」。**先做良性區域，不要從共享 bus / 強耦合 dynamic 叢集開始** | §8.1、§8.3、§11–§13 |
| S2.5 | **IR 資料結構**：`Expr` = `ConstExpr` / `NodeRefExpr` / `NotExpr` / `AndExpr` / `OrExpr` / `MuxExpr` / `HoldExpr`；每個節點一條 next-state 表達式；標記哪些是組合（可拓樸排序）、哪些是序向（在 clock edge 更新） | §8.2 |
| S2.6 | **IR 直譯器**：拓樸排序組合邏輯，在 clock phase 邊界跑序向更新；保留兩相時鐘、half-cycle、必要時迭代 settle | §11 |
| S2.7 | **等價 gate**：IR 直譯器對同一批 test ROM 的 trace 必須與 **S1 開關層級直譯器逐節點完全一致**。任何不一致 → 回 S2.3/S2.4 修。也要能 dump `ir_debug.json` 供人工檢視 | §13.6、附錄 B |

### S2 過關條件

- ✅ IR 抽取對 2A03（至少 CPU 核心的大部分）完成，能 dump `ir_debug.json`
- ✅ IR 直譯器跑得通 S1 跑得過的那批 test ROM
- ✅ IR 直譯器與 S1 開關層級直譯器的 trace **逐節點、逐 cycle 完全一致**（這是硬條件，不是「看起來差不多」）
- ✅ 迴路 / 回授區域有被正確分類（不會因為強制展開而表達式爆炸或不收斂）

> 註：如果某些區域（典型是共享 bus、強耦合 dynamic）抽不出乾淨的 IR，可以暫時對那些區域**保留開關層級求值**（hybrid：大部分跑 IR，少數難搞區域跑 group 演算法），但要記錄下來、標清楚，並在 S2 過關條件裡放寬到「IR 覆蓋率 ≥ X%、其餘 hybrid，整體 trace 仍一致」。

## S3 — CPU 跑、證明可行 + 證明效率大幅改善

**目標**：IR 版**只在 CPU 上**跑，達成兩件事：
1. **證明可行**——IR 版在真實 workload（test ROM、實際遊戲 ROM 的前幾秒）上正確
2. **真的證明效率大幅改善**——用 benchmark 證明 IR-CPU 比 S1 的 netlist-CPU 快**很多**（一個數量級或更多）

這一階段不碰 GPU。它是 S4 的 **go / no-go**：如果 IR 版在 CPU 上沒有顯著快過 netlist 版，那 GPU 加速的前提就動搖了（因為 GPU 版本質上就是「同一個 IR、bit-sliced 平行跑」）——那時應該回頭檢討 IR 抽取夠不夠 aggressive，而不是硬上 GPU。

### S3 子步驟

| # | 內容 |
|---|------|
| S3.1 | **IR 優化**：拓樸排序、常數摺疊（`A && true ⇒ A`）、共同子式消除（CSE）、死碼消除、可能把 IR 攤平成一串線性指令（拿掉 tree 的指標間接），讓 CPU 直譯器跑得快 |
| S3.2 | **benchmark harness**：固定一顆 ROM、固定 cycle 數，量 (i) S1 開關層級直譯器 vs (ii) S3 優化後 IR 直譯器 的 **cycles/sec**（或 NES-frame/sec）。兩者都單 instance、單執行緒、同機器、同編譯設定 |
| S3.3 | **正確性 gate**：IR 直譯器在 benchmark 之外，仍要跑得通 S1/S2 跑得過的全部 test ROM——速度不能用犧牲正確性換 |
| S3.4 | **數字 + 解釋**：寫一份報告，列出 speedup 倍數、量測方法、為什麼變快（不用每 cycle 重建 node group、不用走 graph、只是 evaluate 一個預編譯好的表達式 DAG）。倍數多少算「大幅改善」沒有硬標準——重點是「顯著、可重現、有方法論」，目標心理值是 ≥ 10×（若只有 2–3× 要認真檢討） |
| S3.5 | **（順帶）** 也可以順便 emit C++（S4.1 的暖身），編譯起來跟 IR 直譯器比對，看看「編譯成原生碼」又能再快多少——但這不是 S3 的過關條件 |

### S3 過關條件

- ✅ 一份 benchmark 報告，明確顯示 **IR-CPU 的 throughput ≫ netlist-CPU**（相同 workload、相同正確性），附量測方法與解釋
- ✅ IR 直譯器仍 pass 所有 S1/S2 的 test ROM
- ✅ 對「IR 為什麼快」有清楚的技術解釋（不是「看起來比較快」）

> **這是分水嶺**。過了 → 進 S4，GPU 加速有了堅實前提（「同一個 IR 攤平、bit-sliced、SoA、丟上 GPU 平行跑」）。沒過 → 不要進 S4，回頭看 IR 抽取（S2.3/S2.4）是不是不夠 aggressive、有沒有太多區域 fallback 到 hybrid 開關層級。

## S4 — Codegen + GPU 加速

**目標**：從 IR emit 可執行 backend。主軸是 **CUDA bit-sliced kernel**——一個 `uint32`/`uint64` 字裝 32/64 台 NES 的同一個節點，`&` `|` `~` 取代 `&&` `||` `!`，SoA 布局 `state[node][instance_word]`，一條表達式同時算 32/64 台。

### S4 子步驟

| # | 內容 | 對應設計文件 |
|---|------|--------------|
| S4.1 | **C++ codegen**（sanity check）：從 IR emit C++，編譯，驗證跟 IR 直譯器逐節點一致 | §9.2 |
| S4.2 | **Verilog codegen**（選做）：從 IR emit 結構化 / 可合成 Verilog，給 FPGA 導向流程或第三方等價驗證用 | §9.1 |
| S4.3 | **CUDA codegen（bit-sliced）**：每個節點的 next-state 表達式 → device 端 `uint32/64` 位元運算；所有節點更新收進一個大 device function；SoA layout；一條 word = 32/64 instances | §10.1、§10.2、§23–§24 |
| S4.4 | **CUDA runtime**：時鐘 driver、批次 step（每次 launch 推進固定數量 half-cycle）、記憶體 handler（每 instance 的 ROM/RAM 在 VRAM）、reset；先做「批次 evaluator」**不要**一開始就做「整顆 NES 常駐 kernel」 | §10.3、§14 |
| S4.5 | **等價 gate**：CUDA 直譯器與 CPU IR 直譯器**逐節點等價**——單 instance 要一致，整批（32/64 台）每一台也都要一致。用同一份 IR，跑同樣輸入，比對 | §13.6 |
| S4.6 | **吞吐量 benchmark**：一張 GPU 上能平行跑幾台 NES / 秒；跟 S3 的單 instance CPU 數字對比，算出有效加速 | §26 |
| S4.7 | **（stretch）** 整顆 NES 常駐 GPU kernel——host 只餵 input、讀 framebuffer/audio；給 AI 訓練環境 / Golden Model 大批量驗證用 | §27 |

### S4 過關條件

- ✅ codegen 流程可重現（同一份 IR 跑出同一份 CUDA / C++）
- ✅ C++ backend 與 IR 直譯器逐節點一致
- ✅ CUDA backend 與 CPU IR 直譯器逐節點等價（單 instance + 整批）
- ✅ 有「N 台 NES / 秒」的吞吐量數字，且相對 S3 單 instance CPU 有可量化的加速
- ✅（若做了 stretch）常駐 kernel 跑得起來、輸出正確

## 一句話總結

> **S1** 站在 MetalNES 肩膀上得到一個我們完全掌握的 C# 開關層級模擬器（基礎）；**S2** 把它高階化成 IR（最有價值、最危險）；**S3** 在 CPU 上證明 IR 版又對又快很多（go/no-go 分水嶺）；**S4** 從 IR emit CUDA，做大規模平行（終局）。每一步之間都有逐節點等價驗證，沒過不准前進。
