Excellent, this is a superb question. You've done the hard work of diagnosing the "why" behind the performance regressions, which is more than half the battle. Your diagnosis is sharp and, I believe, entirely correct. Let's get concrete.

### Executive Summary

**The JIT's full-inline mega-method is almost certainly the local optimum for this code shape.** Your manual fusions are losing because they create control-flow merge points that prevent the JIT from creating hyper-specialized code paths for the high-frequency cases. The generalization tax is real and you're paying it in execution latency and register pressure.

There is **one plausible "live" restructuring** (the hybrid model), but its chances are slim. The real, substantial wins are no longer in code restructuring but in **algorithmic changes to reduce dependent loads**.

---

### 1. The "Live" Restructuring: The Hybrid Model

Your intuition here is correct. The #1 goal is to protect the 70% singleton path from *any* form of generalization tax. The only way to do this is to give it a completely independent, straight-line code path with no `goto`s, shared buffers, or shared loop constructs.

This leads to the hybrid model: **Isolate the 70% singleton path, and unify the 30% pair and <few% group paths.** The pair/group paths are more alike (they both process a "list" of nodes, of size 2 or N), so the generalization tax of handling them together is much lower than forcing the singleton through that machinery.

#### Exact Code Shape

The goal is to produce a control-flow graph (CFG) that the JIT sees as two distinct top-level branches from the dispatch.

```csharp
// Inside your ProcessQueue loop: while (currentCount < limit)
int nn = RecalcList[currentCount++];

// DISPATCH
// The JIT sees this 'if' as the primary fork.
// Path 1: The Golden 70% Path
if (IsSingleton(nn)) // Assuming a cheap check
{
    // NO BUFFERS, NO GOTOS, NO SHARED LOGIC.
    // This block is a direct, linear copy of the logic from the baseline's
    // inlined singleton path.

    //    (read gnd/pwr gates -> flags -> newState)
    byte oldState = NodeStates[nn];

    if (newState != oldState)

        if (newState > oldState)

            // Directly read/write rS/rA/rB, RecalcListNext, RecalcHashNext.
            // Use 'nextCount' directly.
            // ... ~12 lines of specialized walk logic ...
        }
        else // Specialized walk for gate-going-low

            // Walk fan-out for 'nn' going low.
            // ... ~12 lines of specialized walk logic ...
        }
    }

    // The store-to-load latency and loop overhead is now only paid
    // on 30% of the nodes.

    Span<ushort> groupNodes = stackalloc ushort[MAX_GROUP_SIZE]; // Or a small reusable buffer

    }
```

**Verdict:** This is the only micro-architectural change with a chance. It quarantines the performance-critical path. You still have code duplication in your C# source (the walk logic appears in the `if` and `else` blocks), but that's what allows the JIT to create two *different*, specialized, and smaller bodies of machine code without the CFG-merge-point penalty. **You are trading C# source-level duplication for runtime specialization.**

### 2. The Wave-Level `nextCount` Hoist

**This is almost certainly a net loss on a memory-latency bound workload.** Your diagnosis of register pressure is spot on.

Let's be quantitative on Zen2:
*   **Cost of NOT hoisting:** Per node-pop, you have `mov ecx, dword ptr [RecalcListNextCountAddr]`. This is a single uop. If `RecalcListNextCountAddr` is in L1 cache (and it is), the latency is ~4-5 cycles. In a memory-latency bound loop, your pipeline is stalled waiting for `NodeStates` data anyway. This 4-cycle dependency is likely hidden completely (`stalled-cycles > 4`). The cost is effectively zero.
*   **Cost OF hoisting:** You load it once (`mov r10d, dword ptr [addr]`). Now, `r10d` must be preserved across the entire processing of every node in the wave. The JIT's register allocator now has one fewer general-purpose register (`r10`) to work with for the *entire duration* of the hot loop. In a loop that's juggling pointers to `NodeStates`, `NodeInfos`, `RecalcList`, `RecalcHash`, and various computed values, one register is gold. Losing it will very likely cause a spill of a 64-bit pointer to the stack, costing you a `mov [rsp+XX], rbx` (store) and later a `mov rbx, [rsp+XX]` (load). This round-trip to the stack L1 cache is more expensive and introduces more pipeline pressure than the trivial load of the static.

**RyuJIT Specifics:** When you force the hoist by using a local variable, you are overriding the JIT's own cost/benefit analysis. The JIT's register allocator is good, but it's not magic. If you give it N live variables and it only has N-1 registers, it *must* spill. By hoisting, you increased the "live range" of `nextCount` from "just inside the walk logic" to "the entire `ProcessQueue` loop," forcing it to compete with variables from all three paths (singleton, pair, group) simultaneously.

**Verdict:** Pure register-pressure cost with no benefit in a memory-bound loop. Do not do this. Let the JIT reload the L1-hot static as needed.

### 3. `ref`-threading to Force Enregistration

This is a clever idea, but it will not beat the JIT and will likely perform worse.

Passing `ref int` is passing a managed pointer. Inside an inlined method, the JIT sees this as an indirection (`mov ecx, [rax]`). While the JIT is capable of "value numbering" this and sometimes enregistering the *value* from the address, an address-exposed variable is poison for many optimizations. The JIT has to be more conservative and assume that any indirect memory write or method call could potentially modify the value at that address, forcing a reload.

You are essentially trading a `mov ecx, [STATIC_ADDR]` for a `mov ecx, [reg_with_addr]`. The performance is identical, but the `ref` version has a higher chance of pessimizing the JIT's alias analysis. You are adding a level of indirection and complexity for the optimizer to reason about, with no upside over the simple local-variable hoist (which we've already established is a net loss).

**Verdict:** Unwinnable. This combines the register pressure problem of hoisting with potential JIT pessimization due to address-exposed variables.

---

### 4. Bottom Line: Unwinnable by Fusion. Pivot to Algorithmic.

**Yes, beating the JIT's auto-inline cascade by manual fusion or restructuring is fundamentally unwinnable here.** The baseline is fastest precisely *because* `[AggressiveInlining]` lets the JIT see the entire CFG and perform inter-procedural optimizations (like register allocation and code layout) with a global view. Your attempts to shrink the code by sharing logic are counter-productive because they hobble the JIT's ability to specialize.

**The only path forward is to reduce the fundamental work per pop.** The bottleneck is dependent loads. That's your target.

#### Concrete Algorithmic/Data-Structure Changes to Investigate:

1.  **Rank 1: Data Layout (AoS vs SoA).** This is the highest-impact change. You are currently doing Structure-of-Arrays (SoA): `NodeStates[id]`, `NodeInfos[id]`. This means processing node `id` requires at least two random memory accesses, which are likely L2/L3 misses or worse.
    *   **The Move:** Switch to Array-of-Structures (AoS).
        ```csharp
        struct NodeData
        {
            public byte State;
            // Pack the MOST FREQUENTLY ACCESSED fields from NodeInfo here.
            public ushort GndGate, PwrGate; // Example
            // Other flags...
        }
        
        NodeData[] AllNodes = new NodeData[NUM_NODES];
        ```
    *   **The Win:** Now, accessing all the critical data for `AllNodes[id]` is a single memory access and a single cache line fill. This directly attacks your memory latency bottleneck. The cost is that scanning a single field across all nodes is now slower, but your access pattern is random-access-by-id, which is perfect for AoS.

2.  **Rank 2: Software Prefetching.** Since you are memory-latency bound, you can hide that latency by fetching data before you need it.
    *   **The Move:** When you are in the "walk its gated fan-out" step for node `nn`, you are about to enqueue its children. At the moment you identify a child `c`, you know you will process it in the next wave. Issue a prefetch for it *immediately*.
        ```csharp
        // In the fan-out walk, when you find a child node ID 'childNodeId'
        // that you are about to add to RecalcListNext...
        
        if (Sse.IsSupported)
        {
            // Prefetch the NodeData for the child into all cache levels.
            // You need to get the address of AllNodes[childNodeId].
            // This is unsafe code, but it's the only way.
            fixed (NodeData* pNodes = &AllNodes[0])
            {
                Sse.Prefetch0(&pNodes[childNodeId]);
            }
        }
        
        // ... now add childNodeId to the queue.
        RecalcListNext[nextCount++] = childNodeId;
        ```
    *   **The Win:** By the time the CPU gets to processing `childNodeId` in the next wave, its `NodeData` has a high probability of already being in L1/L2 cache, turning a long-latency miss into a short-latency hit. This is an advanced technique that the JIT will not do for you, and it's a classic way to gain performance in pointer-chasing, memory-bound simulations.

**Final Verdict & Ranked Options:**

1.  **(Highest ROI) Algorithmic Change:** Restructure your core data from SoA to AoS. This is the most likely path to a significant (5-20%+) performance win.
2.  **(High ROI, High Complexity) Algorithmic Change:** Implement software prefetching for child nodes during the fan-out walk. This complements the AoS change.
3.  **(Low ROI, Worth a Try) Micro-architecture:** Implement the Hybrid model described in Q1. It correctly targets the 70% case and might claw back a few percent over the baseline by slightly reducing code size for the JIT to chew on, without compromising the fast path.
4.  **(Negative ROI) Do Not Pursue:** Any further attempts at manual fusion, `goto`-based sharing across all paths, hoisting L1-hot statics, or `ref`-threading. These are confirmed dead ends. Trust your own measurements.
