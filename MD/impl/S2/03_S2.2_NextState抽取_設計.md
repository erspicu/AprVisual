# S2.2 — `Expr` next-state 抽取（`nextExpr[N]`）

> S2 的第 2 步：把 S2.1 的 `DriveInfo[N]`（`PullDown` 條件 / `PullUpKind` / `Passes`+方向 / `Hybrid` 旗標）合成成 **`nextExpr[N]`** —— node N 下一拍的值的布林表達式。順便：標每個 node **組合 vs 序向**（初判，S2.3 精修）；算 **IR 覆蓋率**。**還不碰 SCC（S2.3）；不建 IR 直譯器（S2.4）；不改模擬行為。**
>
> 工作流程（`00_計畫總覽.md` §3）：① 設計描述 → ② 丟 Gemini → ③ 評估採用 → ④ 實作 → ⑤ 驗證。
>
> **承接 S2.0/S2.1**：node 角色 Supply/Input/Bus/Internal；transistor kind Pass/PullDown/PullUp{Strong,Load,Active}/Dead；depletion load = `node.Pullups` 標記不是電晶體；`DriveInfo` 14649 個（Internal+Bus）；`PullDown` complex=0（全乾淨串並聯）；`PullUp` None=7488/StaticLoad=6396/StrongVcc=5/Conditional=760；transmission-gate pass 7924（Bidir 5574）；**Hybrid 5774（39.4%）**（主因 cross-coupled latch —— S2.3 會 reclassify）；`RefineBuses` 把 40 個 `_d[]` pin 標 Bus。

## ① 設計描述

### 1.1 合成規則（依 NMOS 解析優先鏈：GND 贏 → VCC/pull-up → 外部驅動 → 保留前值 → 0）

對 node N（有 `DriveInfo` di）：

**(A) `di.Hybrid` → `nextExpr[N] = null`**（= 這個 node 由 S2.5 的 hybrid bridge 用 switch-level 算；IR 引擎看到 `nextExpr == null` 就知道）。

**(B) 非 hybrid**：拆 di 的「驅動源」，照優先序組：
- `pd = di.PullDown`（可能 `null` = 無下拉路徑，或一個 `Expr`；不會是 `Complex` —— Complex 已被標 hybrid）。
- `inPasses` = `di.Passes` 裡 `OwnerDrives == false` 的（= 對方驅動 N）；非 hybrid 的 N，passes 的 `OwnerDrives` 都非 null（null 的那些 —— bidirectional / charge-sharing / 碰 bus —— 都已 `MarkHybrid`）。`OwnerDrives == true` 的（N 驅動對方）不影響 N 的值，忽略。
- `puFallback` = pull-up 帶來的「沒被下拉、沒被 pass 驅動時」的值：
  - `StaticLoad` / `StrongVcc` → `Const(true)`（恆拉高；弱或強對布林值都一樣）
  - `Conditional` → `Mux(di.PullUpCond, Const(true), Hold(N))`（precharge：clock on → 1，否則 hold —— domino 求值後保持的 1）
  - `None` → `Hold(N)`（純 dynamic：沒驅動就 hold）
- **組起來**：
  - `passChain` = `inPasses` 由後往前折：`fallback`；對每個 `(M, cond)` → `Mux(cond, Node(M), 上一層)`。沒 inPasses → `passChain = puFallback`。
    - ≥2 個 inPasses：first-wins（巢狀 `Mux`）—— **Gemini 確認這是對的**：mux 的 select 在乾淨數位設計裡必然互斥（one-hot），順序不重要、邏輯等價；同時導通且值不同就是實體 short → 不該 fallback 到 hybrid（會大幅掉覆蓋率）。實作時註解標明「Assumes mutually exclusive (one-hot) pass selects」。
  - `pd == null` → `nextExpr[N] = passChain`（沒下拉路徑 → 永遠不會被拉 0）。
  - `pd != null` → `nextExpr[N] = Mux(pd, Const(false), passChain)`（下拉導通 → 0；GND 贏，蓋過 pull-up 和 pass —— 因為導通的 pass 把 M 併進同一個 group，但 N 自己的下拉讓 group 有 Gnd flag → 0）。
- **特例化（讓常見情況的 Expr 乾淨）**：
  - `pd != null && pu ∈ {StaticLoad, StrongVcc} && inPasses 空` → `nextExpr[N] = Not(pd)` ← 教科書 inverter / NAND / NOR / AOI（`Mux(pd, false, true)` 由 `Mux` 的 smart ctor 收成 `Not(pd)`）。
  - `pd == null && pu ∈ {StaticLoad, StrongVcc} && inPasses 空` → `Const(true)`（恆拉高、無下拉）。
  - 其餘照上面的通式。

> 關鍵語義（為什麼 pull-down 蓋過 pass、pass 蓋過 pull-up）：switch-level 裡導通的 pass 把 N 和 M 併成同一個 group → group 值 = `resolve(group 內所有 flag)`。N 自己的下拉導通 → group 有 Gnd flag → 0（不管 M）。N 沒下拉但 pass 導通 → group = {N, M}，N 的 pull-up 是弱 flag、M 的值是 State/PullUp/Gnd flag → M 的值蓋過 N 的弱 pull-up（M=0 → 0；M=1 → 1）。N 沒下拉、pass 沒導通 → group = {N}，pull-up → 1（或 hold）。

### 1.2 組合 vs 序向（初判；S2.3 精修）

- N 是**序向**若 `nextExpr[N]` 裡出現 `Hold(N)`（它保留自己的前值 → latch）。
- 否則 N 暫標**組合**。
- **註**：cross-coupled latch（N 的 expr 引用 /N、/N 的 expr 引用 N —— 誰都沒 `Hold(self)`，但兩個合起來是個環）—— 這需要 dependency graph + SCC，是 **S2.3** 的事。S2.2 只做「有 `Hold(self)` → 序向」的初判；S2.3 會把 SCC 裡的 node 也標序向、把環打斷成 `Prev(...)`。
- hybrid node（`nextExpr == null`）→ 既不算組合也不算序向（它走 switch-level）。

### 1.3 IR 覆蓋率

- `coverage = #(nextExpr != null) / #(有 DriveInfo 的 node)` = `1 - hybrid率`。S2.2 不改 hybrid 集合（S2.1 標的那些）→ 覆蓋率 ≈ 60.6%（跟 S2.1 一致；S2.3 處理 SCC 後才會跳上去 —— 那些 cross-coupled latch 目前被標 hybrid，S2.3 reclassify 成序向後就「有 nextExpr」了）。
- 也報：`Hold(self)` 引用了哪些 node（序向 node 初判數）、`nextExpr` 是 `Const`/`Not(...)`/`Mux(...)`/`Hold(...)` 各幾個（結構分佈）。

### 1.4 程式落點

- `Expr.cs` 加 smart ctor：`Expr.Mux(cond, a, b)`、`Expr.Hold(id)`（→ `new HoldExpr(id)`）、`Expr.Prev(id)`（→ `new PrevExpr(id)`，給 S2.3 用）。`And`/`Or`/`Not` 已有；`Pretty()` 已涵蓋 `Mux`/`Hold`/`Prev`。`Expr.Mux` 的化簡規則（含 Gemini 建議的 4 條 Mux→And/Or）：
  - `Complex` 傳染 → `Complex`
  - `Mux(True, a, b) = a`；`Mux(False, a, b) = b`；`Mux(c, a, a) = a`
  - `Mux(c, True, False) = c`；`Mux(c, False, True) = Not(c)`
  - **`Mux(c, False, b) = And(Not(c), b)`**；**`Mux(c, a, False) = And(c, a)`**；**`Mux(c, True, b) = Or(c, b)`**；**`Mux(c, a, True) = Or(Not(c), a)`** —— 把「pull-down 樹 / pass 樹解析後初看像巢狀 Mux」的 AOI/OAI/enable 邏輯瞬間壓平成標準 And/Or/Not（可讀性 + S2.4 直譯器效能）。例：`pullUp=StaticLoad` + `pullDown=(A|B)` 的 node → `Mux(Or(A,B), False, True)` → `Not(Or(A,B))` = NOR 閘 ✓。
  - **不化簡 `Mux(c, a, Hold(self))`** —— 保留 Mux 語義（D-latch / enable-latch 的標準特徵，對 S2.3 後續分析最有利）。
  - 其餘 → `new MuxExpr(c, a, b)`。
- 新檔 `src/AprVisual/Sim/Logic/NextStateBuilder.cs`：
  ```csharp
  internal sealed class NextStateModel
  {
      public Expr?[] NextExpr = [];     // [nodeId]; null = hybrid (use switch-level) or no DriveInfo (Supply/Input)
      public bool[] IsSequential = [];  // [nodeId]; true if NextExpr[id] references Hold(id) (S2.3 will add the SCC ones)
      public static NextStateModel Build(NetlistGraph g, DriveInfo?[] di);
      // stats
      public (int total, int comb, int seq, int hybrid, int constE, int notE, int muxE, int holdE, int other) Stats();
      public string Describe(int nodeId);
  }
  ```
- `TestRunner` 加 `--dump-next`（`ComposeSystem` → `NetlistGraph.BuildFrom` → `DriveAnalysis.Analyze` → `NextStateBuilder.Build` → 印：comb / seq / hybrid 數、IR 覆蓋率、`nextExpr` 結構分佈、sample exprs（幾個 `Not(...)` 的 = 邏輯閘、幾個 `Mux(...)` 的 = dynamic/precharge）、spot-check（`ppu.io_ce` 應該 = `Not(n14327)`、某 dynamic latch、某 hybrid node = null）。
- `--selftest` 加 `TestNextState`（手刻 inverter `y=!a` → `nextExpr[y] == Not(Node(a))`；NAND `z=!(b&c)` → `nextExpr[z] == Not(And(Node(b),Node(c)))`；一個 dynamic latch（`d --[ph]--> q`，q 無 pull-up、q 驅動一個假閘）→ `nextExpr[q] == Mux(Node(ph), Node(d), Hold(q))` 且 `IsSequential[q]`；一個 hybrid node（bidirectional pass）→ `nextExpr == null`；precharge node（有 PullUpActive + 串聯下拉）→ `nextExpr == Mux(pd, False, Mux(precharge_cond, True, Hold(self)))`）。

### 1.6 開放問題（丟 Gemini）

1. 合成規則（§1.1）對嗎？特別是「pull-down 蓋過 pass、pass（M 的值）蓋過 pull-up」這個優先序 —— 對應 chipsim 的 group resolution 嗎？有沒有漏掉的情況（N 同時有 inbound pass 和 conditional pull-up？N 有 inbound pass 但對方 M 也是 hybrid？）？
2. ≥2 個 inbound pass 的 first-wins 近似 —— 在 6502/2C02 上會碰到嗎（一個 node 被多個 mux 分支驅動）？該不該標 hybrid（保守）還是 first-wins（樂觀，靠「乾淨設計裡互斥」）？有沒有更好的（把多個 inbound pass 折成「resolve 多個條件驅動」的 Expr）？
3. 「pull-up 帶來的 fallback」對 `Conditional`（precharge）：`Mux(precharge_cond, True, Hold(N))` —— 對嗎？precharge node 在「clock on（precharge phase）」是 1、「clock off（evaluate phase）且下拉不導通」hold 那個 1、「下拉導通」是 0 —— 所以完整是 `Mux(pd, False, Mux(precharge_cond, True, Hold(N)))`。precharge_cond 是 `di.PullUpCond`（= OR of PullUpActive gates）—— 但 PullUpActive 的 gate 可能不是「乾淨的 clock」而是個推導出來的訊號，這樣 OK 嗎？
4. 組合 vs 序向的初判（只看 `Hold(self)`）—— 夠嗎？有沒有「nextExpr 沒 `Hold(self)` 但其實是序向」的情況（除了 cross-coupled SCC，那個 S2.3 處理）？
5. `Expr` 的 smart ctor（`Mux` 收 `Mux(c,True,False)=c` / `Mux(c,False,True)=Not(c)` 之類）—— 夠嗎？要不要更多（`Mux(c, a, Hold(self))` 之類的 idiom）？

每條建議標明「對這個計畫的具體影響 / 該怎麼改」。某設計 OK 不用改也直說。

## ② Gemini 分析回饋

> `gemini-3.1-pro-preview-customtools`，log：`tools/knowledgebase/message/20260512_105945.txt`。摘要 —— **設計邏輯無懈可擊，綠燈放行**：
- **§1.1 合成規則 / 優先序（PD > Pass > Pull-up）**：完全正確，精準對應 switch-level 的 group resolution。原理：pass 導通 → N、M 併入同一 group；N 自己 pull-down 導通 → group 有強烈 Ground flag（NMOS 傳 0 無損）→ 0（不管 M）；N 沒 pull-down 但 pass 導通 → N 的弱 pull-up 輸給 M 傳來的 driven state（0 或 1）→ N = M 的值；都沒導通 → 退回 N 的 pull-up 或 hold。沒漏掉的情況（「M 也 floating / 只有弱 pull-up」已被 S2.1 的 `OwnerDrives`/`Hybrid` 標記防堵 —— 非 hybrid node 的 inbound pass 對方必有明確強驅動值）。**不用改。**
- **§1.6-Q2 ≥2 inbound pass**：用 first-wins（巢狀 Mux）**正確且最佳，不要標 hybrid**。這是標準 mux（ALU 輸入選擇 / 多 reg 寫同 bus）；select 在乾淨設計裡必然 one-hot → 順序不重要、邏輯等價；同時導通且值不同就是實體 short。標 hybrid 會大幅掉覆蓋率（很多純邏輯的資料路徑被誤判）。實作時註解標明「Assumes mutually exclusive (one-hot) pass selects」。
- **§1.6-Q3 precharge 公式 `Mux(pd, False, Mux(precharge_cond, True, Hold(N)))`**：完全正確。2C02 的 OAM/sprite 確實有大量 domino；`precharge_cond` 通常來自時鐘或 clock-gating 出來的訊號；預充期 pull-down 的求值電晶體被 clock 關掉（互斥）→ `pd` 和 `precharge_cond` 正常不會同時 True；真同時了 → 公式讓 `pd`(False) 贏，符合 NMOS 物理（GND 贏 VCC）。**照做。**
- **§1.6-Q4 組合 vs 序向只看 `Hold(self)`**：對 S2.2 絕對夠（排除 cross-coupled 環後，「能記住前一拍狀態」在布林邏輯裡唯一的表現形式就是 next-state 依賴自己 = `Hold(self)`，涵蓋所有 dynamic latch）。
- **§1.6-Q5 Expr smart ctor**：**需要補上 4 條 Mux→And/Or 化簡**（見 §1.4 / ③）—— `Mux(c,False,b)=And(Not(c),b)` / `Mux(c,a,False)=And(c,a)` / `Mux(c,True,b)=Or(c,b)` / `Mux(c,a,True)=Or(Not(c),a)`。因為 `Mux(pd, False, passChain)` 這種結構在 passChain 被化簡後會出現大量 `Mux(cond, False, B)`，這些規則把 AOI/OAI/enable 邏輯瞬間壓平成標準 And/Or/Not（可讀性 + S2.4 效能）。但 **`Mux(c, a, Hold(self))` 不化簡**（D-latch 特徵）。
- 預期：`pullUp=StaticLoad` + `pullDown=(A|B)` 的 node → `Mux(Or(A,B), False, True)` → 化簡成 `Not(Or(A,B))`（NOR 閘）—— spot-check 時看到這個就證明合成規則對了。
