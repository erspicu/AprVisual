# S1 引擎演算法說明 — 資料結構與程式碼對照

> 對象:`src/AprVisual.S1/`(C# golden engine)。本文說明**目前實際在跑的**開關級(switch-level)事件驅動演算法,逐一對照資料結構與程式碼位置。
> 參考來源演算法見 `01_模擬核心演算法.md`(MetalNES / Visual6502 `chipsim.js`);本文聚焦 S1 在其上的實作與優化後的樣貌。

---

## 0. 一句話總覽

引擎把晶片當成「**節點(wire)+ 電晶體(開關)**」的網路。每半週期(half-cycle)切換一次主時脈,讓狀態變化像漣漪一樣在「導通中的電晶體群」裡傳播,反覆重算到全網路穩定(quiescent),再進下一個半週期。**沒有 opcode、沒有行為模型**,CPU 的行為是從電晶體的導通關係「長出來」的。

核心三層,由外而內:

```
StepCycle()                  一個半週期:翻轉時脈 → settle → 時間 +1
  └ ProcessQueue()           settle:一波一波重算到穩定(雙緩衝佇列)
      └ RecalcNode(nn)       單一節點重算:三類 fast-path 派發
          ├ RecalcNodeFast   O(1) 解析(群組可證明只有 {nn})
          └ ComputeNodeGroup 完整 BFS:走訪導通群 → 旗標 OR → LUT 解析
```

## 1. 資料結構總表

所有熱資料都是 **SoA(Structure-of-Arrays)+ 非託管裸指標**,在 `Reset()`(`WireCore.cs`)依節點數一次配置,程式結束一次釋放。索引一律是 node id `0 .. NodeCount-1`。

| 資料結構 | 型別 | 用途 | 宣告位置 |
|---|---|---|---|
| `NodeStates` | `byte*` | 每節點目前邏輯值 0/1。**最常讀** | `WireCore.cs:65` |
| `NodeInfos` | `NodeInfo*` | 每節點 16 bytes:旗標 + 通道鄰接(內聯或索引) | `WireCore.cs:70` / 結構定義 `WireCore.cs:289` |
| `TransistorList` | `ushort*` | 攤平的鄰接表,0 結尾的子清單(高扇出節點用) | `WireCore.cs:90` |
| `NodeConnections` | `int*` | 冷:浮接群組「電容」比大小用(<1% 才碰) | `WireCore.cs:79` |
| `NodeTlistGates` | `int*` | 冷:`SetNodeState` 回寫時用來找下游節點 | `WireCore.cs:80` |
| `RecalcList` / `RecalcListNext` | `int*` | 雙緩衝待重算清單(這波 / 下波) | `WireCore.cs:93-94` |
| `RecalcHash` / `RecalcHashNext` | `byte*` | 上面兩個清單的「在不在清單裡」成員旗標(去重 + 已處理標記) | `WireCore.cs:98-99` |
| `_groupBuf` | `ushort*` | 群組 BFS 的節點清單兼工作佇列 | `WireCore.Group.cs:30` |
| `_inGroup` | `byte*` | 群組 BFS 的 O(1) 去重旗標 | `WireCore.Group.cs:31` |
| `FlagsToState` | `byte*` (256) | OR 後的旗標 → 群組解析值的查表 | `WireCore.cs:84` |
| `IsPureLogic` | `byte*` | fast-path 分類(0/1/2),見 §6 | `WireCore.FastPath.cs:36` |

> 為什麼大量用 `byte*`/`ushort*` 而不是 `int*`:這條熱路徑是**記憶體延遲限制(memory-latency bound)**,縮小工作集讓更多資料留在 L1/L2 比省指令更有效。例如 `TransistorList` 從 `int` 改 `ushort`(node id < 65K)把最熱的陣列從 697KB 砍到 350KB;`RecalcHash` 從 int 改 byte 是 58KB→14KB。

### 1.1 特殊節點 id(`WireCore.cs:34-37`)

```csharp
public const int EmptyNode = -1;   // 不存在
public const int Npwr = 1;         // VCC
public const int Ngnd = 2;         // GND
// 0 保留為 EMPTY(子清單的 0 結尾就靠這個)
```

供電節點(Npwr/Ngnd)**永遠不會進待重算佇列**(`EnqueueNode` 直接過濾),所以它們在 `RecalcHash` 裡恆為 0。

### 1.2 節點旗標 `NodeFlags`(`WireCore.cs:41-52`)

8 個 bit,會在一個連通群裡 **OR 起來**,再拿去查 `FlagsToState`:

```csharp
[Flags] public enum NodeFlags : byte {
    None         = 0,
    State        = 1 << 0,  // 目前邏輯值(也代表「浮接、維持高」)
    PullUp       = 1 << 1,  // 有上拉(segdef '+' / 模組 pullups / weak 電晶體)
    SetHigh      = 1 << 2,  // 外部驅動為高
    SetLow       = 1 << 3,  // 外部驅動為低
    Pwr          = 1 << 4,  // 此節點是 VCC
    Gnd          = 1 << 5,  // 此節點是 GND
    ForceCompute = 1 << 6,  // 群組同時有 Gnd 與 Pwr 時兩者相消(某些匯流排節點)
    HasCallback  = 1 << 7,  // 此節點掛了 callback(記憶體監看等)
}
```

## 2. `NodeInfo` 的兩種佈局(關鍵優化)

`NodeInfo` 是固定 16 bytes 的 explicit-layout 結構(`WireCore.cs:289`),一個 cache line 放 4 個。前 4 bytes 是描述欄,後 12 bytes 是一個 **union**:

```csharp
[StructLayout(LayoutKind.Explicit, Size = 16)]
internal unsafe struct NodeInfo {
    public const int InlineCap = 6;
    [FieldOffset(0)] public NodeFlags Flags;     // 可變(執行期 SetHigh/SetLow)
    [FieldOffset(1)] public byte Inline;         // 1 = 用 InlinePayload;0 = 用 Tlist*
    [FieldOffset(2)] public byte C1c2Count;      // 內聯的 (gate,other) 對數
    [FieldOffset(3)] public byte GndPwr;         // 低 nibble=GndCount,高 nibble=PwrCount

    // ── offset 4 起的 12-byte union ──
    [FieldOffset(4)]  public fixed ushort InlinePayload[6];  // [c1c2 對][gnd 閘][pwr 閘]
    [FieldOffset(4)]  public int TlistC1c2s;   // 否則:索引到 TransistorList:(gate,other,...,0)
    [FieldOffset(8)]  public int TlistC1gnd;   //        通往 GND 的通道 (gate,...,0)
    [FieldOffset(12)] public int TlistC1pwr;   //        通往 VCC 的通道 (gate,...,0)
}
```

**為什麼分兩種**:絕大多數節點(~96%)鄰接很少,塞得進 6 個 ushort 槽(`Inline==1`)。這時整個節點的鄰接資訊**就在這一條 cache line 裡**,熱路徑不必再去 `TransistorList` 追一次指標(no dependent chase)—— 這是延遲限制下最大的省法。少數高扇出節點(`Inline==0`)才把鄰接放到外部的 `TransistorList`,用三個 int 索引指過去。

`InlinePayload` 的排列是 `[c1c2 對 ×C1c2Count][gnd 閘 ×GndCount][pwr 閘 ×PwrCount]`:
- **c1c2 對**:每對是 `(gate, other)` —— 「若 `gate` 導通,本節點就和 `other` 連通」(可能讓群組長大)。
- **gnd 閘 / pwr 閘**:單一 gate —— 「若 `gate` 導通,本節點就接到 GND / VCC」(決定群組旗標)。

三種 Tlist 子清單(高扇出版)語意完全一樣,只是放在外部、以 0 結尾。

## 3. 主迴圈:一個半週期 `StepCycle`

`WireCore.Recalc.cs:278`(`Step(n)` 就是呼叫它 n 次):

```csharp
private static void StepCycle() {
    int clk = ClockNode;                 // AttachClockHandler 解析出的時脈節點
    if (clk != EmptyNode) {
        ref NodeInfo ns = ref NodeInfos[clk];
        int next = NodeStates[clk] ^ 1;  // 時脈一定翻轉
        // 無分支:next=1 → 8>>1=4=SetHigh;next=0 → 8>>0=8=SetLow
        ns.Flags = (ns.Flags & ~(NodeFlags.SetHigh | NodeFlags.SetLow)) | (NodeFlags)(8 >> next);
        EnqueueNode(clk);                // 把時脈節點丟進待重算
        ProcessQueue();                  // settle 到穩定
    }
    Time++;
}
```

時脈翻轉以前是 handler 委派鏈裡的一個項目,現已 **inline 進 StepCycle**(`[H1]`),省掉每半週期的委派呼叫 + null 檢查。

## 4. 事件傳播:雙緩衝 settle `ProcessQueue`

這是引擎的最熱迴圈(`WireCore.Recalc.cs:66` `ProcessQueueInterp`)。settle 是**一波一波(wave by wave)**做的 BFS:

```csharp
private static void ProcessQueueInterp() {
    while (RecalcListNextCount != 0) {
        // 1) 對調:剛累積好的「下一波」變成「這一波」,Next 清空
        int* tmpList = RecalcList; RecalcList = RecalcListNext; RecalcListNext = tmpList;
        byte* tmpHash = RecalcHash; RecalcHash = RecalcHashNext; RecalcHashNext = tmpHash;
        RecalcListCount = RecalcListNextCount;
        RecalcListNextCount = 0;

        // 2) 處理這一波
        for (int i = 0; i < RecalcListCount; i++) {
            int nn = RecalcList[i];
            if (RecalcHash[nn] != 0) {     // 可能已被 AddNodeToGroup 吸收進群組而清掉
                RecalcNode(nn);
                RecalcHash[nn] = 0;        // 只清自己設過的(O(設過數),非整表 memset)
            }
        }
        RecalcListCount = 0;
    }
    InvokeCallbacks();   // 群組解析時排入的 RAM/ROM/影像 callback,settle 完才一起跑
}
```

### 4.1 雙緩衝的兩半

| | 角色 | 誰寫 | 誰讀/清 |
|---|---|---|---|
| `RecalcList` + `RecalcHash` | **正在處理的這一波** | 對調而來 | 主迴圈讀;`AddNodeToGroup` 吸收節點時清 |
| `RecalcListNext` + `RecalcHashNext` | **正在累積的下一波** | `EnqueueNode` / `SetNodeState` 寫 | 對調後變成「這一波」 |

每輪只要**對調兩個指標(O(1))**就完成換波 —— 這是用雙緩衝而非單一陣列的根本原因:同一輪內一個節點可能同時「這波已處理(=0)」又「下波要做(=1)」,兩種狀態必須各自獨立。

> 詳細「為什麼非要兩個」見本目錄相關討論;`RecalcHash` 名字沿用自 `chipsim.js`,**它不是校驗碼**,是「在不在清單裡」的成員旗標。真正驗證用的雜湊是 `NodeStatesChecksum()`(`WireCore.Recalc.cs:20`,FNV-1a),只在 benchmark/測試結尾算一次,不在熱路徑。

### 4.2 入列去重 `EnqueueNode`(`WireCore.Recalc.cs:36`)

```csharp
private static void EnqueueNode(int nn) {
    if (nn == Npwr || nn == Ngnd) return;       // 供電節點不重算
    if (RecalcHashNext[nn] == 0) {              // 去重:同一波最多排一次
        RecalcListNext[RecalcListNextCount++] = nn;
        RecalcHashNext[nn] = 1;
    }
}
```

### 4.3 狀態翻轉 → 通知下游 `SetNodeState`(`WireCore.Recalc.cs:169`)

當一個節點值改變,要把「以它為閘極的電晶體所連到的節點」排進下一波。這段是內聯+特化過的熱碼:

```csharp
private static void SetNodeState(int nn, byte newState) {
    if (NodeStates[nn] == newState) return;     // 沒變就不做事(收斂的關鍵)
    NodeStates[nn] = newState;
    int tlistGates = NodeTlistGates[nn];        // 以 nn 為閘的 (c1,c2,...,0) 子清單
    if (tlistGates != 0) {
        // 把 RecalcListNext / RecalcHashNext 提到區域變數(暫存器)
        // 一次讀兩對 (c1,c2) = 一個 64-bit load,減少分支與載入次數(+~1.2%)
        // #G2:依 newState 0/1 特化兩個迴圈(閘升高時 c2 仍經導通通道連著,只需排 c1)
        ...
        if (newState == 0) { /* 閘降低會「斷開」通道,c1、c2 都要重評 */ }
        else               { /* 閘升高:只排 c1 */ }
    }
}
```

> 注意分工:`NodeTlistGates`(以 nn 為**閘極**的清單)只在這裡用,每個群組成員回寫時碰一次;而 `NodeInfos` 裡的 Tlist*(以 nn 為**通道端點**的清單)是 BFS 走訪時碰。兩者是不同方向的鄰接。

## 5. 單點重算派發 `RecalcNode`

`WireCore.Recalc.cs:111`。先讀分類,再走最省的路徑:

```csharp
private static void RecalcNode(int nn) {
    byte cls = IsPureLogic[nn];
    if (cls == 1) { RecalcNodeFast(nn); return; }      // 靜態純邏輯:群組必為 {nn}
    if (cls == 2) {                                     // 動態 singleton 候選(R-1)
        // 若這半週期它所有 c1c2 通道閘都是 OFF,群組就是 {nn} → 可走 fast
        NodeInfo* ns = NodeInfos + nn;
        if (ns->Inline != 0) { /* 從內聯 payload 讀 c1c2 閘 */
            for (...) if (NodeStates[pay[k]] != 0) goto FallbackBFS; }
        else { /* 從 TransistorList 掃 */
            while (*p != 0) { if (NodeStates[*p] != 0) goto FallbackBFS; p += 2; } }
        RecalcNodeFast(nn); return;                     // 全部 OFF → O(1)
    }
FallbackBFS:                                            // cls==0,或動態候選有閘導通
    byte newState = ComputeNodeGroup(nn);               // 完整 BFS
    for (int i = 0; i < _groupCount; i++) SetNodeState(_groupBuf[i], newState);
    if ((_groupFlags & NodeFlags.HasCallback) != 0) { /* 排入該群組的 callback */ }
}
```

## 6. Fast-path 分類 `IsPureLogic`(§5 的依據)

在 `Reset()` 末尾由 `ClassifyPureLogicNodes()`(`WireCore.FastPath.cs:50`)建好,每節點一個值:

- **`1` 靜態純邏輯**:完全沒有通往「一般節點」的通道(`TlistC1c2s` 空 / `C1c2Count==0`),群組**可證明永遠是 {nn}**。NMOS 上這類很多(depletion-load 閘極輸出:有上拉,只能透過電晶體被拉到 GND)。
- **`2` 動態 singleton 候選**:有 c1c2 通道(群組**可能**長大),但若這半週期那些閘剛好全 OFF,群組就是 {nn} —— RecalcNode 做執行期「閘是否全 OFF」檢查決定。
- **`0` 必須走完整 BFS**:帶有 `HasCallback / ForceCompute / Pwr / Gnd` 任一旗標(callback 必須經正常路徑、forceCompute/供電有特殊解析)。

排除條件(`WireCore.FastPath.cs:53`):

```csharp
const NodeFlags exclude = HasCallback | ForceCompute | Pwr | Gnd;
```

> `PullUp` **不**在排除之列:無上拉的單節點群組若有通道導通就解析成 GND/PWR,否則浮接 →「維持前值」,`RecalcNodeFast` 對 empty-flags 直接回傳 `NodeStates[nn]`,與 `ComputeNodeGroup` 的單節點浮接 tie-break 完全一致。把 PullUp 移出排除,涵蓋率從 23.1% 拉到 26.7%,且 checksum 不變。

### 6.1 `RecalcNodeFast`(`WireCore.FastPath.cs:94`)— O(1) 解析

群組就是 {nn} 時,不必跑 BFS/`_groupBuf`/`_inGroup`,直接算這一個節點的值:

```csharp
private static void RecalcNodeFast(int nn) {
    NodeInfo* ns = NodeInfos + nn;
    int flags = (int)ns->Flags;          // PullUp 與/或執行期 SetHigh/SetLow,或 0(浮接)
    // OR-all 無分支(R4):NodeStates 是 0/1,有任一通往 GND 的閘導通就 |= Gnd(<<5),通往 VCC 就 |= Pwr(<<4)
    if (ns->Inline != 0) { /* 從內聯 payload 的 gnd/pwr 段 OR */ }
    else                 { /* 從 TlistC1gnd / TlistC1pwr OR */ }
    // flags==0 ⇒ 浮接 singleton ⇒ 維持前值;否則查 LUT
    SetNodeState(nn, flags != 0 ? FlagsToState[flags] : NodeStates[nn]);
}
```

## 7. 群組解析:`ComputeNodeGroup` / `AddNodeToGroup` / `GetNodeValue`

這是開關級引擎的心臟(`WireCore.Group.cs`),處理「一群透過導通電晶體連在一起的節點該解析成什麼值」。

### 7.1 `ComputeNodeGroup`(`WireCore.Group.cs:65`)

```csharp
private static byte ComputeNodeGroup(int nn) {
    for (int i = 0; i < _groupCount; i++) _inGroup[_groupBuf[i]] = 0;  // 只清上次用到的
    _groupFlags = NodeFlags.None;
    _groupCount = 0;
    AddNodeToGroup(nn);     // BFS 走訪 + 累積 _groupFlags
    return GetNodeValue();  // 解析
}
```

### 7.2 `AddNodeToGroup`(`WireCore.Group.cs:88`)— 迭代式 BFS

`_groupBuf` **同時是群組清單也是工作佇列**(`readIndex` 讀、`gc` 寫,`readIndex==gc` 即排空)。把所有靜態指標與 `gc/gf` 提到區域變數(整個走訪期間留在暫存器,**+3.2%**):

```csharp
private static void AddNodeToGroup(int seed) {
    byte* inGroup = _inGroup; ushort* groupBuf = _groupBuf;
    byte* nodeStates = NodeStates; byte* recalcHash = RecalcHash;
    NodeInfo* nodeInfos = NodeInfos; ushort* transList = TransistorList;
    int gc = _groupCount; NodeFlags gf = _groupFlags;

    // 加入 seed:去重 + push + 清 recalcHash + OR 旗標
    if (inGroup[seed] == 0) { inGroup[seed] = 1; groupBuf[gc++] = (ushort)seed; recalcHash[seed] = 0; gf |= nodeInfos[seed].Flags; }

    int readIndex = 0;
    while (readIndex < gc) {
        int nn = groupBuf[readIndex++];
        NodeInfo* ns = nodeInfos + nn;
        if (ns->Inline != 0) {
            // c1c2 對:閘導通 ⇒ 把 other 拉進群組
            // gnd 段:任一閘導通 ⇒ gf |= Gnd
            // pwr 段:任一閘導通 ⇒ gf |= Pwr
        } else {
            // 高扇出:c1c2 用 ulong 雙對讀取(+1.4%,只有長走訪才划算);gnd/pwr 子清單同上
        }
    }
    _groupCount = gc; _groupFlags = gf;
}
```

每加入一個節點都做四件事:**`_inGroup` 去重 → push 進 `_groupBuf` → 清該節點的 `RecalcHash`(本波不必再單獨重算)→ OR 它的旗標**。

### 7.3 `GetNodeValue`(`WireCore.Group.cs:155`)— 解析

```csharp
private static byte GetNodeValue() {
    if (_groupFlags != NodeFlags.None) return FlagsToState[(int)_groupFlags];  // 主路徑:查表
    // 純浮接群組(OR 後為空):比 NodeConnections,連接數最多者(電容最大)保留其前值
    int maxConn = -1; byte maxState = 0;
    for (int i = 0; i < _groupCount; i++) {
        int conn = NodeConnections[_groupBuf[i]];
        if (conn > maxConn) { maxState = NodeStates[_groupBuf[i]]; maxConn = conn; }
    }
    return maxState;
}
```

### 7.4 解析優先序與 `FlagsToState` LUT

群組 OR 出來的 8-bit 旗標直接查 256 格的 `FlagsToState`(`WireCore.Group.cs:34` 在 `Reset` 時用 `FlagsToStateOf` 預建)。優先序(NMOS,同 MetalNES / chipsim.js):

```
1. ForceCompute 特例:群組同時有 Gnd 與 Pwr → 兩者相消
2. Gnd     → 0   (接地最強)
3. Pwr     → 1
4. SetHigh → 1   (外部驅動)
5. SetLow  → 0
6. PullUp  → 1
7. State   → 前值
8. 否則    → 0
另:OR 後旗標為空(純浮接)→ 走 GetNodeValue 的「電容最大者保留前值」分支
```

把這套優先序**預先算進 256 格查表**,執行期一次查表取代一串條件判斷。

## 8. 一個半週期的完整資料流

```
StepCycle
  時脈節點翻轉(設 SetHigh/SetLow),EnqueueNode(clk) → 寫 RecalcListNext/HashNext
  │
  ▼  ProcessQueue(while 下一波非空)
  ├─ 對調 List/Hash 指標(這波 ↔ 下波)
  ├─ for 這波每個 nn:
  │     RecalcNode(nn)
  │       ├ cls=1 → RecalcNodeFast ─┐
  │       ├ cls=2 → 閘全OFF? →Fast ─┤  讀 NodeInfos / NodeStates(+TransistorList 高扇出時)
  │       └ 否則  → ComputeNodeGroup┘     ↓ 解出新值
  │                 SetNodeState(群組每個成員, 新值)
  │                   └ 值有變 → 依 NodeTlistGates 把下游 EnqueueNode → 寫「下一波」
  │       (群組有 HasCallback → EnqueueCallback)
  └─ 下一波非空則繼續;空了 → 收斂
  │
  ▼  InvokeCallbacks()  跑這半週期排入的 RAM/ROM/影像 callback
  Time++
```

## 10. 設計取捨備忘(為什麼長這樣)

- **事件驅動而非整批重算**:只重算「這波真的可能變」的節點。網路天生稀疏(每事件平均導通群 ~1.4 節點),整批掃描會做上萬倍白工 —— 這也是 IR/codegen/GPU 全部更慢的根因。
- **記憶體延遲是牆,不是算術**:熱路徑的成本在隨機散讀 `NodeStates`/`NodeInfos`/`TransistorList`,不在運算。因此優化方向一律是**縮小資料 + 提高區域性 + 更緊的 codegen**(ushort 化、內聯 payload、SoA 指標進暫存器、ulong 雙對讀取),而不是更聰明的演算法。
- **fast-path 不縮 D(髒節點數),縮的是每個髒節點的常數**:把「群組必為單點」的情形從完整 BFS 降為 O(1),涵蓋約 1/4 的節點。
- **bit-exact 是硬門檻**:任何改動都要維持整機 checksum `0x794A43ABDF169ADA`(`full_palette` 300k --extra-ram),並以 interleaved-paired A/B 驗證效能(見 `tools/ab_bench.ps1`)。

> 完整的死路清單與優化筆記散見 `MD/`、`MD/S2/` 與 commit 訊息;本文只描述**留下來、現在在跑**的演算法。

## 11. 執行期熱路徑分布(實測 2026-06-07,full_palette 300k,引擎 2325692)

`RecalcNode` 一趟跑下來共 **159,646,522 次呼叫**,每次走哪條:

| 路徑 | 佔呼叫 |
|---|---|
| **Fast-path(O(1) `RecalcNodeFast`)** | **73.6%** |
| ├ 靜態純邏輯(class 1) | 16.0% |
| └ 動態 singleton(class 2,c1c2 閘全關) | **57.6%** |
| **完整 BFS(`ComputeNodeGroup`)** | **26.4%** |

對照**靜態分類**(14,729 live):class 1 = 3,929(26.7%)、class 2 = 10,784(73.2%)、class 0 ≈ 16(callback/ForceCompute/供電)。

重點:
- **靜態佔比 ≠ 執行期佔比**:class 1 佔節點數 26.7%,卻只佔呼叫 16.0% —— 那些靜態純邏輯節點被重算的頻率**低於平均**(較少變)。撐起 fast-path 的是**動態 singleton 的 57.6%**,正是 **R-1** 抓的那一類(沒有 R-1 它們全得走 BFS,fast-path 只剩 16%)。
- **那 26.4% 的 BFS 也不大**:平均導通群 ~1.4 節點,所以「BFS」≠「大走訪」,只是約四分之一的重算需要走群組解析。
- 含意:再要擠效能,槓桿越來越靠「**減少進入 settle 的事件數**」(如 P-1 同態 turn-on 剪枝)而非「加速單次 BFS」。
- 量法:在 `RecalcNode` 三條出口各放一個臨時計數器(已用完移除,不留 hot-path 成本);bat 見 `temp/run_diag_fastpath.bat`。
