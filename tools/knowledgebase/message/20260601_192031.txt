At 14k nodes, the overhead of an event queue (pointers, branching) is higher than just evaluating the boolean equations. ### A & B. The Concrete Pipeline (Solving Problems 1, 2, and 3)

Forget pure static graph theory. Use an **Empirical-Structural Hybrid Pipeline**.

#### Stage 1: Auto-Detect "Analog" Blocks (Problem 2)
You cannot easily detect ratioed logic/charge-sharing by looking at the graph. You detect it by its failure to be boolean.
*   **The Algorithm:**
    1. For every node $N$ in the SCC, use a localized structural search: gather all pass-transistors and pull-down/pull-up paths within a radius of 1 or 2.
    2. Formulate a candidate boolean function for $N$ based on inputs.
    3. Stream a 1-million-cycle trace from the Golden Simulator.
    4. For every cycle, check: Does the candidate boolean function predict the state of $N$?
    5. **The Brutal Cut:** If a node's state depends on a previous floating state (charge sharing), relies on contention where a weak pull-up loses to a pull-down (ratioed logic), or is part of a bidirectional ALU carry chain (like the 6502's), *the boolean prediction will fail against the golden trace.* 
    6. Any node that fails prediction is tagged as an **Analog Island Node**. Group contiguous analog nodes into blocks. Leave them for the switch-level engine.

#### Stage 2: Clock Phase Discovery (Problem 3)
Do not try to statically reverse-engineer the 6502 clock generator.
*   **The Algorithm:**
    1. You know the external CLK pin.
    2. Look at the Golden Trace. Find every node that toggles periodically with a strict, unbroken phase relationship to CLK. (e.g., Node X is always high during CLK half-cycle 1, low during half-cycle 2).
    3. These are your internal clocks ($\phi_1$, $\phi_2$, precharge signals).
    4. Tag these nodes as **Clocks**. They are treated as primary inputs to the logic networks, not as logic to be evaluated.

#### Stage 3: Auto-Detect State Elements (Problem 1)
NMOS state elements are either cross-coupled inverters (static) or pass-gate-isolated capacitors (dynamic). 
*   **Structural Detection (Static):** Run Tarjan’s Strongly Connected Components algorithm. In boolean logic, an SCC of size > 1 is a static feedback loop. Break the loop by identifying the feedback gate and replacing it with a "D-Flip-Flop" equivalent (a state boundary).
*   **Structural Detection (Dynamic):** A dynamic latch in NMOS is simply a gate input driven *only* by the source/drain of a pass transistor.
*   **Empirical Validation:** Look at the Golden Trace. A state node is one that holds its value for multiple cycles while its driver is completely disconnected (all incoming pass transistors are off). If the trace shows a node maintaining '1' or '0' while floating, it is a dynamic state element. Tag it as a **Register**.

#### Stage 4: Levelization
Now that you have cut the graph at **Clocks**, **Registers**, and **Analog Islands**, what remains is strictly feed-forward combinational logic.
*   **The Algorithm:** Run a topological sort on the remaining combinational nodes. If the sort fails, you missed a state element (an un-cut loop). Fall back to the trace to figure out which node in the loop acts as the delay element, cut it, and sort again.

### 4. DOES IT ACTUALLY GET FASTER? (The Performance Reality Check)

**Be highly skeptical of event-driven logic simulation at this scale.** 
Your switch-level engine evaluates 604 nodes per half-cycle, taking ~82 cycles per event. 
Total CPU cycles: $604 \times 82 = \approx 50,000$ CPU cycles per half-cycle.

If you make an *event-driven* logic simulator, the overhead of maintaining the queue, checking if a node changed, and dispatching to a virtual function or delegate in C# will cost you ~20-30 cycles per event. You will save very little.

**The Escape:** Oblivious Compiled Simulation. 
If you compile the 14,000 nodes down to, say, 5,000 boolean equations, and write them as straight-line C# bitwise operations, the C# JIT will map them to native instructions. Evaluating a boolean equation takes ~1-2 CPU cycles. 
Evaluating *all 5,000 equations every single half-cycle* (ignoring sparsity entirely) will take $\approx 10,000$ CPU cycles. 

By touching *everything* but doing it blindly and optimally in L1 cache (no branches, no queues, no pointers), you beat the sparse event-driven engine by a factor of 5x to 10x. **Sparsity is a trap on modern superscalar CPUs for small netlists.** Branch prediction misses and memory latency from pointer chasing cost more than just computing the AND/OR gate.

### 5. VALIDATION + AUTO-LOCALIZATION

When the screenshot fails, how do you find the bug? 
You use a **Dynamic Miter**.

1. Run the Golden Switch-Level simulator and your Compiled Logic simulator side-by-side in the same process, fed by the same inputs.
2. At the end of every half-cycle, compare the states of the **Registers** and the inputs to the **Analog Islands**.
3. **The Trap-Door:** The exact cycle, half-cycle, and specific Register that diverges (Golden = 1, Logic = 0) is your localized bug. 
4. Because the logic is levelized and feed-forward, if Register X is wrong, the bug is *mathematically guaranteed* to be in the combinational logic cone leading to Register X. 
5. You can auto-generate a Graphviz dump of just the logic cone for Register X, overlay the Golden node states vs your Logic node states on the edges, and the exact boolean extraction error will be visually highlighted in red. The pipeline essentially debugs itself.
