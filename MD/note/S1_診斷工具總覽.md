# S1 診斷工具總覽(DEBUG profilers + 效能量測工作流)

> 一份集中索引:S1 引擎內建的診斷 profiler、量測旗標、外部硬體工作流,各自「量什麼 / 怎麼跑 / 詳細在哪」。
> **全部 DEBUG-only(`#if DEBUG`)或 opt-in 旗標 → Release byte-identical**(golden `0x9174E19D961CB6E5` @ full_palette 400k --extra-ram)。最後更新 2026-06-20。

## 怎麼跑(共通)
DEBUG profiler 統一用 Debug build 跑 benchmark,結果印在輸出尾段:
```
dotnet run -c Debug --project src/AprVisual.S1 -- --benchmark <rom> --bench-hc N --extra-ram --system-def-dir <dir>
```
(或先 `dotnet build -c Debug` 再跑 `bin/Debug/net11.0/AprVisual.S1.dll`。)Debug/Release 的「事件計數」相同(同演算法),只有 wall-clock 不同 → 計數對 Release 調校有效。

## A. 引擎內建 DEBUG profilers(輸出尾段的 `# [..]` 區塊)
| Tag | 量什麼 | 用途 / 找到過什麼 |
|---|---|---|
| `[waste-profile]` | RecalcNode pop 中「no-change(零狀態改變)」的比例 + 分類(Float/PullUp/Supply…) | 找出 84% 浪費 pop → 催生 P-2/P-3/P-4 剪枝 |
| `[fast-gate dist]` | fast-path(RecalcNodeFast)pop 的 gnd/pwr/c1c2 gate 數分佈 | 評估 MLP 定寬展開的覆蓋率 + dummy-load 成本 |
| `[B1 pair-path]` | 用 inline 2-node-group 解析的 pop 數(vs 落 BFS) | B1 pair fast-path 命中率 |
| **`[hotpath-calls]`** | 熱路徑 chain 每個 method 的**呼叫次數 / % / 每 pop**(ProcessQueue→RecalcNode→RecalcNodeFast/ComputeNodeGroup→AddNodeToGroup+GetNodeValue→SetNodeState)| SetNodeState 37%/1.27per-pop 最熱;70% pop 走 O(1)。詳:`2026-06-20-熱路徑呼叫次數剖析.md` |
| **`[branch-dist]`** | 各**資料相依分支**的方向分布(taken/not,~50/50=高熵=易誤預測)| 定位 ~6 MPKI branch-miss:enqueue-prune 58/42 最熱、cls2 71/29。詳:`2026-06-20-branch-miss熱點分析.md` |
| **`[co-read]`** | co-read 的兩個 node id 落在**同一 64-bit word** 的比例 | 判 bitset-mirror 可行性(rising 僅 1.9% → Scheme B 死)|
| `[settle-pass-dist]` | 每次 ProcessQueue 跑幾個 settle wave 才收斂 | mean 12.08;檢視 MaxSettlePasses 安全帽。詳:`2026-06-08-settle-pass分佈.md` |
| `[bfs-depth-dist]` | AddNodeToGroup 的 BFS 群組走訪深度分佈 | mean 1.25(群組小)|
| `[co-activity]` | 被 pop 過的「熱集」節點數 + 佔多少 cache line(NodeInfos KB)| 評估 renumber packing(熱集 129KB,近理想)|

## B. 量測旗標(opt-in,Release 也可用)
| 旗標 | 量什麼 | 怎麼用 |
|---|---|---|
| **`--array-footprint`** | 啟動時印各熱路徑非託管陣列的 **base 位址 + 大小**(NodeStates/NodeInfos/TransistorList(+Off)/RecalcList(+Next)/RecalcHash(+Next)/NodeTlistGates(+Off)/IsPureLogic/FlagsToState)| 給硬體 data-address 分桶(IBS/SPE)用 + 看哪些陣列超過 L1d/L2。**Release 可用**(不污染預設輸出)|
| `--pin [N]` | 釘熱執行緒到單核 + High priority + 關 EcoQoS(降變異)| 預設 OFF;leaderboard 公平性。見 `Sim/PerfTuning.cs` |

## C. 外部硬體量測工作流
### C-1. ARM perf(Raspberry Pi 5 / A76)— 最乾淨的 A/B + 分層 cache
- A/B(cyc/hc,頻率無關):`~/pi_freq_lock.sh` 鎖 3.0G → `sudo perf stat -e cycles …` interleaved-paired。見記憶 [[perf-ab-on-pi5]]。
- branch-miss / 分層 cache:`sudo perf stat -e instructions,branches,branch-misses` / `-e L1-dcache-loads,L1-dcache-load-misses,l2d_cache,l2d_cache_refill,LLC-loads,LLC-load-misses -- <Release run>`。
- ⚠️ **ARM SPE(per-data-address 取樣)在此 Pi kernel 沒開**(`CONFIG_ARM_SPE_PMU` 未編 + 無 DT 節點;要重編 kernel)。
- 連線:`ssh -i ~/.ssh/aprvisual_pi pi@<ip>`(ip 由 `curl baxermux.org/service/ip.php`)。

### C-2. AMD uProf IBS(Zen2 / 3700X)— per-data-structure 快取 miss 歸因
- **前置(一次性)**:IBS 需 `bcdedit /set x2apicpolicy Disable` + `uselegacyapicmode Yes` + 重開機(否則 x2APIC 擋住 IBS 中斷 → 0 樣本)。
- 流程:`AMDuProfCLI collect -e event=ibs-op,loadstore,interval=50000 -o OUT <exe> … --array-footprint`(**free-run,勿用 --affinity/-c**)→ `report --ascii ibsop-event-dump` → **`python tools/ibs_bucket.py OUT`** 依 footprint 把 `IbsDcLinAd` 分桶,出 per-array load / L1-miss(IbsDcMiss)/ L2-miss(IbsL2Miss)/ 延遲。
- 完整雷點(os=0 需 Zen6+、min interval 25000、PMU 鎖排除…):**`tools/操作筆記_AMD_uProf.md`**。實測結論:`2026-06-20-L-cache-miss熱點分析.md`。
