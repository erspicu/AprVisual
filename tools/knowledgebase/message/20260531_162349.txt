### A & B. 2005年後有哪些「真正有效 vs. 紙上談兵」的突破？

在單核 Event-driven 領域，幾乎沒有演算法突破，但有 **資料結構與記憶體存取模式** 的突破。

#### 1. 真正有效且被採用的技術：Cache-Conscious Data Layout / Memory-Centric Graph Representation
*   **技術名稱：** Data-Oriented Netlist Layout & Cache-aware Event Queuing
*   **代表學者/論文：** 雖然較少作為單一理論論文發表，但可參考 EDA 軟體架構演進，如 *Geodesic: A cache-efficient logic simulator (Jiang et al., DAC 2007)* 或近年軟體工程中的 Data-Oriented Design (DoD)。
*   **為何真正有效：** 在 activity 僅 1.4 的極端情況下，**「演算法複雜度」根本不是瓶頸，「L1/L2 Cache Miss」和「Branch Misprediction」才是。** 將 Switch/Node 資料從傳統的 Linked-list/OOP pointer graph 打平為連續記憶體陣列（Structure of Arrays, SoA），並將相鄰拓撲的節點透過 Graph Partitioning 排列在同一個 Cache Line 中。這能讓單核 CPU 效能提升 2x-5x，這是近 20 年最有效的單核加速手段。

#### 2. 紙上漂亮但實務沒贏的技術（Incremental / Academic only）：
*   **MIG (Majority-Inverter Graph) / XAG (XOR-AND Graph) 等新邏輯結構：**
    *   *代表：* Amarú et al. (EPFL), *Majority-Inverter Graph: A Novel Data-Structure and Algorithms for Boolean Optimization (DAC 2014)*。
    *   *為何沒贏：* MIG 對於邏輯合成 (Synthesis) 和 FPGA 對映極度有效，能減少 logic depth。但對於你這種以 Pass-transistor 為主的 Switch-level Event-driven 模擬器，將 pass-transistor 硬轉成 MIG 不僅會破壞雙向性 (bidirectionality)，還會增加節點數量，完全拖慢模擬速度。
*   **Machine Learning-directed Event Scheduling:**
    *   *為何沒贏：* 近年有一堆論文試圖用 ML 預測哪些 net 不需要 evaluate。但 ML inference 的 overhead (即使是極小的決策樹) 也遠大於你直接做 BFS 走訪 1.4 個節點的成本。

### C. 關於「Lowering」與「Fast-path」：這兩條路真的成熟了嗎？

你的直覺完全正確，這兩條路在單核序列演算法上已經成熟，你目前做的就是當代最佳實踐。

#### (1) 關於 Lowering（結構化簡）
*   **學術對應巔峰：** Berkeley ABC (*Alan Mishchenko et al., "DAG-aware AIG rewriting a fresh look at combinational logic synthesis", DAC 2006*)。
*   **現狀：** 2006 年的 AIG Rewriting 搭配 SAT-sweeping (Fraiging) 至今仍是單核 Boolean Network 優化的絕對霸主。近 20 年來的進步多為「如何平行化 ABC」或「如何用強化學習 (RL) 找出更好的 ABC command recipe」，其底層的 DAG 演算法沒有根本性改變。
*   **針對你的 94% SCC (Pass-transistor network)：** Bryant 在 1987 年的 COSMOS 模擬器提出了 *Symbolic Switch-Level Analysis*，也就是在靜態時期 (Compile-time) 把 Channel-Connected Components (CCC, 也就是你的 SCC) 透過高斯消去或 BDD 提取成等效的 Boolean 函式。**如果你現在的 Lowering 還沒做到把那 94% 的 SCC「靜態萃取」成無迴圈的有向邏輯（如果可能的話），這是你唯一還能大幅挖掘的演算法紅利。** 如果你因為 6502 類比特性必須保留動態 BFS，那 Lowering 已經到頂了。

#### (2) 關於 Fast-path（執行期特化）
*   **學術對應巔峰：** Activity-directed simulation, Selective Trace (1980s Ulrich 等人)。
*   **現狀：** 你的 Fast-path (辨識單一節點、跳過通用 group resolution) 在編譯器領域稱為 *Fast-pathing* 或 *Guard clauses*，在 EDA 稱為 *Event Filtering* 或 *Activity Bypass*。
*   **為何沒有新東西：** 因為平均 1.4 個節點的 evaluate 意味著「**你沒有時間做任何聰明的事**」。任何稍微複雜的 Heuristic（例如動態分析 SCC 的 active window）其 overhead 都會超過直接做一次單純的 bitwise 運算。
*   **工程推進（近20年）：** 這裡唯一的進步是利用現代 C++ / Rust 的模板元編程 (Template Metaprogramming) 或 JIT 技巧，把 "if (single_node)" 這種判斷去虛擬化 (Devirtualization)，讓 CPU 的 Branch Predictor 能 100% 猜中，避免 pipeline flush。

### D. 給你這種極限模擬器的具體建議（基於現代工程，而非學術幻想）

既然學界在演算法上沒有新把戲，要在你的現有架構上再榨出效能，現代 EDA 工程師會做以下幾件事：

1.  **放棄傳統的 Priority Queue / Event Wheel：**
    如果 activity 只有 1.4，維護 Event Queue 的成本可能佔了總時間的 50% 以上。改用 **Double-buffered Flat Arrays (雙緩衝陣列)**。將本次 event 要觸發的 index 直接 push 到一個連續陣列，下一 cycle 循序掃描這個陣列。沒有排序，沒有 linked-list。
2.  **Bit-Parallel Simulation（雖然你不要多核，但這是在單核上的資料級平行）：**
    這是一個古老但在 2010s 後因為 AVX-512 等指令集被重新重視的技術。即使你只模擬「一台」NES，能否將 64 個獨立的 boolean 節點塞進一個 `uint64_t` 中，在一次 CPU 指令中完成 64 個節點的狀態更新？
3.  **Data-structure flattening (Cache locality)：**
    你的 BFS resolution 慢，通常不是因為 BFS 本身，而是因為 Graph pointer 亂跳導致 L1 cache miss。把那 94% 的大 SCC 節點透過 BFS 順序重新編號（Graph reordering/linearization），讓執行時讀取相鄰節點都在同一個 Cache Line 裡。

**總結你的疑問：**
你並沒有錯過任何近 20 年的「學術神蹟」。Switch-level 的 event-driven simulation 在單核 CPU 上的理論極限，在 Bryant 的年代與 2000 年代初期的高階邏輯合成結合後，就已經畫下句點。你現在面對的是純粹的 **Systems Programming / Micro-architecture Optimization** 挑戰。你的 Lowering 搭配 Fast-path 策略，已經是這條技術路線最正確、最極致的做法。
