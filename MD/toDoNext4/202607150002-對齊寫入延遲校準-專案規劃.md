# 專案規劃:CPU→PPU 對齊 / $2001 寫入延遲校準

**日期**:2026-07-15
**性質**:規劃 + 刻畫(尚未動引擎;這是全戰役風險最高的一塊,需使用者 go/no-go)
**觸發**:StaleSpriteShiftRegs Test 3 根因 = `$2001` 渲染啟用早 3 個 dot;與 BGSerialIn、
8 顆 NMI-edge、even_odd 共用同一個根 = **CPU→PPU 跨晶片寫路徑被理想化為零延遲**。

---

## 1. 問題精確刻畫

**時間單位**(實測):1 PPU dot = **8 hc**;1 CPU cycle = 24 hc = 3 dots。

**共用根因**(KB §2.5,PPUSim 仲裁已定案):兩顆晶片**內部**逐 dot 一致
(VBL@(241,1)、skip@339);~1-dot 偏移在**跨晶片寫路徑**(CPU 寫 $2001 → PPU 內部
生效),S1 把它理想化為零延遲。真機延遲 = **2-5 PPU cycle,依 clock alignment 而異**。

**各測試的延遲需求(皆繞 dot 339 這個決策點)**:
| 測試 | 決策點 | S1 現況 | 需求 | 缺口 |
|---|---|---|---|---|
| even_odd(nes-test-roms) | v=261 dot 339 skip 決策 | 窄窗補 +16 hc | 過(現行) | — |
| StaleSpriteShiftRegs T3(AC) | v=14 dot 339 counter 重設 | enable @dot337 | @dot340 | **+24 hc** |
| BGSerialIn(AC) | 可見行 dot%8 邊界 | 未測 | $2001 延遲 2-5 cyc | 未量 |
| **ALERead(AC)** | LDA $2007 汙染 dot | $2007 讀 @dot224 | 對齊 dot 214→sprite@244 | **~2 dot** |
| NMI-edge ×8(nes-test-roms) | VBL/NMI dot 邊界 | reset-hold 對齊 7 過 | — | 別打壞 |

**注意 16 vs 24 的矛盾**:兩者都是「$2001 效果落在 dot 339 的哪一側」,但寫入指令
落點不同 → 需要的**額外**延遲不同。真機是**單一固定延遲**(在 S1 的固定 alignment 下),
16/24 的差異暗示:窄窗那個 16 是**擬合 even_odd 特定寫入時序**的值,不是真·均勻延遲。
**未解問題**:是否存在單一均勻延遲 D 同時滿足全部四家族?這是計畫的核心未知數。

**ALERead 併入(2026-07-15 實測)**:err2 = Test 2(boing2k7 八位閂鎖汙染→sprite-zero hit)。
八位閂鎖是穩定不動點($FF→$FF,feedback shim 不介入=正確);S1 的 `LDA $2007` 讀取結束
在 v=3 dot 224(測試預期 dot 214 起跑),汙染像素落在 **dot 246** 而 sprite 0 在 **dot 244**
→ 差 2 dot 不重疊 → 無 hit(selPat 也因偏移歪成 3 而非 1)。**同根,同案。**
所以剩餘三顆偏差(StaleSpriteShiftRegs T3、BGSerialIn、ALERead)全是**跨晶片 $2007/$2001
dot 精確時序**,一個校準專案收三顆。方向一致性註記:T3 是 $2001 效果**早** 3 dot、
ALERead 是 $2007 效果**晚** 2 dot —— 皆 ~2-3 dot 量級,符合「寫/讀路徑零延遲理想化」的
單一根因(方向依量測對象不同)。

## 2. 已知死路 + 現行 shim 的實作牆

**死路(KB §3.1 #13 明載)**:「**廣域延遲每個 $2001 轉態會掛住測試**」——
even_odd 自己用逐週期精算的多次 $2001 量測,天真全域延遲污染它的校準迴圈 → 掛死。
→ 這是當初收窄成 `v==261 && h∈[338,339]` 窄窗的原因。

**現行 shim 的實作牆**(WireCore.System.cs PpuWriteDelayStep):
用**單發** clamp/release(`_pwdBkgRel` 是單一值)——
`if (_pwdBkgRel >= 0) {...release...} else {...arm...}`:前一個延遲未放開時,
**第二個轉態的 else 分支不執行 → 轉態被丟棄**。所以「全域化」不是放寬窗那麼簡單,
**必須重建成佇列式**(像 DMA shim 那樣)才能承接重疊/背靠背寫入。

## 3. 提案設計:佇列式均勻寫入延遲

**機制**:把 `bkg_enable`/`spr_enable` 建模成**延遲線** ——
每個轉態進一個 (releaseTime, clampNode, holdValue) 的**小佇列**,而非單一 pending;
每步檢查佇列頭是否到期釋放。用既有 `InstClampLow`(鉗互補節點,GND 贏)承接。
關鍵:重疊寫入時,後到的轉態鉗的是**前一個延遲後的值**(pipeline 語意),
不是原始舊值 —— 這是正確性核心,也是單發版做不到的。

**磁量校準**:先假設單一 D,在 even_odd(掛死試金石)+ StaleSpriteShiftRegs T3
兩顆孤立 ROM 上掃 D ∈ {16, 20, 24}:
- 若某個 D 同時讓兩者過 → **均勻延遲假說成立**,計畫可行;
- 若無單一 D → 需**分區延遲**(依 vpos band 給不同 D)或改從跨晶片模型下手。

### 3.1 第一輪實驗結果(2026-07-15,已 commit 骨架 OFF-by-default)

建了「觀察暫存器 bkg_enable、連續鉗下游 bkg_enable_out 到 N hc 前值」的環形延遲線
(EnablePpuWriteDelayGlobal,--ppu-write-delay-global N)。掃 StaleSpriteShiftRegs:
- **D=8(1 dot)**:err3(不退步,但 T3 需 3 dot 才夠,未修)
- **D=24(3 dot)**:**err1 退步**(打壞 Test 1 基本 sprite-zero hit)

**判決:連續鉗下游 `bkg_enable_out` 是錯機制。** 矛盾 = 修好 T3 所需磁量(24 hc)
正好是打壞 Test 1 的量。真機均勻延遲下 Test 1 照過,所以延遲線注入了「均勻延遲」
以外的假象:`bkg_enable_out` 是**組合節點**(閘控 rendering_disabled,但自身由 bkg_enable
經邏輯衍生),每半週期強鉗打斷它與其他訊號的組合關係(D=8 分歧窗小少衝突;
D=24 持陳舊值夠久就壞)。金 checksum OFF 時不變(骨架安全)。

### 3.2 下個 session 的正確機制方向

**改鉗暫存器 `bkg_enable`/`spr_enable`(而非下游)+ 佇列承接重疊**:
- 在暫存器**轉態**時,鉗舊值 N hc 再放開(乾淨延遲「暫存器→全下游」傳播,
  不 pin 組合節點);
- 用小佇列(ring of releaseTime)承接重疊寫入 —— 這才是避開 even_odd 掛死的正解
  (單發版丟第二個寫入;佇列版逐個延遲)。
- 注意:鉗暫存器會讓「觀察後續轉態」變難(鉗住就看不到暫存器真值)——
  解法:另存一份「暫存器意圖」的影子(每步取 NodeStates 但在鉗定時用佇列裡的預定值),
  或改延遲**寫入 strobe**(write_2001_reg 事件)讓暫存器本身晚 N 更新。後者最乾淨
  但需找到 strobe→register 的 pass 管施力點。
- 現有窄窗 PpuWriteDelay(單發鉗暫存器)已是這個方向的雛形,只差「全域 + 佇列」。

## 4. 驗證成本 + 風險(為何需要 go/no-go)

- **牽動範圍**:任何全域 $2001 時序改動影響 —— even_odd + 8 NMI-edge
  (nes-test-roms ppu_vbl_nmi 家族)+ StaleSpriteShiftRegs + BGSerialIn +
  **所有開關渲染的遊戲畫面**。
- **六顆哨兵擔保不了**:對齊敏感家族在 nes-test-roms,不在 AC 孤立 ROM 集。
- **完整驗證 = 跨兩套件的旗艦級**:AC 141 掛牌跑(8h)+ ppu_vbl_nmi 全家族回歸
  + 遊戲畫面 CRC 語料庫。
- **風險**:可能危及已入帳的 **136/141**(even_odd/NMI 若翻面就是淨負)。
- **試金石不便宜**:even_odd 孤立 ROM 在 400 幀/合理預算內跑不出判定
  (這顆本來就慢)→ 連 crux 實驗都要分鐘級,無法在單一短 session 收斂。

## 5. 建議執行計畫(需一個專屬 full-budget session)

1. **佇列式延遲模型**(取代單發 clamp/release,保留 InstClamp 施力點);
2. **D 掃描**:even_odd + StaleSpriteShiftRegs T3 孤立,掃 D ∈ {16,20,24},
   找同時過的 D(或證明不存在 → 轉分區延遲);
3. **NMI 家族回歸**:8 顆孤立 ROM(nes-test-roms rom_singles)驗不翻面;
4. **BGSerialIn 順檢**:同延遲下是否連帶翻綠(很可能,同根);
5. **旗艦雙套件驗證** + 遊戲 CRC;
6. checksum 不變(shim 測試模式限定,不上 benchmark 路徑)。

## 6. 我的建議

**這一塊值得做**(一次可能收 Test3 + BGSerialIn 兩顆,還把最深的系統債轉正),
**但不該在預算見底時硬幹** —— 它需要一個專屬的 full-budget session:動全域時序 →
掃磁量 → 跨兩套件旗艦驗證,任何環節翻面都要能回頭。**建議排下一個 session 專門打**,
或先去打 **Address2004 err10**(獨立、低風險、錨點最好)累積戰果,把這塊留給
有完整驗證預算的時段。決定權在使用者。
