# perfect6502 研究與比較(2026-06-08)

對 [mist64/perfect6502](https://github.com/mist64/perfect6502) 做了編譯、效能量測與演算法比對。結論先講:**它和我們是同一條 Visual6502 / chipsim 血脈(我們經 MetalNES),它的兩個核心 enqueue 技巧我們 S1 早就有,而且我們多了 perfect6502 沒有的 P-1~P-4 剪枝 —— 在剪枝這條軸上我們嚴格領先。** 沒有可直接搬來「打贏現況」的東西;最大價值是它作為獨立第三方實作,反向驗證了我們事件驅動核心 + 剪枝已達/超過這條參考血脈的水準。

## 1. 編譯(Windows + clang)

- repo 的 Makefile 用 `gcc + make`,本機只有 `clang 22.1.6`(`gcc`/`make`/`cl` 都沒有)。
- 此 fork 其實已含 Windows 支援(`console.c`/`plugin.c`/`runtime.c` 都 `#include <windows.h>`),唯一卡點:`cbmbasic/readdir.c` 的 `#include <windows.h>` 被註解掉了 → 缺 `HANDLE`/`WIN32_FIND_DATAA`。
- 為遵守「不改 `ref/`」,把整份複製到 `temp/p6502_build/`,在副本把那行取消註解,直接用 clang 編譯(`-O3 -Wno-everything`,丟掉 `-Werror`),把 `perfect6502.c netlist_sim.c cbmbasic/*.c readdir.c` 一起連結成 `cbmbasic.exe`。**編譯成功。**

## 2. 效能數據(`--benchmark`,跑到 cbmbasic 的 `READY.`)

| 指標 | 值 |
|---|---|
| Half-cycles | 33,155(固定;到 `READY.` 的指令流是確定的) |
| 時間 | ~1.14 s |
| 速度 | **~28.9K hc/s**(5 次中位數;範圍 28.3K–29.2K) |

對照:真實 1 MHz 6502 跑這段 = 33155 hc / 0.017s ≈ **1.95M hc/s**,所以 perfect6502 ≈ 真機的 1/67(README 自稱高階 2025 CPU 上約 1/30,本機慢一倍,屬正常範圍)。

### 為何不能跟我們的 hc/s 直接比

1. **晶片規模不同**:perfect6502 = 純 6502,**1,725 node / 3,239 transistor**;我們 S1 = 整台 NES(2A03+2C02),lowering 後 **14,723 node / 26,775 transistor** —— 約 8.5 倍節點、8.3 倍電晶體。
2. **hc 單位不同**:perfect6502 的一個 half-cycle = 一次 6502 時鐘 half-cycle(`step()` 切一次 6502 clk);我們的一個 hc = 一次 **NES 主時鐘** half-cycle。2A03 內的 6502 核是主時鐘 /12,PPU 是 /4 —— 兩邊的「hc」代表的晶片推進量差一個數量級以上。**直接比 hc/s 數字無意義**(這點先前已對使用者說明)。
3. 真要比,得用「每秒節點更新量」之類正規化,且還得扣掉事件驅動下大多數節點每 hc 都閒置的事實 —— 我們的 ~109K hc/s × 14.7K node 規模,吞吐並沒有落後。

## 3. 演算法比對 —— 哪些值得參考

`netlist_sim.c` 是 Visual6502 `chipsim.js` 的最佳化 C port,核心和我們同源:

| 機制 | perfect6502 | 我們 S1 |
|---|---|---|
| group 解析 | `addNodeToGroup` 遞迴 + `group_value` 6 級 enum + `getGroupValue` switch | flags-OR → 256-entry `FlagsToState` LUT(MetalNES 衍生,**更強**) |
| **turn-on(gate 升)只 enqueue 單側端點** | `nodes_left_dependant`(每電晶體只收一個非供電端點) | ✅ **已有**(Recalc.cs turn-on 分支只 enqueue c1,BFS 走 ON 通道到 c2) |
| **turn-off(gate 降)收雙端點** | `nodes_dependant`(c1+c2 都收) | ✅ **已有**(turn-off 分支 c1+c2 都 enqueue) |
| enqueue 階段剪枝 | **無** —— 把所有 static dependant 全 enqueue,只靠 `listout` bitmap 去重 + `recalcNode` 內 `value!=newv` 變化檢查過濾 | ✅ **多出 P-1~P-4**(same-state / 單通道 turn-off isolation / 電容守衛 turn-on skip)—— 在「進佇列前」就砍掉,比 perfect6502 早一階,更省 |
| 電晶體去重 | 載入時去重,**含反向 (g,A,B)≡(g,B,A)** | 去重 (g,c1,c2) + 供電正規化到 c2;**反向非供電對沒去**(唯一小缺口) |
| 雙緩衝 ping-pong settle | ✅ `lists_switch` | ✅ |
| group 成員 O(1) 去重 bitmap | ✅ `groupbitmap` | ✅ `_inGroup` |
| settle 迭代上限 | ✅ `max_iterations=50` | ✅ |
| node 值儲存 | **bitmap 1-bit/node**(uint64 packed,8× 密) | byte/node(含多 bit flags) |
| floating group 解析 | **無 hold-previous / 電容**(全浮 → 低) | **有**(LUT hold-previous + largest-capacitance tie-break)—— PPU 動態儲存(OAM/palette)需要,**load-bearing** |

### 三點結論
1. **它的兩個招(單側 turn-on / 雙側 turn-off)我們早就有**,而且我們在其上又疊了 P-1~P-4 —— perfect6502 完全沒有 enqueue 前剪枝。剪枝軸上我們領先。
2. **它的 floating 解析比我們簡單(沒有 hold-previous)** —— 純 6502 撐得過去,但我們要模整台 NES,PPU 的動態儲存非要 MetalNES 那套 LUT + 電容 tie-break 不可。我們的模型更準,而且這準確度是必要的。
3. **bitmap 1-bit node 值**是它唯一明顯不同的資料結構選擇。理論上對 group-walk 的值讀取更省 cache,但 (a) 我們的剪枝/flags 需要 byte;(b) 我們先前 bit-parallel / bitset 實驗都是 dead-end(walk 平均 1.4 節點,bit 操作開銷壓不下來)。不值得。

## 4. 唯一具體的(很小的)候選

**反向重複電晶體移除**:perfect6502 把 `(g,A,B)` 與 `(g,B,A)` 視為同一顆(NMOS pass 對 c1/c2 對稱,電性確實相同);我們只去重 `(g,c1,c2)`,反向非供電對會各留一份。移除是 bit-exact 的(對稱),可略縮 gate-walk 與 group-walk。但:netlist 由 Visual2A03/2C02 自動產生、每顆實體電晶體只出現一次,反向重複只可能來自 module/connection 層,**數量預期極少**(類比 lowering const-fold 那 58 顆的等級,甚至更少)。依「先量候選數再決定」的紀律,要動工前應先量 candidate count;預期收益接近 0,優先度低。

## 5. 對官網的用途

可補進 lineage / comparison 敘述:perfect6502 是同血脈的獨立第三方參考實作,實測證實我們的事件驅動核心 + P-1~P-4 剪枝已達或超過這條血脈;其 hc/s 因晶片規模與 hc 單位不同,不可與我們直接並列比較。
