# 範圍剪枝(+3.6%)與 cls-range(中性)——「查表→ID 區間」公式的適用條件

> 2026-06-10 ~ 06-11。本篇是「範圍化」這條礦脈的完整收尾筆記:一個大勝、一個中性,
> 以及從兩者對照中釘死的適用檢查表。網站版故事見 `WebSite/rcm-revival.html`。

## 背景:三個實驗診斷出真正的瓶頸

2026-06-10 連跑三個記憶體行為實驗(全部 interleaved-paired @400k full_palette `--pin`、位元等價閘門):

| 實驗 | 結果 | 證明 |
|---|---|---|
| 佇列 int→ushort(−60KB 串流) | −0.07%(10/20) | 串流資料被硬體 prefetcher 全蓋,容量不是成本 |
| 彈出迴圈軟體 prefetch(PD=4/8) | −0.29% / −1.05% | 沒有延遲可藏 |
| 純共活性重編號(密度 198.7→109.5 lines/hc,−45%) | −0.36% | 連近乎完美的區域性也不值錢 |

**三個獨立證據指向同一件事:每事件 ~20ns 的地板不是 cache miss,
是「相依載入鏈」——載入→用它載入→依它分支,成本 = 鏈長。**
區域性改變位元組住哪裡,不縮短鏈;唯一的槓桿是**刪鏈節**。

## 勝利:範圍剪枝(range-prune,C# `51e046d` / Rust `665acf5`)

最熱迴圈(`SetNodeState` 入列走訪)每個端點要隨機讀一個 `PruneMask[c]` byte——
L1 命中也是一條 4-5 cycle 的相依載入,而且它坐在**被攜帶的鏈**上
(`nextCount += n`,n 依賴遮罩,下一輪依賴 nextCount)。

**作法:靜態事實不查表,改住在節點 ID 裡。** 自動類別為主鍵重編號
(兩段式載入:pass-1 identity 分類、pass-2 重排重建),四個剪枝類別變四個連續區段:

```
[3,A) skip∩unsafe   [A,S) skip∩safe   [S,B) noskip∩safe   [B,∞) noskip∩unsafe(假節點自然延伸)
turn-off skip ⇔ c < S(supply 1,2 < 3 ≤ S 搭便車)   turn-on unsafe ⇔ c < A || c ≥ B
```

安全機制:每次 Reset 仍算出真遮罩當基準,逐節點驗證範圍推導位元;
不符退安全退化邊界(S=3 只跳 supply、A=∞ 全 unsafe = 剪枝全關但正確)。
位元等價:開機掃描與 checksum 走原序(經排列),三條 golden 全數不變。
區段內 locality 鍵 = clk-BFS 訊號流序(動態 first-touch 的靜態近似;
實測 profile 鍵可再 +1.7%,留作 `--renumber` 診斷,因綁網表不能當預設)。

**實測:C# +3.56%(19/20 & 11/12)、Rust +2.90%(20/20)。**
兩個獨立實作推導出相同邊界 A=460 S=1275 B=7532 = 免費交叉驗證。
Release 驗證後釋放 PruneMask(15KB);Rust 直接 drop。

## 中性:cls-range(2026-06-11,已恢復)

下一張(也是最後一張)熱路徑靜態查表:`IsPureLogic[nn]`(cls 0/1/2,
每彈出讀一次決定 fast-path 分派)。照同一條公式做了完整實作:

- **cls-1 單區間技巧**:block-2 內 cls-1 排頂端、block-3 內 cls-1 排底端,
  隔著 B 相鄰拼成單一連續區間(實測 `cls1=[4124,8053)` 跨 B=7532)
  → dispatch 兩個比較:`(uint)(nn-lo) < len`、`nn >= ClsZeroStart`
- **空槽陷阱**(實作中發現):對齊縫隙的 null 槽會打斷 cls-1 連續性
  → 分類時標 cls=3「萬用牌」,排序跟 cls-2 走、驗證跳過(它們永不入列)
- 基準真相驗證 `[cls-range VERIFIED]` + 全-BFS 安全退化;Release 釋放陣列;
  分支順序/機率與原版相同;golden ×2 + selftest 全過

**實測:median −0.28% / trimmed −0.45%、7/20 → 中性偏負,恢復。**

## 為什麼同一條公式一勝一敗(本篇的重點)

| | PruneMask(勝 +3.6%) | IsPureLogic(中性) |
|---|---|---|
| 載入餵給什麼 | **被攜帶的相依鏈**(`nextCount += n`) | ~70% 預測正確的**分支** |
| 延遲可否被藏 | 不可——鏈一節等一節 | 可——CPU 推測執行先跑了 |
| 刪掉後省到什麼 | 鏈短一節 = 真金 | 本來就被藏住的延遲 = 空氣 |

這也回頭解釋了 2026-06-08「IsPureLogic-fold 中性」:同一條載入,
fold(改從 header 推導)與 range(改比較)兩種刪法都是零——
因為它從來不在關鍵路徑上。

## 範圍化公式的適用檢查表(之後任何候選先過三關)

1. **靜態事實**(載入期可知、執行期不變)
2. **夠熱**(每事件等級的頻率)
3. **坐在被攜帶的相依鏈上** ← cls 敗在這關;遮罩贏在這關

S1 熱路徑已無第三張靜態查表;此礦脈採畢。
更深一層的教訓(同 dominant-bypass、prefetch 系列):
**在這顆 latency-bound 引擎上,一個操作的成本不看它「做什麼」,
看它「擋住誰」——不在串行鏈上的工作,刪了也是白刪。**
