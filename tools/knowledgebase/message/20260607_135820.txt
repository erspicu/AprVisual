### 突破點：電容解耦 (Capacitance Value Decoupling)
你提到「移除安全節點會改變相鄰脆弱節點的 `NodeConnections`，進而破壞 Floating tie-break」。這是一個**自我施加的工程缺陷**。
物理上，節點的寄生電容是 Layout 決定的常數。你的 Lowering 不應該在拓撲改變後重新計算 `NodeConnections`。

**解法**：在執行任何 Lowering 之前，先在最原始的 Pristine Netlist 上計算所有節點的 `NodeConnections`（`c1c2 + gate count`），並將其寫死為一個靜態的 `CapacitanceLUT` 陣列。在 Hot-loop 裡的 Floating tie-break **絕對只讀取這個原始電容值**。
這能徹底將拓撲精簡與電容 Tie-break 脫鉤，保證即使你刪除了安全節點，脆弱節點在 Floating 狀態下的「勝負關係」依然 Bit-exact。

即使解鎖了這個限制，我們來看看你真正能砍掉多少東西：

### 1. 迭代常數折疊 Iterative Constant-Folding (Rail-tied nodes)
**Transformation**:
尋找那些透過 Always-on（`gate == Pwr`）且非 Weak 的晶體管，永久連接到 `Gnd` 或 `Pwr` 的 Normal nodes。將這些節點直接 Fold 進入隱含的 `Gnd` / `Pwr` flags 中。
*   **作法**：若節點 $X$ 永久短路至 `Pwr`，將陣列中所有 $c1/c2 = X$ 或 $gate = X$ 的晶體管端點改寫為 `Pwr`，然後**完全刪除節點 $X$**。改寫 $gate = Pwr$ 會產生新的 Always-on 晶體管，這可能引發進一步的 Static-group merge 或進一步的 Rail-tie。使用 Union-Find 迭代直到 Fixpoint。
*   **為何 Bit-exact**：這些節點在物理上與 Supply rail 無異。它們的狀態被強勢驅動，不會進入 Floating，也不會被 Pull down（否則會觸發 ForceCompute cancel）。把它們當作 Rail 本身完全符合 Bryant 模型的 Flag OR-ing 規則。
*   **節點刪除/重構**：直接刪除節點與冗餘的 Always-on 晶體管，縮小 Working set。
*   **量化預估**：在 Visual6502 的 2A03/2C02 網表中，這類節點主要來自多邊形提取（Polygon extraction）時殘留的 Power distribution network 碎片或未使用的 Pads 綁定。**極限預估能再砍 150 - 300 個節點 (~1% - 2%)**。

### 2. 串聯/並聯晶體管化簡 (Series/Parallel Reduction)
**Transformation**:
*   **Parallel**：你已經做了（Duplicates removal）。
*   **Series**：例如 $A - B - C$ 串聯，且 $B$ 沒有連接其他任何閘極或路徑（純粹的中繼節點）。
*   **為何不可行**：這在純 Switch-level 中是**不可能的**。串聯的兩個晶體管等同於對 Gate 進行 AND 邏輯。要化簡為單一晶體管 $A - C$，這個新晶體管必須有**兩個 Gate**。你的硬性限制禁止了 IR 與複合節點（Composite gates），因此你現有的 `TransistorList` struct 根本無法表達這種結構。
*   **結論**：在不違反 Constraint 1 的情況下，Series reduction 貢獻為 **0**。

### 3. 縮減 Fragile Set (No-PullUp + ForceCompute) 本身
**Transformation**:
*   **ForceCompute 節點 (PPU Data Bus, 709 nodes)**：這些節點依賴 `Gnd + Pwr` 同時存在來觸發 Cancel 機制，以此模擬 NMOS 預充與下拉的類比競爭。如果你改變了任何連接拓撲，導致在特定半週期內無法正確匯集 `Gnd` 與 `Pwr` flags，PPU 讀取就會立刻損毀。**完全不可動**。
*   **No-PullUp 動態與記憶體節點 (OAM/Palette/CPU dynamic, ~8300 nodes)**：這些是晶片儲存狀態的物理載體（Pass-transistor latches）。它們的 State 值必須在 Floating tie-break 中作為歷史記憶向回傳遞（Back-propagation via group walk）。如果你刪除了所謂的「死胡同 (Dead-end)」脆弱節點，它的狀態就不會成為 Group 的 Winner，整個 Latch 的記憶就會丟失。**完全不可動**。

### 4. 非脆弱節點 (~41% PullUp-driven) 的「死胡同」剔除
**為何無效**：
你以為某些 Combinational logic 節點是「Dead-end」（`Gates.Count == 0`），可以被移除。但這在 NMOS 邏輯中是個謬誤。
一個 PullUp 節點 $N$ 即使不驅動任何 Gate，只要它連接在某個 Group 中，它的 `PullUp` flag 就會向整個 Group 廣播。如果你移除了這個「無觀察者」的節點，該 Group 就會失去 PullUp 來源，導致整個 Group 錯誤地解析為 0 或 Floating。
若要安全移除，你必須將 PullUp 屬性轉移到相鄰節點，但如果相鄰節點是 Fragile 的，賦予它 PullUp 會直接改變它的 Resolution 邏輯。這需要極其複雜的布林等價性驗證，在純拓撲操作下幾乎必定打破 Bit-exact。
