# 三個同血脈參考實作的比較:perfect6502 / VisualNes / metalnes(2026-06-08)

把三個 Visual6502 血脈的開關級模擬器都拿來「實際編譯 + 量測 hc/s + 比對演算法」,看有沒有值得學的策略、以及它們的效能水準。**全部在同一台機器、同一顆測試 ROM(`full_palette.nes`)、同樣的 NES 主時鐘 half-cycle 單位下量到的數字**(perfect6502 除外 —— 它只有 6502、hc 單位不同,見下)。

## 0. 結論先講(TL;DR)

| 專案 | 範圍 | 語言 | 本機 hc/s | 演算法特徵 | 相對 S1 |
|---|---|---|---|---|---|
| **VisualNes** (SourMesen) | 整台 NES | C++ | **~24K** | chipsim.js 的逐字 port:group 用 `find()` 線性去重 O(n²)、`std::vector`+`shared_ptr`、**零剪枝** | 4.2× 慢 |
| **metalnes** (iaddis) | 整台 NES | C++ | **~55K** | **S1 的直系祖先**:flags-OR→256-LUT、雙緩衝 ping-pong、單側 turn-on,但**無剪枝、std::vector** | 1.8× 慢 |
| **perfect6502** (mist64) | 只有 6502 | C | **~29K**(註) | 高度最佳化 chipsim:bitmap 狀態、預算 dependant CSR、單側 turn-on | 不可直接比 |
| **我們 S1** | 整台 NES | C# | **~108K** | metalnes 演算法 + R-1 動態單例 + P-1~P-4 剪枝 + SoA 裸指標 + LUT 微優化 | — |
| (我們 Rust) | 整台 NES | Rust | ~101K(已發布) | 同上 port | — |

> **最有價值的發現:量到一條乾淨的「最佳化進度條」** ——
> **VisualNes 24K(未優化 chipsim)→ metalnes 54K(更好的 C++ 祖先)→ S1 108K(我們的剪枝+SoA)**。
> metalnes→S1 的 **約 +100%(2.0×)** 正是我們 R-1 + P-1~P-4 + SoA/native 的成果(同演算法、同範圍、同單位、同機器,bit-exact)；相對未優化 chipsim(VisualNes)約 **4.5×**。沒有任何一個可以「搬過來打贏現況」的東西;三者反而獨立佐證了我們的血脈正確、且優化確實領先。
> **S1 數字 = `--bench-hc 300000 --extra-ram`(專案標準驗證點),3 次 ~107~108.5K(中位數 107.9K),checksum `0x794A43ABDF169ADA` = golden,bit-exact 確認。**(200k 無 --extra-ram 那次 101K 是跑較短、開機暫態攤提較少所致。)

註:perfect6502 的 hc = 一次 6502 時鐘 half-cycle,我們/VisualNes/metalnes 的 hc = 一次 **NES 主時鐘** half-cycle(6502 核是主時鐘 /12)。**單位差一個數量級,29K 不可與其餘三者並列。** 一台 NES frame ≈ 715K 主時鐘 hc(三個整機實作都對得起來)。

## 1. perfect6502(mist64)—— 6502-only,C,高度最佳化

- **編譯**:Windows + clang(無 make/gcc);只需把 `cbmbasic/readdir.c` 被註解掉的 `#include <windows.h>` 取消註解(改在 temp 副本,`ref/` 不動)。
- **量測**:`--benchmark` 跑 cbmbasic 到 `READY.` = 33,155 hc / ~1.14s = **~29K hc/s**(6502-only,1,725 node / 3,239 transistor)。
- **值得學的策略**(我們都已有或更強):
  - **預算 dependant CSR + rising/falling 拆分**:載入時為每個節點預先算好「升緣只收單端點 / 降緣收雙端點」的去重節點清單,熱路徑直接 iterate node 清單而非 walk transistor。→ **S1 的 SetNodeState 已做單側 turn-on / 雙側 turn-off**;perfect6502 把它存成扁平 CSR(更 cache-friendly),但我們的 P-1 same-state 剪枝需要 (c1,c2) 配對,扁平化會失去配對 —— 我們的剪枝省更多。
  - **bitmap 1-bit 節點狀態**(uint64 packed):8× 密、對 group-walk cache 友善。但我們的 flags/剪枝需要 byte,且先前 bit-parallel/bitset 實驗都是 dead-end → 不值得。
  - **反向重複電晶體去重**((g,A,B)≡(g,B,A)):我們只去重 (g,c1,c2)+供電正規化,反向非供電對沒去 —— 唯一小缺口,但 netlist 自動產生、預期數量極少,優先度低。
  - 它**沒有** hold-previous / 電容浮接解析(全浮→低);純 6502 撐得過,我們模整台 NES 非要不可。
- 詳見 `MD/note/2026-06-08-perfect6502-比較.md`。

## 2. VisualNes(SourMesen)—— 整台 NES,C++,未最佳化的 chipsim.js port

- **編譯**:把 flat 的 `Core/*.cpp` + 自寫 headless 驅動(parse iNES → 載入 prg/chr → 上電 reset → soft reset 重抓向量 → 計時 step N)用 clang++ `-std=c++17` 直接編成 exe。**乾淨好編**(Core 沒有 GUI 糾纏)。
- **量測**:`full_palette.nes`,**~24K hc/s**(5 次中位數,整台 NES,與我們同單位)。
- **演算法**(`chipsim.cpp`):Visual6502 `chipsim.js` 的**逐字 C++ port**:
  - `addNodeToGroup` 用 `std::find(group...)` **線性掃描去重 O(group²)** —— 我們/perfect6502 用 O(1) bitmap/旗標。
  - `std::vector`+`shared_ptr` 在熱路徑進進出出。
  - `getNodeValue`:hasGnd&&hasPwr 時用一張**寫死的 node-id 清單**當 ForceCompute(359,566,…);浮接用 **area(電容)大者勝** —— 跟我們同源(這點它有,perfect6502 沒有)。
  - **完全沒有 enqueue 前剪枝**(P-1~P-4)。
- **值得學的**:坦白說沒有 —— 它是「正確性參考」等級,所有它有的我們都有且更快。它的價值是當**未優化 chipsim 基準線**:整台 NES 24K,正好凸顯我們 ~4× 的優化幅度。

## 3. metalnes(iaddis)—— 整台 NES,C++,**S1 的直系祖先**

- **它是什麼**:`MD/note` 早就記載 S1 是「metalnes `wire_compute` 的 headless C# port」。本次直接讀原始碼確認:metalnes 的 `setNodeState` / `processQueue` / `computeNodeGroup` / `_node_infos[].​_tlist_gates` + 0 結尾 `(c1,c2)` `_transistor_list` 佈局 = **S1 `SetNodeState`/`ProcessQueueInterp` 在加剪枝之前的逐字原型**;S1 的 `RecalcListNext`/`RecalcHashNext` 就是它 `_recalc_list_next`/`_recalc_hash_next` 的 port。
- **編譯(本次重點工程)**:metalnes 是 OSX-only 的 Apple **Metal** + imgui + 音訊 GUI app(README 自承「Needs lots of optimization」)。在使用者授權「可修改 metalnes code」下,於 temp 副本把 wire 核心**抽成 headless**:
  - stub 掉 `imgui_support.h`(極簡假 ImGui)、`triangulate.h`(no-op `BuildTriangleList`)、`raster_device.h`/`audio_device.h`(只留型別 + `saveImage`/`saveAudio`,**device 傳 nullptr**;handler_nes_system 用 `if(device)` 守衛,所以不需真實作)。
  - neuter `handler_nes_system::serialize_fields`(避免序列化 stub device)。
  - Windows 相容 shim:`-DWIN32`(原碼用 `WIN32` 非 `_WIN32`)+ 強制 include `compat_win.h`(`strcasecmp→_stricmp`、`mkdir→_mkdir`、`timegm→_mkgmtime`、`localtime_r→localtime_s`),修 `File.cpp` 的 `_mkdir` 兩參數 bug;`-DCATCH_CONFIG_DISABLE` 關掉內嵌的 Catch2 單元測試;stub `SetCurrentThreadName`。
  - 驅動:照 `system_state::Create` 的配方 load `nes-001`+`cart-mmu0`(+prgrom/chrrom)、`setupRom`、attach `handler_rom`/`handler_ram`/`handler_nes_system`、reset、計時 `wires->step(N)`。**跳過 video_out/audio_out 類比輸出 handler**(那才是需要 raster/audio device 的部分),所以這量到的是「引擎 + 邏輯 + RAM/ROM/匯流排」,與 S1 headless step 對等;完整 app 會再慢一截(類比 ladder + 渲染 + imgui)。
- **量測**:`full_palette.nes`,**~54K hc/s**(整台 NES,同單位)。一台 frame ≈ 715K hc,和 S1「101K → 7.07s/frame」對得起來 → 單位一致、確實在跑(穩定 ~18µs/hc;若卡死會快好幾個數量級)。
  - **以我們標準 200,000 hc 複測(回應「沒那麼快吧」的質疑):單跑 200k = 55.8K;拉到 12×200k = 2.4M hc 看 rate 曲線,全程穩在 52~56K**,包含 chunk 7~11(hc>1.42M,已過 full_palette 開機的 2× vblank-wait、進入主迴圈/實際描繪)—— 所以 55K **不是**開機暫態假象,steady-state ~54K 是真的。
  - **重要 caveat**:此數字是「引擎 + 邏輯 + RAM/ROM/匯流排」**核心**,我**跳過了 `video_out`/`audio_out` 類比 ladder handler、也沒有 GUI/imgui/渲染迴圈**。完整的 metalnes 桌面 app(類比複合視訊+音訊+渲染)會**比這慢** —— 那才是一般印象中的「metalnes 很慢」。但對 S1 比較而言這是**公平**的:S1 benchmark 同樣不算類比輸出(只 step + checksum NodeStates),且兩邊都跑同一顆輕量 ROM。
- **值得學的策略**:它就是 S1 演算法的來源,所以**沒有新東西可學 —— 我們全拿了還往前走**。它證明的是:
  1. 我們的 256-LUT group 解析(分辨「driven high」vs「hold previous」)、weak/depletion 旗標、largest-capacitance 浮接 tie-break、behavioral RAM/ROM handler、module(.js)系統、ForceCompute —— 全部源於 metalnes,是這條血脈中最完整的數位模型。
  2. **metalnes 55K → S1 101K = +80%**,就是我們在祖先之上加的:**R-1 動態單例 fast-path + P-1~P-4 事件數剪枝 + SoA 裸指標 + LUT 微優化**。這是我們優化價值的乾淨第三方對照。
  3. metalnes 還模了**類比複合視訊 + 音訊電壓 ladder**(perfect6502/VisualNes/我們都沒有)—— 那是超出我們「數位邏輯 + behavioral 記憶體」範圍的東西,不是效能而是模擬深度。

## 4. 對官網/README 的用途

可補進 lineage / comparison:
- 同血脈三實作實測,排出 **24K(VisualNes,未優化 chipsim)< 55K(metalnes,我們的祖先)< 101K(S1,我們)** 的進度條,客觀量化我們相對直系祖先的 **+80%**、相對未優化 chipsim 的 **~4.2×**。
- perfect6502(29K)因只有 6502 且 hc 單位不同,**不可與整機數字並列**;放在「另一條(6502-only)分支的最佳化參考」即可。

## 5. 量測方法(可重現)

- 機器/ROM/單位一致;每個都先 warmup(跳過開機暫態)再計時、取 5 次中位數。
- 建置副本在 `temp/{p6502_build,vnes_build,mn_build}/`(gitignored);原始 clone 在 `ref/{perfect6502,VisualNes,metalnes-main}/`(gitignored、唯讀;metalnes 僅在副本中為 headless 化而改)。
- S1 數字:已發布的 `AprVisualBenchMark/win/csharp/AprVisual.S1.exe`(version fb97342)。
  - `--bench-hc 300000 --extra-ram`(標準驗證點)= **~108K hc/s**(中位數 107.9K,3 次 107.0~108.5K),checksum `0x794A43ABDF169ADA` = golden ⇒ bit-exact。
  - `--bench-hc 200000`(無 --extra-ram)= 101.1K(較短、攤提較少)。
