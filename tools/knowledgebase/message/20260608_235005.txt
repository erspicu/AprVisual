Here is the blunt, structural truth. The user’s idea is functionally a **fragmented, worse version of Escape-1**. It will not speed up your single-core simulation if you maintain the per-node bit-exact constraint. 

Here is the exact breakdown of why, mapped to your four questions:

### 1. Does restricting extraction to ONLY storage elements escape the Escape-1 failure causes?
**No. It makes the boundary cost strictly worse.** 
In Escape-1, you extracted logic *and* registers, creating macro-blocks. By extracting *only* registers/latches/buses and leaving the logic at switch-level, you are drastically increasing the surface area of the behavioral ↔ switch-level boundary. 

Because 94% of the chip is a giant pass-transistor SCC, a register is rarely cleanly isolated by logic gates. It is coupled to the combinational fabric via pass-transistors (transmission gates). If you abstract the storage nodes into an array:
* **Failure 1 (Min-granularity):** Worsened. The event engine touches ~1.4 nodes per event. Now, instead of resolving a local topological switch-equation, an event hitting a latch input must trigger a behavioral callback, read the array, compute, and push events back into the switch fabric. The overhead of crossing domains happens *more frequently* than in Escape-1.
* **Failure 2 (Internal nodes continuously read):** Still exists. The SCC means the logic natively sees the internal impedance/charge of the latch. If you hide the latch in an array, the switch-fabric evaluating an attached pass-transistor still needs to know the state of that node to resolve its own equations.
* **Failure 3 (Per-node checksum):** Still exists.

### 2. Is there a BIT-EXACT-preserving formulation?
**For single-core emulation speed: No.**
**For analysis/verification: Yes, highly valuable.**

**(a) Reconstruction cost vs. Event-driven silence:** 
To pass your FNV checksum, every node's state must be present in the `NodeStates` array in order. If you pack a 4-node latch into a 1-bit array entry, you must unpack it before the checksum. 
*The baseline engine does exactly zero memory writes for a sleeping latch.* If you unpack behavioral states every half-cycle to satisfy the checksum, you are trading an O(0) operation (event-driven silence) for an unconditional O(N) memory-scatter operation. Even if you only unpack "on change," the branching and dictionary-lookup overhead to map array indices back to FNV-hashed node IDs will cost more memory bandwidth than you save.

**(b) The Analysis pivot:** 
If your actual goal is to build an understandable, verifiable behavioral model (e.g., lifting the netlist to C-code for cycle-accurate emulation or formal verification), this extraction is the *only* way to do it. You just have to drop the per-node bit-exact constraint and check equivalence only at the architectural state boundaries (CPU registers, memory buses). But if the goal is to speed up *this specific switch-level engine* running *this specific per-node testbench*, it is a dead end.

### 3. The specific speed angle: Targeted storage-loop skip?
**Bit-exact: Yes. Beats current engine: No. This is the no-change-prune problem again.**

A dynamic latch hold-loop re-settles because a nearby node (like a pass-transistor gate) wiggled. In your Bryant relaxation solver, evaluating that wiggle is just a few bitwise operations on already-cached L1 data. 

To implement a "targeted skip," the engine must:
1. Fetch the event.
2. Look up metadata: "Is this node part of a dynamic latch?"
3. Fetch the clock/strobe node state for that latch: "Is it currently written to?"
4. Branch: `if (!clocked) continue;`

You are inserting dependent memory reads (metadata + clock state) and a dynamic branch into your tightest inner loop to avoid computing a simple switch-level boolean equation. On a modern CPU, the branch misprediction risk and cache-line fetches for the metadata will cost more cycles than just doing the redundant math and letting the `if (new_state == old_state)` check naturally terminate the event chain. 

### 4. Blunt Summary
**Is there ANY version of this that beats the current ~110K hc/s engine while staying per-node bit-exact?**
**No. 0% chance.**

The honest answer is: **Escape-1 already proved the reducibility but also that it doesn't buy single-core speed, and your detection just re-finds a subset of the exact same reducible elements.** 

You are memory-latency bound. Your event engine runs at the natural minimum granularity of the circuit physics. Any abstraction you build requires bridging a behavioral model back to a switch-level SCC, which introduces boundary-translation overhead and state-reconstruction overhead. Because the baseline engine relies on event-silence (doing nothing when things don't change), replacing that silence with structural array lookups or skip-logic branches will always be a net negative for performance.

Abandon extraction for emulation speed. Keep it strictly if you want to write a static behavioral lifter/compiler.
