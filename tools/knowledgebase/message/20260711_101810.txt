# Consult: deep recursion / stack overflow in a switch-level NES simulator's settle↔callback loop

I want your independent analysis of a stack-overflow bug and a critique of my proposed fix. Please
be skeptical — tell me if the fix is a band-aid over a design flaw, and what the idiomatic fix is.

## Context: the engine

AprVisual.S1 is a switch-level (transistor-level) NES simulator in C# (.NET 11, `unsafe`, x64).
It models the 2A03 CPU + 2C02 PPU as a netlist and settles node states event-driven, the classic
Visual6502/perfect6502 way. It is heavily optimized (SoA, unmanaged arrays, zero-bounds-check hot
loops); it runs bit-exact and we guard that with a golden checksum.

Two mechanisms matter here:

1. **`ProcessQueue()`** — the settle loop. It drains a work-list of dirty nodes, recomputing node
   groups until quiescent (a double-buffered FIFO of "waves"). NES settle normally converges in
   ~7–45 waves. It is the hot method; RecalcNode/ComputeNodeGroup/SetNodeState all inline into it.

2. **behavioral memory** — RAM/ROM aren't modeled transistor-by-transistor; they are "fake
   transistors": a callback fires when a chip-select/clock edge settles, and the callback reads
   address/`R/W` off the (real, switch-level) bus nodes and either latches the data bus into a byte
   array or drives the data-out bus nodes with `mem[addr]`. Driving those nodes re-dirties the
   netlist, so the callback calls `ProcessQueue()` again.

## The recursion cycle (this is the crux)

`ProcessQueue()` calls `InvokeCallbacks()` at the END of settling. A memory callback then drives bus
nodes via `WriteBits()`, which — if any node changed — calls `ProcessQueue()` **again, synchronously,
nested inside the callback**, before the outer `InvokeCallbacks()` loop can continue. So:

```
StepCycle
  └ ProcessQueue            (settle)
       └ InvokeCallbacks
            └ DoMemReadWrite (a callback)
                 └ WriteBits → if(changed) ProcessQueue   ← recurses
                      └ InvokeCallbacks
                           └ DoMemReadWrite → WriteBits → ProcessQueue → …   (nests deeper)
```

Each memory operation that changes the bus adds one level of `ProcessQueue`→`InvokeCallbacks` nesting.

### The actual code

`ProcessQueue()` (tail):
```csharp
private static void ProcessQueue()
{
    while (RecalcListNextCount != 0)
    {
        // swap next↔current wave buffers, then for each dirty node: RecalcNode(nn)
        ...
    }
    InvokeCallbacks();   // <-- runs callbacks after the netlist is quiescent
}
```

`InvokeCallbacks()`:
```csharp
internal static void InvokeCallbacks()
{
    while (_pendingCallbacks.Count > 0)
    {
        (_pendingCallbacks, _processingCallbacks) = (_processingCallbacks, _pendingCallbacks); // swap
        for (int i = 0; i < _processingCallbacks.Count; i++)
        {
            var cb = _processingCallbacks[i];
            cb.Enqueued = false;
            switch (cb.Kind)
            {
                case HandlerKind.MemReadWrite: DoMemReadWrite(cb); break;   // may drive the bus → ProcessQueue → InvokeCallbacks (nested!)
                case HandlerKind.MemRead:      DoMemRead(cb); break;
                case HandlerKind.Video:        DoVideo(cb); break;
                ...
            }
        }
        _processingCallbacks.Clear();
    }
}
```

`DoMemReadWrite()` (a callback body) and the `WriteBits` that closes the loop:
```csharp
private static void DoMemReadWrite(CallbackInfo cb)
{
    if (NodeStates[cb.Cs] != 0) return;                       // chip not selected
    int address = ReadBits(cb.Addr, cb.ALen);
    if (NodeStates[cb.We] == 0) cb.MemData[address & cb.Mask] = (byte)ReadBits(cb.DataOut, cb.DLen); // write
    else                        WriteBits(cb.DataOut, cb.DLen, cb.MemData[address & cb.Mask]);        // read → drive bus
}

public static void WriteBits(int* nodes, int len, int value)
{
    bool changed = false;
    for (int i = 0; i < len; i++)
    {
        if ((value & (1 << i)) != 0) changed |= SetHighQueued(nodes[i]);
        else                          changed |= SetLowQueued(nodes[i]);
    }
    if (changed) ProcessQueue();     // <-- the recursion closes here
}
```

The comment on `InvokeCallbacks` claims it is "re-entrant-safe via swap-and-drain: a callback may
drive nodes → ProcessQueue → InvokeCallbacks recursively; the new pending list gets drained on the
next outer-loop iteration." Note `_pendingCallbacks` / `_processingCallbacks` are **shared static
`List<CallbackInfo>`**, and the nested call swaps them again while the outer `for` loop is mid-iteration
over `_processingCallbacks`.

## The symptom

   at AprVisual.Sim.WireCore.InvokeCallbacks()

So the `ProcessQueue`↔`InvokeCallbacks` nesting reached ~24,021 deep in a single `StepCycle`, blowing

- **Open question I cannot yet answer: is this bounded-but-deep, or an oscillation that would recurse

private static int RunOnBigStack(Func<int> body)

Here is my independent analysis of your engine’s settle/callback loop. 

To answer your questions directly: **Your user is right to flag the design smell, and the 512 MB stack is a dangerous band-aid.** The 24,000-deep recursion is not a legitimate cascade of NES logic; it is a phantom amplification caused by a **severe state-corruption bug** in your re-entrancy handling.

Here is the concrete, code-level breakdown of what is actually happening, followed by the idiomatic fix.

---

### 3. Re-entrancy Correctness (The Root Cause)
*Let's start here, because it answers almost everything else.*

Your comment on `InvokeCallbacks` states it is "re-entrant-safe via swap-and-drain." **It is not.** Because `_pendingCallbacks` and `_processingCallbacks` are shared statics, re-entering `InvokeCallbacks` destroys the iteration state of the outer loop.

Let’s trace a nested call where `_pending` happens to have two callbacks, `A` and `B`:

1. **Outer `Invoke`:** Swaps lists. `_processing` now holds `[A, B]`.
2. **Outer Loop (`i=0`):** Evaluates `A`. `A` dirties nodes and recurses into `ProcessQueue` -> `InvokeCallbacks` (Inner). Let's say `A` triggered a new callback, `C`.
3. **Inner `Invoke`:** Swaps lists! `_processing` becomes `[C]`. `_pending` becomes `[A, B]` (the outer loop's list!).
4. **Inner Loop (`i=0`):** Evaluates `C`. 
5. **Inner End:** The inner loop finishes and calls `_processingCallbacks.Clear()`. **`[C]` is destroyed.**
6. **Inner `while` loop continues:** It sees `_pending` has items `[A, B]`. It swaps again. `_processing` becomes `[A, B]`.
7. **Inner Loop (`i=0`):** **Evaluates `A` a second time!**
8. **Inner Loop (`i=1`):** Evaluates `B` (which the outer loop hasn't even gotten to yet).
9. **Inner End:** Calls `_processingCallbacks.Clear()`. **`[A, B]` is destroyed.** Inner call returns.
10. **Outer Loop Resumes:** `i` increments to `1`. It evaluates `i < _processingCallbacks.Count`. But `_processingCallbacks` was cleared by the inner loop! `1 < 0` is false.
11. **Outer End:** The outer loop terminates prematurely, missing items, having executed things out of order, and having evaluated `A` redundantly.

### 4. Bounded vs Unbounded
You asked if the 24k depth is a bounded cascade or an infinite oscillation. 

It is an **oscillation that terminates only because of physical netlist gating**. 
Because the static list swapping causes callbacks to be evaluated repeatedly, your engine is thrashing in a chaotic blender. It doesn't infinite-loop purely because of this line in your callback:
`if (NodeStates[cb.Cs] != 0) return;`
Eventually, a redundant evaluation hits a stable netlist state that acts as a no-op, breaking the recursion chain. The 24,000 depth is not the NES doing 24,000 dependent memory operations; it is your loop tearing itself apart and accidentally surviving.

### 1. The Big-Stack Thread Fix
This is a pure band-aid. By moving to a 512 MB stack, you are just giving the engine enough memory to survive the chaotic re-evaluation thrashing triggered by AccuracyCoin. It does not fix the state corruption.

### 2. Should `WriteBits` / Callbacks NOT recurse?
You are completely correct. Tying the host application's call-stack depth to simulated physical time/activity is a fundamental anti-pattern in hardware simulation. 

The idiomatic design for an event-driven simulator is a **Trampoline (an outer `do...while` loop)**. `WriteBits` should mutate the netlist state, and the top-level driver should manage the settling.

**Will it be bit-exact?** 
Because your current engine processes nested callbacks *Depth-First*, moving to an iterative loop *might* change the execution order to *Breadth-First*. Given that your golden checksum currently bakes in the results of the "static list tearing" bug described above, fixing the engine to be logically sound *might* alter the checksum for AccuracyCoin. If it does, the new checksum is the correct one. (Your baseline regression suite likely passes because `_pending` rarely exceeds a length of 1, masking the list-tearing bug).

---

### 5. The Fix (Iterative Trampoline Sketch)

We must break the mutual recursion while maintaining exact node-settling timings (zero-allocations, hot-path optimized). 

Here is the minimal, low-risk way to flatten the engine.

**Step 1: Break the cycle in `ProcessQueue`**
Remove the call to `InvokeCallbacks()` at the end of the settle loop.
```csharp
private static void ProcessQueue()
{
    while (RecalcListNextCount != 0)
    {
        // ... swap, RecalcNode(nn) ...
    }
    // REMOVED: InvokeCallbacks(); <-- Cycle broken here
}
```

**Step 2: Leave `WriteBits` EXACTLY as it is**
You *want* memory callbacks to settle the bus synchronously before the next callback reads the bus. Leaving `ProcessQueue` here ensures exact physical timing.
```csharp
public static void WriteBits(int* nodes, int len, int value)
{
    // ... set high/low queued ...
    if (changed) ProcessQueue(); // Keep this! It returns cleanly now.
}
```

**Step 3: Add the Trampoline at the Top Level**
Push the mutual recursion out into an iterative loop inside your main cycle step.
```csharp
public static void StepCycle()
{
    do
    {
        // 1. Settle any dirty nodes first
        ProcessQueue(); 

        // 2. Process ONE wave of callbacks sequentially
        // Because ProcessQueue no longer calls InvokeCallbacks,
        // this is now 100% re-entrant safe.
        InvokeCallbacks(); 

    } 
    // Loop until both the netlist is quiet AND no callbacks are pending
    while (RecalcListNextCount != 0 || _pendingCallbacks.Count > 0);
}
```

### Summary of what this fix achieves:
1. **Zero Recursion:** `ProcessQueue` and `InvokeCallbacks` are now strictly sequential. Maximum stack depth is `StepCycle -> InvokeCallbacks -> DoMemReadWrite -> WriteBits -> ProcessQueue`. Depth is fixed at ~5 frames.
2. **Re-entrancy fixed:** Because `InvokeCallbacks` cannot be paused mid-loop by a nested call to itself, your static `_pending` and `_processing` lists are completely safe and require no allocations or structural changes.
3. **No big stack needed:** You can safely revert the 512 MB thread wrapper.
4. **Performance:** This is functionally identical in overhead to your current code, minus the function call prologue/epilogue overhead of 24k deep stacks. It should be slightly faster.
