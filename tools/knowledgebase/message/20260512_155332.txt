### 1. 怎麼斷環？(Global `Prev` vs. Targeted Cut)

**絕對不要對所有 pass-transistor latch 全局一律 `Node → Prev`**，這會引發嚴重的 over-cut（導致 data-path 的半週期延遲，破壞 critical path）。

**務實的演算法：Targeted SCC Back-edge Cut**
你需要利用圖論（DFS back-edge）只針對這 56 個 SCC 進行精準打擊。

1. 對每一個剩下的 SCC，隨機（或 heuristic，見下）選定一個起點做 DFS 遍歷，標記出所有的**回邊（Back-edges）** $u \rightarrow v$。
2. 檢查回邊的源頭 $u$：如果 $u$ 符合「latch 類 node」的特徵，就把這條回邊切斷。
3. **切斷的具體操作**：在目標節點 $v$（以及 SCC 內其他引用 $u$ 的 current 值且構成環的節點）的 `NextExpr` 中，把 `NodeRef(u)` 替換成 `Prev(u)`。
4. **Heuristic（選誰當回邊最安全）**：

### 2. Clock 訊號在 `Mux(clk_phase, ...)` 裡該用 `Node` 還是 `Prev`？

**必須一律使用 `Node(clk_phase)`（Current Value）。**

*   **物理原因**：Transmission gate 是 level-sensitive，它的透明與否取決於**此時此刻**（這個半週期求值後）的 clock level。如果你用了 `Prev(clk)`，等於你的 latch 永遠看「上一個半週期的 clock」，這會導致所有資料的載入慢半拍，嚴重的會漏掉只有一個 half-cycle 寬的 clock pulse。
*   **會不會繞回 Cycle？**：極大概率不會。因為 clock 產生器（divider chain）通常是單向 feed-forward 給這些 counter 和 register 的。即使 clock 產生器自己也是個 phase ring（例如 Johnson counter），它自己的內部 SCC 會被上一題的機制解掉。解掉後，它對外輸出的 `Node(clk)` 已經是 acyclic 的 DAG 節點，直接讀取絕對安全。

### 3. 「組合邏輯 node」（NOR detector）怎麼辦？

你推測得完全正確：**讓 Detector 讀 `Prev(latch)`，它們就會自動變成 acyclic，這是 physically correct 的。**

*   **物理驗證**：Counter 在 clock edge 發生時更新（load 新值）。Reload detector 的條件（例如 counter == 0）必須在 clock edge **來臨前**就 evaluate 完畢並穩定。
*   因此，Detector 讀取的本來就是「Counter 更新前的穩態值」。在你的 IR 語意下，目前半週期的「更新前穩態值」就是 `Prev(latch)`。
*   把你找到的回邊（通常就是 Latch 灌給 Detector 的邊）改成 `Prev`，Detector 就只依賴上一狀態，不再跟 Latch 的 current 值打架，SCC 瞬間瓦解。

### 4. DMC 的 496-node SCC（包含 Carry Chain）

**NES 的 APU down-counters 絕大多數是 Synchronous Counters，而非 Asynchronous Ripple Counters。**

*   仔細看你抓出來的 log：`13237` 和 `13262` 吃的 clock 都是 `[n11258, n11259]`。這證明了整個 counter chain 的每一個 bit 都是**同相/同步**更新的！
*   所謂的「Carry chain」其實是組合邏輯：`carry_i = carry_{i-1} & bit_{i-1}`。
*   **結論**：因為所有 bit 同步 load，所以 Carry 邏輯必須根據**更新前的狀態**來計算。
*   **演算法套用**：在 SCC 中，把 Latch 灌給 Carry 邏輯（和 Reload 邏輯）的邊切成 `Prev`。由於 Carry chain 內部純粹是組合邏輯（`carry_i` 依賴 `carry_{i-1}` 的 current 值），這部分**不用切**，它本來就是一條單向的 acyclic chain。只切 Latch $\rightarrow$ 組合邏輯 的源頭，整個 496-node SCC 就會漂亮地展開成一棵巨大的 DAG。

### 7. 會不會 Over-cut？(NES 的即時讀取 Edge Cases)

**如果你用 DFS Back-edge 只切 SCC 內部的邊，Over-cut 的機率是 0。**

*   **危險的反例（全局替換的風險）**：假設有一個純 Data-path：Latch A (clocked by $\phi_1$) $\rightarrow$ 組合邏輯 B $\rightarrow$ 輸出給外面。如果在 $\phi_1$ 剛 high 時，Latch A load 了新值，組合邏輯 B 應該在**同一個半週期**立刻反應並傳遞下去。如果你全局把 Latch A 的輸出改成 `Prev`，B 就要等到下個半週期才會有反應。這在 CPU ALU 或 Address Decoder 中會是致命傷。
