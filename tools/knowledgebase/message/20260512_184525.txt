RESPONSE
========================================================================
這是一個非常核心且硬派的模擬器架構問題。你在 $\gamma.4$ 遇到的 rare mismatch 與巨型 SCC，正是因為**「把具有雙向/廣播特性的傳輸閘（Pass Transistor），硬塞進單向的 Boolean DAG 中」**所導致的。

要用 branch-free bit-sliced（64 lanes）來解這個問題，我們不需要完整的整數強度，只需要**「3 個 Bit-Plane 代表 3 種驅動力」**。

以下是針對你問題的精確規格與實作指南：

### 1. ＆ 2. 精確 (Val, Strength) 模型與 Bit-sliced 實作

在 NES (NMOS) 的物理與 S1 模型中，強度階層可以被精簡為 3 種主動驅動態 + 1 種浮接態。因為我們處理的是 64 台機器的 `ulong`，我們不對每個 node 存 `int Strength`，而是為每個 Bus Node $v$ 宣告 3 個 `ulong` 變數來代表它的狀態：

*   `S0` (Strong 0)：強拉低。來自 GND (`PullDown`) 或輸出 0 的邏輯閘。**（最強，優先權最高）**
*   `S1` (Strong 1)：強拉高。來自直接 VCC (`StrongVcc`)、預充閘 (`PullUpCond`) 或輸出 1 的邏輯閘。
*   `W1` (Weak 1) ：弱拉高。來自 Depletion Load (`StaticLoad`)。
*   (如果以上皆為 0，就是 Float / Hold)。

#### Step A: 初始候選驅動 (Local Candidate Generation)
在每一輪 Bus Resolution 開始時，對於每個 Bus Node $v$，先從它**自身**與**非 Bus 的邏輯節點**收集初始驅動力：

```c
// 1. 本身的 PullDown / PullUp
ulong init_S0 = di[v].PullDown ? ~0ULL : 0ULL;
ulong init_S1 = (di[v].PullUp == StrongVcc || di[v].PullUpCond) ? ~0ULL : 0ULL;
ulong init_W1 = (di[v].PullUp == StaticLoad) ? ~0ULL : 0ULL;

// 2. 外部 Handler (若此 bus 是 Board BD，Memory Handler 會在這裡注入)
// inject_handler_mask: 該 lane 正在被 handler 寫入
// inject_handler_val:  寫入的 0/1 值
init_S0 |= (inject_handler_mask & ~inject_handler_val);
init_S1 |= (inject_handler_mask &  inject_handler_val);

// 3. 來自「單向寫入」的 Pass Ports (Other 是被決議過的 Logic Node，非 Bus)
for (pass in di[v].LogicPasses) {
    ulong cond = Eval(pass.Cond);   // 傳輸閘導通條件
    ulong val  = Eval(pass.Other);  // Logic node 的值
    init_S0 |= (cond & ~val);       // Logic node 輸出 0 視為強拉低
    init_S1 |= (cond &  val);       // Logic node 輸出 1 視為強拉高
}

S0_curr[v] = init_S0;
S1_curr[v] = init_S1;
W1_curr[v] = init_W1;
```

#### Step B: 傳輸閘網路廣播 (Bus-to-Bus Propagation)
Bus 之間會透過 Pass Transistors 短路。此時強度不會衰減，雙向互傳。你需要一個 Inner Loop 來讓網路收斂。

```c
for (int pass_iter = 0; pass_iter < K_PASS; pass_iter++) {
    // 必須使用 next buffer 避免計算順序導致的 artifacts
    for (int i = 0; i < NUM_BUSES; i++) {
        S0_next[i] = S0_curr[i];
        S1_next[i] = S1_curr[i];
        W1_next[i] = W1_curr[i];
    }
    
    for (int i = 0; i < NUM_BUSES; i++) {
        for (pass in di[i].BusPasses) {
            ulong cond = Eval(pass.Cond);
            int other_bus = pass.OtherBusIdx;
            
            // 雙向短路：任何一邊有的強度，另一邊也會有
            S0_next[i] |= (cond & S0_curr[other_bus]);
            S1_next[i] |= (cond & S1_curr[other_bus]);
            W1_next[i] |= (cond & W1_curr[other_bus]);
        }
    }
    // Swap next to curr...
}
```

#### Step C: 最終決議 (Resolution)
當廣播結束後，每個 Bus $v$ 會根據 S1 的物理優先權（GND 贏 VCC/PullUp 贏 Depletion 贏 Hold）產生最終的 0/1 值：

```c
for (int i = 0; i < NUM_BUSES; i++) {
    ulong has_s0 = S0_curr[i];
    ulong has_s1 = S1_curr[i];
    ulong has_w1 = W1_curr[i];
    ulong hold_v = PrevState[i]; 

    // Bitwise 決議邏輯：
    // 如果 has_s0 為 1，結果強制為 0 (~has_s0 遮罩一切)
    // 否則如果 has_s1 為 1，結果為 1
    // 否則如果 has_w1 為 1，結果為 1
    // 否則保持 hold_v
    BusVal[i] = (~has_s0 & has_s1) | 
                (~has_s0 & ~has_s1 & has_w1) | 
                (~has_s0 & ~has_s1 & ~has_w1 & hold_v);
}
```

### 3. K_BUS 大概要多少？為什麼比巨型 SCC 小？

**為什麼之前會變巨型 SCC？**
在 $\gamma.4$ 中，你把「Bus 值」當作 DAG 的一個節點，傳輸閘的導通條件與 Bus 值互相依賴。因為 Bus 是雙向的，A 依賴 B，B 依賴 A，DAG 化後整個資料排線和驅動它的 ALU/Registers 全被捲進同一個圖裡。

**改成 Ping-Pong 後的架構與 K 值：**
這個架構把**「邏輯計算 (DAG)」**與**「電路拓撲短路 (Bus Resolver)」**切開。
外層結構是：
```c
for (int outer = 0; outer < K_OUTER; outer++) {
    Eval_DAG();       // 單向邏輯 (AND/OR/NOT)，不含 Bus
    Eval_SCCs();      // 小殘餘環 (Flip-Flops)
    Eval_Buses();     // 裡面包著 K_PASS 的傳輸閘廣播
}
```

*   **`K_PASS` (內部廣播)**：NES 的 Bus-to-Bus 直接短路深度（比如一個 pass 接另一個 pass）通常極淺。`K_PASS = 2~3` 絕對足夠讓同一個 cycle 內短接的 Bus 形成同電位。
*   **`K_OUTER` (外部反饋)**：如果一個 Bus $A$ 變了，觸發一個 logic node $L$，$L$ 又打開另一個傳輸閘驅動 Bus $B$。這種「Bus $\rightarrow$ Logic $\rightarrow$ Bus」的因果鏈在 NES 裡通常也是同步設計（跨 cycle），在半週期瞬態內非常少見。`K_OUTER = 2~3` 就能達到 100% 收斂。
*   **總結：** 這個分離設計讓圖的大小從 $O(V)$ 降到 $O(K_{outer} \times K_{pass})$，因為 90% 的 node 都在 DAG 裡一波帶走了。

### 4. 驗證策略與 Debug 區分

先照上述架構寫死 `K_OUTER = 3`, `K_PASS = 3`。跑 `--trace-cmp` 對決 S1。

**如果發生 Mismatch，如何區分是 K 不夠還是模型錯？**
1.  **加大 K 測試**：直接把 `K_OUTER=10`, `K_PASS=10`。如果 mismatch 消失，那就是 K 不夠。如果還在，就是模型問題。
2.  **觀察 Mismatch 的態樣**：
    *   *模型錯（強度問題）*：S1 說是 0，你的模型說是 1。把該 Node 的 `S0`, `S1`, `W1` dump 出來。如果發現該 Node 同時被 `S0` (e.g. GND) 和 `S1` (e.g. VCC) 驅動，但 S1 的結果卻跟你的規則矛盾，你就抓到 S1 特殊的 corner case 了。
    *   *模型錯（Float 電容問題）*：如果 dump 出來發現你的 `S0`, `S1`, `W1` 全是 0，表示它進入 Hold。如果 S1 的結果與你的 Hold 值不同，通常就是「浮接電容競爭」造成的（見第 5 點）。

### 5. 更簡單的方案與「忠於物理 vs 忠於 S1 Artifact」

你問到是否能接受 $\gamma.4$ 在 `io_db*` 上的 rare mismatch，這是個靈魂拷問。

**為什麼浮接狀態 (Float) S1 會算出不一樣的值？**
在 Visual6502 (S1) 的演算法中，如果 Bus A (電容 10) 值為 1，Bus B (電容 5) 值為 0，當這兩個 Bus 都在**浮接狀態**時把傳輸閘打開，S1 會判斷「大電容吃掉小電容」，兩者都變成 1。
但在上面的 Bit-sliced 模型中，我們只實作了 `Hold`（各自維持前值）。如果我們不加電容判斷，A 會 Hold 1，B 會 Hold 0，這就跟 S1 產生了 Mismatch。

**這真的重要嗎？**
**對 NES 來說，幾乎 100% 不重要。** `io_db` (PPU 的 I/O Data Bus) 上的這些浮點電荷轉移，都發生在時脈相位的「瞬態」或「無效讀取區間」。當真正的 CPU 或 PPU 要讀取這條 Bus 時，一定會有一個主動的 Driver（Strong0 或 Strong1）強勢介入，把這個微弱的浮點電荷蓋掉。

**務實的工程建議：**
1.  **先實作上面的 `S0/S1/W1` 模型**：這解決了 99.9% 的剛性衝突（GND 打架 VCC）。
2.  **無視小電容 Mismatch**：跑 `--trace-cmp`，如果 Mismatch 發生的節點是 Bus，且該 cycle 結束時該 Bus 並未被任何 Logic 鎖存（Latch），這就是一個「不可觀察的 transient state」。
3.  **如何讓 Validator 閉嘴**：如果你確定那是 transient artifact，可以在 Validator 中加入一個白名單：「當該 Node 的 `S0==0 && S1==0 && W1==0` 時，允許模擬器結果與 S1 不一致」。
4.  如果你*非得*要跟 S1 完全一模一樣（例如為了強迫症的零誤差），你就必須在上述 Bit-sliced 邏輯的 `~S0 & ~S1 & ~W1` 分支中，加入比較 `Cap[v]` 與相鄰浮接 Node `Cap[other]` 的邏輯。但在 64-lane 的 bitwise 運算中做整數大小比較非常痛苦且拖慢效能，強烈建議直接把這種 Artifact 當成 S2 的 unnamed junction 一樣**豁免掉**。這才是適合 GPU 執行的「忠於邏輯正確性」之道。
