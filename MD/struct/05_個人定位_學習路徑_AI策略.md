## 1. 我做這個專案的成功率

### 想一次到位的話：成功率不高

「一次到位」指：

- 完整 2A03 / 2C02 抽取
- 整顆 NES 常駐 GPU
- 高信心驗證
- 還要能寫成 paper / 開源專案

這超出單一技能範圍，是一個橫跨多個領域的長期整合任務。

### 分層做的話：成功率其實不錯

換成這個目標：

- 先把工具鏈在本機跑起來
- 從 parser / graph / evaluator / 小區域驗證開始
- 漸進地往「值得信任的模型」靠近
- 把它當**長期技術專案**而非短期交付

成功率反而比直覺想像的高。

理由是這個專案最重要的能力**不是**「已經會 FPGA / IC 設計」，而是：

- 抽象能力
- 建模能力
- 把複雜系統拆成可驗證子問題的能力
- 對工具與基建的耐心
- 對「正確性」的偏執

這些能力其實**比較像成熟軟體工程師**，而不是傳統晶片工程師。

### 我目前缺什麼

- NMOS / switch-level 電路的直覺
- 動態邏輯、回授、charge storage 的硬體直覺
- 對網表、timing、bus 物理意義的熟悉度
- 「驗證邏輯等價」的方法論（不只是「程式跑得起來」）

這些缺口**可以學**。對 CS 背景＋碩士訓練＋實務軟體經驗的人來說，**不算不切實際**。

### 直白判斷

> 我大概**不是那種一上手就做對**的人，
> 但我**有可能**在一年的持續投入後越來越接近成功。

這個專案最終獎勵的不是「開始時懂最多 IC」的人，而是能：

- 持續縮小問題
- 建出驗證方法
- 不被複雜度淹沒
- 把每一層都變成具體中間結果

的人。

## 2. 是否有更適合的人想做這個

**有，但不多**。

更天然適合的人通常是：

- 模擬器或逆向經驗豐富者
- EDA / formal methods / 硬體 compilation 專業者
- 對 CUDA / HPC 很熟、又願意碰硬體抽象的人
- 喜歡工具鏈、analyzer、驗證系統的研究型工程師

但**同時滿足下列條件**的人很少：

- 對古典晶片有興趣
- 願意鑽進髒亂的 switch-level 細節
- 願意自己造大量基建
- 能容忍長時間慢回報

**所以更適合的人存在，但會做、會做完的人很少。** 這意味著我的優勢不只是背景，更可能是**長期投入、好奇心與堅持**。

## 3. 最該補的 5 個知識缺口

### 1) Switch-Level / NMOS 直覺

不必一開始就有完整 IC 設計能力，但要懂：

- transistor 當開關的行為
- pass transistor
- pull-up / depletion load
- 浮接節點
- 動態 storage

這直接決定圖模型與 evaluator 寫得對不對。

### 2) Timing 與兩相時鐘

把「程式步驟」翻譯成「phase 驅動的狀態更新」，特別是：

- half-cycle
- settling
- clock phase
- read / write bus timing

### 3) 把圖論套到電路上

可能已經會圖論演算法，但要熟悉它在電路裡的意義：

- 連通性
- SCC
- 回授分類
- 浮島偵測

這不是普通演算法題，是**圖論 + 電路語意**。

### 4) 邏輯抽取 / IR / Codegen 思維

這個專案非常像 compiler 專案。要培養：

- IR 設計直覺
- 表達式樹
- 化簡
- backend emit
- trace 與等價工具

### 5) 驗證方法論

最容易被低估的部分。要學：

- 設計小型可驗證測試
- 建立參考 trace
- 分層驗證 parser / graph / evaluator / backend
- 避免「看起來對但沒被證明」的系統

## 4. 6 個月學習與建構路徑

### Months 1–2：基本電路與資料理解

**目標：**

- 理解 Visual6502 / Visual2A03 資料格式
- 建立 switch-level / NMOS 直覺
- 寫最小 parser

**產出：**

- `segdefs.js` / `transdefs.js` parser
- 基本 `NetNode` / `NetTransistor` / `NetlistGraph`
- 數個手刻迷你網表：inverter / NAND / pass transistor

**這兩個月不要碰 CUDA。**

### Months 3–4：CPU 參考求值器

**目標：**

- 把 graph 變成可執行的 evaluator
- 支援通電 / hold / settle 行為
- 對小區域產出 trace

**產出：**

- CPU evaluator
- connected-to-gnd / connected-to-high 邏輯
- 浮島偵測
- 收斂式 settle
- trace dump

**這個階段優先正確性，不是速度。**

### Month 5：開始處理真實局部區域

**目標：**

- 不再只跑手刻網表
- 對真實 2A03 局部區域做分析與求值

**產出：**

- reset chain 或 register bit 區域的 trace
- 一個真實局部區域的 graph debug 輸出
- loop / SCC 分析結果

### Month 6：Codegen 與 CUDA 基礎

**目標：**

- 把 evaluator 結果整理成 IR
- 開始建構 CPU 與 CUDA 共享的抽象

**產出：**

- 表達式 IR
- 一個簡單 emitter
- CPU backend 與 IR 比對
- 第一份 bit-slicing 設計草稿

**到第六個月還是不必急著上完整 CUDA**，真正需要的是**穩定的中間層**。

## 5. 怎麼避免燒掉或半途卡死

最危險的三種失敗模式：

### 1) 太早追終局架構

例如：

- 整顆 NES 常駐 GPU
- 同時做 CPU / PPU / APU / mapper
- 過早追求大批次吞吐

→ 在核心模型還沒被證明對之前就被複雜度淹沒。

### 2) 沒有分層驗證

每個階段都沒留下「明確可驗證的結果」 → 大量工程但任何部分都不可信。

### 3) 沒有顯性中間勝利

如果只在「最終完工」才有成就感，這個專案幾乎一定半路停下。要設定里程碑：

- parser 跑通
- graph 建好
- 手刻網表測試通過
- 真實 trace 產出
- IR 存在

**每一小步都要算成果**。

### 防卡死建議

- 一次只解一個核心問題
- 每階段都留下可展示的東西
- 永遠先解小型可驗證問題、再解大的
- 不要同時碰太多未知
- 把它當**長期研究型 side project**，不要當短期產品衝刺

## 6. 怎麼用 AI 工具達成目標

AI 對這個專案幫助很大，**但要把它當加速器、不是替代品**。

### 1) 把 AI 當「跨領域概念翻譯機」

我不是缺能力，而是**跨領域**。AI 很擅長把以下概念，翻譯成軟體工程語言：

- NMOS
- switch-level 邏輯
- 回授
- 浮接節點
- bus timing

加速 ramp-up。

### 2) 把 AI 當「結構設計助手」

請 AI 草擬：

- parser API
- graph 模型
- evaluator class 邊界
- IR 結構
- trace 格式

AI 在「快速產出幾個可行設計形狀」這件事上很強。

### 3) 把 AI 當「文件整理工具」

這個專案有很多素材、長 context、重疊概念。AI 適合：

- 讀書筆記整理
- roadmap 重組
- 把散亂概念整成章節
- 雙語對照

可以**大幅降低 context-switching 成本**。

### 4) 把 AI 當「測試案例產生器」

請 AI 生：

- 最小 inverter 網表
- NAND / NOR / pass transistor 測試
- dynamic latch 場景
- 邊界 case 的期望 trace

特別在早期驗證階段很有用。

### 5) 把 AI 當「程式碼骨架產生器」

例如：

- C# class skeleton
- parser placeholder
- evaluator interface
- IR record types
- CLI / logging / trace dump 基建

「AI 先草稿、人來收尾」對這類 boilerplate 是最佳搭配。

## 7. 不要讓 AI 做的事

### 1) 不要讓 AI 替你判斷正確性

AI 很擅長「聽起來對」。這個專案特別容易栽在「sounds right」上。任何牽涉到：

- 電路語意
- timing 行為
- memory node 分類
- 正確性

都必須**用自己的 trace 與驗證流程證明**，不能只靠 AI 解釋。

### 2) 不要讓 AI 一次生整套系統

請 AI 一次寫完 parser + evaluator + CUDA backend，結果通常是：

- 結構看起來完整
- 實際上沒被驗證過

正確做法：把工作切成**很小、很明確**的子問題。

### 3) 不要把 AI 答案當文獻或標準

AI 可以幫助理解，但不能取代：

- 真實原始資料
- 真實 trace
- 參考模型
- 實驗證據

## 8. 最推薦的 AI 協作模式

```text
1. 先定義「我現在解的這一個」子問題
2. 請 AI 給 2~3 個建模或實作選項
3. 選最保守、最可驗證的那個
4. 請 AI 生鷹架、pseudocode、測試想法
5. 「我自己」做 trace 與正確性檢查
6. 再請 AI 幫總結成果與提下一步
```

這樣最安全，因為：

- 決策權在我
- 加速來自 AI
- 正確性來自驗證

## 9. 結語

我的背景**不是最對盤的**，但這不代表我成功率低。對這個題目來說，更重要的問題不是「一開始懂多少硬體」，而是我能不能：

- 長期維持興趣
- 用軟體工程師的方式拆問題
- 建出可信的驗證
- 抗拒「太早追終局」的誘惑

如果策略對，我有機會做出**令人尊敬的成果**。

最佳路徑不是一次大躍進，而是：

> 先取得**值得信任的局部成功**，再一層層擴張。
> 用 AI 加速理解、鷹架、知識整理、測試生成，
> 但最終正確性的判斷永遠留在自己手上。

**一句話總結：**

> 我**不是**最背景對盤的人，但**很可能**是能做成的那種人 —
> 前提是要用軟體工程師的方式打它，並把 AI 當加速器、不是替代品。
