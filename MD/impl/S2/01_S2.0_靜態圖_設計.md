## ① 設計描述

### 1.1 目標 & 範疇

- 輸入：`WireCore` 在 `ComposeSystem`（含 `EnableLowering` → `LowerNetlist()`）之後的 build-time 表 —— `_nodes: List<Node?>`（dense、`1==Npwr`、`2==Ngnd`、`0` 保留）、`_transistors: List<Transistor>`（每筆 `{Gate, C1, C2, IsWeak, Name}`，`AddTransistor` 已把 supply 正規化到 `C2`）、`_nodeByName` / `_nameByNode`。
- 輸出：一個 `AprVisual.Sim.Logic.NetlistGraph`（靜態 class，跟 `WireCore` 同 assembly，讀 `WireCore.Nodes` / `WireCore.Transistors`），裡面：
  - `NodeRole[] role`（indexed by node id）
  - `TransistorKind[] kind`（indexed by transistor index）+ `bool[] isWeak`（直接抄 `Transistor.IsWeak`，方便）
  - `PassDir[] passDir`（indexed by transistor index；只有 `kind==Pass` 的有意義，初值 `Unknown`，S2.1 填）
  - 一個 `Build()` 把上面填好，加幾個統計查詢（`CountByRole()`、`CountByKind()`、`Describe(nodeId)`）
- **不做**：pull-down 表達式、pass 方向推導、任何 `Expr`、任何模擬行為改動。那是 S2.1+。

### 1.2 `NodeRole`

```
enum NodeRole { Supply, Input, Bus, Internal }
```
- **`Supply`** = `{ Npwr(1), Ngnd(2) }`。直接設。
- **`Input`** = 「從來不被網表本身驅動，只被外部碼（handler / `ResetNes`）`SetHigh`/`SetLow`」的 node。判定 heuristic：`role != Supply` 且 `node.C1c2s.Count == 0`（從來不是任何電晶體的 channel 端）且 `node.Pullups == 0`。→ 抓到 `clk`（clock handler 每 half-cycle toggle 它；它只當一堆電晶體的 gate，自己沒 channel）、可能還有幾個 config strap。
  - 註（設計時的假設）：原以為 `res` 不算 `Input`（有 pull-up + CIC 驅動）。**⚠ 實作後發現不對**：在這份網表裡 `res`（S1.5 把 `res`/`cpu.res`/`ppu.res` 等 always-on connection 併成的 super-node）**沒有任何 channel-端電晶體、也沒有 `Pullups` 標記** → heuristic 把它判成 `Input`。沒關係 —— 標 `Input` 表示「IR 引擎不去算它，維持 handler/`ResetNes`/S1-settle 給它的值」，正是我們要的（`ResetNes` 裡 `SetHigh/SetLow(res)` 就是外部 override）。詳見 ④。
- **`Bus`** = 雙向（既被網表驅動、又被 handler 驅動）—— S2.0 先做**最小但確定對**的版本：把 `cpu.db[7:0]`（CPU 資料匯流排）跟 `ppu.io_db[7:0]`（PPU 對 CPU 的資料線）resolve 出來標 `Bus`（這兩條確定是雙向的：CPU 寫的時候 CPU 驅動、讀的時候 RAM/ROM/PPU handler 驅動）。其餘雙向 node（RAM/PPU 內部的資料線、`_d[7:0]` 那些 handler 寫的腳）**先不標** —— 等 S2.1 有了 `DriveInfo`（知道哪些 node「有 driver」）+ 從 handler 取得「被 `WriteBits` 的 node 集合」，再精準地把 `(有 driver) ∩ (被 handler 寫)` 標成 `Bus`。S2.0 階段 `Bus` 是「定義好的類別、先粗填」。
- **`Internal`** = 其餘全部。

### 1.3 `TransistorKind`

```
enum TransistorKind { PullDown, PullUpStrong, PullUpLoad, PullUpActive, Pass, Dead }
```
對每筆 `Transistor t`（已正規化：若有 supply 在 channel 端，它在 `C2`）：
1. `t.Gate == Ngnd`（gate 接 GND → 永遠 OFF）→ **`Dead`**。（S1.5 lowering 已丟掉這種，理論上不會出現；保留以防萬一。）
2. channel 端碰 GND（`t.C1 == Ngnd || t.C2 == Ngnd`）→ **`PullDown`**。（這是「直接下拉到地」的葉子；經由其他信號 node 串聯才到 GND 的，那些電晶體會是 `Pass`，串並聯結構由 S2.1 算。`gate == Npwr` 的「無條件接地」也歸 `PullDown`，S2.1 看到 `gate` 是 `Npwr` 就知道條件恆真。）
3. channel 端碰 VCC（`t.C1 == Npwr || t.C2 == Npwr`，設 `sig` = 非 VCC 的那一端）：
   - `t.Gate == Npwr`（無條件接 VCC）→ **`PullUpStrong`**。
   - `t.Gate == sig`（gate 接自己 source / 被拉的 node —— 經典 NMOS depletion load 的接法）→ **`PullUpLoad`**。
   - 否則（gate 是別的信號 node → conditional pull-up，superbuffer / push-pull 的上半 / precharge 的 clocked pull-up）→ **`PullUpActive`**。
4. 兩個 channel 端都是信號 node（非 supply）→ **`Pass`**（`passDir = Unknown`，S2.1 填）。
- `isWeak[i] = t.IsWeak`（depletion / weak 旗標）—— S2.1 分 `PullUpLoad` 是強是弱、pass 方向推導時會用。

> 為什麼這樣分（呼應 Gemini review §7-3、§7-4）：S2.1 抽 `Expr` 時要分得出「static load（恆拉高的負載）」「strong VCC tie」「conditional / clocked pull-up（precharge）」「active pull-up（superbuffer，要做 `Mux(pullup, 1, Mux(pulldown, 0, Hold))`）」。S2.0 把這個一級分類做好，S2.1 才有 base。

### 1.4 `PassDir`

```
enum PassDir { Unknown, AtoB, BtoA, Bidirectional }
```
- `Pass` 電晶體上一個欄位，S2.0 一律 `Unknown`。
- S2.1 的方向性推導（Gemini §7-2）會填：典型「`C1` 側有 PULLDOWN 網路、`C2` 側純 dynamic（只有 pass 連進來）→ `AtoB`」（A=有 driver 那端）。推不出單一方向（如 2C02 OAM 讀寫匯流排）→ 留 `Bidirectional`，那個 pass 網路 → hybrid（S2.1 標）。
- S2.0 只是把欄位開出來；不推導。

### 1.5 IR 跟 S1 共用 `NodeStates`（X / 初始化的前置決策）

呼應 `00_計畫總覽.md` §1.4 / Gemini §7-6：S2.0 這步順便**確認**（不需要改 code，只是文件化決策）：IR 引擎（S2.4）會跟 S1 共用那個 `byte* NodeStates`（0/1，不追 X）；power-on 的「未定」由「`ResetNes(full)` 先跑 S1 的 `Reset()` + `RecomputeAllNodes()` settle 到決定性穩態，IR 引擎再接手」這條路解掉。所以 S2.0 不需要為 X 做任何資料結構。

### 1.6 程式落點（依 Gemini §7-4 採用：可實例化、非 static、工廠方法）

- 新檔 `src/AprVisual/Sim/Logic/NetlistGraph.cs`：
  ```csharp
  namespace AprVisual.Sim.Logic
  {
      internal enum NodeRole { Supply, Input, Bus, Internal }
      internal enum TransistorKind { PullDown, PullUpStrong, PullUpLoad, PullUpActive, Pass, Dead }
      internal enum PassDir { Unknown, AtoB, BtoA, Bidirectional }

      // S2 的分析資料結構（跟 S1 的執行期引擎 WireCore 分開 —— 不同生命週期、不同關注點）。
      // 可實例化（非 static）：方便之後 S2.1+ 用小網表跑 unit test 驗證邏輯抽取。
      internal sealed class NetlistGraph
      {
          public NodeRole[] Role = [];           // [nodeId]
          public TransistorKind[] Kind = [];     // [transistorIndex]
          public bool[] TransIsWeak = [];        // [transistorIndex]
          public PassDir[] PassDirection = [];   // [transistorIndex]; 只有 Kind==Pass 的有意義；初值 Unknown，S2.1 填

          /// <summary>從目前 WireCore 已組裝（含 S1.5 lowering）的網表建一個 NetlistGraph。</summary>
          public static NetlistGraph BuildFrom();   // 讀 WireCore.Nodes / WireCore.Transistors / GetNodeName / ResolveNodes

          public string Describe(int nodeId);      // "name#id  role=…  pulldownTrans=…  pullup=…  passTrans=…"
  ```

### 1.7 驗證計畫（含 Gemini §7-5 採用的「窮舉 + sanity 斷言」）

`SelfCheck()` 跑這些斷言（任一違反 → 回傳違規描述，否則 `"OK"`）：
- **窮舉（exhaustive）**：`sum(CountByKind()) == 非空 transistor 數`；`supply + input + bus + internal_ == 非空 node 數`。
- **sanity cross-check**：每顆 `PullDown` 都 `C1==Ngnd || C2==Ngnd`；每顆 `PullUpLoad` 都 `(Gate==C1 || Gate==C2)` 且另一端 `==Npwr`；每顆 `PullUpStrong` 都 `Gate==Npwr` 且 `C1==Npwr || C2==Npwr`；每顆 `Pass` 兩端都非 supply；`CountByKind()[Dead] == 0`（驗 S1.5 lowering 的成果）。
- `Supply` 恰為 `{1, 2}`。

驗收：
1. `dotnet build` 0 warning / 0 error（Debug + Release）。
2. `--dump-graph --system-def-dir ref/metalnes-main/data/system-def`：
   - `SelfCheck()` = `OK`。
   - role 統計：`Supply` = 2；`Input` 很少（個位數，含 `clk`）；`Bus` ≈ 8~16（`cpu.db[7:0]` + `ppu.io_db[7:0]`，視 alias）；`Internal` = 其餘（~14700）。
   - kind 統計：`Pass` 佔不少；`PullDown` 很多；`PullUpLoad` ≈ 跟「有 pull-up 的 node 數」同量級（~6400）；`PullUpStrong`/`PullUpActive` 少；`Dead` = 0。
   - spot-check：`vcc`#1 / `vss`#2 = `Supply`；`clk` = `Input`；`res` = `Internal`；`cpu.db0` = `Bus`。
3. `--selftest` 全 PASS（含新 `TestNetlistGraph`）；`--trace branch_timing/1.Branch_Basics.nes --cycles 24` 輸出跟 commit `529f927`（S1.5）的一致（S2.0 不該改任何模擬輸出）。

## ② Gemini 分析回饋

> `gemini-3.1-pro-preview-customtools`，log：`tools/knowledgebase/message/20260512_102000.txt`（RESPONSE 全文在那；Windows console 印出來會 mojibake）。摘要：

- **整體**：設計很扎實，「先建表、不推導邏輯、不碰模擬」的切入點精準。
- **Q1（分類 / NMOS 結構）**：1-transistor 分類法**正確，不需要新類別**。**不要**在 S2.0 試圖辨識 superbuffer / bootstrap —— 那是陷阱。理由：depletion load 的 `Gate==source`（`PullUpLoad`）vs `Gate==Npwr`（`PullUpStrong`）已經是最關鍵的基底；superbuffer = 「一顆 PullDown + 一顆 PullUpActive」，S2.1 看到一個 node 同時有這兩個 driver 就知道；bootstrap 在 Visual6502 網表裡沒有顯式元件（靠 dynamic node cap + pass transistor 模擬），驅動端看起來就是 `PullUpActive` 或 `Pass`，S2.0 如實標即可、overdrive/timing 交給 S2.1/S2.4 的 Hold 機制；ratioed inverter / tri-state / 下拉鏈都要看多顆電晶體的拓撲 → S2.1 串並聯分析。
- **Q2（Input heuristic / res）**：判斷標準正確，`res` 標 `Internal` 是清醒的決定（它網表內有合法 driver：板上 pull-up + CIC；C# 的 `SetHigh/Low` 是強制覆寫，不是它天生只是輸入腳；保持 `Internal` 能確保 CIC / pull-up 邏輯不被誤刪）。手把輸入腳（`$4016` D0）有 pull-up → 被 heuristic 踢出 `Input` 變 `Internal` → **沒關係**，只要 IR 引擎允許 handler override `Internal` node（像 S1 的蓋值傳播）。heuristic 夠了，維持現狀。
- **Q3（Bus 分階）**：「先粗填 `cpu.db` / `ppu.io_db`、S2.1 精修」**強烈建議採用**。`Bus` 標籤在 S2.0 只是存進陣列、沒邏輯依賴它；直到 S2.1/S2.2 處理雙向 pass 方向 + 驅動強度才需要精確集合。S2.0 寫死那 16 條線（+ alias 解析）即可；S2.1 再寫 `BusResolver.RefineBuses()` 算 `(有內部 driver) ∩ (有外部 handler write)` 來覆寫。
- **Q4（程式落點）**：**絕對用獨立 class，千萬不要放進 `WireCore` 的 partial**（`WireCore` 是 S1 執行期引擎核心；`NetlistGraph` 是 S2 編譯器/分析器資料結構，生命週期、關注點完全不同）。**微調：改成可實例化（非 static）+ 工廠方法 `NetlistGraph.BuildFrom(...)`** —— 避免全域狀態污染；之後想寫 unit test 跑兩塊小網表驗 S2.1 邏輯抽取時，非 static 好用一百倍。
- **Q5（驗證）**：除了 spot-check，**必須加「互斥 + 窮舉」斷言**：`sum(各 kind) == 總 transistor 數`、`sum(各 role) == 總 node 數`；每顆 `PullDown` 都觸 GND、每顆 `PullUpLoad` 都 `Gate==source` 且另一端是 `Npwr`、每顆 `PullUpStrong` 都 `Gate==Npwr` 且觸 VCC、`Dead == 0`。「10 行 LINQ，100% 保證分類沒漏網之魚」。
