# S2.4 — IR 直譯器（`IrEngine`）+ S2.5 hybrid bridge（合併到這步做）+ S2.6 等價 gate

>
> 工作流程（`00_計畫總覽.md` §3）：① 設計描述 → ② 丟 Gemini → ③ 評估採用 → ④ 實作 → ⑤ 驗證。可分多 firing（①寫設計+丟Gemini ②topo+SCC+evaluator ③Step 迴圈+bridge ④等價 harness+驗證+commit）。

## ① 設計描述

### 1.1 IR build（一次性，`LoadSystem` 後 / `IrEngine.Build()`）

從目前的 `WireCore` 網表（已 compose + lowering）：`NetlistGraph.BuildFrom()` → `DriveAnalysis.Analyze()` → `NextStateModel.Build()` → `SccModel.Build()` → 得 `nextExpr[]`、`isSequential[]`、`hybrid[]`。然後：

1. **建「現值依賴圖」**：`nextExpr[v]` 裡每個 `NodeRef(M)` → 一條 `v → M` 的依賴（`Hold(M)`/`Prev(M)` **不算**邊 —— 那是上一拍）。hybrid node（`nextExpr == null`）不在圖裡（它走 switch-level；但別人引用它的 *現值* 是 OK 的 —— bridge 會把它的現值放在 `NodeStates`）。
2. **Tarjan 找殘餘 SCC**（size>1，或 size-1 但 `nextExpr[v]` 引用 `NodeRef(v)`）—— 這些是 S2.3 沒打斷的（複雜回授 / cross-phase pipeline）。第一刀：**把殘餘 SCC 的所有 node 標 hybrid**（`nextExpr[v] = null`、`hybrid[v] = true`），丟回 switch-level。（之後可選做 S2.3 的 Stage D —— 用 `clk0`/`clk1` 互斥認出「假環」、用 `Prev` 打斷 —— 來救回一些；先看殘餘 SCC 有多少。）
3. **拓樸排序剩下的 non-hybrid node**（依現值依賴圖；hybrid 不在裡面）→ `evalOrder[]`（一個 node id 的陣列，按「先算被依賴的、後算依賴別人的」）。
4. **`compactExpr`**：把每個 non-hybrid node 的 `nextExpr` 攤平成「一串 stack-machine 指令」或一棵小樹的指標形式 —— 第一刀直接用 `Expr` 樹 + 遞迴 eval（簡單；S3 才優化成攤平）。
5. **hybrid node 清單** + 「IR node 中引用了 hybrid node 現值的」清單（bridge 用）。

### 1.2 evaluate 一個 node（`EvalExpr(Expr e)`）

遞迴：`Const(b)` → `b ? 1 : 0`；`NodeRef(m)` → `NodeStates[m]`（現值）；`Hold(m)` / `Prev(m)` → `prevStates[m]`（上一拍值）；`Not(x)` → `1 - eval(x)`；`And(l,r)` → `eval(l) & eval(r)`；`Or(l,r)` → `eval(l) | eval(r)`；`Mux(c,a,b)` → `eval(c) != 0 ? eval(a) : eval(b)`（short-circuit OK）；`Complex` → 不該出現（hybrid 的 `nextExpr` 是 null 不是 Complex）。

### 1.3 `Step(1)` 一次（mirror S1 的 `Step` 但用 IR）

```
RunHandlerChain();                              // clock toggle / RAM-ROM callbacks / video — 跟 S1 一樣，會直接寫 NodeStates（外部驅動 + override）
Array.Copy(NodeStates, prevStates, NodeCount);  // 快照「這一拍開始時」的狀態 → Hold/Prev 讀這個
foreach v in evalOrder: NodeStates[v] = EvalExpr(nextExpr[v]);   // 一遍：按拓樸序算 non-hybrid node（讀 NodeStates 的現值 + prevStates 的上一拍值）
// ── bridge：hybrid node 還沒算；用 S1 的 switch-level recalc 補 ──
//   enqueue：(a) handler 改過的 node（handler 內部已 SetHigh/Low → 已 enqueue 了？要確認 handler 路徑），(b) 上面 IR-eval 改了值的 node（讓 ProcessQueue 從它們傳播到相連的 hybrid node），(c) hybrid node 本身（讓它們被重算）。
//   ProcessQueue();   // S1 的 recalc：會 re-settle hybrid node（用 group-BFS）；對 non-hybrid node 它會 re-derive，理論上跟 IR 算的一樣 → 不變 → 不再傳播 → 收斂快。
if (TraceLevel != 0) CaptureTraceLine();
Time++;
```
細節要小心：(1) handler（`SetHigh`/`SetLow`）會走 S1 的 `SetNodeState` → 已經 enqueue 了相鄰 node。所以 handler 改完後 recalc queue 裡已經有東西。(2) IR-eval 直接寫 `NodeStates[v]` —— 沒走 `SetNodeState` → 沒 enqueue。所以 IR-eval 完要把「值有變的 IR node」手動 enqueue（`EnqueueNode(v)`）。(3) 然後 `ProcessQueue()` 跑 S1 的 recalc —— 它會處理 queue 裡的（handler 改的 + IR 改的 + 連帶 enqueue 的 hybrid node），用 group-BFS re-settle。對 non-hybrid node：ProcessQueue 會用 `ComputeNodeGroup` re-derive 它的值 —— 如果 IR 算對了，結果跟現值一樣 → `SetNodeState` 不觸發變化 → 不傳播。對 hybrid node：ProcessQueue 第一次算它（從 raw 0/pull-up 狀態？不對 —— `prevStates` / `NodeStates` 已有它上一拍的值；ProcessQueue 的 `getNodeValue` 對「無 flag 的浮島」會回 `_maxState`（最大 cap 的 node 的現值）= 上一拍值 —— 所以 hybrid node 的 hold 行為由 S1 的浮島機制自然處理）。→ 整個收斂後 `NodeStates` 就是這一拍的正確結果。
- **第一刀的「ProcessQueue 也 re-derive non-hybrid node」是浪費**（IR 已經算對了）—— 但**正確**，而且 ProcessQueue 對「值沒變」的 node 不傳播 → 不會無限。S3 才優化（讓 ProcessQueue 只碰 hybrid 區）。S2.4 要的是「跑得對」。
- **更大的擔心**：IR-eval 的「一遍拓樸序」夠不夠？non-hybrid node 之間 —— topo 排好了 → 一遍夠。但 non-hybrid node 引用 hybrid node 的*現值* —— hybrid node 是 ProcessQueue 之後才算的，IR-eval 時 hybrid node 還是上一拍值。所以 IR node 用了「上一拍的 hybrid 值」算自己 → 然後 ProcessQueue 算 hybrid → hybrid 變了 → ProcessQueue 傳播到相連的 IR node → 那個 IR node 被 ProcessQueue re-derive（用 group-BFS，不是 IR）→ 變對。Hmm —— 所以「IR node 緊鄰 hybrid node」的會被 ProcessQueue 用 switch-level 重算。OK，正確（只是那些 node 沒享受到 IR 加速）。或者：IR-eval → ProcessQueue → 再 IR-eval（吃 ProcessQueue 算好的 hybrid 現值）→ 再 ProcessQueue → 迭代到收斂。第一刀先用「IR-eval 一遍 → ProcessQueue 一遍」+ cap-and-warn（若 ProcessQueue 改了 > N 個 IR node，警告 —— 表示 IR/hybrid 邊界耦合太緊，要看）。

### 1. Hybrid Bridge：IR-eval 一遍 → ProcessQueue 一遍（無迭代）
**設計評估：完全 OK，第一刀就該這樣做。**
*   **會不會收斂？** 會。因為 `ProcessQueue` 跑的是 S1 的 Switch-level group-BFS，S1 本身就會收斂。IR 只是幫 94% 的 Node 提供了「預先算好的精確值」。
*   **IR 緊鄰 Hybrid 的 Node 被 ProcessQueue 重算，可接受嗎？** **完全可接受**。這在功能上叫做「IR 預測，Switch-level 校正」。雖然這些邊界 Node 沒享受到 IR 加速，但確保了絕對的物理正確性。
*   **該不該 IR↔ProcessQueue 迭代？** **絕對不要**。這會讓系統變得極度複雜且難以 debug（可能引發無窮震盪）。S2 的目標是「驗證 IR 邏輯的正確性」，不是壓榨極致效能。
*   **對計畫的具體影響 / 該怎麼改**：
    維持 `IR 拓樸一遍` → `Enqueue 變動的 IR Node` → `ProcessQueue() 跑到底`。
    **額外加一行 Log**：統計 `ProcessQueue` 改變了多少個 *Non-hybrid (IR)* node 的值。如果數量很少（< 50），代表 IR 的邊界預測很準；如果數量極大，記下來留到 S3 再看，S2 不去修效能。

### 2. S1 Step 中的 Recalc 時機（關鍵行為確認）
**設計評估：這會是你實作時最大的地雷，必須先看 Code。**
*   **分析**：在 Visual6502 / MetalNES 架構中，外部 Handler（例如 Clock toggle 或外部 RAM 餵資料）通常是呼叫 `SetHigh(clk)` 或 `SetNodeState()`。**通常這些函式內部就會直接觸發 `ProcessQueue` 或 `Recalc()`**。
*   如果 S1 的 Handler 已經觸發了 Recalc，那當你執行 `RunHandlerChain()` 後，**整個晶片其實已經由 S1 算完一遍並收斂了**。如果此時你再跑 IR Eval，你只是在「重新證明一次 S1 算出來的值是對的」，這對驗證沒差，但這表示 IR 根本沒在「驅動」網表。
*   **對計畫的具體影響 / 該怎麼改**：
    去查 S1 的 `SetNodeState` 或 `Handler`。
    *   **情況 A（最好）**：Handler 只負責 `Enqueue`，外部 `Step()` 才顯式呼叫 `ProcessQueue`。那你的 `IrEngine.Step` 就照你的計畫：`Handlers` → `快照 prevStates` → `IR` → `Enqueue` → `ProcessQueue`。
    *   **情況 B（常見）**：Handler 寫值會立刻 settle 網表。如果不能改 S1，那你就在 `IrEngine.Step` 裡照跑 Handler（讓 S1 settle 局部），然後**強行快照 `prevStates`**，接著跑 IR，如果 IR 算出來跟 S1 有差異，覆寫它，最後再 call 一次 `ProcessQueue` 確保 Hybrid 區段同步。S2 先求對，S3 再來砍 S1 的多餘 Recalc。

### 3. Hold(M) 與 Prev(M) 的語意（讀 prevStates 快照）
**設計評估：OK 不用改，你的理解是正確的。**
*   **分析**：
    *   `Prev(M)` 用來打斷假環，讀上一個 Half-cycle 的值，這毫無懸念。
    *   `Hold(M)` 是 Pass transistor 關閉時的電容保值。你的 IR 是 `Mux(we, data, Hold(self))`。在一個拓樸排序、零延遲的 Boolean 網路中，如果 `we` (Write Enable) 是 1，它選 `data`（當前拍的最新值）；如果 `we` 是 0，它必須維持「這一拍開始前」的值。
    *   如果在同一個 Half-cycle 內讀取 `Hold(M)`，它**絕對不該**讀到同一個 Half-cycle 內其他分支對 M 的臨時修改（因為拓樸排序不保證同層的先後順序）。
*   **對計畫的具體影響 / 該怎麼改**：
    維持現狀。`Hold(M)` 和 `Prev(M)` 在 Eval 時統一讀取 `prevStates[M]`（Half-cycle 初的快照）。這對於 6502 的 Dynamic latch 行為是嚴格正確的。

### 4. 等價 Harness：Snapshot-compare vs Lockstep
**設計評估：Snapshot-compare 是最優解，OK 不用改。**
*   **分析**：Lockstep（跑兩份 NodeStates 陣列）會牽扯到 Handler 改寫硬體狀態的巨大重構。你不想改 S1 既有行為，所以 Snapshot-compare 是唯一解。
*   2000 個 Half-cycle 聽起來很少（約 300 個 CPU 指令），但對於 Switch-level to Boolean 轉換的 Bug 來說，**只要有錯，通常在 Reset 序列的前 50 拍就會全面炸裂**（例如某個 Bus 衝突沒解好、某個 Latch 寫錯邊）。
*   **對計畫的具體影響 / 該怎麼改**：
    *   **N-capped Snapshot (2000拍)**：專門用來抓 IR 邏輯轉譯錯誤。發生 Mismatch 時，印出「哪一拍、哪幾個 Node、該 Node 的 nextExpr 是什麼」，這對 Debug 是救命神器。
    *   **長跑測試**：用 `--test --engine ir` 跑 blargg test ROM。只要 CPU state（A, X, Y, PC, 記憶體簽章）是對的，就代表整個 NES 邏輯完美等價。不需要在長跑時比對每一個內部 Node。

### 5. 殘餘 SCC 與 Tarjan（S2.3 D 階段的取捨）
**設計評估：先全部標為 Hybrid，S2.4 不做 Stage D。**
*   **分析**：如果你現在卡下去做 clk0/clk1 的互斥分析（Stage D），你會迷失在圖論演算法裡，而延誤了最核心的 IR Engine 驗證。
*   **對計畫的具體影響 / 該怎麼改**：
    S2.4 中，Tarjan 掃出 Size > 1 或自我引用的 SCC，**無腦把它們全部標記為 Hybrid**（`nextExpr = null`）。
    這可能會讓你的 Hybrid Node 數量從 847 暴增到 2000，**這完全沒關係**。只要 `ProcessQueue` Bridge 能正常工作，跑起來功能正確，你的 S2 就過關了。把「消滅殘餘 SCC」留到確保這套系統能動之後（S2.7 或 S3）再做。

### 6. 合併 S2.4, S2.5, S2.6 成一步
**設計評估：必須合併，分開無法驗證。**
*   **分析**：IR Engine (S2.4) 算出來的值必須透過 Bridge (S2.5) 餵給 Hybrid 網路，否則 Hybrid 網路處於浮接狀態，下一拍系統直接崩潰。沒有等價驗證 (S2.6)，你根本不知道 S2.4 寫對了沒。
*   **對計畫的具體影響 / 該怎麼改**：
    當作同一個 PR/Commit 來做。開發順序建議：
    1. 實作 `IrEngine.Build()` (Tarjan + Topo Sort + prevStates 配置)。
    2. 實作 Snapshot-compare 的 Test Runner（先拿 S1 自己比自己確保 Harness 沒寫錯）。
    3. 實作 `IrEngine.EvalExpr()` 與 `Step()`。
    4. 掛上去跑，看 2000 拍能過幾拍，開始無窮盡的 Debug。

### 總結 Actions：
1. **去查 S1 的 `SetNodeState`/Handler 到底在哪裡觸發 Recalc**，這決定了你 `IrEngine.Step` 的順序。
2. **照原計畫使用 Snapshot-compare**，不搞 Lockstep，不搞 IR/ProcessQueue 迭代。
3. **殘餘 SCC 直接判死刑（標 Hybrid）**，不要在此刻優化它。
4. **`Hold/Prev` 統一吃 `prevStates` 快照**。
