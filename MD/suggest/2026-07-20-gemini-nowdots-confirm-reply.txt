這是一個極度經典的模擬器開發情境：**「宿主（Host）內部狀態與客體（Guest）硬體狀態的定義錯位」**。

我非常仔細地審視了你的重新診斷與修法。先說結論：**你的重新歸因 100% 正確，且你的修法方向非常精準、安全。**

以下為針對你 4 個問題的嚴謹回覆與細節確認：

### 1. 我的重新歸因對嗎？
**對，完全正確。Bug 純粹在你的 `NowDots` 函數公式，與模擬器本身的精度無關。**

你的公式 `frame_count * 89342 + scanline * 341 + ppu_cycles_x` 在數學上**嚴格要求** `frame_count` 必須且只能在 `scanline` 發生 `261 -> 0` 的那一瞬間進位（++）。
既然 `frame_count` 在 SL240 就進位了，當 PPU 執行到 SL240 的第 0 個 dot 時，你的 `NowDots` 計算結果會瞬間暴增 `1 * 89342`（跳躍了一整幀的時長），這導致了嚴重的**非單調性（Non-monotonic）**。當你的衰減邏輯計算 `NowDots() - stamp` 時，這個巨大跳躍會讓系統誤以為時間已經過了很久，進而提早觸發衰減。

### 2. `frame_count++` 在 SL240 是不是對遊戲/精度完全無影響？
**對，毫無影響。模擬器作者不需要、也不應該為了這個問題去改 `frame_count` 的時序。**

在真正的 NES 硬體中，**不存在**一個叫做「Frame Counter」的暫存器可以讓遊戲 CPU 讀取。遊戲感知「一幀結束」的唯一方式，是透過 PPU 在 SL241 觸發 NMI 或是 `$2002` 的 VBLANK flag 升起。
模擬器內部的 `frame_count` 變數純粹是「宿主端（Host-side）」的計數器，通常用來控制前端渲染（例如累積到 240 條可見線就丟給 GPU 畫）、計算 FPS 或是處理音訊同步。既然遊戲讀不到它，它在 SL240 還是 SL261 ++，對 AccuracyCoin 測試和遊戲運作完全透明。要求作者改動這個運作良好的核心時序，反而可能打破他現有的渲染或同步邏輯。

### 3. 改用 `mcCycleCount * 3` 是安全的嗎？需要重驗什麼？
**這做法非常安全。**

因為 `mcCycleCount` 是一個唯讀（Read-only）的依賴。你在判定 open-bus 衰減時去「讀取」這個變數，並**不會反向修改**任何 CPU/PPU 的狀態、週期長度或中斷時序。
- **對遊戲可見時序的影響**：**0**。
- **對 AccuracyCoin / blargg 的影響**：**0**（因為核心時序根本沒變）。
- **需要重驗的部分**：你**只需要**重新驗證「open-bus 衰減」相關的測試 ROM（如 `ppu_open_bus` 系列），確認電容衰減的時長（75~91°C 週期）是否回到正確的範圍，且不再出現提前衰減的 Bug。

### 4. 有沒有漏掉的細節？（嚴謹把關）
你的思路很清晰，但作為資深工程師，我必須提醒你幾個實作上可能遇到的邊角細節（Edge Cases），請在實作時確認：

*   **細節 A：確認 `mcCycleCount` 究竟是 CPU Cycle 還是 Master Clock？**
    你提到 `mcCycleCount` 是「每 CPU cycle...++」，但它的命名 `mc` 通常代表 Master Clock。
    *   **如果是 CPU Cycle**：NTSC 下 `CPU * 3 = PPU Dots`，你的公式 `mcCycleCount * 3` 是**正確**的。
    *   **如果是 Master Clock**：NTSC 下，PPU divider 是 4，CPU divider 是 12。所以公式應該是 `mcCycleCount / 4` 才會等於 PPU Dots。請務必確認該變數的實際定義。

*   **細節 B：DMA Cycle Steal 與暫停（Halt）狀態**
    當發生 OAM DMA 或 DMC DMA 時，CPU 會暫停執行（Steal cycles）。
    *   請確認該模擬器在處理 DMA 時，`mcCycleCount` 是否**依然會繼續遞增**？
    *   一個高精度的模擬器在 DMA 期間，全局時鐘/CPU 週期計數器依然會前進（OAM DMA 會消耗約 513/514 CPU cycles）。如果 `mcCycleCount` 在 DMA 期間凍結，你的時間計算就會稍微變慢。不過這對於長達數十毫秒的「物理電容衰減」來說誤差極小，通常可以忽略，但從架構正確性來看值得確認。

*   **細節 C：PPU 的 Missing Dot（少一個 Dot 的機制）**
    NTSC NES 在開啟渲染（BG/OBJ enable）時，奇數幀（Odd frame）的 pre-render scanline 會跳過最後一個 dot（Dot 340）。
    *   這意味著，如果你用 CPU cycle 乘 3 來推算 PPU Dot，隨著時間推移，你的 `NowDots` 會跟 PPU 內部的真實 `ppu_cycles_x` 產生**每兩幀 1 dot 的微小漂移（Drift）**。
    *   **這對你有影響嗎？沒有。** open-bus 的衰減是模擬物理 RC 電路的放電時間（動輒上萬 dots 的量級），這種 1 dot 的漂移對衰減閾值來說完全是九牛一毛。你的需求是「單調遞增的相對流逝時間」，而不是「絕對像素對齊」，所以這點可以放心。

*   **細節 D：非 NTSC 區域（PAL / Dendy）**
    如果這個模擬器支援 PAL（1 CPU = 3.2 PPU dots）或 Dendy，`mcCycleCount * 3` 的硬編碼會產生較大的比例誤差。
    如果你想讓寫法更 robust，可以考慮不要轉換成 PPU dots，**直接把 open-bus 衰減判定的閾值（decay_period）也換算成 CPU Cycles 基準**，這樣判定式就會變成單純的 `mcCycleCount - stamp >= CPU_DECAY_PERIOD`，連乘 3 的運算和區域差異的耦合都能直接拔除，架構上會最乾淨。

### 總結
你的除錯邏輯非常完美：分清了 Host 狀態（`frame_count`）與 Guest 狀態（真實 PPU 時序），並將依賴從「容易帶有副作用與邊界條件的前端計數」轉移到了「底層全局單調時鐘（Cycle Count）」。請安心按照你的修法執行（留意上述細節 A 與 D 即可），這是非常專業的模擬器架構優化。
