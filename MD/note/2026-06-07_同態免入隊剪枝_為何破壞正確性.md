# 同態免入隊剪枝(same-state turn-on prune):從「破壞正確性」到「PullUp-guarded 成功 +11.85%」

> 日期:2026-06-07。**結論:天真版會破 bit-exact;加上 PullUp 守衛後 bit-exact 且 +11.85%(14/14 配對全勝),已併入 main(commit ed8c457)。**
> 來源:2026-06-06 Gemini 演算法諮詢的點子 3。對照演算法說明 `06_S1引擎演算法_資料結構與程式碼對照.md`。
> 注意:本文先前的版本把原因歸給 ForceCompute —— **那是錯的**,真因是「無 PullUp 的動態/儲存節點走浮接 hold-previous」。以下是完整修正。

---

## 1. 想法

事件驅動 settle 裡,閘極 `nn` 變高、它控制的電晶體 `T(c1,c2)` 導通時,會把 `c1` 入隊(單側入隊,BFS 會穿過導通的 T 到 c2)。

剪枝:**若此刻 `NodeStates[c1] == NodeStates[c2]`,連 `c1` 都不入隊** —— 直覺是「把兩個同值群組接起來,值不會變,免得多做一次 settle」。

## 2. 天真版:壞(checksum 不符、畫面全黑)

- checksum `0xB60D…` ≠ 黃金;`--frame-dump` 第 20 格**全黑**。
- 一度看到 +51%,但那是「跳過必要計算、模擬幾乎停擺」的假象。

## 3. 找真因的過程(方法論)

加診斷開關 `--no-prune`(關剪枝當對照)+ `--dump-states`(倒出每個節點狀態),在 **hc=100** 把 golden vs pruned **逐節點比對**,再用 `--names` 把分歧節點 id 轉成名字:

```
ppu.oam_ram_100_a0..a7 / b0..b7   ← OAM(精靈)RAM 儲存節點
ppu.pal_ram_11_* / pal_ram_13_*    ← 調色盤 RAM 儲存節點
cpu.p2 / cpu.Pout2 / cpu.pipeUNK17 ← 動態 CPU 節點
```

分歧只有 ~50 個節點,**幾乎全是「有記憶性質」的儲存/動態節點**。這直接指向真因。

## 4. 真因:無 PullUp 的節點會走「浮接 hold-previous」,剪枝對它們不安全

群組解析 `FlagsToStateOf`(`WireCore.Group.cs:42`)在「合併 = 旗標 OR」下**幾乎單調**(合併只加旗標、取最高優先序),所以「兩同值群組合併後同值」對**被驅動**的群組成立。但有兩個非單調來源:

- **(a) 無 PullUp 的節點**:它的群組可能變成**純浮接**,值由 `GetNodeValue` 的「**連接數(電容)最大的成員保留前值**」tie-break 決定 —— 也就是**動態邏輯與儲存單元**(OAM/調色盤 RAM、動態 CPU 節點)靠電荷保持狀態的機制。這類節點的「目前值」是**儲存狀態**,不是輸入的純函數,所以「輸入同值 ⇒ 不變」不成立;連接儲存節點到 bitline 的存取**必須**被求值才能正確 settle。這是**主因**(8,331 個節點)。
- **(b) ForceCompute 通道分量**:FC 的 Gnd+Pwr 抵消會**移除**驅動,非單調(僅 709 個節點,且多與 (a) 重疊)。

> 反面教材:先前只怪 (b) ForceCompute —— 實測卻發現整顆網表只有 8 個 FC 節點(全在一個 709 節點分量),光 guard 掉 FC **仍然壞**(checksum `0x3C8F…`、畫面仍黑)。是逐節點 diff 才揪出真因 (a)。**教訓:用實測 diff 找根因,別只靠推理。**

## 5. 修正:PullUp-guarded —— 只在「保證被驅動」的靜態節點上剪枝

判準很乾淨:**有 PullUp 的節點**,任何包含它的群組都帶 PullUp 旗標 ⇒ 一定走被驅動的 LUT(不會走浮接 tie-break)⇒ 單調 ⇒ 剪枝可證明 bit-exact。

`ClassifyPruneTaint()`(`WireCore.FastPath.cs`,在 Reset 跑一次、非 hot path)建一張 per-node `PruneUnsafe` 遮罩:無 PullUp **或** 在 FC 分量 ⇒ taint。`SetNodeState` 的 turn-on 入隊只對未 taint 的節點剪枝:

```csharp
if ((pruneUnsafe[c1a] != 0 || nodeStates[c1a] != nodeStates[c2a]) && nextHash[c1a] == 0) { enqueue c1a; }
```

taint 比例:58.7%(8,331 無 PullUp + 709 FC),剪枝只作用在其餘 41.3%。

## 6. 結果(嚴謹驗證)

- **bit-exact**:整機 checksum `0x794A43ABDF169ADA` @300k、`0x6D4CCBCE2E9CD599` @1M,皆與 golden 完全相同。
- **+11.85%**:interleaved-paired(同一 binary,`--no-prune` vs 剪枝,14 回合 300k full_palette)中位數 90,084 → 100,762 hc/s,**14/14 全勝**。
- 專案自 R-1 以來最大的單一改善。已併入 main(commit ed8c457)。

## 7. 教訓

- 「在事件源頭剪掉傳播」**必須對群組解析函式的單調性成立**;本引擎的非單調來源是「浮接 hold-previous(無 PullUp 動態/儲存節點)」與 ForceCompute。
- **PullUp = 是否被靜態驅動**,是區分「安全可剪(組合邏輯)」與「不可剪(動態/儲存)」最乾淨的靜態判準。
- 之前 Gemini 諮詢說「演算法層已榨乾」其實**不對** —— 點子 3 經過「找出正確的安全子集」後就是一個真實的大勝。方法論價值:**負面結果要查到根因,可能藏著正面結果。**
