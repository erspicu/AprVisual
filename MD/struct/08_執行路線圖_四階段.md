範圍策略（貫穿四階段）：

- **S2 的抽取是「一套演算法跑整張網表」，從一開始就涵蓋整片板子** —— 不是「2A03 專用 / 2C02 專用」，也不刻意把 CPU 和 PPU 隔開。整片 NES 是一張互連的電晶體網（2A03 + 2C02 + 板上 74xx 膠合 + CIC + 手把 + NROM mapper 邏輯；記憶體陣列維持行為級、不在抽取範圍內，是模型的邊緣）。**人為地「先抽 2A03、PPU 留 switch-level」反而糟**：那要設計一個「IR 區 ↔ switch-level 區」的跨區協定（共享 bus 何時看到對方更新、clock phase、settle 順序），而那正是設計文件失敗點 #1（dynamic / 回授 / 共享 bus 抽象錯誤）的溫床；而且 S3 的加速會被 Amdahl 卡住（PPU 是電晶體數更大的那塊）。
  - **「分階段」改成按結構複雜度（在同一套抽取器裡），不是按晶片**：先處理良性結構（反相器 / NAND / NOR / 串並聯下拉 / 明確的 precharge-evaluate）→ 那已 cover 晶片大部分 → 得到高覆蓋率 IR + 少數還沒處理的區域 fallback 回 switch-level（hybrid）→ 隨著處理更多結構類型，覆蓋率往 100% 爬。所以**很早就有東西在跑**（高覆蓋率 hybrid），逐節點 diff 當 debug 工具（第一個發散的 node 在哪 / 何時 → 知道是哪類結構抽錯）。「最難的結構」（shift register / OAM DRAM / 共享 bus / 強耦合 dynamic）剛好大多落在 2C02，所以實務上仍是「2A03 那塊先全乾淨、2C02 那塊最後收尾」—— 但這是結構難度的自然結果，不是計畫上的晶片切割。
  - **hybrid = fallback，不是計畫**：給個別頑固區域用的逃生口，不是「CPU 跑 IR、PPU 跑 switch-level」這種刻意切割。S2 過關條件可放寬到「IR 覆蓋率 ≥ X%、其餘 hybrid、整體 trace（整片板子）逐節點仍一致」。
- **只 NROM（mapper 0）**。跟 MetalNES 一樣（它就是 "No MMU support"）。
- mapper、AI 大規模訓練環境、「整顆 NES 常駐 GPU kernel」都是 S4 之後或 stretch goal。

> **路線圖調整 #1（2026-05-12，user）**：原本寫「S2 的 IR 抽取從 2A03 開始、PPU 延後」—— 改成上面這版（抽取從一開始就整張網表、分階段按結構不按晶片、hybrid 只當 per-region fallback）。理由：MetalNES 基礎本來就是整片板子一起跑的、晶片是一個互連整體；單獨優化一塊、delay 另一塊會引入「IR↔switch-level 跨區邊界」這個 side-effect 來源。下面 §S2 的子步驟敘述同理理解（S2.4「先做良性區域」= 先做良性**結構**，不分晶片）。

> **路線圖調整 #2（2026-05，study + 實作後修正）**：S1、S2 照計畫完成（S2 過關 gate 過 —— `IrEngine` checking + driving 兩種模式都 0 mismatch）。**S3 從「CPU proof」拓寬成「whole-system IR 最佳化」**（user：「每顆晶片都要優化處理，不是只有 cpu，最後展現效能是一整體的」），且 **Gemini review 後把「IR-CPU 比 S1 快」從硬 gate 拿掉** —— 因為：(1) GPU bit-slicing 跑不了 event-driven 引擎（SIMT 討厭 branch divergence / queue），只能跑 brute-force flat program（無條件全算）+ 固定 K 次迭代（無 early-out）；(2) brute-force 在單實例 CPU 上**預期比 S1 慢**（多做 ~10–100× node-work），但成本**不隨並行度長** → 真正的加速來自 S4 的 bit-sliced 並行（CPU `ulong`=64 台 / AVX-512 更多 / GPU 上千台），每實例吞吐量 = 單實例成本 ÷ 並行度；(3) 所以 S3 的目標改成「把 IR 弄成盡量 acyclic 的高覆蓋率 DAG」（GPU 友善的前提）、殘餘的小 SCC / 多驅動 bus 不強求 S3 100% 拆掉、留給 S4 的 codegen 用 Fixed-K micro-block 包起來。**S3 ↔ S4 的交接物（中間產物）= 那份 IR**：`IrEngine.NextExpr[]`（每 node 的 next-state 表達式）+ `EvalOrder[]`（拓樸序，driving-covered 的 ~85%）+ `InScc[]` + 殘餘 SCC components + hybrid-bus 集合。S4 從這裡接手做 codegen。下面 §S3 的子步驟與過關條件已照這版改寫。

## S1 — C# 改寫 MetalNES 核心（基礎平台）

**目標**：得到一個能跑、且驗證過的 C# 開關層級模擬器。這是「繼續前進的基礎」——後面 S2/S3/S4 都站在它上面。

**為什麼是改寫 MetalNES 而不是從 chipsim.js**：MetalNES（`ref/metalnes-main`）已經是 chipsim.js 演算法的工程化升級版（flags 查表、transistor 預分類、SoA、雙緩衝 recalc list、收斂式 settle），而且把 bus 整合、真實 reset、support chip、handler 機制、test ROM 驗證全做完了。詳見 `MD/note/`。我們抄它的架構，不重新摸索那 4 個月。

## S2 — netlist 高階化成 IR

**目標**：從開關層級網表**抽取**出一個更高階的表示——每個節點的「下一個狀態 = 某個布林表達式 / 保留前值」——並用一個「IR 直譯器」執行它，行為要跟 S1 的開關層級直譯器一模一樣。

**這是整個專案最有價值、也最危險的部分**（設計文件 §15.3 + `04` 號文件的失敗點 #1：dynamic node 抽象錯誤）。MetalNES **完全沒做這層**——它永遠是直譯開關層級。我們要走到這裡。

### S2 子步驟

| # | 內容 | 對應設計文件 |
|---|------|--------------|
| S2.1 | **靜態圖建構**：標記供電節點（VCC/VSS/GND）、外部 pin（`clk0` `res` `rw` `ab[]` `db[]` …）、建鄰接 | §6.1 |
| S2.2 | **通電圖 / 浮島分析**：對每個 clock phase，決定哪些電晶體 ON，算連通分量；判斷每個節點/分量是「連到 GND」「連到 VCC/pull-up」還是「浮島」 | §6.2、§7.3 |
| S2.3 | **迴路 / 回授偵測**（MetalNES 跳過的部分）：三色 DFS / Tarjan SCC，把回授群縮成超節點，分類為 **static cross-coupled latch / dynamic storage / bus-style 回授**。停止遞迴防無限展開，但更重要的是**把序向邏輯與組合邏輯分開** | §6.4、§10 |
| S2.4 | **布林表達式抽取**：對良性區域（反相器、NAND/NOR、串並聯下拉網路、明確的 precharge/evaluate 區）→ 抽成 `Expr` 樹（串聯=AND、並聯=OR、有路徑到 GND→0、無下拉且有 pull-up→1）；對 dynamic 區域 → `HoldExpr`；對回授 → `NodeRefExpr` 指向「上一輪狀態」。**先做良性區域，不要從共享 bus / 強耦合 dynamic 叢集開始** | §8.1、§8.3、§11–§13 |
| S2.5 | **IR 資料結構**：`Expr` = `ConstExpr` / `NodeRefExpr` / `NotExpr` / `AndExpr` / `OrExpr` / `MuxExpr` / `HoldExpr`；每個節點一條 next-state 表達式；標記哪些是組合（可拓樸排序）、哪些是序向（在 clock edge 更新） | §8.2 |
| S2.6 | **IR 直譯器**：拓樸排序組合邏輯，在 clock phase 邊界跑序向更新；保留兩相時鐘、half-cycle、必要時迭代 settle | §11 |
| S2.7 | **等價 gate**：IR 直譯器對同一批 test ROM 的 trace 必須與 **S1 開關層級直譯器逐節點完全一致**。任何不一致 → 回 S2.3/S2.4 修。也要能 dump `ir_debug.json` 供人工檢視 | §13.6、附錄 B |

### S2 過關條件

- ✅ IR 抽取對 2A03（至少 CPU 核心的大部分）完成，能 dump `ir_debug.json`
- ✅ IR 直譯器跑得通 S1 跑得過的那批 test ROM
- ✅ IR 直譯器與 S1 開關層級直譯器的 trace **逐節點、逐 cycle 完全一致**（這是硬條件，不是「看起來差不多」）
- ✅ 迴路 / 回授區域有被正確分類（不會因為強制展開而表達式爆炸或不收斂）

> 註：如果某些區域（典型是共享 bus、強耦合 dynamic）抽不出乾淨的 IR，可以暫時對那些區域**保留開關層級求值**（hybrid：大部分跑 IR，少數難搞區域跑 group 演算法），但要記錄下來、標清楚，並在 S2 過關條件裡放寬到「IR 覆蓋率 ≥ X%、其餘 hybrid，整體 trace 仍一致」。

## S3 — whole-system IR 最佳化（拆環、拉覆蓋率，往 GPU 友善的 acyclic DAG 靠攏）

> **2026-05 修正**：原本這階段叫「CPU proof」（在 CPU 上 benchmark 證明 IR 比 S1 快很多、當 S4 的 go/no-go）。實際做下去 + Gemini review 後改成現在這版（理由見上「路線圖調整 #2」）。詳細的 firing-by-firing 過程在 `MD/impl/S3/00_S3_效率與優化.md`。

**目標**：把 S2 抽出來的整片網表 IR（每 node 一個 next-state 表達式）**最佳化成一個盡量 acyclic、高覆蓋率的 DAG** —— 這是 GPU bit-slicing 的前提（GPU 跑不了 event-driven 引擎，只能跑「無條件全算的 flat program」+「固定 K 次迭代」）。順便量 CPU benchmark，但**不**把「IR-CPU 比 S1 快」當硬 gate（單實例 brute-force flat IR 預期比 event-driven S1 慢；加速來自 S4 的並行）。殘餘沒拆掉的小 SCC / 多驅動 bus 不強求在 S3 100% 拆掉 —— 留給 S4 的 codegen 用 Fixed-K micro-block 包。

- ✅ S2 的逐節點等價 gate 仍全過（IR ≡ S1，checking + driving 兩種模式，每 node 每半週期）—— γ.0/γ.1 不破壞它。

## S4 — Codegen + GPU 加速

> **狀態：未開始**。設計草稿（含要問 Gemini 的問題）在 `MD/impl/S4/00_codegen_設計.md`（在 S3 後期草擬 —— Gemini 判「別繼續鑽 γ.2、把 codegen 提前想/動」，但它在 chain 上仍是 S4）。

**目標**：從 S3 交來的那份 IR emit 可執行 backend。主軸是 **bit-sliced kernel**——一個 `uint32`/`uint64` 字（或 AVX2 `Vector256` / AVX-512 `Vector512`）裝 32/64/256 台 NES 的同一個節點，`&` `|` `^` `~` 取代 `&&` `||` `!`、`Mux(c,a,b)`→`(c&a)|(~c&b)`，SoA 布局，一條表達式同時算 32/64/256 台。

**這階段要處理的三類 node**：(1) **driving-covered 的 ~85%**（有 NextExpr、acyclic）→ 直接把 tree 編成 `node[v] = <expr>`、照 `EvalOrder` 排（一個 pass）—— 就是 `RunFlatProgram` 的 emitted 版本；(2) **殘餘 56 個小 SCC（843 node）→ Fixed-K Micro-Evaluation**：給 SCC 任意拓樸序、emit `for k in 0..K(=3~8): { node[v] = <NextExpr[v]>; ... }`（`#pragma unroll` / 手動展開）—— SCC 內 `Node(x)` 讀同一 K-loop 內最新值（Gauss-Seidel）、init = `Prev`。**GPU 友善（每個 thread 無腦跑 K 次、零 divergence）、物理正確（吸收 settling ripple、不需知道 derived-clock 順序）**；(3) **~1379 個 hybrid 多驅動 tri-state bus node**（`NextExpr==null`）→ 可能先抽 pseudo-NextExpr（`bus = Mux(drv1_en, drv1_val, Mux(drv2_en, drv2_val, …, Hold(bus)))`，從 DriveAnalysis 的 `Passes` + pull-down 組）→ 跟殘餘 SCC 一起丟 Fixed-K block；或暫時混合呼叫 S1（CPU 上勉強可、GPU 不行）。

**後端選擇**（user 定調，2026-05）：**做兩份 —— 一份純 C#（Roslyn-emit-C#-source）+ 一份 LLVM（LLVM-via-.NET）—— 互相做效能對照**。codegen 工具本身都用 C# 寫（不引入 C++ toolchain 依賴）。
- **C# 後端**：emit 一個巨大的 `static void Step(...)`（bit-sliced `& | ^ ~`）、Roslyn 動態編譯。陽春、快 prototype、量 baseline、邏輯正確性驗證。風險：~100K 行 / 數萬 local 的方法可能把 RyuJIT 的 register allocation 搞爆（甚至 fallback unoptimized）—— 量出來看。
- **LLVM 後端**（`LLVMSharp` / LLVM-C API via P/Invoke）：emit LLVM IR → `-O3`（跨全域 const-folding / DCE / instruction-combining、再壓縮網路圖一輪；greedy regalloc 處理巨型 basic block 遠超 RyuJIT）→ JIT/AOT 成 native。**用法參考 user 抓下來的另一個 .NET+LLVM 專案**（同 owner 的 repo，例如 github.com/erspicu/AprGba —— .NET 裡用 LLVM 當 codegen 後端）。**最關鍵的賣點：同一份 LLVM IR 後面可以 retarget 到 GPU（NVPTX / 其他 target），只維護一套 `IRBuilder` 邏輯。**
- 兩份的等價 gate（vs S1）都要過；benchmark 報告列「S1 / C#-emit / LLVM-emit / LLVM-emit + bit-sliced ×N」的 throughput 對照。

**GPU 化（尚待討論，不急著定）**：CUDA / NVPTX **不是唯一選項**。考量：(1) C# 對 CUDA 的支援/library 不一定好處理；(2) **GPU compute shader / shading language**（HLSL compute shader、GLSL/SPIR-V compute、WGSL、…）可能更適合 .NET 生態 —— bit-sliced 的 boolean kernel 本來就只是一堆 `& | ^ ~`，compute shader 跑得動；(3) **參考 user 的 `AprNesAvalonia`**（AprNes 的 .NET 版、用 Avalonia UI 框架、自己發展的 shader script）—— 那套「在 .NET 裡跑自訂 shader」的做法到時候可以借。所以 S4 的 GPU 那一步先不寫死 target，等 C#/LLVM 兩份 CPU 後端做出來、量過、再討論 GPU 用 CUDA-via-LLVM-NVPTX 還是 compute-shader 路線。

## 一句話總結

> **S1** 站在 MetalNES 肩膀上得到一個我們完全掌握的 C# 開關層級模擬器（基礎，✅ 完成）；**S2** 把它高階化成 IR —— 每 node 一個 next-state 布林表達式 + 一個 IR 直譯器，行為跟 S1 逐節點一致（最有價值、最危險，✅ 完成、等價 gate 過）；**S3** 把整片網表的 IR 最佳化成盡量 acyclic 的高覆蓋率 DAG（GPU 友善的前提；🚧 ~85%、在穩定停點 —— 殘餘小 SCC / 多驅動 bus 留給 S4）；**S4** 從那份 IR emit bit-sliced codegen（C# / LLVM），殘餘的環編成固定 K 次的 micro-block，做大規模平行（終局；⬚ 未開始）。每一步之間都有逐節點等價驗證，沒過不准前進。
>
> （原本 S3 寫成「在 CPU 上證明 IR 又對又快很多 = go/no-go 分水嶺」—— study + 實作 + Gemini review 後修正：「IR-CPU 比 S1 快」不再當硬 gate，因為 GPU 跑不了 event-driven 引擎、只能跑 brute-force flat program + 固定迭代，那在單實例 CPU 上預期比 S1 慢但成本不隨並行度長 → 加速來自 S4 的並行；S3 的真正產出是「acyclic 的高覆蓋率 IR」這個中間產物，S4 從它接手。詳見「路線圖調整 #2」。）
