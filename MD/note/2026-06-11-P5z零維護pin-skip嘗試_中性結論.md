# P-5z:零維護 pin-skip 翻盤嘗試 —— 中性結論(2026-06-11)

## 一句話

P-5 的「維護 ~7% 地板」這次被**真正繞過了**(零維護、skip 現場推導、bit-exact),但量測仍是**中性**(+0.16% / −0.17%,10/20):
可以免維護推導的那一腿(gnd 腿)壓的全是**引擎裡最便宜的 pop**,測試成本恰好吃光獎賞;
而值錢的那一腿(PullUp 腿,佔舊 P-5 壓制量 64%)被二分法**證明本質上需要維護狀態** —— 牆的位置這次標到了公分級。

## 設計(留在分支上的最終形)

- **類別**(載入時靜態分類,每次 Reset 從結構重算 = 基準真相,免驗證、免 safe-degenerate、與編號無關):
  恰好 1 個自有 gnd 通道、0 pwr、有 pass 通道、乾淨元件(排除 FC / callback / driven / clk / res / **閘極回饋**元件)。
  全機 6,338 個節點(43.0%);PullUp 與否不影響測試(只是舊分類的 M/D 來源)。
- **測試**(關斷走訪端點 c):`skip ⇔ NodeStates[ProbeGate[c]] != 0`(該 gnd 通道的閘極目前導通 ⇒ Gnd 是 LUT 最高優先 ⇒ 任何 pass 斷開都不改變解析 ⇒ 重算必為 no-op;**暫態也成立**)。
- **編碼**:全尺寸唯讀表 `ProbeGate[id]`,非類別節點 = 0 → `NodeStates[0]==0` 自動失效 —— 無範圍比較、無遮罩 ALU、無新分支(摺進既有 nextHash 條件)、零寫入、零每解析成本。c1 側用 `(2-c2)>>31` 保住自有供電關斷與 callback 假電晶體路徑;c2 側免守衛(其對側 c1 永非供電)。
- **自我守衛失效**:probe 閘自己關斷時,SetNodeState 已先寫 0 → 測試自然失敗 → 照常入列。

## 為什麼中性(算式)

- 關斷端點測試量:**50.1M 次/10 萬 hc**(c1 25.05M + c2 25.05M),壓制 5.3M → **每 9.4 次測試換 1 次跳過**。
- 被跳過的 pop 是單 gnd 葉節點的 fast-path 解析(~30 cycles 級,引擎裡最便宜的一種)→ 損益兩平需要每次測試 < ~3 cycles —— 2 個 branch-fed 載入 + OR 剛好就在這條線上。兩個變體一個略虧一個剛好打平,夾出結論。
- E0 的換算(0.9M 壓制 ≈ +1%)對這個類別**高估**:那是用舊 P-5 混合類別(含貴的 BFS 成員 pop)校準的。

## PullUp 腿的負結果(本次最重要的新知識)

二分法實測:把「c==1(PullUp 撐住)⇒ skip」也放進去 → **checksum 100k 就發散**(0xABF16ED2…)。
原因:該腿的證明(「值為 1 ⇒ 群組無 gnd 可達」)只對 **c 上次解析時的值**成立;
零維護版讀的是**即時** NodeStates[c],會撿到 settle 中途、群組值還不一致的暫態 —— 此時壓掉重算會丟失殘餘群組裡**其他成員**的修正。
維護版 pinned 位元(每次解析重算+寫入)恰好就是「上次解析時的值」的快取 —— **這就是它非要維護不可的本質**。
⇒ P-5 的牆精確化:**值錢的腿(64%)需要維護;免維護的腿(~43% 事件量)利潤為零。** 不要再嘗試以即時狀態替代 pinned。

## 對抗式評審的其他產出(已落地)

- **舊 dominant-bypass 分支有兩個潛伏洞**(當時 300k/1M golden 通過是**倖存者偏差**):
  (1) FC 元件隊友:pinned-high 的證明在 ForceCompute 的 Gnd+Pwr 互消下不成立;FC 集合 = 2C02 spr_d 精靈/OAM 匯流排,full_palette 幾乎不運動它(SMB1 才會)。
  (2) 成員迴圈內過期窗:群組解析的成員 SetNodeState 走訪會讀到同組成員尚未刷新的 pinned。
  → 未來任何 pinned 類設計必須:FC/driven/callback **元件級**排除 + 閘極回饋元件排除(或成員預清除)。
- **E0 類別拆解**(dominant-bypass 分支 DEBUG 儀器,12.37M 壓制):merged(PullUp+1gnd)64.0% / D2-gnd 7.1% / D2-pwr 1.7% / 多供電殘餘 24.5% / 健全性排除僅 2.6%;**被消費 pin 的 83.2% 由 BFS 成員迴圈寫入**(→「只在 fast path 維護」的省成本路線確定死)。
- hash 位元同居方案(pinned 塞進雙緩衝 RecalcHash)被成本分析否決:雙緩衝迫使每解析 RMW 兩條陣列,抵銷讀側免費。
- 成本模型代理對 V0(維護版改良)的預測 +2%±4 —— 與實測一致(我們直接跳過了它)。

## 規則更新

1. **「零維護」不是免費**:刪掉維護後,**測試端的每端點成本 × 測試/跳過比** 成為新的地板;對便宜 pop 類別,2 個載入就足以打平。評估任何 skip 設計先算:跳過一次省幾 cycles ÷ 每次跳過要做幾次測試。
2. **「即時狀態」≠「上次解析時的狀態」**:所有依賴「值 ⇒ 群組性質」的證明都綁定解析時刻;免維護推導只能用**暫態也成立**的事實(如 Gnd 最高優先)。
3. golden 通過 ≠ 健全(survivorship):工作負載蓋不到的區域(FC/精靈)要嘛結構性排除、要嘛換 ROM 加門(SMB1 已加入本次門檻)。
