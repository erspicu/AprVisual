## Full implementation detail + cache/working-set characterization

### Golden engine (12.0 µs/hc)
- Struct-of-arrays, unmanaged pointers: `byte* NodeStates` (1B/node, ~14.7 KB total), `NodeInfo*` (32B/node: flags + an inline payload of channel adjacency = (gate,far-end) pairs + gnd/pwr gate lists), `ushort* TransistorList` (flattened overflow adjacency), `byte FlagsToState[256]` LUT.
- Hot path: a dirty queue; for each dirty node, ComputeNodeGroup does a tiny BFS over ON transistors, collects the connected group (avg 1.4 nodes), OR's their flags, indexes FlagsToState → resolved 0/1; SetNodeState writes it and enqueues downstream gated nodes.
- **Memory profile: the CODE is tiny (one BFS loop + LUT) → fits L1i trivially. The DATA touched per hc is only ~600 nodes' worth of NodeInfo/NodeStates → fits L1d/L2. This is why it's fast: tiny code, tiny hot working set, event-driven.**

### Oblivious model — INTERPRETER (EvalObliviousRelax), 539 µs/hc (45× slower)
- `byte* LogicState` (parallel state, ~14.7 KB), `int* _logicOrder` (the ~6,000 oblivious node ids, in a fixed sweep order), `int* _covBase` + `ushort* _covInputs` (flat per-node input-id lists), `byte* _logicTT` (**~1 MB** of dense truth tables), a `Dictionary<ulong,byte>[]` for sparse (17–60 input) nodes, `byte* _logicBus` (flag), and a structural BusResolve (a small BFS like golden's) for wide nodes.
- Per relaxation iteration: for each of ~6,000 nodes in `_logicOrder`: pointer-chase `_covInputs+_covBase[nn]`, loop k inputs GATHERING bits from `LogicState`/`NodeStates` (SCATTERED random reads by input node-id), pack a key, then dense `_logicTT[base+key]` OR a Dictionary lookup OR a BusResolve group-walk. ~6.5 iterations to fixpoint.
- **Memory profile: CODE is small (one loop) → L1i fine. DATA is the problem: the per-node input GATHER is scattered random access across LogicState/NodeStates by node-id (no locality), and `_logicTT` is ~1 MB (exceeds L1d, ~half of a typical 256KB–1MB L2). So d-cache misses on the gather + TT. Plus we re-touch all 6,000 nodes 6.5× regardless of what changed (no sparsity).**

### Oblivious model — COMPILED (Roslyn straight-line), 1004 µs/hc (84× slower; 2× SLOWER THAN THE INTERPRETER)
- We emit ~6,000 straight-line statements (one per node), e.g. `{ int i = ls[100] | (ns[205]<<1) | (ls[88]<<2) | ...; byte v = tt[1234+i]; if (v!=2 && ls[NN]!=v){ls[NN]=v; ch++;} }`, bus/sparse nodes as a call to a helper. Total **~709 KB of generated C# source**, compiled to native in 15 methods. The relaxation calls all 15 methods in order, ~6.5 times per half-cycle.
- **Memory profile: the CODE is now HUGE — ~6,000 baked statements ≈ hundreds of KB to >1 MB of machine code, and we stream through ALL of it 6.5× per half-cycle. A typical L1 instruction cache is only 32–48 KB. So the front-end is almost certainly thrashing the i-cache — every sweep re-fetches code that cannot fit.**

## THE HYPOTHESIS (please confirm/correct and factor in)
The compiled version being SLOWER than the interpreter is the tell. The interpreter is a small code loop over data (L1i fine; d-cache is the cost). The compiled version is a ~MB code blob streamed 6.5×/hc, far exceeding L1i → front-end / i-cache stalls dominate and exceed the dense straight-line savings. In other words, **"compile the oblivious sweep to straight-line code" is self-defeating on a single core precisely because oblivious means touching the WHOLE circuit every cycle, so the code footprint scales with the whole circuit and blows the i-cache.** Golden wins not just on event-count but on having a TINY code+data hot footprint.
