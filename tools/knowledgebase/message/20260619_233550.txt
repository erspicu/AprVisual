Here is the concrete, bit-exact, no-reorder blueprint for destroying your ~6 MPKI branch mispredicts, prioritized for Zen2/Cortex-A76.

At 245M misses and a typical 14-cycle mispredict penalty, you are losing **~3.4 billion cycles** to this. Out of your 20.5G total cycles (41G / 2.0 IPC), this is **~17% of your total execution time**. It is nearly tied with your 19.5% dependent-load stall.

### 1. The DISPATCH Branch (Prediction-friendly, zero reorder)
**Verdict: Use a `switch` backed by a 1D Look-Up Table (LUT). Do NOT use `cmov` for the dispatch itself.**

*Why predication/cmov fails here:* To use `cmov` on dispatch, you'd have to speculatively execute *both/all* paths (size-1, pair, general) and `cmov` the final simulator state. This is disastrous on Zen2/A76 because it bloats your uops and puts the *expensive* "general" path on the critical data-dependency chain for every single pop, destroying your IPC.

*What actually helps the OoO core:* **Flatten the if-else tree into a single direct table lookup -> single indirect jump (or switch).**
Instead of:
```c
if (flags & SIZE1) { ... }
else if (flags & PAIR) { ... }
else { ... }
```
Do:
```c
// Precompute a 256-byte LUT at init time that maps all combination of node class/flags to an enum (0=Size1, 1=Pair, 2=General).
uint8_t handler_idx = DispatchLUT[node->flags & FLAG_MASK];
switch (handler_idx) { ... }
```
**Effect:** You convert a tree of multiple conditional branch mispredicts into a *single* branch instruction (an indirect jump or bounds-checked jump). The Zen2/A76 BTB (Branch Target Buffer) and indirect predictors (TAGE) are much better at finding correlations in a single instruction's history than tracking paths through a nested if-tree. Even if it mispredicts, it only misses *once* per pop, strictly capping the latency penalty.

### 2. Group-Resolution PRIORITY (Branchless & Bit-Exact)
**Verdict: YES, a branchless formulation is possible, bit-exact, and will violently outperform the if/else tree.**

Instead of an `if/else` tree checking `GND > VCC > driven > hold > float`, map each node's contribution to a **single 32-bit (or 64-bit) integer score** and take the `max()`. 

```c
// Encode each node's state into a score. 
// Bits 28-31: Base Priority (GND=5, VCC=4, Driven=3, Hold=2, Float=1)
// Bits 0-27: Capacitance (or other tie-breakers)
uint32_t current_score = (PriorityLUT[node_state] << 28) | node->capacitance;

// Branchless max (preserves bit-exact queue order tie-breaking because '>' ignores later exact ties)
if (current_score > best_score) {
    best_score = current_score;
    winning_node_val = node_val;
}
```
**Effect:** A Zen2/A76 core compiles `if (A > B) best = A` to a `CMP` and a `CMOV`. This executes in ~1 cycle per node with **zero branch mispredicts**. Because a `CMOV` only adds a 1-cycle data dependency and doesn't flush the pipeline, it will trivially beat a mispredicted branch tree. 

### 3. SEGREGATION without Reorder (Filtering the 70%)
**Verdict: Yes, "removing the easy cases" is exactly how you fix the dispatch.**

When you handle the 70% size-1 cases via the co-consult's branchless gate-OR, you effectively bypass the complex dispatch. 
*   **The Math:** If the size-1 cases were highly predictable, your branch mispredict *rate (%)* on the remaining 30% will skyrocket (because only the high-entropy, complex cases are left).
*   **The Reality:** *You don't care about the rate, you care about the absolute cycle penalty.* Removing the 70% drops the absolute volume of branches flowing through the predictor. Furthermore, diluting a branch predictor with 70% "noise" (even easy noise) evicts useful history for the 30% hard cases. By removing the 70%, the BTB and TAGE predictors can dedicate their capacity entirely to learning the patterns of the 30% complex pops. 

### 4. The C#-vs-Rust Branchless Sign-Flip (When to Predicate)
**The Principled Rule:** A correctly predicted branch has **zero latency** on the data path, because the CPU executes the predicted path speculatively. A `cmov` has **non-zero latency** because the CPU *must* wait for the predicate condition to resolve before it can pass the selected data forward.

*   **Why Rust hurt (-3.2%):** Native compilers (LLVM) optimize branch layout extremely well. The branch was probably predicted correctly >90% of the time. When you forced it to be branchless (`cmov` or bitwise masking), you serialized the data dependency. In a simulator severely bottlenecked by a per-pop dependent-load chain, adding *any* latency to the critical path via `cmov` causes the backend to stall further.
*   **Why C# helped (+0.6%):** The JIT produced a clumsy branch structure (perhaps with bounds checks or poor block layout) that the CPU failed to predict. The 14-cycle mispredict penalty outweighed the 1-cycle `cmov` latency.
*   **How to decide:** If the code is inside your #1 dependent-load chain (e.g., fetching the next node), **prefer branches** unless the mispredict rate is astronomically high (>20%). If the code is *off* the critical path (e.g., computing the state to write back *after* the next pop is already being fetched—like the Priority Resolver in Q2), **prefer branchless `cmov`**.
