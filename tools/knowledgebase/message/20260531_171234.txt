### A. 演算法盤點 (名稱 + 作者/年份 + 對微小動態 component 之效用)

1.  **Direction-Optimizing BFS (Beamer, 2012)**
    *   **效用：完全無效。** 它的核心是當 frontier 增長到一定比例時切換為 bottom-up 掃描全圖，對於永遠只有 2~4 個節點的微小走訪，frontier 從未跨過閾值，連 branch predictor 都會被 if-else 檢查拖累。
2.  **Fully-Dynamic Connectivity / HDT Algorithm (Holm, de Lichtenberg, Thorup, 2001)**
    *   **效用：完全無效。** 透過維護多層級的 Spanning Forests（通常用 Euler-tour trees）來達成 $O(\log^2 V)$ 邊更新。但在 4 個節點的 component 中，$O(\log^2 V)$ 的常數成本（樹旋轉、層級推移）比跑完 BFS 慢上千倍。
3.  **Link-Cut Trees (Sleator, Tarjan, 1983) / Euler-Tour Trees (Henzinger, King, 1995)**
    *   **效用：完全無效。** 專為動態維護「樹 / 森林」結構設計。NES 電路圖是高度環狀的（SCC 佔 94%），維護非樹邊的成本極高，且每節點需增加至少 3 個指標（Left, Right, Parent），直接撐爆 32KB L1d。
4.  **Ligra / GraphIt (Shun, Blelloch, 2013 / Zhang et al., 2018)**
    *   **效用：完全無效。** 這些是針對大圖的 Cache-oblivious 框架與排程 DSL，為了抽象化犧牲的 function call 與 iterator overhead 甚至大於你 4 個節點 BFS 的總時鐘週期。
5.  **Incremental Connectivity / Disjoint-Set Union-Find (Tarjan, 1975)**
    *   **效用：完全無效。** Union-Find 對動態「增加」邊極快，但 switch-level 模擬有大量的「電晶體關閉（邊刪除）」。支援刪除的 Union-Find 會退化或需要極大的額外結構，完全不適合不斷開關的 pass-transistor。

### B. 針對「平均 1.4、最多個位數節點 + 必須留在 L1d」的殘酷評估

**你的假設完全正確：上述所有演算法的漸進複雜度優勢，在個位數節點的 walk 中，會被常數成本與記憶體開銷完全壓垮。**

*   **(1) 操作的額外常數成本：** 對於 2~4 個節點的 BFS，你的 CPU 只需要執行大約 10~30 個指令（讀鄰接表、判斷 gate=high、推入陣列）。HDT 或 Link-Cut Tree 在做任何事之前，光是追蹤一個 pointer 造成的潛在 pipeline stall 或多重條件分支，就已經超過 30 個指令了。
*   **(2) 額外記憶體（L1d residency 破壞）：** 這是最致命的。要在 32KB 內塞入 14.7K nodes + 27K edges，代表平均每個 node 的 budget 只有 ~2.1 bytes（你肯定已經做了極致的 bit-packing 或 CSR 壓縮）。
    任何 Dynamic Connectivity 演算法都需要至少每個 node 儲存 Level (1 byte)、Parent Pointer (2 bytes)、Size/Weight (1-2 bytes)。這會瞬間讓你的圖結構膨脹一倍，直接跨越 32KB 邊界。一旦熱資料溢出到 L2 cache，L2 的 latency（~12 cycles）會吃掉你從演算法省下的所有時間（L1d latency 僅 ~4 cycles）。

### C. 我的誠實判斷：是否還有更聰明的演算法？

**沒有更聰明的「圖搜尋演算法」能贏過 plain BFS/DFS 了。** 但有兩個與圖搜尋無關的思路可以檢視：

1.  **純陣列堆疊的 DFS (Array-based Stack DFS) 取代 BFS Queue**
    *   既然只有 2~4 個節點，DFS（使用 L1d 內的簡單小陣列當 Stack）與 BFS 在走訪節點數上毫無差異。但 Queue 需要維護 `head` 與 `tail` 兩個指標並做 ring-buffer 檢查；Stack 只需要一個 `top` 指標。在極限的 micro-level，這能省下幾個 ALU 指令與暫存器壓力。
2.  **Transistor-Level Memoization (Group Caching)？**
    *   *原理：* 將常出現的 pass-transistor 導通組合直接 Hash 到結果。
    *   *為何可能輸：* Hash key 的計算成本與 Hash table 會佔用 L1d。對於 4 節點的計算，算 Hash 可能比直接跑 BFS + flags-OR 還慢。除非你針對「特定極高頻、大於 2 節點的硬體模組（例如 ALU 的 carry chain）」做硬編碼的 pattern matching，否則通用 memoization 在這裡不划算。

結論：針對這個尺度，通用的高級圖演算法全軍覆沒。

### D. 最終結論

**沒有。plain BFS (或最極簡的 Stack DFS) 在這個尺度已最優，進步只能來自 (i) 減少 BFS 呼叫次數、(ii) micro-level 記憶體/分支工程。**

既然你已經排除了絕大多數常規最佳化，如果你還要榨取最後的效能，建議只能往以下非圖論的方向鑽研：
*   **Micro-level 記憶體工程：** 確保 Graph Topology 陣列與 Node State 陣列在記憶體中完美對齊 (Cache-line aligned)，避免 false sharing 或跨 cache-line 讀取。
*   **分支工程 (Branchless programming)：** 在 BFS 迴圈內，將 `if (gate == high && !visited) queue.push()` 改寫為無分支 (branchless) 的位元運算與條件寫入 (Conditional Move, `CMOV`)，因為小 component 走訪時間極短，Branch Misprediction 帶來的 15~20 cycles 懲罰佔比極大。
*   **減少呼叫次數 (Event Bounding)：** 確保只有當 pass-transistor 的 gate 發生「實質翻轉 (toggle)」，且兩端邏輯準位「不一致」時，才觸發 dirty node evaluation。
