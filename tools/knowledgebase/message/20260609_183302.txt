### 1. Pin the whole PROCESS vs just the THREAD?
**Do not pin the process.** Pin the **specific hot thread**. 
.NET spawns background threads (GC server threads, ThreadPool, Finalizer, ETW, Timer). If you pin the whole *process* to a single core, you force all of .NET's background infrastructure to timeslice on the exact same core as your hot loop. If a background timer or threadpool task wakes up, it preempts your hot loop, trashing your L1/L2 caches.

Because your workload is memory-latency bound, L1/L2 eviction is your #1 enemy. Give the OS freedom to put background .NET threads on other cores, and give your hot thread an exclusive core.

### 2. WHICH core to pick? (And why the user's heuristic is fatal)
The user's heuristic (`ProcessorCount - 1`) was fine in 2015. Today, **it will destroy your benchmark on Intel 12th/13th/14th Gen CPUs.** 
On Intel Hybrid architectures, the highest-numbered logical cores are the **E-cores**. They have smaller caches, higher memory latency, and lower clock speeds. Pinning to an E-core will artificially tank the benchmark.

Furthermore, on CPUs with SMT/HyperThreading, sibling logical cores share L1/L2 caches and execution units. You want a **primary physical core**. Avoiding Core 0 is a good instinct (it handles the bulk of Windows DPCs and hardware interrupts), but you must also guarantee it's a Performance core (P-core).

To do this portably without shipping a CPU database, you must P/Invoke `GetLogicalProcessorInformationEx`, look at `RelationProcessorCore`, and read the `EfficiencyClass` (higher is better).

### 3. Priority
- **Process:** `ProcessPriorityClass.High`. (Do *not* use `RealTime`. Realtime runs higher than OS keyboard/mouse input and crucial hardware watchdogs. It can hard-lock the system).
- **Thread:** `ThreadPriority.Highest`.
Because you are pinning only the *thread*, priority inversion against .NET background threads is avoided (they can run on other cores).

### 4. GC Mode
**Drop the GC tweaks. They are a placebo.**
If your hot loop does zero managed allocations and uses `NativeMemory` / `byte*`, the GC will naturally never run during the timing phase. `GCLatencyMode.SustainedLowLatency` and `GC.TryStartNoGCRegion` only change how the GC behaves *when allocations occur*. Since you allocate nothing, they do absolutely nothing. 

Instead, force a blocking GC *before* your timed region to clean up any garbage left over from the netlist load phase, ensuring the Finalizer queue is empty.

### 5. Default Policy for a Public Leaderboard
**Do NOT pin by default.** Make it opt-in (e.g., `--pin-best-core`). 
If you auto-pin blindly, you take responsibility for OS scheduling bugs. If a user runs your benchmark while playing a game or running OBS, forcing your benchmark onto a specific core might clash with another pinned heavy app.

**Standard benchmarking practice (e.g., BenchmarkDotNet):** Rely on process isolation, High priority, and multiple iterations to smooth variance. Let the OS scheduler do its job by default. Allow power-users to opt-in to affinity for the absolute physical limit, and **record the affinity state in the leaderboard submission payload** so you can compare apples to apples.

### The Code: Finding the "God Core" in .NET 11
Here is the exact `unsafe` C# implementation that dynamically queries the Windows kernel, identifies the highest `EfficiencyClass` (P-cores), filters out Core 0, and extracts the affinity mask for a *single thread* (ignoring the SMT sibling). 

```csharp
using System;
using System.Diagnostics;
using System.Runtime.CompilerServices;
using System.Runtime.InteropServices;
using System.Threading;

public unsafe static class NativePerfTuning
{
    [DllImport("kernel32.dll")]
    private static extern IntPtr GetCurrentThread();

    [DllImport("kernel32.dll")]
    private static extern nuint SetThreadAffinityMask(IntPtr hThread, nuint dwThreadAffinityMask);

    [DllImport("kernel32.dll", SetLastError = true)]
    private static extern bool GetLogicalProcessorInformationEx(
        int RelationshipType, 
        IntPtr Buffer, 
        ref uint ReturnedLength);

    /// <summary>
    /// Locks the current thread to the best available physical P-Core, avoiding Core 0 if possible.
    /// </summary>
    public static void PinToBestPhysicalCore()
    {
        // 1. Elevate priority
        using (var process = Process.GetCurrentProcess())
        {
            process.PriorityClass = ProcessPriorityClass.High;
        }
        Thread.CurrentThread.Priority = ThreadPriority.Highest;

        // 2. Find the best core
        nuint bestCoreMask = GetBestCoreMask();
        if (bestCoreMask == 0)
        {
            Console.WriteLine("Warning: Topology query failed. Falling back to OS scheduling.");
            return;
        }

        // 3. Pin the thread
        SetThreadAffinityMask(GetCurrentThread(), bestCoreMask);
        Console.WriteLine($"[Perf] Pinned hot thread to mask: 0x{bestCoreMask:X}");
    }

    private static nuint GetBestCoreMask()
    {
        uint len = 0;
        // RelationProcessorCore == 0
        GetLogicalProcessorInformationEx(0, IntPtr.Zero, ref len);
        if (len == 0) return 0;

        IntPtr buffer = Marshal.AllocHGlobal((int)len);
        try
        {
            if (!GetLogicalProcessorInformationEx(0, buffer, ref len))
                return 0;

            byte maxEfficiency = 0;
            nuint bestMask = 0;
            bool foundNonZeroCore = false;

            byte* ptr = (byte*)buffer;
            byte* end = ptr + len;

            while (ptr < end)
            {
                // SYSTEM_LOGICAL_PROCESSOR_INFORMATION_EX header
                int relationship = *(int*)ptr;
                uint size = *(uint*)(ptr + 4);

                if (relationship == 0) // RelationProcessorCore
                {
                    // PROCESSOR_RELATIONSHIP struct layout (x64/x86):
                    // Flags (1 byte) @ offset 8
                    // EfficiencyClass (1 byte) @ offset 9
                    // GroupCount (2 bytes) @ offset 30
                    // GroupMasks array @ offset 32
                    
                    byte efficiencyClass = *(ptr + 9);
                    
                    // GROUP_AFFINITY struct layout:
                    // Mask (nuint) @ offset 0
                    // Group (ushort) @ offset sizeof(nuint)
                    nuint groupMask = *(nuint*)(ptr + 32);

                    // Extract just the FIRST logical processor of this physical core (ignore HT sibling)
                    nuint isolatedThreadMask = groupMask & (nuint)(-(nint)groupMask); // Isolate lowest set bit

                    bool isCore0 = (isolatedThreadMask & 1) != 0;

                    // We want the highest efficiency class. 
                    // If tied, we prefer avoiding Core 0.
                    if (efficiencyClass > maxEfficiency || 
                       (efficiencyClass == maxEfficiency && !foundNonZeroCore && !isCore0))
                    {
                        maxEfficiency = efficiencyClass;
                        bestMask = isolatedThreadMask;
                        if (!isCore0) foundNonZeroCore = true;
                    }
                }
                ptr += size;
            }
            return bestMask;
        }
        finally
        {
            Marshal.FreeHGlobal(buffer);
        }
    }
}
```

### Other things that actually matter for this profile:
1. **Windows Power Throttling / EcoQoS:** Ensure Windows hasn't silently dumped your console app into Efficiency mode. Elevating the ProcessPriority usually mitigates this, but you can strictly prevent it by calling `SetProcessInformation` with `ProcessPowerThrottling` to disable `EXECUTION_SPEED` throttling.
2. **Code Alignment:** .NET 11 still suffers from random 32-byte loop alignment variance. If your tight loop happens to cross a cache-line boundary differently between Build A and Build B (because of unrelated code changes shifting the method in memory), you'll see a 1-3% drift. You cannot easily force method alignment in C#, but be aware of it when doing A/B testing. If Build B is mysteriously slower, swap the execution order of A and B, or run them in separate processes.
3. **Timer Resolution:** You don't need `timeBeginPeriod(1)`. `Stopwatch.GetTimestamp()` uses the High Precision Event Timer (HPET) or Invariant TSC (Time Stamp Counter). It runs at 10MHz natively and is totally immune to OS tick resolution.
