Here are the exact, copy-pasteable `AMDuProfCLI` 5.3 commands for your Windows 11 / Zen 2 / .NET environment. 

**Prerequisite:** You **must** run your command prompt or PowerShell as **Administrator** for IBS, PMU, and ETW (managed symbol) collection to work.

To make the commands readable, I will use a variable for your launch string. In PowerShell:
`$APP = "dotnet C:\path\AprVisual.S1.dll --benchmark rom.nes --bench-hc 400000 --extra-ram --system-def-dir DIR"`

---

### 1. List Available Profiling Options
To see what your Zen 2 CPU supports and the predefined configurations:
```powershell
# List predefined profile configs (e.g., assess, ebp, timechart)
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" info --list-profiles

# List all raw PMU events available on Zen 2 (Family 17h)
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" info --list-events
```

### 2. COLLECT using IBS (Instruction-Based Sampling)
For IBS, the predefined config is **`assess`**. This automatically uses IBS Op sampling (which captures load/store latencies and data sources). 
*Note: `--jnc` is required to capture .NET JIT symbols via ETW.*

```powershell
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" collect --config assess --ibs-op-maxcnt 100000 --jnc --launch $APP -o C:\uProf_IBS
```
* **`--config assess`**: Enables IBS Op and Fetch sampling.
* **`--ibs-op-maxcnt 100000`**: Sets the IBS Op sampling interval (lower = more samples, 100,000 is a good high-resolution baseline for a 5-10s run).
* **`--jnc`**: Enables Java/.NET/CoreCLR ETW providers to resolve JIT symbols.
* **`--launch`**: Automatically limits profiling to the launched process and its children.

### 3. COLLECT using Core PMU Events (EBP)
For Event-Based Profiling (EBP), we use `--config ebp` and specify the raw Zen 2 (Family 17h) hex events.
* `0x76` = CPU Cycles
* `0xC0` = Retired Instructions
* `0x41` = L1 Data Cache Misses
* `0x87, umask=0x02` = Dispatch Stalls (Backend Bound)

```powershell
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" collect --config ebp --event event=0x76,umask=0x00,user=1,os=0 --event event=0xc0,umask=0x00,user=1,os=0 --event event=0x41,umask=0x00,user=1,os=0 --event event=0x87,umask=0x02,user=1,os=0 --jnc --launch $APP -o C:\uProf_EBP
```

### 4. .NET / Managed Symbol Resolution
As included in the commands above, the exact flag is **`--jnc`**. 
**Crucial .NET Tip for short runs:** Because your app runs for only 5-10 seconds, ETW symbol rundown might miss methods if Tiered Compilation recompiles them mid-run. It is highly recommended to force RyuJIT to compile at max optimization immediately by setting this environment variable before running the `collect` command:
```powershell
$env:DOTNET_TieredCompilation="0"
```

### 5. REPORT: Generate CSV / Text Reports
Reports are generated from the `.cxperf` file created in your output directory.

**A. Function-level Hotspot Report (CSV):**
```powershell
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" report -i C:\uProf_IBS\AMDuProf-assess-*.cxperf --report detail --format csv -o C:\uProf_IBS\Reports
```

**B. Instruction/Assembly-level Report (CSV):**
To get the per-instruction breakdown (which you need to see the exact `movzx` instruction), generate an ASM report. You should filter it to your specific JIT'd module or function to avoid massive files.
```powershell
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" report -i C:\uProf_IBS\AMDuProf-assess-*.cxperf --report asm --func "YourHotMethodName" --format csv -o C:\uProf_IBS\Reports
```

### 6. Reading IBS Output for Load Latency
Open the generated ASM CSV report. Because you used the `assess` (IBS) config, the report will contain specific IBS columns next to the disassembled instructions. 

Locate your `movzx reg, byte ptr [reg + reg]` instruction. Look at these specific columns:
* **`IbsOpDcLoadLat`** (Data Cache Load Latency): This is your golden metric. It shows the average latency (in cycles) from when the load was dispatched to when the data was returned. If this is high (e.g., >4-5 cycles for L1), you are seeing dependency stalls or bank conflicts.
* **`IbsOpDcAcc`**: Number of times this instruction accessed the Data Cache.
* **`IbsOpDcMiss`**: Number of times it missed L1. (You expect this to be near 0 since you said it's L1-resident).
* **`Data Source` / `IbsOpDcMissLat`**: If it *did* miss, this shows where it fetched from (L2, L3, DRAM) and the miss latency.

### 7. BONUS: Bypassing the CLI Pager
The `AMDuProfCLI` pager uses a Windows console API check (`isatty`) to see if it's printing directly to a screen. You can disable the interactive "Press any key..." prompt by redirecting standard output to a file or piping it to a non-interactive stream.

**Option A (Dump to file - Best for automation):**
```powershell
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" collect --help > uProf_Collect_Help.txt
```

**Option B (Pipe to `cat` or `Out-String` in PowerShell):**
```powershell
"C:\Program Files\AMD\AMDuProf\bin\AMDuProfCLI.exe" collect --help | Out-String
```
This tricks the CLI into thinking it's running headlessly, completely bypassing the pager hang.
