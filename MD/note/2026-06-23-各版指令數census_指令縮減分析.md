# 各版 instruction census — 指令縮減分析(2026-06-23)

## 目的
用 **retired instructions**(執行的原生指令數)當衡量,看歷代 early-return + 預計算(剪枝家族 / B1 pair / tagged gate 等)在**同樣 step 數**下到底把每個 half-cycle 的工作量減少了多少。retired instructions 在「同 binary + 確定控制流」下是**確定值**(不受頻率 / cache / 漂移影響),所以是「邏輯有沒有簡化」最純粹的指標。

## 方法
- 平台:**Raspberry Pi 5(Cortex-A76)、Linux perf**。instruction count 跨 ISA 不可比,只在同平台跨版比趨勢。
- 量法:**相減法** —— 每版各跑 `--bench-hc 100000` 與 `400000`,`perf stat -e instructions` 取兩者相減 → 純 **300k hc 迴圈指令**,消除 startup + load(compose netlist + power-on settle)的一次性 overhead。
- `DOTNET_TieredCompilation=0`(一開始就 FullOpts,消除 tier-up 重編譯干擾)。
- `per_hc = (instr_400k − instr_100k) / 300000`。
- 每版徹底 clean + build retry(net11→net10 跨版切換的 obj 殘留會 CS2012);早期 net10 版在 `sudo perf` 下需補 `DOTNET_ROLL_FORWARD=Major` 才能 roll-forward 到 net11 runtime,否則 benchmark 不會跑(只量到 startup)。
- 全版皆 bit-exact(checksum `0x9174E19D961CB6E5` @400k --extra-ram)。

## 重點
1. **總縮減 −49.9% —— 每 half-cycle 的執行指令數整整砍了一半**(127,023 → 63,579)。early-return + 預計算這條路確實把工作量減半。
2. **砍最兇的是早期演算法剪枝**:記憶體佈局(−16%)、P-1(−11.8%)、P-2~4 事件剪枝(連兩刀 −10.5% / −10.7%)、B1 pair path(−6.6%)。後期(06.18+)是精修(每刀 1~3%)。
3. **.NET 11 不是靠減指令**:06.09(net10)→ 06.09b(net11)指令數幾乎不變(76,085 → 76,225)。官網標的 .NET 11 +1% 來自 runtime/JIT 的 cycles/IPC 改善,**不是**減少指令。印證「instruction count 量邏輯簡化、cycles 量執行效率」是兩件事。
4. **本 session 兩個 win 也現形**:06.20 → 06.23 −1.4%(−897 指令/hc),tagged gate list 的「省 per-transistor range check + all-unsafe 少讀一次 NodeStates」直接反映在指令減少。
