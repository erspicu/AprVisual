# Thread 實驗:CPU‖PPU 雙執行緒切分(分支 thread-experiment)

> 目的:**學術驗證**。實作失敗也有價值 —— 量到「為什麼失敗、失敗多少」就是論文邊界地圖
> 那章的素材。本分支不求合併回 main,求一份可引用的數據與結論。

## 背景與假設

單核 ~136K hc/s,離即時 316×。過去 4 次平行化全死(per-chip per-wave 15×、bit-parallel
156×、per-chip rayon 15×、codegen batch 3–6×),共同死因=**細粒度同步成本**。

2026-06-13 的邊界量測(`MD/note/2026-06-13-CPU-PPU雙執行緒切分_可行性實測.md`)顯示:粗粒度
功能切分(2A03‖2C02 當兩個 PDES 行程,只在匯流排事件同步)是唯一「碰到不立刻死」的方向 ——
真正的 CPU↔PPU 交握僅 ~0.007/hc,切點窄(~13 條線),可位元精確。**但工作嚴重偏 PPU
(PPU 每 hc 滿載、CPU 只 12.5% hc 動、cpu-only=0),預測 2-way 加速上限僅 ~1.12x。**

**主假設 H1**:即便耦合稀疏、可位元精確,2-thread 切分的實測也會 ≤1x(同步/喚醒/快取一致性
成本 > 受限於負載不平衡的微薄理論增益)。本實驗要量出實際倍率與成本分解。

**對立假設 H0**:某種粗粒度設計能達 >1.05x 位元精確 → 意外正向,值得追。

## 階段

- **S1 精確分割**:flood-fill 把每個節點歸 {CPU, PPU, cut, neutral};驗證 cut 恰為耦合集、
  兩側除 cut 外通道不相連。輸出精確兩側工作平衡(補足上次無名節點的缺口)。
- **S2 Oracle 上限(真實計時)**:單執行緒下,每 hc 分別計時兩側 settle,算
  T_parallel = Σ max(t_cpu, t_ppu) + 同步成本模型 vs T_serial。給出帶真實 ns 的加速上限。
- **S3 真 2-thread 原型**:per-wave barrier lockstep(CPU thread + PPU thread,每波屏障交換
  cut)。位元精確閘(golden checksum)。量實際 hc/s + 成本分解。預期:slowdown(負結果)。
  **核心綁定(使用者指定 2026-06-13):兩條 thread 各鎖定不同的最佳 P-core —— 主/CPU 線程
  一顆、PPU 線程另一顆(避開 E-core、避開同一實體核的 SMT 雙線程),用既有 --pin 的選核邏輯
  挑兩顆,High priority + EcoQoS-off。乾淨量測用。**
- **S4(選配)**:hc 粗粒度樂觀執行 + rollback(使用者原構想的最忠實版本)。

## S1 結果(2026-06-13,SMB 3M hc,flood-fill 分割)

**Cut 完整性:contested = 0** —— flood-fill 後沒有節點同時被兩晶片觸及 → CPU/PPU 兩側除了
14 條 cut 線外通道完全不相連,**PDES 切分結構乾淨**(S3 能把每節點明確分某 thread)。
- 節點數:cpu 3,361 / ppu 7,671 / cut 14 / contested 0 / 中性(時鐘樹等)3,682

**工作平衡(pop):cpu-side 16.9% / ppu-side 54.3% / shared(時鐘樹,不可切)28.8%。**
**加速上限:1.84x(shared 免費,不可能)~ 1.20x(shared 兩側都跑)→ 實際 ≈ 1.20x。**
(比上次 ~1.12x 精確:flood-fill 正確把較多內部節點歸 CPU 側,cpu 10.5%→16.9%。)

工具:`BuildPartition()`(WireCore.Recalc.cs,DEBUG),輸出 `[cpu-ppu-boundary]` 的
partition NODES / PDES side split 兩行。

## S2/A 結果(2026-06-13):真實雙核屏障成本 + 效能模型

`--thread-bench N`(Sim/ThreadExp.cs,Release):兩條 thread 各釘一顆最高 EfficiencyClass
P-core(本機 core 14 / 12),ping-pong 量「每波跨核握手 round-trip」:

**實測 round-trip ≈ 115–157 ns(中位 ~134,不鎖頻有變異)。** 純同步、未含共用 NodeStates
cache line 的一致性流量 → 真實每波等效成本只會更高。

模型(serial 135K hc/s = 7.41 µs/hc、work-ceiling 1.20x、12.06 waves/hc):
| 方案 | T_par | 倍率 |
|---|---|---|
| 平行工作(零同步,不可達) | 6.17 µs/hc | 1.20x |
| **naive(每波都屏障)** | 6.17 + ~1.6 = ~7.7 µs | **~0.96x SLOWDOWN** |
| **smart(只在 CPU 有活動的 12.5% hc 同步)** | 6.17 + ~0.2 = ~6.4 µs | **~1.16x(模型樂觀值)** |

門檻:>1x 需 barrier < 102 ns(naive)/ 819 ns(smart)。實測 ~134ns → naive 必輸、smart 有戲。

**A 階段結論:把判決從「鐵定變慢」推到「刀鋒上」。** naive 每波屏障穩定 slowdown(印證
attempt #1);但「跳過 CPU 閒置 hc 的屏障」模型樂觀預測 ~1.16x。模型忽略了:(i) 共用
NodeStates 跨核一致性流量、(ii) group 暫存 thread-local 化的 per-pop 成本、(iii) both-hc 內
波數可能更多、skip 偵測成本。這些只會往下侵蝕。**所以值得做 B(真實作)來解這個刀鋒** ——
看現實落在 >1x 還是被這些成本拉回 <1x。

## Lock 機制分析(B 階段,2026-06-13)

熱路徑會被兩 thread 同時碰的共用狀態,逐一看要不要 lock:

| 共用狀態 | 兩 thread 行為 | 要 lock 嗎 |
|---|---|---|
| `NodeStates`(寫) | 兩側群組 disjoint(S1 證 contested=0)→ 各寫各側節點 | **不用**(寫互斥);只有 cache-line false-sharing 的一致性流量(硬體保證正確,純效能) |
| `_groupBuf/_inGroup/_groupCount/_groupFlags`(group 走訪暫存) | 兩側同時各跑一個 BFS 走訪 | **必須 thread-local**(各自一份),但不是 lock —— 分開的緩衝即可 |
| `RecalcListNext/Hash/Count`(下一波清單) | 各 append 各側節點 | **各 thread 一份 next-list**;hash 位元組域 disjoint(A 只設 cpu+cut、B 只設 ppu)→ 不用 lock |
| `NodeInfos/TransistorList/FlagsToState/NodeConnections/PruneMask/PartSide/CutNodeKind` | 只讀 | **不用**,自由共用 |
| 時鐘節點 + handlers + callbacks | 主線程在 hc 邊界寫;波內穩定 | **不用**(波內只讀) |

**關鍵:14 條 cut 線是唯一跨側耦合,且 cut 節點會 gate 兩側電晶體(io_ce/rw 改變會 enqueue
兩側)。處理方式 = 任何含 cut 節點的「波」整波退回主線程序列處理(cut 活動稀疏 ~0.007/hc)。**
如此一來平行波保證 cut-free → 每側 enqueue 只碰自己側 → **沒有跨 thread 寫入 → 完全不需要 mutex。**

**結論:本實驗不需要任何 lock/mutex。唯一的同步是「每波一個 barrier」(無鎖自旋 / sense-reversing)
讓兩 thread 同意一波做完、主線程再 swap+merge 清單並決定下一波是平行還是序列(看有無 cut)。**
外加 worker 啟停的一次性 volatile handshake。這正是 S1 花力氣證 contested=0 的回報 —— 乾淨切分
讓我們用「barrier-only、零 lock」就能位元精確並行。

## S3/B 結果(2026-06-14):真 2-thread 引擎 —— bit-exact 但 0% 可平行

`--thread-run N`(WireCore.ThreadRun.cs):主線程=CPU 側、worker=PPU 側,各釘一顆 P-core
(本機 14/12),共用 hash + NodeStates(寫互斥)、各自 next-list/group 暫存,**零 mutex,只有
每波一個自旋 barrier**。每波只在「io_ce 非啟用 + 無 cut 節點 + 無 neutral 節點 + 兩側都有
工作」時才平行,否則整波序列(保 bit-exact)。

實測 full_palette 400k(對照單執行緒同 binary 133.5K hc/s):
- **位元精確 ✅:checksum = golden 0x9174E19D961CB6E5。**
- **可平行波 = 0%**(serial-coupled 92.5% + serial-empty 7.5%)。退序列主因:**neutral 節點
  85%**、io_ce-active 7%、cut 0.2%。
- **2-thread 118.3K hc/s = 0.89x(慢 11%)** —— 純粹是每波分類 + 全序列(worker 閒置)+
  [ThreadStatic] 的額外開銷,沒有任何平行收益。

**比 Stage A 模型更深刻的負結果**:不是「同步成本 > 微薄增益」,而是 **bit-exact 下可平行的波
= 0**。原因有二、都回到本專案核心命題:
1. **時鐘衍生的共用(neutral)節點瀰漫近 92% 的波** —— shared clock distribution 每 hc 都動,
   且它 gate 兩側。
2. **波內 Gauss-Seidel 順序是 observable semantics** —— 實證:把 neutral 抽到 phase-1 先做
   的 3-phase 變體(想救回平行)立刻破壞 checksum(0xBAA1…≠golden),因為 neutral 節點與
   cpu/ppu 節點之間的相對處理順序會改變後者的群組(neutral 改 gate 狀態)。

**裁決(B 階段)**:CPU‖PPU 切分結構乾淨(contested=0、14 條 cut、零鎖可位元精確),但**一個波
都平行不起來** —— 故 2-thread 必然慢於單執行緒。H1 成立、且比預期更強。要讓它平行,唯一路是
**複製時鐘樹到兩執行緒(各自本地 tick)+ io_ce 耦合 hc 用 rollback 樂觀執行**(= 使用者最初
「兩份 netlist」構想,S4),但 io_ce 耦合 hc + 工作不平衡 + 開銷,模型上限仍 <1.2x,CP 值不值得。
**這是乾淨、可證偽、bit-exact 驗證過的負結果,直接強化「波內順序=語意」的論文論點。**

## S4 速度探針(2026-06-14):per-wave 平行即使放寬 neutral 仍 0.68x

要驗 S4(複製時鐘樹 + io_ce rollback)是否值得做完整版,先用便宜探針(`--thread-probe`)
放寬 neutral 限制(把 neutral 併到 CPU 執行緒、main 按列序處理 cpu+neutral、ppu 在 worker),
量「平行真的跑起來時的速度」:

- 平行波 **0% → 53.3%**(neutral 不再 bail;另 8.2% io_ce/cut 退序列、38.5% 一側空)。
- 速度 **90.1K hc/s = 0.68x**(比 B2 的 0.89x 更慢,遠低於單執行緒 133.5K)。
- checksum 在 full_palette 上湊巧仍 = golden(main 按列序處理 cpu+neutral 保住了 cpu↔neutral
  順序;ppu 獨立)—— 但不保證通用,速度才是重點。

**決定性結論:per-wave barrier 粒度太細。** 即使平行波拉到 53%,每波一個 ~134ns 跨核屏障 ×
每 hc 十幾波,同步成本遠超省下的工作 → 越平行越慢。這就是 attempt #1(慢 15×)的同一道牆,
現在用乾淨切分精確量到。

**唯一可能 >1x 的路 = 粗粒度(每 hc 一個屏障,而非每波)**:兩執行緒在一個 hc 內各自獨立
settle 到靜止、只在 hc 邊界同步。這正是「複製時鐘樹」的正解 —— 但它需要**每執行緒一份獨立
NodeStates**(否則 neutral 的跨執行緒讀寫順序破壞 bit-exact),即把最熱的 NodeStates 陣列
[ThreadStatic] 化(每次存取 TLS 成本)+ io_ce 耦合 hc 的快照/rollback/merge。模型上限 ~1.16x
(Stage A smart),而 [ThreadStatic]-NodeStates + rollback 開銷極可能把這個邊際吃光。

**S4 裁決:per-wave 版實測死(0.68x);粗粒度版是唯一理論可能,但需要「兩份獨立引擎狀態」的
大重構,且開銷極可能讓它仍 <1x。證據強烈不建議投入;若仍要做,那是把整個引擎去靜態化的另一個
專案。** 整個 thread 實驗(S1→A→B1→B2→S4 探針)是一致、可證偽、bit-exact 驗證過的負結果:
**這顆晶片的開關級模擬,在維持 bit-exact 下無法用多執行緒加速 —— 細粒度死於屏障成本,粗粒度
死於工作不平衡(1.2x 天花板)+ 狀態複製開銷。**

## 判定準則

| 結果 | 解讀 |
|---|---|
| >1.05x 且位元精確 | 意外正向 → 追 S4 / 擴大 |
| 位元精確但 <1x | **預期負結果** → 記錄倍率 + 成本分解(同步 vs 閒置 vs 屏障) |
| 無法位元精確 | 語意發現 → 波內 Gauss-Seidel 順序無法跨 thread 保序 |

任一結果都記成數據與 note。失敗不刪分支,留作論文引用。
