# MetalNES — 驗證與測試策略

我們設計文件把「驗證不足」列為三大失敗點之一（`MD/struct/04_風險_驗證_可行性.md`）。MetalNES 怎麼驗證自己？答案分三層，而且每一層都值得抄。

---

## 1. 第一層：子電路單元測試（`chip_tests.cpp`）

用一個小框架（看起來是 Catch / doctest 風格的 `TEST_CASE_METHOD`）對**單一支援晶片**跑 truth-table 測試。`CircuitUnitTest` 基底類別：

```cpp
wire_module_ptr LoadCircuit(const char *defs_name) {
    wire_defs_ptr defs = wire_defs::Load(getSystemDefDir(), defs_name);
    wire_module_ptr wires = WiresCreate();
    wires->addInstance(nullptr, defs, "");
    wires->resetState();
    return wires;
}
class CircuitUnitTest : public wire_handler {
    void step()  { _wires->step(1); }
    void setTrace(std::string nodes_expr) { _log->setTrace(nodes_expr); }
};
```

### 範例：`4021`（控制器的 8-bit 並轉串移位暫存器）

```cpp
TEST_CASE_METHOD(UnitTest_4021, "4021") {
    writeBits(_d, 0); setLow("clk"); setLow("PL"); step(); step();
    REQUIRE(!isNodeHigh("q7"));
    for (auto data : {0xFF, 0xCC, 0x00, 0x12}) {
        writeBits(_d, data);
        setHigh("PL"); /* parallel load */  setLow("PL");
        for (int i = 0; i < 8; i++) {
            REQUIRE(isNodeHigh("q7") == (((data << i) & 0x80) != 0));   // 逐 bit 對照預期
            REQUIRE(isNodeHigh("q6") == (((data << i) & 0x40) != 0));
            REQUIRE(isNodeHigh("q5") == (((data << i) & 0x20) != 0));
            step(); step();   // 一個 shift = 兩個 half-cycle
        }
    }
}
```

### 範例：`pslatch`（一個 latch 電路）

```cpp
TEST_CASE_METHOD(UnitTest_pslatch, "pslatch") {
    setLow("clk"); setLow("d"); setLow("p"); setLow("ps");
    REQUIRE(!isNodeHigh("q"));
    setHigh("d"); step(); REQUIRE(isNodeHigh("q"));   // d=1 → q 跟著
    setLow("d");  step(); REQUIRE(!isNodeHigh("q"));
    setHigh("p"); setHigh("ps"); REQUIRE(isNodeHigh("q"));   // preset
    setHigh("ps"); REQUIRE(!isNodeHigh("q"));                 // ...
}
```

→ **對照設計文件 §13.3**（手刻迷你網表：inverter / NAND / NOR / pass transistor / dynamic latch）。MetalNES 做的就是這個 —— 只是它的「迷你網表」是真實的支援晶片（4021、pslatch、74LS139…）的網表，每個附 truth table。**這比抽象的手刻電路更有用，因為它同時驗證了「parser + graph + evaluator」和「這個晶片的網表本身是對的」。**

## 2. 第二層：nesdev 測試 ROM（`data/roms/nes-test-roms/`）

裝了一整套經典 nesdev 測試 ROM：

- **CPU**：`blargg_nes_cpu_test5`、`instr_test-v3`、`instr_timing`、`instr_misc`、`cpu_dummy_reads`、`cpu_interrupts_v2`、`cpu_exec_space`、`cpu_reset`、`cpu_timing_test6`、`branch_timing_tests`、`nes_instr_test`
- **PPU**：`blargg_ppu_tests`、`ppu_vbl_nmi`、`ppu_open_bus`、`oam_read`、`oam_stress`、`sprite_hit_tests`、`sprite_overflow_tests`、`vbl_nmi_timing`、`scanline`、`full_palette`
- **APU**：`apu_test`、`apu_mixer`（含 `.mp3` 參考錄音！）、`apu_reset`、`blargg_apu_2005.07.30`、`pal_apu_tests`、`dmc_tests`、`dmc_dma_during_read4`、`volume_tests`
- **其他**：`MMC1_A12`、`mmc3_test`、`dpcmletterbox`、`nmi_sync`、`read_joy3`、`PaddleTest3`…（雖然 MetalNES「No MMU support」，但放著）

這些 ROM 是 blargg 等人寫的，會自己跑測試、把結果（pass/fail + 文字訊息）寫到記憶體裡。

## 3. 第三層：自動偵測 test ROM 結果（`handler_nes_system::check_unit_test`）

blargg 的測試 ROM 有個慣例：把結果寫到 `$6000`（卡匣的 work RAM / extra RAM）：
- `$6001-$6003` = magic `0xDE 0xB0 0x61`
- `$6000` = 結果碼（`< 0x80` = 完成；`0x80` = 執行中；`0x81` = 需要 reset）
- `$6004...` = NUL 結尾的文字訊息

```cpp
void check_unit_test() {
    auto eram = _wires->resolveMemory("cart.eram.ram");   // $6000 區
    const uint8_t *data = eram->getDataPtr();
    if (data[1]==0xDE && data[2]==0xB0 && data[3]==0x61) {   // magic 對上
        if (!_unit_test_found) { log_printf("[nes-test] unit-test signature found\n"); _unit_test_found = true; }
        int result = data[0];
        if (result < 127) {                                  // 測試完成
            std::string output;
            for (int i = 0; i < 1024; i++) { char c = data[0x0004+i]; if (!c) break; if (c=='\r') continue; output += c; }
            onUnitTestComplete(result, output);              // 印結果、存圖（.test.png）、存音訊（.test.wav）
        }
    }
}
```

`check_unit_test()` 每幀（`onFrameEnd`）呼叫一次。`system_state::shouldQuit()` 會在 test 完成時回 true → 自動退出。所以 MetalNES 可以**批次跑整套測試 ROM、自動收集結果**（含畫面截圖、音訊錄音）。

→ **對照設計文件 §13.5**（對照參考模型 / 已知 reset/fetch/opcode 序列）。MetalNES 用 blargg ROM + 自動結果偵測達成「跑得對嗎」的高層驗證。`apu_mixer` 還附了 `.mp3` 參考錄音，等於連音訊都有 golden reference。

## 4. 輔助：trace log（`handler_log` + `logger.cpp` + `wire_gui.cpp`）

`setTrace("cpu.clk0,cpu.ab[],cpu.db[],cpu.rw,prg.a,prg.d,...")` 可以指定一串節點/register 當「欄位」，每個 trace 點 dump 一行（像邏輯分析儀）。`logger.cpp` 是環形緩衝。`system.cpp` 裡有一大堆註解掉的 `_tracedColumns` 範例，顯示開發時是怎麼盯著特定訊號 debug 的（PPU palette bus、控制器移位、74LS139 解碼、卡匣位址解碼…）。

→ **對照設計文件 §13** 整章：每層驗證都要留下 trace 證據。MetalNES 有完整的 trace 基建。

## 5. 對我們的啟示

| 設計文件主張 | MetalNES 怎麼做 | 我們該抄的 |
|--------------|-----------------|------------|
| §13.3 手刻迷你網表測試 | `chip_tests.cpp`：4021 / pslatch 等支援晶片的 truth-table 測試 | 用真實小晶片（或 6502 的小區域）當測試案例，每個附預期 |
| §13.5 對照參考模型 | blargg test ROM + `$6000` 簽章自動偵測 | 直接用同一套 nesdev test ROM；6502 階段可對 Perfect6502 |
| §13.6 CPU/CUDA 等價 | ❌ MetalNES 沒做（沒有 CUDA backend） | **這是我們要補的** —— 同一份 IR 跑兩個 backend，node-by-node 比對 |
| §13 trace 證據 | `handler_log` + 環形緩衝 + GUI 欄位選擇 | 我們的 evaluator 也要有 trace dump（設計文件附錄 B 的 `cpu_eval_trace.json`） |
| 靠命名節點讀內部狀態 | `cpu.a[7:0]` 等直接讀 | 善用 Visual2A03 已命名好的暫存器節點，每幀 dump 對照 |

**一句話**：MetalNES 的驗證策略是「子電路 truth-table（低層）+ blargg test ROM 自動偵測（高層）+ trace log（隨時）」。我們可以**直接複用 blargg ROM 和 `$6000` 簽章機制**（甚至直接用 MetalNES 的 `data/roms/`），並在它的基礎上**加一層它沒有的「CPU evaluator vs CUDA backend 節點對節點等價」**——那才是我們設計文件主張的、MetalNES 還沒走到的驗證層級。
