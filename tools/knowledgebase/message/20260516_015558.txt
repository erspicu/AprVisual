## 1. S1 目前的核心演算法（baseline、要替換 / 增強的對象）

**每半週期的工作流程**（`WireCore.Recalc.cs` + `.Group.cs`）：

```
StepCycle:
  RunHandlerChain()                              # clk toggle、memory/video callbacks
  ProcessQueue():                                # BFS over dirty-set
    while RecalcListNext non-empty:
      swap current/next queue (double-buffered)
      for each nn in current queue:
        if RecalcHash[nn] != 0:                  # dirty flag #1（佇列 dedup）
          RecalcHash[nn] = 0
          RecalcNode(nn):
            ComputeNodeGroup(nn):
              clear _inGroup of previous group   # O(prev group size)
              AddNodeToGroup(nn) (recursive):
                if _inGroup[nn] != 0: return     # dirty flag #2 / early return（群 dedup）
                _inGroup[nn] = 1
                _groupBuf[_groupCount++] = nn
                _groupFlags |= NodeInfos[nn].Flags
                # walk channels to normal nodes / GND / VCC:
                for each transistor t in TlistC1c2s[nn]:
                  (gate, other) = t
                  if NodeStates[gate] != 0:      # transistor conducts iff its gate is high (NMOS)
                    AddNodeToGroup(other)
                for each t in TlistC1gnd[nn]: if NodeStates[gate]: _groupFlags |= Gnd
                for each t in TlistC1pwr[nn]: if NodeStates[gate]: _groupFlags |= Pwr
              newState = FlagsToState[_groupFlags & 0xFF]   # 256-entry LUT
            for each m in _groupBuf:
              SetNodeState(m, newState):
                if NodeStates[m] == newState: return        # early return #1（最關鍵！）
                NodeStates[m] = newState
                for each t in TlistGates[m]:                # m gates these transistors
                  enqueue both channel endpoints of t       # gate-fanout enqueue
  InvokeCallbacks()                              # memory writes、video frame writes
  Time++
```

**關鍵速度來源**：
- **Dirty-flag**（兩個 O(1) dedup hash：`RecalcHash` 佇列 dedup、`_inGroup` 群 dedup）
- **Early return**（三個 —— 最關鍵的是 `SetNodeState` 的「值沒變、不傳播」 —— 確保 dirty-set 只在「值真的變了」的地方擴張、不雪崩）

- **Unmanaged hot path**：`byte* NodeStates` / `int* TransistorList` / `NodeInfo* NodeInfos` / 各 `int*` 佇列、零配置零 bounds-check

### 候選 A：Union-Find（with path compression + union by rank）替代 BFS group walk

**怎麼做**：每半週期、不做 DFS。而是維護一個 Union-Find 結構，當一個 transistor 切換導通狀態時、做對應的 `Union(c1, c2)` 或 `分離`（這後者 UF 不支援、要 Link-Cut Tree 或 Euler-tour）。Group 解析時 `Find(nn)` O(α(N))。

**我的複雜度估算**：
- 純 incremental Union（不支援刪邊）：O((D + M_conducting_changed) · α(N))/半週期
- 但 conduction 會*兩向*變（gate 改變既加邊也刪邊）→ 需要 Link-Cut Tree、O(log N) per op
- 對比 S1：S1 每 RecalcNode 重做 DFS、O(group size)；對 group size > log N (~14) 時 LCT 贏

**贏面**：算法上同階、cache 可能好一點。**輸面**：實作複雜（LCT）；group size *通常很小*（~1），那 α(N) ≈ log N ≈ 14、可能輸 S1 的 O(1)。

### 候選 B：稀疏 boolean 矩陣 SpMV-style「frontier expansion」

**怎麼做**：把當下「導通的 transistor channel 圖」表示成 N×N 稀疏 boolean 矩陣 A（CSR）。一個 group BFS = 從 dirty bitvector 出發、反覆 `frontier' = A · frontier`（boolean SpMV）直到 fixed point。所有 dirty group 可以*同時*expand（一次 SpMV、所有 group 平行進展一步）。state 用 bit-packed 表示（N/64 個 ulong）→ SpMV 變 bitwise OR + AND 的 SIMD 操作。

**我的複雜度估算**：
- 每次 SpMV：O(nnz(A) / 64) ≈ O(M / 64) ≈ ~420 個 AVX2/SIMD 字 / half-cycle
- 收斂 iteration：O(max group diameter) ≈ O(log G_max) ≈ 3-5 次
- 半週期總 cost：~2K SIMD ops vs S1 的 ~10K scalar transistor 檢查 ≈ 算上 SIMD 寬度（AVX2 = 4× ulong64）大概 *似乎* 贏 5-10×

**贏面**：SIMD 友好、cache 友好（CSR 順序存取）、bit-packing 省 32× 記憶體。**輸面**：(1) 每半週期 A 都要*重建*（哪些 transistor 導通取決於當下 NodeStates），rebuild cost 可能吃掉 SpMV 省的；(2) bit-packed state 跟原本 byte state 之間要轉換；(3) 「dirty-set 只算變的」這個 invariant 在 SpMV 下變成 `frontier 是 sparse vector`、稀疏 SpMV 對 GPU 友好但對 CPU 純 SIMD 不見得。

### 3. Bit-packing 與 AVX2 Layout (針對不規則 Degree)

如果你非得試試看矩陣/SIMD路線，絕對**不能用 CSR**。CSR 導致每個 Row length 不同，SIMD lane 會發散（Divergence），且不利於 Vectorization。

最佳的 Layout 是 **SELL-C-$\sigma$ (Sliced ELLPACK)**：
1.  將所有 Node 依照 degree 排序。
2.  將相近 degree 的 Node 切成 Chunk（例如 $C=32$ 或 $64$ 配合 AVX2/AVX-512）。
3.  在 Chunk 內部 Padding 到該 Chunk 的最大 degree。
4.  因為 NES 晶片 max degree 才 30+，Padding 浪費極小。

**但致命傷依然存在**：即便你完美 pack 了矩陣，SpMV 的 `x` (NodeStates) 是隨機存取的。AVX2 必須用 `VPGATHERDD` 靠 index 去讀取 state，這會成為 Pipeline 的絕對瓶頸。

### 4. Conduction Matrix Rebuild 是不是 Deal-Breaker？

**是，絕對是 Deal-breaker。**

*   **全重建成本**：掃過所有 Transistor (O(M) = 27K) 更新矩陣，以記憶體頻寬估算，大約需要 10µs - 15µs。你原本 S1 整個半週期才 21µs！這直接把天花板砍半。
*   **增量更新 (Incremental) 的幻想**：你說「只更新 Gate 變了的列」。這要求你維護一個反向索引：`Gate -> Controlled Matrix Elements`。當 Gate 改變時，你去修改 Bit-packed 矩陣裡的特定 bit。
    *   這代表你要做：`Read Matrix Word -> Bitwise AND/OR -> Write Back`。
    *   由於 SpMV 矩陣是被轉置 / Packing 過的，這通常意味著 Scatter 寫入。
    *   你等於寫了一個純量 (Scalar) 的複雜演算法，去維護一個矩陣，**只為了讓後面能發射幾發 SIMD 指令**。Amdahl's Law 告訴我們：這個純量維護的 overhead，將成為新的瓶頸。

### 5. 比 H 更好的方向：有沒有完全沒想到的招？

有的。既然 86% 是 Logic gate（單向），只有 14% 是 Pass-Transistor Bus（雙向、大 Group），你應該**打破完全一致的 Graph 視角，採用「異質模擬 (Heterogeneous Simulation)」加上「Oblivious Logic Evaluation」。**

**方向一：Bipartite 分離 (Logic vs. Bus)**
1.  **Build-time**：把網路切成純 Logic Subgraphs (NAND, NOR, Inverter) 和 Pass-Transistor Clusters。
2.  **Runtime**：
    *   對 86% 的 Logic，**放棄 BFS！** 把 Logic 拓樸排序（Topological Sort），然後轉成 **Bitwise Parallel Vector Simulation**。
    *   這叫 **Oblivious Simulation**：不管有沒有變 (dirty)，只要 Input 準備好，我就用 SIMD 暴力算。例如 `State_C = State_A & ~State_B`。利用 AVX2，你一條指令可以同時模擬 256 個 Logic Gates！這完全消滅了 Queue、Hash Hash、Branch Mispredict。
    *   對 14% 的 Bus / 雙向開關，保留你 S1 霸道無比的 BFS Dirty-set 演算法。

**方向二：Vectorized Event Queue (針對你原有的 S1)**
不要搞矩陣。保留你的 Queue，但用 SIMD 去加速 Queue 的處理：
1.  從 Queue 中一次 pop 8 個 dirty nodes。
2.  用 AVX2 `VPGATHERDD` 並發載入它們的 Flags 與 Neighbors。
3.  用 `VPSHUFB` (平行查表) 替代你的 256-entry LUT，一次算出 8 個 new state。
4.  如果有變化，再 enqueue。
*雖然這很吃組語功力，但這保留了 $O(D)$ 的優勢，又利用了 SIMD 的寬度。*

### 6. 整體判斷與天花板估計

**這條 `math-algos` branch 的結論：**
如果你硬要走「把整個電路視為一個大矩陣」的純數學/線性代數路線（候選 B, C, D, H），**這條路是死胡同，效能大概率會倒退（變成 0.5x 到 0.1x）。** 原型寫出來後，你會發現 Profiler 裡滿滿的都是 Array Unpacking、Gather/Scatter Latency 和 Matrix Update Overhead。

**合理的天花板（在不觸碰 IR / Codegen 的前提下）：**
*   目前的 47K hc/s (~21µs/hc) 已經是純量 C# 的頂尖水準。
*   如果實作 **G (Static Graph Layout 重排)**：預計可達 **~55K - 60K hc/s (1.2x - 1.3x)**。
*   如果實作我提議的 **Bipartite 分離 (86% Logic 用 SIMD 暴力算 + 14% Bus 用 BFS)**：極限約落在 **80K - 100K hc/s (約 2x)**。

**為什麼極限是 2x？**
因為記憶體延遲的物理極限就擺在那裡。你的 D=500 決定了你必須做指標追蹤。在 5GHz CPU 上，一次 L3 Cache Miss 就是 40-50ns，相當於數百個 ALU 指令。任何沒有從根本上消滅「狀態讀寫」的演算法（例如 JIT 融合成單一函數，這是你排除了的 main branch 作法），都跨不過這道物理之牆。

**給 User 的建議：**
停止對「全域稀疏矩陣 SpMV」的浪漫幻想。對於 Event-driven、極度稀疏且拓樸不規則的 Switch-level 電路，矩陣是錯誤的抽象。把精力投資在 **Data Layout 重排 (G)** 與 **邏輯閘的批次化/Oblivious SIMD** 上，這才是能帶來量化改善的務實做法。
