### 1. 「EvalExpr 取代 group walk」對 event-driven 模型 fundamentally sound 嗎？

**答案：Fundamentally UNSOUND（根本上不健全），除非有極其嚴格的靜態保證。**

你的直覺非常敏銳，問題就出在**導通群（DC-connected component）的原子性被破壞了**。

*   **S1 (Group Walk) 的物理意義**：當一個 transistor 導通，兩端的 nodes 在物理上就變成了「同一個電氣節點」（共享電荷與驅動力）。S1 的 `ComputeNodeGroup` 保證了這個導通群裡的所有節點會**同時、原子性地**更新為相同的值。
*   **$\beta$ (EvalExpr) 的問題**：`EvalExpr` 是典型的 gate-level 邏輯。它預設了「單向驅動」與「節點孤立性」。當節點 `nn` 透過 pass transistor 連到混合總線（hybrid bus）時，如果只對 `nn` 做 `EvalExpr`，你只更新了 `nn` 的值，卻沒有把電荷/狀態同步給總線上的其他成員。
*   **為什麼靠 enqueue (b) channel-fanout 補救無效？**
    這會引發**暫態危害（Transient Hazard）**。在 S1 中，節點 A 和 B 是「同時」變成 1 的。在 $\beta$ 中，A 先變成 1，把 B 放進 queue；在 B 被處理之前，系統處於一個「A=1, B=0 且 A, B 導通」的**物理不可能狀態**。如果這個瞬間，某個 callback 或其他 fanout 被觸發，就會讀到錯誤的狀態，導致雪崩式的 mismatch（這就是為什麼你的 mismatch 會在幾百個半週期後爆炸）。

**能用 EvalExpr 走快路的嚴格條件是：**
該節點必須是**完全孤立的 CMOS 邏輯閘輸出（Static Gate Output）**。也就是說，它的 channel 端點**只連接到 VCC/GND**（透過 pull-up/down network），**絕對不包含**任何 OwnerDrives == null 的 pass transistor，也絕對不會在任何條件下與其他狀態未知的節點形成導通群。一旦有條件導通的 pass pass，就必須退回 Group Walk。

---

### 2. $\beta$ 正確需要哪些 fanout？

如果硬要修補 $\beta$（非常不建議），除了你加的 (a) `irFanout` 和 (b) channel-fanout 之外，你還面臨一個無底洞：

在 switch-level 中，信號傳播有兩種：
1.  **Topological Change（拓樸改變）**：Gate 端改變，導致開關切換，這會改變導通群的形狀。這時需要 S1 的 `gate-fanout`。
2.  **Value Injection（值注入）**：Channel 端的 Source 改變，透過**已經導通**的 transistor 傳遞給 Drain。這時需要 channel-fanout。

**你缺的是「Transitive Channel Propagation」（傳遞性通道廣播）。**
如果 A 改變了，且 A 連著導通的 T1 到 B，B 又連著導通的 T2 到 C... 你不能只 enqueue B。你必須 enqueue B、C 以及整個導通群。而要找出這個導通群需要什麼？**這不就是 S1 的 Group Walk 嗎！**

所以，企圖用單點 EvalExpr + 精細的 fanout enqueue 來模擬 switch-level 的導通行為，最終必然會演化成「重新發明了一次充滿 Bug 的 Group Walk」。

---

### 3. 有沒有更乾淨的 formulation？

如果你的目標只是「讓 S1 變快一點」，你提的 **(b) IR 當「memoization」** 是相對最乾淨且安全的做法。

*   **做法 (b) 的細節**：
    保留 S1 的 `RecalcNode`（不破壞 event-driven 和 group walk 的原子性）。但是，對於 IR-covered 且是純單向驅動的節點，建立一個 `LastInputs[]` 快取。當 `RecalcNode` 被喚醒時，先檢查其 `NextExpr` 所依賴的 input nodes 是否與上次相同。
    *   如果相同：直接 return（Skip group walk）。
    *   如果不同：執行原本的 S1 `ComputeNodeGroup`。
*   **優點**：絕對保證正確性（沒有破壞 group walk 的原子性），且濾掉了無效的 event 觸發。
*   **缺點**：維護 input 快取的 Overhead 可能不小，整體加速上限極低（大概頂多 1.2x ~ 1.5x）。

### 4. $\beta$ 值得做嗎？（關鍵結論）

**極度不建議繼續投入 $\beta$。請立刻收尾止血。**

我們來算一筆務實的帳：
1.  **效能目標差距過大**：你的目標是 real-time（~16ms/幀），現在是 ~15000ms/幀。你需要的是 **900 倍**的加速。
2.  **$\beta$ 的天花板極低**：就算 $\beta$ 完美實作，沒有 bug。因為你還有大量 hybrid node (SCC, bus) 必須跑 S1，且 event-queue 本身的 overhead 佔比很大。把 1-node group walk 換成 tree eval，撐死只能給你 **2 到 3 倍**的加速（降到 ~5000ms/幀）。這對「能玩」這個 user goal 沒有任何實質意義。
3.  **複雜度與除錯成本失控**：為了一個 3x 的加速，你要處理 switch-level 與 boolean-level 之間極難搞的邊界狀態（如上述的暫態危害）。這在工程上是 ROI（投資報酬率）極低的行為。

### 下一步的明確建議（Pivot Direction）

既然你的大目標是「單台 correct + real-time-usable」，你需要的是**架構上的降維打擊**，而不是微觀的 runtime 最佳化。

1.  **放棄「Event-Driven 混合靜態 IR」**：
    Switch-level 模擬器要跑到 real-time，歷史上的成功案例（像 Verilator 的做法）都是把整個系統「完全攤平編譯成靜態的 C/C++ 程式碼」。
2.  **擁抱 S4 的 Batch Driving，但要解決 Latch 延遲問題**：
    你之前放棄 S4（Batch re-eval）是因為速度比 S1 慢，且全靠 S1 擦屁股。
    如果你能把 **IR 提取的品質提升**（例如：正確識別出 latch 的 mid-phase clock，把 Latch 寫成帶 clock edge trigger 的 IR，而不是依賴 `Hold/Prev` 和 S1 的延遲修補），讓 **100% 的節點都能轉成 IR**（把 SCC 解開成 fixed-point iteration 迴圈）。
    一旦 100% 都是 IR，你就可以直接生成一包純粹的 C# / C++ function，裡面沒有 queue，沒有 dynamic transistor walk，只有純粹的 bitwise 運算。這是唯一能達到 900x 加速（~16ms/幀）的路徑。
3.  **把 S1 當作「開發期」的 Golden Reference**：
    S1 現在 ~15s/幀 且絕對正確。這是一個完美的**離線對照組**。不要試圖加速它了。把它封裝好，用來自動驗證你未來提取的「純靜態 C Code」每一 cycle 的行為是否與物理模型一致。

**總結**：$\beta$ 是一個誘人但錯誤的兔子洞（Rabbit hole）。它試圖在不相容的物理抽象層（電路網 vs 邏輯閘）之間建立動態橋樑。**停掉 $\beta$，保留 S1 作為神主牌，把精力轉移到「如何提升 IR 的提取率到 100% 並直接 AOT 編譯」上。**
