# S2.3 — 回授 / SCC 分析（S2.3a 帶條件依賴圖 + Tarjan；S2.3b 打斷假 SCC + cross-coupled latch 回收 → 序向）

> S2 的第 3 步。S2.2 後我們有：~8875 個非 hybrid node 的 `nextExpr`（5455 組合 + 3420 序向（含 `Hold(self)`））+ **5774 個 hybrid node（`nextExpr == null`）—— 其中 5534 個是「bidirectional pass between two driven nodes」，絕大多數是 register-bit / flag 的 cross-coupled latch**（兩個 cross-coupled inverter + 一個 write pass 到 driven 的 data；S2.1 認不出方向就標 hybrid）。S2.3 要：① 從 `nextExpr` + `DriveInfo` 建**帶條件的依賴圖**、Tarjan 找 SCC、按結構分類；② 把 cross-coupled latch 那一坨從 hybrid **回收成序向**（合成 `nextExpr`、用 `Prev(...)` 打斷環）→ IR 覆蓋率跳上去；剩下打不開的 → 維持 hybrid。**不建 IR 直譯器（S2.4）；不改模擬行為。**

> 工作流程（`00_計畫總覽.md` §3）：① 設計描述 → ② 丟 Gemini → ③ 評估採用 → ④ 實作 → ⑤ 驗證。可分多 firing（①寫設計+丟Gemini ②S2.3a 實作 ③S2.3b 實作 ④收尾+驗證+commit）。
>
> **承接**：node 角色 Supply/Input/Bus/Internal；`DriveInfo`{PullDown, PullUpKind{None,StaticLoad,StrongVcc,Conditional}, PullUpCond, Passes{Other,Cond,`bool? ## ① 設計描述

### 1.0 執行順序（採用 Gemini 的重排 —— pattern 回收先於建圖）

**範疇（採用 Gemini Q3）：本輪 S2.3 只做最簡單的 2-node cross-coupled latch + write pass 的回收**（5534 個「bidirectional pass between two driven nodes」hybrid 裡能符合這 pattern 的）—— 那幾千個處理掉，覆蓋率就從 ~60% 衝到 ~85-90%+。**SR latch（6502 裡多半是中斷 flip-flop，~幾十個）、multi-write、dynamic latch with feedback —— 暫緩**（不符 2-node 的維持 hybrid 跑 S1，對效能拖累微乎其微）。**做完 2-node 回收跑 `--dump-scc`，若覆蓋率 >85% → 收工進 S2.4，不在 corner case 泥淖打轉。**

階段（順序重排）：
- **Stage A — pattern 靜態回收（建圖前）**：直接遍歷所有 node，用 §1.2 的 heuristic 找 `CrossCoupledLatch` → 改 `NextExpr[Q] = Mux(we, Node(data), Prev(Q))`、`IsSequential[Q] = true`、從 hybrid 剔除 Q（另一半 `R` = `Not(Node(Q))` 維持組合）。理由：這些 latch 結構極固定（兩 node 互依），不用 Tarjan 就找得到；提早回收大幅減少建圖/Tarjan 的 node 數。
- **Stage B — 建帶條件的依賴圖**：對剩下的（非 hybrid + 剛轉序向的）建有向圖；`Prev(X)` / `Hold(X)` 不產生邊。
- **Stage C — Tarjan 找 SCC**：找到的環 = 「Stage A 漏掉的未知結構」或「cross-phase 假環」。
- **Stage D — SCC 條件分析 + 打斷（cross-phase 假環）**：走訪環上每條邊的啟用條件；若一條含 `NodeRef(phi1)`、另一條含 `NodeRef(phi2)` 且 `(phi1, phi2)` 在 Mutex 名單 → 假環 → 挑一顆 clock-gated node 把引用換 `Prev(...)`、標序向、打斷。**phi1/phi2 用「Mutex Pair 名單」硬寫死 node id/name**（從 nodenames 找 `clk0`/`clk1` 等），不從 AST 推（脆弱）。**—— 但若 Stage A 後覆蓋率已 >85%，Stage D 可先跳過**（那些 SCC 維持 hybrid，少數而已）。
- **Stage E — 無解 SCC → hybrid**：剩下既不是 Mutex 假環、也解不開的 SCC（複雜組合回授 / bus 結構，典型在 2C02 OAM/sprite bus）→ 把該 SCC 內所有 node `Hybrid = true`，丟回 S1 的 WireCore（保留 hybrid fallback 的最大價值 —— bus fight 靠 S1 的群組解析 + GND/VCC/pull-up 優先鏈處理）。

### 1.1 S2.3a — 帶條件的依賴圖（Conditional Dependency Graph）+ Tarjan SCC

**為什麼不建純靜態圖**（Gemini 的 S2-總覽 review 已採用）：在 NMOS 網表上對純靜態圖跑 Tarjan 會得到「跨多個 latch」的巨大無意義 SCC —— 因為 pass transistor 在硬體上常雙向 / 串成環。要建**帶條件的依賴圖**：

- **node 集合**：所有 `nextExpr[N] != null` 的 node（含 §1.2 回收進來的 cross-coupled latch）。hybrid 的（回收不了的）不進圖（它們走 switch-level）。
- **邊 + 條件**：`nextExpr[N]` 裡每個 `NodeRef(M)` → 一條 `N → M` 的邊，**邊上帶這個引用的「啟用條件」**（在 `nextExpr` 的 AST 裡，這個 `NodeRef(M)` 出現在哪個 `Mux` 的哪一分支下 → 條件 = 那些 Mux cond 的 AND/它們的 Not 的組合；最外層無條件 → `True`）。`Hold(M)` / `Prev(M)` **不產生組合邊**（那是「上一輪的值」，跨 cycle）。
- **Tarjan SCC**（對組合邊；忽略邊條件先找 SCC，再用條件判哪些是假環）。size-1 無自環 → 不算 SCC（純組合 node）；size-1 有自環（`nextExpr[N]` 引用 `NodeRef(N)` —— 不是 `Hold(N)` 的話 → 真的自己引用自己 → 怪，可能是 ratioed feedback → 標 hybrid 或當 latch）；size≥2 → SCC。

### 1.2 cross-coupled latch 回收（從 hybrid → 序向）

S2.1 把「bidirectional pass between two driven nodes」標 hybrid。其中最常見的是 **2-node cross-coupled latch + write pass**：node `Q` 有下拉（條件 `pd_Q`，通常 = `NodeRef(/Q)` 或一個含 `/Q` 的式子）、static pull-up、一個 pass 到 `data`（gated by `we`）；對稱地 `/Q` 有下拉 `pd_/Q`（含 `Q`）、static pull-up（可能沒有 write pass，或有第二個 pass）。

**辨識 pattern**（保守 —— 認不出的維持 hybrid）：
- `Q` 是 hybrid、`HybridReason` 含 "bidirectional pass between two driven nodes"；
- `Q` 的 hybrid 來源 pass 連到 `data`、gated by `we`；`data` 是「有 driver 的真 node」（`HasDriver(di[data])`，非 hybrid 或就算 hybrid 也有值）；
- `Q` 有下拉 `pd_Q`，且 `pd_Q` 引用某個 node `R`（= `/Q` 的角色），而 `R` 的下拉 `pd_R` 引用 `Q` → 確認是 2-node cross-couple；
- `Q` 有 static pull-up（`PullUpKind.StaticLoad`/`StrongVcc`）—— 沒有的話可能是 dynamic latch，另一種 pattern。

**合成**（把 `Q` 當「state node」—— 它有 write port）：
- `nextExpr[Q] = Mux(we, Node(data), Not(prev_pd_Q))`，其中 `prev_pd_Q` = `pd_Q` 但裡面的 `NodeRef(R)`（= `/Q`）換成 `Prev(R)`（打斷環）。化簡：若 `pd_Q == NodeRef(R)`（最常見：Q 的下拉就是 /Q 一個 gate）→ `Not(Prev(R)) = Prev(Q)`（因 `R = !Q` 恆成立）→ **`nextExpr[Q] = Mux(we, Node(data), Prev(Q))`** = 標準 D-latch。
- `R`（= `/Q`）：它沒 write port → 當組合，跟原本 S2.2 給的一樣（`nextExpr[R] = Not(pd_R)`，`pd_R` 引用 `Node(Q)` —— 讀 Q 的現值）。但 `R` 在 S2.1 也可能被標 hybrid（如果 `R` 也有個 bidirectional pass）—— 那 `R` 也要回收（對稱）。
- `Q` 標序向（`IsSequential[Q] = true`）；`R` 標組合。
- 把 `Q` 從 hybrid 名單拿掉（`di[Q].Hybrid = false`）；`nextExpr[Q]` 設成上面的。

**其他 pattern**（先列、firing 看狀況做）：
- **dynamic latch with feedback**（沒 static pull-up，但有個 feedback inverter 維持值 + write pass）—— 類似，`nextExpr[Q] = Mux(we, Node(data), Prev(Q))`。
- **set/reset latch**（有兩個 input pass，set 和 reset）—— `nextExpr[Q] = Mux(set, True, Mux(reset, False, Prev(Q)))` 之類。
- **多 write port**（一個 reg bit 被多個來源寫，各自的 we）—— `Mux(we1, data1, Mux(we2, data2, Prev(Q)))`（one-hot 假設，跟 S2.2 一樣）。
- 認不出的 / >2-node 強耦合 / bus-style 回授 → **維持 hybrid**，記 reason。

### 1.3 S2.3b — 打斷剩下的真 SCC

回收完 cross-coupled latch 後，重建依賴圖，再 Tarjan。剩下的 SCC：
- **cross-phase pipeline 假環**：環上有一條邊的條件需要 `clk0`（或含 `clk0` 的 AND）、另一條需要 `clk1`，而 `clk0`/`clk1` 互斥 → 這個環在邏輯上不存在（它是個跨 phase 的 pipeline，不是同一拍的組合回授）→ **打斷**：把環上「在 phase 邊界更新」的那些 node 標序向，環內引用換 `Prev(...)`。判定「`clk0`/`clk1` 互斥」：先找出 `clk0`/`clk1`（或它們的對應 node）—— 從 nodenames 或 from 「`clk1 = !clk0`」這種結構推；保守起見也可以只認「直接是 `clk0` 或 `Not(clk0)` / `clk1`」的條件。
- **同 phase async loop（RS latch、真回授）**：環裡沒有 clock 互斥 → 真的是個 latch / 回授結構。若能認出「state node + set/reset/data input」→ 改寫成序向（同 §1.2 的精神）；認不出 → **維持 hybrid**（整個 SCC 的 node 都標 hybrid）。
- **bus-style 回授 / >2 強耦合**（典型在 2C02 的 OAM bus / sprite bus）→ 維持 hybrid（沒 strength 維度，bus-fight 解不了）。

### 1.6 開放問題（丟 Gemini）

1. cross-coupled latch 的辨識 pattern（§1.2）—— 6502/2C02 的 register bit 實際長這樣嗎？「state node = 有 write pass 的那個」這個 heuristic 對嗎（會不會兩個都有 write pass、或 write port 不是 pass 而是某種 gate）？2C02 的 VRAM-addr / scroll / OAM-addr 那些 latch 結構一樣嗎？
2. 合成 `nextExpr[Q] = Mux(we, Node(data), Prev(Q))` 對嗎？ratioed write（`data` 強驅動、weak pull-up + feedback inverter 輸給它）—— 我的 IR 沒有 strength 維度，這樣模型化（write 時 `data` 直接覆蓋）對嗎？6502 裡有沒有「write 但不是 ratioed、而是先把 feedback 切斷再寫」的 latch（那種更乾淨，但 pattern 不同）？
3. set/reset latch / 多 write port / dynamic latch with feedback —— 6502/2C02 上常見嗎？該不該 S2.3 就做，還是先只做最簡單的 2-node cross-couple、其他維持 hybrid（等覆蓋率不夠再補）？
4. 「cross-phase pipeline 假環」的判定（環上有邊需 clk0、有邊需 clk1，clk0/clk1 互斥）—— 怎麼穩健地識別 `clk0`/`clk1`？從 nodenames（`clk0`/`clk1` 直接有名字）？從「`clk1 = Not(clk0)`」的結構？2A03/2C02 的兩相時鐘實際是怎麼產生的（一個外部 `clk` → 內部分相？）—— 這影響我怎麼找它們。
5. 剩餘的 bus-style / >2 強耦合 SCC 維持 hybrid —— 對嗎？還是有辦法把 2C02 的 OAM bus 那種也抽出來（例如把整個 bus 當一個「多 driver 的 wired node」、`nextExpr` = `Mux` 鏈 over 各 driver 的 enable）？

每條建議標明「對這個計畫的具體影響 / 該怎麼改」。某設計 OK 不用改也直說。

## ② Gemini 分析回饋

> `gemini-3.1-pro-preview-customtools`，log：`tools/knowledgebase/message/20260512_111240.txt`。摘要 —— **方向務實精準、大膽放行**：
- **Q1/Q2 register-bit 結構 & 寫入**：6502/2C02 的靜態暫存器（A/X/Y/PC…）**絕大多數就是 ratioed-write 的 cross-coupled inverter pair** —— 沒有 break-before-make；寫入端（pass + driver）的尺寸/驅動力被設計得比回授 inverter 大很多，`we` 開時外部訊號直接蓋過弱回授。**`nextExpr[Q] = Mux(we, Node(data), Prev(Q))` 完全正確** —— 不用引入 strength 維度（硬體靠物理尺寸、IR 靠 Mux 分支優先級）。**state node = 「有 write pass 的那個」，heuristic 穩健，直接採用。** 2C02 特例：VRAM-addr(v/t)、sprite shift reg 多半是 **dynamic latch**（無 static pull-up、靠寄生電容 hold、過一個 inverter 輸出）—— S2.2 可能根本沒標 hybrid（單向 pass 寫入的話只會是 `Hold(Q)`）；若被誤標 hybrid，用 dynamic-latch-with-feedback pattern 抽（暫緩）。
- **Q3 做到哪個程度**：S2 的核心目標是「大幅提覆蓋率換效能」不是「100% 抹除 hybrid」。5534 個 bidirectional pass 佔 hybrid 的 >95%。**先只做最簡單的 2-node cross-coupled + write pass** → 覆蓋率從 60% 衝到 85-90%+。**暫緩 SR latch（6502 裡多是中斷 flip-flop，~幾十個）、multi-write、dynamic latch** —— 不符 2-node 的維持 hybrid 跑 S1，對整體效能拖累微乎其微。**做完 2-node 回收跑 `--dump-scc`，>85% 就收工進 S2.4，不在 corner case 泥淖打轉。**
- **Q4 cross-phase 假環 & 時鐘識別**：6502 的兩相時鐘 phi1/phi2 是外部 `CLK0` 經內部分相產生的**非重疊**時鐘 → `phi1 AND phi2 == False` 恆成立。**不要從 AST 硬推 Mutex（脆弱，中間可能過好幾個 gate）；用「Mutex Pair 名單」—— 直接寫死關鍵 clock node 的 id/name。** 打斷：Tarjan 找出 SCC 後走訪環上邊的啟用條件，若邊 A 含 `NodeRef(phi1)`、邊 B 含 `NodeRef(phi2)` 且 `(phi1,phi2)` 在名單 → 假環 → 挑一顆 clock-gated node 引用換 `Prev(...)`、標序向。
- **Q5 bus-style / >2 強耦合 SCC**：**絕對維持 hybrid** —— 2C02 的 OAM bus / sprite bus 是 wired-OR/AND 或複雜 priority 總線，沒精細 strength 模型硬轉布林極易出錯（sprite-zero-hit timing 容易爛）。S1 的群組解析 + GND/VCC/pull-up 優先鏈完美處理 bus fight —— 這正是保留 hybrid fallback 的最大價值。Tarjan 找出的 SCC 不是認識的 latch pattern → 整個 SCC 的 node 標回 hybrid。
- **Q6 流程微調 —— 重排（採用）**：Stage A pattern 靜態回收（建圖**前**，不用 Tarjan）→ B 建帶條件依賴圖（`Prev`/`Hold` 不產生邊）→ C Tarjan → D SCC 條件分析 + 打斷 cross-phase 假環（用 Mutex 名單）→ E 無解 SCC → hybrid。提早回收大幅減少建圖/Tarjan 的 node 數。
