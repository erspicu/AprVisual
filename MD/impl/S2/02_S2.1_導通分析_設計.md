# S2.1 — 導通分析 / `DriveInfo`（每個 node 的「驅動結構」）

> S2 的第 1 步：用 S2.0 的 `NetlistGraph`（node 角色 + transistor kind + pass 方向欄位），算出每個非 `Input`/`Supply` node 的「驅動結構」`DriveInfo` —— 它被拉到 GND 的布林條件、它的 pull-up 種類、透過 pass 連到哪些別的信號 node（含方向）。**還不組 `Expr` 的 next-state（那是 S2.2）；還不碰 SCC（S2.3）；不改模擬行為。**
>
> 工作流程（`00_計畫總覽.md` §3）：① 設計描述（本段）→ ② 丟 Gemini 分析 → ③ 評估 & 採用 → ④ 實作 → ⑤ 驗證。
>
> **承接 S2.0 學到的**：depletion load 在這份網表裡**不是電晶體、是 `node.Pullups > 0` 標記**（→ static pull-up 看 `Pullups`，不要去 `TransistorKind` 找 `PullUpLoad`）；`PullUpStrong` 很少（5 顆，= S1.5 留的「綁到 vcc」connection）；`PullUpActive` ≈ 897 顆（真正「條件拉到 vcc」的電晶體 = precharge / push-pull / 動態邏輯）；`PullDown` ≈ 16126（直接到 GND 的葉子）、`Pass` ≈ 9747（transmission gate + 下拉鏈中段）。`Input` = 72（含 handler hook node / reset 網路 / 板上腳 / gate-only 內部 node —— 這些 S2.1 直接跳過、不算 `DriveInfo`）。

## ① 設計描述

### 1.1 目標 & 輸出

對每個 node `N`（`role ∈ {Internal, Bus}`，跳過 `Supply` / `Input`）算 `DriveInfo[N]`：

```csharp
sealed class DriveInfo
{
    public Expr? PullDown;          // 「N 被拉到 GND」的布林條件（沿 PullDown+Pass 子圖做串並聯歸約）；
                                    //   null = 無下拉路徑；ComplexExpr = 歸不出乾淨式子（→ S2.2 標 hybrid）
    public PullUpKind PullUp;        // None / StaticLoad（node.Pullups>0）/ StrongVcc（有 PullUpStrong 電晶體）
                                    //   / Conditional（有 PullUpActive 電晶體 → 條件式；Expr = OR 那些 gate）
    public Expr? PullUpCond;         // 只有 PullUp==Conditional 時有意義
    public List<PassLink> Passes;    // 透過 Pass 電晶體連到的別的信號 node
    public bool Hybrid;              // 這個 node 的驅動結構太亂、抽不出乾淨式子 → 整個 node 留 switch-level
    public string? HybridReason;     // debug 用
}
enum PullUpKind { None, StaticLoad, StrongVcc, Conditional }
struct PassLink { public int Other; public Expr Cond; public PassDir Dir; }   // Dir 從 NetlistGraph.PassDirection 來（這步會把 Unknown 推導掉）
```
`Expr` 型別本身在 S2.2 才正式定（這步先用一個最小子集：`ConstExpr(bool)` / `NodeRefExpr(int)` / `AndExpr` / `OrExpr`；`ComplexExpr` 當 sentinel「歸不出來」）。S2.1 不引入 `NotExpr` / `MuxExpr` / `HoldExpr`（那些在 S2.2 組 next-state 時才用）。

### 1.2 `PullDown` 的算法 —— 串並聯歸約 + 路徑展開 fallback

「N 被拉到 GND」= 在「目前哪些電晶體 ON」這個 symbolic 設定下，存在一條從 N 到 `vss` 的導通路徑。把相關電晶體當邊、gate signal 當邊的「啟用條件」：

- **相關電晶體** = N 的 `C1c2s` 裡 kind 為 `PullDown` 或 `Pass` 的（`PullDown` 的另一端是 `vss`；`Pass` 的另一端是別的信號 node —— 可能要遞迴往那邊找到 `vss`）。
- **遞迴邊界（採用 Gemini §7-Q2 的修正 —— 用「佈局物理特徵」而不是「有沒有 pull-up」）**：往 `Pass` 的另一端 `M` 走時，**只有當「`M` 不是 `Bus` 且 `M` 從來不驅動任何電晶體的 gate（`WireCore.Nodes[M].Gates.Count == 0`）」才繼續**。理由：下拉鏈的中間 node 純粹是電晶體串聯的物理節點，**絕不會去接任何電晶體的 gate**；反之，transmission-gate / mux 的輸出是個真正的邏輯訊號，**一定會去接某個 gate（fan-out > 0）**。所以 `M.Gates.Count > 0` ⟺ `M` 是獨立的邏輯 node ⟺ 那個 `Pass` 是 mux / TG → 歸到 `Passes`、停止這條路徑往 `vss` 的搜尋。這個判定**不會被 precharge node / bootstrap 騙倒**（原本「看 M 有沒有 pull-up」會把「有 PullUpActive 預充、再串聯下拉求值」的動態 NAND/NOR 腰斬）。
- **串並聯歸約**：從 N 開始 DFS 到 `vss`，把走過的子圖做 series-parallel reduction：
  - 一條鏈（每個中間 node 度數 = 2、只在這條鏈上）→ 鏈上所有電晶體 gate 的 **AND**。
  - 同一對端點之間的多條鏈 → **OR**。
  - 遞迴歸約。`gate == Npwr` 的電晶體 → 該邊條件恆 `Const(true)`（在 AND 裡可省、在 OR 裡讓整個 OR 變 true）。
- **非串並聯（橋接）fallback**：若子圖歸不成純串並聯（Wheatstone-bridge 式結構）→ 退成「枚舉所有 N→vss 的簡單路徑，每條路徑 = 路徑上 gate 的 AND，全部 OR 起來」。**路徑數上限 = 32**（Gemini §7-Q1 採用：原本想用 64，下修到 32 —— 一個 node 到 GND 若 >32 條路徑，99% 是 Q2 的 heuristic 判錯、把一坨 mux/pass logic 當下拉網路展開了；提早報 `Complex`+`Hybrid` 反而幫我抓出分類錯誤）；超過 → `PullDown = ComplexExpr`、`Hybrid = true`、`HybridReason = "pulldown net not series-parallel and >32 paths (likely a mis-classified pass network)"`。
  - 註：6502/2C02 的下拉網路 99.9% 是乾淨的串並聯（NMOS AOI/OAI）；NMOS 裡的「橋接」結構通常是 pass-transistor logic（ALU 進位鏈 / barrel shifter / BCD 校正），不是直通 GND 的下拉網路。

### 1.3 `PullUp` 的判定

對 node `N`：
- `node.Pullups > 0` → `PullUp = StaticLoad`（depletion load 標記；弱拉高，輸給任何下拉）。
- N 的 `C1c2s` 裡有 kind == `PullUpStrong` 的電晶體（gate==Npwr、另一端 vcc）→ `PullUp = StrongVcc`（無條件接 VCC —— 但因 NMOS 的「GND 贏」優先鏈，行為上跟 StaticLoad 一樣是「下拉導通則 0、否則 1」，差別只在 bus-fight strength，那種情況反正 hybrid）。
- N 的 `C1c2s` 裡有 kind == `PullUpActive` 的電晶體（gate 是信號）→ `PullUp = Conditional`、`PullUpCond = OR(那些電晶體的 gate)`（precharge / 動態 / push-pull 上半）。
- 以上同時成立的優先序：`StaticLoad` / `StrongVcc` 蓋過 `Conditional`（有靜態上拉的話，conditional 那個是多餘的；實際上一個 node 通常只有一種）。都沒有 → `PullUp = None`（純 dynamic / pass 驅動的 node）。

### 1.4 `Passes` + pass 方向推導（Gemini §7-2 採用）

對 N 的每顆「不算進下拉網路」的 `Pass` 電晶體（另一端 `M`、gate `g`）：加 `PassLink { Other = M, Cond = NodeRef(g), Dir = ? }`。方向 `Dir`：
- N 有下拉網路（`PullDown != null && != Complex`）或有 pull-up，而 `M` 是「純 dynamic」（無下拉、無 pull-up、`role` 非 `Bus`）→ `Dir = AtoB`（A = 有 driver 那端 = N → 驅動 M）。從 M 的角度看同一顆 → `Dir = BtoA`（一致）。
- 兩端都有 driver（或都是 `Bus`）→ `Dir = Bidirectional`、那個 pass 網路 → `Hybrid = true`（兩端的 node 都標 hybrid），`HybridReason = "bidirectional pass between two driven nodes"`。
- 兩端都「純 dynamic」（charge sharing）→ `Dir = Bidirectional`、`Hybrid = true`、`HybridReason = "pass between two floating dynamic nodes (charge sharing)"`（Gemini §7-5：6502 真依賴的場景極少，先警告 + hybrid，看 test ROM）。
- 一個 `Bus` node 連到一個普通 node 的 pass：方向看普通 node 那邊有沒有 driver；通常 `Bus` 是「外部 + 內部都驅動」→ 偏 `Bidirectional` → hybrid（呼應 S2.0 把 `cpu.db`/`ppu.io_db` 標 `Bus` 的理由）。

> 這步順便把 `NetlistGraph.PassDirection[]` 從 `Unknown` 填掉（推不出的留 `Bidirectional`）。

### 1.5 `BusResolver.RefineBuses()`（S2.0 留的待辦，這步做掉）

用 `(N 有內部 driver：`PullDown != null` 或 `PullUp != None` 或被 `AtoB` 的 pass 驅動) ∩ (N 被某個行為級 handler `WriteBits`)` 把 `NetlistGraph.Role` 裡更多 node 標成 `Bus`。「被 handler `WriteBits`」的清單從哪來：`WireCore.Handlers.cs` 的 `AttachRamLikeHandler` / `AttachVideoHandler` 解析出來的 `_d[7:0]` / `d[]` / dataOut node list —— 這些目前沒存起來，這步要在那邊加個 `HandlerWrittenNodes` 集合（或讓 `AttachMemoryHandlers` 回傳 / 累積）。被新標成 `Bus` 的 node → 連帶 `DriveInfo.Hybrid = true`（雙向 bus 先 hybrid，呼應 §1.4 + Gemini §7-... bus-fight 沒 strength 維度）。

### 1.6 程式落點

- 新檔 `src/AprVisual/Sim/Logic/Expr.cs`：`abstract record Expr` + `ConstExpr` / `NodeRefExpr` / `NotExpr` / `AndExpr` / `OrExpr` / `MuxExpr` / `HoldExpr` / `PrevExpr` + `ComplexExpr`（sentinel）。簡單的 smart constructor（`Expr.And(a,b)` 做 `a && true ⇒ a` 之類的 trivial 化簡 —— 不是 S3 的完整 CSE，只是別讓樹太醜）。**S2.1 只用 `Const`/`NodeRef`/`And`/`Or`/`Complex`**；其餘給 S2.2+。
- 新檔 `src/AprVisual/Sim/Logic/DriveAnalysis.cs`：`DriveInfo`、`PullUpKind`、`PassLink` + `DriveAnalysis.Analyze(NetlistGraph g) → DriveInfo?[]`（indexed by node id；`Supply`/`Input` 的是 null）。內含串並聯歸約 + 路徑展開 fallback + pass 方向推導。
- `BusResolver.RefineBuses(NetlistGraph g, DriveInfo?[] di)`（同檔或獨立檔）—— 需要 `WireCore` 那邊提供「被 handler 寫的 node 集合」（小改 `WireCore.Handlers.cs`）。

- `--selftest` 加 `TestDriveAnalysis`（手刻 inverter / NAND / NOR / AOI / 一個 pass-mux（**mux 輸出端故意去接一顆假電晶體的 gate**，才能驗 Gemini §7-Q2 的新 heuristic 有正確把它判成 `Passes` 而不是當成下拉鏈中段）/ 一個 bidirectional pass，驗 `PullDown` 式子、`PullUpKind`、`Passes`/方向、`Hybrid` 標記）。

  - `Hybrid` node 數：希望**不多**（主要是 `Bus`（refine 後 ~16-32：`cpu.db`/`ppu.io_db` + RAM/PPU 內部資料線）+ 少數 bidirectional/charge-sharing pass + 少數 Complex 下拉網）。把前 ~30 個 hybrid node + reason 印出來人工看 —— 若一大票都在 2C02 的某區（OAM / sprite bus）那符合預期；若 CPU 裡冒出一堆要查。
  - **覆蓋率**：`1 - (Hybrid + Complex) / (有 DriveInfo 的 node 數)` —— 目標 ≥ 90%（S2.2~S2.3 之後會更高）。
  - spot-check：某確定是反相器的 `cpu./xxx` node：`DriveInfo.PullDown == NodeRef(它的閘輸入)`、`PullUp == StaticLoad`、`Passes` 空；某 register bit 的兩個 cross-coupled node：彼此在對方的 `PullDown` 式子裡（透過 pass-gated 的 latch 結構 —— 這正是 S2.3 要處理的 SCC，S2.1 只要把結構算出來）。

### 1.8 開放問題（丟 Gemini 一起問）

1. 串並聯歸約 + 路徑展開 fallback 的設計 OK 嗎？6502/2C02 的下拉網路有多少是非串並聯（橋接）的、路徑上限 64 夠不夠、有沒有更好的作法（BDD？直接對下拉子圖做 reachability 的 fixpoint？）？
2. 「pass transistor 是下拉鏈中段 vs 是 transmission-gate mux」的 disambiguation heuristic（往另一端走只有當那端「有到 GND 的下拉、無自己的 pull-up、非 Bus」才繼續）—— 會不會分錯？6502/2C02 裡有沒有「pass transistor 串聯下拉、但中間 node 也有 pull-up」的結構（precharged 的串聯下拉？）？
3. pass 方向推導 heuristic（一端有 driver、另一端純 dynamic → 定向；都有 driver → bidirectional/hybrid；都 dynamic → charge-sharing/hybrid）—— 在 6502/2C02 上會把多少 pass 留成 `Bidirectional`？bootstrap node 的 pass（Gemini 之前說 bootstrap 在 Visual6502 沒顯式元件、看起來像 `PullUpActive`/`Pass`）會怎麼被分？
4. `RefineBuses` 把雙向 bus node 一律標 `Hybrid` —— 太保守嗎？`cpu.db[7:0]` 其實「CPU 寫的時候 CPU 驅動、handler 讀的時候 handler 驅動」，這兩個 phase 互斥 —— 能不能不 hybrid，而是把 db node 的 `nextExpr` 寫成 `Mux(cpu_is_writing, cpu_drives_db, Hold(db))`、handler 那邊當外部 override？還是這太細、先 hybrid 之後再說？
5. `DriveInfo` 的資料結構 / `Expr` 的最小子集（S2.1 只 `Const`/`NodeRef`/`And`/`Or`/`Complex`）—— 合理嗎？有沒有 S2.1 就該有的東西被我留到 S2.2 了（例如下拉網路裡如果有 `gate` 本身是個 inverted 信號 `/xxx` —— 那是個獨立 node、不是 `Not(xxx)`，所以 `NodeRef(/xxx)` 就對；確認一下）？

每條建議標明「對這個計畫的具體影響 / 該怎麼改」。某設計 OK 不用改也直說。

## ② Gemini 分析回饋

> `gemini-3.1-pro-preview-customtools`，log：`tools/knowledgebase/message/20260512_103626.txt`（RESPONSE 全文）。摘要：

- **整體**：S2.1 的邊界切得很漂亮（「只處理靜態拓撲與導通條件，不組 next-state、不解 SCC」）。唯一盲點是 Q2（pass 鏈中段 vs mux 輸出的區分）。
- **Q1（串並聯 vs 路徑展開 vs BDD）**：設計 OK，**保持串並聯 + 枚舉 fallback、不需 BDD**（殺雞用牛刀、難轉回人類可讀 Expr）。非串並聯（橋接）在 6502/2C02 下拉網路**極少**（99.9% 是純串並聯 AOI/OAI）；NMOS 裡的橋接通常是 pass-transistor logic（ALU 進位鏈 / barrel shifter / BCD），不是直通 GND 的下拉。**微調：路徑上限 64 → 16~32**（>16 條路徑 99% 是 Q2 判錯、把一坨 mux/pass logic 當下拉展開；提早報 `Complex`+`Hybrid` 幫抓分類錯誤）。
- **Q2（pass 鏈中段 vs mux 的 disambiguation）**：**原 heuristic（看 M 有無下拉/pull-up/Bus）有危險** —— precharge node（有 `PullUpActive`）串聯 NMOS 求值拉到 GND，若因它有 pull-up 就停住會把動態 NAND/NOR 腰斬。**更好的物理特徵**：下拉鏈中間 node **絕不去驅動任何電晶體的 gate**；transmission-gate/mux 的輸出是真正的邏輯訊號、**一定會去接某個 gate（fan-out > 0）**。→ 遞迴條件改成 `繼續 = (M 不是 Bus) && (M 從不驅動任何 gate，即 WireCore.Nodes[M].Gates.Count == 0)`；M 有驅動 gate → 它是獨立邏輯 node、那個 Pass 是 mux/TG → 歸 `Passes`、停止該路徑往 vss。**不會被 precharge/bootstrap 騙倒**。
- **Q3（pass 方向推導）**：**設計極佳，原封不動保留**。會有一些 pass 留成 `Bidirectional`/`Hybrid` 但量在可接受範圍；真 charge sharing 在 6502 確實存在（主要靠 phase1/phase2 隔離）；bootstrap node 在 Visual6502 通常標 PullUpActive + 某驅動，兩端打架就 Hybrid 最安全。S2.1 的目標不是「完美解析所有方向」，是「把顯然有向的 pass 找出來（90%+）、難的用 Hybrid 框起來」。
- **Q4（雙向 bus 一律 Hybrid 太保守？）**：**以 S2.1 的階段這樣剛好，不要在這做 Mux**。`Mux(cpu_writing, cpu_drives, Hold)` 是 next-state 邏輯（S2.2），甚至牽涉外部 handler 的行為模型。S2.1 的職責是物理拓撲；物理上 `cpu.db` 就是個被多方驅動 + 帶大量 pass 連線的巨大 node。→ 保持 `RefineBuses` 後一律 `Hybrid = true`；**不要在 S2.1 引入 `HoldExpr`/`MuxExpr`**（會破壞「只做靜態導通分析」的純粹性）；handler 覆寫 + bus-fight 解析留給 S2.2/S2.4。現在標 Hybrid 只是讓 DFS 歸約不要跨過 data bus 亂竄。
- **Q5（DriveInfo / Expr 最小子集）**：**精準，沒漏 corner case**。`/xxx` 本身是獨立 node、它的下拉條件就是 `NodeRef(xxx)`；描述「導通條件」本來就不需要 `Not`（`Not` 是電壓翻轉的結果 = S2.2 的事）。維持 S2.1 嚴格只 `Const`/`NodeRef`/`And`/`Or`/`Complex`。
- **補充**：`StaticLoad/StrongVcc 蓋過 Conditional` 正確（NMOS depletion 負載常開 = Static；有 PullUpActive 通常是動態邏輯的 precharge；兩者很少同時存在）。`TestDriveAnalysis` 的 mux node 要故意讓其輸出去接一顆假電晶體的 gate，才能驗新 heuristic。效能：~14600 node 的 DFS+串並聯歸約，C# 跑 < 100ms（O(V+E)），fallback 提早（32 條路徑）截斷就沒問題。
