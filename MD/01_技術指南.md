## 1. 抽象層次：Visual6502 是什麼

Visual6502 不是普通的高階模擬器。它是用解封裝、影像、向量重建出來的**晶片實體拓樸**。

對本專案最關鍵的事實：

- 它描述的是**開關層級（switch-level）網表**
- `Node`：電氣連通的線網（一段相連的金屬／矽）
- `Transistor`：以 gate 節點控制的開／關通道（理想開關，不含類比參數）

抽象層次位於：

- **比 RTL 低**：仍保留電氣節點與開關方向性
- **比 SPICE 高**：沒有實際的類比製程參數

正因為位於這個夾層，它**適合做圖論分析與數位邏輯抽取**，但又不能直接當作門級網表處理。

## 2. 為什麼這個格式不是業界標準反而是優點

Visual6502 不是 Cadence / Synopsys / SPICE / Verilog 任何標準格式。它混合了邏輯資訊、節點 ID、電晶體連通性、以及多邊形座標。

實務上：

- 對傳統 EDA 工具不友善
- 但**非常適合自己寫 parser** 載入記憶體（C# / Python / C++ 都行）
- 比 SPICE 容易解析得多，不必處理一堆製程參數

## 3. 該選哪一份資料？

如果目標是 NES，正確的資料來源是：

| 用途 | 資料集 |
|------|--------|
| CPU | **Visual2A03** |
| PPU | **Visual2C02** |

不是 Visual6502 的標準 6502。理由：

- 2A03 是 NES 改版的 6502，**沒有 decimal mode**
- 2A03 還包含 **APU 與 DMA** 邏輯
- 2C02 是 NES 的 PPU，遠比 CPU 複雜

## 4. 解析器最小輸出

Parser 至少要保留這些欄位：

- node id
- 可選的節點名稱（pin 名稱）
- **`PullType`（`+` / `-`）— 這個絕對不能丟**
- transistor id
- gate / source / drain 的節點 id

可選保留：原始 index、polygon 座標（debug 與視覺化用）。

> ⚠️ `PullType` 是最常被忽略卻又最關鍵的欄位。許多 NMOS 內部沒有顯式 pMOS pull-up，而是靠 depletion-mode load。一旦 `+` 沒被保留，後續模擬會嚴重漂移。

## 5. 記憶體中的圖模型

解析完之後不要繼續操作原始陣列，要立刻轉成顯式的圖物件：

```csharp
public sealed class NetNode
{
    public int Id { get; init; }
    public string? Name { get; set; }
    public char? PullType { get; set; }

    public List<NetTransistor> GatesControlled { get; } = new();   // 控制關係
    public List<NetTransistor> ConnectedChannels { get; } = new(); // 通電關係
}

public sealed class NetTransistor
{
    public int Id { get; init; }
    public NetNode Gate   { get; init; } = null!;
    public NetNode Source { get; init; } = null!;
    public NetNode Drain  { get; init; } = null!;
}

public sealed class NetlistGraph
{
    public Dictionary<int, NetNode> Nodes { get; } = new();
    public List<NetTransistor>      Transistors { get; } = new();
}
```

**必須同時保留兩種不同的關係**，後面所有分析都建立在這個區分上：

| 關係 | 意義 |
|------|------|
| **控制（control）** | 哪一個節點當作哪一顆電晶體的 gate |
| **通電（conduction）** | 當電晶體導通時，哪些節點被短接成同電位 |

## 6. 最大概念陷阱：這不是普通的邏輯樹

對話中反覆撞到的同一個問題是：**switch-level NMOS 邏輯不能當作有方向的布林樹**。

理由：

- Pass transistor 是雙向的
- 有些節點會浮接、用寄生電容保留狀態
- Pull-up 是弱驅動
- Bus 是共享多源，可能爭用
- 回授結構不一定是直觀的 latch

所以下面這個策略**全域上是不安全的**：

> 從輸出節點出發，遞迴往回展開成一棵布林表達式。

它在獨立的 NAND / NOR 區域可行，**在整顆晶片上會出事**。

## 7. 推薦的分析管線

### 7.1 Stage A：靜態圖建構

時間無關的結構：

- 載入所有 nodes / transistors
- 建鄰接表
- 標記固定電源節點：`VCC` / `VSS` / `GND`
- 標記重要外部 pin：`clk0` / `res` / `rw` / `ab0–ab15` / `db0–db7`

### 7.2 Stage B：通電圖分析

對給定的控制條件：

1. 先決定哪些電晶體 ON
2. ON 的電晶體把 source 與 drain 視為連通
3. 計算連通分量

得到三件事：

- 該節點是否連到 GND
- 該節點是否連到 VCC（或 pull-up 高源）
- 該節點是否落在浮島（floating island）

### 7.3 Stage C：辨識「狀態保留」區域

原始材料用 DFS 找浮接節點，方向正確，但要注意兩件事：

- 一個浮接節點**未必**是有意義的記憶體元件
- 真正的儲存元件常常是**整個浮接連通分量**，不是單一節點

更穩健的做法：

1. 找出在某個 phase 與電源網路斷開的連通分量
2. 檢查這些分量在另一個 phase 是被充電還是放電
3. 只把符合條件的分量分類為 dynamic state

過早把單一節點分類為儲存元件，會切壞 bus 島。

### 7.4 Stage D：回授與迴路偵測

原始材料提到三色 DFS、SCC / Tarjan，方向正確。

實務做法：

- 建一張**邏輯依賴圖**
- 把 SCC 收縮成超節點
- 把每個 SCC 分類為以下三種之一：
  - 靜態 cross-coupled storage（兩個 inverter 互鎖）
  - 動態 storage
  - bus-style 回授／共享

> 「遇到灰色節點就停止遞迴」可以避免無限展開，但**不保證語意正確**。

## 8. NMOS 求值規則（核心）

最重要的一條規則。對話中後期才完整成形的版本：

```text
if connected_to_gnd:
    next = 0
else if connected_to_vcc_or_pullup:
    next = 1
else:
    next = previous   # ← 這就是 dynamic node 行為
```

幾個關鍵觀念：

### 8.1 Pull-up 整合

`PullType == '+'` **不代表「永遠是 1」**，它代表「沒有被主動拉低時會回到高」。

所以「`next = !pulldown`」這條只有在某些靜態 gate 形狀下安全，**對 dynamic 區域與 bus 結構並不普遍成立**。

### 8.2 GND wins

「強路徑到 GND 一律贏過弱 pull-up」對 ratioed NMOS 通常合理，但**這是工程簡化、不是類比定律**。真實電路還有：

- weak-high vs strong-low 競爭
- threshold loss
- 充電時間常數
- charge sharing

對 cycle-accurate 數位語意夠用，對 analog fidelity 不夠。

### 8.3 完整的求值器要有三條 query

僅靠 `FindPathsToGnd` 不夠。完整版至少需要：

- `FindPathsToGnd`
- `FindPathsToHigh` / `FindPathsToVcc`
- `IsFloatingIsland` (回傳 previous)

少了任何一個，dynamic 傳輸行為都會被誤分類。

## 9. 布林表達式抽取

### 9.1 安全可靠的起點

從規則良好的區域開始：

- 標準反相器
- NAND / NOR
- 簡單串聯／並聯下拉網路
- 明顯的 precharge / evaluate 時鐘區

**不要從這些開始**：

- 跨多個浮島的共享 bus
- 強耦合動態邏輯叢集
- 還沒有 high-path 分類的區域

### 9.2 不要直接生字串，要先有 IR

原始材料常常直接拼 C++ / CUDA 字串，這對 demo 可以，對長期維護不行。建議先建 IR：

```csharp
public abstract record Expr;
public record ConstExpr(bool Value)              : Expr;
public record NodeRefExpr(int NodeId)            : Expr;
public record NotExpr(Expr Inner)                : Expr;
public record AndExpr(IReadOnlyList<Expr> Terms) : Expr;
public record OrExpr (IReadOnlyList<Expr> Terms) : Expr;
public record MuxExpr(Expr Sel, Expr T, Expr F)  : Expr;
public record HoldExpr(int NodeId)               : Expr;  // 對應 dynamic node
```

好處：

- 可以化簡（`A && true ⇒ A` 等）
- 可以 emit 多種 backend
- 可以在多個 backend 之間互相比對

進階化簡（BDD / AIG / SAT 等價驗證）以後再說，**不要一開始就上**。

### 9.3 串並聯規則

對於良性 NMOS 區域：

- 電晶體串聯 → AND
- 電晶體並聯 → OR
- 有導通路徑到 GND → 下拉為 0
- 沒有下拉路徑且有 pull-up → 回升為 1

這就是把實體 switch network 翻譯成可編譯數學式的方式。

## 10. CUDA Backend：晚一點再寫

CUDA **不該**是第一個驗證環境。先在 CPU 上把模型驗對，CUDA 才是吞吐量後端。

正確順序：

1. CPU evaluator
2. 在手刻迷你網表上驗證
3. 在真實小區域上驗證
4. 才 emit CUDA
5. CPU 與 CUDA 結果**節點對節點**比對

否則一次要 debug 解析、邏輯抽取、時序、記憶體布局、branch divergence — 五樣同時，會卡死。

### 10.1 記憶體布局：SoA 勝過 AoS

```text
state[node_id][instance_id]   ← 推薦
state[instance_id][node_id]   ← 不推薦
```

理由：一個 warp 通常是更新「同一個 node 在多個 instance」，SoA 才能 coalesced。

### 10.2 不要用 `bool*` 當儲存

`bool` 對齊與封裝差、頻寬利用差。實務上應該用：

- `uint32_t` bit-slicing：1 個 word = 32 個 instance
- 或 `uint64_t`：1 個 word = 64 個 instance

把 `&&`、`||`、`!` 換成 `&`、`|`、`~`，一條表達式就同時算 32 / 64 個 NES instance。

### 10.3 一個 thread 跑一整個 NES 比較安全

把 CPU / PPU / APU 拆給不同 thread 看似漂亮，實際上會出問題：

- 共享 RAM 同步成本高
- branch divergence 變嚴重
- data race 變難 debug

第一版**一個 thread（或一個 warp）負責一個完整 instance**，profile 之後才談切割。

### 10.4 別一開始就做「整顆 NES 常駐 GPU」

把 NES 全部塞進 GPU、host 只送 input 收 framebuffer，是有道理的研究方向，但是**最後階段**：

- 互動延遲差
- debug 困難
- watchdog 與 scheduler 可能搗亂

漸進式分期：

- Phase 1：每次 launch 推進固定數量 half-cycle
- Phase 2：回傳小型 trace snapshot
- Phase 3：考慮長駐 kernel

## 11. 時脈、穩態與週期精確

### 11.1 「週期精確」不等於「一次布林展開」

Switch-level 網路有些區域要**多次 settle / relaxation** 才會穩定，因為通電與回授不會在一次內就解開。

實作建議：

```text
while changed and iteration < max_settle:
    evaluate_all_nodes()
```

不要把「跑三次」寫死。某些區域一次就穩，某些要更多次。**設上限 + 超過要 log warning**。

### 11.2 兩相時鐘

6502 / 2A03 用的是非重疊兩相時鐘。執行系統需要：

- `current_state`
- `next_state`
- 每個 half-cycle 跑一次完整 evaluation
- phase 切換時做 buffer swap

`clk0` 通常是外部輸入；`phi1` / `phi2` 是內部產生。**實際 pin 名稱要去資料集裡核對，不能憑文字描述假設。**

### 11.3 Reset 不等於「假裝資料 bus 讀到 NOP」

這是原始材料中最危險的捷徑之一。

- 6502 / 2A03 在 reset 期間**會初始化內部狀態**
- 然後從 reset vector 取指
- 假裝 bus 讀到 `0xEA` 只是 bring-up 暫時 stub，**不是正確硬體行為**

註解上必須明寫「這是暫時 stub」，未來的真實版本要實作 reset vector 與真實 bus 行為。

## 12. Bus 整合與外部世界

### 12.1 Bus 必須照 phase 與驅動方向處理

- 在地址穩定的 phase 讀 address bus
- 看 `R/W` 決定外部行為
- 把資料推回 data bus 節點

兩個務實警告：

- Bus 方向與三態行為不是單純覆寫語意
- 內外部 driver 會有 precharge / float / 爭用

第一版的數位簡化：

> 當外部記憶體回應時，假設它是唯一 driver；否則由內部晶片邏輯決定 bus 狀態。

### 12.2 Mapper：先別碰

GPU 整合 mapper state 是很後面的事。穩健順序：

1. 只支援 **NROM**
2. 加上 RAM mirroring
3. 才考慮 **MMC1** / **MMC3**

每多一個 mapper 就多一台 state machine 與一份驗證負擔。

### 12.3 PPU：第二階段才做

務實順序：

1. CPU 2A03
2. Memory bus
3. PPU 2C02

PPU 難在 shift register、sprite evaluation、CHR bus 行為。

## 13. 驗證策略（最容易被低估的部分）

沒有驗證計畫，整個專案會變成「看起來很厲害但其實不對」的系統。驗證必須**分層**。

### 13.1 Parser 驗證
- node / transistor 計數
- 重要 pin 都能找到
- pull marker 沒被丟掉

### 13.2 Graph 驗證
- gate / source / drain mapping 正確
- 鄰接建構正確
- 通電條件下的連通分量看起來合理

### 13.3 手刻迷你網表
- 反相器、NAND、NOR
- 三態 pass transistor
- dynamic latch
- 每個都要有期望真值表 + 期望 phase 行為

### 13.4 真實小區域
- 不要從整顆 CPU 開始
- 選 reset chain、單一 register bit、ALU carry path 一段

### 13.5 對照參考模型
- Perfect6502 風格的實作
- Visual6502 自己跑出來的 trace
- 已知的 reset / fetch / opcode 序列

### 13.6 CPU / CUDA 等價
- 用**同一份 IR**
- 同樣輸入跑兩個 backend
- 結果**節點對節點**完全一致

這是必要條件。否則你無法分辨 bug 是在模型還是在 backend。

## 14. 推薦交付分期

| Phase | 內容 |
|-------|------|
| 0 | 解析 `segdefs.js` / `transdefs.js`，輸出可快取格式 |
| 1 | CPU 參考求值器，只做節點狀態更新，在迷你網表上驗證 |
| 2 | SCC / 連通分量分析、pull-up / float 分類、收斂式 settle |
| 3 | IR + 簡單 C++ / Verilog 輸出 |
| 4 | CUDA bit-sliced 求值器，SoA + uint32/64 |
| 5 | CPU bus 與 reset vector 整合，僅 NROM，驗證 reset / fetch |
| 6 | PPU、整機、AI 大規模平行 |

## 15. 對原始討論的修正與風險警語

下面這些不是反對原始想法，而是**避免實作者照抄脆弱版本**。

### 15.1 「先轉成 FPGA 再用 GPU 模擬 FPGA」通常不是主路線

更好的做法：

- 直接抽出邏輯 / IR
- 從 IR emit CPU / CUDA / Verilog

加一層虛擬 FPGA 通常只增加 routing overhead，無助於核心問題。

### 15.2 「有 pull-up 就回 true」只在窄條件下成立

當：

- 沒有 GND 路徑
- 有 pull-up

通常 → high。但若有：

- 競爭性弱源
- charge sharing
- 未穩定 phase 行為

「直接回 true」就過於樂觀。

### 15.3 「dynamic node 等於 previous state」是工程抽象

對 cycle-level 數位行為大致夠用。對更精細的物理保真度（保留時間、洩漏、charge sharing）不夠。

### 15.4 「整台 NES 常駐 GPU kernel」是終局，不是 MVP

一開始就做這個 = 一個極難 debug 的黑盒。沒有前面分層驗證，這個東西完全不能信。

### 15.5 來源章節有重複

`GPU 內全封裝晶片架構藍圖` 與 `GPU 實作全封裝晶片架構` 內容大致重複。保留其中一份當正規章節即可。

## 16. 給未來實作者的短建議

如果要建一個同類系統，穩健順序**不是「先寫 CUDA」**，而是：

1. 把網表乾淨地解析成 graph
2. 寫含通電 / pull-up / float / 收斂的 CPU evaluator
3. 在手刻迷你電路上驗證
4. 在真實 2A03 小區域驗證
5. **才**做 CUDA bit-slicing

如果上面任何一步沒完成，**不要**跳去做「在 GPU 上同時跑一萬台 NES」。

## 17. 結語

這個專案真正的價值不是「用了很多 GPU」，而是建立一條值得信任的翻譯鏈：

- 矽片連通性
- → 圖模型
- → 邏輯與時序抽象
- → 可驗證的執行 backend

中間模型對的話，CPU / CUDA / Verilog 只是不同輸出格式。
中間模型錯的話，再華麗的 CUDA 也只是更快地算出錯的答案。
