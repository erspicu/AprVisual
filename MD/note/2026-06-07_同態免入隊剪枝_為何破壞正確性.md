# 死路筆記:同態免入隊剪枝(same-state turn-on prune)為何破壞正確性

> 日期:2026-06-07。狀態:**已實測、失敗、已退回。請勿重試天真版本。**
> 相關:演算法說明見 `06_S1引擎演算法_資料結構與程式碼對照.md`;這條來自 2026-06-06 的 Gemini 演算法諮詢(它的點子 3)。

---

## 1. 想法是什麼

事件驅動 settle 裡,當閘極節點 `nn` 變高、它控制的電晶體 `T(c1,c2)` **導通**時,目前的做法是把 `c1` 入隊到下一波(單側入隊,因為 BFS 會穿過導通的 T 到達 c2)。

提出的剪枝:**若此刻 `NodeStates[c1] == NodeStates[c2]`(兩端同值),就連 `c1` 都不入隊** —— 直覺是「把兩個同值的群組接在一起,值不會變,免得多做一次 settle」。

實作位置:`WireCore.Recalc.cs` 的 `SetNodeState` turn-on 分支,改成多讀 `c2`、加守衛 `if (nodeStates[c1] != nodeStates[c2] && ...) enqueue`。改動很小(約 4 行)。

## 2. 實測結果:壞

- **整機 checksum 跑掉**:`0xB60DB145F4C7A44B` ≠ 黃金 `0x794A43ABDF169ADA`(full_palette 300k `--extra-ram`)。
- **畫面全黑**:`--frame-dump` 第 20 格整片黑(每格 PNG 只有 259 bytes = 純色),golden 同格是正常的 NES 全調色盤(~2.7 KB)。
- **那個 +51% 是假的**:速度暴增是因為「跳過了大量必要計算、模擬幾乎停擺」,不是真加速。

## 3. 根本原因:解析函式的「單調性」被 ForceCompute 破壞

剪枝的隱含前提是:**兩個群組各自解析成同一個值 ⇒ 合併後也是那個值**。這等價於要求群組解析函式 `FlagsToStateOf`(`WireCore.Group.cs:42`)在「合併 = 旗標 OR」之下**單調(monotone)**。

解析優先序(高 → 低):`Gnd→0 > Pwr→1 > SetHigh→1 > SetLow→0 > PullUp→1 > State→1 > (無)→0`。

### 一般節點(沒 ForceCompute)→ 單調 → 剪枝正確

合併兩群只會**增加**旗標;結果取「優先序最高的旗標」。合併後最高的旗標,必是原本兩群最高旗標的其中一個。兩群本來都解成同值 `s`,所以合併後仍是 `s`。
→ **合併只會加東西、不會拿掉強驅動**,兩個 1 必合成 1、兩個 0 必合成 0。✅(可嚴格證明。)

### ForceCompute 節點 → 非單調 → 剪枝錯誤

`ForceCompute` 的唯一作用(`Group.cs:44-49`):

```csharp
if ((flags & ForceCompute) != 0 && (flags & Gnd) != 0 && (flags & Pwr) != 0) {
    flags &= ~Gnd;   // 兩條電源軌互相抵消
    flags &= ~Pwr;
}
```

這是整個解析函式裡**唯一會「移除」旗標**的地方。合併若湊齊 `FC+Gnd+Pwr`,反而會**抹掉原本決定結果的最強驅動(Gnd/Pwr)**,讓一個更弱的旗標冒出來主導 → 結果翻盤。單調性就此破壞。❌

## 4. 精確的「能用 / 不能用」條件

| | 條件 | 剪枝 |
|---|---|---|
| **能用** | 合併後的群組**沒有觸發 FC 抵消**;或就算觸發,抵消後的主導旗標解出的值與原本相同 | ✅ bit-exact |
| **不能用** | 合併**新湊齊了 FC+Gnd+Pwr**(把缺的電源軌/FC 補上),抵消後露出不同的主導旗標,使結果改變 | ❌ 翻值 |

一句話:**剪枝只在「解析保持單調」時正確;ForceCompute 是唯一讓解析變非單調的機制。**

### 數值例子(兩個 1 合併成 0)

- G1 = `{Pwr, SetLow}` → Pwr 最高 → **1**
- G2 = `{ForceCompute, Gnd, Pwr, PullUp}` → 自身 FC 抵消 Gnd+Pwr → 剩 `{FC, PullUp}` → **1**
- 兩端都 1 → 剪枝判「免入隊」
- 實際合併 = `{FC, Gnd, Pwr, PullUp, SetLow}` → 抵消 Gnd+Pwr → 剩 `{FC, PullUp, SetLow}` → **SetLow 優先序高於 PullUp → 0**
- 應變 0 卻被跳過 → 錯。(0 合併成 1 也能同樣構造:如 G1=`{FC,Gnd,Pwr,SetLow}`→0、G2=`{Gnd,SetHigh}`→0,合併抵消後 SetHigh 主導→1。)

## 5. 為什麼補不了(便宜的 guard 行不通)

要安全就得保證「合併後的群組不含會觸發抵消的 FC 節點」。但:

1. **觸發抵消的 FC 節點可能是群組的第三個成員**,不是 c1/c2 本身 —— 入隊當下還沒跑 BFS,無從得知整個群組組成。要知道就得先跑 BFS,剪枝即失去意義。
2. 就算預先標記「此節點可能與 FC 同群」,在 **94% 雙向 SCC** 的網表裡,幾乎每個節點透過某種開關組態都能連到 FC 節點 → 標記幾乎全為真 → 剪枝永不觸發 → 零效益。

所以這不是「加個 if 就好」,是**結構性死路**。

## 6. 為什麼是「全黑」而不是小走樣

`ForceCompute` 正是用來模擬「匯流排上同時被推高又推低該如何收斂」的特殊節點(PPU 的 precharge / bus 衝突處理;來源是網表 `.js` 的 `forceCompute` 清單,見 `WireCore.Module.cs:309`)。**2C02 PPU 送像素的內部資料匯流排大量倚賴這類節點** —— 剪枝一旦在這些節點上錯誤跳過合併,整條視訊路徑就崩潰,所以畫面整片產不出來。

## 7. 結論與教訓

- 這是專案第 9 個「**一擾動 floating / tie-break 類解析就破 bit-exact**」的死路(同類前例:lowering 階段 stub-removal、dead-end-skip)。
- 教訓:任何「在事件源頭剪掉傳播」的點子,**必須對群組解析函式的單調性成立**才安全;而本引擎的解析因 ForceCompute 而非單調。
- 也是一個正面訊號:Gemini 給的 3 個演算法級點子,前 2 個本專案早已實作(同波次 BFS 去重、單側入隊),第 3 個在本晶片上會破壞正確性 → **演算法層已接近榨乾**,沒有不犧牲 bit-exact 又能省計算的單核槓桿。
