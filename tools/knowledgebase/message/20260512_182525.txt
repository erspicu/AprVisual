這是一個非常經典的「Switch-level 抽象」撞上「Gate-level/RTL 抽象」的牆。

你的分析非常精準：那 3 個 mismatch 不是偶然，10K 巨型 SCC 更是架構上的強烈警訊。**Bidirectional pass-transistor bus 在物理上是「動態的電路節點整併（Group Merge）與電容/驅動強度決議」，試圖把它壓平到純粹方向性的 `NextExpr` Boolean Logic，注定會產生假性循環（False Combinational Loops）並丟失強度資訊。**

以下針對你的四個問題，給出明確、務實的工程建議：

### 1. 正確的 multi-driver bus 模型是什麼？

**解答：(d) 別的——放棄用靜態 Boolean Logic（Wired-AND）來逼近 bidirectional bus。改用「驅動強度 + 狀態」的分層解析（Strength-Aware Resolution）。**

*   **為什麼 Wired-AND 會錯（0.07% mismatch 的真相）：** 
    你的 `!Node(Other_i)` 條件把「對方是 0」等同於「對方正在強烈輸出 0」。但在 Switch-level 中，`Other_i` 可能只是一根懸空的線（Floating 0），而你的 Bus 是 Floating 1 且寄生電容比它大。物理上，Bus 的 1 會吃掉對方的 0；但你的 Wired-AND 邏輯卻判定 Bus 被拉成了 0。這就是產生 artifact 的根本原因。
*   **正確的模型：** Bus 不是一個邏輯閘，而是一個**「多工器 + 強度比較器」**。
    一個 Bus node 的值不只要傳遞 `0` 或 `1`，還必須帶有「強度（Strength）」。
    Switch-level 的強度通常分 3~4 級：`GND/VCC (Strong)` > `Pull-up (Weak)` > `Capacitance (Hold)` > `High-Z`。
    如果要在 IR 中表達，你需要將 Bus 相關節點的輸出從 `bool` 升級為 `int (2-bit)`（例如：`0=Float0, 1=Float1, 2=Drive0, 3=Drive1`）。但這會讓 IR 複雜度暴增。
*   **結論：** **不要試圖把它變成 `NextExpr`。** Bus 就是 Bus，它在硬體架構中是特殊的互連線，在你的 Compiler 中也應該是特殊的 First-class citizen，而不是普通的 Boolean Node。

### 2. 那個 ~10K-node 巨型 SCC：fixed-K 划算嗎？

**解答：絕對不划算，這是災難。**

*   **為什麼會變 10K？** 因為 Pass-transistor 的雙向性。`CPU_ALU -> CPU_DB <-> Board_DB <-> PPU_IO_DB -> PPU_OAM`。在 Boolean Graph 中，雙向連線會把原本單向的 Logic Cones 全部串成一個有向環（Cycle）。
*   **K 會是多少？** Fixed-K 的 $K$ 值取決於 SCC 內最長的邏輯深度（包含 Feedback path）。ALU 的 carry chain 加上 bus 的 bounce，$K$ 可能高達 20~50。
*   **效能代價：** 10,000 nodes $\times$ 30 (K) $\times$ 2 (半週期) = 每週期 600,000 次 node evaluation。就算在 GPU 上算也是極度浪費算力（且 register 壓力極大）。這違背了你 S2/S3 辛苦做 IR 最佳化（DAG 化）的初衷。

### 3. GPU 那邊的 data-bus 怎麼辦？ (Inline bus-resolve)

**解答：你的直覺完全正確！用「Inline, Branch-free Bus Resolver」切斷 Graph，把它當作執行期的迭代屏障（Iteration Barrier）。**

這是解決 10K SCC 的完美方案。你可以這樣做：

1.  **把 Bus 當作 Cut-points：** 在建構 IR Graph 時，**不要**把 `NextExpr == null` 的 bus 視為 Graph 的一般節點，也不要產生雙向 edge。把它們視為「邊界輸入/輸出」。這樣 10K 巨型 SCC 就會立刻碎裂回 843 個 node / 56 個小 SCC 的健康狀態。
2.  **Codegen 一個獨立的 Bus-Resolve Block：** 對於那 1379 個 bus node，提取出它們的 Pass-neighbours（誰可能連到誰）和 Pull-up/down。
    寫一段 **Branch-free 的 C#/GPU 程式碼** 來模擬 Group Merge。
    *   *Branch-free 技巧：* 利用 Bitwise 運算。
    *   每個 Bus 節點準備兩個變數：`Val` (0/1), `Strength` (0=Hold, 1=Pull, 2=Drive)。
    *   Resolve 邏輯（偽碼）：
        ```c
        // 評估 pass 導通狀態
        int pass_A_en = Eval(Cond_A); 
        // 收集各方強度與值 (bitwise max 或查表)
        int max_str = ...; 
        int final_val = ...;
        ```
3.  **Flat Program 的交錯迭代架構 (Data-flow Ping-Pong)：**
    你的 Main Loop 長這樣：
    ```c
    // 外層：處理 Bus 傳遞 (通常 Bus 延遲收斂極快，K_bus ≈ 2-3)
    for (int b = 0; b < K_BUS; b++) {
        // 1. 評估 Acyclic DAG (Logic cones driven by buses)
        Eval_DAG_Nodes(); 
        
        // 2. 評估 小 SCCs (Fixed-K, K_scc ≈ 3-5)
        for(int s = 0; s < K_SCC; s++) Eval_Small_SCCs(); 
        
        // 3. Inline Branch-free Bus Resolver
        // 根據剛算完的 Node 值，重新決議 Bus 的值
        Resolve_Buses(); 
    }
    ```
*   **優勢：** 
    *   沒有 Warp Divergence（全部都是 bitwise 運算和陣列讀寫）。
    *   10K 巨型 SCC 被打破。
    *   Bus 行為 100% 忠於物理現象（強度與電容決議可以直接寫進 `Resolve_Buses` 裡）。
