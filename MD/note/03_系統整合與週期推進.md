# MetalNES — 系統整合與週期推進

本文件講「電路引擎之外」的東西：怎麼把它變成一台會跑的 NES —— 時鐘怎麼推、reset 怎麼做、RAM/ROM 怎麼接、影像/音訊怎麼出、執行緒怎麼安排。

相關檔案：`system.h/.cpp`、`handler_*.h`、`wire_handler.h`、`wire_thread.h`、`dispatch_queue.h`、`nesrom.cpp`、`raster_device.cpp`、`audio_device.cpp`。

---

## 1. handler 機制：行為掛在電路上

`wire_handler`（`wire_handler.h`）是所有「行為元件」的基底。它提供：

- `resolveNode("name")` / `resolveRegister("a[7:0]")` / `resolveMemory("ram")` — 用名字（自動加 prefix）拿到 handle
- `wire_node_handle` / `wire_register_handle` / `wire_memory_handle` — 包裝過的 handle，可以 `=`/`bool()`/`int()` 直接用，超方便：
  ```cpp
  wire_field(_cs, "cs");                       // wire_node_handle
  wire_register_field(_a, "a[]");              // wire_register_handle
  wire_memory_field(_ram, "ram");              // wire_memory_handle
  // 用起來像普通變數：
  if (!_cs) { int addr = _a; int data = _ram->readByte(addr); _d = data; }
  ```
- `add_handler<T>(prefix, args...)` — new 一個子 handler
- `register_handlers<T>("*func<rom>", args...)` — 用 wildcard 找出所有掛鉤節點，每個都 new 一個 handler
- `add_callback(nodes, fn)` / `add_edge_callback(node, state, fn)` — 註冊「這些節點變動時呼叫 fn」

### callback 的實作（聰明）

`Wires::add_callback(node_ids, fn)`（`wire_module.cpp:681`）：
1. 建一個假節點 `target_node`
2. 對每個 watched node，加一顆電晶體 `(gate=watched_node, c1=target_node, c2=gnd)`
3. 把 `fn` 掛在 `target_node` 上

→ 任何 watched node 變動 → target_node 被加進 recalc → `recalcNode` 發現它有 callback → `enqueue_callback(fn)`。等 `processQueue` 整個收斂後，`invoke_callbacks()` 才真的呼叫 fn。**又是「重用電晶體傳播機制」的招式** —— 連「事件監聽」都不用另寫一套。

## 2. 三種「記憶體 / IO」handler

### `handler_ram`（`handler_ram.h`）

```cpp
class handler_ram : public wire_handler {
    wire_memory_field(_ram, "ram");
    wire_register_field(_a, "a[]");  wire_register_field(__d, "_d[7:0]");
    wire_field(_cs, "cs");  wire_field(__oe, "/oe");  wire_field(__we, "/we");
    wire_register_field(_trigger, "cs|/we|a[]|d[]");   // ★ watch 這些
public:
    handler_ram(...) { if (_ram) add_callback(_trigger, [this]{ check_ram(); }); }
    void check_ram() {
        if (!_ram) return;
        if (!_cs) {                          // chip selected (active low)
            if (__we) { __d = _ram->readByte(_a); }       // /WE 高 = read：把資料推回 bus
            else      { _ram->writeByte(_a, __d); }       // /WE 低 = write：抓 bus 寫進去
        }
    }
};
```

→ RAM 完全是行為級：watch cs / /we / 位址 / 資料 bus，一變動就 array 讀寫並回驅資料腳。`system.cpp` 用 `register_handlers<handler_ram>("*func<ram>")` 把每個有 `func<ram>` 掛鉤的記憶體模組都裝上一個這種 handler。`handler_rom` 同理（唯讀）。

### `handler_palette_ram` / `handler_sprite_ram`

PPU 的 palette RAM（32 bytes）和 OAM（256 bytes）本來在 Visual2C02 網表裡是有電晶體的，但 MetalNES 似乎把它們也用特殊 handler 處理（讀 `2c02/nodenames_palram.js` / `nodenames_oamram.js` 列出的節點）。這算是性能優化 —— 把規律的記憶體陣列從電晶體模擬抽出來。

### `handler_clock`（`handler_clock.h`）

```cpp
class handler_clock : public wire_handler {
    wire_field(_clk, "clk");
public:
    handler_clock(...) { _wires->add_handler([this]{ handle(); }); }
    void handle() { if (_clk) setLow(_clk); else setHigh(_clk); }   // 每步翻轉 clk
};
```

→ 每個 half-cycle 翻轉一次 21.47MHz 主時鐘。`add_handler` 把它串進 `_handler` 鏈，`step_cycle()` 每步呼叫。

> 註：`Wires::step_cycle()` 自己也會翻轉一個內部 `_clk` 節點 + 跑 `_handler` 鏈。實際是哪個在動主時鐘要看 `func<clock>` 怎麼接 —— 重點是「每 step 翻一次時鐘節點，然後 recalc 連鎖傳播到整個電路穩定」。

## 3. 頂層 orchestrator：`handler_nes_system`（`handler_nes_system.h`, 550 行）

它是「這台 NES 的大腦之外的部分」。建構時：

```cpp
_palram = add_handler<handler_palette_ram>("");
_sprite_ram = add_handler<handler_sprite_ram>("");
add_edge_callback(_ppu_in_vblank, 1, [this]{ handle_vblank_flag(); });   // VBL 上升沿 → onFrameEnd
```

它持有一大堆 handle，**全部用命名節點抓內部狀態**（這就是 Visual2A03 nodenames 已經把暫存器命名好的好處）：

```cpp
wire_register_field(_cpu_a,  "cpu.a[7:0]");      // 累加器
wire_register_field(_cpu_x,  "cpu.x[7:0]");      // X
wire_register_field(_cpu_y,  "cpu.y[7:0]");      // Y
wire_register_field(_cpu_p,  "cpu.p[7:0]");      // 狀態暫存器
wire_register_field(_cpu_s,  "cpu.s[7:0]");      // SP
wire_register_field(_cpu_ir, "cpu.ir[7:0]");     // 指令暫存器
wire_register_field(_cpu_pc, "cpu.pcl[7:0]|cpu.pch[7:0]");   // PC（低+高拼成 16-bit）
wire_register_field(_cpu_ab, "cpu.ab[15:0]");    // 位址 bus
wire_register_field(_cpu_db, "cpu.db[7:0]");     // 資料 bus
wire_field(_cpu_sync, "cpu.sync");               // SYNC（指令 fetch 週期標記）
wire_register_field(_ppu_vpos, "ppu.vpos[7:0]"); // PPU 掃描線
wire_register_field(_ppu_hpos, "ppu.hpos[7:0]"); // PPU 像素位置
wire_register_field(_ppu_vbl_flag, "ppu.vbl_flag"); ...
wire_memory_field(_cpuRam, "u1.ram");  wire_memory_field(_ppuRam, "u4.ram");
wire_memory_field(_prgRom, "cart.prg.rom");  wire_memory_field(_chrRom, "cart.chr.rom"); ...
wire_register_field(_port0_buttons, "port0.buttons[7:0]");  ...
```

### Reset（真實版本，不是「假裝 NOP」）

```cpp
void reset() {
    clear(_cpuRam); clear(_ppuRam); clear(_chrRam);
    _wires->resetState();              // 全節點重置（pull-up 的設 high，其餘 0/float）
    setHigh(_res);                     // 拉高 reset 線
    _wires->step( 12 * 8 * 2 );        // ★ 跑 192 個 half-cycle，讓晶片自己跑 reset sequence
    setLow(_res);                      // 放開 reset
}
void softReset() { setHigh(_res); _wires->step(12*8*2); setLow(_res); }
```

→ **對照設計文件 §11.3**：設計文件警告「reset 視為 bus 讀到 0xEA 只是 bring-up stub，不是正確硬體行為」。MetalNES 做的就是正確版本：拉 reset 線、讓電路跑、再放開。reset vector 的讀取是電路自己做的，因為 PRG ROM（`handler_rom`）會在電路驅動 `$FFFC/$FFFD` 位址時自動回應。

### Frame 結束處理（`onFrameEnd`）

PPU 進 VBL 時觸發：

```cpp
void onFrameEnd() {
    EmulationState s; getState(&s);                       // 抓 cpu_a/x/y/p/sp/pc、ppu_vpos/hpos...
    _port0_buttons = ~_pad0;  _port1_buttons = ~_pad1;    // 把按鍵狀態寫進控制器節點
    log_printf("frame[%d] time:%d pc:%04X a:%02X x:%02X y:%02X p:%02X s:%02X ...", ...);
    check_unit_test();                                     // 檢查 test ROM 結果簽章
    if (_save_frame_states) SaveState(...);
    _frame++;
}
```

### 影像 / 音訊輸出

- `handler_video_out`：watch PPU 的 `vid_[11:0]`（composite 電壓階梯的 11 bit 編碼）+ `pclk`，每個像素時鐘把電壓值寫進 `raster_device` 的 framebuffer。`raster_device` 再交給 NTSC shader 解碼成 RGB 畫面。
- `handler_audio_out`：watch CPU 的 `snd1`/`snd2`（APU 的 DAC ladder 輸出，類比電壓），定時取樣丟給 `audio_device`，可寫成 `.wav`。

→ 這就是 README 說的 "voltage ladders for composite output and audio" —— 連類比輸出階段都模擬到電壓階梯，再由 host 端做訊號處理。

## 4. 執行緒模型：`wire_thread`

```cpp
void mainThread() {
    SetCurrentThreadName("system-simulation");
    while (!_done) {
        if (_running) step(_stepCount /* 預設 1024 */);   // 跑 1024 個 half-cycle
        else std::this_thread::sleep_for(1ms);
        _queue.dispatch();                                 // 處理 GUI 執行緒丟來的任務（step / reset / saveState...）
    }
}
```

- 模擬跑在獨立執行緒，每輪 step 一批（1024 步），mutex 保護
- GUI 執行緒（ImGui）透過 `dispatch_queue` 把指令（Step / SoftReset / LoadState / SaveState）丟給模擬執行緒執行 —— 不直接碰電路狀態，避免 race
- `_stepCount = 1024`：批次大小是 throughput 與互動延遲的折衷（一批越大越快、但越不即時）

## 5. iNES 載入（`nesrom.cpp`）

```cpp
nesrom_ptr nesrom::LoadFromFile(path) {
    // 讀 "NES\x1A" magic
    // prgCount16K, chrCount8K, flags, flags2, pad[8]
    // prg_rom = 接下來 prgCount16K * 16KB
    // chr_rom = 接下來 chrCount8K  *  8KB
}
```

極簡 iNES header parser。然後 `system.cpp::setupRom()` 把 `prg_rom` 灌進 `cart.prg.rom` 記憶體（16KB 的話 mirror 兩份），`chr_rom` 灌進 `cart.chr.rom`。**「No MMU support」** —— 只支援 NROM（mapper 0），跟我們設計文件 MVP 的範圍一樣。

## 6. 對我們的啟示

1. **「電路引擎 + handler」的分層很乾淨**。電路引擎只管「電晶體狀態傳播到穩定」，所有「外部世界行為」（時鐘、記憶體、IO、reset 排程、影像音訊、test 偵測）都是 handler，透過命名節點 / wildcard 掛鉤 / callback 接上去。我們的 C# 架構應該照抄這個分層。
2. **callback 機制重用電晶體傳播** —— 監聽節點變動 = 加一顆假電晶體連到假節點。簡潔到值得直接搬。
3. **靠命名節點抓內部暫存器** —— `cpu.a[7:0]` 等都是現成的（Visual2A03 nodenames 命名好了）。這讓「驗證」變得超簡單：每幀 dump A/X/Y/P/SP/PC 對照已知正確值。我們的驗證策略要善用這點。
4. **真實 reset**：拉 reset 線 → step 一段 → 放開。不要走 bring-up stub 的捷徑。
5. **記憶體行為級 + GUI/模擬分執行緒 + dispatch queue** —— 都是務實工程，可以照搬。
6. **批次 step（1024）** —— throughput vs 互動延遲的折衷。我們做 GPU 版時，「每次 kernel launch 推進幾個 half-cycle」也是同一個 trade-off（設計文件 §10.3）。
