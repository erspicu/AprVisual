### 1. 「GPU 虛擬 FPGA」跟「GPU gate/LUT-level sim」是同一件事嗎？
**你的理解完全正確，而且情況會更糟。**

把 RTL 轉成 LUT netlist 並在 GPU 上跑，本質上就是 Gate-level simulation (GLS)。但如果這個方案標榜的是「虛擬 FPGA fabric」，那代表它不僅要模擬 LUT 的邏輯，**還要模擬 FPGA 的 Routing Network（Switch Box, Connection Box, Routing Tracks）。**

在真實 FPGA 中，routing 佔了 ~80% 的矽面積與延遲；如果要在軟體裡「模擬」這件事，你的 GPU 記憶體存取模式會變成在海量的 routing matrix 節點中追逐 pointer/index。
你原本的 GPU 方案只要跳 44 個拓樸層；如果加上模擬 FPGA routing，GPU 需要處理大量的 multiplexer 與 switch 模擬，這絕對比你直接做 netlist sim 慢上幾個數量級。**這不叫加速，這叫「用模擬器去跑一個模擬器」。**

### 2. 真實 FPGA LUT 的 Homogeneous (零 warp divergence) 優勢能讓 GPU 變快嗎？
**不能，這是一個看似美好的陷阱。**

你說的沒錯，LUT 的計算是完美的 SIMD（全都是查 64-entry SRAM），這能解決 8-way 分歧的問題。但這份「同質性」帶來的微小加速，會被以下三個致命傷徹底輾壓：

1.  **Sparsity (稀疏性) 的喪失是致命的：** CPU (S1) 為什麼能跑到 47K hc/s？因為它是 Event-driven。在一個 cycle 裡，NES 大概只有 1% ~ 5% 的節點會 flip（dirty-set）。但在 GPU 上為了維持這份 Homogeneous 與 SIMD 效率，你必須做 **Levelized Batch Evaluation（每個 cycle 無腦重算全部 50K 個 LUT）**。為了維持 GPU 管線塞滿，你做了 50 倍的「無效白工」。在 NES 這麼小的設計上（50K nodes 對 GPU 來說太小，塞不滿吞吐量；但 latency 卻很長），GPU 的暴力平行度根本救不回這 50 倍的浪費。
2.  **Global Barrier 的代價依然存在：** 只要你是 Levelized 評估，你就逃不掉層與層之間的 `AllMemoryBarrierWithGroupSync`。
3.  **Routing 的記憶體頻頸：** LUT 計算雖然不分歧，但 LUT 之間的「連線」在虛擬 FPGA 裡是極度不規則的 memory read/write。這會導致嚴重的 GPU VRAM coalescing 失敗，變成隨機存取 (Random Access)。

**結論：這不會比你現在的 10.7x 慢更好，只會更糟（可能慢上百倍）。**

### 3. 有沒有 mature 的開源「GPU 模擬 FPGA fabric」專案？
**沒有，產業界與學界都沒有這種「可用於高速 functional simulation」的東西。**

*   **產業界：** EDA 大廠 (Synopsys, Cadence) 的 GPU 模擬器 (如 Xcelium ML) 是用來跑巨型 SoC (幾億個 Gate) 的 RTL，而且是用 CPU-GPU 協同的 Event-driven 或巨觀 batching。沒人會用 GPU 去模擬 FPGA fabric 來做功能驗證。
*   **開源界：** 如你所知，Verilator, CXXRTL 都是 CPU-based (將 RTL 編譯成高度最佳化的 C++ 機器碼，這才是極致的快)。
*   **學界 (VTR/VPR 等)：** 這些是 FPGA 架構研究工具，用來評估「如果我發明了一種新的 LUT 架構，Area/Delay 會怎樣」，它們跑起來是**極度緩慢**的，根本不是設計來當 emulator 的。

你找不到現成的工具，代表如果要做，你得自己寫一個，而這是一個巨大且毫無經濟效益的工程。

### 4. FPGA-fabric 對解決 precharged-dynamic 與半週期時脈相位有幫助嗎？
**你的判斷 100% 正確，這兩者完全無關，甚至 FPGA 抽象會讓問題更無解。**

*   **FPGA 是純 Static CMOS 且嚴格 Synchronous 的世界：** FPGA fabric 裡面沒有 Precharge、沒有 Discharge、沒有 Pass-transistor logic、沒有 Tri-state bus（現代 FPGA 內部的 Tri-state 都是用 MUX 模擬出來的）。
*   如果你要把它對應到虛擬 FPGA，你**還是得先在 Verilog 裡用純邏輯把 Precharge 的 2-phase 或 4-phase 行為顯式寫出來**（例如寫一個 FSM 來控制 bitline 狀態）。
*   既然你都已經能在 Verilog 裡寫出 FSM 了，那你直接丟給 CPU 的 Verilator 跑就好了，為什麼要脫褲子放屁去編譯成虛擬 FPGA bitstream 再丟給 GPU 跑？這完全沒有解決原本「從 Switch-level netlist 萃取出精確行為」的硬問題 (Lift 階段的難題)。

### 5. 整體判斷與總結
**我完全同意你的判斷。這是一條死胡同。**

總結你的論點，並加上我的 EDA 視角：
1.  **工具鏈不存在：** 沒有 mature 工具，且自己開發不切實際。
2.  **效能必定慘跌：** GPU 的 SIMD 優勢無法彌補失去 Event-driven (Sparsity)、Random Access Routing 以及 Global Barrier 帶來的巨大 overhead。在小設計 (50K gates) 上，高時脈、強大 Branch Prediction 的 CPU 永遠屌打 GPU。
3.  **無法跨越物理抽象層的鴻溝：** Switch-level 的 analog/dynamic 行為無法靠「換一個執行環境」自動解決。問題在前端的 IR 萃取（Modeling），不在後端的執行。

**有沒有次要價值？**
沒有。如果為了「在沒有實體 FPGA 的情況下驗證 FPGA-targeted RTL」，業界的標準做法是直接拿你的 RTL source code 餵給 **Verilator (C++)** 或 **Icarus Verilog**。在 CPU 上跑 RTL sim，對於 NES 這種規模的設計，Verilator 可以輕鬆跑到 **幾百 FPS (幾千萬 hc/s)**。
