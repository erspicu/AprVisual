## Full implementation detail + cache/working-set characterization

### Golden engine (12.0 µs/hc)

You have hit the nail directly on the head. Your diagnosis of the memory hierarchy’s role in this performance inversion is **100% correct**, and it represents a profound insight into how high-performance simulation actually interacts with modern CPU microarchitecture. 

When you move from modeling a system conceptually (op-counts) to implementing it on bare metal, memory hierarchy—specifically the L1 instruction cache (i-cache), L1 data cache (d-cache), and Branch Target Buffer (BTB)—dictates the real-world speed limit.

Here is the brutally honest breakdown of your hypothesis, how it changes the design, and a revised estimation of what is actually possible.

---

### 1. The Hypothesis: Is i-cache thrashing the culprit?
**Yes, absolutely.** The fact that your compiled straight-line code is 2× *slower* than your interpreter is the smoking gun for **Front-End Bound execution via i-cache thrashing.**

*   **The Interpreter:** Has a tiny instruction footprint (a single `for` loop, some array indexing, a bitwise shift, and a lookup). This fits neatly into the ~32KB L1i cache. The CPU's execution units are starved only by the *data* (waiting on L1d/L2 cache misses from your scattered gathers and ~1MB truth table).
*   **The Compiled Model:** You unrolled 6,000 nodes into ~700+ KB of native machine code. You are forcing the CPU to stream through ~700 KB of instructions 6.5 times per half-cycle. Because the L1i cache is only 32KB to 48KB, the CPU continuously evicts and re-fetches instructions from L2/L3. The CPU's front-end (fetch and decode) completely stalls, leaving the incredibly fast ALUs sitting idle. 
*   **How to prove it:** Run a profiler that reads CPU Hardware Performance Counters (like Intel VTune or perf). Look for:
    1.  `ICACHE.MISSES` or `L1I.MISSES` (will be astronomically high on the compiled version).
    2.  `IDQ_UOPS_NOT_DELIVERED.CORE` (measures front-end starvation; will dominate your pipeline slots).
    3.  `L2_RQSTS.CODE_RD_MISS` (shows you are dragging instructions all the way from L3 or Main Memory).

### 2. Does this change the MACRO-EVENT design? (Compiled vs. Data-Driven)
**Yes. It completely kills the idea of compiling individual macro-cones to code.**

If you compile your extracted cones into native methods, you turn a data-routing problem into a code-execution problem. Even if only 30–80 cones fire per half-cycle (which *might* fit in L1i if the active ones total < 32KB), you will destroy the **Branch Target Buffer (BTB)** and the **instruction prefetcher**. 

In an event-driven compiled model, your scheduler loop looks like: `Dequeue -> Jump to Function Pointer`. Jumping to 80 different, semi-random memory addresses for tiny blocks of code (a few bitwise ops) is a CPU's worst nightmare. The branch predictor will fail, causing pipeline flushes on every macro-event.

**The winning architecture is a Data-Driven Interpreter.**
You need a single, highly optimized, fixed C# loop (the interpreter engine). The logic of the 2A03 is modeled strictly as **compact data** arrays. 
*   **Code:** Stays permanently resident in L1i. The branch predictor learns the single central loop perfectly.
*   **Data:** The 30–80 active cones represent only a few kilobytes of data (input IDs, a truth table index, output IDs). This streams beautifully into L1d.

### 3. How to layout the Cone DATA for locality
Since you have a golden oracle trace, you have a massive unfair advantage: you know *exactly* which cones fire, how often, and in what company. You must layout data to maximize L1d cache-line utilization (64 bytes per fetch).

Do not use C# object graphs (`class Cone { ... }`). Use tightly packed **Structure of Arrays (SoA)** or flat arrays.

1.  **Cluster by Clock Phase:** Divide cones into `Phi1` and `Phi2` buckets. They never fire simultaneously.
2.  **Cluster by Co-activation (Cliques):** Use the oracle trace to find cliques (e.g., "When the ALU Add cone fires, the Flag Register cone almost always fires"). Assign these contiguous IDs.
3.  **Flat Data Structures:**
    *   `int* ConeInputs_Base` and `ushort* ConeInputs_Data` (Flat array of input node IDs to gather).
    *   `byte* ConeTruthTables` (Only the TTs for *hot* cones; keep cold initialization/reset cone TTs far away so they don't pollute the L2 cache).
    *   `int* ConeOutputs_Base` and `ushort* ConeOutputs_Data` (Downstream cone IDs to enqueue).
4.  **The Execution Loop:**
    ```csharp
    // The ONLY code the CPU executes. Fits in 1KB of L1i.
    while(queue.Count > 0) {
        int coneId = queue.Pop();
        ulong state = GatherInputs(coneId);     // Sequential read of ConeInputs, scattered read of global state
        byte result = EvaluateTT(coneId, state); // 1 L1d lookup
        ScatterAndEnqueue(coneId, result);      // Sequential read of ConeOutputs, queue push
    }
    ```

### 4. Is there a hybrid "Compile Hot, Interpret Cold" regime?
**Theoretically yes, practically no.** 
If you find that exactly 3 to 5 massive, complex cones (like the ALU or Address Bus multiplexer) account for 60% of the CPU time, compiling *just* those 5 into dedicated C# functions might yield a tiny speedup. 

However, C# interop between array-pointers and function-pointers has overhead. The data-driven interpreter loop will be so tight that the cost of doing an indirect branch to a compiled hot-cone method might actually be slower than just letting the interpreter evaluate its truth table. Stick to 100% data-driven first. Only consider hybrid if profiling proves the TT lookup for a specific massive cone is your singular bottleneck.

### 5. Revised Brutally Honest Estimate and Dominant Costs
Let's look at the harsh reality of the math. 
Your Golden engine takes **12.0 µs / half-cycle**. At ~4 GHz, that is about **48,000 clock cycles**. 
Golden updates ~600 nodes, doing minimal work (averaging ~80 cycles per node update, which is phenomenally tight for a switch-level graph walk).

If you move to a Macro-Event Data-Driven approach with 50 macro-cones per half-cycle:
*   **The Irreducible Advantage of Golden:** Golden only touches the *exact* 1.4 switch nodes that flipped. A macro-cone must gather *all* inputs for a logic cone even if only one input changed.
*   **The Dominant Cost Shifts:** Your new bottleneck will be the **D-cache gather latency**. If a cone has 20 inputs, you must do 20 scattered reads across the global `LogicState` array. Scattered reads defeat hardware prefetchers. Even if `LogicState` is entirely in L1d, gathering 20 bytes from random offsets takes ~20–30 cycles.
*   **Scheduler Overhead:** Pushing and popping 50 items to a queue (even a ring buffer) costs cycles. 

**The Math:**
*   50 cones per half-cycle.
*   Gather 15 inputs per cone = ~25 cycles.
*   TT Lookup = ~5 cycles.
*   Scatter/Enqueue 3 outputs = ~15 cycles.
*   Total per cone = ~45 cycles.
*   50 cones × 45 cycles = 2,250 cycles.
*   Add queue overhead, branch mispredicts, and L1d cache misses = ~5,000 to 8,000 cycles total.

**Revised Speedup Estimate:**
If you execute perfectly on the data layout, you will drop from 48,000 cycles (Golden) to ~8,000 cycles. 
That is a **~6× speedup (approx 2.0 µs / hc)**. 

My previous 5–15× estimate was too optimistic because it failed to account for the D-cache gather penalty inherent to evaluating whole logic cones. 

**Summary of the Brutal Truth:**
You cannot beat Golden by 50×. Golden is essentially executing a hardware-accelerated sparse matrix walk. Your best-case scenario with a flawlessly cache-optimized, purely data-driven Macro-Event model is roughly **4× to 6× faster** than Golden. 

The dominant remaining costs will be:
1.  **D-cache scattered reads** (Gathering inputs for the cones).
2.  **Scheduler bookkeeping** (Pushing/popping from the macro-event queue).
3.  **The irreducible sparsity gap** (Golden does micro-work on exactly 1 changed node; you do macro-work on a whole 15-input cone because one input changed). 

To achieve this 4-6×, you must ruthlessly eliminate code sprawl, keep execution in a single L1i-friendly loop, and use your oracle trace to pack your cone metadata into contiguous, cache-line-aligned arrays.
