# Thread 實驗:CPU‖PPU 雙執行緒切分(分支 thread-experiment)

> 目的:**學術驗證**。實作失敗也有價值 —— 量到「為什麼失敗、失敗多少」就是論文邊界地圖
> 那章的素材。本分支不求合併回 main,求一份可引用的數據與結論。

## 背景與假設

單核 ~136K hc/s,離即時 316×。過去 4 次平行化全死(per-chip per-wave 15×、bit-parallel
156×、per-chip rayon 15×、codegen batch 3–6×),共同死因=**細粒度同步成本**。

2026-06-13 的邊界量測(`MD/note/2026-06-13-CPU-PPU雙執行緒切分_可行性實測.md`)顯示:粗粒度
功能切分(2A03‖2C02 當兩個 PDES 行程,只在匯流排事件同步)是唯一「碰到不立刻死」的方向 ——
真正的 CPU↔PPU 交握僅 ~0.007/hc,切點窄(~13 條線),可位元精確。**但工作嚴重偏 PPU
(PPU 每 hc 滿載、CPU 只 12.5% hc 動、cpu-only=0),預測 2-way 加速上限僅 ~1.12x。**

**主假設 H1**:即便耦合稀疏、可位元精確,2-thread 切分的實測也會 ≤1x(同步/喚醒/快取一致性
成本 > 受限於負載不平衡的微薄理論增益)。本實驗要量出實際倍率與成本分解。

**對立假設 H0**:某種粗粒度設計能達 >1.05x 位元精確 → 意外正向,值得追。

## 階段

- **S1 精確分割**:flood-fill 把每個節點歸 {CPU, PPU, cut, neutral};驗證 cut 恰為耦合集、
  兩側除 cut 外通道不相連。輸出精確兩側工作平衡(補足上次無名節點的缺口)。
- **S2 Oracle 上限(真實計時)**:單執行緒下,每 hc 分別計時兩側 settle,算
  T_parallel = Σ max(t_cpu, t_ppu) + 同步成本模型 vs T_serial。給出帶真實 ns 的加速上限。
- **S3 真 2-thread 原型**:per-wave barrier lockstep(CPU thread + PPU thread,每波屏障交換
  cut)。位元精確閘(golden checksum)。量實際 hc/s + 成本分解。預期:slowdown(負結果)。
  **核心綁定(使用者指定 2026-06-13):兩條 thread 各鎖定不同的最佳 P-core —— 主/CPU 線程
  一顆、PPU 線程另一顆(避開 E-core、避開同一實體核的 SMT 雙線程),用既有 --pin 的選核邏輯
  挑兩顆,High priority + EcoQoS-off。乾淨量測用。**
- **S4(選配)**:hc 粗粒度樂觀執行 + rollback(使用者原構想的最忠實版本)。

## S1 結果(2026-06-13,SMB 3M hc,flood-fill 分割)

**Cut 完整性:contested = 0** —— flood-fill 後沒有節點同時被兩晶片觸及 → CPU/PPU 兩側除了
14 條 cut 線外通道完全不相連,**PDES 切分結構乾淨**(S3 能把每節點明確分某 thread)。
- 節點數:cpu 3,361 / ppu 7,671 / cut 14 / contested 0 / 中性(時鐘樹等)3,682

**工作平衡(pop):cpu-side 16.9% / ppu-side 54.3% / shared(時鐘樹,不可切)28.8%。**
**加速上限:1.84x(shared 免費,不可能)~ 1.20x(shared 兩側都跑)→ 實際 ≈ 1.20x。**
(比上次 ~1.12x 精確:flood-fill 正確把較多內部節點歸 CPU 側,cpu 10.5%→16.9%。)

工具:`BuildPartition()`(WireCore.Recalc.cs,DEBUG),輸出 `[cpu-ppu-boundary]` 的
partition NODES / PDES side split 兩行。

## 判定準則

| 結果 | 解讀 |
|---|---|
| >1.05x 且位元精確 | 意外正向 → 追 S4 / 擴大 |
| 位元精確但 <1x | **預期負結果** → 記錄倍率 + 成本分解(同步 vs 閒置 vs 屏障) |
| 無法位元精確 | 語意發現 → 波內 Gauss-Seidel 順序無法跨 thread 保序 |

任一結果都記成數據與 note。失敗不刪分支,留作論文引用。
