## 1. NES 主引擎(AprVisual.S1,full_palette,400k hc)

| 機器 | rate | checksum @400k | 距即時 |
|---|---|---|---|
| 桌機 Zen 2(中位數) | ~132K hc/s(峰值 135.9K) | `0x9174E19D961CB6E5` | ~316× |
| **Pi 5 A76**(best-of-8) | **best 71,991 / 中位數 ~71,546 hc/s** | **`0x9174E19D961CB6E5`** | **~600×** |

- **跨 ISA 位元精確**:Pi 的 checksum 與桌機 golden **一字不差** → 引擎在 x64 與 arm64 上行為完全相同(純整數/位元運算,無浮點)。這是論文「第二顆微架構」最關鍵的證據。
- 絕對吞吐 Pi ≈ 桌機的 **0.54×**(時脈 2.4 GHz vs ~4 GHz boost + 微架構差異)。變異極小(8 跑落在 71.0–72.0K,CV ~0.5%)。
- **已上傳排行榜**:`api.php?action=upload`(POST,FormData 欄位 `logfile`),回 `{"ok":true,"id":38,"rank":36,"total":38,"hcPerSec":71666.9}`——排行榜**第一個 ARM / Raspberry Pi 條目**(`cpu_model="Raspberry Pi 5 (Arm Cortex-A76)"`, `arch=Arm64`, os Debian 13)。上傳前修正了引擎在 Linux 沒抓到的兩個 metadata 欄位(`cpuModel` "unknown"、`commitDate` "%cd");效能數字 71,666.9 為實測未動。

## 2b. 三種無-ROM 工作負載(NOP / fuzz / reset-hold,我們的引擎)

裸 CPU 沒有 test ROM,用三種 pin 邊界驅動;以下為 Pi 上**我們的引擎**(C# ours,500k hc/round,3 rounds):

| 晶片 | NOP sled | Random fuzz(排除鎖死碼) | Reset-hold |
|---|---|---|---|
| MOS 6502 | 74,955 | 38,566 | 100,154 |
| Motorola 6800 | 38,116 | 68,207 | 112,957 |
| Zilog Z80 | 28,356 | 24,991 | 94,812 |

- **fuzz vs NOP 的正負號在 ARM 上完全重現桌機**:6502 與 z80 在 fuzz 下**較慢**(真壓力——隨機執行翻動更多節點),6800 在 fuzz 下**較快**(它的 NOP=0x01 是相對重的指令)。所以「fuzz=最壞情況」只對三顆中的兩顆成立——與桌機結論一致。
- **reset-hold 每顆都最快**(量的是引擎基礎時鐘樹開銷);Pi 上落在 ~95–113K,比桌機(~156–166K)離散一點,但「reset 最快」的次序不變。
- 每種工作負載的 checksum 各自固定(不同工作負載狀態軌跡本就不同),此處不跨工作負載比對。

## 3. 關鍵比值分析(Pi vs 桌機)

| 量 | 6502 | 6800 | z80 | 桌機對照 | 結論 |
|---|---|---|---|---|---|
| **語言紅利** JS→C# naive | 74× | 77× | 71× | ~70–85× | ✅ **重現**:直譯器懲罰在 ARM 上同樣是 ~70–80× |
| **語言紅利** C++naive/C#naive(native vs managed,naive) | 1.53× | 1.63× | 1.74× | 1.36–1.51× | native 在 ARM 上略更領先 |
| **演算法增益** C# ours/C# naive(同語言) | 6.9× | 5.6× | 4.0× | 8.1 / 7.1 / 5.1× | ✅ 仍是數倍,但 ARM 上**略小** |
| **反轉** C# ours / C++ ours-full | 1.08× | 1.07× | 1.03× | 1.27–1.56× | ✅ 仍成立(C# 領先),但**幅度大幅縮小** |
| 絕對速度 Pi / 桌機(C# ours) | 0.49× | 0.42× | 0.45× | — | Pi ≈ 桌機的 ~一半 |

### 三個值得記的觀察
1. **論文的核心分解在 ARM 上完整重現**:JS→AprVisual 的巨大比值(286–506×)一樣乾淨地拆成
   「~70–80× 語言紅利 × ~4–7× 演算法」。演算法貢獻(同語言 naive→ours)在 ARM 上是 **~4–7×**,
   與桌機 ~5–8× 同量級、略低。
2. **「反轉」在 ARM 上仍然成立但縮到 ~1.03–1.08×**(桌機是 ~1.27–1.56×)。也就是說:
   重度微調的 C#(.NET 11)在 ARM 上只比忠實的 g++ `-O3` C++ 移植快個位數百分比,
   遠不如它在 x64 對 clang 的領先。可能原因:.NET 的 arm64 JIT/動態 PGO 相對 x64 較不成熟、
   或 g++ 在 A76 上相對 clang-on-x64 更強。**無論如何,「語言這軸從不超過 ~1.5×」在 ARM 上更成立**
   ——兩個編譯語言落在 ~1.08× 內。
3. **演算法增益在 ARM 上略小**(z80 尤其,4.0× vs 桌機 5.1×)。A76 的記憶體延遲/快取行為與 Zen 2 不同
   (L1d 反而較大 64 KiB),latency-bound 的本質不變,但剪枝/fast-path 的相對回報略降。

## 4. 量測條件

- 每引擎 hc 依速度調整(回報的是 hc/s 速率,跨不同 hc 可比):JS naive `hc≈5000–6000, warmup 1000, rounds 3`;
  C#/C++ naive `hc=100000, warmup 20000, rounds 3`;各 ours `hc=500000, warmup 50000, rounds 3`。
- 未用 `--pin`(Linux 上 PerfTuning 不支援硬綁核;Pi 預設排程)。clock 未鎖(A76 scaling ~75%)。
- C++ 用 **g++ 14.2 `-O3 -std=c++17`**(非桌機的 clang)。C++ 載入的是 C# `--export-*` 在 Pi 上重生的網表/引擎檔,資料與 C# 端一致。
- 原始輸出留在 Pi 的背景執行 log;本機 task 輸出 `b99gukx0n`(6800+z80)、6502 為前景單跑。
