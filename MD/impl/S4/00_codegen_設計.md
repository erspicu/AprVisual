## 1. 目前的 IR 資產

- `IrEngine`：每個 node 的 `NextExpr[v]` = 「下半週期 settle 後 v 的值」的 boolean tree（`NodeRef(id)`=current 值 / `Hold(id)`/`Prev(id)`=上半週期值 / `Mux/And/Or/Not/Const`）。
- driving 模式（已驗證 0 mismatch）：每半週期 ① `PrevStates = NodeStates` snapshot ② `RunHandlerChain`（master clock toggle + 行為式 RAM/ROM）deferred-enqueue ③ `ProcessQueueOneLevel`（flush 一層）④ `RunFlatProgram`（按 `EvalOrder` 拓樸序評估 12501 個 driving-covered node 的 NextExpr → flat stack-machine instruction stream，~99K 條）⑤ `ProcessQueue`（S1 的 switch-level BFS 處理剩下的：843 個 SCC node + ~2200 個 hybrid bus node + 它們的 fanout）。
- coverage：14723 個 node —— **12501（85%）driving-evaluated**（有 NextExpr 且不在 SCC，γ.0+γ.1）、**843 在 56 個殘餘 SCC**（counter / shift-register / phase-ring；γ.2 暫停）、**~1379 hybrid bus**（`NextExpr==null`；BD0-7 / io_db / exp_in 那種 multi-driver tri-state bus）。

## 2. 目標 / milestone

**最終目標（user 確認 2026-05）= 正確 + 即時堪用**（不追求上千台並行）。最終會有兩個執行版本（共用 S1/S2/S3 的 IR、在「執行模型」分岔）：

- **(A) GPU bit-sliced 版 = S4，先做** —— brute-force flat program + 殘餘環 fixed-K + bit-sliced。
  - **M_codegen-cpu**：IR → 一個 C# 的 `Step(...)`（per-instance，先非 bit-sliced），benchmark vs S1（2 frame branch_timing = 29.7s）。100% 等價。
  - **M_codegen-llvm**：同上但 LLVM 後端（emit LLVM IR → `-O3` → JIT/AOT）。benchmark 跟 C#-emit 版對照。
  - **M_bitslice-cpu**：把 `Step` bit-sliced（`ulong` → 64 台 / AVX2-512 更多），證明吞吐量隨並行度 scale。
  - **M_gpu**（最後、target 待定）：同一份邏輯 retarget GPU（NVPTX-via-LLVM、或 compute shader —— 見 §4b）。
- **(B) CPU-optimized 版 = S4 之後、從較早 commit branch** —— event-driven IR 直譯器（β：dirty-set、只重算 input 變了的 node、hybrid bus 仍 S1 group resolution）。在「單實例 CPU」前提下盡可能快（互動式使用 / debug / 不開 GPU 的場景）。**M_cpuopt**：等價 gate vs S1；benchmark 單實例 vs S1，目標心理值 ≥ 5-10×。為什麼 branch 而非接 S4：S4 為 GPU 把 IR 攤平 / fixed-K / bit-sliced 改了很多對 CPU event-driven 反而是負擔的東西；(B) 想要「乾淨 IR + S1 風格 event loop」。詳見 `MD/struct/08` 的「目標與最終形態」+「S4 之後」段。

註：對電晶體級準確度，**單實例「即時」≈ ~43M 半週期/sec，比目前 S1 的 ~46k/sec 快 ~900-1000×** —— 這是硬目標；兩版一起逼近（GPU 版 = 多台 + 確認管線、CPU-opt 版 = 單實例最快）。

## 3. codegen 要處理的三類 node

1. **driving-covered（85%，有 NextExpr、acyclic）**：直接把 `NextExpr[v]` 的 tree 編成 `node[v] = <expr>`，按 `EvalOrder` 順序排（拓樸 → 一個 pass）。`NodeRef(x)` → `node_cur[x]`、`Hold(x)`/`Prev(x)` → `node_prev[x]`、`Mux(c,a,b)` → bit-sliced 是 `(c & a) | (~c & b)`、`And`/`Or`/`Not` → `& | ~`。這部分就是 `RunFlatProgram` 的編譯版本（現在是 interpreter、要變成 emitted code）。
2. **殘餘 SCC（843 個 node）→ Fixed-K Micro-Evaluation**（Gemini）：給 SCC 一個任意拓樸序，emit `for k in 0..K(=3~8): { node_cur[v] = <NextExpr[v]>; ... }`（`#pragma unroll` / 手動展開）—— `Node(x in SCC)` 讀 `node_cur[x]`（同一 K-loop 內最新值，Gauss-Seidel）、`Node(x not in SCC)` 讀已算好的 `node_cur[x]`、`Hold/Prev` 讀 `node_prev[x]`。init `node_cur[v] = node_prev[v]`。K 要夠大讓 settling ripple 收斂（小 SCC K=3~4、496-node DMC counter 可能要 K=~16~50；用等價 gate 決定）。GPU 友善（零 divergence）。
3. **hybrid bus（~1379 個 node）→ ???**（待 Gemini）：選項：
   - (a) 也用 Fixed-K micro-eval —— 但它們 `NextExpr==null`，沒 expr 可 eval。要先給它們一個「pseudo-NextExpr」：tri-state bus = `bus = Mux(drv1_en, drv1_val, Mux(drv2_en, drv2_val, ..., Hold(bus)))`（從 DriveAnalysis 的 `Passes` + pull-down 資訊組）—— 然後跟殘餘 SCC 一起丟進 Fixed-K block（bus 跟 SCC node 常常互相耦合，例如 PPU io_db ↔ palette ↔ ...）。
   - (b) 留給 S1（codegen 出來的東西混合呼叫一個小 S1 kernel for the bus nodes）—— CPU 上勉強可、GPU 上不行（warp divergence）。
   - (c) 把 bus 也當「狀態」、用 switch-level group resolution 的 bit-sliced 版（GND wins → VCC → drive → hold → 0 的 256-entry LUT）—— 但 group 是動態的（哪些 transistor 導通隨值變），bit-slice 不了。
   - 傾向 (a)：給 hybrid bus 抽 pseudo-NextExpr（`Mux` chain over drivers）→ 它們就變成「有 NextExpr 但可能在 SCC 裡」→ Fixed-K 處理。這也順便把它們從「S1-only」變成「IR-modeled」→ coverage 衝到 ~100%。

## 4. 後端：C# emit + LLVM emit，兩份並行做效能對照（user 定調 2026-05）

- **做兩份**：(1) **C# 後端** —— emit C#-source（一個巨大的 `static void Step(...)`、bit-sliced `& | ^ ~`）→ Roslyn 動態編譯。陽春、快 prototype、量 baseline、驗證邏輯正確。風險：~100K 行 / 數萬 local 的方法可能把 RyuJIT 的 register allocation 搞爆（量出來看）。(2) **LLVM 後端**（`LLVMSharp` / LLVM-C API via P/Invoke）—— emit LLVM IR → `-O3`（跨全域 const-folding / DCE / instruction-combining + greedy regalloc 處理巨型 basic block 遠超 RyuJIT）→ JIT/AOT 成 native。整個 codegen 工具都用 C# 寫（不引入 C++ toolchain）。**LLVM-in-.NET 的用法參考 user 抓下來的另一個 .NET+LLVM 專案**（同 owner、例如 github.com/erspicu/AprGba）。
- 兩份的等價 gate（逐 node vs S1）都要過；benchmark 報告列「S1 / C#-emit / LLVM-emit / + bit-sliced ×N」的 throughput 對照。Gemini（`..._152404.txt` §6 / `..._162049.txt`）強烈推 LLVM 是因為「同一份 LLVM IR 一鍵 retarget GPU、只維護一套 IRBuilder」—— 但 user 想兩份都做、量出差距，所以兩份並行。

## 4b. GPU 化 —— target 尚待討論（不急著定）

CUDA / NVPTX **不是唯一選項**。考量：(1) C# 對 CUDA 的 library 支援不一定好處理；(2) **GPU compute shader / shading language**（HLSL compute shader、GLSL/SPIR-V compute、WGSL …）可能更貼 .NET 生態 —— bit-sliced boolean kernel 本來就只是一堆 `& | ^ ~`，compute shader 跑得動；(3) **參考 user 的 `AprNesAvalonia`**（AprNes 的 .NET 版、用 Avalonia UI 框架、自己發展的 shader script）—— 那套「在 .NET 裡跑自訂 shader」的做法可以借。→ S4 的 GPU 那一步先不寫死 target；等 C#/LLVM 兩份 CPU 後端做出來、量過，再討論「CUDA-via-LLVM-NVPTX」vs「compute-shader」路線。

## 5. 驗證計畫

- bit-sliced `Step` 跑 N 個 NES instance（先 1 個，再 64 個都載同一個 ROM）→ 每半週期跟 S1（per-instance）比對每個 observable node。0 mismatch = 等價。
- benchmark：2 frame branch_timing，bit-sliced（64 instance）的 wall-clock / 64 vs S1 的 wall-clock → per-instance 吞吐量比。
- 階段：先 emit + interpret-equivalent 的 `Step`（驗證 codegen 的邏輯）→ 再 bit-slice（驗證並行）→ 再 LLVM（驗證後端）→ 再 GPU（bonus）。

## 7. S4 步驟拆解（MVP-first，Gemini review 後的修正順序 —— `/loop 5m` 跑到 S4.4）

> 每步照 per-step 工作流程（寫設計 → 必要時丟 Gemini → 評估採用 → 實作 → build 0/0 + 功能驗證 + 不破壞既有（`--selftest` 全 PASS、`--trace branch_timing/1 --cycles 8` 跟前一 commit 一致、`--trace-cmp` / `--trace-cmp --engine ir` 等價 0 mismatch）→ 更新④⑤+進度日誌 → commit & push）。

- **S4.0 — design review**：✅ 完成（firing 1）—— 把 §6 問題丟 Gemini（log `..._174739.txt`），全採用（見 §8）。
- **S4.1 — C# emit：先只 emit IR-covered 的 ~85%、scalar（非 bit-sliced）、殘餘 SCC+bus 仍走 S1** ✅ **MVP done（firing 2）** —— 用 `System.Linq.Expressions` 把 EvalOrder 編成 chunked `Action<byte[] cur, byte[] prev>` delegates（每 `ChunkSize=512` node 一個、`.Compile()` → JIT'd IL）—— `NodeRef(id)`→`cur[id]`、`Hold/Prev(id)`→`prev[id]`、`Mux(c,a,b)`→`c!=0?a:b`、`Not`→`1-x`、`And/Or`→`&/|`（全用 int、store 時 `(byte)`）。`StepOneDriving` step-4 = `RunCompiledStep`（snapshot `NodeStates→_evalCur`、跑 23 個 chunk、再 loop EvalOrder 把變了的 `SetNodeState+EnqueueNode`）取代 `RunFlatProgram`（`UseCompiledStep` flag 可 A/B、`--no-compiled-step` 切回直譯器）。也 emit 等效的 **C# 原始碼字串**（`IrEngine.EmitCsharpSource()` + `--dump-emitted-cs <file>`）—— 給 Roslyn 後端 / GPU compute-shader 雛形 / 眼睛看用（11680 node、23 chunks、~11.7K 行、~905KB）。驗收：`--trace-cmp --engine ir` 0 mismatch（3000 hc）、`--no-compiled-step` 也 0 mismatch、checking 模式不受影響、`--selftest` 全 PASS、`--trace` 跟 HEAD 一致。**benchmark（branch_timing/1 frame = 654,719 master 半週期，這台機器）：S1 = 14.9s（~44k hc/s）；直譯器（`RunFlatProgram`，`--no-compiled-step`）= 179.7s（~3.6k hc/s）；compiled（`RunCompiledStep`）= 100.6s（~6.5k hc/s）→ compiled 比直譯器快 ~1.8×** ✓（達成「應更快」）。compiled 仍比 S1 慢 ~6.8×（預期）。**firing 3：試了在每個 compiled chunk 開頭加 `if (cur.Length < n || prev.Length < n) throw;` 的 length-guard（RyuJIT BCE 的標準 pattern，讓 chunk 內 `cur[id]`/`prev[id]` 的 bounds check 可消）—— benchmark 不變（101.7s）**：診斷出 **瓶頸不在 step-4（compiled chunks）、而在 step-5（`WireCore.ProcessQueue` 的 bridge，對 ~843 SCC + ~1379 hybrid bus + fanout 跑 S1 的 switch-level BFS）** —— 那 ~100s 幾乎都是 step-5。所以 compiled step-4 的 micro-opt（bounds check / `_evalCur` copy）報酬有限；**真正讓 driving 變快的是 S4.2/S4.3（把 SCC + bus node 也 model 進 IR → step-5 縮到 ~0）**。length-guard 留著（harmless、S4.2/S4.3 後會有用）。（註：直譯器現在比之前慢，是 γ.1 把 flat program 從 ~35K 條 bloat 到 ~99K 條（size-2 代換複製子式）造成的 —— compiled 把同樣的邏輯編成 JIT'd IL、操作數少很多。）
- **S4.2 — γ.4 hybrid bus lowering（IR pass，放 S3↔S4 之間）**：把 ~1379 個 multi-driver tri-state bus node 抽 pseudo-NextExpr —— **NMOS 物理式（Wired-AND）、不是巢狀 Mux**：`has_pd = (en_1 & !val_1) | (en_2 & !val_2) | …`（任何人輸出 0）、`has_pu = (en_1 & val_1) | …`（任何人輸出 1）、`bus_next = has_pd ? 0 : (has_pu ? 1 : Hold(bus))`（`en_i` = DriveAnalysis 的 `Passes[i].Cond`、`val_i` = `Node(Passes[i].Other)`；`Hold(bus)` = 寄生電容）。抽完它們有 NextExpr → acyclic 的進 `EvalOrder`、跟其他 node 形成環的**無腦丟進 SCC**（io_db ↔ palette 那種）。驗收：`--trace-cmp` 0 mismatch；coverage 衝到 ~100%（IR-covered = 全部、driving-evaluated = 全部減掉殘餘 SCC）。
- **S4.3 — fixed-K micro-block：把殘餘 SCC（含 γ.4 後落進 SCC 的 bus）都 emit**：對每個 SCC：DFS 斷 back-edge → 對剩下的 DAG 拓樸排序（= 該 SCC Gauss-Seidel 的最佳 in-place 順序）；emit `for k in 0..K_thisSCC: { cur[v]=<NextExpr[v]>; ... }`（`Node(x in SCC)` 讀 in-place `cur[x]`、`Node(x not in SCC)` 讀已算好的、`Hold/Prev` 讀 prev、init `cur[v]=prev[v]`）。**K 用 training 求**：S4 掛著 S1 跑遊戲 ROM、每半週期每個 SCC 跑 `k=1,2,3…` 直到 fixpoint（SCC 內 node 不再變）、記下整局跑下來每個 SCC 的最大 k、寫死（+1 保險）成 per-SCC K（K=2 就直接 unroll 兩份、省 loop branch）。驗收：`--trace-cmp` 0 mismatch。**這步後沒有 S1 fallback**。⚠️ 最有風險（K sizing / Gauss-Seidel 收斂）—— training + 等價 gate 是安全網。
- **S4.4 — bit-slicing：scalar `byte` → `ulong`（64 台並行）**：emit 改 bit-sliced（`Mux(c,a,b)`→`(c&a)|(~c&b)`、`& | ~ ^`、SoA `ulong[nodeCount]`、每個 `ulong` = 64 台的那個 node 的值）；handler chain 也 bit-sliced（master clock toggle `clk^=~0UL`；行為式 RAM/ROM = **per-lane gather/scatter transpose** —— ⚠️**巨坑**：16-bit address 散在 16 個 `ulong` 裡 → AoS↔SoA matrix transpose 出 64 個 `ushort` 位址 → 查各自的 mem → 64 個 `byte` 結果 transpose 回 8 個 `ulong`；用 Hacker's-Delight divide-and-conquer transpose 或 x86 `PDEP`/`PEXT`(BMI2)，**不能** 用純 scalar for-loop（transpose 會比 14723 個 node 的計算還久））。驗收：64 台跑同一 ROM、每台 trace == S1；benchmark 64-wide step 時間 / 64 vs S1。**之後升級 `Vector256<ulong>`（AVX2，256 台）—— C# emit 裡 `ulong`→`Vector256<ulong>` + `Vector256.BitwiseAnd` 等；不要用 `Vector<T>`（runtime width、transpose 沒法寫死）。**`/loop` 跑到 S4.4（`ulong` 版）等價驗過就停（CronDelete）。**
- S4.5（loop 外、之後）— LLVM 後端（LLVMSharp / LLVM-C P/Invoke、emit LLVM IR → `-O3` → JIT/AOT；用法參考 user 的 .NET+LLVM 專案 AprGba）；benchmark 報告「S1 / C#-emit / LLVM-emit / + bit-sliced ×64 / ×256(AVX2)」對照。
- S4.6（loop 外、之後）— **GPU = SIMT compute shader、不是 bit-slice GPU！**（Gemini：bit-slice 上 GPU 會 register pressure 爆 + memory divergence）。Compute Shader（HLSL / ComputeSharp / SPIR-V；參考 `AprNesAvalonia` 的 shader script）：thread ID = 模擬機台 ID，state array = `uint states[NUM_INSTANCES][NUM_NODES]`（或 bit-pack），每個 thread 內部**純 scalar**（`state[15] = state[10] & ~state[5];`）—— GPU 硬體 warp 自動把 32/64 threads 合併成 vector。**所以 S4.1 的 scalar C# emit code 幾乎可以字串替換直接變成 GPU compute shader kernel！**（這也是 S4.1 寫成「scalar、chunked」的理由 —— 它同時是 CPU scalar 路線 + GPU kernel 雛形。）逐 instance 等價；聚合吞吐 benchmark；先批次 evaluator、不做常駐 kernel。
- S4 之後 — `cpu-opt` branch：event-driven IR 直譯器（β），從 ~S3-結束 commit fork。

## 8. Gemini S4 design review（log `tools/knowledgebase/message/20260512_174739.txt`）+ 採用

「S4 的核心挑戰從『如何解析電路』變成『如何餵飽現代微架構（CPU JIT/Cache/ALU 或 GPU 執行緒）』」。逐題（全採用，已反映進 §7）：

1. **S4.1 巨型 C# 方法 → 絕對會出事**（RyuJIT regalloc 崩潰/超時、或 bailout to MinOpts）→ **切 chunk（每 ~500-1000 node 一個 `Step_chunk_N(ref byte state)`，主方法依序呼叫）**；**不要宣告幾千個 local**（炸 stack）；用 `ref byte state` 陣列指標（RyuJIT 對連續存取最佳化好、消 bounds check）；chunk 邊界靠 L1 cache 傳遞、不會變慢（狀態本來就溢出 register）。
2. **fixed-K → per-SCC K、training-based**：每個 SCC DFS 斷 back-edge → 拓樸排序剩下的 DAG = 最佳 Gauss-Seidel in-place 順序；K = 在該順序下「訊號逆 back-edge 漣波傳遞的最大次數」（counter 排 LSB→MSB 通常 K=2）；**不要純靜態算（太複雜悲觀）**—— S4 掛著 S1 跑遊戲、每半週期每 SCC 跑 k=1,2,3… 到 fixpoint、記整局最大 k、寫死（+1）；K=2 就完全 unroll（emit 兩份、連 loop branch 都省）。
3. **hybrid bus → 放 S3↔S4 之間的 IR lowering pass（γ.4），規則用 NMOS 物理式（Wired-AND）不是巢狀 Mux**：`has_pd = OR_i(en_i & !val_i)`、`has_pu = OR_i(en_i & val_i)`、`bus = has_pd ? 0 : (has_pu ? 1 : Hold(bus))`（GND 拉低能力遠強 → pull-down 優先；比巢狀 Mux 扁平、布林更乾淨）；`Hold(bus)` = 寄生電容（C# 就是不重新賦值 / `cur[bus]=prev[bus]`）；跟其他 node 形成環就無腦丟 SCC（Gauss-Seidel 解 bus-latch 耦合）。
4. **bit-slicing word width → 目標 `Vector256<ulong>`(AVX2、256 台)，先用 `ulong`(64) 開發**：`ulong` → state ~117KB（L2/L1）、ALU-bound；`Vector256<ulong>` → ~470KB（L2/L3）、ALU throughput ×4、開始 memory-bound（甜蜜點）。**不要用 `Vector<T>`**（runtime width → transpose 寫不死、噩夢）。路徑：MVP `ulong`（最好除錯）→ 進化 `Vector256<ulong>` + `Vector256.BitwiseAnd` 等（近 10 年 CPU 都有 AVX2）。
5. **漏掉的坑**：(a) **行為式 RAM/ROM 的 gather/scatter transpose = 巨坑** —— bit-slicing 的阿基里斯腱；16-bit address 散在 16 個 ulong → AoS↔SoA matrix transpose → 64 個 ushort 位址 → 查表 → 64 byte 結果 transpose 回 8 ulong；用 Hacker's-Delight divide-and-conquer transpose 或 `PDEP`/`PEXT`(BMI2)，**純 scalar for-loop 轉置會比那 14723 個 node 的計算總和還久**。(b) reset / 不同 ROM per lane → bit-slicing 本來就支援（gather 時用 lane index 查各自的 `byte[][] roms`）。(c) **GPU：千萬不要把 bit-slicing 搬上 GPU！** CPU 適合 bit-slicing（SWAR、core 少、ALU 寬度有限）；GPU 適合 SIMT（硬體 1 thread = 1 instance）；bit-slice 上 GPU = register pressure 爆 + memory divergence。**GPU 正確做法 = compute shader（HLSL / ComputeSharp）**：thread ID = 機台 ID，每 thread 純 scalar（跟 S4.1 的 C# scalar code 一模一樣！），GPU 硬體 warp 自動合併 32/64 threads 成 vector。**S4.1 的 scalar C# emit 幾乎可以字串替換變成 GPU kernel**（這是 S4.1 寫成「scalar、chunked」的理由）。

**總結（Gemini 的修正路線）**：(1) S4.1 Scalar C# 切 chunk + `ref byte state`（= 未來 GPU compute shader 雛形）；(2) S4.2 = γ.4 IR Lowering（bus → Wired-AND/OR pseudo-NextExpr、有環丟 SCC）；(3) S4.3 Fixed-K（DFS 斷環求拓樸序、K 掛 S1 profiling 求出來寫死）；(4) S4.4 Bit-slicing CPU（重點解 gather/scatter transpose、`ulong` 起手後升 `Vector256`）。「這個管線設計非常現代且精準，繼續推進你會得到一個效能怪物」。
