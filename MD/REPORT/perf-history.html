🧱 ProcessQueue 機械碼大小(bytes)— 哪些版真的動了 C#

關鍵洞察:碼變小 ≠ 變快。 最早 05.30 的 native 最小(2,985 B)卻最慢;06.12 的 B1 讓 native 變大(3,807→5,381 B)卻更快(刪的是相依載入鏈)。 而 06.09b/c/d/e 的 native 完全相同(3,860 B) → 這幾版沒動 C# hot path(Rust/診斷/小改),它們之間的 cyc/hc 差異純屬量測噪音。

🎯 量測噪音底線(誠實揭露)

06.09b/c/d/e 的 C# hot path 位元相同(native 全 = 3,860 B),鎖頻下卻量到 cyc/hc 33,661 ~ 35,951(約 7% 散布)。 這給出本量測的相鄰版本噪音底線 ≈ ±5-7% —— memory-bound code 即使鎖頻,cycle 仍受系統干擾/快取狀態影響。 判讀原則:相鄰版差異在 ~5% 內視為持平,只有大跳點(★)是真改善。

✅ 18 vs 19:有退步嗎?沒有

06.19 全是 bit-exact codegen 微優化,native 5,579→5,314 B(更小)。專門的鎖頻 20 輪 interleaved-paired:median 18=32,317 / 19=32,205 cyc/hc, 19 反而快 0.35%、paired 12/20 勝 → 非退步,中性偏微正。(全版掃描裡 06.19 偶爾顯示 +4% 是 per-block ~5% 噪音;paired 才是定論。)

🔬 方法論與限制

  • cycle:QueryProcessCycleTime(Win32,程序實際 CPU cycle,頻率/排程無關)+ 鎖頻 3.6GHz(PERFBOOSTMODE 0)消掉「memory latency 換算 cycle 隨頻率變」的二階效應。
  • load 扣除:cyc/hc=(cycles@400k−cycles@40k)/360000,隔離純 hot-loop(netlist 組裝/power-on 與 hc 無關)。
  • bit-exact 基準:18 版在 400k 同 checksum → 工作量(事件數)完全相同,cyc/hc 差異純為「每事件成本」優化。
  • 噪音:相鄰 ±5-7%(見上),單機單顆 Zen2。boost hc/s 為不鎖頻 best-of,僅供參考(有熱噪)。
  • 未納入:Rust 引擎(本報告只測 C# 發布二進位);per-instruction IBS load 延遲(uProf IBS 不歸因 .NET JIT 程式碼)。
  • 距即時:NES NTSC 需 42,954,552 hc/s → 末版仍約 332× 之遙(這正是持續優化的理由)。
