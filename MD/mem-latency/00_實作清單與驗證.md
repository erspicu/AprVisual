# 相依載入鏈 + branch-miss 架構級優化 — 實作清單與驗證

> branch: **`mem-latency-relayout`**。目標:打破熱路徑「每 pop 的相依記憶體載入鏈」(backend-stall 綁定)+ 降低資料相依 branch 誤預測(#2 痛點)。全程 **bit-exact**、單核、event-driven 不變。

## 1. 瓶頸定位(perf 實證,Pi 5 A76 @ 鎖頻 3.0GHz,最新 S1,400k hc)
- 熱路徑 = `ProcessQueue` 的 RecalcNode:每 pop 一串**相依載入** `nn → NodeInfo[nn] → TransistorList[] → NodeStates[gate]`,位址都取決於上一筆 → pointer-chasing,序列化等記憶體延遲。
- `IPC 2.00` · **backend stall 19.5%** vs frontend 5.1%(→ memory-latency 綁定,非算術/前端)· L1d miss 1.13%(整個 NodeStates ~25KB 已 L1-resident)· **branch-miss ~6 MPKI(~245M,#2 痛點)** · i-cache 0.09% / TLB ~0(皆非痛點)。
- 結論:微優化(ALU/重排/hoist)見頂;剩**架構級**兩條 — (a) 縮短相依鏈跳數、(b) 砍資料相依分支誤預測。

## 2. Gemini 諮詢結論(2026-06-19;log `tools/knowledgebase/message/20260619_2333*.txt`,prompt 在 `temp/gemini_*_prompt.txt`)
- **該做**:① gate-OR 打包(把相依鏈轉成 MLP)② branchless 優先級 max(群組解析)③ LUT 化 dispatch。
- **否決(別做)**:RCM / 空間排序 —— NodeStates 已全進 L1,L1 命中固定 ~4 cycle,聚簇不會更快(對「延遲」無用;我們 pop-order capture +6% 拿的是 OoO 視窗的時間局部性)。denormalize / fat-node = 我們試過的 -6% 計數器(成本從讀側搬到寫側)。
- **存疑**:投機平行兩個 size-1 pop —— Gemini 稱 bit-exact,但 size-1 仍會寫 NodeStates + enqueue 鄰居,平行化可能踩「within-wave reorder = 壞 checksum」死路 → **預設跳過,除非能嚴格證明順序無關**。
- **keeper 規則**:在 #1 相依鏈上 → 用分支(預測對=零延遲);離開關鍵路徑(如解析回寫)→ 用 branchless cmov。

## 4. 逐項驗證流程(每一項都跑完才併)
1. **實作**:在 C# S1(canonical)做最小改動;留 `--no-<feature>` 或編譯開關方便 A/B(若可)。
2. **bit-exact 閘(硬性,先過才看速度)**:`full_palette 400k --extra-ram` 的 NodeStates checksum **必須 == `0x9174E19D961CB6E5`**;不符即否決,不看速度。
3. **A/B —— Zen2 先**:interleaved-paired(預編 base/exp 兩個 binary、每輪交替跑、報 median + trimmed-mean + paired-win-count);sub-1% 一律鎖頻(`tools/cpu_lock_3.6ghz.bat`,測完 restore)。
4. **若 Zen2 無明顯勝負(模稜兩可)→ 拿到 Pi 5 測**(依新規則 [[perf-ab-on-pi5]]):兩版 build arm64、`~/pi_freq_lock.sh` 鎖 3.0G、paired 跑、閘 = checksum + `perf stat` cycles(變異極小);另看 `stall_backend` / `branch-misses` 有沒有實際下降(對應 I-1 / I-2/I-3)。
5. **Rust port**:依 C#/Rust parity policy —— 演算法層 parity、實作層 per-platform 各取所長(branchless 可能 sign-flip,各自決定;Rust 一律 `get_unchecked`)。
6. **記錄**:每項在本檔狀態欄填結果(checksum 通過?Δ% median、paired-win、哪個平台判定);正面確認才併回 main,負面/中性就 revert 並記原因(進死路清單)。

## 5. 執行順序(建議)
**I-2(最易、隔離,先驗整條 A/B+checksum 流程)→ I-1(最高 EV、主菜,直擊 #1 瓶頸)→ I-3(收尾 dispatch)→ 評估 I-4(高風險,多半跳過)。**

預期合計上限 ~10-15%(backend stall 19.5% + branch ~17% 但兩者重疊,非純加總);報酬遞減,以「確認正面的小勝累積」為原則。
