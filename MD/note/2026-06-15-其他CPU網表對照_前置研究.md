# 其他 CPU 網表對照組 — 前置研究(STUDY,2026-06-15)

把 **AprVisual / S1 開關級引擎**套到 6502 以外的 visual6502 系網表,做跨晶片對照,
目的是回答評審最常見的質疑:**「這套 ~136K hc/s 的引擎是不是只對 NES 的 6502 寫死(hardcoded / overfitted)?」**
本筆記整理候選晶片的硬體特性、各自能展示的「火力」、以及不需要 Test ROM 的合成基準測試策略。

> ⚠️ **本筆記是規劃/研究素材(來源:Gemini 諮詢),不是實測結果。** 凡標 **【推測】** 者皆為尚未在本引擎上量測的預測;
> 寫進論文前一律照本專案規矩:**先實際跑、checksum 閘、interleaved-paired**,再把數字補上。事實性硬體規格另標 **【史實】**。

---

## 一、為什麼要做(對照組的學術意義)

- **核心賣點 = 泛用性**:若在**完全不改底層圖演算法與 C# 排程器**的前提下,把別顆網表丟進引擎就能跑出同級距效能,
  即可證明「零組態分類管線 + 電荷分享解析語意」是**通用於整個 NMOS/CMOS 時代的框架**,而非為 6502/NES 客製。
- 三顆候選剛好湊成評估模擬引擎的**黃金三角**(極速 / 剪枝率 / 穩健度),見第三節。
- 接點:本引擎現況 C# 峰值 **135,872 hc/s**(~316× 距即時)、剪枝管線 P-1…P-4(載入期靜態安全證明)、
  自我捕捉初次觸碰重排(relayout)、MOSSIM II 譜系的「最大電容浮接 tie-break」。對照組就是要在這些維度上各推到極端。

## 二、三顆候選晶片

### 1. Motorola 6800 — 防「過度擬合」的同級距 A/B 測試
- **規模/製程【史實】**:約 **4,000 顆電晶體**,**NMOS**(與 6502 同製程、同時代)。開源成熟專案 `perfect6800`。
- **對照意義**:6800 與 6502 同時代、同製程、相近規模,但**微指令架構與匯流排控制邏輯完全不同** → 最乾淨的 A/B。
- **展示指標 = 絕對極速(raw hc/s)**:
  - 節點數僅約 NES(~14.7K)的 1/4 → 整個網路狀態 + 電容矩陣 + 事件佇列幾乎可完全塞進 Zen 2 的 32KB L1d。
  - **【推測】** 在演算法負載大減、近乎零 cache miss 下,hc/s 可能較 NES 倍增,**衝破 300K+**。用來證明底層 C# 框架的輕量化。
- **圖表敘事**:一張「效能垂直起飛」的圖。

### 2. ARM1 — 32-bit RISC + CMOS 的「剪枝奇蹟」
- **規模/製程【史實】**:約 **25,000 顆電晶體**,**CMOS**。visual6502 團隊已逆向(visualarm);人類第一顆商用 RISC、現代手機晶片共祖。
- **與 NES/6502 的三個毀滅性差異**:① 32-bit 資料匯流排/暫存器;② 3 階管線(Fetch/Decode/Execute)→ 每拍內部翻轉劇烈;③ 高度規律 RISC 佈局(重複 ALU + 暫存器檔)。
- **展示指標 = 演算法效率(剪枝率)**:
  - **【推測】** 純 CMOS → 多數節點為「強驅動(strongly driven)」,P-1…P-4 靜態證明可把無效重估率從 NES 的 ~21% 飆到 **80–90%**。「用數學砍掉 80% 執行期運算」比單純跑得快更具震撼力。
  - **【推測】** 絕對 hc/s **未必**比 NES 快(32-bit + 25K 節點,時脈邊緣的事件風暴遠大於 8-bit)→ 正好壓力測試 C# 事件佇列在「高併發節點翻轉」下的吞吐極限。
  - **【推測】** 規律佈局 → 依空間局部性重排後,L1d 命中率對 `cs.AR` 評審極具說服力(relayout 的終極展現)。
- **圖表敘事**:一張「執行期重估次數雪崩式下降」的圖。

### 3. Zilog Z80 — 微排程 / 事件佇列吞吐量的「穩健度試金石」
- **規模/製程【史實】**:約 **8,500 顆電晶體**,**NMOS**(Federico Faggin 設計)。visual6502 有 Z80 網表。
- **三個折磨引擎的特性**:
  1. **披著 8-bit 外皮的 4-bit ALU**:8-bit 加法分兩個子週期跑高/低 4 位 → 每個外部時脈內,網路圖**極頻繁密集的局部翻轉(graph churn)**。考驗事件驅動排程器是否真「零成本抽象」。
  2. **龐大影子暫存器 + 電荷保持**:`AF'/BC'/DE'/HL'`;`EXX` 切換時一大塊網路瞬間被隔離成浮接(floating)。最嚴苛考驗「最大電容浮接 tie-break」——長時間休眠的 NMOS 記憶節點電荷不得異常流失,位元精確不得有失。
  3. **DRAM 更新背景噪音(`R` 暫存器)**:即使沒讀寫外部記憶體,Z80 仍在匯流排空檔狂推 `R` 值到地址線 → 一種「無時無刻的無效翻轉」。考驗靜態剪枝能否精準把無關子網路擋在重估佇列外。
- **展示指標 = 穩健度(robustness)**:絕對速度不會贏 6800、剪枝率不會贏 ARM1,但它是最完整的「最壞情況仍不崩」證明。

| 晶片 | 規模/製程 | 證明的維度 | 一句話 |
|---|---|---|---|

## 四、不需要 ROM 的合成基準測試(pin-level)

開關級模擬下,引擎**不知道何謂 ROM/RAM/指令**,眼中只有節點電壓高低。因此只要在**引腳層級**動手腳,就能造出有學術價值的合成基準。

### 1. 永遠的 NOP(Infinite NOP Sled)
- **目的**:量「基準管線吞吐 + 最佳快取命中」。
- **作法**:CPU 發出讀取信號(R/W=Read)時,**不查 ROM,直接強制 Data Bus 節點 = 該處理器的 NOP 指令值**。
  - Z80 → `0x00`;Motorola 6800 → `0x01`;ARM1 → `0xE1A00000`(`MOV R0, R0`)。
- **效果**:CPU 以為在無邊 ROM 裡狂讀 NOP,PC 無窮 +1,產生極規律週期性翻轉 → 測「最小圖論擾動」下的**最高 hc/s 物理極速**。

### 2. 最大熵亂數注入(Random Bus Fuzzing)
- **目的**:對事件佇列 + BFS 做終極壓力測試。
- **作法**:CPU 讀外部時,用 `Random.Next()` **隨機塞 8/32-bit 垃圾值**到 Data Bus 引腳。
- **效果**:CPU 執行非法指令、亂跳、亂觸發中斷甚至當機;但**物理模擬器不在乎 CPU 當機**,仍忠實計算瘋狂切換的電容與邏輯閘 → 最大 graph churn,暴露**最壞情況效能底線**。
- **注意**:亂數會破壞決定性 → 量測前要固定 seed(可由 `args` 帶入,本引擎腳本不可用 `Random` 於需可重現處;benchmark 需可重跑)。

### 3. 時脈空轉(Reset Hold / Clock-only)
- **目的**:精測「靜態剪枝 + C# 框架」的基礎開銷。
- **作法**:強制 `RESET` 拉到觸發態,只對 `CLK` 灌 0/1 方波。
- **效果**:CPU 卡在準備重置,內部邏輯大多安靜,每拍只有時脈樹相關節點翻轉。**【推測】** 若 ARM1(25K)在 Reset 下重估數被剪到近零,可畫出「零閒置成本(zero idle cost)」的漂亮圖表。

### 論文 Evaluation 可直接引用的英文段落(來源建議)
> *"To isolate the core simulation performance from the memory subsystem and specific workload biases, we employed three synthetic benchmarking strategies at the pin-boundary: (1) **NOP Sled Throughput** to measure optimal pipeline evaluation latency, (2) **Random Fuzzing** to enforce maximum entropy and stress-test the event queue, and (3) **Reset Hold** to evaluate base clock-tree traversal overhead."*

## 八、visual6502 原始 JS 驅動模型 + Node 基準(2026-06-15 實測)

### 8.1 visual6502 怎麼驅動裸 CPU(讀 `ref/visual6502-master` 原始碼)
原始專案是瀏覽器網頁,核心三檔:`chipsim.js`(模擬核心)、`wires.js`(建表)、`macros.js`(驅動)。驅動模型極簡:
- **`setupNodes()`/`setupTransistors()`**:把 `segdefs`/`transdefs` 灌進 `nodes[]`/`transistors{}`;`ngnd=nodenames['vss']`、`npwr=nodenames['vcc']`。
- **`initChip()`(上電重置)**:所有節點 float、gnd/pwr 釘住 → `setLow('res')`、`setLow('clk0')`、`setHigh('rdy')`、`setHigh('irq'/'nmi')` → `recalcNodeList(allNodes())` 全網重算 → 切 `clk0` 8 次 → `setHigh('res')` → `halfStep()` 18 次。
- **`halfStep()`(半週期)**:讀 `clk0`;高→`setLow('clk0')`+`handleBusRead()`,低→`setHigh('clk0')`+`handleBusWrite()`。
- **`handleBusRead()`**:若 `rw` 高 → `a=readBits('ab',16)`、`d=mRead(a)`、`writeDataBus(d)`。**`handleBusWrite()`**:若 `rw` 低 → `mWrite(a, readBits('db',8))`。
- **記憶體**:就是平面 `memory[]` 陣列(`mRead`/`mWrite`),不懂 ROM/RAM/指令。
- **benchmark 迴圈(`goForN`)**:`while(n--){ halfStep(); cycle++; }`——無 trace、無繪圖。
> 重點:驅動裸 CPU **不需要 Test ROM**,只要在 pin 層級餵 data bus(NOP sled = 整片記憶體填 NOP)。這正是第四節「合成基準」的來源。

### 8.2 重要發現:我們的 `.Parse` 已經會讀 raw visual6502 格式
`Sim/WireCore.Parse.cs` 的 `ReadSegDef`(`[node,'+'/'-',layer,…]`)、`ReadTransDef`(含第 7 欄 weak)、nodenames 物件、`LoadExternalArray`(吃 `var segdefs=`/`var transdefs=`/`var nodenames=` 標頭)**全部都在**,只是目前包在 MetalNES `var module={…}` 外層。⇒ 我們引擎載入裸 CPU 網表是「小增量」(直接用這些 reader 組一個 ModuleDef 即可)。但時鐘節點是 `clk0`(非 NES 的 `clk`),且裸 CPU 要自寫 visual6502 式的 reset + halfStep + 平面記憶體驅動(不走 NES 的 cart/PPU handler)。

### 8.3 Node 基準結果(原始 JS 的效能基線)
工具:`tools/visual6502-node/bench.js`(原封不動載入 chipsim/wires/macros,只 stub UI,NOP sled 計時)。Node v24,本機:

| 引擎 | 晶片 | 節點 | 電晶體 | NOP | hc/s(中位) |
|---|---|---|---|---|---|
| **visual6502 原始 JS**(chipsim.js 遞迴 group-walk) | 6502 | 1704 | 3510 | 0xEA | **約 249** |
| | 6800 | 2012 | 3923 | 0x01 | **約 149** |
| | z80 | 3595 | 6781 | 0x00 | **約 166** |
| AprVisual **S1**(最佳化 C# 事件驅動) | 整台 NES(2A03+2C02) | ~14,700 | — | — | **~135,872** |

- **三顆都用同一份未改的 `chipsim.js` 核心跑**;唯一的 chip 專屬碼是各自的 `support.js`(時鐘相位 + 匯流排協定 + setupTransistors 跳過 weak 電晶體),最後載入覆寫 `setupTransistors`/`halfStep`/`initChip`。
- **速度不是只看節點數**:z80 最大(3595 節點)反而比 6800(149)快;6800 的兩相時鐘 `phi1`/`phi2`/`dbe` 每半週期翻動更多節點。決定速度的是 NOP-sled 下「每半週期的 group-walk churn」,不是晶片大小。
- 每顆都**極穩定**(輪間 ±1–3 hc/s),故量級可以很小(6502 8000×3、z80 3000×3 就夠)。
- 這些是**純核心**數字(已剝 UI);官網互動版更慢。
- **對照錨點**:S1 跑**整台 NES**(~8.6× 6502 的矽)達 ~135,872 hc/s,是 6502-JS 的 **~545×**——把「教科書級 JS 參考 → 最佳化引擎」的距離量化出來,可與 perfect6502/metalnes/VisualNes 那組參考基準並列(論文/官網)。

**注意事項(寫進論文前)**:
- **官網沒有「更快的核心」(2026-06-15 查證)**:expert.html 載入順序為 segdefs/transdefs/nodenames/`wires.js`/`expertWires.js`/**`chipsim.js`**/memtable/`macros.js`/testprogram——**模擬核心就是 `chipsim.js`**(我們測的那個遞迴 group-walk)。`expertWires.js` 只是 expert 模式 UI(縮放/滑鼠/URL 參數/canvas 繪圖),不含任何模擬函式;`expert-allinone.js`(1.5MB)只是把同一份核心打包成單檔(`recalcNodeList`/`recalcNode`/`addNodeToGroup`/`getNodeValue` 與 chipsim.js **逐字相同**)。⇒ **249 hc/s 就已經是官網用的演算法**,且官網互動頁實際更慢(每步還跑 canvas/DOM,我們已剝掉)。沒有更快的 JS 核心可量。
- 6502/6800/z80 三顆皆完成;6800/z80 各自載入 `support.js`(其 `setupTransistors` 跳過 weak 電晶體,與上游一致)。
- 與 S1 非 apples-to-apples(不同晶片/語言/簡單 vs 最佳化演算法),只當**參考基線**,不是 head-to-head。

### 8.4 我們引擎(AprVisual.etc)實測 + 直接對照(2026-06-15)
工具:`Sim/WireCore.RawCpu.cs` + CLI `--cpu-bench <dir> --chip <name> [--bench-hc N] [--warmup N] [--rounds N] [--pin] [--no-renumber] [--no-capture]`。沿用既有 `.Parse`(`ReadSegDef`/`ReadTransDef`/`LoadExternalArray`)組 ModuleDef → `AddInstance` → `LowerNetlist` → **class-major renumber(3 趟,如 S1 的 LoadSystem:pass0 抓 prune class、pass1 真跑晶片以自我捕捉 first-touch locality、最終套 range-prune + 該 key)** → `Reset()`(建 fast-path + P-2/3/4 prune 遮罩)→ `RecomputeAllNodes()` 上電。**memory/cache 優化完全與 S1 同步:hot path 全非託管(`int*` 匯流排節點、`byte*` 64K 記憶體;進 timed loop 前 `ClearPostLoadBuildState`+`ReleaseBenchResidualState` 釋放所有託管 build 狀態),小 hot 函式掛 `[MethodImpl(AggressiveInlining)]`(大的 `ProcessQueue` 不 inline,同 S1)。** 再以 visual6502 式 `initChip`/`halfStep` + NOP-sled 驅動。**同機、`--bench-hc 1000000 --warmup 200000 --rounds 6 --pin`,取穩態中位:**

| 晶片 | 節點 | visual6502 JS | **我們引擎(峰值,self-capture)** | **加速** |
|---|---|---|---|---|
| 6502 | 1704 | ~249 | **~145,337**(best 145,436) | **~584×** |
| 6800 | 2012 | ~149 | **~89,233**(best 89,262) | **~599×** |
| z80 | 3595 | ~166 | **~62,491**(best 62,725) | **~376×** |

- **renumber 是大關鍵**:沒有 class-major renumber 時 range-prune 走 safe-degenerate(**P-2/3/4 prune 等於關閉**),6502 只有 109,020 hc/s;一開 renumber → ~145K(**+33%**)。`--no-renumber`/renumber-on/`--no-capture`/self-capture **四種設定 NodeStates checksum 全相同(6502 = 0x0E71…BB3)= bit-exact**(renumber/locality 純屬 perf,不動語意);range-prune verified=True。
- **自我捕捉 locality(S1 峰值法)已加入**:pass1 用我手刻的 settle 副本(`ProcessQueueCapturing`,記每個節點首次被 pop 的順序)在真執行的 NOP-sled 上捕捉熱節點順序。同 session A/B(6502,pin):BFS-key 143,175 → self-capture 145,337 = **+1.5%、bit-exact**。增益比 NES 的 +6.2% 小(這些裸晶片 1704–3595 節點已大致 L1 常駐,locality 空間小);self-capture 只 key 真正觸碰的熱節點(6502 900 個,BFS 是 1686)。
- **三顆「零改演算法」可跑**,AB 位址匯流排確實前進(NOP-sled → PC 遞增,例:6502 0xEAEA→0xAE3A),overfitting 防禦的實證。節點數與 JS 完全一致;電晶體較少是我們去重 (gate,c1,c2)+丟 c1==c2、6800/z80 跳 weak。

**換算成模擬 CPU 時脈**(裸 CPU 不像 NES 要 master/12 分頻:`clk` 直接就是 CPU 時脈,1 CPU 週期 = clk 高+低 = 2 half-cycle,故 **模擬時脈 = hc/s ÷ 2**):

| 晶片 | 峰值 hc/s | **模擬 CPU 時脈** | 真實晶片典型 | 距真實 |
|---|---|---|---|---|
| 6502 | 145,337 | **~72.7 kHz** | 1–2 MHz | ~14–28× |
| 6800 | 89,233 | **~44.6 kHz** | ~1 MHz | ~22× |
| z80 | 62,491 | **~31.2 kHz** | 2.5–4 MHz | ~80–128× |

- 對比 NES:S1 模擬 2A03 的 CPU 時脈僅 **~5.66 kHz**(135,872 hc/s ÷ 24,因 master/12 再 ×2 half-cycle;距 1.789 MHz 真實 ~316×)。**裸 6502 ~72.7 kHz 比 NES 2A03 高約 13×**——正是「非 NES 不用分頻 + 只有一顆晶片(無 PPU)」的效果,離真實時脈也近得多(~14–28× vs NES 的 ~316×)。
- 趣味換算:visual6502 原始 JS 跑 6502 只有 ~249 hc/s ≈ **模擬 ~124 Hz** 的 6502。
- **已是 S1 完整方法學**:lowering + P-2/3/4 prune + class-major renumber + **自我捕捉 first-touch locality** + 非託管 hot path + inline cascade + pin,全部到位(見上)。原先「需 ClockNode/callback 才能自我捕捉」的卡點,已用裸 CPU 自己的 settle 捕捉副本解決。
- 一句話:**同一顆 6502,visual6502 原始 JS 249 hc/s(≈ 0.12 kHz)vs 我們引擎 145,337 hc/s(≈ 72.7 kHz)= ~584×**。

### 8.5 語言 vs 演算法分解(C# naive 移植,排除語言差異)
JS 249 vs 我們 145K 混了兩個變因:(a)JS 直譯器 vs C#、(b)naive 遞迴 group-walk vs 我們的事件驅動+prune。為了**純看演算法貢獻**,把原始 `chipsim.js` 演算法**忠實移植成 C#**(`Sim/NaiveSim.cs`:object-per-node/transistor、遞迴 `addNodeToGroup`、線性 group 成員判定、每次全群重解、**無 prune/fast-path/SoA**;電晶體不去重、不丟 c1==c2,計數與 JS 完全相同)。CLI:`--cpu-bench <dir> --chip <name> --naive`。同機、同 NOP-sled、`--pin`:

| 晶片 | JS naive | **C# naive** | C# ours(峰值) | **語言(JS→C#)** | **演算法(naive→ours)** | 總計 |
|---|---|---|---|---|---|---|
| 6502 | ~249 | **~17,914** | ~145,337 | **~72×** | **~8.1×** | ~584× |
| 6800 | ~149 | **~12,582** | ~89,233 | **~84×** | **~7.1×** | ~599× |
| z80 | ~166 | **~12,255** | ~62,491 | **~74×** | **~5.1×** | ~376× |

- **誠實的關鍵發現**:總加速的**大頭是語言**(JS 直譯器 → C#,同演算法,**~70–85×**);我們的**演算法本身只貢獻 ~5–8×**(同語言,naive → 事件驅動+P-2/3/4 prune+renumber locality)。這正是論文最在意的「語言 vs 演算法」混淆——之前 JS 對 C# 的比較會把語言紅利誤算進演算法;現在用同語言量,演算法的真實貢獻是 ~5–8×。
- C# naive 的電晶體數與 JS **完全一致**(6502=3510、6800=3923、z80=6781),確認是忠實移植、非偷工。三顆 AB 皆前進。
- naive 版很慢(~12–18K hc/s),故用小量級(2–3 萬 hc);穩定度高。
- 寫論文時:**主張應是「在受控的同語言(C#)下,事件驅動+剪枝較 naive 開關級快 ~5–8×」**,而非把 584× 當演算法功勞。

### 8.6 對照 1980 年代實機時脈(我們模擬 vs 當年機器)
> 註:這裡的「6800」是 Motorola **6800(8-bit)**,非 68000(16/32-bit,我們無此網表)。我們的模擬 CPU 時脈取 §8.4 峰值 ÷ 2。

| 晶片 | 我們模擬時脈 | 代表 1980s 個人電腦(實機時脈) | **距當年實機** |
|---|---|---|---|
| 6502 | ~72.7 kHz | Apple II / Commodore 64(~1.02 MHz)、BBC Micro(2 MHz) | **~14×**(C64)/ ~27×(BBC) |
| 6800 | ~44.6 kHz | SWTPC 6800 / Altair 680 等(~1 MHz) | **~22×** |
| z80 | ~31.2 kHz | ZX Spectrum(3.5 MHz)、MSX(3.58 MHz)、Amstrad CPC(4 MHz) | **~112–128×** |

- 解讀:我們在單核把**裸 6502 模擬到約當年 Apple II/C64 的 1/14 速度**;6800 約 1/22;z80 最差(約 1/112–1/128,因節點/電晶體最多、且當年 z80 機器時脈最高 3.5–4 MHz)。
- 對比 NES:S1 模擬整台 NES 距真實 ~316×;裸 CPU(無 PPU、無 master/12 分頻)離「當年實機速度」近得多(6502 僅 ~14×)。
- 真要追平當年實機(例:6502 再 ~14×、z80 再 ~128×)單核仍不可達,但這量化了「離可互動模擬有多遠」,比抽象的 hc/s 更直觀。

### 8.7 三種 pin-level 合成負載實測(NOP / Fuzz / Reset-Hold,2026-06-15)
第四節設計的三種合成基準現已實作(C# `--workload nop|fuzz|reset`、node `--workload`),三晶片各跑。

**關鍵:fuzz 必須排除「鎖死碼」否則量到的是鎖死的 CPU、不是壓力。** 隨機 opcode 會餵到讓 CPU 永久停擺的非法碼;未排除時 6502 fuzz 假快到 231,844 hc/s(post-warmup AB=0xFFFF 卡死),排除後才 70,941(真壓力)。鎖死碼清單由 **Gemini(2026-06-15)** 給(已寫進程式,re-roll 跳過):
- **6502**:`02 12 22 32 42 52 62 72 92 B2 D2 F2`(12 個 KIL/JAM;鎖死後 AB 持續驅動 0xFFFF,只有 RESET 能救)。ref: Adam Vardy《Extra Instructions of the 65XX》/ Visual6502。
- **6800**:`3E(WAI) 9D DD ED FD(HCF, halt-and-catch-fire)`(5 個)。ref: Wheeler《Undocumented M6800》BYTE 1977。
- **z80**:`76(HALT)`(1 個;z80 無 crash opcode,未指派碼皆為 NOP/鏡像)。ref: Young《Undocumented Z80 Documented》。

**C#(我們引擎,self-capture + pin,1M hc × 6 round 中位,同 session):**

| 晶片 | NOP | Fuzz(排除鎖死碼) | Reset-Hold |
|---|---|---|---|
| 6502 | 149,736 | **70,941** | 155,915 |
| 6800 | 88,313 | **114,222** | 165,764 |
| z80 | 62,952 | **43,955** | 161,838 |

**JS(原始 visual6502,Node;小量級,同模式):** 6502 215/182/311、6800 152/212/248、z80 168/152/224(NOP/Fuzz/Reset)。

觀察:
- **Reset-Hold 全部最快、且三顆相近(C# ~156–166K)**——chip 卡在 reset、只剩時鐘樹活動,量到的是「引擎基礎開銷 + 時鐘分配」,近乎與晶片無關。
- **Fuzz vs NOP 隨晶片、且 C# 與 JS 同號**:6502、z80 的 fuzz **比 NOP 慢**(真壓力:隨機執行的節點 churn 比規律 NOP 多);**6800 的 fuzz 反而快**(6800 的 NOP=0x01 是相對重的指令,隨機平均更輕)。⇒ 第四節「fuzz = 最壞情況」的先驗預期**只對 6502/z80 成立**,6800 反例。
- fuzz 餵值改走 LCG(固定 seed,可重現)不讀記憶體陣列;reset-hold 跳過整個 bus、且 init 不釋放 reset。

### 8.8 三語言 naive 對照(加入 C++,2026-06-15)
為把「語言」這軸量到底,把原始演算法(chipsim.js 遞迴 group-walk)也移植成 **C++**(`tools/cpp-naive/naive.cpp`,clang++ -O3;載入 C# 匯出的**同一份** raw 網表,保證資料一致),NOP sled、三晶片:

| 晶片 | JS naive | C# naive | **C++ naive** | C# ours | C++/C# naive | 演算法(C#naive→ours) |
|---|---|---|---|---|---|---|
| 6502 | 249 | 17,914 | **26,239** | 149,736 | **1.46×** | ~8.4× |
| 6800 | 149 | 12,582 | **17,137** | 88,313 | **1.36×** | ~7.0× |
| z80 | 166 | 12,255 | **18,452** | 62,952 | **1.51×** | ~5.1× |

**最強的語言結論**:**同一個 naive 演算法下,C++ 只比 C# 快 ~1.4–1.5×**(native vs managed)。所以 managed C# **不是瓶頸**——native 也只多買到 ~1.5×;真正的離群值是 **JS 直譯器(比 C# 慢 ~70–85×、比 C++ 慢 ~105×)**。這直接、量化地佐證論文的「受控語言可行、增益來自方法」論點(比原本只用 S0≈C++ MetalNES 更乾淨:這是三語言、同 naive 演算法、同網表)。
- 換句話說:JS→ours 的 ~376–599× 真正拆解是 **語言(JS 直譯器,~70–105×)× 演算法(~5–8×)**,而「managed vs native」只佔語言那軸的 ~1.5×。
- 工具:`src/AprVisual.etc --export-netlist`(匯出 naive 網表)→ `tools/cpp-naive/naive.cpp`(C++ 同演算法載入同網表)。C++ 未 `--pin`(無此旗標),但語言差距遠大於釘核 ~10% 效應,結論穩健。

### 8.9 用 C++ 寫「我們的(最佳化)演算法」(bit-exact 驗證,2026-06-15)
把 **OUR 演算法**(非 naive)也移植成 C++(`tools/cpp-ours/ours.cpp`)。做法:C# `--export-engine` 匯出**已建好的引擎狀態**(post-lower、identity id=`--no-renumber`、初始化後的 per-node flags/state/pruneMask/connections + lowered 電晶體 + name→id);C++ 重建鄰接(同 `Reset`)、複製正規熱路徑 **SoA + 事件驅動 double-buffer settle + 256-LUT 群解析 + 浮接 largest-cap tie-break + P-1..P-4 遮罩 prune**。

**移植範圍(誠實)**:含上述架構核心;**刻意不含** bit-exact 中性的微優化——inline-payload 節點佈局、cls1/cls2 fast-path、B1 pair path、class-major range-prune、self-capture locality。所以這是「我們演算法的**架構**在 C++」,**不是 peak-vs-peak**。

**bit-exact 驗證:三顆的 final NodeStates checksum 與 C# `--no-renumber` 完全相同** ⇒ 忠實移植:

| 晶片 | **C++ ours-core** | C# `--no-renumber` | C# ours(完整微優化) | final checksum(C++==C#) |
|---|---|---|---|---|
| 6502 | **69,319** | 110,160 | ~149,736 | 0x1948E581C9666553 ✓ |
| 6800 | **36,828** | 73,358 | ~88,313 | 0x2216CCB5330A83DB ✓ |
| z80 | **25,603** | 49,775 | ~62,952 | 0x8F396CDADAD74F9C ✓ |

**關鍵解讀(避免誤讀)**:
- C++ ours-core **比 C# 慢**,但**這不是語言輸**——是 C++ 版**少了那一整套微優化**(inline-payload / fast-path / B1 / range-prune / self-capture)。C# `--no-renumber` 還有 fast-path/inline(只是關了 range-prune),C# 完整版更全;C++ 核心兩者都沒有。
- 對照 §8.8:**同 opt 等級(naive)時 C++ 反而比 C# 快 ~1.4–1.5×**。所以「語言」這軸 native 領先;C++ ours-core 落後純粹因為缺微優化。
- ⇒ C++ ours-core 落後純粹因為缺微優化;**完整移植見 §8.10**。
- 工具:`--export-engine` + `tools/cpp-ours/ours.cpp`(clang -O3)。匯出檔/執行檔 gitignore(授權衍生 + 本地二進位)。

### 8.10 C++ 完整最佳化版(全微優化)+ 兩個現象(2026-06-15)
把 §8.9 補成**完整**:`tools/cpp-ours/ours_full.cpp` 移植了引擎的**整條熱路徑**——**packed 16-byte NodeInfo 結構**(inline-payload 與 overflow Tlist 共用 union,4 個一條 cache line)、cls1/cls2 fast-path 派發 + RecalcNodeFast、B1 兩節點 pair path、SetNodeState 的 class-major **range-prune**、double-buffer settle、以 perm 走**原序 checksum**。靠 `--export-engine-full` 載入 post-renumber+self-capture 的引擎狀態,**三顆 final checksum 與 C# 完整版完全相同**(bit-exact):

| 晶片 | **C++ ours-FULL** | C# ours FULL | C#/C++ | final checksum(一致) |
|---|---|---|---|---|
| 6502 | **118,918** | 151,561 | **1.27×** | 0x1948E581C9666553 ✓ |
| 6800 | **61,630** | 89,924 | **1.46×** | 0x2216CCB5330A83DB ✓ |
| z80 | **40,255** | 62,631 | **1.56×** | 0x8F396CDADAD74F9C ✓ |

**現象一:cache 佈局本身就是優化。** 一開始 C++ 用**分離 SoA 陣列**(flags[]、inline[]、payload[]… 各一條),讀一個節點要碰好幾條 cache line;6502 ~117K。改成**packed 16-byte 結構**(對齊 C# 的 NodeInfo,一條 cache line 裝 4 個)後 ~119–124K(這些 1.7–3.6K 節點的小晶片已大致 L1 常駐,差異被熱雜訊蓋住、約 +0~6%;在更大工作集(如整台 NES,當初 S2-A 量到的場景)才會明顯)。教訓:忠實移植「我們的演算法」必須連**佈局**一起移植,不只邏輯。

**現象二:naive→完整,語言優勢反轉。**
- naive(§8.8):C++ 比 C# **快** ~1.4–1.5×(簡單迴圈,native 省掉 GC/JIT 開銷)。
- 完整(本節):**C# 反而比忠實 C++ 移植快 ~1.27–1.56×**。
- 原因:C# 完整熱路徑是針對 **.NET JIT + 動態 PGO** **重度微調**的(`[AggressiveInlining]` cascade、ulong 雙對讀、按 `[cond-profile]` 排序的 `&&` 分支……),而我的 clang `-O3` 是**忠實但未做指令級微調**的移植(雙對讀簡化成逐對、無 PGO)。⇒ **「重度微調的 managed」贏過「忠實但未微調的 native」,不是語言天花板**;同等微調投入(把 ulong 讀、clang PGO、針對 optimizer 重構也做上)native 很可能追平或反超。
- 誠實總結:**在這個尺度,語言差距很小(~1.5× 任一方向),且由「逐指令微調 + PGO」主導,不是 managed-vs-native 的本質差異**;連把整套微優化移植到 native,也只在 ~1.5× 內。
- 進 hot path 前 free 掉非熱用的 array(name→id map),比照 C# 的 `ClearPostLoadBuildState`/`ReleaseBenchResidualState`。工具:`--export-engine-full` + `tools/cpp-ours/ours_full.cpp`。
