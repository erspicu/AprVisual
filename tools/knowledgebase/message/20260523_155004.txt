## 3. **Step 3.5 撞牆 —— 重點在這**

### 3.1 嘗試:把 133 個 ALU 內部節點都 mark CodegenOwned

原始 `--dump-block` 找到 ALU 的 reverse-closure 是 133 個 node + 477 個 transistor(reverse BFS from alu outputs,stop at pull-ups + declared inputs)。

我把這 133 個都標 `CodegenOwned`,讓 RecalcNode 看到就 return,並 dispatcher 寫 alu/notalu/alucout outputs。

**結果**:
| Mode | rate | D (RecalcNode/hc) | trace |
|---|---|---|---|
| baseline (S1) | 38,662 hc/s | 614.2 | reference |
| own 133 closure | **47,049 hc/s (+22%)** | **521.1 (-15%)** | **CPU 卡死 PC=00FF** |

22% speedup **是 fake** —— CPU 停在 reset 沒進 ROM,所以「省」掉很多 work。

### 3.2 為什麼:reverse-closure 自然包含 CPU 關鍵節點

```
# 實際的 133 closure 名單(部分):
cpu.#aluresult0..7    ← 真的 ALU mid (8 nodes)
cpu.a0, cpu.a4        ← 累加器 bits! 不該屬於 ALU
cpu.adh0..7  (8)      ← address bus high bits!
cpu.adl0..7  (8)      ← address bus low bits!
cpu.C78.phi2          ← clock phase 2!
cpu.cclk              ← CPU clock!
cpu.cp1               ← clock phase 1!
... 93 個 anonymous mids (有些是 ALU 真內部,有些是 shared bus/clock mids)
```

所有上述「不該」節點的 `pullups=0`。原 `--dump-block` 的「stop at pull-up」啟發式擋不下它們。它們透過 pass transistor 跟 ALU output 共群。

OWN 它們 → S1 不更新它們 → `cpu.adh*` (memory address bus) 卡 0 → CPU 讀錯記憶體 → 卡死。

### 3.3 保守 subset(只 own 8 個 `cpu.#aluresult` mids)

| Mode | rate | D | trace |
|---|---|---|---|
| own 8 #aluresult only | 37,635 hc/s (-2.7%) | 612.9 (-0.2%) | 64-line trace diff |

**完全沒省到工作**。D 幾乎沒變。

### 3.4 為什麼:`CodegenOwned` 在當前架構下的真實語意

S1 的 group-resolution(看 `WireCore.Group.cs`):

```csharp
private static byte ComputeNodeGroup(int nn) {
    _groupFlags = NodeFlags.None;
    _groupCount = 0;
    AddNodeToGroup(nn);              // BFS BFS BFS the entire connected group
    return GetNodeValue();
}

private static void AddNodeToGroup(int nn) {
    if (_inGroup[nn] != 0) return;
    _inGroup[nn] = 1;
    _groupBuf[_groupCount++] = nn;
    _groupFlags |= NodeInfos[nn].Flags;
    // walk channels to normal nodes
    int* p = TransistorList + NodeInfos[nn].TlistC1c2s;
    while (*p != 0) {
        int gate = *p++; int other = *p++;
        if (NodeStates[gate] != 0) AddNodeToGroup(other);   // recurse through ON transistors
    }
    // GND / VCC checks ...
}

private static void RecalcNode(int nn) {
    if (CodegenOwned[nn]) return;    // ★ skip entry, but...
    byte v = ComputeNodeGroup(nn);
    for (int i = 0; i < _groupCount; i++) SetNodeState(_groupBuf[i], v);   // writes ALL group members
}
```

**問題**:`CodegenOwned[nn]` 只 skip `RecalcNode(nn)` 這個入口。可是 `ComputeNodeGroup(mm)`(另一個非 owned 節點 mm)的 BFS 還是會 traverse nn(透過 channel),把 nn 拉進群裡共同決定 value,然後寫回 nn。所以 OWN nn 不阻止 group walk 工作。

**只有當 owned region 的每個 starter 都被 skip 時**才看到 D 下降 —— 但這時 region 內部群也不再被 resolve,所以裡頭那些「不該 OWN 但被掃進來」的關鍵節點停留在錯誤值。

### 3.5 推論:Step 4(LLVM emit)在當前架構下沒意義

LLVM emit 是讓 dispatcher 的 per-block eval 從手寫 C++(3.97 ns/call)變成 LLVM 自動 emit。但 dispatcher 本身已經比 S1 group walk 快 9-18×。**瓶頸不在 dispatcher**,在「S1 group walk 對 owned region 沒被 skip 掉」。

沒架構改變,Step 4 也不會加速。

## 4. 三個架構級選項(請你判斷)

### A. 物理移除 transistor

在 LoadSystem + Reset 之後、跑模擬之前,從 `TransistorList` / `Nodes[].C1c2s` / `Nodes[].Gates` 中刪掉 owned region 的內部 transistor。
S1 看到的 netlist 變小 —— owned 區只剩 boundary node 直接連 supply rails。Dispatcher 負責整個 owned region 的所有節點計算。

**優點**:S1 真實做更少工作。speedup 直接可量。
**風險**:
  - 邊界節點的 group 解析會被破壞(pull-up / capacitance 計算邏輯依賴完整的 transistor 結構)
  - 必須完整接管 owned region 的所有 NodeStates 寫(否則下游讀到陳舊值)
  - 不可逆(rebuild 才能恢復原 netlist)
  - **核心難題**:如何處理「owned region 跟 boundary 之間」的 transistor?保留或刪除?

### B. 修改 ComputeNodeGroup 的 BFS,遇到 CodegenOwned 就停

```csharp
private static void AddNodeToGroup(int nn) {
    if (CodegenOwned[nn]) {
        // 視為「supply-like」固定值:把 NodeStates[nn] 當作 driver flag 加進去
        _groupFlags |= NodeStates[nn] == 0 ? NodeFlags.Gnd : NodeFlags.PullUp;
        return;   // 不展開、不加入 _groupBuf
    }
    // ... 原邏輯
}
```

Dispatcher 在 input 變動時先寫 NodeStates 給 owned region 的 boundary,group walk 從邊界看到固定值,跳過 owned 內部。

**優點**:
  - 不改 netlist topology(可逆,A/B 比較容易)
  - 比 A 工程量小很多
  - 跟現有 Phase 2 P2.3 `IrBoundaryDriver` 路徑類似(已有先例)
**風險**:
  - capacitance / "largest connections wins" 邏輯失準(因為 owned 內部沒被算進 `_maxConnections`)
  - 對動態節點(無 pull-up 的 dynamic-hold storage)語意可能不對

### C. Branch wind-down

承認 S1 group-resolution 模型不適合 codegen 接管。Codegen 已驗證在「dispatcher 黑盒 + functional 一致」層級可行,但加速天花板來自 S1 架構,不是 dispatcher 不夠快。Memory 補一筆,branch 收。

### Q2 & Q3: 架構級選項分析 (A vs B vs C vs D)

#### 為什麼不選 A (物理移除 Transistor)?
- **核心難題無法解決**：如果你刪除 owned 區內部的電晶體，boundary node 上的 capacitance (面積/pull-up 等效強度) 邏輯會被徹底破壞。Visual6502 的 "largest group wins" 依賴完整的 graph 來比較驅動能力。
- **文獻先例**：在 Switch-level sim (如 R.E. Bryant 的 COSMOS) 中，確實有將子圖替換為 boolean block 的做法。但他們的做法是**嚴格定義方向性 (Directionality)**，把網路轉成有向圖後再分割。Visual6502 是無向圖（除 Gate 外），物理切割極易引發訊號無法正確雙向傳遞（如 pass-transistor）的 Bug。

#### 我提議的 Option D:「嚴格 Gate 邊界」+ Option B 阻斷
你目前的困境（133個節點拉了一堆 CPU bus 進來）是因為 BFS 沿著 source/drain channel 亂竄。
**唯一在物理與邏輯上安全的 Macro-block 邊界，是「單向隔離」的節點。在 MOSFET 裡，就是 Gate。**

如果你要做最後一次架構突破嘗試，請試這個 **Option D**：
1. **重新定義 Partitioning (Step 3)**：
   - 區塊的 Boundary 只能是 **Transistor Gate** 或者是 **強驅動輸出 (Pull-up/Pull-down network 的最終匯出點)**，絕對不能經過 Channel (pass-transistor) 切割。
   - 如果 ALU 的輸出是透過 pass-transistor 連接 address bus，那麼 address bus *必須*留在 S1 的管轄區。
2. **實作 Option B (BFS 阻斷)**：
   ```csharp
   private static void AddNodeToGroup(int nn) {
       if (CodegenOwned[nn]) {
           // 將 Dispatcher 算好的結果當作「無限強度的供電軌」
           _groupFlags |= NodeStates[nn] == 0 ? NodeFlags.Gnd : NodeFlags.PullUp;
           return; // 🛑 絕對不把它加進 _groupBuf，阻斷 BFS
       }
       // ... 原邏輯
   }
   ```
3. **Dispatcher 的職責**：Dispatcher 監聽 ALU 的輸入 (Gates)，算出 ALU 的輸出 (8 個 node)，然後寫入 `NodeStates`。當 S1 掃描到這 8 個 node 時，Option B 的邏輯會讓 S1 以為遇到了 VCC/GND，從而終止 BFS，並吸收 Dispatcher 的正確數值。

**這條路徑的優點**：不改 Topology、可逆、S1 真實 workload 下降、解決了 133 closure 亂吸節點的問題。

### Q5: 專案戰略與工程價值

既然 CPU 算不動底層，我們就不要讓 CPU 算底層。目標應該轉向 **Static AOT Transpilation (靜態重編譯)**，而 `math-algos` 是這個編譯器的前端。

3. **這就是 MetalNES 的路線**：MetalNES 不是在 runtime 做 BFS，它是 pre-compiled 的。
