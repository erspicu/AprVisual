## 1. S1 目前的核心演算法（baseline、要替換 / 增強的對象）

**每半週期的工作流程**（`WireCore.Recalc.cs` + `.Group.cs`）：

```
StepCycle:
  RunHandlerChain()                              # clk toggle、memory/video callbacks
  ProcessQueue():                                # BFS over dirty-set
    while RecalcListNext non-empty:
      swap current/next queue (double-buffered)
      for each nn in current queue:
        if RecalcHash[nn] != 0:                  # dirty flag #1（佇列 dedup）
          RecalcHash[nn] = 0
          RecalcNode(nn):
            ComputeNodeGroup(nn):
              clear _inGroup of previous group   # O(prev group size)
              AddNodeToGroup(nn) (recursive):
                if _inGroup[nn] != 0: return     # dirty flag #2 / early return（群 dedup）
                _inGroup[nn] = 1
                _groupBuf[_groupCount++] = nn
                _groupFlags |= NodeInfos[nn].Flags
                # walk channels to normal nodes / GND / VCC:
                for each transistor t in TlistC1c2s[nn]:
                  (gate, other) = t
                  if NodeStates[gate] != 0:      # transistor conducts iff its gate is high (NMOS)
                    AddNodeToGroup(other)
                for each t in TlistC1gnd[nn]: if NodeStates[gate]: _groupFlags |= Gnd
                for each t in TlistC1pwr[nn]: if NodeStates[gate]: _groupFlags |= Pwr
              newState = FlagsToState[_groupFlags & 0xFF]   # 256-entry LUT
            for each m in _groupBuf:
              SetNodeState(m, newState):
                if NodeStates[m] == newState: return        # early return #1（最關鍵！）
                NodeStates[m] = newState
                for each t in TlistGates[m]:                # m gates these transistors
                  enqueue both channel endpoints of t       # gate-fanout enqueue
  InvokeCallbacks()                              # memory writes、video frame writes
  Time++
```

**關鍵速度來源**：

### 候選 A：Union-Find（with path compression + union by rank）替代 BFS group walk

**怎麼做**：每半週期、不做 DFS。而是維護一個 Union-Find 結構，當一個 transistor 切換導通狀態時、做對應的 `Union(c1, c2)` 或 `分離`（這後者 UF 不支援、要 Link-Cut Tree 或 Euler-tour）。Group 解析時 `Find(nn)` O(α(N))。

**我的複雜度估算**：
- 純 incremental Union（不支援刪邊）：O((D + M_conducting_changed) · α(N))/半週期
- 但 conduction 會*兩向*變（gate 改變既加邊也刪邊）→ 需要 Link-Cut Tree、O(log N) per op

**贏面**：算法上同階、cache 可能好一點。**輸面**：實作複雜（LCT）；group size *通常很小*（~1），那 α(N) ≈ log N ≈ 14、可能輸 S1 的 O(1)。

### 候選 B：稀疏 boolean 矩陣 SpMV-style「frontier expansion」

**怎麼做**：把當下「導通的 transistor channel 圖」表示成 N×N 稀疏 boolean 矩陣 A（CSR）。一個 group BFS = 從 dirty bitvector 出發、反覆 `frontier' = A · frontier`（boolean SpMV）直到 fixed point。所有 dirty group 可以*同時*expand（一次 SpMV、所有 group 平行進展一步）。state 用 bit-packed 表示（N/64 個 ulong）→ SpMV 變 bitwise OR + AND 的 SIMD 操作。

**我的複雜度估算**：
- 每次 SpMV：O(nnz(A) / 64) ≈ O(M / 64) ≈ ~420 個 AVX2/SIMD 字 / half-cycle
- 收斂 iteration：O(max group diameter) ≈ O(log G_max) ≈ 3-5 次
- 半週期總 cost：~2K SIMD ops vs S1 的 ~10K scalar transistor 檢查 ≈ 算上 SIMD 寬度（AVX2 = 4× ulong64）大概 *似乎* 贏 5-10×

### 2. 關於最被看好的 H (Incremental SIMD Frontier) 與矩陣重建 (回答 Q4, H)

### 3. Bit-packing 與 Sparse Matrix Layout (回答 Q3)

如果你堅持要玩 SIMD（作為探索），在 $N=14.7K$，平均 Degree 3-5，Max 30+ 的極度不規則分佈下：

*   **CSR:** 不利於 SIMD，因為每個 Row 長度不同，SIMD Lane 利用率低，且分支發散嚴重。
*   **ELLPACK:** 絕對不行。為了迎合 Max Degree 30，你的每一列都要 Padding 到 30。平均只有 3-5 的資料會讓你浪費 80% 以上的記憶體頻寬在讀取 `0`。
*   **SELL-C-$\sigma$ (Sliced ELLPACK):** **這是唯一適合你 Workload 的格式。**
    它會先把 Nodes 依照 Degree 排序（或近似排序），然後切塊（Chunking，例如 C=4 對應 AVX2 的 4 個 64-bit integer）。每塊內部的 Padding 只取決於該塊的最大 Degree。
    *但注意：因為你做了排序，Node ID 邏輯上不再連續，你需要維護一個 `Original_ID <-> Sorted_ID` 的 LUT，每次存取 State 都要多一次查表（Cache Miss 熱點）。*

### 4. 真正有希望的方向 (回答 Q2, Q5)

如果限制在 `math-algos` 這個不碰 IR 的沙盒裡，只有以下幾種做法能擠出效能：

#### 策略一：你提的 G (Static Graph Analysis + Cache Layout)
**這非常有用，無副作用。**
1.  用 Cuthill-McKee (RCM) 或 Spectral Bisection 對 Node 重新編號。
2.  目標：讓經常一起進入 Dirty-Queue 的 Node 在 `byte* NodeStates` 和 `NodeInfos` 中的記憶體位置相鄰。
3.  這樣 BFS 推進時，Cache-line (64 bytes = 64 個 Node states) 會被完美利用。
**預期收益：1.1x ~ 1.25x。**

#### 策略二：混合架構（邏輯閘靜態排序 vs 傳輸閘動態擴散）(我的建議)
你提到：**86% 是 logic gate（小 group），只有 14% 是 pass-transistor bus。**
你現在 S1 的做法，是把 100% 的電路都當成「雙向傳輸閘」在跑動態 BFS。這太虧了！

Logic Gate (NAND/NOR/INV) 是 **單向 (Unidirectional)** 的。
就算你不做 IR 轉譯，你可以在 Build-time 分析圖形：
1.  找出那些純粹由 Pull-up 網路和 Pull-down 網路組成、沒有 Pass-transistor 介入的 Nodes（這佔了 86%）。
2.  對這些 Nodes 進行 **拓撲排序 (Topological Sort)**。
3.  把它們的計算變成一長串的 Dense Array 操作，不再用 Queue，不再用 BFS。
4.  只對那 14% 的 Pass-transistor Bus 啟動 BFS Group 解析。
*這不是把 Gate 抽象化，只是把 Update 順序靜態化，避開 Queue 操作。*

#### 策略三：Bit-Slicing 平行宇宙 (SIMD 的正確用法)
如果你想把 SIMD 用到極致，既然單一 NES 內部的線性代數化行不通（因為 $D \ll N$），那就 **平行跑 64 台或 256 台 NES**。
把你所有的 `byte* NodeStates` 變成 `ulong* NodeStates`（代表 64 個獨立的模擬實例）。
你的 S1 BFS 邏輯完全不用改，只要把 `newState` 的計算變成 bitwise 操作。
這種叫做 **Bit-sliced Simulation**。雖然沒辦法讓「單一使用者的 0.07 FPS」變快，但可以讓「總體吞吐量 (Throughput)」直接暴增 64 倍（適合拿來跑機器學習訓練或大規模自動測試）。

### 5. 整體判斷與天花板 (回答 Q6)

**這條 branch（純矩陣/代數化、無視電路特徵的通用演算法）的極限在哪？**

*   **願意潑冷水的結論：這條路的盡頭是「變慢」。**
    你試圖用處理 Dense/Data-parallel 的數學工具 (矩陣、SpMV)，去硬解一個高度 Sparse/Control-flow-heavy 的問題 (Event-driven Netlist)。這在計算機結構上是逆天而行。
*   **S1 的 47K hc/s 能靠這些演算法推到哪？**
    如果硬上 H 或 B，速度會掉到 ~10K - 20K hc/s (0.2x ~ 0.5x)。
    如果只上 G (記憶體重排)，大約能到 **~55K - 60K hc/s (1.2x)**。
    如果你願意採用「策略二 (區分 86% 單向邏輯做靜態排程)」，這依然沒有動到 IR，但可能可以推到 **~100K - 150K hc/s (2x - 3x)**。

**給 User 的最終建議：**
「我們已經評估了線性代數與 SpMV 的方向。由於 NES 切換級模擬的每個半週期活動率極低 (~3%)，將其轉化為矩陣運算會導致計算量從 $O(Dirty)$ 膨脹到 $O(Nodes)$，即使有 SIMD 加持，基礎計算量的 30 倍暴增也無法彌補。目前主流的 SpMV 優化無法對抗 Baseline 已經極限最佳化的 $21\mu s$ 事件驅動迴圈。這條 Branch 作為技術探索可宣告『此路不通』。後續應專注於靜態圖形 Cache Line 佈局最佳化 (G)。」
