### 1. 核心問題：為什麼 `Prev(data)` 會錯？如何解？

你把 `Node(data)` 換成 `Prev(data)` 的前提是「`data` 在 clock edge 前就穩定了」。
但在 NES（特別是 NMOS 邏輯）中，derived clock（如 `cclk`, `cp1`）是在同一個半週期內從 master clock 衍生出來的。
如果 `9118` 的 latch 在這個半週期是 **Transparent（導通的）**，且它上游的 register 也在這個半週期更新了值，那麼 `9118` 物理上**必須**捕捉到上游更新後的「新值」（`Node(data)`），而不是上個半週期的「舊值」（`Prev(data)`）。

你強行用 `Prev` 切斷邊，等於人為引入了一個半週期（half-cycle）的延遲，這在 pipeline 或 carry chain 中是致命的。

#### 務實解法：選 (d) - 在 IR 層級引入「固定 K 次微型迭代（Fixed-K Micro-Evaluation）」

你擔心交給 GPU 會 divergence，但我們**不需要用 S1 引擎來 fallback**，我們可以直接**把這 843 個 SCC node 編譯成一段「固定迴圈次數」的 IR**。

對於這 843 個 node（約 56 個 SCC，可以每個 SCC 獨立，或全包在一起），給定一個任意的拓樸序（因為有環，必定有 back-edge），我們在生成的 Code / IR 裡這樣寫：

```csharp
// 演算法：Fixed-K Relaxation 編譯生成
// 對於一個包含 N 個 node 的 SCC：

// 1. 準備 Current 陣列（初始值為上一半週期的狀態，或從拓樸上游剛算出的值）
for node in SCC:
    val[node] = Prev[node] // 或 Hold[node]

// 2. 固定迭代 K 次（K=3 或 4 通常足以讓 Latch 環或 Carry Chain 穩定）
// 【注意】這裡不用 while(changed)，直接 for 展開，GPU 最喜歡這種沒有 divergence 的寫法！
#pragma unroll
for (int k = 0; k < 3; k++) {
    // 按照某個固定的順序評估
    val[n_A] = Expr_A( val[n_B], Prev[n_C], ... ); // Node(x) 映射到 val[x]
    val[n_B] = Expr_B( val[n_A], Prev[n_D], ... );
    // ... 
}

// 3. 將最終結果寫出
for node in SCC:
    Node[node] = val[node]
```

**為什麼這極度划算？**
* **GPU 友善**：843 個 node 對 GPU 來說連塞牙縫都不夠。Fixed 3 次 iteration 完全沒有 Branch Divergence（warp 裡的每個 thread 都無腦跑 3 次）。
* **物理正確**：這完美模擬了電子的 Settling 過程。不管是誰先誰後，只要迭代 3 次，組合邏輯和 transparent latch 的訊號一定能傳遞並收斂。
* **開發成本低**：你不需要再寫任何複雜的 Graph 演算法，直接把沒解掉的 SCC 打包輸出即可。

---

### 2. Clock Phase 順序：能自動推導嗎？該推導嗎？

**不要試圖從零延遲（Zero-delay）的 Netlist 自動推導 Derived Clock 的精確拓樸序。**

在 Static Timing Analysis (STA) 裡，這叫 Clock Network Delay。在我們從 switch-level 抽出來的 Boolean IR 裡，所有的 gate 都是 0 delay 的。
* 物理上：`master` 變了 -> `divider` 狀態更新 -> 經過 2 個 NOR gate -> 產生 `cclk` -> 再經過 1 個 inverter -> 產生 `cp1`。
* IR 上：它們都在同一個 `Step()` 內被求值。如果你用靜態的拓樸去硬切，遇到環就必須猜哪裡是「真正的」先後。

`9118` 跟 `n10390` 的案例告訴我們：**你的暫存器判斷和 SCC 分組是沒錯的，錯的是「假設同一半週期內，所有 latch 更新都是同時/原子的」。** 事實上它是像水波一樣傳遞的。用上述的 **Fixed-K Relaxation** 就能完美吸收這種水波傳遞（Ripple），不需要知道順序。

### 3. Palette-RAM Cascade：如何 Debug `n824` 是被誰害的？

如果你想為以後的開發留一個除錯工具，找出「到底是哪個 cut 導致了 cascade mismatch」，最聰明也最標準的演算法是 **Delta Debugging (自動二分法除錯)**。

**自動化二分尋找兇手演算法（Self-Validating Bisection）：**
1. **收集嫌疑犯**：記錄 γ.2 一共對多少條邊進行了 `Node -> Prev` 的替換（假設有 100 條邊）。
2. **切分集合**：將這 100 條邊分成兩半（A 區 50 條，B 區 50 條）。
3. **編譯與測試**：
   * 只對 A 區的 50 條邊實施 γ.2 cut，其他邊保留環（跑 S1 或 Fixed-K）。
   * 跑 3000 個半週期，看 `ppu.pal_ram_08_b4` 是否 mismatch。
4. **二分縮小**：
   * 如果 mismatch，說明「兇手」在 A 區。把 A 區再分一半（25/25）繼續遞迴。
   * 如果沒 mismatch，說明 A 區的 cut 是安全的！兇手在 B 區。
5. **定位**：大約只需要 $\log_2(100) \approx 7$ 次跑檔，你就能精確定位到是哪「一條」邊的 cut 導致了 `n824` 算錯。這比盯著波形圖看快 100 倍。
