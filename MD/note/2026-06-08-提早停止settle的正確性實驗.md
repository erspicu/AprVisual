# 「提早停止 settle 會不會損害正確性?」實驗(2026-06-08)

## 問題

settle 迴圈(`ProcessQueue`)會跑到收斂(`RecalcListNextCount == 0`)才停。一個冒險的想法:**如果不讓它 iteration 完,是否一定損害正確性?** 也就是把每次 settle 強制截在 N 個 wave,看會怎樣。

## 方法(全程 DEBUG,`#if DEBUG` 包好、不汙染 Release)

加了 DEBUG-only 的 `--settle-cap N`:把 `MaxSettlePasses`(原本是 128 的純防呆 tripwire)降成 N,讓每次 settle 跑超過 N 波就**直接放棄**(清掉 pending、break,維持當下未收斂的狀態),`SettleCapSilent` 抑制每次觸發的 abort 訊息(低 cap 會觸發數十萬次)。Release 連這個欄位和 cap 區塊都沒有(整段 `#if DEBUG`)。

cap 值用前一份 settle-pass 分佈量到的百分位來挑(mean=12、p90=19、p99=33、max=45,見 `2026-06-08-settle-pass分佈.md`)。

## 結果

### 1. checksum 掃描(`full_palette` 300k `--extra-ram`,golden = `0x794A43ABDF169ADA`)

| cap | 放棄的 settle | checksum @300k | |
|---:|---|---|---|
| 45(=真實最大深度) | 無 | `0x794A43ABDF169ADA` | **== golden** ✓ |
| 33(p99) | 最深 ~0.58% | `0x024848…` | **DIVERGED** |
| 19(p90) | ~10% | `0xFD113B…` | DIVERGED |
| 12(mean) | ~50% | `0xA20187…` | DIVERGED |
| 8 | ~64% | `0xA89533…` | DIVERGED |

→ **只要真的砍掉任何一次會改變節點的 settle(連最深的 0.58% 都算),就破壞 bit-exact。** cap=45 之所以還對,是因為 `iteration > 45` 永不觸發(最大就是 45)。

### 2. 發散速度(cap=33 vs 未截斷)

cap=33 在 **hc < 1000 就已偏離**(boot 期間就有 ≥34-pass 的 settle 被截斷)。不是慢慢累積,是幾乎立刻。

### 3. 嚴重度 —— 不是「微糊」,是「CPU 跑飛」

同一張 frame 的 CPU 暫存器(`--screenshot ... --frames 4`):

| | uncapped | cap=8 |
|---|---|---|
| PC | **8007**(在 PRG-ROM 執行) | **0040**(掉進 zero-page RAM) |
| IR | 2C(`BIT`,正常指令) | **00(`BRK`)** |
| S | FD | **00**(堆疊指標繞回 0) |

cap=8 的 CPU 在**第一張 frame 內**就崩進 BRK 迴圈、開始執行 RAM 裡的垃圾。

## 結論

**會,而且是「立刻且災難性」的。** 對這顆晶片:
- 「沒 iteration 完」= 模擬一顆時脈開太快、組合邏輯來不及 settle 的晶片(timing violation);真實 NES 的時脈週期就是設計成讓組合邏輯 settle 完,所以完整 settle = 真晶片該半週期的穩定態。
- CPU 是緊耦合回授迴路:一個被提早截斷、栓錯的 bit 落進 PC / 旗標 / 指令暫存器,下一拍抓錯指令 → 雪崩,幾百個 hc 內跑飛。
- **沒有「截短換速度、只掉幾個像素」的優雅退化區間** —— settle 深度(7–45)就是晶片真實的邏輯關鍵路徑,砍它 = 砍真實的訊號傳播。

唯一理論上安全的提早停(只跳「可證明不會被觀測」的 closed dead-end),專案早試過 → `dead-end-skip` dead end:看似 38% 可跳,但安全子集 <1%,過度套用就把 CPU 弄壞,根因同樣是 **~94% 的 netlist 是一個雙向耦合大 SCC**,大多數節點都能傳到匯流排。

## 工具

`--settle-cap N` 留作 DEBUG 實驗工具(跟 `[settle-pass-dist]` profiler 一組),Release 無此旗標。重現:

```
dotnet run -c Debug --project src/AprVisual.S1 -- --benchmark <rom> --bench-hc 300000 --extra-ram --system-def-dir <data> --settle-cap 33
# 看 "# NodeStates checksum @ t=...": 與 golden 0x794A43ABDF169ADA 比對
```
