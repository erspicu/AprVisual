以下我將根據 Visual2C02/RP2C02 的電路結構與已知的時序行為，逐一回答你的問題。

### 總結在前

你遇到的矛盾是真實的硬體現象。**根本原因在於：PPU 內部不同功能區塊，是從 $2001 暫存器的輸出，經過不同長度的邏輯路徑，在各自的關鍵時點去「取樣」渲染啟用狀態。** 因此，「$2001 寫入的延遲」並不是一個全域常數，而是依賴於「哪個內部電路在什麼時候去讀取這個狀態」。Dot 339 的 sprite counter reset 決策，恰好是一個取樣路徑特別長、時序特別晚的例子。

### Q1: $2001 寫入的精確延遲是多少？

這個問題需要拆解成兩個部分：CPU 寫入到 PPU 內部 latch，以及 PPU latch 變化到渲染邏輯生效。

#### 已確證事實

1.  **CPU to PPU Latch (非同步邊界):**
    *   CPU 和 PPU 時脈是非同步的。CPU 的 `/WR` 信號在一個 CPU 週期 (3 PPU dots) 內為低電位。PPU 在其自身的時脈 `CLK` 的驅動下，監視位址匯流排 (`PA0-2`) 和 `/PPUWR` 信號。
    *   當 PPU 偵測到有效寫入位址 ($2001) 和 `/PPUWR` 有效時，它會在某個 `CLK` 的上升沿或下降沿（取決於具體電路設計）將資料匯流排上的值鎖存 (latch) 到 $2001 暫存器的輸入級。
    *   由於 CPU 寫入週期與 PPU 時脈的相位隨機，從 CPU 執行 `STA $2001` 指令的週期開始，到 PPU 內部 $2001 暫存器的位元翻轉，這個延遲是**可變的**。nesdev wiki 上「2-5 PPU cycles (dots)」的說法，就是對這個非同步邊界延遲的經驗性總結。它涵蓋了 CPU 指令週期的剩餘時間、PPU 的取樣延遲、以及內部第一級鎖存器的建立時間。**對於一個 bit-exact 的模擬器，將這個非同步邊界理想化為零延遲是可以接受的起點，但必須在後續的內部傳播中補償。**

2.  **PPU Latch to Logic (內部傳播延遲):**
    *   一旦 $2001 暫存器的 D-latch/flip-flop 狀態改變，它的輸出 (Q 和 /Q) 會開始在晶片上傳播。這不是瞬間的。信號需要通過多個反相器(inverter)組成的緩衝器(buffer)、NAND/NOR 邏輯閘，才能到達最終目的地。
    *   **這個傳播延遲是確定性的，取決於物理路徑長度和經過的邏輯閘數量。** 這正是你遇到的問題核心。

### Q2: 為什麼均勻延遲會失敗？

#### 已確證事實

你的實驗完美地復現了硬體行為。均勻延遲之所以失敗，是因為 PPU 內部對「渲染是否啟用」(`rendering enabled`) 這個信號的使用點和使用時機不同。

*   **`rendering enabled` 信號**：這個內部信號通常是 `$2001` 的 `spr_enable` (bit 4) 和 `bkg_enable` (bit 3) 經過 `OR` 邏輯後的結果。我們稱之為 `ren_en`。
*   **Test 1 (dot 15/33 附近)**：這個測試驗證的是 scanline 開頭的 sprite-zero hit。相關的 PPU 決策（如開始背景 tile data 提取）發生在 scanline 的早期。當 CPU 在前一條 scanline 的末尾或 VBLANK 期間寫入 `$2001` 時，到下一條 scanline 的 dot 15 或 33，`ren_en` 信號有**充足的時間** (超過 50 個 dots) 在整個 PPU 內部穩定下來。因此，一個 1-dot (8hc) 的延遲是無傷大雅的，但一個 3-dot (24hc) 的延遲可能會讓 PPU 錯過 scanline 開始時的一些初始設定，導致渲染管線出錯。
*   **Test 3 (dot 339)**：這個測試的關鍵在於，CPU 在 dot ~324 寫入 `$2001`，而決策點在 dot 339。這是一個非常窄的時間窗口 (~15 dots)。`ren_en` 信號從 `$2001` 暫存器出發，需要傳播到 **Sprite Control Logic** 中負責重設 shifter 計數器的部分。這段物理路徑在 die 上相對較長，經過了多級緩衝。因此，它需要更長的時間才能到達。**硬體實測和電路分析表明，這個特定路徑的傳播延遲恰好在 2-3 個 PPU dots 左右。**

**結論：** 不是 `$2001` 寫入延遲本身不均勻，而是 **`ren_en` 信號到達 PPU 不同功能單元的傳播延遲不同**。你試圖用一個全域變數來模擬一個在物理空間上分佈的信號，因此產生了矛盾。

### Q3: Dot 339 的兩個決策是否使用同一個信號？

#### 已確證事實

是的，這兩個決策——(a) pre-render line 的 odd-frame dot-skip 和 (b) visible line 的 sprite shifter counter reset——它們的決策邏輯都依賴於同一個源頭：`ren_en` 信號。

*   **邏輯結構**：
    *   Odd-frame skip logic ≈ `(v_is_prerender) AND (h_eq_339) AND (is_odd_frame) AND (ren_en)`
    *   Sprite reset logic ≈ `(v_is_visible) AND (h_eq_339) AND (ren_en)`

*   **取樣時機**：兩者的觸發條件都包含一個由水平計數器產生的、在 dot 339 生效的脈衝信號 `h_eq_339` (或類似的 `h_is_339-340` 之類的)。這意味著它們是在**完全相同的 PPU 時脈相位**去「看」`ren_en` 信號當前的狀態。
*   **物理位置**：在 Visual2C02 die 上，控制 scanline 計數器（與 odd/even frame 相關）的邏輯和控制 sprite pipeline 的邏輯物理上是分開的，但它們都會接收來自 `$2001` 暫存器區域扇出 (fan-out) 的 `ren_en` 信號。雖然路徑長度可能略有微米級的差異，但在 dot/hc 的時間尺度上，可以認為它們的傳播延遲是**幾乎一樣長**的。

你用「v=261, dot 338-339 窄窗延遲 16hc」修好了 (a)，這非常有力地證明了 (b) 也需要一個相似的、較長的延遲。你的 `AccuracyCoin` Test 3 需要 3-dot (24hc) 延遲，這與 16hc 在同一個數量級，差異可能源於模擬器模型的其他細節或測試 ROM 的精確邊界條件。

### Q4: 對開關級模擬器的修法建議

你的方向是對的：不應該延遲 `$2001` 寫入本身，而是要**模擬 `ren_en` 信號在 PPU 內部的傳播延遲**，並且是**有區別的傳播延遲**。

對於一個二值、settle-to-quiescence 的模擬器，最乾淨、最符合硬體真實情況的做法如下：

1.  **建立分級的 `ren_en` 內部信號節點**：
    *   CPU 寫入 `$2001` 時，立即更新代表 `$2001` flip-flop 輸出的**核心節點**，例如 `ppu.mask.spr_en_raw` 和 `ppu.mask.bkg_en_raw`。
    *   從這兩個 `_raw` 節點，衍生出一個 `ppu.rendering_enabled_raw` 節點 (`OR` 邏輯)。這是 0 延遲的信號源。

2.  **模擬傳播路徑**：
    *   建立一個或多個「延遲後的」`ren_en` 節點。在開關級模擬中，最物理的方式是插入一個**反相器鏈 (inverter chain)**。一對反相器構成一個緩衝器，會產生一個小的傳播延遲。
        *   `ppu.rendering_enabled_fast`: `ppu.rendering_enabled_raw` -> **[Buffer x1]** -> `ppu.rendering_enabled_fast`。
          這個信號可以有一個較短的延遲，比如 8-12 hc (1-1.5 dot)。將 PPU 中那些需要快速響應 `ren_en` 的電路（例如 scanline 初期的tile fetch控制邏輯）連接到這個節點。
        *   `ppu.rendering_enabled_slow`: `ppu.rendering_enabled_raw` -> **[Buffer x3]** -> `ppu.rendering_enabled_slow`。
          這個信號可以有一個較長的延遲，比如 20-24 hc (2.5-3 dot)。將 dot 339 的決策邏輯（sprite counter reset 和 odd-frame skip）連接到這個節點。

3.  **連接到目標邏輯**：
    *   修改你的 sprite counter reset 邏輯和 odd-frame skip 邏輯，讓它們的與門 (AND gate) 輸入不再是 `ppu.rendering_enabled_raw`，而是 `ppu.rendering_enabled_slow`。
    *   檢查 Test 1 失敗的原因，很可能是它所依賴的某個渲染管線邏輯被錯誤地連接到了 `_slow` 信號，或者全域延遲過長。將它連到 `_fast` 或 `_raw` 應該能修復。

#### 矽片層面的佐證 (RP2C02)

在 die shot 上，`$2001` 暫存器位於晶片的控制邏輯區域。從這裡扇出的控制信號到位於晶片另一側的 Sprite DMA/Evaluation/Rendering pipeline，確實需要經過一段長長的 metal layer 走線和多個 repeater/buffer (由反相器對構成)。這段物理距離和緩衝器就是延遲的來源。你的 `_slow` 信號模型，正是在功能上模擬了這個物理現實。

**總結建議：**
放棄單一的全域 `$2001` 寫入延遲。在你的 S1 引擎中，實現一個**分層的內部渲染啟用信號** (`..._raw`, `..._fast`, `..._slow`)。這不僅能同時滿足 Test 1 和 Test 3 的要求，也是對 PPU 內部信號傳播這一物理現象的更精確模擬，是通往 bit-exact 的必經之路。根據 `AccuracyCoin` 的反饋，將 `..._slow` 的延遲調整到 **3 個 PPU dots (24 個 half-cycles)** 應該是正確的目標值。
