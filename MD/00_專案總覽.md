# 專案總覽

## 這個專案到底是什麼

不是「再寫一個 NES 模擬器」。這個方向是要：

> 把 **Visual6502 / Visual2A03 / Visual2C02** 這種開關層級（switch-level）晶片網表，
> 轉換成「可分析、可驗證、可執行」的邏輯模型，
> 最終可以 emit 到 **CPU、CUDA、Verilog** 三種後端。

**重點不是 GPU 本身，重點是中間那條翻譯鏈**：

```
原始網表
  ↓ 解析
  Graph 模型 (NetNode / NetTransistor)
  ↓ 分析
  動態節點 / 回授結構辨識
  ↓ 抽取
  組合邏輯 + 時序邏輯
  ↓ 產生
  Backend (CPU / CUDA / Verilog)
  ↓ 驗證
  跨 backend 等價比對
```

CUDA 不是難點。FPGA 不是難點。難點是中間那層**邏輯抽取與狀態分類**。

## 適用場景

這條路適合的目標：

- **Golden Model 驗證**：替手寫的 NES 模擬器當對照組
- **晶片逆向工程**：把實體晶片拓樸還原成可讀邏輯
- **週期精確（cycle-accurate）研究**：找出 illegal opcode、邊界 timing 行為
- **大規模 GPU 批次模擬**：上千個 NES 平行跑，給 AI 訓練環境用

如果只是想「玩遊戲」，傳統手寫模擬器永遠是更好的選擇。這條路是大砲打蚊子。

## 核心技術觀念（最容易被誤解的部分）

1. **開關層級邏輯不是普通的布林樹**
   - Pass transistor 是雙向的（source / drain 沒有固定方向）
   - 有些節點會「浮接」並靠寄生電容保留前一個狀態（dynamic node）
   - Pull-up 不等於「永遠是 1」，而是「沒有強拉低時才回升到高」
   - Bus 是多驅動共享，不是單向覆寫

2. **單純做 `FindPathsToGnd` 是不夠的**
   - 完整的 NMOS 評估規則需要三條：
     1. 有強路徑通往 GND → 下一輪是 0
     2. 否則，有路徑通往 VCC 或 pull-up → 下一輪是 1
     3. 否則，**保留前一個狀態**（這就是 dynamic node 行為）

3. **`dynamic node = previous state` 是工程抽象，不是完整物理**
   - 對 cycle-level 數位行為通常夠用
   - 對真實電容洩漏 / charge sharing / 時間常數則不夠

## 最小可行版本（MVP）速覽

只做下面這個範圍：

- 只處理 **2A03**（不碰 PPU 2C02）
- 只支援 **NROM**（不做 mapper）
- 只驗證 reset 後**前幾個 fetch cycle**
- 先做 **CPU evaluator**（不寫 CUDA）
- 之後才寫 CUDA evaluator，**節點對節點**比對等價

詳見 [03_MVP與CSharp骨架.md](./03_MVP與CSharp骨架.md)。

| Phase 0 | 資料清洗：解析 `segdefs.js` / `transdefs.js` |

## 目前已知的常見錯誤路徑

- 把「reset 視為 `NOP`」當成最終模型 — 它只是開機 bring-up 的暫時短路
- 把「跑三輪 settle pass 就收斂」當成定律 — 它只是經驗法則
- 一開始就寫一顆巨型 CUDA kernel — 幾乎不可能 debug
- 一開始就試圖把整個 NES（CPU + PPU + APU + mapper）塞進 GPU
- 把 PPU 跟 CPU 一起做 — PPU 比 CPU 複雜得多，必須延後
