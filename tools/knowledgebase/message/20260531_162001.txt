### C. 針對「結構化簡 (Lowering)」與「Event-driven 特化 (Fast-path)」：近 20 年有新東西嗎？

**結論：沒有超越 1990-2000s 的「破壞性創新」，近 20 年全靠工程實作（Memory Hierarchy / Cache Locality）推進。**

1.  **Switch-Level 的死因：**
    Bryant 的 MOSSIM (1981) 與後續基於 BFS 的開關級模擬器，在 1990 年代末期被 EDA 工業界徹底放棄。原因是深次微米（Deep Sub-micron）效應讓純邏輯的 Switch 模型失去物理準確性，工業界全面轉向 FastSPICE（簡化矩陣求解）或純 Gate-Level/RTL。因此，2000 年後學界幾乎**沒有人**在研究 Switch-level event-driven BFS 的演算法加速。
2.  **Lowering (AIG / SAT) 的極限：**
    你提到的 Berkeley ABC (Mishchenko, 2006) 幾乎是 Logic Synthesis 結構化簡的巔峰。後來唯一算得上「學術突破」且被業界吸收的是 **MIG (Majority-Inverter Graph)** (Amarú, EPFL, 2014)。但這主要用於減少邏輯深度（Logic Depth）以利於時序優化，對於你這種「極低 Activity」的模擬器來說，MIG 能減少的 Event 數量與 AIG 差異不大。
3.  **Fast-path 的極限：**
    Selective-trace 與 Activity-directed 是 1980 年代的產物。當平均每次 Event 只有 ~1.4 個節點被解析時，代表**「調度開銷（Scheduling Overhead）遠大於計算本身」**。在單核 CPU 上，只要你還維持「Event Queue -> Dequeue -> Resolve -> Enqueue」的演算法框架，你的瓶頸就永遠是 **Cache Miss、Branch Misprediction 與 Pointer Chasing**，這些是馮紐曼架構的物理限制，不存在演算法魔術。

### A. 2005 年後「真正有效、被工業 EDA 採用」的技術突破 (退役/替代性技術)

既然單純的 Event-Driven 沒有突破，工業界是如何解決巨大 Netlist 與低 Activity 模擬的？他們採用了以下轉換思維的技術（符合你單核 CPU 的限制）：

**1. Channel-Connected Component (CCC) Macro-modeling / BDD Extraction**
*   **作者/代表：** R.E. Bryant (COSMOS 早期), 後續被 Synopsys NanoSim / CustomSim (FastSPICE) 團隊在 2000s 中後期極致化。
*   **真正有效的原因：** 你的 NES netlist 有 ~94% 是一個巨大的 Pass-transistor 耦合 SCC。在 runtime 做 BFS 解析這 94% 是效能殺手。真正有效的做法是在 Lowering 階段，使用 **Tarjan 演算法或 Union-Find** 將由 Pass-transistor 雙向連接的區域切分為 CCC（通道連接單元），然後將這個巨大的 SCC **靜態提取（Static Extraction）** 為一個或多個 BDD（二元決策樹）或 Truth Table。
*   **對你的意義：** 將「Runtime BFS 解析拓撲」變成「Runtime 查表 / BDD Traversal」。這能把 1.4 個節點的 BFS 尋訪，壓縮成 O(1) 的狀態轉移。

**2. Two-Queue Time-Wheel 的極簡化 (Data-Oriented Design in EDA)**
*   **作者/代表：** 這是工業界 VCS / Xcelium 內部不斷重構的工程技術，學術界可參考類似 "Cache-conscious Event-driven Logic Simulation" 的文獻 (e.g., Li, 2010s)。
*   **真正有效的原因：** 拋棄 Calendar Queue 或複雜的 Priority Queue。對於零延遲（Zero-delay）或單位延遲（Unit-delay）的開關級模擬，只維持 Current Array 和 Next Array。將 Netlist 從 Linked Nodes（指標）轉化為 **SoA (Structure of Arrays)**，保證 Fan-out 遍歷時完全順序讀取記憶體（Linear Memory Access）。這不是演算法突破，但能帶來 2x - 5x 的硬核加速。

### B. 誠實區分：哪些是紙上談兵？哪些真的有效？

在 2005-2025 年間，你在論文庫如果搜尋 "Event-driven logic simulation acceleration"，會看到很多垃圾，以下為你排雷：

*   **❌ 絕對無效 / 紙上談兵 (Incremental / Hype)：**
    1.  **AI/ML-based Event Prediction (機器學習事件預測)：** 近年很多論文聲稱用 GNN 或 RNN 預測哪些節點會翻轉來跳過模擬。**實務上完全無效**，因為推論（Inference）的運算成本比你那 1.4 個節點的 BFS 高出幾百倍。
    2.  **複雜的 Event Scheduling Trees (如 Splay tree, Heap-based queue)：** 維護樹的平衡開銷遠大於直接用簡單的 Ring Buffer / Array。
    3.  **Asynchronous Relaxation (非同步鬆弛法)：** 在單核軟體上做這個會導致大量的冗餘計算，完全不適合低 Activity 場景。

*   **✅ 真正有效 (但你可能已經在做，或屬於工程領域)：**
    1.  **Constant / Dead-code Elimination (如你所說的 Lowering)。**
    2.  **Topological Levelization 結合 Partial Compiled Code：** 即便你因為 activity 太低排除了全編譯型模擬 (Verilator)，但在你的 94% SCC 外部，那些穩定的單向邏輯閘，依然可以拓撲排序後編成一段 Inline C code。遇到 Event 時，直接 Call 該段函數，避免 Event Queue 的 Enqueue/Dequeue 開銷。
    3.  **Memory Pooling & Contiguous Fanout Lists：** 確保一個節點的扇出指標在記憶體中是連續存放的。
