這是一份非常硬核且極具價值的工程分析。你們能用電晶體級網表（Netlist）推演並證明「4 種相對相位空間」與「互斥現象」，這對 NES 模擬社群是非常珍貴的數據。

針對你的問題，基於目前 NESdev 社群的共識、真機行為，以及硬體逆向的經驗，以下是詳細的檢驗與回答：

### 1. 真機事實查核：單次上電能否全過？互斥存在嗎？

**結論：真機在「幸運的上電相位（單次上電）」下，是可以 10 項全過的。互斥現象在真機上不成立。**

*   **blargg 的測試設計**：blargg 的測試是在真實 NTSC NES 上開發並反覆驗證的。他發布 `ppu_vbl_nmi` 時的目標就是「精確還原真機行為」。雖然他在文檔中提到了上電對齊（Alignment）會導致某些測試失敗（使用者需要按 Reset 刷首抽），但**存在一個「Golden Alignment」能讓這套測試全過**，這是這套 ROM 存在的基礎。如果真機本質上互斥，他當年就會把它拆成兩支 ROM 並註明「這兩支不可能同時過」。
*   **社群共識**：在 NESdev 論壇上，處理過 blargg 測試的模擬器開發者（如 Mesen, puNES, BizHawk 的開發者）都經歷過這個掙扎。最終解決方案都是找準那個對的 PPU-CPU 相位。沒有文獻指出 `05-nmi_timing` 與 `10-even_odd` 存在物理上的互斥。
*   **你提到的 TriCNES**：TriCNES 將 offset 設為 7 能全過，這其實在數學上**證明了軟體層面的判定區間是存在交集的**。這意味著「0 交集」是你們兩顆晶片網表**合併時的系統偏差**造成的，而非測試 ROM 邏輯互斥。

### 2. 對齊空間的真實大小機制

**結論：你們的模型觀察到的「4 種物理類」以及「CPU 自由運轉、PPU 釋放重啟」機制，與社群最頂層的硬體共識完全一致。**

*   **相位的來源**：NTSC 系統有一個 21.477272 MHz 的主時鐘（Master Clock）。CPU (2A03) 內建 `÷12` 除頻器產生 Phi2；PPU (2C02) 內建 `÷4` 除頻器產生 Pixel Clock (Dot)。這意味著一個 CPU 週期等於 3 個 PPU Dot。
*   **為何是 4 種**：由於 `12 / 4 = 3`，這兩者的時鐘邊緣相對關係有 4 種可能（通常被稱為 Alignment 0, 1, 2, 3）。
*   **重啟機制**：社群硬體測試證實，拉低 `/RES` 引腳時，2A03 的 `÷12` 除頻器**不會**被重置（自由運轉）；但 2C02 的時鐘發生器狀態**會**受 reset 影響（或者更精確地說，Reset 釋放的瞬間決定了 PPU 狀態機相對於 Master Clock 的起點）。因此，每次按 Reset，相對相位就會在這 4 種狀態中隨機切換。
*   **你們的驗證非常漂亮**：你們用網表全掃描 K=0..5 證明偶數塌縮為 4 種，這從電晶體層面完美驗證了社群多年來透過外部黑箱量測得到的結論。

### 3. 「忠實偏差」的定性是否恰當？

**結論：這個定性「下得太重了」，且可能會誤導後人。建議改為「系統整合偏差（Integration Offset）待仲裁」。**

*   **為什麼不恰當**：「忠實偏差（Faithful Deviation）」暗示「真實的 RP2A03G + RP2C02G 晶片就是這樣互斥的，是 blargg 的測試寫錯了或取巧了」。但如前所述，真機能在一次上電中全過。
*   **更嚴謹的說法**：這應該定性為**「多晶片網表整合時的絕對相位偏移（Unarbitrated Absolute Phase Offset in Multi-Chip Integration）」**。
*   也就是說，Visual2A03 和 Visual2C02 作為單獨的晶片模型是非常準確的，但在你們的模擬器板級（Board-level）整合中，決定「CPU 週期起點」和「PPU Dot 起點」絕對位置的「黏合邏輯（Glue Logic / Reset Release Timing）」，存在 1 到 2 個 Master Clock 的微小偏移。

### 4. 仲裁實驗建議：如何找出這個 1-dot 偏差？

你們手上擁有極好的工具。我建議的最少決定性實驗如下：

**實驗核心目標**：定標 `BIT $2002`（CPU 讀取 PPU 狀態）在兩顆晶片間的絕對 Master Clock 延遲。

*   **實驗 A（純網表層面對比）—— 尋找 PPU/CPU 讀寫交接的絕對延遲**：
    1.  使用你們的雙網表系統，執行 `BIT $2002` 指令。
    2.  記錄：CPU 的 `Phi2` 邊緣 -> CPU 輸出地址 `$2002` -> CPU 輸出 `/RD` (或者 R/W 變低) -> PPU 網表接收到 `/RD` -> PPU 網表將 D7 (VBL flag) 放到數據匯流排上的**精確 Master Clock tick 數**。
    3.  將這個 tick 數與 emu-russia 的 PPUSim 進行對比。如果 PPUSim 在處理 `/RD` 到資料輸出的延遲比你們的 Visual2C02 快/慢了 1-2 個 Master Clock（可能因為 bonding pad 延遲、或者鎖存邊緣的判定不同），這就是導致 1-dot 偏移的元凶。
*   **實驗 B（真機探針與網表對齊）—— Reset 釋放的絕對起點**：
    1.  用高階探針量測真機 NES-001：Trigger 設在 `/RES` 上升沿。
    2.  量測從 `/RES` 越過閾值（釋放）到：(a) 第一個 CPU `M2` (Phi2 外露) 上升沿 的時間；(b) PPU 第一個有意義的外部匯流排動作的時間。
    3.  將這個時序圖對比你們網表中 `/res` 節點變 high 到內部除頻器啟動的 Master Clock 數。網表模擬器通常缺乏真實主板上的寄生電容與引腳上升時間（Rise time）模型，這極易導致 PPU 除頻器在網表中「過早」或「過晚」啟動 1 個 Master Clock。

### 5. 如果是假象，最可能的偏移來源在哪裡？

由於 blargg NMI 測試（05）和偶數幀跳點測試（10）都對 1-dot 極度敏感，且兩者在你們的模型中相位正好錯開（{7,5} vs {1,3}），這幾乎 100% 是一個 **1 個 PPU Dot（即 4 個 Master Clock）或 1 個 CPU 週期** 的系統層面滑移。

**最可能的來源（按可能性排序）：**

1.  **VBL Flag 讀取時序（$2002 Read Latency）**：
    *   blargg 的 `05-nmi_timing` 依賴於：CPU 在特定週期執行讀取 `$2002`，剛好卡在 VBL flag set 的同一個 dot 或是後一個 dot。
    *   在真實硬體中，CPU 的 `/RD` 抵達 PPU 且 PPU 將資料送回資料匯流排，這個過程橫跨了兩個除頻域。如果網表模擬環境中，數據從 PPU 送回 CPU 的「板級傳輸延遲」被理想化（設為 0），或者 `/RD` 同步邏輯差了 1 個半週期，會讓 CPU 「提早 1 dot」讀到 VBL，從而讓 NMI 測試的通過相位發生平移，導致與 `10-even_odd` 互斥。
2.  **Even/Odd 跳點判定時刻（Skip Evaluation Point）**：
    *   NTSC 2C02 在 Pre-render scanline 的 dot 339 決定是否跳過 dot 340（取決於奇數幀與 BG rendering enabled）。
    *   網表中這個判定點是湧現的。如果在你們的系統裡，因為 `/RES` 的啟動邊緣差異，導致「BG Enable 寫入生效」的 CPU 週期相對於這個 dot 339 晚了或早了一個 Master Clock，就會導致跳點測試（10）需要的相位偏移。
3.  **Master Clock 到 PPU internal clock 的緩衝延遲**：
    *   Visual2C02 網表是剝離了 pad ring 提取的嗎？輸入時鐘引腳到內部真正除頻器的傳播延遲，在 CPU 和 PPU 上可能不同（物理距離、buffer 級數）。如果不加上這個微小的 offset 參數，兩個網表自由運轉時就會有一個固定的相差。

### 總結建議

*   **修改文件聲明**：將「永久忠實偏差」改為「已知的網表整合限制（Known Netlist Integration Limitation）」。
*   **解釋原因**：明確寫出「經窮舉證明，在此系統絕對相位下，這兩個測試集合沒有交集。這很可能源於雙晶片網表在 Reset 釋放時，或外部匯流排通訊時，存在約 1-dot 的絕對時序偏差（Absolute timing offset），目前尚未透過真機微觀探針仲裁。」
*   這樣寫不僅保護了你們的科學嚴謹性（我們確實測量到互斥），也保留了硬體事實的正確性（真機能全過，是我們的系統黏合有 1-dot 誤差），展現出世界級模擬器開發者的專業高度。
