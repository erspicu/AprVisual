# S4.6 — GPU compute shader 加速 IR（設計 + 步驟）

> 狀態：**規劃**（2026-05-12，user 要求「驗證用 GPU shader 跑 IR」）。實作未開始。
> 前置：S4.1–S4.5 完成（IR codegen：C# Expression-tree JIT / LLVM-MCJIT / bit-sliced emit / fixed-K SCC block / Resolve_Buses block —— 見 `MD/impl/S4/00`）。
> 目標機器：user 的 **RTX 4080**。

---

## 0. TL;DR / 老實的期望值

- **「一台 NES、即時、用 GPU」是有可能的，但只有用對方法**：不是 naive 的「每個拓樸層 dispatch 一次」（dispatch overhead 會吃光一切）；而是 **「一個 workgroup、state 全放 groupshared、`GroupMemoryBarrier` 隔開 DAG 的每一層、一個 dispatch 內 loop 跑 N 個半週期、memory handler 也在 GPU 上做」**。這樣每幀只有 ~數百個 dispatch、GPU 跑得滿、沒有 per-half-cycle 的 CPU↔GPU round-trip。粗估在 4080 上 **可能跟 S1（~14.5s/幀 ≈ ~45K hc/s）打平、甚至更快** —— 但這要做出來量才知道（半週期太小、bottleneck 是 barrier latency 跟 tiny-kernel occupancy，不是 FLOPS）。
- **用 AprNesAvalonia 的 SKSL（SkiaSharp runtime fragment shader）跑 IR 是錯的工具** —— SKSL 是 *fragment* shader（per-pixel → 一個 color），沒有 compute 語意（沒有 RWStructuredBuffer、沒有 groupshared、沒有 dynamic loop、`uniform int` 都不行），而且 memory handler 每半週期要 CPU↔GPU 來回 = 致命。SKSL 那套只適合它原本的用途：**CRT 後處理顯示 shader**（如果 GPU-sim 版之後要加 UI，那個拿來顯示畫面剛好）。
- **對的工具**：HLSL compute shader（DirectCompute / D3D11/12）或 GLSL/SPIR-V compute（Vulkan）。.NET 端最乾淨的選擇 = **[ComputeSharp](https://github.com/Sergio0694/ComputeSharp)**（用 C# 寫 HLSL compute shader、跑在 D3D12、**可 headless 不用開視窗**）。次選 Silk.NET（D3D11/12/Vulkan 原生 binding）或 ILGPU（C# IL → CUDA/OpenCL）。
- **「多台 NES 同時跑」（bit-sliced，一條 GPU thread = 一台 NES，每條 thread 用 32-lane SWAR per `uint`）才是 GPU 真正壓倒性贏的場景** —— embarrassingly parallel、4080 跑幾千台輕鬆 —— 但 user 對「多台」沒興趣，所以這條只當 fallback 記著（S4.4a 的 bit-sliced emit 已經是它的起點）。

## 1. 為什麼 naive GPU 做法慢、聰明做法可能不慢

**IR 的一個半週期** = 按拓樸序評估 ~11680 個 EvalOrder node（深度 ~數十層、層內獨立 → 可平行）+ 843 個殘餘 SCC node（fixed-K=32 迭代）+ 2086 個 bus（S0/S1/W1 + K_PASS≈3）+ handler chain（master clock toggle + 行為式 64KB RAM / cart ROM 的讀寫）。

**Naive 做法（每層一個 dispatch）**：~數十個 dispatch / 半週期 × ~3-10µs/dispatch（D3D12 CPU-side command 開銷 + UAV barrier drain）≈ ~tens of µs / 半週期 → ×654719 半週期/幀 ≈ **~10-50s/幀 純 dispatch overhead** → 比 S1 慢。**而且** memory handler 在 CPU 上的話、每半週期 CPU↔GPU round-trip（~10-100µs）≈ ~6-65s/幀 → 更死。所以 naive 做法不行。

**聰明做法**：
1. **state 全放 GPU**：node-state 是 ~14738 bytes（一個 byte/node，0/1）—— 用 `groupshared byte`（或打包成 `uint`）放在 on-chip。RAM（64KB）+ ROM（PRG/CHR，幾十 KB）放在 `RWStructuredBuffer` / `groupshared`（64KB 超過 groupshared 的 32-48KB 上限 → RAM 放 buffer、其餘放 groupshared）。
2. **一個 workgroup**（≤1024 threads）做整片晶片：precompute 一個 **schedule**（CPU 端、build 時算好）= 每一層 (level) 有哪些 node、第 t 條 thread 在第 L 層負責哪個 node id。kernel 裡 `for level in 0..numLevels: { 每條 thread 算自己這層的 node（讀 groupshared、算 Expr、寫 groupshared）; GroupMemoryBarrierWithGroupSync(); }`。`GroupMemoryBarrier` 只有 ~ns 等級、不是 dispatch barrier。
3. **SCC fixed-K + bus resolver 同理**：DAG 評估完 → `for k in 0..K_SCC: { 算 SCC node; barrier }` → `for kb in 0..K_BUS: { 算 bus S0/S1/W1; barrier }` —— 全在同一個 kernel 裡。
4. **handler chain 也在 GPU 上**：master clock toggle = 一條 thread 翻一個 bit；memory R/W = 從 `RWStructuredBuffer ram[]` / `rom[]` 讀寫（addr/we 已 settle 在 groupshared） —— 純 array index，compute shader 易如反掌。**沒有 CPU↔GPU round-trip**。
5. **一個 dispatch 跑 N 個半週期**：kernel 最外層再包一個 `for hc in 0..N(~1000): { 上面整套 }`（半週期之間 `GroupMemoryBarrier`）。N=1000 → 每幀只 ~654 個 dispatch（純 GPU 跑、CPU 只管 submit）。framebuffer 只在要顯示時 readback。
6. **per-半週期成本** ≈ ~數十個 `GroupMemoryBarrier`（~ns 各）+ 實際 compute（每 node 幾個 boolean op、總共 ~11680+843×32+2086×3 ≈ ~50K op/半週期、4080 一個 SM 一個 cycle 幾十個 op → ~µs 級）→ ~低個位數 µs / 半週期 → ×654719 ≈ **~1-5s/幀** → **可能跟 S1 打平或更快**。但這是樂觀估計；實測才算數。

**風險 / 為什麼可能還是慢**：(a) 只用「一個 workgroup」= 只用了 4080 一個 SM（4080 有 ~76 個 SM）—— 嚴重 underutilize；要嘛跑「一台」就接受這個（反正一個 SM 也夠快？），要嘛把 schedule 攤到多個 workgroup（但跨 workgroup 沒有 barrier、要用 global memory + device-wide barrier 或多個 dispatch → 回到 naive 問題的一半）。(b) `GroupMemoryBarrier` × ~數十層 × 654719 半週期 ≈ ~千萬次 barrier —— 每個 barrier 讓整個 workgroup 等齊、warp divergence + 不均勻層寬會放大。(c) groupshared bank conflict（14738 bytes 的散亂讀寫）。—— 這些都是 GPU sim 的老問題，所以「不一定贏」是誠實的；但 user 想試、4080 也夠猛、值得做個 prototype 量一下。

## 2. 為什麼不用 AprNesAvalonia 的 SKSL

AprNesAvalonia（AprNes 的 .NET Avalonia UI 版）用 **SkiaSharp 3.119.3-preview 的 `SKRuntimeEffect`（SKSL = Skia Shading Language）** 做 **CRT 後處理 fragment shader**（`Shaders/crt_core_*.sksl`、`ShaderLoader.cs` 讀檔、`CrtGpuRenderThread.cs` 在 Avalonia render thread（`ISkiaSharpApiLeaseFeature.Lease()` 拿到 GPU-backed `SKCanvas`、Windows 上是 D3D11）跑）。emu loop 自己在另一條 `Thread`、不在 UI thread；但 SKSL 的執行要在那個 leased canvas 上 → 要有 Avalonia render thread → 要開視窗。

**SKSL 拿來跑邏輯 IR 不行的理由**：
- SKSL 是 **fragment shader**：`main(float2 coord) → half4 color`，per-pixel。沒有 compute kernel、沒有 workgroup、沒有 `groupshared`、沒有 `RWStructuredBuffer`（README 明寫「無 structured buffer，用 uniform / child shader 傳資料」）、`uniform int` 不行、整數 `%` 不行、無 dynamic loop bound。
- 拿它做 GPGPU = 老式「fragment-shader-as-GPGPU」hack：node-state 編成 texture、render 一個 quad、fragment shader 讀 prev-state texture + 算每個 node 的 next → 輸出到新 texture、ping-pong。每個 draw 只能算「一層」（fragment 之間不能讀彼此這次 draw 算的值）→ 每半週期 ~數十個 draw call + texture ping-pong overhead。
- **memory handler**：fragment shader 不能讀寫任意 buffer（只能 sample texture）→ memory R/W 得在 CPU 上做 → 每半週期 CPU 讀回 addr texture、做 memory op、寫 data texture → CPU↔GPU round-trip → 死。
- 所以 SKSL 那套**只留給它本來的用途**：如果 GPU-sim 版之後要加個視窗顯示畫面，那個 CRT shader 拿來顯示剛好（reuse `ShaderLoader` + `CrtGpuRenderThread` 的 pattern）。**邏輯 IR 的 compute 不走 SKSL。**

## 4. GPU compute shader IR 的設計

### 4.1 state layout（GPU 端）

- `node_state` : `uint[ceil(14738/?)]` 或乾脆 `groupshared uint node[14738]`（一個 uint 存一個 node 的 0/1 —— 浪費，但避免 sub-word 讀寫的麻煩；14738 × 4 bytes ≈ 59KB > groupshared 上限 48KB ⇒ 改 byte-pack：`groupshared uint node[ceil(14738/4)]` ≈ 14.7KB，或 bit-pack ≈ 1.8KB —— bit-pack 最省但讀寫要 mask/shift，先 byte-pack）。
- `prev_state` : 同樣大小（Hold/Prev 讀這個 —— 半週期開頭 snapshot）。
- `ram` : `RWStructuredBuffer<uint> ram[16384]`（64KB / 4，或 byte-pack）—— global memory（太大放不進 groupshared）。
- `rom` : `StructuredBuffer<uint> prg[], chr[]`（read-only，cart ROM）。
- `framebuffer` : `RWStructuredBuffer<uint> fb[256*240]`（ARGB；要顯示時 readback）。
- `schedule`（read-only，build 時算好）：
  - `level_starts[]` / `level_node_ids[]` —— 每層的 node id list（按 EvalOrder 拓樸分層）。
  - 每層第 t 條 thread 負責哪些 node（thread t 處理 `level_node_ids[level_starts[L] + t], [+ numThreads], [+ 2*numThreads], ...`）。
  - 每個 node 的 NextExpr 編成一段「迷你 bytecode / 直接 inline 在 HLSL 裡」—— 兩條路：(A) **codegen 整個 HLSL kernel**（每個 node 一行 HLSL `node[v] = (...)`，按拓樸序，層之間 `GroupMemoryBarrierWithGroupSync()`）—— 跟 `EmitCsharpSource` / LLVM emit 一樣的做法、HLSL compiler 去優化；(B) **bytecode interpreter**（schedule 裡放 Expr 的 RPN bytecode，kernel 跑一個 stack-machine）—— 慢但 kernel 小、不用每換 ROM 重編。**先做 (A)**（跟現有 codegen 一致，HLSL→DXIL 由 dxc 編）。
- SCC：`scc_eval_orders[]`（每個殘餘 SCC 的 Gauss-Seidel 順序，已有 `IrEngine.SccEvalOrders`）—— codegen 成 `for k in 0..32: { 每個 SCC 的 node 按順序 node[v]=(...); barrier }`。
- bus：`BusNodes`（已有 `IrEngine.BusNodes`：LogicPasses / BusPasses / PullDown / PullUpCond / StrongVcc / StaticLoad）—— codegen 成 §4.3 的 `Resolve_Buses()`（已有 C# / bit-sliced emit，照搬成 HLSL）。

### 4.2 `step` kernel（一個半週期）

```hlsl
[numthreads(NUM_THREADS, 1, 1)]   // NUM_THREADS = 256 或 512
void step(uint tid : SV_GroupThreadID, ...) {
    // 0. snapshot: prev_state = node_state  (每 thread 抄一段)
    for (i = tid; i < N; i += NUM_THREADS) prev[i] = node[i];
    GroupMemoryBarrierWithGroupSync();
    // 1. handler chain: master clock toggle + memory R/W (從 ram[]/rom[]、addr 還是上半週期的值)
    if (tid == 0) { node[CLK] = ~node[CLK] & 1; ... }
    GroupMemoryBarrierWithGroupSync();
    // (memory handler 其實要在 addr settle 後才跑 —— 放在 step 4b 之後，見下)
    // 2. flush boundary changes —— 在 GPU 上其實不需要「ProcessQueueOneLevel」這步，直接進 DAG eval（DAG 會把 clk 的影響傳開）
    // 3. DAG eval（EvalOrder，按拓樸分層）
    for (level = 0; level < NUM_LEVELS; level++) {
        for (i = tid; i < level_count[level]; i += NUM_THREADS) {
            int v = level_node_ids[level_start[level] + i];
            node[v] = <inline lowered NextExpr[v], 讀 node[]/prev[]>;
        }
        GroupMemoryBarrierWithGroupSync();
    }
    // 4. SCC fixed-K
    for (k = 0; k < K_SCC; k++) {
        for (each SCC s) for (i = tid-strided over SCC s's Gauss-Seidel order) node[v] = <NextExpr[v]>;
        GroupMemoryBarrierWithGroupSync();   // (per-SCC barrier, or one barrier per k —— Gauss-Seidel 要序列、所以其實 SCC 內不能 thread-parallel；殘餘 SCC 小（最大 496），一條 thread 跑一個 SCC、SCC 之間 parallel)
    }
    // 5. bus resolver (S0/S1/W1) + handler 注入(memory handler 在這之前跑、設好 SetHigh/SetLow flags)
    if (tid == 0 || strided) { do memory R/W: node[BD0..7] ← rom/ram[addr]; }
    GroupMemoryBarrierWithGroupSync();
    for (kb = 0; kb < K_BUS; kb++) { Resolve_Buses(); GroupMemoryBarrierWithGroupSync(); }
    // 6. 再跑一遍 DAG（讓 bus / memory data 傳開）—— 或整個 3→6 包進一個 outer K_PINGPONG loop
    // 7. (可選) framebuffer 寫入 —— PPU 的 pixel-out node 變了就寫 fb[]
}
```

注意：**PPU 的 precharged-dynamic 讀出路徑**（palette / sprite RAM —— 就是 ping-pong run-path（firing 13）撞牆的那個）—— 在 GPU 上一樣是問題（「settle 到 fixpoint」vs S1 的 within-half-cycle event sequencing）。**選項**：(a) 跟 CPU runtime 一樣留個 fallback —— 但 GPU 上沒有 S1 的 event-driven ProcessQueue 可以 call → 不行；(b) **接受 float-artifact 豁免 + 那 ~130 個 precharged bitline 用更精細的「2-phase」模型**（precharge phase 一個 barrier、discharge phase 一個 barrier —— 把那個 within-half-cycle sequence 顯式做出來）—— 工程量中等、但可行；(c) 先**只驗 CPU 路（2A03）或不含 PPU 讀出的部分**、PPU display 之後再說。先 (b) 或 (c)。

### 4.3 `frame` dispatch（N 個半週期）

最外層再包：`for hc in 0..N: { step 的內容 }`（N=1000 之類）。一幀 ~654719 半週期 → ~654 個 dispatch。或乾脆一個 dispatch 跑滿一幀（654719 次內層 loop）—— 但 D3D12 有 TDR（GPU timeout，預設 ~2s）→ 太久會被 OS 砍 → 所以分 N=1000~10000 一批比較安全。

## 5. codegen 路徑

新增 `IrEngine.EmitHlslCompute()`（跟 `EmitCsharpSource` / `LlvmCodegen.EmitIR` 平行）：
- `HlslExpr` lowering：`NodeRef(id)`→`node[id]`、`Hold/Prev(id)`→`prev[id]`、`Const`→`0`/`1`、`Not(x)`→`(~{x} & 1)`（或 `(1-{x})`，工作在 0/1）、`And`→`({a} & {b})`、`Or`→`({a} | {b})`、`Mux(c,a,b)`→`({c} != 0 ? {a} : {b})`。
- emit 出：`[numthreads(...)] void step(...)` —— DAG 部分按 `IrEngine.EvalOrder` **再分層**（拓樸 level）、層之間 `GroupMemoryBarrierWithGroupSync()`；SCC 部分照 `IrEngine.SccEvalOrders` emit `Step_scc_fixedK`；bus 部分照 `IrEngine.BusNodes` emit `Resolve_Buses`（已有 emit 邏輯、改 HLSL 語法）；handler 部分（clock toggle + memory R/W）—— 這部分不是 IR、要 hand-write 進 kernel template（master clk node id、memory module 的 cs/we/addr/data node id list、`u1.ram`/`cart.prg`/`cart.chr` 的對應 buffer）。
- + schedule 生成：`level_start[]` / `level_node_ids[]`（從 EvalOrder 算拓樸層）—— 寫成 HLSL `static const int[]` 或上傳成 buffer。
- ComputeSharp 那邊：把 emit 出來的 HLSL 字串餵給 ComputeSharp 的 `GraphicsDevice.LoadComputeShader`（如果它支援 raw HLSL string；不然用 Silk.NET + dxc 編 DXIL + D3D12 create PSO）。

`--dump-emitted-hlsl [file]`（看 HLSL）、`--gpu-step` / `--gpu-frame`（runtime 用 GPU step）、`--gpu-trace-cmp`（GPU step vs S1 lockstep 比對）。

## 7. 步驟拆解（phases）

| phase | 內容 |
|---|---|
| **G.0** | setup：加 ComputeSharp NuGet（或 Silk.NET）；寫一個「spike」compute kernel（`out[i] = in[i] + 1` 之類）、headless dispatch、readback 驗 → 確認 D3D12 compute toolchain 在這台機器 / 4080 上能跑。（類比 S4.5 的 `LlvmSpike`。）|
| **G.1** | 拓樸分層 EvalOrder（`level_start[]`/`level_node_ids[]`）+ schedule 生成；state layout 定案（byte-pack）；`EmitHlslCompute` 的骨架（只 emit DAG 部分、層之間 barrier、handler / SCC / bus 先 stub 成「跟 CPU 一樣留 S1」—— 但 GPU 上不能 call S1，所以先只跑「純 DAG、其餘 node 維持 prev」、`--gpu-trace-cmp` 看 DAG node 對不對）。|
| **G.2** | 加 SCC fixed-K block（HLSL）+ bus Resolve_Buses block（HLSL）+ handler chain on-GPU（clock toggle + memory R/W from buffers）→ `--gpu-trace-cmp` 0 mismatch（含 float 豁免；PPU precharged 路徑用 §4.2(b) 的 2-phase 模型 或先 (c) 跳過 PPU 讀出）。|
| **G.3** | `frame` dispatch（N 個半週期/dispatch、TDR-safe 分批）；framebuffer on-GPU 寫入 + readback → 跑得出畫面。|
| **G.4** | benchmark vs S1 / LLVM / C# —— **量出「GPU 一台」到底比 S1 快還慢**。如果慢：profile（barrier latency? occupancy? bank conflict?）、試多 workgroup 攤 schedule、試 bit-pack state。如果快：🎉。|
| **G.5**（可選）| bit-sliced GPU 變體（一條 thread = 一台 NES、32-lane SWAR per `uint`、dispatch N threads）—— 這個 GPU 一定壓倒性贏（embarrassingly parallel）；只在 user 想要「多台」時做（S4.4a 的 bit-sliced emit 是起點）。|

## 8. 附：硬體描述語言（HDL）輸出 —— 可行嗎？

**可行，而且其實很自然。** IR（`NextExpr[v]` 的 boolean tree over `NodeRef`/`Hold`/`Prev`/`And`/`Or`/`Not`/`Mux` + 殘餘 SCC + bus + 行為式 memory）基本上就是一份**可綜合的 RTL netlist**：

- **acyclic 部分（EvalOrder ~11680 node）= 純組合邏輯** → 直接對應 Verilog `assign` / VHDL concurrent assignment。`NextExpr[v] = And(NodeRef(a), Not(NodeRef(b)))` → `assign n{v} = n{a} & ~n{b};`。`Mux(c,a,b)` → `assign n{v} = c ? a : b;`（或 `(a & c) | (b & ~c)`）。
- **sequential node（`Hold(self)` / `Mux(clk, data, Hold(self))` 那種透明 latch）** → `always @* if (clk) n{v} = data;`（透明 latch）或 `always @(posedge clk) n{v} <= data;`（edge-triggered FF）—— 哪種要看 S2 抽出來的 NextExpr 形狀（多半是 latch）。`Hold`/`Prev` 的「半週期開頭的值」語意在 Verilog 裡沒有原生對應 → 用一個 registered copy（`n{v}_prev`）+ clk 邊緣更新來模型化、或直接讓 sequential node 由 master clock 驅動。
- **殘餘 SCC（cross-coupled inverter = SR latch、counter、shift-reg）** → `always @(posedge clk)` block 或實例化 standard cell；fixed-K 那套是「模擬器怎麼算」、HDL 裡這些就是真正有狀態的 reg。
- **hybrid pass-transistor bus（tri-state）** → Verilog `tri`/`wire` + 多個 `assign x = en ? val : 1'bz;` driver；或用 S0/S1/W1 model 解成 priority mux（`x = has_s0 ? 0 : has_s1 ? 1 : has_w1 ? 1 : x`） —— **後者比較好**（綜合工具對 `1'bz` 跟 NMOS depletion-load pull-up 會頭痛；用 resolved IR、不用 raw transistor netlist）。
- **行為式 RAM/ROM** → `reg [7:0] mem [0:65535];` block（+ `$readmemh` 載 ROM）。
- **handler 的 clock toggle** → testbench 的 clock generator。

**所以**：可以 emit 一個 Verilog module（2A03 + 2C02 + board），邏輯等價於 S1（modulo float-artifact 豁免）。**用途**：
1. **餵給 Verilog 模擬器**（Verilator / Icarus）—— 又一個跑 / 驗的後端（Verilator 把 Verilog 編成 C++→native，常常超快 —— 可能比我們現在的 S1 還快！）；
2. **餵給 FPGA 綜合流程**（Yosys / Vivado）—— 把真正的 NES 晶片跑在 FPGA 上（!）；
3. 純當「這份 IR 是 synthesizable」的 demonstration。

**caveat**：那會是個**從 switch-level 推導出來的 gate-level netlist**、不是手寫的乾淨 RTL —— 但因為 IR 已經把 bus（S0/S1/W1 解掉了）跟 latch（SCC solver 處理了）resolve 得差不多了，emit 一個「cleaned-up」Verilog（用 resolved IR、不用 raw transistor）是可行的、綜合工具應該吞得下。module 會很大（~14738 wire、~27K transistor→gate）、綜合慢、但是個真的 HDL artifact。

**規劃**：`--dump-emitted-verilog [file]` —— `EmitCsharpSource` 的 `CsExpr` lowering → 一個 `VerilogExpr` lowering（`NodeRef(id)`→`n{id}`、`Not(x)`→`~({x})`、`And`→`({a} & {b})`、`Or`→`({a} | {b})`、`Mux(c,a,b)`→`({c} ? {a} : {b})`、`Hold(id)`→`n{id}_q`（一個 registered copy））；EvalOrder → `assign` block、sequential/SCC node → `always` block、bus → resolved priority mux、memory → `reg [] mem []`。**優先序低**（不在 S4.6 的關鍵路徑上），但是個「順手 1-2 天能做」的 bonus artifact —— user 想要的話隨時可以加。
