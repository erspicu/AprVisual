這是一個非常精準且具備極高可行性的優化計畫。從 Stage A（乾淨 SRAM cell）推進到 Stage A2（混入 load 的 NOR-NOR latch），再用 Stage D（拓樸學上的暴力解環）來收底，完全符合 EDA 工具（如 Verilator）處理 switch-level 或 un-clocked netlist 的標準降階策略。

針對你的 5 個開放問題，以下是逐一的深入分析與具體建議：

### 1. Stage A2 的 pattern 判據與 NAND-NAND 支援
*   **分析**：6502 與 2C02 都是 **NMOS 製程**。NMOS 的基本邏輯閘是 NOR（多個 transistor 並聯下地），串聯（NAND）會造成電壓降與速度變慢，因此 NMOS 晶片裡絕大多數的 latch 都是 **NOR-NOR cross-coupled**。至於板上的 TTL 晶片（如 74LS373），雖然邏輯上可能是 NAND-NAND 構成，但在 Visual6502/chipsim 的 switch-level 萃取下，很多時候也被等效成了 pull-down 網路配上 static load。我們目前的 `DriveInfo` 缺乏「PullUp 依賴 partner」的表達能力（NMOS 的 PullUp 通常只是 depletion mode FET，永遠導通）。
*   **對計畫的影響**：若硬要支援 NAND-NAND，會需要大改 `DriveInfo` 的萃取邏輯，投資報酬率極低。
*   **該怎麼改**：**維持現狀，只抓 NOR-NOR。** Stage A2 專心抓「`PullDown` 含 `NodeRef(partner)` 且有 static pull-up」的 pattern 就足夠吃下 6502/2C02 絕大多數的殘餘 latch。漏網的（或 TTL 的特例）直接放給 Stage D 去暴力打斷即可。

### 2. Stage A2 的 contention model 與 Circular Risk
*   **分析**：你提出的 model 是 `NextExpr[S] = Mux(rest_a, 0, Mux(rest_b, 1, Prev(S)))` 與 `NextExpr[C] = Not(Node(S))`。
    *   **Contention 議題**：當 `rest_a` 與 `rest_b` 同時為 1（兩邊 PullDown 同時導通），真實硬體（GND wins）會讓 a 和 b 都變成 0。但在你的 model 裡，S 會變 0，而 C 被強制 `Not(S)` 變成了 1。這在暫態（transient state）會產生一個與真實 S1 不同的行為。
    *   **Circular Risk 議題**：`rest_a` 確實有可能不小心引用了 `a` 自己的狀態（雖然少見），若發生，直接代入會產生自己依賴自己的 cycle。
*   **對計畫的影響**：`Not(S)` 在極端寫入衝突時可能導致錯誤狀態鎖死。
*   **該怎麼改**：
    1.  **放棄 `Not(Node(S))`，改採對稱生成**：
        `NextExpr[a] = Mux(rest_a, 0, Mux(rest_b, 1, Prev(a)))`
        `NextExpr[b] = Mux(rest_b, 0, Mux(rest_a, 1, Prev(b)))`
        這樣當兩邊 `rest` 都為 1 時，a 和 b 都會 evaluation 出 0，完美符合 GND wins 的物理現實，也省去區分誰是 S、誰是 C 的麻煩。
    2.  **加入防禦性檢查**：在 Stage A2 決定套用前，掃描提取出的 `rest_a` 與 `rest_b` Expr，若發現裡面含有 `NodeRef(a)` 或 `NodeRef(b)`，則 **放棄該 pair，交給 Stage D**。

### 3. Stage D 的 blind back-edge `Prev` 打斷
*   **分析**：這一步非常聰明，也是 Verilator 等工具處理 combinational loop 的做法。在沒有 true within-phase 組合邏輯迴圈的前提下，環一定是「跨 cycle 延遲」或「被漏掉的 latch」。
    *   **Blind DFS 選邊安全嗎？** 安全。在一個環狀結構中，切斷「任何一條邊」並補上 `Prev`，在數學上等同於在該處插入一個 1-tick delay（在我們的系統中就是半個 clock phase）。因為狀態最終會收斂，切斷哪條邊頂多影響該半週期的「暫態傳遞速度」，不會改變穩態結果。
    *   **需要引入 clk0/clk1 互斥來挑邊嗎？** 不需要。這會大幅增加複雜度，且違反了 IR engine 作為 local 拓樸分析的純粹性。
*   **對計畫的影響**：Blind 打斷效能極高，實作簡單，能完美切開剩下的 pipeline 環。
*   **該怎麼改**：**維持 Blind 打斷的設計。** 不用花時間去算哪條 back-edge 比較「對」。直接在 Tarjan DFS 中遇到 `visited` 但還沒 `finished` 的 node 時（這就是 back-edge 的定義），將該 NodeRef 替換為 PrevExpr 即可。

### 4. Pipeline 結構的放置位置
*   **分析**：Stage A2 依賴電氣特性（`DriveInfo` 的 PullDown/Up），屬於語意層次；Stage D 依賴圖論拓樸（Tarjan SCC），屬於結構層次。
*   **對計畫的影響**：放對位置會讓程式碼解耦（Decoupling），易於維護與除錯。
*   **該怎麼改**：**你的設計完全正確，不用改。**
    *   Stage A2 放在 `SccModel.Build`：在這裡它能輕易存取 `DriveInfo`，做完後標記為 Hybrid 移出待處理清單。
    *   Stage D 放在 `IrEngine.BuildEvalOrder`：在這裡專心算圖論，不需要知道 `DriveInfo` 是什麼，只要看到環就砍。

### 5. 關於預期：713-node 大 SCC 的真面目
*   **分析**：在 2C02 這種晶片中，出現 713 個 node 的 SCC，絕對不會是一個 713-deep 的真實邏輯環。這通常被稱為 **"Tangle"（糾纏區）**。它的成因是：
    1.  未被打斷的 2-cycle latch 彼此透過雙向 pass-transistor（傳輸閘）或 bus 網路連在一起。
    2.  多相位時脈（phi1/phi2）控制的交錯 shift-register（例如 2C02 的背景渲染 pipeline），因為 IR 看不到時脈互斥，導致 A → B → C → A 在無條件評估下形成巨大迴圈。
*   **對計畫的影響**：
    *   Stage A2 會像拆解積木一樣，把這個 713-node tangle 裡面的「節點黏合劑」（NOR-NOR latch）大量消除。
    *   做完 Stage A2 後，713 很可能會立刻碎裂成數個 10~50 node 的中型 SCC。
    *   接著 Stage D 的 DFS 進場，會在這些剩下的中/大型 SCC 中找到 back-edge 砍斷，幾輪 Tarjan 迭代後就會完全消滅。
*   **預期指標**：預計 Stage A2 + Stage D 實作後，residual SCC node 數量會從 5824 驟降至 **0**（或者非常接近 0 的邊緣 case）。IR driving coverage 會非常順利地衝破 80% 甚至更高。
