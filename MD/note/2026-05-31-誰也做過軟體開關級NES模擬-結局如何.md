# 誰也做過「軟體 switch-level NES / 6502 模擬」—— 以及他們最後成功了嗎?

> 2026-05-31 的領域盤點筆記。源於一個問題:「訓練資料裡是不是很多人在處理這種 issue?他們最後成功了嗎?」
> 結論先講:**這是個很小的利基,不是熱門題目;做過軟體版的那幾位都做出了「正確、可驗證」的模擬器(那層成功了),但沒有任何一個達到 real-time;想追速度的最後多半放棄或回去做傳統模擬器。real-time 只在離開「軟體 switch-level」約束(改用 FPGA / RTL-compile)時才有人達成。**
> 數字的可追溯來源見 `WebSite/comparison.html`(整理自公開、可查證的 README / nesdev thread / 媒體報導)。

## 1. 學術界:這條線在 1990 年代後冷掉了

純 switch-level 模擬加速的學術研究,在深次微米時代後基本被工業界放棄 —— 純邏輯的 switch 模型失去物理準確性,工業界全面轉向 FastSPICE(簡化矩陣求解)、gate-level、RTL。因此 2000 年後幾乎沒有人在研究「switch-level event-driven BFS 的演算法加速」。

**所以「很多人在處理這個 issue」是錯覺。** 真正在搞「軟體、單核、switch-level NES / 6502」的,是少數幾個 hobbyist / 研究型專案,不是活躍的學術社群。(此結論與 2026-05-31 對 Gemini 3.1 Pro 的兩次諮詢 + 我們自己的窮舉一致;見 `WebSite/landscape.html`。)

## 2. 實際做過軟體 transistor / switch-level NES / 6502 的人,結局如何

| 專案 / 人 | 成果 | 速度 | 「成功」了嗎? |
|---|---|---|---|
| **Visual6502**(Greg James / Silverman 兄弟等,2010–11) | 原始 JS switch-level 6502(chipsim.js)+ 視覺化 | ~1 clock/s(動畫)→ expert mode ~250Hz → C port ~1kHz | ✅ 達成「正確 + 可視化教育」目標;❌ 本來就沒打算追 real-time |
| **perfect6502**(Michael Steil + Brian Silverman) | 乾淨的 C golden 6502 NMOS netlist,half-cycle 精準 | ~1/30 of 1MHz 6502(單顆 CPU,無 PPU) | ✅ 成為公認的 golden CPU model;❌ README 明說「電晶體級就是貴」,不追速度 |
| **Visual NES**(Sour —— 傳統快速模擬器 Mesen 的作者) | 合併 Visual 2A03 + 2C02、能跑 NES ROM 的 C++/C# | ~1/1000 實機、~5000Hz、~1 frame/min;data-shrink + PGO 後 ~7500Hz | ⚠️ 能跑,但 **2022-05-13 archived**;作者**回去維護傳統模擬器 Mesen** —— 等於放棄了這條速度路 |
| **MetalNES**(iaddis —— 我們 S1 的參考) | 完整 NES-001 主機板電晶體級模擬(macOS,含支援晶片 + composite/audio ladder) | 媒體/使用者轉述「分鐘/frame」 | ⚠️ 能跑(很厲害),但作者自註「needs lots of optimization」,開發停滯 |

**核心觀察:做過軟體 switch-level NES 的人,沒有一個達到 real-time。** 大多把它當研究 / 可視化 / 正確性目標(那層都成功了);而真正想追速度的(Sour),最後選擇放棄、回去做傳統行為級模擬器。

### Sour 的獨立發現(2017 nesdev thread)—— 與我們一模一樣
- ~50–60% 時間花在 recursive group function(= 我們的 BFS group-walk 熱點)。
- 多執行緒很難(大量 lock contention)。
- vector → hashset 反而更慢;bool presence array 沒幫助(慢 1–2%)。
- 真正的進步來自 int → short + PGO(~5000 → ~7500Hz)。

→ **兩個獨立專案撞同一面牆、得到同一組教訓(data-shrink / cache / PGO 贏,多執行緒與花俏資料結構輸),這本身就證明那是真的天花板,不是我們漏了什麼招。**

## 3. 唯一「達到 real-time」的,都離開了這個約束

FPGA / Verilator(RTL-compile)能跑到 1MHz+ real-time —— **但那是硬體 / RTL 編譯,正是 AprVisual 刻意排除的路**。換句話說:real-time 只在「不堅持軟體 switch-level」時才有人成功;堅持軟體 switch-level 的,全部停在數十秒~數分鐘/frame。

## 4. 那 AprVisual 算成功還是失敗?(看定義)

- **real-time 軟體 switch-level NES**:**沒有人成功過**;而且分析(~94% 雙向 pass-transistor SCC、~550× 差距、所有獨立專案撞同一牆)指向這是**結構性天花板**,不是「沒人夠拚」。
- **正確、可驗證、而且夠快的 switch-level 模擬**:這層可達成,而 **AprVisual 很可能是目前公開最快的軟體版**(R-1 後 ~78K hc/s、~9 s/frame,明顯快過 Visual NES 與 MetalNES 的公開數字;C# 與 Rust 雙引擎位元一致互相驗證)。

## 5. 誠實總結

我們不是在做一件「別人做成功、我們還沒追上」的事 —— 而是在一個**小眾、且 real-time 被結構性封死**的題目上,站到了「軟體 switch-level」這群的**最快前緣**。

因此「成功」的合理定義應是:**正確 + 可驗證 + 在這條路上最快 + 一份誠實的優化知識庫(含完整的負面案例 / dead-end 清單)**,而不是 real-time(那個沒人能在軟體裡做到)。把 S1 衝到天花板、留下 `landscape.html` 與「beat-S1 守則」,本身就是這個小領域裡少見的、完整且誠實的收尾。
