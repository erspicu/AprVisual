Your diagnosis of why Escape A failed is **100% correct**. You bloated your hottest L1 array (`NodeStates`/Prune-mask) from 14.7 KB to 59 KB. Zen 2 has a 32 KB L1d cache. You forced your absolute highest-traffic random reads to spill to L2 (12 cycle penalty vs 4 cycle). No amount of instruction reduction can outrun a geometric increase in L1 capacity misses on a latency-bound loop. 

Here are the crisp answers to your questions, including the **only remaining novel lever (Q3)**. If Q3 doesn't get you in the green, it is structurally dead.

### 1. Shift maintenance to event-driven (supply-gate edge)?
**CRISP NO.** 
To incrementally maintain `dom`, you'd need to store the running count of ON supply transistors for each node (another array). When a gate toggles, you'd walk its transistors, do a read-modify-write on those counts, and conditionally update `dom`. 
In event-driven switch-level sims, nodes evaluate *because* gates toggled. Toggling gates touch roughly the same order of nodes as the resolution loop. You are just trading a random gather (reading supply gates during resolution) for a random scatter (RMW on counts/dom during gate toggle). Scattered RMWs are strictly worse for your store buffers and cache than gathers.

### 2. Conditional write (`if (dom != x) dom = x`)?
**CRISP NO.**
On Zen 2 (and all modern x86), a write to a cold cache line triggers a Read-For-Ownership (RFO). A read miss *also* fetches the line. 
If `Bypass[nn]` is not in L1, the conditional read stalls the pipeline for 12-40 cycles to fetch it from L2/L3. If you just did the store, the hardware would fetch it (RFO) with the same latency (though slightly more decoupled via the store buffer). Because you are single-threaded, there is zero MESI invalidation traffic to other cores to save. You pay the miss latency either way, but now you've added a branch and a dependent load.

### 3. A cheaper 1-bit invariant? (THE NOVEL LEVER)
**YES. This is your silver bullet for the memory problem.**
You can entirely drop the `dom` gate ID, drop the `Bypass` array, and achieve a sound skip using a 1-bit `IsPinned` flag packed into your existing 1-byte `NodeStates` array. **Zero cache bloat, zero new writes.**

**The Logic:**
An `IsPinned` node is one driven by exactly ONE supply transistor, and NO opposing supply transistors. 
When gate `nn` turns off, the walk iterates over the transistors `T` controlled by `nn`. 
Suppose `T` connects endpoint `c` to `d`.
*   **Case A (`d` is GND/PWR):** `T` is a supply transistor. Since it is turning off, `nn` *might* be the single gate pinning `c`. You **CANNOT** skip. Enqueue `c`.
*   **Case B (`d` is a regular node):** `T` is a pass transistor. Because it's a pass transistor, `T` is *not* connecting `c` to supply. Therefore, `nn` is mathematically **NOT** the gate pinning `c` via this specific connection. If `c` has `IsPinned == 1`, you are 100% certain it is pinned by some *other* gate. You **CAN** safely skip enqueuing `c`.
*(Note: If `nn` controls BOTH a pass transistor and a supply transistor connected to `c`, Case A will enqueue it and Case B will skip it. The node still gets evaluated, which is sound).*

**The Implementation:**
1. Revert `NodeStates` back to 1 byte (restore your 14.7 KB L1 footprint).
2. Steal 1 bit in `NodeStates` for `Pinned` (e.g., `0x80`).
3. In `RecalcNodeFastDom`, maintain it for free in registers before the state write:
   ```csharp
   bool pinned = (gndCount == 1 && pwrCount == 0) || (gndCount == 0 && pwrCount == 1);
   if (pinned) flags |= 0x80; // or fold it into your FlagsToState lookup table!
   SetNodeState(nn, FlagsToState[flags]); // Writes state + pinned bit simultaneously
   // REMOVE the Bypass[nn].Dom write entirely.
   ```
4. In the turn-off walk (gate `nn` went low, processing pass-transistor to `c`):
   ```csharp
   // Only for PASS transistors (not supply transistors)
   if ((NodeStates[c] & PinnedBit) != 0) { /* SKIP */ }
   ```

### 4. The Honest Verdict
If the 1-bit `IsPinned` trick above does not yield a net positive, **STOP. It is structurally dead.**

**The One-Line Reason:** 
*The architectural cost of fetching a cold cache line (~12-40 cycles) or executing branchy metadata-maintenance instructions strictly exceeds the compute cycles saved by a 3.4% prune rate.*

You are hunting for a +3.4% win. That means for every 100 nodes you evaluate, you save 3.4 evaluations. If an evaluation costs ~30 cycles, you are saving ~100 cycles per 100 nodes. That gives you an overhead budget of exactly **1 cycle per node**. 
You cannot write *any* runtime metadata to a separate array in 1 cycle per node due to L1 spills and RFOs. The only mathematically possible way to win is if the metadata is updated *in the exact same instruction/cache-line* as the mandatory state update, which the 1-bit `IsPinned` trick does. If that still fails, your engine is too fast and the P-5 branch is just overhead. Rip it out.
