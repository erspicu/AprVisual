## 一、這顆是什麼(測試原始碼 `AC_ref/AccuracyCoin-main/AccuracyCoin.asm` → `TEST_ALERead`)

源自 Damian Yerrick 的 `boing2k7.nes`(https://forums.nesdev.org/viewtopic.php?p=62806#p62806)。
**Test 2:「一個抓準時機的 `LDA $2007` 能不能污染背景的低位面 fetch?」答案:能。**

### 測試如何布置(逐行)
```asm
JSR ClearNametable2_WithF0 ; CHR $F0 透明 tile,但特意取 CHR 資料的 $Fx 列
; palette 3 色 1 = 亮藍(讓 artifact 可見)
JSR SetUpSpriteZero
.byte $02, $C0, $00, $F4    ; Y=掃描線3, CHR=$C0(左上單一像素), X=$F4=244
JSR Sync_ToLine0Dot1
JSR ClockslideFromWord / .word 412   ; 對齊 2
LDA $2007          ; 讀 $2007 污染背景讀取節奏
NOP / NOP
LDA $2002 / AND #$40 / BNE Pass   ; 檢查 sprite-0 hit
; 沒中 → INC $200(sprite 下移一線)→ clockslide 526(對齊 3)→ 再試 → 沒中就 FAIL
```
**通過條件**:兩個對齊(412 / 526 clockslide)**任一**產生 sprite-0 hit 即 PASS。

### 硬體上該發生什麼(測試註解,關鍵)
- dot 227:ALE 由背景讀取節奏設起,PAR 決定下週期讀 $2FC7(屬性表);位址匯流排=$2FC7,八位閂鎖=$C7。
- dot 228:背景節奏讀 $2FC7 → 得 $FF;位址/資料共用 8 pin → 14 pin 全變 $2FFF。
- **dot 229:ALE 和 Read 同時設起 → 八位閂鎖進入回授迴圈**:
  - 匯流排 $2FFF + ALE 設 → $FF 進閂鎖(本來就 $FF)。
  - 同時在讀 → 讀 $2FFF 得 $FF 上資料匯流排 → ALE 設 → $FF 又回閂鎖 → 讀 $2FFF …(作者刻意讓這不穩定迴圈**穩定在 $FF**)。
  - PAR 決定讀 $0F03(CHR $F0 第 3 列),位址匯流排變 $0F03,**但八位閂鎖仍是 $FF**(作者原話:「I'm not sure why it wasn't updated to $03, but that's how it appears to work out」)。
  - 更新讀取緩衝:八位閂鎖 $FF → **從 $0FFF 讀**,PPU 位址匯流排變 $0FFF。
- dot 230:PAR 說 $0F03,但八位閂鎖 $FF → **實際從 $0FFF 讀 → 低位面載入 $FF**。
- 結果:shift register = `%0000000011111111`(儘管 nametable 全透明),**dots 241-248 畫成 palette 3 色 1**。
- sprite 0 單像素在 X=244(dot 244,落在 241-248 內)→ **sprite-0 hit 偵測得到!**

## 三、根因分類與為何我的 shim 失敗

**分類:A 類引擎語意極限 + 板級建模(74LS373 類比回授)**。這不是 netlist 缺管(E 類)——
2C02 幾何審計 0 洞。是**板上外部八位閂鎖('373)在 $2007-讀取與背景-讀取節奏巧合時的類比回授**,
S1 的節點圖建模 PPU **內部**讀取,不含板上這顆晶片的類比行為。**測試作者本人都說不完全懂**這個更新怪癖。

**我這次的失敗嘗試(已還原)**:在 `WireCore.Handlers.cs DoMemRead` 加「追蹤八位閂鎖低位元組
`_octalLatchLow` + ALE+Read 重疊時從它起算不動點」。**掛錯點**:
- shim 的重疊條件 `NodeStates[cb.PpuAle]!=0 && NodeStates[cb.PpuRead]==0`(rd=/RD,低=讀取中)。
- 但 [ae] 顯示 dot 229 是 **ale=1 && rd=1**(ale/rd 在此 trace 永遠同值:奇 dot 都 1、偶 dot 都 0)。
- 更關鍵:**CHR ROM 回呼只在讀取事件觸發那一刻執行**,而那一刻 settle 已把節點解到不重疊;
  [shimIN]/[shimfp] 在汙染窗口(t 41M-43M)**零輸出** → 重疊在回呼視角根本不存在。
- 金 checksum 不變(full_palette 不觸發),但對 ALERead **完全無效**。→ 已 `git checkout` 還原。

## 四、下一個 session 的建議施力點

目標:讓 S1 在此測試的低位面 fetch 從 **$0FFF** 讀(而非 $0F03),使背景在 dot 241-248 不透明、
涵蓋 dot 244(sprite 0 單像素)→ sprite-0 hit。兩個對齊(412/526)任一中即可。

1. **不要再掛 CHR 回呼的 ale/rd 重疊**(已證回呼視角看不到)。改用**步進層**(像 dot-339 shim 那樣,
   在 `Recalc.cs` step 尾端跑):偵測 `read_2007_trigger` 觸發 + 當前在可見線背景 fetch 節奏 →
   強制下一個低位面 fetch 的 CHR 位址低位元組 = 前一次 PPU AD 匯流排低位元組(八位閂鎖持有值)。
2. **八位閂鎖持有值追蹤本身是對的**:只在 PPU AD 匯流排(`cb.PpuAle != EmptyNode`,即 cart.chr)更新,
   不被 CPU 側讀取覆寫。這段邏輯可留用,只是觸發點要換。
3. **驗證預算**:孤立 ROM `--max-frames 200` 夠(artifact 在 v=3,早期);AprNes oracle 對照
   低位面 fetch 位址。金 checksum 必須維持 0x794A43ABDF169ADA。
4. **診斷探針**:`[ae]`(逐 dot ale/rd/chrAb/selPat,armed on read_2007_trigger)和 `[ar]`
   (LDA $2007 END + artifact/hit)都在 `WireCore.System.cs DmaProbeStep`,OB_DEBUG-gated,已 commit,可重用。
5. **風險提醒**:這是脆弱的板級巧合 shim。兩個對齊(412/526)可能落在不同 dot,shim 要對兩者都成立
   或至少一個。先確認 alignment-2(clockslide 412)在 S1 落在哪個 dot,對準它。
