### 1. 「Reuse 現成的 vs Rewrite」的判準？

你的判準（「你要改它嗎 + 它要跟你後面的東西緊密耦合嗎」）是非常精準的。但在「我不確定」的灰色地帶，「不確定就 rewrite」通常是**壞建議**（對一般人而言）。

為什麼？因為重寫（Rewrite）的成本通常不是「行數」，而是**「重新踩雷的成本（Bug-for-bug compatibility）」**。MetalNES 累積了許多邊角料的修復，重寫意味著你可能會漏掉原作者用血淚換來的 edge case。

但**為什麼你在 S1 的 Rewrite 是絕對正確的？**
因為這個案例的關鍵不在於「你要不要改它」，而在於**「執行邊界與狀態擁有權 (Execution Boundary & State Ownership)」**。
*   如果 S1 是 P/Invoke，S1 跟 S2-S4 之間隔著一道 **FFI (Foreign Function Interface) 牆**。
*   在一般的 App，這道牆沒問題（例如 C# 呼叫 C++ 的 SQLite）。
*   但在這個專案裡，S1 跟 S2-S4 是要在 **Tick-by-Tick 的微觀層級**（每半個 clock cycle）進行 `--trace-cmp` 狀態比對的！如果隔著 FFI，你要麼得每秒幾百萬次把 C++ 的 state marshal 到 C#，要麼得搞複雜的 shared memory (memory-mapped file)。這不僅慢，更會讓「驗證正確性」這個這專案最神聖的任務，變得脆弱無比。

所以我的判準會加一條：**「當兩者需要在極高頻率（hot loop 內部）進行狀態同步或深度 Interception（如 `DeferRecalc` hook）時，跨越 FFI 邊界的成本將大於重寫的成本，此時必須 Rewrite。」**
