One engine, one parser — but S1a has diverged on env vars. The S1a fork (src/AprVisual.S1A/) still shares all 71 CLI flags with S1, but no longer the env-var surface below: as of 2026-07-21 S1a runs its M1–M6 physics mechanisms unconditionally (no NO_* toggles, no ALEREAD_MUX/MUX_* — the ALERead mux is baked in) and had its forensic debug probes (OB_DEBUG/…) removed. These env vars are still live on this S1 engine (the golden standard, unchanged). 8 · Diagnostic probes8 · 診斷探針The forensic toolbox取證工具箱

Three probe rules (hard-won): (1) trust a PC sample only when two consecutive half-cycles agree — mid-instruction the PC register half-updates. (2) "budget exhausted" ≠ "hang" — get a frame baseline from AprNes first. (3) sample a bus transaction's value on its last half-cycle — the first half still carries the operand byte. 探針三鐵律(戰役學費):(1) PC 取樣要連續兩拍同值才可信 —— 指令執行中 PC 暫存器會半更新。(2)「預算耗盡」≠「卡死」—— 先用 AprNes 拿幀數基準。(3) 匯流排交易的值取樣「最後一個」半週期 —— 首半的 db 還是運算元位元組。

Budget & timeout — how a run is bounded預算與 timeout —— 一次跑怎麼設界

  • --max-frames N is the primary budget (simulated frames; default 900 ≈ 15 sim-seconds). Exhaust it and the verdict is timeout, exit code 3. A too-small budget looks exactly like a hang — get the true frame count from the AprNes oracle first (tools/estimate_frames.py), don't just raise it blindly.--max-frames N 是主要預算(模擬幀;預設 900 ≈ 15 模擬秒)。耗盡即判 timeout、exit code 3。預算太小看起來就跟卡死一模一樣 —— 先用 AprNes oracle(tools/estimate_frames.py)拿真幀數,別盲目調高。
  • --max-wait N is a wall-clock safety cap in seconds (0 = off); it is secondary to --max-frames.--max-wait N 是 wall-clock 安全上限(秒;0 = 關);次於 --max-frames。
  • The batch runner adds two more guards on top: it caps each ROM at min(maxFrames, 1.5×typicalFrames+5) (a hung/regressed test is killed at ~1.5× its known verdict frame) and a subprocess wall-guard of maxFrames×10+600 s.批次 runner 再加兩道保險:每顆封頂 min(maxFrames, 1.5×typicalFrames+5)(卡死/退化的測試約 1.5 倍就砍),另有子行程 wall-guard maxFrames×10+600 秒。

The detection=none trap. If a class-A ROM reports detection=none / "budget exhausted, no $6000 signature", the engine never saw the blargg signature — almost always the $6000 work RAM wasn't mounted, not a frame-budget problem (raising --max-frames just runs the failure longer). Test mode mounts it automatically; only suspect this if you modified the engine. detection=none 陷阱。若某 A 類 ROM 報 detection=none /「budget exhausted, no $6000 signature」,代表引擎沒看到 blargg 簽名 —— 幾乎都是 $6000 工作 RAM 沒掛,不是幀預算問題(調高 --max-frames 只會讓錯誤跑更久)。test 模式會自動掛;只有你改過引擎才需懷疑這點。
