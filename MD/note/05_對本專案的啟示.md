## 3. 我們要做得不一樣 / 往前推進的（MetalNES 沒做的）

這 5 點正好是我們設計文件主張的「真正有價值的中間層」，也是這個專案相對 MetalNES 的差異化價值：

### 3.1 顯式的迴路偵測 / SCC / 狀態分類

MetalNES 沒做 —— 純靠 `processQueue` 迭代收斂 + 100 次警告（不停手），賭電路會穩定。設計文件 §6.4 / §10 主張：建邏輯依賴圖 → Tarjan 縮 SCC → 分類為 static latch / dynamic storage / bus loop。**做這個的好處不只是「不會 hang」，而是把序向邏輯與組合邏輯分開**，這是後面抽 IR 的前提。

### 3.2 IR（中間表示）

MetalNES 是直譯：每個 half-cycle 重跑 group 演算法。它從來沒把電路「編譯」成布林表達式樹。設計文件 §8.2 主張先建 `Expr` IR（`ConstExpr` / `NodeRefExpr` / `NotExpr` / `AndExpr` / `OrExpr` / `MuxExpr` / `HoldExpr`），這樣才能化簡、emit 多 backend、跨 backend 比對。

### 3.3 Codegen（emit C++ / Verilog / CUDA）

MetalNES 完全沒有。設計文件 §9 主張從 IR emit。

### 3.4 Bit-slicing / 多 instance / GPU 平行

MetalNES 是一個 byte 一個 instance、單執行緒一台 NES。設計文件 §10.2 主張用 `uint32/64` 一條表達式算 32/64 台，SoA layout，GPU kernel。**這是「跑一萬台 NES 給 AI 訓練」這個賣點的技術核心，MetalNES 完全沒碰。**

### 3.5 等價驗證框架

MetalNES 靠跑 test ROM 對結果，不是 node-by-node 比對 CPU vs CUDA（因為沒有 CUDA backend）。設計文件 §13.6 主張用同一份 IR 跑兩個 backend、節點對節點比對 —— 這是「我們的模型可信」的硬證據。

## 5. 建議的下一步（如果要動手）

1. **先把 MetalNES 跑起來、讀懂 `wire_module.cpp` 那 2340 行**（特別是 `wire_compute` 那 ~500 行核心）。這是最快理解「switch-level evaluator 長什麼樣」的方式 —— 比看 chipsim.js 那 204 行更接近一個認真的實作。
2. **把它的 `data/system-def/*.js` 模組格式當作我們 C# parser 的目標格式之一**（或至少參考它怎麼分離 module wrapper / 外部網表 / connection list）。
3. **C# 重實作 `wire_compute` 等價物**作為我們的 Phase 1 CPU evaluator，但從一開始就：
   - `NodeValue` 分四態（Low / High / HoldPrevious / Undefined），不要學它把 hold 跟 high 混在一個 flag
   - 保留 transdefs 第 7 欄的 weak/depletion flag（MetalNES 的 `transdef::unknown_2` 有讀但沒怎麼用 —— 我們應該真的用它區分強/弱驅動）
4. **用同一套 `data/roms/nes-test-roms/` + `$6000` 簽章機制**做高層驗證。
5. **然後才往設計文件 Phase 2→3→4 走**（SCC 分類 → IR → CUDA codegen → bit-slicing），這才是我們相對 MetalNES 的增量價值。

> **核心訊息**：不要把 MetalNES 當競爭對手或要超越的東西 —— 把它當「前 4 個月的免費地圖 + CPU evaluator 的參考解答」。我們的價值在它停下來的地方開始：把「能跑的 CPU 模擬器」變成「能 emit、能比對、能在 GPU 上跑一萬台的可信翻譯流水線」。
