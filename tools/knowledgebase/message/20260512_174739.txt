### 1. S4.1 的「巨大 C# 方法」會不會出事？
**會，絕對會出事。**
*   **RyuJIT 的天花板：** 雖然 Roslyn 編譯這個 AST 沒問題，但 .NET 的 RyuJIT 在面對超過幾千個 statement 或 local variables 的單一 method 時，會遇到兩個問題：(1) **暫存器配置器（Register Allocator）崩潰或超時**；(2) **Bailout to MinOpts**（JIT 放棄最佳化，直接生成極慢的機器碼）。
*   **建議做法（Chunking & Memory Access）：**
    1.  **切 Chunk：** 按 `EvalOrder` 每 500 ~ 1000 個 node 拆成一個 `Step_chunk_N(ref byte state)` 方法。最後用一個主方法循序呼叫它們。
    2.  **不要用 Local Variables 接狀態：** 千萬別宣告 12500 個區域變數！這會炸掉 stack。
    3.  **使用 `ref byte` 陣列指標消除邊界檢查（Bounds Checking）：**
        在 C# emit 裡，傳入狀態陣列，並在每個 Chunk 開頭拿指標：
        ```csharp
        // Emit 這樣的 C# 碼
        static void Step_chunk_1(ref byte state) {
            // state[0] 是 node 0, state[1] 是 node 1
            // RyuJIT 對 ref byte 的連續存取最佳化極好
            state[15] = (byte)(state[10] & ~state[5]);
            // ... 500 行
        }
        ```
    這樣切 chunk 不會變慢，因為狀態本來就塞不進 CPU 的暫存器（12500 bytes 早就溢出到 L1 甚至 L2 cache 了）。依賴 L1 cache 來做 chunk 間的資料傳遞是最快且最穩定的。

---

### 2. fixed-K 的 K 與 SCC 拓樸序
**建議：Per-SCC 決定 K，並利用 S1 (Golden) 來做「Training」。**
*   **拓樸序排法：** 對於每個 SCC，跑一次 DFS，遇到 back-edge 就「暫時剪掉」。對剪掉後的 DAG 進行拓樸排序。這就是該 SCC 內部 Gauss-Seidel 的最佳 in-place 執行順序。
*   **K 的物理意義：** 在這個最佳拓樸序下，K 就是「訊號在同一個半週期內，逆著 back-edge 漣波（Ripple）傳遞的最大次數」。對於大部份計數器，如果拓樸序是 LSB 到 MSB，K=2 通常就穩了（甚至 1 加上下個半週期就能收斂）。
*   **如何算 K？務實解法（Training-based）：**
    不用純靜態分析去算最大 back-edge 路徑，太複雜且悲觀。
    **實作你的想法：** S4 跑的時候掛著 S1。每半週期，SCC 內部跑 `k=1,2,3...` 次，每次跑完檢查「這個 SCC 內的 node state 有沒有變動？（Fixpoint）」。
    記錄下跑完整個馬力歐 / 薩爾達的 ROM 後，**每個 SCC 遇到過的最大 k 值**。然後把這個最大值寫死（或 +1 當保險）進 Emit 裡 `for (int k = 0; k < K_for_this_SCC; k++)`。
    甚至可以完全 Unroll：如果 K=2，就直接 Emit 兩次該 SCC 的程式碼，連 loop branch 都省了。

---

### 3. hybrid bus 抽 pseudo-NextExpr
**建議：(d) 放在 S3 與 S4 之間作為「IR LoweringPass」，且 (a) 規則要符合 NMOS 物理特性。**
*   *(a) 優先序與衝突解決：* NES 是 NMOS 邏輯，Tri-state bus 衝突時通常表現為 **Wired-AND** 或依賴強弱。GND（拉低）的能力遠強於 VCC/Pull-up。
    因此，Pseudo-NextExpr 的構建不該是單純優先序，而應該是：
    `has_pull_down = (en_1 & !val_1) | (en_2 & !val_2) ...` (有任何人輸出 0)
    `has_pull_up = (en_1 & val_1) | (en_2 & val_2) ...` (有任何人輸出 1)
    `bus_next = has_pull_down ? 0 : (has_pull_up ? 1 : Hold(bus))`
    這比巢狀 Mux 更貼近物理，且布林邏輯更扁平。
*   *(b) Hold(bus)：* 對，這就是寄生電容效應。`Hold(bus)` 轉譯成 C# 就是 `state[bus]` 不重新賦值（或 `cur[bus] = prev[bus]`）。
*   *(c) 跟 SCC 耦合：* 沒錯，如果抽完發現它跟其他 node 形成環，**無腦丟進 SCC 交給 fixed-K 處理**。Gauss-Seidel 能完美解掉這種 bus-latch 耦合。

---

### 4. Bit-slicing 的 word width (C# / .NET)
**建議：S4.4 唯一指定 `Vector256<ulong>` (AVX2)。先用 `ulong` 開發驗證。**
*   **分析 Bottleneck：** 14723 nodes，如果用 `ulong` (64台)，狀態陣列約 117 KB（完全塞進 L2 cache，部分在 L1）。這時是 **ALU-bound**（每個 node 至少 1-2 個 ALU ops）。
*   如果上 `Vector256<ulong>` (256台)，狀態陣列變 470 KB（仍在現代 CPU L2/L3 內），但 ALU throughput 直接變 4 倍。這時才會開始轉向 **Memory-bound / L2 Cache Bandwidth-bound**。
*   **開發路徑：**
    1.  **MVP:** 先 Emit `ulong` (64-way)。這最好除錯，C# 的 `& | ^ ~` 直觀。
    2.  **進化:** 在 C# Emit 裡把 `ulong` 替換成 `Vector256<ulong>`，用 `Vector256.BitwiseAnd` 等。只要 CPU 支援 AVX2（近 10 年 CPU 都支援），這是效能甜蜜點。
    3.  **不要用 `Vector<T>`：** 長度由 Runtime 決定，這對你在實作 Memory Gather/Scatter Transpose 時是個噩夢（你不知道一次要 transpose 幾個 lane）。`Vector256` 寬度固定，好寫死最佳化。

---

### 5. 整體 / 漏掉的坑

*(a) 行為式 RAM/ROM 的 Gather/Scatter Transpose 成本*
這是一個 **巨坑**。Bit-slicing 的阿基里斯腱就是 Memory 存取。
在 64-way (`ulong`) 下，每個 clock 你要拿 16-bit x 64 台的 Address，這代表位址被分散在 16 個 `ulong` 裡。
你要做 **AoS to SoA Matrix Transpose**，轉出 64 個 `ushort` 實際記憶體位址，去查表，再把 64 個 `byte` 結果 Transpose 回 8 個 `ulong` 餵給 Data Bus。
**建議：** 這個 Transpose 必須用神仙級的 Bit-twiddling 演算法（例如 Hacker's Delight 裡的 Divide and Conquer transpose），或是直接呼叫 x86 `PDEP`/`PEXT` (BMI2 指令集)。如果用純 Scalar C# 跑 for loop 轉置，**Transpose 的時間會超過那 14723 個 node 計算的總和**。

*(b) Reset / 初始狀態*
如果 64 台跑不同 ROM / Input：完全沒問題。Bit-slicing 架構本來就支援所有 lane 的 state 完全不同。ROM 的差異只要在 Gather 時，用 Lane Index 去查各自的 `byte[][] roms` 即可。

*(c) GPU 路線：SIMT vs Bit-sliced*
**這是最關鍵的架構分水嶺。千萬不要把 Bit-slicing 搬上 GPU！**
*   **CPU 適合 Bit-slicing (SWAR)：** 因為 CPU core 少，ALU 寬度有限（AVX2 = 256bit）。我們用 1 bit 代表一台機器，強行向量化。
*   **GPU 適合 SIMT：** GPU 本身就是硬體層級的 1 thread = 1 instance。
    如果你在 GPU 寫 Bit-sliced（一個 Thread 算 64 台的 ulong），你會遭遇極為可怕的 **Register Pressure** 和 **Memory Divergence**。
*   **GPU 的正確做法：** Compute Shader (HLSL/ComputeSharp)。
    `Thread ID = 模擬機台 ID`。
    State Array 就是 `uint states[NUM_INSTANCES][NUM_NODES]`（或壓縮成 bit-field `uint states[NUM_INSTANCES][NUM_NODES/32]`）。
    每個 Thread 內部**純 Scalar 執行**：`state[15] = state[10] & ~state[5];`
    GPU 的硬體 Warp/Wavefront 會自動把 32/64 個 Threads 合併成向量指令執行。這跟你 S4.1 寫的 C# Scalar 程式碼一模一樣！所以 **S4.1 的 C# Emit 程式碼，幾乎可以字串替換直接變成 GPU Compute Shader 的 Kernel！**
