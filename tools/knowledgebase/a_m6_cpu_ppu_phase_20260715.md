這是一個非常精彩且深入的架構級問題。前四輪我們把 `ALERead` 的實體機制（八位閂鎖與半週期行為）拆解得非常透徹，現在你將這三顆看似不相關的測試（`ALERead`, `dot-339`, `even_odd`）統一收束到同一個 **「跨晶片介面早 1 CPU cycle (3 PPU dot)」** 的根因上，這展現了極高的系統洞察力。

作為你的硬體/架構顧問，我的結論非常明確：**這個「統一相位偏移」是值得挑戰的，但必須用「精準的介面交握延遲（Interface Handshake Delay）」來解，而不是調整全域上電相位或全域時鐘。**

以下是對你四個問題的深度剖析與實作建議：

### 1. 為什麼「早 1 Cycle」？為什麼前人做全域延遲會打壞別的測試？

*   **跨晶片物理延遲被抹零：** 在真實的 NES 主機板上，CPU（2A03）發出 Address、R/W 和 /CE 訊號，經過走線電容、PPU（2C02）的 Pad Buffer（輸入緩衝），再進入 PPU 內部的位址解碼器，最後觸發內部狀態機，這中間有**真實的物理傳播延遲（Propagation Delay）與同步化（Synchronization）過程**。Switch-level 引擎（如 S1）是理想化的，邏輯閘幾乎在同一個 half-cycle 內瞬間 settle，導致 PPU 提早了約 1 個 CPU cycle（24 hc）「看見」並「反應」了這些訊號。
*   **為什麼「全域延遲」會打壞已通過的測試（NMI 陷阱）？**
    這正是模擬器開發的深水區。你目前對齊的相位 `--reset-hold-extra 1` (Offset=7) 讓 `$2002` 的讀取（特別是 VBL flag 邊緣與 NMI 的競態條件）達到了完美的對齊。
    如果你「無腦」地將 CPU 到 PPU 的**所有**訊號（包含 `$2002` 的讀取）都延遲 1 cycle，你雖然修好了 `$2001` 寫入（`dot-339/even_odd`）和 `$2007` 讀取觸發（`ALERead`），但你也把 `$2002` 的讀取時間往後推了 3 dots！這會立刻打破那些對 1-dot 精度要求極高的 NMI-edge 測試。
    **結論：** 不是延錯了量，而是**延遲的粒度太粗**。在真實晶片中，讀取 `$2002`（非同步狀態直通輸出）和觸發 `$2007` 狀態機/寫入 `$2001` 暫存器（需要內部同步與時脈邊緣捕捉），其內部傳播延遲是**不對稱**的。

### 2. 哪種「上游根因」修法最乾淨？

我強烈建議選擇 **(a) 延遲特定的跨晶片交握**。

*   **為什麼不選 (b) 調上電相位？** 上電相位決定的是 CPU 分頻器與 PPU 分頻器的「相對齒輪咬合位置」（四種可能之一）。你現在在 Offset=7 下能過 8 顆 NMI 測試，證明**這個齒輪咬合位置（Golden Alignment）是完全正確的**。你遇到的 3-dot 誤差不是齒輪咬錯格，而是「訊號從一顆晶片傳到另一顆晶片的導線/介面延遲」未建模。
*   **為什麼選 (a)？** 因為這最符合真實硬體的行為。真實硬體的 PPU 在接收到 CPU 的寫入（Write）或狀態機觸發（$2007 Read ALE）時，訊號通常需要經過 PPU 內部的主時脈（Master Clock）重取樣（Re-sampling）或管線化延遲。在 S1 中顯式地對這幾個特定介面補上 1 CPU cycle 的延遲，是架構上最乾淨的做法。
*   **(c) TriCNES 的做法啟示：** TriCNES 屬於 Behavioral/Cycle-accurate 模擬器，他們在程式碼中用 Pipeline 陣列（例如 `delay_queue[3]`）顯式地實作了這個「交握延遲」。S1 作為 netlist 模擬器，本質上依賴網表本身的延遲，但由於網表缺少 Pad 延遲模型，我們必須在 S1 的 **CPU-PPU 橋接層 (Bus Interface)** 手動加上這個 Pipeline。

### 3. S1 中的具體施力點（Implementation Strategy）

要在 S1 (半週期 `hc` 引擎) 中乾淨地實作跨晶片延遲，請實施 **「非對稱匯流排介面延遲 (Asymmetric Bus Delay)」**：

**施力點：在 PPU 的輸入端 (Input Pins) 進行訊號攔截與延遲。**
*   **延遲對象：** 當 CPU 嘗試寫入（`R/W = 0`）任何 PPU 暫存器，或讀取（`R/W = 1`）`$2007` 時。
*   **不要延遲的對象：** CPU 讀取 `$2002` 的狀態時（避免破壞 NMI 時序）。
*   **延遲量：** 1 CPU Cycle = **24 hc** (3 PPU dots)。

**具體做法（虛擬 Shim 層）：**
在 S1 把 CPU 的接腳狀態灌給 PPU 網表之前，插入一個 24hc 的 Delay Buffer。
```c
// 概念碼：在 S1 的 half-cycle step 中
bool is_2001_write = (cpu_addr == 0x2001) && (cpu_rw == WRITE);
bool is_2007_access = (cpu_addr == 0x2007); // 包含 Read 和 Write

if (is_2001_write || is_2007_access) {
    // 延遲 PPU 看到 /CS (Chip Select) 的時間 24hc
    ppu_cs_pin = delay_buffer(cpu_cs_pin, 24); 
} else {
    // $2002 讀取等其他操作，保持 0 延遲 (維持現有 NMI Golden Alignment)
    ppu_cs_pin = cpu_cs_pin;
}
```
**架構意義：** 這樣做等於告訴 S1：「CPU 發出 `$2001` 寫入或 `$2007` 操作後，PPU 的內部邏輯要等 24hc（1 CPU cycle）才會意識到並開始處理。」
*   對於 `dot-339`：`$2001` 的寫入晚 24hc 生效，完美抵銷了之前「早 3 dot 開渲染」的問題。
*   對於 `even_odd`：同上，背景開關晚 24hc 生效，完美修補。
*   對於 `ALERead`：`$2007` 讀取觸發晚 24hc，ALE 發生在正確的點（從 dot 226 退回到 dot 229），八位閂鎖自然抓到正確的 `$FF`，**下游完全不需要再掛 Shim**！

**理由：**
