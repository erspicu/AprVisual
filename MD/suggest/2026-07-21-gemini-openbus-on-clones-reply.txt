這是一個非常硬核且極具創意的 test ROM！利用 NMOS 寄生電容與接面漏電流（Junction Leakage）的 Arrhenius 溫度相依性來做「數位溫度計」，在原廠硬體上是完全成立的物理現象。

然而，針對你的問題：**在較新款或第三方 NES 盜版/相容主機上，這個 open-bus 衰減特性會不會消失或大幅改變？**

**答案是：會的，這個特性在絕大多數的非原廠（及較新）硬體上會「完全消失」，或變成「毫無溫度意義的假象」。**

以下基於半導體物理、ASIC/FPGA 設計實務以及 NESdev 社群的已知驗證，為你逐點分析：

---

### 1. NOAC (NES-on-a-chip) Famiclone (確定會消失/大幅改變)

NOAC（包含各種 Plug-and-play、山寨機、現代廉價掌機）是將 CPU、PPU 邏輯整合成單顆 ASIC。它們幾乎全部採用較舊成熟的 **CMOS** 製程（例如 0.35µm 或 0.18µm）。

*   **物理層面的改變：** CMOS 的漏電機制與早期 8µm / 5µm NMOS 完全不同。雖然 CMOS 也有漏電，但對於「內部匯流排浮接」的處理，現代 ASIC 設計有嚴格的 Design Rule。CMOS 的輸入閘極如果處於浮接（Floating）狀態，會因為電壓漂移到 VDD/2 附近而導致 CMOS 結構同時導通，產生巨大的短路電流（Shoot-through current）燒毀晶片或耗光電量。
*   **ASIC 實作方式（推測與已知行為）：** 為了避免上述問題，NOAC 的設計師在處理 PPU 內部讀寫匯流排時，通常會採用以下三種作法之一：
    1.  **加入 Bus-Keeper (Hold cell) 或真實 Latch：** 只要沒有人驅動匯流排，就用弱回授電路把值鎖死。**結果：永遠不衰減（永不掉 Bit）。**
    2.  **主動 Pull-up / Pull-down：** 讓未驅動的線路直接被弱電阻拉到 $FF 或 $00。**結果：瞬間衰減（在奈秒/微秒級掉到 0，或永遠讀到 $FF）。**
    3.  **單純的串擾（Crosstalk）覆寫：** 如果真的留下浮接節點，由於現代 CMOS 內部線距極小，寄生電容很小，旁邊線路的數位切換雜訊（Coupling noise）會瞬間改變浮接節點的值。它不會是優雅的 RC 放電，而是隨機被其他訊號蓋掉。
*   **結論：** 在 NOAC 上，你的溫度計**絕對量不到溫度**。最可能的結果是讀到死值（Timeout）。

### 2. FPGA / 全數位重製硬體 (確定會消失，可能讀到固定假溫度)

包含 Analogue Nt mini、MiSTer NES core、FPGANES 等。這些是純數位硬體描述語言（HDL）的實作。

*   **數位邏輯沒有 RC 放電：** FPGA 內部的連線不可能讓你讀取「類比漏電」。PPU 的 open-bus 在 HDL 中必須被明確定義為一個 Register（暫存器）。
*   **社群開發者的模擬（已知事實）：** 為了通過 Blargg 的 `ppu_open_bus.nes` 等嚴格的 Test ROM，FPGA 開發者（如 Kevtris 或 MiSTer 團隊）**確實有刻意模擬**這個衰減！
*   **模擬方式：** 他們通常是寫一個數位計數器（Counter）。當 PPU 暫存器被寫入時，計數器重置；經過**固定的 CPU/PPU Clock cycles** 後，強迫將該 Register 清零或逐 bit 清零。
*   **結論：** 在這類高級 FPGA 主機上，你的 ROM 會完美跑完，但會給出一個**「固定不變的假溫度」**（例如永遠顯示 25°C 或開發者當初 hardcode 的某個衰減時間）。它完全不受環境真實溫度影響。

### 3. 現代原廠風格重製 / 新製 NMOS？ (這類硬體不存在)

*   **半導體產業現實：** NMOS 邏輯製程早在 1980 年代末期就被 CMOS 完全淘汰了。現在世界上沒有任何一家晶圓代工廠會為了做 NES clone 而去重啟純 NMOS 產線。
*   **90 年代的 Discrete Clones（如 UMC UA6527/UA6528）：** 這些早期的台灣仿製晶片雖然是獨立封裝，但通常也是早期 CMOS 或混合製程。社群已知 UMC 晶片的 PPU 時序和 open-bus 行為與原廠 Ricoh 晶片有微妙差異。
*   **結論：** 即使你在 90 年代的 UMC 晶片上跑，雖然它*可能*還有浮接漏電現象，但因為晶片 Layout 的寄生電容大小不同、摻雜濃度不同（導致 $E_a$ 活化能不同），你的 ROM 計算出來的溫度會是**完全失準的亂數**（例如在室溫下顯示 80°C 或 -20°C）。

### 4. 你的 ROM 實際執行的預期結果總結

如果把你的「NES 溫度計 ROM」插到各種機器上，這是有根據的預測：

| 主機類型 | 預期讀數 / 執行結果 | 原因解釋 |
| :--- | :--- | :--- |
| **原廠 NES/Famicom (Ricoh)** | **準確的室溫 (會隨吹風機加熱而變)** | NMOS 浮接節點 + 接面漏電的物理現象完美呈現。 |
| **典型 NOAC (廉價山寨機)** | **(a) 永不衰減 (當機/逾時)** 或 **瞬間掉零** | 晶片設計使用 Latch 鎖死，或用了 Pull-down 電阻避免 CMOS 漏電。 |
| **高階 FPGA (MiSTer / Analogue)**| **(b) 某個固定的假溫度 (永遠不變)** | 內部用數位 Counter 模擬了衰減時間，但 Tick 數是寫死的，無視物理溫度。 |
| **多數軟體模擬器 (FCEUX / Mesen)**| **(b) 固定的假溫度** 或 **(a) 永不衰減** | 視模擬器精確度而定。Mesen 有模擬 decay timer (固定)，舊版模擬器則不會掉 bit。 |
| **90年代早期相容晶片 (UMC)** | **(c) 極端異常的溫度讀數** | 可能保有類比漏電，但寄生電容與漏電常數 ($RC$) 與原廠完全不同，公式無法套用。 |

#### 社群佐證 (NESdev)
在 NESdev 的歷史中，Blargg 曾經寫過測試 PPU open-bus 衰減時間的 Test ROM。當時社群就發現：
1. 這個衰減時間在每台**原廠**紅白機上都有個體差異（製程變異）。
2. 開機越久（機殼內部變熱），衰減越快，這直接證實了你利用的溫度特性。
3. **沒有任何一台 Famiclone** 能原生通過這個測試，直到後來高階硬體模擬器（如 FPGA）手動加入了假 timer 才 pass。

### 總結給你的建議
你的 Test ROM 是非常出色的**「正版 / 原廠晶片驗證器」**！
與其把它當作能在任何機器上跑的溫度計，不如把它包裝成：**「地表最硬核的原廠 Ricoh 晶片檢測 ROM」**。
只要讀出來的溫度會隨環境改變，就**100% 證明這是一顆原廠的 1980 年代 NMOS 晶片**；如果讀出來是死值或死溫度，那就是 Clone 或 FPGA！
